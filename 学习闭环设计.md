# CRM 智能分析系统 · 学习闭环设计

> 日期：2026-06-23
> 用途：定义"越用越聪明"能力的设计基线，与 `重构方案与计划.md` 配套，作为重构第 5 期的独立设计输入
> 定位：本文件只描述学习闭环；功能边界、视觉与页面结构仍以 `DESIGN.md` 和 `specs/001-crm-intelligent-analytics/` 为准
> 原则：自动沉淀的都是"候选"，必须管理员审核才生效；学习只动知识资产表，不动 AI 模型权重；学习过程本身全程审计

## 1. 设计目标

### 1.1 要解决的问题

当前系统是开环：用户提问 → AI 理解 → 返回结果 → 审计记录（止步）。
知识资产（别名、口径提示、正例、负例）只靠管理员手工维护或代码硬编码，使用记录进了审计就不再回流。
结果就是：同样的歧义会被反复踩、同样的成功问法不会沉淀为复用资产、口径冲突不会自动暴露、推荐只基于点击热度。

### 1.2 目标能力（能学到的）

- **业务知识越用越准**：术语归一、口径提示、正例、负例随使用自动增长（候选态），管理员审核后生效
- **推荐越用越贴**：基于真实使用画像 + 反馈信号 + 时间场景的个性化推荐
- **重复问题越用越快**：相似问题命中已验证知识，跳过补问直接进入执行
- **高频查询自动沉淀为看板模板候选**：高频成功查询自动生成看板模板候选，管理员审核后一键复用
- **口径冲突自动暴露**：同一术语在不同查询里被解析成不同口径时，自动生成"待收敛"待办

### 1.3 边界（做不到、不做的）

- **不更新 AI 模型权重**：学习只动知识资产表和画像表，不微调、不重训、不改模型参数。这是企业系统能上线的底线，保证稳定、可回滚、可审计
- **不自动改口径**：口径收敛必须走"冲突发现 → 待办提醒 → 管理员裁定"，不自动改写已发布口径，防止把局部误判收敛成全局错误
- **不做通用 AGI 式自我进化**：学习能力限定在 CRM 经营分析领域，不追求跨领域泛化
- **不绕过权限与审计**：学习过程产生的所有候选、审核、发布行为都进审计；学习产物（知识资产、看板模板）继承现有权限模型
- **不收集敏感数据**：学习只记录问题文本、执行轨迹、反馈信号、口径冲突，不记录结果数据本体（结果数据已在审计里，不重复存储）

### 2.2 关键差距（需要补的）

| 差距 | 说明 | 影响 |
|------|------|------|
| 知识资产无"候选"状态 | 当前只有 ACTIVE / INACTIVE，自动沉淀的资产无法与人工维护的资产区分 | 自动沉淀无法实现，只能全靠手工 |

| 无语义相似匹配 | 问题匹配靠关键词，没有 embedding 向量检索 | 同义不同说法的问题无法命中已验证知识 |

## 3. 闭环架构：四个学习层次

### 3.1 层次总览

| 层次 | 名称 | 难度 | 收益 | 是否进主干 |
|------|------|------|------|-----------|
| 第 1 层 | 使用统计 | 低（已具备） | 推荐更贴 | 已上线 |
| 第 2 层 | 显式反馈 | 低（前端加） | 知道对错 | 进主干，并入重构第 1 期 |
| 第 3 层 | 自动沉淀 | 中（核心闭环） | 越用越准 | 进主干，重构第 5 期 |
| 第 4 层 | 语义检索 | 高（可选） | 泛化强 | 视第 3 层效果再定 |

### 3.2 第 1 层：使用统计（已有，不动）

已有能力：记录每次查询的模板点击、时间槽分布、重跑次数、成功次数，用于"猜你想查"推荐。
本设计不改动这一层，只在第 2 层为其补充反馈维度。

### 3.3 第 2 层：显式反馈（前端小改 + 审计接入）

#### 3.3.1 用户可见入口

在 Web 分析结果详情页和企微报告页底部，增加轻量反馈区：

- **结果有用 / 没用**：两个按钮，点击即提交，无需理由
- **口径不对**：点击后展开一个文本框，让用户描述"哪个数字/口径不对"，可选填
- **想要更多维度**：点击后展开多选，让用户勾选"想加时间趋势/想加区域对比/想加负责人明细"等

反馈区必须遵守 DESIGN.md：
- 不长期占据主结果区，放在结果区底部、口径说明之后
- 用 ElMessage 反馈"已收到，感谢"，3 秒自动消失
- 反馈区不能挡住导出、追问等主操作

#### 3.3.2 后端处理

新增 `AnalysisResultFeedbackService`：

- 接收反馈，写入审计（新事件类型 `ANALYSIS_RESULT_FEEDBACK`），事件体含 queryId、feedbackType（USEFUL / NOT_USEFUL / CALIBRATION_ISSUE / DIMENSION_REQUEST）、可选文本
- 同步更新使用画像：positiveFeedbackCount / negativeFeedbackCount +1
- 口径纠错反馈（CALIBRATION_ISSUE）额外生成一条"口径待核"待办，进治理待办队列（见 3.4.4）
- 反馈不立即改动任何知识资产，只作为第 3 层自动沉淀的输入信号

#### 3.3.3 企微端反馈（基于企微原生 feedback_event，无需自建）

> 重要发现：企微官方原生支持用户对机器人回复进行反馈（参考企微开发者文档 `path/101027` 接收事件 - 用户反馈事件）。
> 反馈类型包括"准确(1)/不准确(2)/取消(3)"，不准确时可附带原因（与问题无关/内容不完整/内容有错误/数据分析错误）和用户输入的反馈内容。
> **企微端无需自建反馈按钮，直接复用企微原生能力，体验更好、用户更愿意点。**

企微端反馈实现：

1. **回复时设置 feedback.id**：在所有流式回复（stream）和模板卡片回复（template_card）中设置 `feedback.id`，值为本次查询的 queryId。设置后企微会在消息上展示"准确/不准确"反馈入口。

2. **处理 feedback_event 回调**：新增 `WecomFeedbackEventService` 处理企微推送的 feedback_event：
   - `type=1`（准确）→ 记录正面反馈，写入审计 `ANALYSIS_RESULT_FEEDBACK`（feedbackType=USEFUL）
   - `type=2`（不准确）+ `inaccurate_reason_list` → 记录负面反馈和原因，写入审计 `ANALYSIS_RESULT_FEEDBACK`（feedbackType=NOT_USEFUL + 原因映射）
     - 原因 1（与问题无关）→ 对应"问题理解错误"
     - 原因 2（内容不完整）→ 对应"结果缺失维度"
     - 原因 3（内容有错误）→ 对应"数据有误"
     - 原因 4（数据分析错误）→ 对应"口径错误"（CALIBRATION_ISSUE）
   - `type=2` + `content`（用户自述文本）→ 写入审计 `ANALYSIS_RESULT_FEEDBACK`（feedbackType=CALIBRATION_ISSUE + 用户文本）
   - `type=3`（取消）→ 忽略，不记录

3. **feedback_event 约束**：企微官方明确该事件**仅支持回复空包**（不能同时回复新消息或更新卡片），处理逻辑需兼容这一约束——收到反馈后只记录，不回复。

4. **与 Web 端反馈统一**：企微 feedback_event 和 Web 端自建反馈都写入同一个审计事件类型 `ANALYSIS_RESULT_FEEDBACK`，后端 `AnalysisResultFeedbackService` 统一处理，学习闭环沉淀器无需区分反馈来源。

对比原方案的改进：
- 原方案考虑"企微附反馈按钮 + 引导到 Web 端反馈页"，用户需跳出企微，体验差、转化低
- 新方案直接用企微原生 feedback_event，用户在企微内一键反馈，转化高、原因结构化（企微原生 4 类原因）
- 去掉了"在会话里收有用/没用短回复"的复杂语义识别，避免误判

### 3.4 第 3 层：自动沉淀（核心闭环）

这是"越用越聪明"的主引擎。新增 `KnowledgeSedimentationService`（知识沉淀器），定时 + 事件触发地从执行轨迹和反馈信号中生成候选知识资产。

#### 3.4.1 触发方式

- **定时触发**：每天凌晨跑一次，扫描前一天的执行轨迹
- **事件触发**：QUERY_SUCCEEDED / CLARIFICATION_REQUESTED / ANALYSIS_RESULT_FEEDBACK 事件发生时，异步触发增量沉淀（不阻塞主链路）

#### 3.4.2 沉淀规则（5 类候选）

| 候选类型 | 触发条件 | 生成内容 | 去重与置信度 |
|---------|---------|---------|-------------|
| 候选正例（VALIDATED_EXAMPLE） | 同一问题文本（归一化后）成功执行 ≥ 3 次，且无负面反馈，且未命中已有正例 | questionText + sqlHint（从执行轨迹提取）+ 适用主题 | 按 questionText 归一化去重；confidence = 成功次数 / 总尝试次数；evidenceCount = 成功次数 |
| 候选负例（NEGATIVE_EXAMPLE） | 同一问题文本被补问 ≥ 2 次仍无法补全，或被 QUERY_BLOCKED ≥ 1 次 | questionText + blockReason（从补问或阻断原因提取） | 按 questionText 归一化去重；confidence = 阻断次数 / 总尝试次数 |
| 候选别名（ALIAS） | 同一规范标签在用户问题里出现 ≥ 2 种不同说法，且这些说法未在已有 ALIAS 资产里 | canonicalLabel + synonyms + matchKeywords + hint | 按 canonicalLabel 去重；confidence = 不同说法种类数 |
| 候选看板模板 | 自由查询（非模板点击）成功执行 ≥ 5 次，且至少 1 次正面反馈，且问题可归一化为模板化问法 | 模板名 + 默认筛选 + 结果 block 组合 + 适用角色 | 按 questionText 归一化去重；confidence = 成功次数 × (1 + 正面反馈数) |
| 口径冲突待办 | 同一术语（如"新增商机"）在不同查询里被解析成不同字段或不同时间口径，且冲突出现 ≥ 2 次 | 冲突术语 + 冲突的两种解析 + 涉及的 queryId 列表 | 不生成资产，生成治理待办；进"口径收敛"队列 |

#### 3.4.3 候选状态机

```
[执行轨迹/反馈] --沉淀器--> PROPOSED(候选) --管理员审核--> ACTIVE(生效)
                                      \--管理员驳回--> REJECTED(驳回，保留记录)
ACTIVE --管理员停用--> INACTIVE --管理员启用--> ACTIVE
ACTIVE --发布--> 进入 published 快照，供 AI 理解层使用
```

关键约束：
- `PROPOSED` 状态的资产**不进入 AI 理解层的 few-shot 注入**，只有 `ACTIVE` 且已发布的资产才注入
- `REJECTED` 的候选保留记录，用于后续去重（避免同样的候选反复生成）和沉淀规则调优
- 候选有 `expiresAt` 字段，默认 30 天未被审核自动过期，避免候选堆积

#### 3.4.4 治理后台新增页

在治理后台新增"知识沉淀"一级菜单，含三个子页：

1. **候选审核**：列出所有 PROPOSED 候选，支持批量审核/驳回，每条候选展示 evidenceCount、confidence、来源 queryId（可跳转审计详情）、与已有资产的相似度
2. **口径收敛**：列出所有口径冲突待办，每条展示冲突术语、两种解析、涉及查询、建议收敛方向（由沉淀器给出，管理员裁定）
3. **沉淀效果**：展示沉淀器的运行报表——本周生成候选数、审核通过率、候选对补问率的下降贡献、候选对推荐点击率的提升贡献

治理后台页面必须遵守 DESIGN.md：
- 用紧凑页头 + 表格 + 右侧抽屉详情
- 候选列表用状态标签（PROPOSED 琥珀色、ACTIVE 绿色、REJECTED 灰色）
- 候选审核动作放页头右侧同行，不放第二层横幅
- 冲突待办用风险徽标（RiskLevelBadge）标识优先级

### 3.5 第 4 层：语义检索（可选，进阶）

如果第 3 层上线后，发现关键词匹配仍漏掉大量同义问题，再考虑引入 embedding：

- 用现有 AI 网关的 embedding 接口（OpenAI 兼容）把已发布的 VALIDATED_EXAMPLE 和高频问题向量化
- 新查询进来时，先做语义相似检索，命中相似度 > 阈值的已验证正例，直接跳过补问
- 向量存储优先用 SQLite + sqlite-vec 或 PostgreSQL + pgvector，不引入独立向量数据库

第 4 层**不进主干计划**，作为第 5 期上线后的效果观察项再决定。

## 4. 数据模型变更

### 4.1 语义知识资产表扩展（向后兼容）

在现有 `AnalysisSemanticKnowledgeAssetRecord` 基础上新增字段：

```typescript
export interface AnalysisSemanticKnowledgeAssetRecord {
  // 现有字段保留
  id: string;
  type: 'ALIAS' | 'TEMPORAL_FIELD_HINT' | 'ORGANIZATION_NORMALIZATION' | 'VALIDATED_EXAMPLE' | 'NEGATIVE_EXAMPLE';
  name: string;
  status: 'ACTIVE' | 'INACTIVE';
  matchKeywords: string[];
  canonicalLabel?: string;
  synonyms?: string[];
  questionText?: string;
  sqlHint?: string;
  hint?: string;
  blockReason?: string;
  updatedBy: string;
  updatedAt: string;

  // 新增字段（学习闭环用）
  source: 'MANUAL' | 'AUTO_DERIVED';           // 来源：手工维护 or 自动沉淀
  reviewStatus: 'PROPOSED' | 'ACTIVE' | 'REJECTED' | 'EXPIRED';  // 审核状态
  derivedFromQueryIds?: string[];               // 候选来源的查询 ID（仅 AUTO_DERIVED）
  evidenceCount?: number;                        // 证据次数（仅 AUTO_DERIVED）
  confidence?: number;                           // 置信度 0-1（仅 AUTO_DERIVED）
  proposedAt?: string;                           // 候选生成时间（仅 AUTO_DERIVED）
  reviewedBy?: string;                           // 审核人（仅 AUTO_DERIVED）
  reviewedAt?: string;                           // 审核时间（仅 AUTO_DERIVED）
  expiresAt?: string;                            // 候选过期时间（仅 PROPOSED）
}
```

兼容策略：
- 现有资产默认 `source = 'MANUAL'`、`reviewStatus = 'ACTIVE'`
- 新增字段全部可选，不破坏现有数据
- `status` 字段保留（ACTIVE / INACTIVE），与 `reviewStatus` 并存：`status` 控制是否生效，`reviewStatus` 记录审核生命周期

## 5. 关键流程设计

### 5.1 流程一：候选正例沉淀与生效

```
用户提问"本月各销售负责人新增商机金额排名"
  → AI 理解执行（QUERY_SUCCEEDED，记录 questionText、scope、resultSummary）
  → 用户点"结果有用"（ANALYSIS_RESULT_FEEDBACK，USEFUL）
  → [当天凌晨] 沉淀器扫描：该 questionText 归一化后成功 ≥ 3 次、无负面反馈
  → 生成候选正例（KNOWLEDGE_ASSET_PROPOSED，source=AUTO_DERIVED，confidence=1.0，evidenceCount=3）
  → 治理后台"候选审核"页出现该候选
  → 管理员审核通过（KNOWLEDGE_ASSET_APPROVED）
  → 候选转 ACTIVE，进入下次发布快照
  → [下次发布] AI 理解层 few-shot 注入该正例
  → 后续相似问题命中，跳过补问，直接执行
```

### 5.2 流程二：候选负例沉淀与生效

```
用户提问"看一下商机转化怎么样"（缺时间、缺对象）
  → AI 识别歧义，补问（CLARIFICATION_REQUESTED，记录 questionText、缺失条件）
  → 用户未补全或多次补问仍失败
  → [当天凌晨] 沉淀器扫描：该 questionText 归一化后被补问 ≥ 2 次仍无法补全
  → 生成候选负例（KNOWLEDGE_ASSET_PROPOSED，type=NEGATIVE_EXAMPLE，blockReason="缺少时间范围和分析对象"）
  → 管理员审核通过
  → [下次发布] AI 理解层注入该负例
  → 后续类似问题，AI 直接补问"请补充时间范围和分析对象"，不再尝试执行
```

### 5.3 流程三：候选别名沉淀与生效

```
多个用户提问时用了不同说法指代"销售负责人"：
  "各销售排名"、"负责人业绩"、"业务员排行"、"销售员表现"
  → [当天凌晨] 沉淀器扫描：这些说法归一化后都命中 owner-performance-ranking 主题，但不在已有 ALIAS 资产里
  → 生成候选别名（type=ALIAS，canonicalLabel="销售负责人"，synonyms=["销售","负责人","业务员","销售员"]）
  → 管理员审核通过
  → [下次发布] AI 理解层注入该别名
  → 后续用户说"业务员排行"，AI 直接归一为"销售负责人"，无需补问
```

### 5.4 流程四：口径冲突发现与收敛

```
用户 A 提问"新增商机" → AI 解析为 opportunities.created_at（按创建时间）
用户 B 提问"新增商机" → AI 解析为 opportunities.updated_at（按更新时间）
  → [当天凌晨] 沉淀器扫描：同一术语"新增商机"被解析成两种不同字段
  → 生成口径冲突待办（CALIBRATION_CONFLICT_DETECTED，含两种解析、涉及 queryId）
  → 治理后台"口径收敛"页出现该待办，风险徽标标为"中"
  → 管理员裁定：新增商机统一按 created_at
  → 管理员新建/更新 ORGANIZATION_NORMALIZATION 资产，固化口径（CALIBRATION_CONFLICT_RESOLVED）
  → [下次发布] AI 理解层注入该口径提示
  → 后续所有"新增商机"统一按 created_at
```

### 5.5 流程五：候选看板模板沉淀

```
多个用户自由提问"广州办渠道下单汇总"，成功执行 ≥ 5 次，且至少 1 次正面反馈
  → [当天凌晨] 沉淀器扫描：该自由查询符合候选看板模板条件
  → 生成候选看板模板（DASHBOARD_TEMPLATE_PROPOSED，含默认筛选、结果 block 组合）
  → 治理后台"候选审核"页出现该候选
  → 管理员审核通过，配置适用角色（DASHBOARD_TEMPLATE_APPROVED）
  → 候选转 ACTIVE，进入"常用查询"看板分类
  → 后续用户在常用查询区看到"广州办渠道下单汇总"看板模板，一键生成
```

## 6. 治理与安全约束

### 6.1 候选必须审核

- `PROPOSED` 状态的候选**绝不**进入 AI 理解层 few-shot 注入，绝不影响查询执行
- 只有 `ACTIVE` 且已发布的资产才注入
- 审核动作（通过/驳回）全部进审计，含审核人、审核时间、原始候选、审核理由

### 6.2 候选去重与过期

- 沉淀器按 questionText 归一化（去空格、去标点、同义词替换）去重，避免同一候选反复生成
- 已被驳回的候选，30 天内不再生成（避免反复打扰）
- 候选默认 30 天未审核自动过期（KNOWLEDGE_ASSET_EXPIRED），过期后可重新生成

### 6.3 口径收敛不自动

- 口径冲突只生成待办，**绝不**自动改写已发布口径
- 收敛动作必须管理员显式裁定，并新建/更新 ORGANIZATION_NORMALIZATION 资产固化
- 收敛后的口径通过发布机制生效，可回滚

### 6.4 学习产物继承权限

- 知识资产、看板模板的可见范围继承现有权限模型
- 候选看板模板的适用角色由管理员在审核时配置
- 候选正例/负例/别名是全局知识，不按角色隔离（口径统一是系统级要求）

### 6.5 不收集结果数据本体

- 学习只记录问题文本、执行轨迹摘要（scope、resultSummary 中的指标名和数值范围）、反馈信号
- 不把结果明细表、客户名单、合同详情存入候选资产
- 候选资产的 questionText 最大 200 字符（与现有 schema 一致），sqlHint 最大 200 字符，避免存大段结果

### 6.6 学习过程可关停

- 沉淀器有全局开关（治理后台配置项），可随时关停
- 关停后不影响现有查询、推荐、知识资产使用，只停止生成新候选
- 关停动作进审计

## 7. 与 AI 理解层的对接

### 7.1 few-shot 注入改造

当前 `analysis-topic-report.registry.ts` 和 `ai-capability-pack.runtime.ts` 的 validatedExamples / fewShotExamples 硬编码在代码里。
改造为：从已发布的语义知识资产表中动态读取 VALIDATED_EXAMPLE 和 NEGATIVE_EXAMPLE，与代码里的基线正例合并注入。

注入优先级：
1. 代码基线正例（保底，确保系统始终有最低能力）
2. 已发布的人工维护正例
3. 已发布的自动沉淀正例（审核通过的候选）
4. 已发布的负例（作为 few-shot 的负样本，提升拒答准确性）

### 7.2 别名与口径提示注入

ALIAS、TEMPORAL_FIELD_HINT、ORGANIZATION_NORMALIZATION 资产在 AI 理解层的提示词中作为"术语归一规则"和"口径约束"注入。
当前 `analysis-query-knowledge.service.ts` 已有 `listPublishedAssets()` 方法读取已发布资产，只需确保新类型资产被正确读取。

### 7.3 不动 AI 模型权重

学习闭环的所有产物都通过 prompt 注入（few-shot、术语规则、口径约束），不涉及模型微调、不涉及权重更新。
这保证学习可回滚（回滚发布版本即撤销学习产物）、可审计（每个产物有来源 queryId 和审核记录）。

## 8. 实施分期

学习闭环作为重构的第 5 期，内部再拆为三个子阶段：

| 子阶段 | 内容 | 依赖 |
|--------|------|------|
| 5.1 数据模型与审计 | 扩展知识资产表、使用画像表、新增审计事件类型、候选状态机 | 无（可先行） |
| 5.2 显式反馈 | 前端反馈入口、`AnalysisResultFeedbackService`、使用画像反馈维度 | 5.1 |
| 5.3 自动沉淀 | `KnowledgeSedimentationService`、定时 Job、治理后台候选审核页、口径收敛页、沉淀效果页、AI 理解层注入改造 | 5.1 + 5.2 |

第 4 层（语义检索）不进本计划，作为 5.3 上线后的效果观察项。

## 9. 验收口径

| 项目 | 验收点 |
|------|--------|
| 显式反馈 | Web 和企微结果页有反馈入口，反馈写入审计，使用画像反馈维度有值 |
| 候选沉淀 | 成功查询 ≥ 3 次自动生成候选正例；补问失败 ≥ 2 次自动生成候选负例；同义说法 ≥ 2 种自动生成候选别名；自由查询 ≥ 5 次自动生成候选看板模板 |
| 候选审核 | 治理后台有候选审核页，支持批量审核/驳回，审核动作进审计，候选 30 天未审核自动过期 |
| 口径收敛 | 同一术语两种解析自动生成口径冲突待办，管理员可裁定并固化，固化后全系统统一 |
| 注入生效 | 审核通过的正例/负例/别名在下次发布后注入 AI 理解层，相似问题命中后跳过补问 |
| 效果可观测 | 治理后台有沉淀效果报表，展示候选生成数、审核通过率、补问率下降、推荐点击率提升 |
| 安全约束 | PROPOSED 状态资产不注入 AI 理解层；候选不收集结果数据本体；学习可全局关停；全部行为可审计 |
| 回归 | 现有查询、推荐、知识资产治理主链路回归用例全部通过；现有 ACTIVE 资产不受影响 |

## 10. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 候选堆积，管理员审核不过来 | 候选 30 天自动过期；按 confidence 降序排；支持批量审核；低 confidence 候选可配置自动忽略 |
| 脏知识污染（用户误反馈导致候选错误） | 候选必须审核；evidenceCount ≥ 3 才生成候选；单一用户反馈不直接生成候选（需多人或多次） |
| 口径收敛误判（管理员裁定错误） | 收敛走发布机制，可回滚；每次收敛进审计，可追溯 |
| 沉淀器拖慢系统 | 沉淀器异步执行，定时 Job 在凌晨；事件触发用队列，不阻塞主链路 |
| 注入过多 few-shot 导致 prompt 膨胀 | 每类资产注入上限（如正例最多 20 条），按 confidence 降序取 Top N；超限的资产不注入但不删除 |
| 用户不点反馈，反馈数据稀疏 | 反馈是辅助信号，不是唯一信号；沉淀器主要依赖执行轨迹（成功/补问/阻断），反馈只提升 confidence |
