# CRM 智能分析最终目标落地方案

## 一、最终目标定义

本项目最终目标不是单点看板，也不是普通聊天机器人，而是把「渠道 CRM 系统机器人」建设成企业微信里的 CRM 智能经营分析入口。

用户在企业微信里用自然语言提问，机器人按当前企微用户映射到 CRM 用户，继承 CRM 权限范围，读取真实 CRM OpenAPI 数据，返回可解释、可追问、可看板化、可审计的经营分析结果。

最终交付形态：

1. 用户能在企业微信里问经营数据、看风险、看排行、看管理看板。
2. 机器人回答必须包含数据口径、权限口径、核心指标、经营判断、明细摘要、下一步动作。
3. 复杂问题以「企微长连接文本摘要 + 图片看板 + 模板卡片 + 可选完整 HTML 报告」组合呈现；最终验收不依赖正式域名。
4. 同类问题使用统一模板，不按 200 多个问题逐条硬编码。
5. 事实数据只来自 CRM 标准开放接口或经过批准的只读受控接口，AI 不编造数字。
6. 所有提问、接口调用、权限裁剪、结果摘要、异常和用户反馈均可审计。
7. 后续支持日报、周报、月报主动推送和风险预警，逐步形成管理层经营节奏助手。

## 三、目标拆解

### 3.1 用户体验目标

用户提问示例：

```text
本月经营总览情况
报备到订单转化漏斗数据看板
本月渠道贡献排名前十
哪些商机 30 天内预计签约但还没有报价
哪些报价最可能本周转订单
生成本周经营会复盘要点
帮我看全国代理商发展运营数据看板
```

机器人理想答复结构：

```text
【问题结论】
一句话说明当前经营状态。

【数据口径】
说明时间范围、权限范围、筛选条件、数据来源。

【核心指标】
展示报备、商机、报价、订单、金额、转化率、排名等关键指标。

【关键发现】
指出断点、风险、高贡献对象、低活跃对象、异常变化。

【建议动作】
给出负责人可执行的下一步动作。

【完整看板】
优先在企微长连接内返回核心看板内容；如已配置报告地址，再提供完整 HTML 报告链接。

【边界说明】
说明权限不足、字段缺失、样本不足或接口缺口。
```

### 3.2 业务能力目标

| 能力域 | 目标能力 | 最终验收口径 |
| --- | --- | --- |
| 经营总览 | 报备、商机、报价、订单、金额、转化率、目标缺口 | 能回答当前周期经营健康度和最大风险 |
| 业务漏斗 | 报备到商机、商机到报价、报价到订单 | 能指出最大流失环节和改进动作 |
| 渠道排行 | 渠道贡献、渠道集中度、渠道活跃度 | 能返回当前权限范围内渠道 TopN 和风险渠道 |
| 区域对比 | 大区、区域、负责人维度贡献 | 能识别高贡献、低活跃、可追回区域 |
| 渠道画像 | 合作级别、分销层级、技术服务商类型 | 能给出升级、激活、淘汰候选 |
| 客户报备 | 待审批、保护期到期、重复报备 | 能输出待处理清单和优先级 |
| 商机风险 | 阶段停滞、预计签约过期、高金额未报价 | 能输出风险清单、负责人和建议动作 |
| 报价订单 | 高金额报价未转订单、订单待确认 | 能输出临门一脚清单和转化建议 |
| 权限口径 | 当前用户可见范围、不可见原因 | 能明确说明为什么不能看全国或全量 |
| 经营节奏 | 日报、周报、月报、经营会复盘 | 能生成管理汇报摘要和风险待办 |

### 3.3 技术能力目标

| 技术模块 | 最终目标 |
| --- | --- |
| 企微接入层 | 长连接稳定运行，支持文本、模板卡片、图片、反馈事件、重连和去重 |
| 身份映射层 | 企微用户到 CRM 用户稳定绑定，支持异常诊断和人工修复 |
| 权限层 | 每次查询都继承 CRM 权限裁剪，机器人侧补充准入控制和口径说明 |
| 意图层 | 支持自由问法映射到受控分析模板，低置信时澄清，不直接乱查 |
| 查询编排层 | 优先 CRM OpenAPI，接口不足时只允许进入受控只读兜底 |
| 事实计算层 | 指标、转化率、排行、风险阈值由程序计算，不由 AI 编造 |
| AI 解读层 | 基于事实结果生成中文解释、建议、追问，不新增事实 |
| 模板展示层 | 统一输出企微文本、图片、模板卡片、HTML 报告 |
| 审计诊断层 | 保留问题、用户、权限、接口、结果摘要、异常、反馈 |
| 管理前端 | 支持 AI、企微、CRM OpenAPI、用户映射、灰度、日志和诊断配置 |

## 四、整体实施路线

建议分为五个阶段推进。前两个阶段已经完成核心闭环，但仍需补齐验收矩阵。

### 阶段 0：收敛与基线确认

目标：确保项目只围绕 CRM 智能分析推进，不再被历史写回、合同、日报旧能力带偏。

已完成和需保持：

1. 收敛老方案，明确以更新版落地方案为准。
2. 第一阶段不纳入合同口径、不纳入证书认证口径。
3. 以 CRM OpenAPI 只读数据作为事实来源。
4. 以企微机器人作为第一入口。
5. 历史能力只作为参考，不整包恢复。

验收标准：

1. 文档中不再承诺合同金额、合同阶段、证书认证。
2. 新增实现都能对应最终需求目标。
3. 所有分析口径有明确数据来源和边界。

### 阶段 1：企微问数基础闭环

目标：用户能在企微里问核心经营数据，并得到可信答案。

主要任务：

1. 企微机器人长连接稳定接入。
2. CRM OpenAPI 鉴权、令牌刷新、超时和错误处理。
3. 企微用户到 CRM 用户映射。
4. 经营总览、漏斗、渠道排行、区域排行、负责人排行。
5. 报备、商机、报价、订单风险清单。
6. 回答中展示数据范围、权限范围、筛选条件和数据来源。
7. 无法回答时说明字段缺失、接口缺失、权限不足或样本不足。
8. 记录问答、接口调用、权限口径和异常审计。

当前状态：

1. 核心问数链路已有。
2. 企微真实会话已可返回 CRM 分析。
3. 仍需补齐多角色权限矩阵验收和管理前端入口。

补齐任务：

| 任务 | 优先级 | 负责人角色 | 验收方式 |
| --- | --- | --- | --- |
| 超管、区域、渠道、员工四类账号真实问数测试 | P0 | 测试负责人 | 同一问题四类账号结果范围不同且说明清楚 |
| 未绑定用户和无权限用户提示 | P0 | 后端开发 | 返回绑定提示或权限不足，不调用业务数据 |
| CRM OpenAPI 自检接口整理 | P0 | 后端开发 | 能一键检测令牌、身份、权限、统计接口 |
| 企微问答日志查询 | P1 | 前端开发 | 管理员可查看最近问答和异常 |
| 灰度准入配置 | P1 | 后端与前端 | 可按人员、群、角色开启或关闭能力 |

### 阶段 3：完整问题集与模板层落地

目标：把 200 多个业务问题收敛成 12 套标准模板，不再靠单点特判。

12 套模板：

| 模板 | 状态目标 | 优先级 |
| --- | --- | --- |
| 经营总览卡 | 完整落地 | P0 |
| 业务漏斗诊断卡 | 完整落地 | P0 |
| 渠道贡献排行卡 | 完整落地 | P0 |
| 商机风险清单卡 | 完整落地 | P0 |
| 数据质量与权限口径卡 | 完整落地 | P0 |
| 客户报备与保护期卡 | 完整落地 | P1 |
| 报价与订单转化卡 | 完整落地 | P1 |
| 经营节奏日报/周报/月报卡 | 问询版落地 | P1 |
| 区域经营对比卡 | 完整落地 | P1 |
| 渠道画像诊断卡 | 完整落地 | P2 |
| 分销层级健康卡 | 字段确认后落地 | P2 |
| 技术服务商生态卡 | 先类型分析，证书不纳入 | P2 |

实施步骤：

1. 建立「问题意图 -> 模板类型 -> 数据口径 -> 查询计划 -> 输出组件」映射表。
2. 每套模板定义固定输入、输出、指标、明细、边界说明。
3. 每套模板至少准备 3 到 5 条验收提问。
4. 对需求问题清单按模板归类，先覆盖前 30 条高频问题，再扩展到 100 条，再扩展到全量。
5. 对无法支持的问题沉淀为接口缺口、字段缺口或权限缺口。

验收标准：

1. 30 条高频问题真实企微验收通过率不低于 90%。
2. 每条问题必须命中标准模板或明确说明不支持原因。
3. 同一模板不同问法输出结构一致。
4. AI 只做解释和建议，不新增事实指标。

### 阶段 4：管理前端与运维诊断

目标：让客户现场可配置、可诊断、可运维，而不是每次依赖开发改环境变量。

管理入口建议：`/governance/integrations`

功能范围：

| 功能 | 说明 | 优先级 |
| --- | --- | --- |
| AI 配置 | 沿用现有模型配置、健康检查、激活策略 | P0 |
| 企微机器人配置 | 机器人编号、密钥替换、长连接状态、重连策略、测试消息 | P0 |
| CRM OpenAPI 配置 | 接口地址、凭证替换、令牌刷新、自检 | P0 |
| 用户映射 | 企微用户与 CRM 用户绑定、解绑、导入、异常诊断 | P0 |
| 灰度准入 | 人员、群、角色、能力开关 | P1 |
| 消息日志 | 查询问题、回复摘要、耗时、状态、错误 | P1 |
| 审计日志 | 查询权限口径、接口调用、异常和用户反馈 | P1 |
| 诊断页 | AI、企微、CRM、用户映射、权限范围一键检查 | P0 |

安全要求：

1. 密钥类字段只允许保存和替换，不允许明文回显。
2. 日志中不得展示访问令牌、AppSecret、企微 Secret、AI Key。
3. 所有配置修改必须记录操作人、时间、修改对象和结果。
4. 测试消息和接口自检必须脱敏展示。

### 阶段 5：主动推送与经营节奏助手

目标：从「用户主动问」升级为「系统主动提醒和复盘」。

主要能力：

1. 每日晨会风险清单。
2. 每周经营会复盘报告。
3. 每月经营复盘报告。
4. 保护期到期提醒。
5. 预计签约过期提醒。
6. 高金额商机未报价提醒。
7. 报价长期未转订单提醒。
8. 二级订单待确认提醒。
9. 用户反馈「准确 / 不准确」闭环。
10. 常用追问推荐。

验收标准：

1. 推送对象、推送时间、推送群可配置。
2. 每条主动推送都说明数据范围和触发原因。
3. 用户可以关闭、调整或反馈推送。
4. 风险提醒不泄露无权限数据。
5. 管理员可查看推送历史和失败原因。

## 六、验收标准

### 6.1 阶段性验收标准

| 阶段 | 必须通过 |
| --- | --- |
| 阶段 1 | 企微能问核心经营数据，权限正确，回答有口径 |
| 阶段 2 | 企微能返回看板卡片、图片或报告链接，完整报告可打开 |
| 阶段 3 | 高频 30 问命中标准模板，问题集可持续扩展 |
| 阶段 4 | 管理员能在前端完成配置、诊断、映射和日志排查 |
| 阶段 5 | 日报、周报、月报和风险提醒可配置、可追踪、可反馈 |

### 6.2 最终验收标准

最终通过需要满足：

1. 企微机器人连续运行稳定，无频繁断连。
2. 超管、区域、渠道、员工四类账号权限验证通过。
3. 高频 30 问真实企微验收通过率不低于 90%。
4. 12 套模板至少 P0/P1 模板完成可用版本。
5. 每次回答都有数据口径、权限口径和边界说明。
6. 看板类问题可通过企微长连接直接返回模板化完整摘要，并可返回图片或卡片。
7. 如果启用 HTML 报告入口，链接必须使用企微客户端可访问的受控地址；未启用时不得影响企微长连接验收。
8. 日志中不泄露密钥、令牌、内部堆栈或敏感字段。
9. 管理前端可完成三方配置、用户映射和诊断。
10. 所有无法回答的问题必须明确说明原因，不编造数据。

## 九、关键风险与处理方案

| 风险 | 影响 | 处理方案 |
| --- | --- | --- |
| CRM 接口字段不足 | 某些问题无法回答 | 返回缺口说明，整理接口补充清单 |
| 订单样本过少 | 趋势和成交质量判断不稳 | 提示样本不足，只展示能力和当前事实 |
| 权限映射错误 | 可能越权或误判数据 | 每次查询强制绑定 CRM 权限，补多角色验收 |
| AI 编造数据 | 经营结论失真 | AI 只消费事实包，数字由程序计算 |
| 企微消息过长 | 用户阅读困难 | 分段输出，默认只展示前 5 到 10 条 |
| 未配置报告地址 | 用户无法打开 HTML 报告 | 不作为必需验收项；企微长连接内必须返回足够完整的模板化结果，HTML 报告只作为可选增强 |
| 密钥泄露 | 安全事故 | 前端不回显密钥，日志脱敏，报告不记录密钥 |
| 旧能力回流导致发散 | 项目复杂化 | 坚持只读分析主线，写回和合同单独评审 |

## 十一、结论

距离最终目标已有可运行的第二阶段核心闭环，但还没有达到完整产品化交付。

当前最重要的不是继续增加零散问法，而是把能力收敛成「权限正确、模板统一、长连接可读、问题集可验收、管理可配置」五条主线。

只要按本方案补齐多角色权限矩阵、统一模板层、图片看板、企微长连接完整回复、管理诊断前端和高频问题集验收，项目就可以从当前的阶段性联调成果推进到可试点、可运维、可持续扩展的 CRM 智能经营分析助手。
