# 原始需求

## 1. 项目背景

公司已经有一套 CRM 系统，核心数据沉淀在商机、合同、客户、组织、部门、用户、角色、权限等模块中。当前商机分析主要依赖人工查数、手工汇总和导出，效率低，且存在数据被大批量导出的安全风险。

管理层和销售角色希望能够更方便地查询经营数据，通过自然语言输入直接获得商机分析结果，但该能力必须严格受控，不能绕开现有 CRM 的权限体系，也不能演变为对生产数据库的任意 SQL 查询入口。

同时，销售角色有很大一部分时间在外部拜访客户、出差或移动办公，无法始终通过 Web 页面进入系统。因此首期需要新增企业微信机器人作为核心查询入口之一，让用户能够在企业微信中以对话形式直接向机器人提问，并实时获得分析结果。

## 2. 项目目标

建设一个“AI 商机分析助手”，面向公司管理层和授权销售管理人员，提供以下能力：

1. 基于现有 CRM 数据做商机分析和合同关联分析。
2. 支持自然语言输入，自动转换为受控的只读 SQL 查询。
3. 查询结果自动继承现有 CRM 的组织、部门、角色和数据权限。
4. 支持通过企业微信机器人以对话形式发起查询，并将结果流式返回给企业微信会话。
5. 严格限制大批量导出，降低数据泄露风险。
6. 对所有查询、导出、连接异常和安全拦截操作进行审计留痕。

## 3. MVP 范围

### 3.1 数据范围

MVP 只允许访问以下业务域和表的白名单数据：

- 商机：`opportunities`
- 合同：`contracts`
- 客户：`customers`
- 用户与组织：`users`、`departments`、`organizations`
- 权限基础信息：`roles`、`permissions`

### 3.2 业务范围

MVP 只支持以下商机分析场景：

1. 商机数量、金额、阶段分布、阶段转化情况
2. 按负责人、部门、组织查看商机分析结果
3. 商机与合同的关联分析，如商机转合同数量、金额、转化率
4. 时间维度分析，如本周、本月、本季度的新增趋势和成交趋势
5. 排行类分析，如销售负责人商机金额排名、部门成交排名

### 3.3 交互入口范围

MVP 首期包含以下交互入口：

1. 企业微信机器人对话入口，面向授权销售和管理角色进行自然语言查询
2. Web 管理入口，面向系统管理员和数据管理员进行白名单、权限、审计和策略配置

### 3.4 不在 MVP 范围内

以下能力不属于首期范围：

1. 任意表、任意字段的全库开放查询
2. 非只读 SQL 执行
3. 写入、更新、删除 CRM 数据
4. 复杂跨域经营分析，例如回款、发票、费用全链路分析
5. 无限制导出
6. 面向普通员工的开放问数入口
7. 企业微信之外的外部 IM 平台接入
8. 机器人脱离 CRM 权限体系独立提供问数能力

## 4. 目标用户

1. 销售总监：查看全公司或授权范围内的商机整体情况
2. 区域经理/部门负责人：查看本部门或本团队的商机分析结果
3. 外勤销售或授权销售人员：在企业微信中查询本人或授权范围内的客户、商机和合同分析结果
4. 系统管理员/数据管理员：配置白名单、权限、审计、导出策略和企业微信机器人连接策略

## 5. 核心用户故事

### 用户故事 1

作为销售总监，我希望直接在企业微信中输入“本月各销售负责人新增商机金额排名”，机器人可以以对话形式持续输出可读的分析结果，这样我不用再让业务同学手工出报表。

### 用户故事 2

作为区域经理，我希望在企业微信中查询“我负责部门近 3 个月商机阶段分布和转合同率”，系统只返回我有权限查看的数据，并在结果较长时分段流式输出。

### 用户故事 3

作为外勤销售，我希望在拜访客户途中通过企业微信向机器人追问“这个客户近 30 天有多少新增商机、当前负责人是谁”，机器人能够在同一会话中继续理解上下文并返回结果，这样我不需要回到电脑前操作 Web 系统。

### 用户故事 4

作为系统管理员，我希望限制 AI 查询的可访问表和字段，并对查询、导出、连接状态和异常行为做审计，这样可以保证数据安全和机器人服务稳定性。

## 6. 功能需求

### FR-1 商机分析查询

系统应支持用户通过自然语言查询商机分析数据，并返回结构化结果。

验收标准：

1. 用户可以输入商机分析类自然语言问题。
2. 系统能够返回表格结果、汇总指标或简要结论。
3. 查询结果至少支持时间、负责人、部门、组织四类常见筛选维度。
4. 当问题超出系统支持范围时，系统给出明确提示，而不是直接报错。

### FR-1A 企业微信机器人入口

系统应提供企业微信机器人作为首期自然语言查询入口，支持用户通过企业微信会话发起问数请求。

验收标准：

1. 已授权用户可以在企业微信中与机器人建立会话并发送自然语言问题。
2. 机器人能够识别当前会话所属用户身份，并映射到 CRM 的用户、角色和数据权限。
3. 机器人支持在同一会话中承接连续追问，而不是把每次消息都当作完全独立的问题。
4. 未授权用户或未完成身份映射的用户无法通过机器人获取 CRM 分析结果。

### FR-2 自然语言转受控 SQL

系统应将自然语言问题转换为受控的只读 SQL，并在执行前进行校验。

验收标准：

1. 系统只允许生成和执行 `SELECT` 查询。
2. 系统禁止执行 `INSERT`、`UPDATE`、`DELETE`、`ALTER`、`DROP`、`TRUNCATE`、`OUTFILE` 等语句。
3. 系统只允许访问白名单表和白名单字段。
4. 系统在执行前对生成 SQL 做语法和安全校验。
5. 当 SQL 不合法或超出安全边界时，系统拒绝执行并返回原因。

### FR-3 权限控制

系统必须复用现有 CRM 权限体系，不能另起一套独立权限模型。

验收标准：

1. 查询结果必须自动继承当前用户的组织、部门、负责人等数据范围。
2. 无权限用户不能通过 AI 查询获取越权数据。
3. 管理员可以配置哪些角色可以使用 AI 查询功能。
4. 管理员可以配置哪些角色允许导出数据。

### FR-4 导出控制

系统应限制大批量导出，降低数据外泄风险。

验收标准：

1. 默认查询结果仅在页面内查看，不自动导出。
2. 单次导出条数应受限，建议首版上限为 1000 行。
3. 超出阈值时系统直接拒绝导出，或进入审批流程。
4. 同一用户每日导出次数应受限，建议首版上限为 3 次。
5. 导出行为必须记录导出人、导出时间、导出条件、导出条数。

### FR-5 审计与追踪

系统应记录所有关键操作，满足安全审计要求。

验收标准：

1. 记录用户原始问题、生成 SQL、执行时间、返回条数、执行结果。
2. 记录导出行为和失败原因。
3. 记录越权访问尝试和安全拦截事件。
4. 管理员可以按用户、时间范围查看审计记录。

### FR-6 结果展示与可解释性

系统应提供便于管理层理解的结果展示，并让查询过程具备一定可解释性。

验收标准：

1. 系统返回的结果包含业务化标题，而不是只展示原始字段名。
2. 系统能够提示本次查询基于哪些数据范围执行。
3. 系统在需要时可以展示“查询口径说明”或“SQL 已受控执行”的提示。
4. 当没有查询到数据时，系统返回明确说明。

### FR-7 企业微信流式输出

系统应支持机器人将查询结果以流式方式返回到企业微信会话，而不是长时间无响应后一次性输出全部结果。

验收标准：

1. 对于耗时较长的查询，机器人先返回“已开始分析”或处理中提示，再逐步输出结果内容。
2. 当结果较长时，机器人能够按段落、表格摘要或结论块分段返回，避免企业微信消息展示异常。
3. 流式输出过程中若查询失败、超时或被拦截，机器人必须立即终止后续输出并返回明确原因。
4. 流式输出的内容仍需遵守权限范围、敏感字段屏蔽和导出限制。

### FR-8 会话连接与保活

系统应管理企业微信机器人与后端分析服务之间的长连接状态，保证会话稳定性和结果返回可靠性。

验收标准：

1. 系统支持通过 WebSocket 等长连接方式维持查询会话。
2. 系统具备连接心跳与超时检测能力，能识别失活连接并及时释放资源。
3. 当连接中断时，系统应记录异常并向用户返回可理解的失败或重试提示。
4. 连接恢复后，新请求不受旧失活连接影响。
5. 多个用户同时连接机器人时，不同会话之间的上下文、权限范围和流式输出内容必须严格隔离，不能出现串话、串数据或结果错发。
6. 当同一用户在短时间内发起多次查询时，系统必须能够识别并管理会话顺序，避免旧查询结果覆盖新查询上下文。
7. 当并发连接或并发查询超过系统可承载范围时，系统必须优先保证会话可用性，并向用户返回排队、稍后重试或处理中提示，而不是直接失去响应。

## 7. 非功能需求

### NFR-1 安全性

1. AI 查询只允许只读访问。
2. 所有查询必须经过 SQL 安全校验。
3. 所有数据访问必须遵守现有权限模型。
4. 敏感字段应支持屏蔽或不纳入白名单。

### NFR-2 性能

1. 常见聚合查询应在 5 秒内返回结果。
2. 复杂查询超时上限建议为 15 秒。
3. 超时后应给出失败提示或建议改写问题。
4. 系统应支持多个企业微信用户同时在线查询，不因单个长查询阻塞其他用户会话。
5. 长连接会话应具备心跳检测、超时回收和异常断开处理能力，避免连接泄漏。
6. 流式输出期间系统应保证消息顺序正确，避免重复推送、乱序推送或消息丢失。
7. 首期建议至少支持 200 个并发在线企业微信连接会话，以及 50 个并发进行中的查询任务。
8. 在并发高峰下，系统仍应优先保证首条响应及时返回，避免用户长时间感知为“无响应”。
9. 并发场景下同一用户与不同用户之间的会话上下文必须严格隔离，不能出现消息串流、结果串发或权限串用。
10. 高并发场景下系统应支持限流、排队或降级提示，避免因瞬时流量导致整体服务雪崩。

### NFR-3 可审计性

1. 所有关键操作必须留痕。
2. 审计日志应支持检索和追踪。

### NFR-4 可维护性

1. 白名单表、字段、角色权限应可配置。
2. 查询模板和问法示例应可扩展。
3. 企业微信机器人配置、连接阈值、心跳周期和并发策略应可配置。

## 8. 风险与约束

1. 生产库中业务表较多，且显式外键约束较少，SQL 生成必须严格限定范围。
2. AI 查询能力如果不设边界，容易成为越权查数入口。
3. 首期建议只聚焦商机和合同域，不直接覆盖回款、发票、费用等更复杂链路。
4. 企业微信机器人需要处理多用户同时在线、流式返回和长连接保活，若连接治理不完善，容易出现消息堆积、会话错乱或资源泄漏。
5. 企业微信会话天然偏移动场景，结果展示必须兼顾可读性和分段输出能力，避免长消息影响使用体验。
6. 并发高峰期若缺少限流、排队和会话隔离机制，容易出现机器人响应变慢、结果串话或大面积连接断开。
7. 首期以企业微信机器人问数为主要查询入口，以 Web 管理端为治理入口，不对外开放。

## 9. 首期成功标准

1. 管理层和授权销售角色可以直接在企业微信中查询常见商机分析问题，不再依赖人工临时报表。
2. 查询结果不突破现有 CRM 权限边界。
3. 机器人能够以流式方式稳定返回结果，且在长查询时不给用户造成“无响应”体验。
4. 多用户同时通过企业微信连接机器人时，查询性能和结果正确性保持稳定。
5. 导出、连接异常、越权访问和查询行为均可控且可审计。
6. 首期并发压测达到目标值后，多个用户同时连接与查询时，不出现串话、串数据、重复推送或明显消息乱序。
7. 首期上线后，至少覆盖 10 个高频商机分析问法。
