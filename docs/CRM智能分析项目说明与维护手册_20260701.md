## 二、项目定位和能力边界

### 2.1 项目定位

本项目是“CRM 智能分析 + 企业微信机器人 + Web 工作台”的一体化交付项目，目标是让业务用户在企业微信里用自然语言询问 CRM 经营问题，并得到：

1. 企微模板卡片：首屏摘要和完整报告入口。
2. 长连接 Markdown 正文：可独立阅读的经营分析、维度判断和建议动作。
3. 图片看板：用于复杂场景的视觉化摘要。
4. HTML 完整报告：用于查看指标、明细、地图和分区内容。
5. 300 问模板覆盖：覆盖经营总览、漏斗、排行、区域、渠道画像、报备保护、商机风险、报价转订单、续费、产品方案、技术服务商、分销层级、经营节奏、数据质量权限等问题。

### 2.2 明确边界

| 边界 | 当前口径 |
| --- | --- |
| 数据源优先级 | CRM OpenAPI 实时数据优先，快照仅用于本地验收、降级或历史回放口径 |
| 权限范围 | 只分析当前企微用户绑定 CRM 账号可见范围，不补齐或推断无权限数据 |
| 订单口径 | 订单不足时必须明确说明，不能把商机或报价等同真实成交 |
| 图片看板 | 是展示增强，不是唯一交付形态；图片失败不能影响卡片和正文 |
| 真实企微验收 | 自动化结构验收通过后，仍需要真实企业微信端截图留档 |

## 六、数据源和口径维护

### 6.1 数据源优先级

1. CRM OpenAPI 实时数据：当前正式主链优先使用。
2. CRM MySQL 只读库：用于受控查询、历史能力和分析仓库相关路径。
3. OpenAPI Markdown 快照：用于本地验收、开发调试或明确降级场景。
4. `.runtime/app-storage.json`：保存本地应用态、会话、配置和查询结果元数据。

### 6.2 快照使用说明

如果日志或回复出现“快照”字样，必须确认是否处于以下场景：

- 本地验收脚本显式启用了快照。
- OpenAPI 实时链路不可用后进入降级口径。
- 开发环境为了避免真实 CRM 调用而使用本地样本。

正式交付口径中，应优先说明“CRM OpenAPI 实时数据”。如果使用快照，回复中必须明确提示“当前快照边界”，不能伪装成实时数据。

### 6.3 权限口径

所有分析回复必须保留以下原则：

- 只使用当前用户可见范围。
- 无权限数据不补齐、不推断。
- 全国、全平台、全部渠道类问题必须确认当前用户是否具备对应权限。
- 字段缺失、接口异常、订单不足时必须说明影响。

## 九、常见故障和排查

### 9.1 后端启动失败：`EADDRINUSE: address already in use :::3001`

原因：3001 端口已有后端或其他进程占用。

处理：

```bash
lsof -nP -iTCP:3001 -sTCP:LISTEN
kill <PID>
PORT=3001 pnpm --dir backend dev:raw
```

### 9.2 企业微信监听锁提示已有进程持有

典型日志：

```text
企业微信机器人入站监听未启动，因为本机已有另一个进程持有监听锁
```

处理步骤：

1. 查看锁文件：`cat .runtime/wecom-bot-listener.lock`
2. 检查 PID 是否存在：`ps -p <PID>`
3. 如果 PID 已不存在，删除锁：`rm -f .runtime/wecom-bot-listener.lock`
4. 如果 PID 仍存在，先判断是否旧后端，再决定是否停止。

### 9.3 机器人只回复文字，没有卡片

优先检查：

1. `WecomStreamDispatcherService` 是否收到 `templateCards`。
2. `WecomTransportService.sendTemplateCardMessage` 是否被调用。
3. 企业微信模板卡片是否包含 `card_action`、`jump_list`、`feedback.id`。
4. 当前是否处于 mock transport 或 SDK transport。
5. 企业微信是否拒收模板卡片，查看 delivery record 和日志。

当前设计中，Markdown 正文是必须可读的兜底；卡片和图片是增强，但模板卡片应正常发送。

### 9.4 回复只有数据，没有分析

当前已在 `WecomDashboardMarkdownRendererService` 增加：

- `【分析结论】`
- `【分维度分析】`
- `建议动作：`
- 各模板专属维度判断

如果后续再次出现只列数据，应检查是否绕过了动态看板主链，或是否走到了旧的普通分析链路。

重点检查文件：

```text
backend/src/modules/wecom/wecom-dashboard-markdown-renderer.service.ts
backend/src/modules/wecom/wecom-bot.service.ts
backend/src/modules/analysis/analysis-channel-presenter.service.ts
```

### 9.5 指定渠道商却返回区域总览

排查方向：

1. 实体识别是否识别到了渠道商名称。
2. 查询规划是否把渠道商条件传给执行器。
3. 执行结果是否仍返回区域聚合。
4. 企微展示层不能修复查询条件错误，只能呈现已有结果。

重点检查：

```text
backend/src/modules/analysis/analysis-intent.service.ts
backend/src/modules/analysis/analysis-query-planner.service.ts
backend/src/modules/analysis/analysis-query-executor.service.ts
backend/src/modules/analysis/lianruan-crm-analysis-executor.service.ts
```

### 9.6 OpenAPI 快照被误用

排查：

```bash
rg "OPENAPI_MARKDOWN_SNAPSHOT|CRM_OPENAPI_MARKDOWN_SNAPSHOT" backend/src backend/scripts
```

原则：

- 生产主链优先实时 CRM OpenAPI。
- 快照必须明确标注为开发、验收、降级或历史回放。
- 不允许在正式回复里把快照伪装成实时数据。

### 9.7 前端无法访问后端

排查：

1. 后端是否监听 3001。
2. 前端代理是否指向 `http://127.0.0.1:3001`。
3. 浏览器是否访问 `http://127.0.0.1:5173`。
4. 登录 Cookie 是否受跨域影响。

本地推荐让前端走 Vite 同源代理，不直接跨域访问后端。

## 十一、生产部署维护摘要

生产部署详细说明见：

```text
docs/architecture/生产部署指南.md
```

核心原则：

1. 前端通过 `pnpm --dir frontend build` 生成静态资源，由 Nginx 承载。
2. 后端通过 `pnpm --dir backend build` 生成 `backend/dist`，由 systemd 或 pm2 托管。
3. 后端默认端口 3001，只建议内网或本机 Nginx 回源访问。
4. `.runtime/` 必须放在持久化目录并定期备份。
5. 生产环境敏感配置放入独立环境变量文件，不要散落在仓库目录。
6. 企业微信回调、机器人密钥、CRM OpenAPI、AI 网关和数据库配置必须按环境隔离。

## 十三、维护红线

1. 不允许绕过权限范围直接查询或展示无权限 CRM 数据。
2. 不允许把快照数据伪装成实时 OpenAPI 数据。
3. 不允许把商机金额或报价金额解释成真实成交金额。
4. 不允许让图片看板失败阻断文字回复。
5. 不允许随意删除企业微信监听锁，必须先确认锁持有进程已退出。
6. 不允许在前端环境变量中放 AI、数据库、企业微信密钥。
7. 不允许改动态模板后不跑 14 类模板测试和 300 问验收。
8. 不允许只优化某一道问题而破坏 300 问全局模板映射。
