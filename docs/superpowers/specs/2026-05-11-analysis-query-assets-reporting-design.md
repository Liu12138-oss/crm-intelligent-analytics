# 智能分析结果报表化与查询资产升级方案

## 1. 背景

本方案面向当前 CRM 智能分析系统的一期工作台升级，目标是在不破坏现有自由 AI 问数主链路的前提下，补齐以下能力：

- 智能分析结果改为“数据在前，AI 分析报告在后”的分阶段展示。
- 常用查询与最近查询升级为可治理、可执行、可推荐的查询资产系统。
- 治理后台支持维护固定 SQL 查询模板、展示方式和推荐策略。
- 最近查询统一沉淀自由 AI 问数与固定 SQL 模板查询，并基于用户点击习惯与时间场景做排序推荐。

本次方案直接吸收 `公司 2026-1778481025047.json` 中的 Grafana 看板定义，作为首批查询模板的重要来源。需要特别说明的是，这份 JSON 是看板定义导出而不是结果数据导出，能够直接复用的是其 SQL 口径、面板分类、展示类型和经营分析主题，而不是某次运行时的统计值。

## 2. 现状与问题

### 2.1 当前系统已有基础

- `analysis` 模块已经具备自由问数、结果报告组装、结果详情、导出与审计链路。
- `query-assets` 模块已经具备常用查询模板、最近查询列表和历史重跑的基础模型。
- `management-report` 模块已经具备一定的区块化报表渲染能力。
- Web 工作台已经具备问数输入区、结果区、常用查询区、最近查询区的基础页面结构。

### 2.2 当前主要问题

- 当前结果页更接近“AI 分析页”，不够像“可复用经营报表工作台”。
- 常用查询模板更多是问法入口，而不是可执行、可治理的小报表资产。
- 最近查询更多是历史记录，没有沉淀成用户习惯与推荐输入。
- 固定查询、自由问数、经营报表三类结果协议尚未完全统一。
- 当前工作台容易让用户先等待 AI，再看到数据，不符合“先看到一点东西”的体验目标。

### 2.3 Grafana 看板定义给出的启发

这份 Grafana JSON 暴露出一批高价值、高频经营查询主题，适合作为首批模板来源，例如：

- 各团队每月新增商机情况
- 最近一周新增商机
- 最近一周跟进商机
- 最近输单项目回顾
- 2026 全年完成预测
- 2026 各团队完成预测
- 商机概率按团队 / 按负责人 / 按客户
- 价值客户历史提单数据
- 客户维度提单数据

同时它也暴露出若干口径问题，提醒系统必须把模板治理和口径治理放进正式方案，而不是仅仅把 SQL 搬过来：

- 标题年份与 SQL 年份不一致。
- “近一周”与“全年”口径混在同一区块。
- 部分 SQL 依赖全局时间配置，部分直接写死年份。
- 团队目标使用硬编码方式维护，缺乏统一治理口径。

## 3. 目标与非目标

### 3.1 目标

1. 让工作台的结果展示先返回确定性数据，再返回 AI 报告。
2. 让常用查询模板成为可治理、可执行、可推荐的固定 SQL 资产。
3. 统一最近查询口径，覆盖自由问数与模板查询。
4. 引入用户点击习惯与时间场景推荐，但不破坏主链路稳定性。
5. 保持现有权限、审计、导出和白名单治理边界不被绕过。

### 3.2 非目标

1. 不把本系统升级成一个通用 Grafana 替代品。
2. 不开放任意写操作 SQL 或任意多语句执行能力。
3. 不把推荐系统做成黑盒模型驱动，第一期以可解释规则和统计排序为主。
4. 不改动当前稳定的认证、登录和统一鉴权主链。

## 4. 总体设计原则

### 4.1 数据优先

所有查询结果必须优先返回结构化数据区块。AI 报告只能建立在数据结果之上，不能反客为主。

### 4.2 统一舞台，不统一入口

自由问数、模板查询、历史重跑是不同入口，但必须进入统一结果舞台。前端只关心统一结果协议，不关心执行来源。

### 4.3 模板稳定优先

常用查询和最近查询的模板入口采用固定 SQL + 参数装配 + 权限注入 + 展示配置的方式执行，不再经过 AI 意图理解。

### 4.4 推荐可解释

排序和推荐必须能给出简短、友好的原因文案，不做不可解释的黑盒推荐。

### 4.5 安全栈程序化

SQL 只读校验、表字段白名单、权限边界、实时权限注入、导出限制和审计留痕必须程序化处理，不能交给 AI。

## 5. 整体架构

### 5.1 两条执行主链

系统保留两条核心执行主链：

1. 自由 AI 问数链路  
   `用户输入问题 -> AI 理解与受控编排 -> 受控查询执行 -> 统一结果区块 -> AI 报告`

2. 固定 SQL 模板链路  
   `点击模板 / 最近查询 -> 参数装配 -> SQL 只读校验 -> 权限注入 -> 数据查询 -> 统一结果区块 -> AI 报告`

### 5.2 最近查询重跑

最近查询统一记录所有查询来源，但重跑时按来源回到正确链路：

- 来源为模板查询：重新执行当前模板
- 来源为自由问数：重新走 AI 编排

重跑不能直接复用旧结果，必须始终按当前实时权限、当前数据与当前模板版本重新执行。

### 5.3 统一结果协议

结果协议统一拆成两层：

1. `resultBundle`
   - 指标卡
   - 图表块
   - 表格块
   - 空态块
   - 口径说明
   - 筛选摘要

2. `insightBundle`
   - AI 报告状态
   - Markdown 报告正文
   - 关键判断
   - 风险与缺口
   - 建议动作
   - 空结果解释

前端必须始终先消费 `resultBundle` 渲染，再展示 `insightBundle`。

## 6. 结果展示设计

### 6.1 展示顺序

结果页严格按以下顺序展示：

1. 查询标题与口径摘要
2. 指标卡
3. 一个 100% 宽的主数据模块
4. AI Markdown 报告区

### 6.2 主数据模块规则

一次查询只突出一个主模块，避免首屏同时堆叠多个图表和多个表格。主模块类型由模板或结果协议决定：

- `STAT_ONLY`
- `TABLE`
- `BAR_CHART`
- `LINE_CHART`
- `RANKING_TABLE`

主模块需占满整行，桌面端高度建议 380 到 460 像素，移动端保持整行主视图，不拆成双列拼盘。

### 6.3 分阶段加载

结果区加载过程分三段：

1. 查询发出，显示数据区骨架屏。
2. 数据返回，立刻渲染指标卡和主模块。
3. AI 报告返回，报告区从轻量加载态切换成正式 Markdown。

若 AI 报告失败，不得影响数据区展示。

### 6.4 空结果规则

模板查询或自由问数查不到数据时，仍需返回空态数据模块，至少说明：

- 当前时间范围
- 当前筛选条件
- 可能为空的原因
- 下一步建议动作

空结果场景下的 AI 输出仅用于解释和建议，不允许硬凑经营分析结论。

## 7. 查询模板设计

### 7.1 模板定位

查询模板升级为“查询资产”，不仅包含模板文案，还包含完整执行和展示定义：

- 只读 SQL
- 参数定义
- 主展示类型
- 指标卡映射
- 表格列定义
- 图表字段映射
- AI 报告配置
- 推荐配置

### 7.2 模板关键字段

建议在 `CommonQueryTemplate` 基础上扩展：

| 字段 | 说明 |
| --- | --- |
| `queryMode` | 查询模式，当前固定为 `FIXED_SQL` |
| `sqlText` | 模板 SQL 原文 |
| `sqlVersion` | 模板版本号 |
| `parameterSchema` | 参数定义数组 |
| `renderConfig` | 展示配置 |
| `aiConfig` | AI 报告配置 |
| `recommendationConfig` | 推荐配置 |
| `governanceFlags` | 导出、推荐、首页展示等治理开关 |
| `validationSnapshot` | 最近一次 SQL 校验摘要 |
| `lastValidatedAt` | 最近一次校验时间 |

### 7.3 参数策略

允许管理员自由配置查询 SQL 和参数占位符，但运行时只能执行查询语句。参数本身不做“只能标准槽位”的前置限制，但仍需在模板发布前通过参数定义、SQL 校验和白名单检查。

## 8. 最近查询与推荐设计

### 8.1 最近查询统一口径

最近查询统一记录所有来源：

- `AI_QUERY`
- `TEMPLATE_QUERY`
- `RERUN_HISTORY`

建议为 `RecentQueryRecord` 补充：

| 字段 | 说明 |
| --- | --- |
| `sourceType` | 来源类型 |
| `templateId` | 来源模板 ID |
| `templateVersion` | 执行时模板版本 |
| `queryMode` | `AI_INTERPRETED` / `FIXED_SQL` |
| `parameterSnapshot` | 参数快照 |
| `renderSnapshot` | 展示快照 |
| `temporalContext` | 时间场景标签 |
| `summaryTags` | 主题标签 |
| `emptyState` | 是否空结果及原因 |
| `rerunCount` | 被重跑次数 |

### 8.2 排序分模型

常用查询与最近查询展示顺序采用三层分值叠加：

1. 用户个人行为分
   - 近 7 天点击次数
   - 近 30 天点击次数
   - 最近一次点击时间
   - 最近重跑成功率

2. 时间场景分
   - 当前是否处于月初、月中、月末、季初、季末、年初、年底等时间窗
   - 模板在该时间窗中的历史点击热度

3. 治理分
   - 模板推荐优先级
   - 模板命中率
   - 模板健康状态
   - 模板是否近期重点推广

### 8.3 推荐区块

首页资产区建议拆成三层：

1. 猜你想查
2. 常用查询
3. 最近查询

推荐文案应采用助手语气，例如：

- 猜你这会儿会先看这个
- 你最近常查，帮你提前放上来了
- 快到月底了，建议先看完成率预测

### 8.4 时间场景规则

第一期采用规则驱动时间窗，不引入复杂模型：

- 月初：1 日到 5 日
- 月中：11 日到 20 日
- 月末：25 日到月末
- 季初 / 季中 / 季末
- 年初：1 月
- 年底：11 月到 12 月

## 9. 治理与安全设计

### 9.1 SQL 只读约束

治理后台允许管理员自由填写查询 SQL，但运行时必须满足：

- 只允许 `SELECT`
- 允许 `WITH ... SELECT`
- 禁止多语句
- 禁止 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`TRUNCATE` 等写操作或 DDL
- 禁止访问非白名单表
- 禁止访问非白名单字段
- 强制增加行数上限
- 强制增加执行超时

### 9.2 权限与审计

模板查询不能绕过现有分析链路安全栈，必须继承：

- 当前实时权限
- 当前角色可见性
- 导出限制
- 审计留痕

需要额外记录以下审计事件：

- 模板发布
- SQL 校验通过 / 失败
- 模板执行
- 模板执行空结果
- 推荐命中
- 最近查询重跑

## 11. 接口调整建议

### 11.1 工作台初始化接口

扩展工作台初始化能力快照，返回：

- 权限摘要
- 猜你想查
- 常用查询
- 最近查询
- 推荐原因文案

### 11.2 模板执行接口

新增模板执行接口：

- `POST /api/v1/query-assets/templates/{templateId}/execute`

请求体建议包含：

- 参数对象
- 是否生成 AI 报告
- 是否写入最近查询
- 分页参数

### 11.3 最近查询接口

保留统一最近查询列表接口，并扩展响应字段：

- 来源类型
- 模板版本
- 参数快照摘要
- 推荐原因

重跑接口需要根据来源类型选择正确执行链路。

### 11.4 治理模板接口

治理模板接口需扩展支持以下能力：

- SQL 原文编辑
- 参数定义
- 展示配置
- AI 报告配置
- 推荐配置
- 最近校验结果
- 模板预览

建议新增：

- `POST /api/v1/governance/query-templates/{templateId}/validate`
- `POST /api/v1/governance/query-templates/{templateId}/preview`

## 12. 前端页面行为

### 12.1 工作台结构

工作台首页建议拆成四段：

1. 问数输入区
2. 查询资产区
3. 当前结果区
4. 辅助动作区

### 12.2 查询资产区

资产区按以下层级展示：

1. 猜你想查
2. 常用查询
3. 最近查询

模板卡片需展示：

- 模板名
- 模板说明
- 主展示类型
- 推荐标签
- 点击热度提示

### 12.3 最近查询区

最近查询统一展示模板查询和自由问数，但需显式标记来源类型，并支持：

- 再次执行
- 查看详情
- 查看空结果原因

### 12.4 结果页行为

- 先渲染指标卡和主数据模块
- AI 报告单独成区块
- 空结果优先展示空态模块
- 移动端仍保持数据主模块整行

### 12.5 治理页行为

治理页采用“配置 + 校验 + 预览”三段式：

1. 基础配置
2. SQL 与参数配置
3. 展示与 AI 配置
4. 推荐配置
5. 预览区

## 15. 验收标准

### 15.1 功能验收

1. 管理员可新增固定 SQL 模板并发布。
2. 模板点击后可直接执行查询，不经过 AI 意图理解。
3. 结果页先展示数据，再展示 AI 报告。
4. 空结果先展示空态模块，再展示解释型 AI 文案。
5. 最近查询统一记录模板查询和自由问数。
6. 最近查询重跑可按来源返回正确链路。
7. 首页可基于用户行为和时间场景做推荐排序。

### 15.2 安全验收

1. 非只读 SQL 无法通过校验和执行。
2. 非白名单表字段无法通过校验。
3. 模板查询执行时会注入当前实时权限范围。
4. 模板执行、推荐命中、导出和重跑均有审计记录。

### 15.3 体验验收

1. 数据区在 AI 报告之前可见。
2. AI 报告失败不影响数据区可用性。
3. 推荐提示文案简短、自然、可解释。
4. 移动端下主数据模块仍保持整行主视图。

## 16. 风险与注意事项

1. 若模板 SQL 治理不严，容易把查询资产页演化成“后台自由查库工具”，必须坚决阻断。
2. 若结果协议不统一，模板查询和自由问数会再次形成两套页面体系。
3. 若最近查询只存结果不存来源和参数快照，后续推荐与重跑都会失真。
4. 若推荐逻辑过重，容易干扰用户使用，因此第一期必须坚持可解释、可回退、规则优先。

## 17. 结论

推荐采用“中度重构、统一结果协议”的路线：

- 保留自由 AI 问数主链路
- 新增固定 SQL 模板直查链路
- 统一进入同一结果舞台
- 先沉淀 Grafana 看板里的高频查询模板
- 先做好数据优先展示，再逐步加推荐和治理预览

这条路线能在控制复杂度的前提下，同时提升稳定性、响应速度、可治理性和日常复用效率，最符合当前一期 CRM 智能分析系统的边界与节奏。
