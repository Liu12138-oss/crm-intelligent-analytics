# 智能分析富报告与单页展示改造方案

## 1. 背景

当前智能分析工作台已经完成“数据结果区在前，AI 分析报告区在后”的基础改造，但现有 AI 报告仍存在三个明显问题：

1. 自由问数与查询模板在报告生成链路上不统一，自由问数已经具备一定的结构化报告能力，而查询模板仍然停留在简单拼接 Markdown 的阶段。
2. AI 分析报告内容过浅，通常只有“返回了多少条数据 + 核心指标 + 一句提示”的摘要，缺少趋势判断、异常识别、短期预测、经营建议和结果依据。
3. Web 端 Markdown 展示仍然是基础文本预览器，不具备专业经营报告应有的层级、语义强调和可读性；企业微信机器人虽然已有兼容渲染链路，但输出内容依然偏短、偏弱。

本方案的目标不是单纯把一段 Markdown 写得更长，而是把智能分析结果升级成“统一事实、统一洞察、分渠道渲染”的富报告体系：

- 自由问数与查询模板共用统一报告对象与统一洞察生成链路。
- Web 查询页直接展示完整报告，不再要求用户跳转详情页才能看到长报告。
- 企业微信机器人继续保留轻量渲染，但其摘要、建议和趋势结论同样来自统一事实与统一报告编排。

## 2. 目标

### 2.1 功能目标

1. 自由问数和查询模板都返回统一的富报告结构，而不是分别维护两套分析文案逻辑。
2. 查询结果页直接展示完整 AI 经营分析报告，不再依赖结果详情页承接完整版阅读。
3. AI 报告必须补齐以下核心内容：
   - 执行摘要
   - 关键发现
   - 趋势分析
   - 短期区间预测
   - 异常与风险
   - 经营建议
   - 建议下一步
   - 结果依据
4. 趋势预测必须采用“平衡型预测”策略：允许给出短期区间预测，但必须同时给出依据、置信提示和限制说明，不允许伪精确预测。
5. Web Markdown 组件升级为专业报告阅读组件，支持标题层级、列表、引用说明、语义块和受控表格展示。
6. 企业微信机器人渲染必须保持兼容，不得因为 Web 富报告升级而打坏现有企微问数结果交付。

### 2.2 非目标

1. 本轮不改变现有 SQL 白名单、权限注入、只读校验、审计留痕和导出控制主链。
2. 本轮不移除现有结果详情页路由；详情页可以保留为兼容入口，但不再作为完整报告唯一阅读面。
3. 本轮不引入复杂时序预测算法平台，不做中长期预测，不做带训练过程的模型服务。
4. 本轮不把企业微信机器人输出升级成与 Web 完全等长的长报告页面。

## 3. 核心决策

### 决策一：统一报告事实，分渠道渲染

系统统一生成一份结构化富报告对象，Web 查询页、Web 详情页兼容入口、企业微信机器人都从该对象派生最终展示内容。

- Web 查询页：展示完整结构化报告与完整版 Markdown 阅读稿。
- Web 详情页：保留兼容，但不再承担“只有这里才能看完整版”的职责。
- 企业微信机器人：继续输出压缩版 Markdown，但其摘要、建议与趋势提示必须来自同一份报告事实。

### 决策二：查询页直接展示完整报告

不再采用“工作台只看摘要、详情页看完整版”的默认交互。查询页内直接展示完整报告，但通过强分区和可折叠区块控制阅读负担。

默认在查询页展开的高价值区块：

- 执行摘要
- 关键发现
- 趋势分析
- 趋势预测
- 经营建议

允许折叠但必须在同页可见入口的区块：

- 异常与风险
- 结果依据
- 完整 Markdown 阅读稿

### 决策三：程序先算事实，AI 只做归纳表达与建议编排

趋势、异常、结构失衡、预测区间和置信提示不得完全交给大模型自由发挥。必须先由程序基于统一结果包计算“洞察事实包”，再由 AI 基于这些事实生成 grounded 解释、建议和下一步追问。

### 决策四：查询模板链路停止手拼 Markdown

查询模板执行完成后，不再在模板执行服务中直接拼接“当前模板已返回多少条数据”的 Markdown。模板结果必须进入与自由问数相同的：

1. 洞察事实计算器
2. 富报告编排器
3. Markdown 编译器
4. 渠道压缩器

### 决策五：预测是可降级能力，不是默认强制输出

只有在满足时间序列长度、粒度一致性、样本完整度等条件时，才生成短期区间预测。否则只输出趋势判断或明确提示“当前结果不具备预测条件”。

## 4. 统一富报告对象

### 4.1 基础元信息

在现有 `AnalysisReportPayload` 基础上补充以下信息：

- `analysisConfidence`：整体分析可信度摘要，例如 `HIGH` / `MEDIUM` / `LOW`
- `predictionMode`：固定为 `BALANCED_RANGE_FORECAST`
- `predictionHorizon`：预测周期标签，例如“下一周”“下一月”
- `dataFreshnessAt`：结果对应的数据更新时间
- `evidenceSummary`：本次结论的依据说明摘要

### 4.2 核心洞察层

统一报告对象必须支持以下高阶字段：

- `trendInsight`
- `forecastInsight`
- `anomalyInsights`
- `riskInsights`
- `recommendations`
- `confidenceSummary`

字段职责如下：

#### `trendInsight`

用于承接趋势方向、变化速度、峰谷位置、连续升降期数、主要驱动因素等内容。

建议字段：

- `status`
- `direction`
- `changeValue`
- `changeRate`
- `peakLabel`
- `troughLabel`
- `volatilityLevel`
- `drivers`
- `summary`

#### `forecastInsight`

用于承接短期区间预测，只在满足预测条件时返回有效结果。

建议字段：

- `status`
- `horizonLabel`
- `predictedValue`
- `predictedRangeLow`
- `predictedRangeHigh`
- `confidenceLevel`
- `drivers`
- `caveats`
- `summary`

#### `anomalyInsights`

用于承接异常波动、离群点、数据缺口、周期不连续等问题。

建议字段：

- `type`
- `title`
- `detail`
- `severity`
- `datasetId`

#### `riskInsights`

用于承接经营风险与结果风险，二者必须区分。

建议字段：

- `riskType`：`BUSINESS_RISK` 或 `RESULT_RISK`
- `title`
- `detail`
- `severity`
- `datasetId`

#### `recommendations`

用于承接经营动作建议，必须是动作导向而非空泛口号。

建议字段：

- `priority`：`HIGH` / `MEDIUM` / `LOW`
- `title`
- `action`
- `reason`
- `evidenceKeys`

### 4.3 展示区块层

在现有 `sections` 基础上新增更明确的区块类型，用于驱动 Web 单页展示：

- `trend`
- `forecast`
- `anomaly`
- `risk`
- `recommendation`
- `evidence`

这些区块不替代现有 `summary`、`metric-strip`、`detail-table`，而是在其基础上扩展富报告能力。

### 4.4 Markdown 衍生层

Markdown 不再是唯一输出本体，而是从结构化富报告派生出来的阅读稿。建议统一补充：

- `workbenchMarkdown`
- `detailMarkdown`
- `wecomMarkdown`
- `markdownOutline`

虽然查询页现在直接展示完整报告，但保留 `workbenchMarkdown` 与 `detailMarkdown` 仍有价值：

1. 兼容现有前端字段选择逻辑。
2. 为企业微信压缩版、导出版和未来多端展示预留不同粒度的文本产物。
3. 避免把结构化渲染与阅读稿完全绑定成一个字段。

## 5. 洞察事实包与生成职责

### 5.1 总体原则

统一采用“程序先产出可校验事实，AI 再做 grounded 表达”的模式。

程序负责：

- 趋势计算
- 预测区间计算
- 异常识别
- 风险识别
- 建议输入事实整合

AI 负责：

- 把事实翻译成经营语言
- 组织更自然的摘要与建议
- 生成 groundedExplanation
- 生成 nextBestQuestions

### 5.2 程序必须先计算的事实

#### 趋势事实

- 当前周期值
- 上一周期值
- 环比变化值
- 环比变化率
- 连续上升或下降期数
- 峰值周期与谷值周期
- 波动程度
- 趋势方向

#### 结构事实

- Top1 / Top3 贡献占比
- 集中度是否过高
- 排名头尾差距
- 排名变化幅度
- 关键对象份额变化

#### 异常事实

- 单周期突增或突降
- 明显离群点
- 数据缺口
- 样本量过少
- 时间口径不连续

#### 预测事实

仅在以下条件满足时生成：

1. 时间序列点数不少于 4 个。
2. 时间粒度一致。
3. 指标值可连续比较。
4. 缺失值占比处于可接受范围。

第一版预测算法控制在轻量、可解释范围内：

- 加权移动平均
- 简单线性趋势
- 波动修正

输出必须包含：

- 中心预测值
- 预测区间上下界
- 置信等级
- 驱动因素
- 限制说明

### 5.3 AI 只允许做的事情

AI 只能基于程序已计算的事实做以下增强：

1. `趋势解读`
   把趋势方向、变化率、拐点和驱动因素翻译成管理层可读的语言。
2. `预测表述`
   把区间预测与置信提示写成谨慎表达，不允许写出伪精确未来事实。
3. `经营建议`
   基于趋势、风险、集中度和异常做优先级建议。
4. `结果依据组织`
   明确说明“为什么得出这些判断”，强化可信感。

### 5.4 预测降级规则

以下场景不允许硬输出预测：

1. 没有时间序列 -> 仅输出趋势缺失说明。
2. 时间点不足 -> 仅输出趋势判断，不输出预测区间。
3. 波动过大或缺口明显 -> 允许输出低置信预测，但必须附带限制说明。
4. 模板仅返回明细表且不具备连续时序 -> 默认不做预测。

## 6. Web 查询页单页展示方案

### 6.1 页面结构

查询页在现有“数据结果区在前，AI 报告区在后”的顺序基础上，统一展示以下区块：

1. 数据结果区
2. 指标卡
3. 关键发现
4. 趋势分析
5. 趋势预测
6. 异常与风险
7. 经营建议
8. 建议下一步
9. 结果依据
10. 完整 Markdown 阅读稿

### 6.2 默认展开与折叠策略

为避免单页内容过长失控，建议如下：

默认展开：

- 执行摘要
- 关键发现
- 趋势分析
- 趋势预测
- 经营建议

默认折叠但必须可见入口：

- 异常与风险
- 结果依据
- 完整 Markdown 阅读稿

### 6.3 结果详情页的角色

本轮不移除现有结果详情页路由，但其角色调整为：

- 兼容历史入口
- 供调试和复核使用
- 不再作为完整长报告唯一阅读面

新的主交互必须在查询页内完成，不能要求用户先在工作台看摘要，再跳详情页看完整版。

## 7. Markdown 阅读组件升级

### 7.1 Markdown 的新定位

Markdown 从“简单文本预览”升级为“统一报告阅读稿”。其定位是：

- 结构化报告的文本化阅读产物
- 可复制、可导出、可在企微中压缩派生的统一源
- Web 查询页中的专业阅读组件输入

### 7.2 建议支持的受控 Markdown 子集

为了兼顾美观和安全，前端阅读组件建议支持：

- `##` / `###` 标题
- 无序列表
- 有序列表
- 引用块
- 粗体
- 行内代码
- 受控表格

不支持：

- 任意 HTML
- 脚本
- 未受控的样式嵌入

### 7.3 建议的章节结构

完整版 Markdown 建议固定包含：

- `## 执行摘要`
- `## 关键发现`
- `## 趋势分析`
- `## 趋势预测`
- `## 异常与风险`
- `## 经营建议`
- `## 建议下一步`
- `## 结果依据`

### 7.4 视觉呈现要求

Markdown 阅读组件必须遵循根级 `DESIGN.md`：

1. 标题层级清晰，避免浏览器默认样式。
2. 风险、预测、建议、依据等区块有语义化视觉样式。
3. 引用块用于承接适用范围、预测限制、口径说明，不再使用默认重边框 blockquote。
4. 列表项间距足够，避免长文挤成一团。
5. 表格用于小型说明矩阵，例如预测区间与置信等级，不承接大型原始明细。

## 8. 企业微信机器人兼容策略

### 8.1 统一事实，不统一消息长度

企业微信机器人必须复用同一份结构化富报告对象，但不能直接照搬 Web 单页完整报告。

企业微信仍然走压缩版 Markdown 或分段消息输出，只保留：

- 执行摘要
- 最重要的 2 到 4 条发现
- 趋势或预测结论
- 核心建议
- 建议下一步

### 8.2 兼容字段必须保留

以下字段不能被破坏：

- `groundedMarkdown`
- `report.groundedMarkdown`
- `wecomMarkdown`
- `nextBestQuestions`
- `reportTitle`
- `executiveSummary`

这些字段将从新的统一富报告对象派生，但对现有企微渲染和回退逻辑仍保持兼容。

### 8.3 现有企微渲染链路的影响

本方案不会要求企微消息改成长篇完整报告，也不要求企微新增 Web 风格组件。企微侧只升级内容事实来源和摘要质量，保持聊天窗口内的阅读效率与兼容稳定性。

## 9. 协议兼容策略

### 9.1 增量扩展，不做破坏性替换

统一采用“新增 richer report 字段 + 保留旧字段”的方式。

保留旧字段：

- `title`
- `summary`
- `report.reportTitle`
- `report.executiveSummary`
- `groundedMarkdown`
- `report.groundedMarkdown`
- `wecomMarkdown`
- `nextBestQuestions`

新增高阶字段：

- `analysisConfidence`
- `trendInsight`
- `forecastInsight`
- `anomalyInsights`
- `riskInsights`
- `recommendations`
- `evidenceSummary`
- `confidenceSummary`
- `workbenchMarkdown`
- `detailMarkdown`

### 9.2 模板链路改造边界

查询模板执行链路本轮只改“结果出来之后的报告编排阶段”，不改变以下主链：

- 模板 SQL 只读校验
- 权限模式注入
- 真实 CRM 查询执行
- 结果主视图构建
- 导出与审计

## 10. 测试策略

### 10.1 后端单元测试

必须新增或补齐以下单测：

- 趋势事实计算器
- 预测区间计算器
- 异常识别器
- 风险识别器
- 建议编排器
- Markdown 编译器
- 渠道压缩器

### 10.2 后端集成测试

必须覆盖：

1. 自由问数返回 richer report。
2. 查询模板返回 richer report。
3. 缺少时间序列时预测降级正确。
4. 样本过少时只输出趋势不输出预测。
5. 企业微信仍可从新报告对象派生兼容 Markdown。
6. 旧字段仍存在且语义不变。

### 10.3 前端单元测试

必须覆盖：

1. 查询页直接渲染完整富报告。
2. Markdown 阅读组件支持标题、列表、引用、表格和语义区块。
3. 风险块、预测块、建议块样式映射正确。
4. 新字段缺失时可自动回退旧字段。

### 10.4 前端 E2E

必须覆盖：

1. 自由问数在查询页直接看到完整报告。
2. 查询模板在查询页直接看到完整报告。
3. 折叠区块可展开，不跳详情页也可完成阅读。
4. 企业微信链路契约测试保持通过。

## 11. 风险与控制

### 11.1 预测内容看起来强，但依据不足

控制方式：

- 预测必须由程序前置校验。
- 不满足条件时只输出趋势判断或明确降级说明。
- 预测文案必须带置信提示和限制说明。

### 11.2 查询页单页内容过长，阅读负担过重

控制方式：

- 采用强分区与折叠区块。
- 默认只展开高价值区块。
- Markdown 阅读稿放在靠后位置或默认折叠容器中。

### 11.3 企业微信消息失焦

控制方式：

- 保持企业微信压缩版输出。
- 统一事实来源，但不统一消息长度。
- 必要时按条分段输出。

### 11.4 模板链路与自由问数风格继续分叉

控制方式：

- 模板链路强制进入统一洞察事实计算器与统一富报告编排器。
- 禁止继续在模板执行服务中直接拼接简单 Markdown。

### 11.5 成本与性能上升

控制方式：

- 趋势、预测、异常优先由程序计算。
- AI 只消费浓缩后的事实上下文，而不是再次吞整张大表。
- 企业微信和兼容字段优先复用压缩产物，避免重复生成。

## 12. 落地顺序

建议按以下顺序灰度实施：

1. 扩展统一富报告协议，保留旧字段不动。
2. 自由问数接入洞察事实计算器与 richer report 编排。
3. 查询模板链路切换到统一 richer report 编排。
4. 查询页升级为单页完整报告展示。
5. Markdown 阅读组件升级为专业报告阅读组件。
6. 企业微信压缩版从统一富报告对象派生。
7. 灰度验证 Web，再验证模板，再验证企业微信增强输出。

## 13. 最终结论

本次改造的最终方向确定为：

1. 统一自由问数与查询模板的富报告事实模型。
2. 统一趋势、预测、异常、风险和建议的洞察生成职责分工。
3. Web 查询页直接展示完整报告，不再依赖详情页承接完整版阅读。
4. Markdown 从普通文本预览升级为专业报告阅读稿与受控阅读组件。
5. 企业微信机器人保持兼容压缩渲染，但内容来源统一升级。

这样可以在不破坏现有权限、审计和只读查询主链的前提下，把“AI 分析太简单”的问题真正解决为“结果更深、更稳、更像专业经营报告”，而不是仅仅把现有摘要改写得更长。
