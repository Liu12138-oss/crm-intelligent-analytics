# CRM 智能分析 300 问剩余工作落地实施与验收方案

## 一、目标边界

本方案面向 `业务运营发展分析问题清单_20260627.md` 中 30 个大类、300 个问题，目标是把企业微信「渠道 CRM 系统机器人」建设成可真实提问、可按模板回答、可追踪数据来源、可用截图举证验收的 CRM 智能经营分析入口。

最终目标不是把 300 个问题硬编码成 300 段答案，而是完成以下闭环：

1. 每个问题都有标准模板类型。
2. 每个模板都有明确数据源。
3. 每个问题都有可验收提问。
4. 每个问题都有企微截图留存路径。
5. 每个回复都必须有展示模板、回复结构、数据口径、权限口径、核心指标、明细摘要、风险建议和建议追问。
6. 无法真实计算的问题必须说明字段缺口、接口缺口、权限缺口或样本不足，不能让 AI 编造事实。

## 六、回复结构硬性要求

所有企微回复必须按以下结构输出，不允许只返回泛化聊天文本：

```text
【展示模板】模板名称
【回复结构】问题复述 / 数据口径 / 权限口径 / 核心指标 / 明细摘要 / 风险建议 / 建议追问
【问题复述】复述用户要看的对象、时间和范围
【数据口径】说明时间、数据源、金额单位、统计对象
【权限口径】说明当前用户可见范围，不能看全国时说明原因
【核心指标】输出 3 到 6 个最关键指标
【明细摘要】输出 TOP3 到 TOP10，不在企微塞全量明细
【风险建议】说明异常、断点、字段缺口和下一步动作
【建议追问】给出 2 到 3 个可继续追问的问题
```

## 八、300 问验收流程

1. 运行 `pnpm generate:crm-analysis-300-matrix` 生成最新矩阵。
2. 使用 `docs/testing/CRM智能分析300问企微验收问题清单_20260630.txt` 作为发送清单。
3. 逐条在企业微信「渠道 CRM 系统机器人」发送问题。
4. 每条问题保存企微截图到矩阵指定路径。
5. 对每条回复检查模板、口径、权限、指标、明细、风险、建议追问。
6. 每 30 条形成一份批次小结，避免 300 条最后才发现路由偏差。
7. 失败项登记缺陷编号、失败原因、责任模块和复测截图。
8. 全量完成后输出最终验收报告。

## 九、验收通过标准

| 层级 | 标准 |
| --- | --- |
| P0 高频问题 | 通过率不低于 90%，失败项必须有明确修复计划 |
| 300 问模板命中 | 100% 问题必须命中标准模板或明确不支持原因 |
| 数据事实 | 指标必须来自 CRM OpenAPI 或受控只读接口，AI 不编造数字 |
| 权限口径 | 任何问题都不能越权展示全国或全量数据 |
| 展示结构 | 所有回复必须带固定模板段落 |
| 截图举证 | 每条问题都有提问和答复截图 |
| 缺陷闭环 | 失败项修复后必须用同一问题复测 |

## 十一、风险与处理

| 风险 | 处理方式 |
| --- | --- |
| 真实 CRM 字段不足 | 回复中明确字段缺口，矩阵状态标注为需补外部字段 |
| 订单样本过少 | 用报备、商机、报价作为前置经营口径，不能伪装成真实成交 |
| 普通问法被帮助菜单抢答 | 提高 300 问模板路由优先级 |
| 预测问题缺模型 | 使用规则评分兜底，并标注非机器学习模型 |
| 企微截图量大 | 按模板分目录、按 Q 编号保存，每 30 条批次复核 |
| 多角色权限复杂 | 先超管跑通，再区域、渠道、员工逐步复测 |
| 图片或 HTML 无法直接发送 | 长连接内以 Markdown 和模板卡片为主，图片卡片作为类 HTML 展示增强 |
