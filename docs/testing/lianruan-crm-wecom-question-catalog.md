# 联软 CRM 企微智能分析问法与业务目录

## 目标

本文用于校准企业微信 AI 问数的业务理解、OpenAPI Markdown 快照对象和结果呈现结构。联软提供的测试环境 SQLite 只作为字段、对象关系和回归样例来源；正式企微问答必须读取由联软标准 OpenAPI 刷新的本地 Markdown 快照明细，再聚合分析，不在问答阶段实时请求 OpenAPI，也不读取 SQLite/MySQL 分析库作为正式结果。

## 真实测试库校准结论

测试库采用 `entities(entity_name, id, data_json)` 存储业务对象，核心对象和记录数如下：

| 对象 | 记录数 | 正式取数对象 | 主要用途 |
| --- | ---: | --- | --- |
| `partners` | 173 | `details/partners.md` | 合作伙伴、服务商、渠道商、代理商主数据和发展运营分析 |
| `registrations` | 152 | `details/registrations.md` | 客户报备、保护期、报备状态、报备与商机关联分析 |
| `opportunities` | 44 | `details/opportunities.md` | 商机总览、阶段分布、停滞风险、渠道商维度商机分析 |
| `quotes` | 4 | `details/quotes.md` | 报价、报价转订单、报价未下单风险分析 |
| `orders` | 3 | `details/orders.md` | 订单、下单、成交、渠道商贡献和订单汇总分析 |
| `users` | 60 | `details/users.md` 或权限上下文 | 负责人、分配人员、审计主体映射 |

测试库字段校准：

| 对象 | 关键字段 | 说明 |
| --- | --- | --- |
| 合作伙伴 | `id/name/region/bigRegion/joinDate/partnerLevel/level/isTechService/techServiceType/status/orderCount/quoteCount/totalAmt/parentPartnerId` | `level/isTechService/techServiceType/totalAmt` 是历史兼容字段；OpenAPI 标准字段可用时优先使用 `partnerLevel/isTechnicalServiceProvider/technicalServiceProviderType/totalAmount`。 |
| 客户报备 | `id/customer/partnerId/partnerName/status/createdAt/approvedAt/expireAt/hasOpportunity/industry/estimatedAmt/assignedStaffName` | 报备问题默认带渠道商维度，保护期问题看 `expireAt`。 |
| 商机 | `id/name/customer/partnerId/partnerName/stage/amount/owner/assignedStaffName/regId/quoteId/expectedClose/lastFollowAt/updatedAt` | 阶段既可能是中文，也可能是 `contacted/registered/quoted/cancelled` 等编码，展示前必须转成中文释义。 |
| 报价 | `id/customer/oppId/regId/partnerId/partnerName/status/total/originalTotal/quoteMode/workloadSummary` | 报价金额优先取 `total/originalTotal`，报价转订单需要和订单的 `quoteId/oppId/regId/customer` 关联。 |
| 订单 | `id/customer/oppId/quoteId/regId/partnerId/partnerName/status/total/createdAt/primaryConfirmedAt` | 订单金额只取订单对象金额字段，不能用商机或报价金额代替。 |

## 主链原则

1. 企业微信接收问题后，先做固定安全检查、身份和权限装载。
2. AI 只负责理解自然语言，输出业务对象、指标、维度、时间、分析模式和呈现偏好。
3. 程序把 AI 结果映射到 OpenAPI Markdown 快照主链，不允许 AI 生成 SQL。
4. OpenAPI 只负责启动、定时或手动刷新 Markdown 快照；正式问答阶段只读 `latest/details/*.md`，按本次请求临时聚合，摘要、模板卡片、表格和 HTML 报告复用同一个结果包。
5. SQLite 测试库只用于理解业务结构、补字段兼容、设计问法目录和做回归样例，不作为企微正式答复的数据源。
6. 企微机器人默认发送模板卡片摘要、短文本导读和只读 HTML 报告链接，不再上传结果图片。

## 默认业务理解

| 用户说法 | 默认业务对象 | 默认输出 |
| --- | --- | --- |
| 商机、机会、项目机会 | 商机 + 渠道商 | 商机数、商机金额、阶段分布、渠道商维度、重点商机明细 |
| 渠道商、服务商、合作伙伴、代理商、经销商 | 合作伙伴 | 如果是“开拓/画像/等级/技术服务商”，查服务商主数据；如果是“经营/运营/贡献/业绩/下单”，拉通报备、商机、订单 |
| 客户报备、报备 | 客户报备 + 渠道商 | 报备数、状态分布、关联商机、未关联商机报备、保护期风险 |
| 报价、报价单 | 报价 + 客户 + 渠道商 + 订单 | 报价数、报价金额、状态分布、报价未下单、报价转订单 |
| 订单、下单、成单、成交、签单 | 订单 + 渠道商 | 订单数、订单金额、状态分布、渠道商贡献、订单明细 |
| 漏斗、转化 | 报备 + 商机 + 报价 + 订单 | 各环节数量、转化率、断点和后续建议 |
| 风险、停滞、未进展、超期、快到期 | 对应对象风险 | 停滞商机、报备保护期到期、报价未下单、沉默客户等风险清单 |
| 看板、报告、汇总分析 | 对应对象组合报告 | 模板卡片摘要、核心指标、图表/表格、HTML 报告链接 |

## 高频问法目录

### 1. 综合经营问题

| 典型问法 | 应识别对象 | 默认结构 |
| --- | --- | --- |
| 帮我查询分析下，公司当前的合作伙伴开拓情况、客户商机报备及订单情况，并且给出后续经营建议 | 合作伙伴、客户报备、商机、订单 | 合作伙伴开拓情况、客户报备情况、客户商机及渠道商维度、订单情况及渠道商贡献、经营建议 |
| 公司当前渠道经营情况怎么样 | 合作伙伴、客户报备、商机、订单 | 渠道商经营贡献汇总、报备/商机/订单明细、断点建议 |
| 全国代理商发展运营数据看板 | 合作伙伴、客户报备、商机、订单 | 服务商发展画像、区域覆盖、技术服务商、报备、商机、订单、重点服务商 |
| 广州办渠道下单汇总分析 | 订单、渠道商、区域、月份 | 订单总览、渠道商下单排名、月度趋势、订单明细 |

## 回归校验清单

每次改动企微 AI 分析链路后，至少验证以下问题：

1. “帮我查询分析下，公司当前的合作伙伴开拓情况、客户商机报备及订单情况，并且给出后续经营建议”必须输出四个业务区块，不得回复合同转化报告。
2. “帮我分析一下全部的商机情况”必须输出商机总览、阶段分布和渠道商维度，不得扩成订单或合同报告。
3. “分析当前 3 个月没有进展的商机情况”必须走商机停滞风险，不得被综合经营快照抢走。
4. “服务商发展和开拓、运营情况”必须进入服务商经营链，不能只输出服务商数量画像。
5. “客户报备情况”必须带渠道商维度和真实报备明细。
6. “订单情况”必须按订单对象统计订单数和订单金额，不能用商机金额或报价金额代替。
7. “报价情况，特别是有报价但未下单的客户”必须识别报价和订单差集，不能误归为合同金额总览。
8. 明细必须来自 OpenAPI 真实名称，不能出现“渠道商001”这类占位名。
9. 阶段、状态、类型必须展示中文业务释义，不能直接暴露 `contacted`、`quoted`、`active` 等内部枚举。
10. 企微交付必须以模板卡片和只读 HTML 报告为主，不上传分析结果图片。
