# 权限中心与企业微信灰度演练清单

## 1. 适用范围

本文档用于在预发环境、联调环境或真实测试环境中执行以下能力的人工灰度演练：

- 权限中心角色能力矩阵
- 企业微信机器人灰度开通与停用
- 用户权限预览
- 身份映射诊断
- 合同审核扩展授权即时生效

本文档对应 OpenSpec 变更 `add-access-governance-and-wecom-pilot-control` 的最后一项人工验收任务。由于该项依赖真实角色、真实企业微信账号或准真实测试账号，本仓库无法通过本地自动化直接完成，需要由测试人员按本文档执行并回填结果。

## 3. 演练步骤

### 3.1 步骤一：验证 Web 可用但企业微信未开通

目标：证明“企业微信灰度只影响入口，不影响 Web 工作台”。

执行步骤：

1. 管理员登录 `权限中心`，确认待测角色已具备：
   - `analysis.use`
   - `analysis.follow_up`
   - Web 入口开关开启
   - 企业微信入口资格开启
2. 在 `企业微信灰度` 分区将模式设为 `白名单开放`。
3. 不为 `A 组账号` 加入任何白名单。
4. 使用 `A 组账号` 登录 Web 工作台，验证：
   - 能看到 `智能分析`
   - 可以进入 `/analysis`
   - 可以发起一次自然语言分析
5. 使用同一账号在企业微信中给机器人发消息，验证：
   - 被阻断
   - 提示为“当前企业微信机器人仍在灰度开放中，你的账号暂未开通体验资格”或等价语义
   - 不应误提示为“未映射 CRM 身份”或“无数据”
6. 在权限中心的 `用户权限预览` 中输入该账号，验证：
   - `wecomBotAccessState = PILOT_REQUIRED`
   - 角色、菜单、动作、作用域摘要正确

验证结果：

- [ ] Web 可正常使用
- [ ] 企业微信被灰度阻断
- [ ] 预览结果显示 `PILOT_REQUIRED`

### 3.2 步骤二：验证企业微信白名单立即生效

目标：证明命中白名单的用户可立即进入企业微信机器人。

执行步骤：

1. 在 `企业微信灰度` 分区为 `B 组账号` 添加白名单：
   可选择用户、角色或部门三种方式之一。
2. 保存灰度策略。
3. 使用 `B 组账号` 在企业微信中重新发消息，验证：
   - 能进入正式机器人链路
   - 可以返回正常问数、帮助提示或补问提示
4. 在 `用户权限预览` 中验证：
   - `wecomBotAccessState = ALLOWED`
5. 在 `身份映射诊断` 中验证：
   - `mappingStatus = MAPPED`
   - `wecomBotAccessState = ALLOWED`

验证结果：

- [ ] 白名单保存后无需重启服务即可生效
- [ ] 企业微信账号可立即通过入口门闩
- [ ] 预览与诊断均显示 `ALLOWED`

### 3.3 步骤三：验证单用户紧急停用

目标：证明停用名单优先级高于白名单，并且只影响企业微信入口。

执行步骤：

1. 将 `C 组账号` 加入白名单，确认其当前能够正常使用企业微信机器人。
2. 在 `企业微信灰度` 分区将该账号加入停用名单。
3. 立即重新用 `C 组账号` 在企业微信中发消息，验证：
   - 被阻断
   - 提示为“当前账号已被暂停企业微信机器人使用资格，请联系管理员确认”或等价语义
4. 再使用该账号登录 Web 工作台，验证：
   - Web 功能仍保持可用
   - 不应因企业微信停用而连带失去 Web 菜单与分析能力
5. 在 `用户权限预览` 中验证：
   - `wecomBotAccessState = EXPLICITLY_DENIED`

验证结果：

- [ ] 停用名单优先级高于白名单
- [ ] 停用后企业微信立即失效
- [ ] Web 不受影响

### 3.4 步骤四：验证合同审核扩展授权与撤销

目标：证明“查看他人合同任务”“下载他人产物”按统一权限中心即时生效。

执行步骤：

1. 使用普通上传者创建一条合同审核任务，确保任务已完成且存在可下载产物。
2. 使用 `D 组账号` 在未授权状态下访问他人任务详情和下载地址，验证：
   - 详情访问被拒绝
   - 下载被拒绝
3. 管理员在 `角色权限` 中为 `D 组账号` 所属角色开启：
   - `contract.review.cross_view`
   - `contract.review.cross_download`
4. 重新使用 `D 组账号` 访问同一任务，验证：
   - 可以查看详情
   - 可以下载产物
5. 管理员撤销上述权限后，立即再次访问，验证：
   - 详情再次被拒绝
   - 下载再次被拒绝

验证结果：

- [ ] 授权后立即生效
- [ ] 撤销后立即失效
- [ ] 下载仍然执行实时二次鉴权

### 3.5 步骤五：验证身份映射诊断与拒绝原因

目标：证明权限中心能解释“为什么这个人不能用”。

执行步骤：

1. 在 `身份映射诊断` 中查询 A/B/C 组账号。
2. 对照三类状态检查：
   - `A 组账号`：`PILOT_REQUIRED`
   - `B 组账号`：`ALLOWED`
   - `C 组账号`：`EXPLICITLY_DENIED`
3. 若存在未映射用户，再查询一名未绑定 CRM 身份的企业微信账号，验证：
   - `mappingStatus = UNMAPPED`
   - `wecomBotAccessState = UNMAPPED_CRM_IDENTITY`
   - 失败原因明确区分为身份映射缺失，而不是普通未开通

验证结果：

- [ ] 三类灰度状态能在诊断页被正确区分
- [ ] 未映射用户不会被误判成普通无权限
