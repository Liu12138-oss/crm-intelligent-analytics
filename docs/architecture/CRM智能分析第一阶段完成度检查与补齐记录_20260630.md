# CRM 智能分析第一阶段完成度检查与补齐记录

检查日期：2026-06-30

## 一、第一阶段完成度结论

第一阶段主链路已补齐并通过验证。当前后端已能从 `/api/v1/wecom/messages` 进入完整企微 CRM 智能分析链路，覆盖企微验签、消息解析、CRM 身份映射、权限拦截、分析查询、企微 Markdown/模板展示、长连接流式回包和审计记录。

本次检查前的主要缺口是：企微消息控制器仍停留在核心机器人服务口径，完整分析服务没有被根模块和企微模块稳定装配，导致 CRM 问数入口可能只返回“业务能力暂未启用”或在 Nest 依赖解析阶段失败。

## 二、本次补齐内容

1. 补齐分析模块装配。
   - 新增 `AnalysisModule`，集中注册并导出 `AnalysisService`、`AiGatewayService`、`AnalysisChannelPresenterService`、`CrmAnalysisPresentationTemplateService`、CRM 标准 OpenAPI 读取链路、查询编排、权限、审计、上下文策略等第一阶段问数必需依赖。
   - 补齐 `PermissionEnforcementService`、`ClarificationService`、`AiContextPolicyService`、`SqlAuditContextService` 等跨链路依赖导出。

2. 补齐企微完整机器人入口。
   - `/wecom/messages` 控制器改为注入完整 `WecomBotService`，不再走只保留核心能力的 `WecomCoreBotService`。
   - `WecomModule` 导入 `AnalysisModule`、`DailyReportModule`、`NotificationModule`，并补齐企微完整服务启动所需的客户查询、商机查询、身份映射、权限审计、流式调度等依赖。

3. 收敛重复服务实例。
   - 企微模块复用分析模块导出的 `AnalysisService`、`AnalysisRequestRepository`、`AnalysisLoggerService`，避免重复注册导致不同模块上下文中出现两套分析实例。
   - 企微模块复用通知模块导出的 `WecomTransportService`，避免流式调度器和测试/通知侧拿到不同传输实例。

4. 恢复第一阶段默认业务口径。
   - 企业微信 CRM 问数默认启用业务动作。
   - 只有显式配置 `WECOM_BUSINESS_ACTIONS_ENABLED=false` 时，才回到普通 AI 核心聊天模式。
   - 未绑定 CRM 身份的企业微信用户默认拦截并返回可读提示，符合第一阶段“企微问数必须落到 CRM 用户与权限口径”的要求。

5. 保留并验证展示模板能力。
   - 已有展示模板匹配服务继续接入分析结果展示。
   - 企微 Markdown 顶部保留展示模板标识，后续可继续扩展图片卡片和模板卡片呈现。
