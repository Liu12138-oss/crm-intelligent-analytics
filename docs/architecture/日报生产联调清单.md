# 日报生产联调清单

## 1. 目的

本文档用于把销售日报生产发送链路从“功能已开发”推进到“生产可联调、可验证、可上线”。

适用范围只覆盖：

- 企业微信目录同步
- 22:00 个人日报确认 / 催报
- 23:59 封账
- 次日 08:00 销售小组团队汇总
- 权限中心中的日报部门启停、区域默认收件人、小组收件覆盖和发送预览

## 2. 联调前提

### 2.1 环境变量

生产或联调环境至少确认以下变量已正确配置：

- `DAILY_REPORT_ENABLED=true`
- `DAILY_REPORT_INTERNAL_SCHEDULER_ENABLED=true`
- `DAILY_REPORT_SCHEDULER_ACTOR_USER_ID=<系统执行人 CRM 用户 ID>`
- `DAILY_REPORT_REMINDER_TIME=22:00`
- `DAILY_REPORT_CLOSE_TIME=23:59`
- `DAILY_REPORT_SUMMARY_TIME=08:00`
- `WECOM_NOTIFY_REAL_MESSAGE_ENABLED=false`（首次联调建议保持关闭）
- `WECOM_NOTIFY_TEST_USER_ID=<联调测试接收人>`
- `WECOM_NOTIFY_CORP_ID=<企业微信自建应用 CorpID>`
- `WECOM_NOTIFY_AGENT_ID=<企业微信自建应用 AgentID>`
- `WECOM_NOTIFY_SECRET=<企业微信自建应用 Secret>`
- `WECOM_APP_MESSAGE_MIN_INTERVAL_MS=3000`
- `WECOM_APP_MESSAGE_MAX_RETRIES=2`
- `WECOM_APP_MESSAGE_RATE_LIMIT_RETRY_DELAYS_MS=60000,180000,300000`
- `WECOM_BOT_PROACTIVE_MIN_INTERVAL_MS=15000`
- `WECOM_BOT_RATE_LIMIT_RETRY_DELAYS_MS=60000,180000,300000`
- `WECOM_DIRECTORY_AGENT_ID`
- `WECOM_DIRECTORY_SECRET`
- `WECOM_DIRECTORY_ROOT_DEPARTMENT_NAME`

说明：

- 日报提醒、封账和团队汇总默认由后端内置调度触发，不再依赖外部脚本登录态。
- 外部调度脚本只作为运维兜底方案；如果继续保留脚本，必须避免与内置调度在同一分钟重复触发同一任务。
- 日报催报、个人确认、正式对上推送和团队汇总属于固定周期系统通知，默认通过企业微信自建应用消息投递；机器人仅保留会话型异步结果回推。
- `WECOM_NOTIFY_REAL_MESSAGE_ENABLED=false` 时，非自动化测试环境仍会把消息真实发送给 `WECOM_NOTIFY_TEST_USER_ID`，用于验证自建应用消息链路。
- 批量发送、测试改投和手工补发都必须进入同一个自建应用消息队列，相邻消息不得小于 `WECOM_APP_MESSAGE_MIN_INTERVAL_MS`。
- 2026-05-18 首日试点出现的 `846607` 属于企业微信机器人发送频率超限；代码上线前可临时把 `WECOM_BOT_DELIVERY_RETRY_DELAY_MS` 调整为 `60000` 止血，但长期修复以通道路由和限流队列为准。

### 2.2 组织与映射

联调前必须确认：

- 企业微信通讯录应用已授权读取部门负责人、成员所属部门、直属上级；
- 目标销售成员都能映射到 CRM 用户；
- 目标区域负责人都能映射到可发送的企微账号；
- 后台发送预览能正确解析出 `销售小组 -> 最终收件人`。

### 2.3 后台规则初始化

联调前必须先在权限中心完成以下配置：

1. 启用目标销售小组，例如 `山东销售`
2. 停用非销售部门，例如 `山东技术团队`、`山东价值客户运营团队`、`山东职能`
3. 为父级区域配置默认收件人，例如 `山东区 -> 牛劲`
4. 执行一次发送预览并保存截图或结果记录

## 3. 推荐联调顺序

### 3.1 第一步：目录同步

先执行一次企业微信目录同步，确保部门树、负责人和成员映射使用最新事实：

```bash
bash docs/architecture/deploy-examples/run-daily-report-jobs.sh sync-all
```

验收要点：

- 返回 `accepted: true`
- 同步结果中没有负责人字段缺失或大量 `unmatched`
- 目标销售成员、区域负责人都能在身份映射诊断里看到

### 3.2 第二步：后台发送预览

用生产或联调管理员账号执行发送预览：

```bash
bash docs/architecture/deploy-examples/run-daily-report-jobs.sh preview 2026-04-28
```

验收要点：

- 预览结果中出现目标销售小组
- 最终收件人与后台配置一致，例如 `山东销售 -> 牛劲`
- 被停用的非销售部门不出现在结果中
- 若缺少负责人或企微映射，返回明确 `deliveryReason`

### 3.3 第三步：22 点个人日报链路

用当天业务日期执行 reminders：

```bash
bash docs/architecture/deploy-examples/run-daily-report-jobs.sh reminders 2026-04-28
```

验收要点：

- 有 CRM 源数据的销售成员只收到个人日报确认
- 无 CRM 源数据的销售成员只收到友好催报提醒
- 22:00 不会给区域负责人发团队汇总
- 主动通知任务的最终通道应为 `WECOM_APP_MESSAGE`
- 审计快照应包含计划触发时间、实际发送开始时间、通知任务 ID、最终通道和最后尝试时间
- 若一次产生多条提醒，发送时间戳应满足自建应用消息最小间隔

### 3.4 第四步：23:59 封账

```bash
bash docs/architecture/deploy-examples/run-daily-report-jobs.sh close 2026-04-28
```

验收要点：

- 未确认日报被封账
- 封账后不额外产生主管发送

### 3.5 第五步：次日 08:00 团队汇总

```bash
bash docs/architecture/deploy-examples/run-daily-report-jobs.sh summaries 2026-04-28
```

验收要点：

- 每个已启用销售小组只生成 1 条团队汇总
- 团队汇总正文包含“日报汇总”和“日报明细”
- 测试模式下仍改投测试接收人，但预览与审计保留真实目标收件人
- 每条团队汇总必须逐条进入自建应用消息队列，不能并发发送
- 消息正文必须说明后续处理入口仍为 CRM 智能分析机器人或 Web 工作台

## 4. 真实发送切换前检查

在把 `WECOM_NOTIFY_REAL_MESSAGE_ENABLED` 从 `false` 切到 `true` 之前，至少完成：

1. 后台发送预览通过
2. 目标销售小组与收件人已由业务负责人确认
3. 联调测试接收人已收到完整的团队汇总样例
4. 审计里能看到真实目标收件人、最终通道和通知任务 ID
5. 自建应用消息凭证、应用可见范围和测试接收人均已验证
6. 阻断场景文案已由运维或业务确认

建议切换步骤：

1. 保持 `DAILY_REPORT_ENABLED=true`
2. 将 `WECOM_NOTIFY_REAL_MESSAGE_ENABLED` 从 `false` 改为 `true`
3. 重启后端服务
4. 只选择一个试点销售小组做一次真实发送验证
5. 试点通过后再扩大到全部启用销售小组

## 5. 生产调度建议

建议按以下节奏配置：

```cron
55 21 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh sync-all >> /srv/crm-intelligent-analytics/shared/logs/daily-report-sync-2155.log 2>&1
00 22 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh reminders >> /srv/crm-intelligent-analytics/shared/logs/daily-report-reminders.log 2>&1
59 23 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh close >> /srv/crm-intelligent-analytics/shared/logs/daily-report-close.log 2>&1
50 07 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh sync-all >> /srv/crm-intelligent-analytics/shared/logs/daily-report-sync-0750.log 2>&1
00 08 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh summaries >> /srv/crm-intelligent-analytics/shared/logs/daily-report-summaries.log 2>&1
```

说明：

- 22:00 和 08:00 前都先跑目录同步，避免按旧组织树发错人。
- `close` 保持独立调度，不建议与 `reminders` 合并为一个夜间批次。

## 6. 失败排查顺序

### 6.1 预览没有结果

先检查：

1. 目标销售小组是否已启用
2. 目标部门是否被误标为 `NON_SALES`
3. 目录同步是否成功
4. 目标成员是否完成 CRM / 企微映射

### 6.2 团队汇总没有发给正确负责人

先检查：

1. 区域默认收件人是否配置正确
2. 销售小组是否存在单独覆盖规则
3. 区域负责人是否具备企微映射
4. 发送预览里的 `ruleSource` 是否符合预期

### 6.3 真实发送没有触达

先检查：

1. `WECOM_NOTIFY_REAL_MESSAGE_ENABLED` 是否已切为 `true`
2. 企业微信主动通知 agent/secret 是否有效
3. 审计和主动通知任务里是否出现 `BLOCKED` 或 `FAILED`
4. 日志中是否存在无效接收人或连接失败

### 6.4 企业微信频率限制或批量发送失败

先检查：

1. 审计中的外部错误码是否为 `846607`，该错误应显示为“企业微信机器人发送频率超限”
2. 日报固定周期通知是否仍误走 `WECOM_BOT_MESSAGE`
3. `WECOM_APP_MESSAGE_MIN_INTERVAL_MS` 是否低于上线要求
4. `WECOM_APP_MESSAGE_RATE_LIMIT_RETRY_DELAYS_MS` 是否仍为分钟级退避
5. 手工补发是否逐条执行，且每条至少间隔 1 到 2 分钟

## 7. 最终上线前必须留档的证据

建议至少保留：

- 一次目录同步成功结果
- 一次发送预览结果
- 一次 reminders 联调结果
- 一次 summaries 联调结果
- 试点真实发送结果
- 当前后台日报部门启停与区域收件人配置截图

这些证据应与 `backend.env`、部署参数清单、发布版本号一起归档。
