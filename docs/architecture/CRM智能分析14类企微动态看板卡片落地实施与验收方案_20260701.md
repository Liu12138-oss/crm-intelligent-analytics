# CRM 智能分析 14 类企微动态看板卡片落地实施与验收方案

> 编写日期：2026-07-01  
> 适用范围：企业微信机器人 CRM 智能分析回复、模板卡片、长连接正文、图片看板和完整 HTML 报告入口。  
> 关联文档：`docs/architecture/CRM智能分析问答结构模板与分析维度大类方案_20260630.md`、`docs/CRM智能分析300问剩余工作落地实施与验收方案_20260630.md`、`docs/CRM智能分析最终目标落地方案_20260630.md`。

## 一、目标与边界

### 1.1 目标

将当前企微看板卡片从“基础标题识别 + 通用 KPI 摘要”升级为“14 类展示模板动态卡片”，使不同问题在企业微信内呈现不同的经营门面、主指标、辅助指标和分析结构。

最终效果：

1. 用户问经营总览、漏斗、排行、区域、渠道画像、报备保护、商机风险、报价转订单、续费、产品、技术服务商、分销层级、经营节奏、数据质量权限时，卡片标题和指标自动匹配问题类型。
2. 企微模板卡片只承载首屏摘要：一个主指标、三到六个辅助指标、完整看板入口和反馈标识。
3. 长连接正文承载完整分析：问题复述、数据口径、权限口径、核心判断、明细摘要、风险建议和建议追问。
4. 图片看板和 HTML 报告作为增强展示，不替代企微正文兜底。
5. 卡片和正文使用同一个模板识别结果，避免“卡片展示”和“分析内容”不一致。

### 1.2 不做范围

本方案不改变以下主链：

1. 不改变 CRM 数据源优先级，仍以 CRM OpenAPI 实时数据为主，快照数据只作为明确降级或历史回放口径。
2. 不绕过现有权限、审计、白名单和企微身份映射。
3. 不把长篇分析塞入企微模板卡片字段。
4. 不在本阶段新增大量临时关键词散落在 `WecomBotService` 中。
5. 不把图片看板作为唯一交付形态，企微正文必须能独立读懂。

## 二、现状问题

### 2.1 已具备能力

当前项目已经具备以下能力：

| 能力 | 当前状态 | 位置 |
| --- | --- | --- |
| 企微长连接文本回复 | 已具备 | `backend/src/modules/wecom` |
| 企微模板卡片下发 | 已具备 | `WecomTransportService.sendTemplateCardMessage` |
| 看板结果生成 | 已具备 | `DashboardReportComposer` |
| 看板转企微格式 | 已具备 | `WecomBotService.convertDashboardResultToWecomFormat` |
| 看板卡片构建器 | 已具备基础版 | `WecomDashboardCardBuilder` |
| 完整报告入口 | 已具备 | `/api/v1/public/analysis-results/:queryId/file` |

### 2.2 当前缺口

| 缺口 | 影响 |
| --- | --- |
| 模板类型识别较粗 | 不同问题容易都落到经营总览卡 |
| 卡片指标选择不够语义化 | 漏斗问题可能没有突出最大断点，排行问题可能没有突出 TOP 占比 |
| 卡片与正文模板没有统一注册表 | 后续迭代容易再次出现设计和实际发送分叉 |
| 14 类模板没有逐类测试 | 只能证明通用卡片可发，不能证明各类问题展示正确 |
| 图片看板未按 14 类分层 | 类 HTML 展示增强还不能覆盖重点场景 |

## 三、总体设计

### 3.1 分层架构

```text
用户问题
  ↓
固定安全前置检查：签名、来源、身份、权限、危险动作
  ↓
分析意图和展示模板解析
  ↓
CRM OpenAPI / 受控查询执行
  ↓
DashboardComposeResult / AnalysisResult
  ↓
14 类展示模板注册表
  ↓
┌───────────────────────┬────────────────────────┬────────────────────┐
│ 企微模板卡片           │ 长连接 Markdown 正文     │ 图片看板 / HTML 报告 │
│ 首屏摘要入口           │ 完整经营分析             │ 类看板视觉和明细下钻 │
└───────────────────────┴────────────────────────┴────────────────────┘
```

### 3.2 核心模块拆分

| 模块 | 职责 | 建议文件 |
| --- | --- | --- |
| 模板注册表 | 定义 14 类模板的标题、关键词、指标优先级、正文段落和验收口径 | `wecom-dashboard-template.registry.ts` |
| 模板解析服务 | 根据问题、报告标题和看板区块解析模板类型 | `wecom-dashboard-template-resolver.service.ts` |
| 指标选择器 | 从 KPI、漏斗、排行、区域、分布等区块选择主指标和辅助指标 | `wecom-dashboard-kpi-selector.service.ts` |
| 卡片构建器 | 统一生成企微 `text_notice` 卡片 | `wecom-dashboard-card-builder.service.ts` |
| 正文渲染器 | 按模板输出 Markdown 主答复结构 | `wecom-dashboard-markdown-renderer.service.ts` |
| 图片看板渲染器 | 按模板输出图片看板输入数据 | `wecom-analysis-table-image.service.ts` 或新增独立服务 |

### 3.3 数据对象建议

```ts
type DashboardTemplateCode =
  | 'BUSINESS_OVERVIEW'
  | 'FUNNEL_DIAGNOSIS'
  | 'CHANNEL_RANKING'
  | 'REGION_COMPARISON'
  | 'CHANNEL_PROFILE'
  | 'REGISTRATION_PROTECTION'
  | 'OPPORTUNITY_RISK'
  | 'QUOTE_TO_ORDER'
  | 'RENEWAL_SUCCESS'
  | 'PRODUCT_SOLUTION'
  | 'SERVICE_ECOSYSTEM'
  | 'DISTRIBUTION_HEALTH'
  | 'CADENCE_REPORT'
  | 'DATA_SCOPE_QUALITY';

interface DashboardTemplateDefinition {
  code: DashboardTemplateCode;
  displayName: string;
  cardTitle: string;
  questionPatterns: RegExp[];
  reportTitlePatterns: RegExp[];
  requiredSections: string[];
  primaryMetricPriority: RegExp[];
  secondaryMetricPriority: RegExp[];
  markdownSections: string[];
  acceptancePoints: string[];
}
```

说明：

1. `questionPatterns` 用于用户原问题识别。
2. `reportTitlePatterns` 用于报告标题兜底识别。
3. `primaryMetricPriority` 用于选择 `emphasis_content`。
4. `secondaryMetricPriority` 用于选择 `horizontal_content_list`。
5. `markdownSections` 用于正文模板动态化。
6. `acceptancePoints` 用于单测和人工验收 checklist。

## 四、14 类模板定义

| 编码 | 展示模板 | 适用问题 | 主指标优先级 | 辅助指标优先级 | 正文重点 |
| --- | --- | --- | --- | --- | --- |
| `BUSINESS_OVERVIEW` | 经营总览看板卡 | 整体经营、全国渠道、发展运营、当前总览 | 商机金额、渠道商总数、综合经营指标 | 渠道商、活跃渠道、报备、报价金额、订单金额、最大风险 | 核心判断、核心指标、漏斗趋势、区域渠道、风险建议 |
| `FUNNEL_DIAGNOSIS` | 业务漏斗诊断卡 | 报备到订单、转化率、漏斗断点、掉点阶段 | 最大断点、关键转化率 | 报备、商机、报价、订单、断点阶段 | 漏斗总览、三段转化率、最大断点、优先动作 |
| `CHANNEL_RANKING` | 渠道贡献排行卡 | 渠道排行、贡献最大、TOP10、集中度 | TOP3 占比、头部渠道金额 | TOP1、TOP3 占比、商机金额、报价金额、长尾渠道 | 排序口径、多口径榜单、集中度、头部长尾动作 |
| `REGION_COMPARISON` | 区域经营对比卡 | 山东区、北京区、各区域、覆盖省份、区域缺口 | 领先区域、异常区域 | 覆盖省份、活跃渠道、商机金额、报价金额、订单金额 | 区域分层、重点区域、异常区域、负责人动作 |
| `CHANNEL_PROFILE` | 渠道画像诊断卡 | 渠道活跃、沉睡、激活、淘汰、高潜低转化 | 活跃渠道、沉睡渠道 | 新增、活跃、沉睡、待激活、高潜低转化 | 渠道分层、代表渠道、生命周期、激活动作 |
| `REGISTRATION_PROTECTION` | 客户报备与保护期卡 | 报备到期、重复报备、客户归属、渠道冲突 | 即将到期、重复报备数 | 即将到期、重复报备、待审批、驳回、归属风险 | 到期统计、冲突清单、归属风险、处理建议 |
| `OPPORTUNITY_RISK` | 商机风险清单卡 | 未报价商机、推进慢、高金额风险、预计签约 | 高风险商机数、风险金额 | 未报价数、停滞数、30 天内预计签约、负责人 | 核心风险、风险分层、商机清单、负责人动作 |
| `QUOTE_TO_ORDER` | 报价与订单转化卡 | 报价转订单、本周可能成交、报价优先级 | 高优先级报价数、报价金额 | 平均评分、超期报价、本周预计、负责人 | 预测声明、转化指标、优先级清单、评分解释 |
| `RENEWAL_SUCCESS` | 续费与客户成功卡 | 快续费客户、续费风险、存量客户经营 | 30 天到期客户、预计续费金额 | 高风险客户、已报价续费、已续费订单、负责人 | 续费概览、风险客户、客户成功动作 |
| `PRODUCT_SOLUTION` | 产品与解决方案结构卡 | 产品线表现、终端安全、方案推进、产品转化 | 高潜产品线、商机金额 | 产品线数、商机金额、报价金额、订单金额、转化率 | 产品结构、高潜方案、转化断点、经营建议 |
| `SERVICE_ECOSYSTEM` | 技术服务商生态卡 | 技术服务商、签约服务商、交付生态覆盖 | 签约技术服务商数、转签约候选数 | 签约、提名、未参与、覆盖区域、贡献金额 | 生态覆盖、贡献对比、覆盖缺口、发展建议 |
| `DISTRIBUTION_HEALTH` | 分销层级健康卡 | 一级二级渠道、上级确认、订单归属、层级协同 | 异常归属数、待确认订单数 | 一级渠道、二级渠道、跨区协同、待确认订单 | 层级结构、链路健康、异常清单、处理建议 |
| `CADENCE_REPORT` | 经营节奏日报/周报/月报卡 | 周报、月报、复盘、晨会重点、经营会议摘要 | 新增风险、完成待办 | 本期新增、本期推进、本期成交、新增风险、完成待办 | 会议摘要、本期变化、风险待办、负责人动作 |
| `DATA_SCOPE_QUALITY` | 数据质量与权限口径卡 | 看不到数据、数据准不准、字段缺失、权限口径 | 字段完整率、接口成功率 | 可见范围、缺失字段、异常接口、替代口径 | 可见范围、限制说明、影响分析、替代问题 |

## 五、分阶段实施方案

### 5.1 P0：14 类动态卡片注册表

#### 目标

先把企微模板卡片从基础识别升级为 14 类动态展示，保证用户第一眼看到的卡片标题、主指标、辅助指标与问题一致。

#### 实施内容

| 序号 | 实施项 | 说明 | 涉及文件 |
| --- | --- | --- | --- |
| P0-1 | 新增模板注册表 | 定义 14 类模板元数据、关键词、标题、指标优先级 | `backend/src/modules/wecom/wecom-dashboard-template.registry.ts` |
| P0-2 | 新增模板解析服务 | 输入用户问题、报告标题、看板区块，输出模板编码 | `backend/src/modules/wecom/wecom-dashboard-template-resolver.service.ts` |
| P0-3 | 新增指标选择器 | 从 `DashboardComposeResult.blocks` 中选择主指标和辅助指标 | `backend/src/modules/wecom/wecom-dashboard-kpi-selector.service.ts` |
| P0-4 | 扩展卡片构建器 | 支持传入模板定义、主指标、辅助指标、摘要说明 | `wecom-dashboard-card-builder.service.ts` |
| P0-5 | 改造看板桥接 | `convertDashboardResultToWecomFormat` 使用模板解析结果构建卡片 | `wecom-bot.service.ts` |
| P0-6 | 注册服务 | 在 `WecomModule` 中注册解析服务和指标选择器 | `wecom.module.ts` |
| P0-7 | 补 14 类单测 | 每类至少一条卡片构建和识别测试 | `backend/test/modules/wecom` |

#### 卡片字段规范

| 字段 | P0 要求 |
| --- | --- |
| `card_type` | 固定为 `text_notice` |
| `source.desc` | 固定为 `CRM智能助手` |
| `main_title.title` | 使用 14 类模板卡片标题 |
| `main_title.desc` | 当前权限范围和数据来源 |
| `emphasis_content.title` | 当前模板主指标值 |
| `emphasis_content.desc` | 当前模板主指标解释 |
| `sub_title_text` | 一句话经营摘要，不超过字段限制 |
| `horizontal_content_list` | 三到六个辅助指标，字段名必须是 `keyname` |
| `jump_list` | 至少包含 `查看完整看板` |
| `card_action` | 必须存在，避免企微拒收卡片 |
| `feedback.id` | 必须使用当前 `queryId`，不得使用随机时间戳 |

#### P0 验收

| 验收项 | 通过标准 |
| --- | --- |
| 模板覆盖 | 14 类模板注册表完整，编码唯一、标题唯一 |
| 识别准确 | 14 条代表问题全部识别到正确模板 |
| 卡片字段 | 每张卡片包含 `source`、`main_title`、`emphasis_content`、`horizontal_content_list`、`jump_list`、`card_action`、`feedback` |
| 指标数量 | 横向指标不少于 3 个、不超过 6 个；无足够指标时使用明确兜底项 |
| 主指标语义 | 漏斗问题突出断点或转化率；排行问题突出 TOP；区域问题突出区域；权限问题突出可见范围或数据质量 |
| 降级可读 | 指标缺失时卡片仍说明“已生成分析结果”，不展示空白或内部字段 |
| 自动化测试 | 14 类单测全部通过，传输层卡片测试通过，后端构建通过 |

#### P0 推荐测试问题

| 模板 | 推荐问题 |
| --- | --- |
| 经营总览看板卡 | 全国渠道商发展运营情况 |
| 业务漏斗诊断卡 | 报备到订单转化漏斗断点在哪里 |
| 渠道贡献排行卡 | 哪些渠道贡献最大，前十渠道是谁 |
| 区域经营对比卡 | 山东区渠道经营表现如何 |
| 渠道画像诊断卡 | 哪些渠道活跃，哪些渠道需要激活 |
| 客户报备与保护期卡 | 哪些客户报备快到期，有没有重复报备 |
| 商机风险清单卡 | 预计签约但还没有报价的商机有哪些 |
| 报价与订单转化卡 | 本周哪些报价最可能转订单 |
| 续费与客户成功卡 | 哪些客户 30 天内需要续费 |
| 产品与解决方案结构卡 | 终端安全相关商机和报价情况怎么样 |
| 技术服务商生态卡 | 技术服务商发展情况怎么样 |
| 分销层级健康卡 | 一级二级渠道协同是否正常 |
| 经营节奏日报/周报/月报卡 | 生成本周经营复盘 |
| 数据质量与权限口径卡 | 当前数据口径是否受我的权限影响 |

### 5.2 P1：14 类 Markdown 正文动态化

#### 目标

让长连接正文与卡片模板保持一致。卡片负责摘要入口，正文负责完整分析维度，避免“卡片像排行，正文却还是经营总览”的割裂。

#### 实施内容

| 序号 | 实施项 | 说明 | 涉及文件 |
| --- | --- | --- | --- |
| P1-1 | 新增正文渲染器 | 按模板编码输出不同正文结构 | `wecom-dashboard-markdown-renderer.service.ts` |
| P1-2 | 拆分通用段落 | 数据口径、权限口径、建议追问、查看完整报告保留公共实现 | 新增 renderer 或 helper |
| P1-3 | 拆分模板段落 | 漏斗、排行、区域、风险、权限等输出不同段落 | 新增 renderer |
| P1-4 | 保留通用兜底 | 未命中特定模板时使用经营总览结构 | `wecom-dashboard-template.registry.ts` |
| P1-5 | 补正文测试 | 14 类正文都检查必备段落和关键业务词 | `backend/test/modules/wecom` |

#### P1 正文结构要求

所有正文都必须包含：

```text
【展示模板】
【回复结构】
【问题复述】
【数据口径】
【权限口径】
【核心指标】或对应模板核心指标段
【明细摘要】或对应模板明细段
【风险建议】或对应模板动作段
【建议追问】
```

各模板允许增加特有段落：

| 模板 | 特有段落 |
| --- | --- |
| 业务漏斗诊断卡 | 漏斗总览、三段转化率、最大断点 |
| 渠道贡献排行卡 | 多口径榜单、集中度判断、头部长尾判断 |
| 区域经营对比卡 | 区域分层、重点区域、异常区域 |
| 渠道画像诊断卡 | 渠道分层、代表渠道、高潜低转化 |
| 客户报备与保护期卡 | 到期统计、冲突清单、归属风险 |
| 商机风险清单卡 | 核心风险、风险分层、负责人动作 |
| 报价与订单转化卡 | 预测声明、优先级清单、评分解释 |
| 数据质量与权限口径卡 | 可见范围、限制说明、影响分析、替代问题 |

#### P1 验收

| 验收项 | 通过标准 |
| --- | --- |
| 卡片正文一致 | 14 类样例中，卡片标题和正文展示模板一致 |
| 结构完整 | 每条正文包含硬性结构字段 |
| 分析维度匹配 | 漏斗问题必须有断点；排行问题必须有排序口径；权限问题必须有可见范围 |
| 不越权 | 不因用户问全国就绕过当前用户权限范围 |
| 不误导 | 订单不足时不得把报价或商机解释成真实成交 |
| 自动化测试 | 14 类正文结构测试全部通过 |

### 5.3 P2：图片看板和 HTML 报告增强

#### 目标

在企微模板卡片和正文之外，为复杂问题提供更接近看板的视觉表达，尤其是漏斗、排行、区域、风险清单和经营节奏报告。

#### 实施内容

| 序号 | 实施项 | 说明 | 涉及文件 |
| --- | --- | --- | --- |
| P2-1 | 定义图片看板输入模型 | 标题、口径、核心指标、图表数据、风险建议 | 新增图片看板类型 |
| P2-2 | 优先覆盖 6 类图片 | 经营总览、漏斗、排行、区域、商机风险、经营节奏 | 图片渲染服务 |
| P2-3 | HTML 报告分区增强 | 根据模板调整页面分区顺序 | 公开报告页面或生成器 |
| P2-4 | 图片失败兜底 | 图片生成失败不影响卡片和正文投递 | 企微分发层 |
| P2-5 | 截图验收 | 保存 6 类图片看板截图和 HTML 截图 | `docs/testing` |

#### P2 优先图片范围

| 5 | 商机风险清单卡 | 风险清单需要可执行优先级 |
| 6 | 经营节奏日报/周报/月报卡 | 可直接用于会议 |

#### P2 验收

| 验收项 | 通过标准 |
| --- | --- |
| 图片可读 | 标题、口径、核心指标、图表、风险建议在企微端可读 |

### 5.4 P3：300 问严格验收和治理闭环

#### 目标

在 14 类动态模板稳定后，按 300 问进行严格验收，找出模板误判、指标缺失、口径不清、明细不足和建议不可执行的问题，并通过反馈闭环持续优化。

#### 实施内容

| 序号 | 实施项 | 说明 |
| --- | --- | --- |
| P3-1 | 300 问模板映射 | 为每个问题标注预期模板编码 |
| P3-2 | 自动结构验收 | 检查展示模板、口径、权限、指标、明细、风险、追问 |
| P3-3 | 人工严格抽检 | 每类至少抽检 5 条，重点看业务语义 |
| P3-4 | 失败问题归因 | 分为模板误判、数据不足、指标缺失、权限口径、展示问题 |
| P3-5 | 修复回归 | 每个失败类别补至少一个回归测试 |
| P3-6 | 形成验收报告 | 输出通过率、失败清单、修复记录、遗留风险 |

#### P3 严格通过标准

| 维度 | 通过标准 |
| --- | --- |
| 结构完整率 | 300/300 必须包含硬性结构字段 |
| 模板匹配率 | 严格验收目标不低于 95%，P0 覆盖模板样例必须 100% |

## 八、风险与兜底

| 风险 | 表现 | 兜底方案 |
| --- | --- | --- |
| 模板误判 | 问区域却返回经营总览 | 解析服务按问题、报告标题、区块三层打分；低置信回退经营总览并记录日志 |
| 指标缺失 | 卡片主指标为空 | 指标选择器用模板兜底项，例如“已生成分析结果” |
| 卡片过长 | 企微截断或拒收 | 构建器统一裁剪字段长度 |
| 卡片发送失败 | 用户只看到正文 | 分发层记录错误，正文继续投递 |
| 数据不足 | 无订单、无报价、字段缺失 | 正文明确替代口径，不能把商机或报价解释为成交 |
| 权限不足 | 用户问全国但只能看局部 | 正文和卡片都显示当前可见范围，不越权推断 |
| 规则膨胀 | 关键词散落各处 | 所有模板规则集中到注册表，不在业务主链零散追加 |

## 十一、最终验收口径

本方案完成后，CRM 智能分析企微侧应满足以下最终验收口径：

1. 企微机器人不是固定卡片，而是根据问题动态选择 14 类模板。
2. 卡片、正文、图片和 HTML 使用同一模板识别结果。
3. 卡片用于摘要入口，正文用于完整分析，图片和 HTML 用于增强展示。
4. 所有结果都说明数据口径和权限口径。
5. 所有结果都能区分真实订单、报价、商机和报备，不误导成交判断。
6. 所有模板都有自动化测试和真实企微验收样例。
7. 300 问验收中，结构完整率达到 100%，模板匹配率进入可量化跟踪。
