## 二、第二阶段最终目标

第二阶段目标不是简单增加几个接口，而是把第一阶段“能在企微里问到”升级为最终需求要求的“真实用户可用、权限可信、分析更准、展示更强、Web 与企微双入口一致、治理可追踪”。

第二阶段完成后，系统应达到以下效果：

1. 用户可以用真实 CRM 账号登录 Web 工作台，并继承 CRM 真实角色、组织、部门、区域、渠道、负责人和企业微信映射。
2. 企业微信机器人和 Web 智能分析工作台使用同一套身份、权限、语义层、结果包和审计规则。
3. 高频经营问题不再只依赖六类对象列表聚合，而是优先通过分析镜像库、指标语义层和受控模板生成更稳定的统计结果。
4. 结果展示从纯 Markdown 升级为“企微摘要 + 图片卡片/模板卡片 + 完整 Web 报告链接”的组合。
5. Web 端具备可演示的智能分析报告页，展示指标卡、趋势图、排行表、明细表、口径说明、权限范围和数据更新时间。
6. 管理员可以在治理页面管理 OpenAPI 配置、真实登录配置、用户映射、权限诊断、模板命中、审计追踪和问题回归。
7. 第一阶段残留的高频问法误判、接口异常提示和模板匹配偏差得到修正。

## 四、总体架构

第二阶段采用“双数据路径、统一语义层、统一结果包”的架构。

```text
用户
  |
  | 企业微信机器人 / Web 智能分析工作台
  v
入口层
  |
  | 企微验签 / Web 登录态 / 真实 CRM 登录态
  v
身份权限层
  |
  | CRM 用户、角色、组织、部门、区域、渠道、负责人、企微 userid
  v
语义理解层
  |
  | 意图、指标、维度、时间、筛选、排序、追问上下文
  v
执行编排层
  |
  | 高频模板优先 -> 分析镜像库 -> 标准 OpenAPI -> 安全兜底
  v
统一结果包
  |
  | 指标、图表、表格、明细、口径、权限、数据新鲜度、审计 ID
  v
展示层
  |
  | 企微 Markdown / 图片卡片 / 模板卡片 / Web 报告页
```

核心原则：

1. 真实身份和权限优先来自 CRM，不允许用户手工声明权限。
2. 正式结果必须来自受控查询或标准 OpenAPI，不允许模型编造数值。
3. AI 负责理解、规划和解释，不能越过 SQL Guard、字段白名单和权限注入。
4. 企微和 Web 使用同一个 `AnalysisReportPayload`，避免两个入口口径不一致。
5. 分析库不可用时允许回退 OpenAPI，但必须记录回退原因。

## 五、第二阶段研发拆解

### 5.1 第 0 步：收口第一阶段尾项

目标：避免第二阶段建立在未修复的高频误判上。

任务：

1. 修复“负责人商机金额排名”槽位解析，不把“商机金额”识别为负责人名称。
2. 修复“客户未报备和未建商机情况”路由，优先进入客户生命周期、未关联商机或客户风险口径。
3. 对 `/analytics/customers/lifecycle` 增加非 JSON 响应保护，返回明确中文提示，并记录 CRM 接口异常。
4. 与 CRM 对接方确认客户生命周期接口真实路径、鉴权方式和返回样例。
5. 补充 3 条回归测试：负责人排名、客户未报备、客户生命周期异常。

验收：

1. 上述 3 类问法不再错误进入漏斗或负责人筛选。
2. 接口返回 HTML 时用户看到“CRM 接口返回格式异常”，而不是技术报错。

### 5.2 第 1 步：真实 CRM 登录与身份落位

目标：从本地 mock 登录和 client 绑定用户，升级为真实 CRM 用户登录。

任务：

1. 在联调管理页增加真实登录配置区：
   - `CRM_OPEN_API_BASE_URL`
   - `CRM_OPEN_API_CORP_ID`
   - `CRM_OPEN_API_DEVICE`
   - `CRM_OPEN_API_VERSION_CODE`
   - `CRM_OPEN_API_TIMEOUT_MS`
2. 接入真实登录接口，保存 `user_id`、`user_token`、过期时间和登录来源。
3. 登录成功后调用身份查询 API 或只读身份库，装载 CRM 用户快照。
4. 将真实用户身份映射到系统 `CrmUser`：
   - 用户 ID
   - 姓名
   - 角色
   - 组织
   - 部门
   - 区域
   - 大区
   - 渠道
   - 负责人范围
   - 企业微信 userid
5. 登录失败时按账号不存在、密码错误、账号禁用、接口超时分别展示中文提示。
6. 在审计中记录真实登录成功、失败、身份解析失败和权限装载失败。

验收：

1. 至少 2 个真实账号可登录 Web 工作台。
2. 高权限账号和受限账号看到的权限摘要不同。
3. 登录失败不会创建有效系统会话。

### 5.3 第 2 步：多角色权限矩阵

目标：让问数结果严格按真实角色裁剪。

任务：

1. 建立 4 类测试用户矩阵：
   - 超管
   - 区域管理员
   - 渠道管理员
   - 普通员工
2. 为每类用户沉淀预期可见范围：
   - 可见区域
   - 可见渠道
   - 可见负责人
   - 可见对象类型
   - 是否可导出
3. 对标准 OpenAPI 和分析镜像库分别验证权限一致性。
4. 在 `ScopeSnapshot` 中增加更清晰的中文权限说明。
5. 查询结果统一展示“当前权限范围”。
6. 越权问题必须拒答并审计，例如“看全国所有渠道明细”。

验收：

1. 同一问题由不同角色提问，结果范围不同且符合权限矩阵。
2. 受限用户不能通过追问、最近查询重跑或报告链接看到越权数据。

### 5.4 第 3 步：分析镜像库与语义层增强

目标：支撑复杂统计、跨对象漏斗和高频经营分析。

任务：

1. 扩展 OpenAPI 同步到 MySQL 分析库：
   - 用户维表
   - 渠道维表
   - 报备事实表
   - 商机事实表
   - 报价事实表
   - 订单事实表
   - 用户权限桥表
2. 建立字段目录：
   - 字段中文名
   - 来源对象
   - 指标含义
   - 可筛选
   - 可聚合
   - 可排序
   - 敏感等级
   - 权限注入字段
3. 建立指标目录：
   - 新增商机金额
   - 商机数量
   - 报备数量
   - 报备转商机率
   - 报价金额
   - 报价转订单率
   - 订单金额
   - 客户活跃度
   - 超期未跟进数量
4. 优先固化 P0 受控 SQL 模板：
   - 经营总览
   - 渠道贡献排名
   - 区域贡献对比
   - 负责人商机金额排名

1. P0 模板在超管和受限角色下都能执行。
2. 权限字段缺失时自动回退或明确提示，不硬查。
3. SQL 审计能追踪模板、参数、权限范围和返回条数。

### 5.5 第 4 步：自由问数与模板路由

目标：让用户不用背固定问题，也能稳定命中正确分析口径。

任务：

1. 建立问题归一化流程：
   - 业务对象识别

   - 追问动作识别

3. 对以下歧义问题先补问：
   - 缺少时间范围
   - 缺少分析对象
   - 指标可能多义
   - 维度不明确
4. 建立负例样本：
   - 合同口径和订单口径混淆
   - 证书认证和技术服务商类型混淆
   - 负责人名称和指标名称混淆

5. 把模板命中结果写入审计，便于后续优化。

验收：

1. 20 到 30 条最终问题集命中率达到可验收标准。
2. 命中失败时能补问或明确说明，不直接编造。

## 九、风险与兜底

| 风险 | 影响 | 兜底 |
| --- | --- | --- |
| 真实登录参数延迟 | Web 真实用户无法上线 | 继续使用第一阶段 client 绑定用户和多 client 角色联调 |
| 身份权限接口不稳定 | 权限无法安全落位 | 保守拒答或回退到已确认 client 权限 |
| 分析库同步失败 | 复杂问数不可用 | 回退标准 OpenAPI 问数链路 |
| AI SQL 生成不稳定 | 结果不准或被 Guard 拦截 | 高频问题优先固化受控 SQL 模板 |
| 图表或图片生成失败 | 企微展示降级 | 回退 Markdown 和 Web 报告链接 |
| 用户追问歧义 | 误解上下文 | 最小补问，不直接执行 |
| 对方接口返回 HTML 或非 JSON | 运行时报错 | 统一异常包装，返回中文提示并审计 |
