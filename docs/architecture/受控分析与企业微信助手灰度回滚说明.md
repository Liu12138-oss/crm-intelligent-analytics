# 受控分析与企业微信助手灰度回滚说明

## 1. 适用范围

本文档用于配套 `docs/architecture/受控分析与企业微信助手统一实施清单.md` 的第一轮落地，说明受控 AI 直查、grounded 洞察、企业微信结构化草稿、候选重排和日报洞察的开关、验证与回滚口径。

## 2. 灰度开关

| 开关 | 默认建议 | 影响范围 | 关闭后的回退路径 |
| --- | --- | --- | --- |
| `AI_GUARDED_DIRECT_QUERY_ENABLED` | `false` | Web / 企业微信问数显式受控直查模式 | 回退到 `PLAN_EXECUTION` 计划执行模式 |
| `ANALYSIS_QUERY_KNOWLEDGE_ENABLED` | `true` | 自由问数的别名、模板、码表、已验证示例和已发布语义资产知识层注入 | 关闭后仍允许受控直查，但不再注入统一问数知识层提示 |
| 无独立关闭开关 | 强制开启 | 自由问数标准时间槽、执行前时间字段白名单、结果包 `temporalScope` 一致性校验 | 若必须临时降级，只允许回退到补问或 `PLAN_EXECUTION`，不得恢复本地时间词表主链 |
| `AI_ANALYSIS_FOLLOW_UP_INTENT_ENABLED` | `true` | Web / 企业微信分析结果后的解释型追问与改条件追问 AI 分流 | 回退到既有关键词与默认追问分流 |
| `AI_GROUNDED_EXPLANATION_ENABLED` | `true` | 结果包 grounded 解释与建议追问 | 仅返回事实结果包与既有解释 |
| `ANALYSIS_MARKDOWN_DELIVERY_ENABLED` | `true` | Web Markdown 预览、统一结果包 Markdown 载荷与企业微信 Markdown 摘要交付 | 关闭后回退到现有结构化结果块与流式摘要下发 |
| `WECOM_AI_ENTRY_INTENT_ENABLED` | `true` | 企业微信空闲态帮助、日报查看、创建入口、显式项目查询，以及活跃任务下的帮助、取消、切换任务、直接提交、继续执行与修改意图 AI 分流 | 回退到最小安全 fallback：帮助、澄清或阻断；不得恢复旧规则主链 |
| `WECOM_AI_STRUCTURED_DRAFT_ENABLED` | `true` | 企业微信自由文本四段草稿、非必填缺项提示、直接提交 | 回退到原四段模板必填收集 |
| `WECOM_AI_ENTITY_RERANK_ENABLED` | `true` | 候选实体受控重排与推荐候选 | 回退到规则召回原排序 |
| `WECOM_AI_DAILY_INSIGHT_ENABLED` | `true` | 个人日报、团队预览、团队汇总的 grounded 洞察快照 | 回退到既有事实摘要 |

补充超时参数：

- `WECOM_AI_IDLE_INTENT_TIMEOUT_MS`
  用于企业微信空闲态消息分类超时；默认建议 `8000`
- `WECOM_AI_TASK_REPLY_TIMEOUT_MS`
  用于企业微信活跃任务下“确认 / 继续 / 提交 / 修改 / 取消 / 切换任务”等回复语义的 AI 理解超时；默认建议 `8000`
- `WECOM_AI_STRUCTURED_DRAFT_TIMEOUT_MS`
  用于企业微信自由文本四段草稿抽取超时；默认建议 `8000`

## 3. 快速验证

1. 打开 `AI_GUARDED_DIRECT_QUERY_ENABLED=true`，调用 Web 问数接口并传入 `executionMode=GUARDED_DIRECT_QUERY`，确认详情返回 `executionSnapshot`、`resultBundleSnapshot`、`insightSnapshot` 和 `deliverySnapshot`。
1.1 同时确认 Web 详情返回 `entryInterpretationSnapshot` 与 `workflowRoutingSnapshot`，可区分 AI 主链命中、fallback 原因、目标工作流和最终固定程序。
1.2 打开 `ANALYSIS_QUERY_KNOWLEDGE_ENABLED=true`，确认受控直查任务生成会优先注入已发布语义资产，并在存在静态兜底时继续补充别名、模板和已验证示例提示；关闭后应仅关闭知识层注入，不影响安全栈和基础直查能力。
1.2.1 在治理后台更新并发布一条语义资产后，确认新的问数请求能命中该资产，且结果详情和审计详情中的 `executionTraceSummary.knowledgeHits` 可看到对应资产名称。
1.3 打开 `ANALYSIS_MARKDOWN_DELIVERY_ENABLED=true`，确认 Web 返回完整 Markdown 结果预览，企业微信优先下发裁剪 Markdown 摘要；关闭后应回退为结构化结果块和流式摘要。
1.4 验证包含“最近四个月、前四个月、近 6 月、过去 12 个月、上季度、去年同期、2026 年一季度、本财年”的问数请求，详情页必须返回同一份 `temporalScope`，且执行快照中的任务时间槽、Markdown、图表标题和明细表标题不得出现不同时间口径。
1.5 验证“从那段时间开始看一下商机情况”等低置信表达，系统必须补问或阻断，不得默认查询全量历史数据。
2. 打开 `AI_ANALYSIS_FOLLOW_UP_INTENT_ENABLED=true`，分别在 Web 和企业微信中对已有结果追问非关键词形式的解释型问题与改条件问题，确认系统能按 AI 分流复用结果或重新分析。
3. 打开 `WECOM_AI_ENTRY_INTENT_ENABLED=true`，在企业微信活跃任务中发送非关键词帮助、取消、切换任务、直接提交和“我改一下”这类回复，确认系统能按 AI 分流保留或结束当前任务。
3.0 对空闲态帮助、跟进主题入口、日报查看、解释追问、改条件追问、新增客户、新增商机、小组日报预览和显式项目/客户查询，优先确认已走统一 AI 分类，而不是直接命中旧关键词分流。
3.0.1 关闭 `WECOM_AI_ENTRY_INTENT_ENABLED` 后，确认 `新增客户`、`把王文定小组日报发给我`、`查安恒信息项目`、活跃任务中的“我想先看看今天的日报”等旧规则入口不会直接恢复业务主链，而是统一回到帮助、澄清或阻断。
3.1 若确认类回复仍频繁因超时回退，可先把 `WECOM_AI_TASK_REPLY_TIMEOUT_MS` 调高到 `10000` 或 `12000` 再观察，不要直接扩规则词表。
3.2 若空闲态入口分类频繁超时，可先把 `WECOM_AI_IDLE_INTENT_TIMEOUT_MS` 调高到 `10000` 或 `12000`，优先保障 AI-first 路由。
3.3 审计中必须能直接看见 `entryInterpretationSnapshot.language`、`usedFallback`、`fallbackReason`、`targetWorkflow` 和 `workflowRoutingSnapshot.gateResult`，否则视为灰度不可放量。
4. 使用包含 `information_schema`、`DROP` 或 `load_file` 的 SQL 风险样例触发安全栈，确认请求被阻断且审计中保留阻断快照。
5. 通过企业微信跟进入口发送仅包含跟进内容的自由文本，确认系统展示结构化草稿、友好缺项提示和“也可以直接提交”的出口。
5.1 若四段草稿抽取经常超时，可先把 `WECOM_AI_STRUCTURED_DRAFT_TIMEOUT_MS` 调高到 `10000` 或 `12000`，优先保障 AI 主链，再评估模型与网关性能。
6. 在多候选场景下确认候选推荐只来自已召回集合，写回或创建场景仍需用户明确选择。
7. 生成个人日报或小组日报预览，确认 AI 洞察快照只记录当前事实数量、降级状态和失败原因，不触发新的正式汇总或通知。
8. 访问合同审核详情接口，确认返回 `naturalLanguageEntryCapability` 预留信息，明确“文件类型校验、文件大小校验、任务创建、权限校验”仍是固定程序前置步骤。

## 4. 回滚步骤

1. 先关闭 `AI_GUARDED_DIRECT_QUERY_ENABLED`，让新增直查请求回退到计划执行。
2. 若问数知识层引入后命中率或解释质量异常，关闭 `ANALYSIS_QUERY_KNOWLEDGE_ENABLED`，仅停用别名、模板、已验证示例和已发布语义资产提示，保留现有安全栈和受控直查能力。
3. 若 AI 时间槽质量短期不稳定，可临时将自由问数固定回 `PLAN_EXECUTION` 并保留同一 `TemporalSlot` contract；不得把 `time-range.util` 或本地时间词表恢复为自由问数主理解链路。
4. 若 `temporalScope` 展示异常，可暂时只在审计快照和 `appliedFilters` 中展示时间口径，但执行侧仍必须使用标准时间槽和时间字段白名单。
5. 若 Markdown 预览或企业微信 Markdown 摘要交付异常，关闭 `ANALYSIS_MARKDOWN_DELIVERY_ENABLED`，回退到现有结构化结果块与流式摘要。
6. 若分析 follow-up 分流不稳定，关闭 `AI_ANALYSIS_FOLLOW_UP_INTENT_ENABLED`，回退到既有关键词与默认追问分流。
7. 若企业微信空闲态与活跃任务入口分流不稳定，关闭 `WECOM_AI_ENTRY_INTENT_ENABLED`，统一回退到最小安全 fallback，只返回帮助、澄清或阻断，不得恢复旧的关键词主分流。
8. 若解释或建议追问质量异常，关闭 `AI_GROUNDED_EXPLANATION_ENABLED`，保留已落库快照用于复盘。
9. 若企业微信自由文本拆分误判率偏高，关闭 `WECOM_AI_STRUCTURED_DRAFT_ENABLED`，恢复原模板必填收集。
10. 若候选重排排序不稳定，关闭 `WECOM_AI_ENTITY_RERANK_ENABLED`，继续使用规则召回结果。
11. 若日报洞察低置信或事实不足，关闭 `WECOM_AI_DAILY_INSIGHT_ENABLED`，系统继续输出既有事实摘要。

## 5. 审计排障口径

- 分析问数优先查看 `executionSnapshot.executionMode`、`executionSource`、`matchedAdapter`、`gapReason`、`resultBundleSnapshot.consistencyToken` 和 `executionTraceSummary`。
- 若结果与口径理解不一致，优先查看 `executionTraceSummary.knowledgeHits`、`report.sourceNotes` 和 `report.footnotes`，确认命中的已发布语义资产、来源说明和脚注说明是否符合预期。
- 时间槽问题优先查看 `entryInterpretationSnapshot.structuredSlots.temporalSlot`、`executionSnapshot.taskSnapshots[].temporalSlot`、结果包 `temporalScope`、Markdown 时间口径和最近查询 `lastTemporalScope`，确认是否为同一边界。
- 统一入口优先查看 `entryInterpretationSnapshot.scene`、`targetWorkflow`、`confidence`、`usedFallback` 和 `workflowRoutingSnapshot.finalProgram`，确认是否发生了 AI 低置信或规则 fallback。
- grounded 洞察优先查看 `insightSnapshot.grounded`、`reusedResultBundle`、`failureReason`，确认是否发生二次查数或事实脱钩。
- 企业微信日报优先查看 `aiInsightSnapshot.scene`、`grounded`、`degraded` 和 `factCount`，确认是否已按事实不足自动降级。
- 企业微信统一入口优先查看 `entryInterpretationSnapshot.language`、`intent`、`replyIntent`、`targetWorkflow`、`usedFallback` 和 `workflowRoutingSnapshot.gateResult`，确认当前是 AI 主链命中、最小安全 fallback，还是执行门闩拦截。
- 合同审核当前仅开放固定上传入口；若后续增加自然语言解释追问或补充说明，必须先命中统一 AI 理解层，再进入固定合同审核程序。
- 导出与审计检索应复用同一份结果包快照，不允许为导出重新生成另一套分析口径。

## 6. 放量阈值建议

- `todayAiFallbackRatePercent < 20%`：可继续按当前灰度范围放量。
- `todayAiFallbackRatePercent >= 20%`：暂停放量，先检查 `entryFallbackReasonBreakdown` 与 `entryLanguageBreakdown`。
- 单一 `entryScene` 的 `fallbackRatePercent >= 30%`：暂停该入口灰度，优先排查对应 prompt、超时和网关稳定性。
- 单一 `fallbackReason` 连续两天位居第一，且占比超过 40%：优先修复该原因，未修复前不继续扩大灰度。
