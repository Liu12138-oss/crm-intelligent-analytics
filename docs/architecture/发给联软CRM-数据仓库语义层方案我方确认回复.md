## 1. 我方总体确认结论

我方确认认可联软侧提出的长期方向：

```text
联软 CRM 标准 OpenAPI / 可信只读数据源
  -> AI-agent 数据同步
  -> AI-agent 分析数据仓库
  -> 标准数据模型
  -> 业务语义层
  -> AI 查询计划
  -> 受控 SQL
  -> 经营分析报告
  -> 企业微信机器人 / Web 工作台展示
```

我方确认以下原则：

- 第一阶段以联软标准 OpenAPI 作为正式数据同步源和权限来源。
- 不要求联软第一阶段直接开放生产业务 SQLite 或生产业务库裸表。
- Text-to-SQL 只面向 AI-agent 侧分析库或只读语义视图执行，不直接访问联软 CRM 生产库。
- AI 负责自然语言理解、查询计划生成和报告表达；程序负责 SQL 安全校验、权限注入、执行限制和审计。
- 当前标准 OpenAPI 链路继续保留，用作第一阶段联调基线、同步源和 AI 离线兜底能力。
- 对于 AI 低置信、字段缺失或权限不明的问题，系统应追问或阻断，不再默认返回同一类兜底数据。

## 2. 对联软提出的 6 个需确认事项的我方回复

### 2.1 数仓是否由 AI-agent 侧承建，数据库是否必须为 MySQL

我方确认：

- 数据仓库、标准数据模型、语义层和受控 SQL 执行层由 AI-agent 侧承建。
- 这里的 MySQL 是指 AI-agent 侧分析库的推荐承载形态，不是要求联软 CRM 把现有 SQLite 改造成 MySQL。
- 联软 CRM 当前如果是 SQLite，可以继续保持 SQLite。第一阶段我方兼容“标准 OpenAPI + SQLite 只读副本 / SQLite 快照文件 / 文件快照”作为同步来源。
- AI-agent 侧第一阶段建议使用 MySQL 或 MySQL 兼容数据库作为分析库，便于后续建设 ODS、DWD、DWS、ADS 分层、索引、权限注入、SQL 审计和报表查询。
- 当前项目后端已有 `mysql2`、只读 SQL 校验、SQL 审计和受控查询相关基础，因此 AI-agent 分析库采用 MySQL 兼容形态落地成本更低。
- 后续如果数据量明显增大，可再评估 ClickHouse、StarRocks、Doris 或 PostgreSQL 等分析型存储。

建议第一阶段命名：

```text
AI-agent analysis warehouse / AI-agent 分析库
```

建议第一阶段分层：

| 层级 | 说明 |
| --- | --- |
| ODS | 保存联软 OpenAPI 同步原始数据和同步批次 |
| DWD | 清洗后的客户、伙伴、报备、商机、订单等标准明细 |
| DWS | 高频问题预聚合，例如伙伴贡献、客户生命周期、商机风险 |
| ADS | 面向企微和 Web 报告的应用结果或缓存 |

建议双方区分两个概念：

| 概念 | 建议形态 | 责任方 | 说明 |
| --- | --- | --- | --- |
| 联软数据源 | 标准 OpenAPI、SQLite 只读副本、SQLite 快照文件、文件快照 | 联软 CRM | 负责提供可信数据、字段口径、权限口径和同步条件 |
| AI-agent 分析库 | MySQL / MySQL 兼容库优先，后续可升级分析型数据库 | AI-agent | 负责承接同步数据、标准模型、语义层和受控 SQL |

如果联软只能提供 SQLite，建议提供以下方式之一：

| 方式 | 说明 | 我方处理 |
| --- | --- | --- |
| SQLite 只读副本文件 | 定时生成只读副本，不直接读取生产运行文件 | AI-agent 定时拉取或挂载后导入分析库 |
| SQLite 快照文件 | 按批次导出 `.sqlite` / `.db` 文件 | AI-agent 按批次解析并同步到 ODS |
| SQLite 表导出文件 | 导出 CSV / JSON / Parquet | AI-agent 按对象导入 ODS |
| OpenAPI + SQLite 补充 | OpenAPI 覆盖主链，SQLite 只补接口缺失对象 | AI-agent 合并同步并标记来源 |

不建议：

- 让 AI-agent 直接查询联软生产运行中的 SQLite 文件。
- 让 AI 直接对联软 SQLite 生成并执行 SQL。
- 要求联软为第一阶段额外改造成 MySQL 后才能联调。

### 2.2 OpenAPI 增量同步是否满足第一阶段数据接入

- 第一阶段不要求联软开放生产库直连，也不要求联软把 SQLite 改造成 MySQL。
- 如果部分 OpenAPI 对象暂时无法满足同步或字段完整性要求，可以由联软提供 SQLite 只读副本或快照文件作为补充数据源。

### 2.4 Text-to-SQL 是否只针对 AI-agent 分析库执行

我方确认：

- Text-to-SQL 只针对 AI-agent 分析库或受控只读语义视图执行。
- 不直接针对联软 CRM 生产库执行。
- 不直接针对联软业务 SQLite 生产文件执行。
- 不允许 AI 生成 SQL 后绕过程序校验直接执行。

推荐执行链路：

```text
用户问题
  -> AI 生成结构化查询计划
  -> 程序根据语义层生成或校验 SQL
  -> 程序注入当前 CRM 用户权限条件
  -> SQL 只读与白名单校验
  -> AI-agent 分析库执行
  -> 查询结果进入 AI 报告生成
```

AI 可以参与：

- 识别业务对象。
- 识别指标、维度、筛选、时间范围。
- 拆解复杂问题为多个子任务。
- 生成候选查询计划。
- 基于真实结果生成报告。

AI 不负责：

- 自行决定权限条件。
- 绕过字段白名单。
- 执行最终 SQL。
- 查询敏感字段。
- 编造查询结果。

### 2.5 SQL 安全层、权限注入和审计由 AI-agent 哪个模块承担

我方确认由 AI-agent 后端统一承担，建议模块边界如下：

| 能力 | 承担模块 | 说明 |
| --- | --- | --- |
| 语义目录 | `analysis semantic layer` | 对象、指标、维度、字段、字典、同义词 |
| 查询计划 | `AI capability pack / intent planner` | AI 输出结构化查询计划 |
| SQL 编译 | `query compiler` | 将查询计划转为受控 SQL 或 AST |
| SQL 安全校验 | `query risk guard`、`query AST validator` | 阻断写入、DDL、危险函数、未授权表字段 |
| 字段白名单 | `query whitelist / semantic field registry` | 只允许语义层登记字段 |
| 权限注入 | `scope injector / permission bridge` | 根据当前 CRM 用户权限强制追加查询条件 |
| 查询执行 | `analysis warehouse executor` | 连接 AI-agent 分析库执行只读查询 |
| 审计留痕 | `sql audit service`、`analysis audit` | 记录问题、计划、SQL、参数、权限快照、行数和耗时 |
| 脱敏处理 | `result normalizer / presenter` | 对敏感字段做脱敏或禁显 |

安全要求：

- 只允许单条 `SELECT`。
- 禁止 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`。
- 禁止跨库、系统表、危险函数。
- 所有用户输入必须参数化。
- 所有查询必须有 `LIMIT` 或分页。
- 所有查询必须有超时。
- 所有查询必须记录审计。
- 未完成权限注入的查询不得执行。
