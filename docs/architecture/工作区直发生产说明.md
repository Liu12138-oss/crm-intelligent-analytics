# 工作区直发生产说明

## 适用场景

- 当前本地工作区存在未提交代码，需要直接整体发布到生产环境。
- 线上服务器已完成基础安装，且使用 `/srv/crm-intelligent-analytics/releases/<版本号>` + `current` 软链接的方式切换版本。
- 生产环境变量统一放在 `/srv/crm-intelligent-analytics/shared/backend.env`，不跟随源码包上传。

## 发布脚本

- 脚本路径：`scripts/deploy-workspace-to-production.py`
- 运行环境：本地 `Python 3` + `paramiko`

## 首次使用前检查

1. 确认本机可执行 `py -3` 或 `python`。
2. 确认本机已安装 `paramiko`。
3. 确认服务器已安装 `Node.js 20`、`pnpm 8.15.9`、`nginx`、`systemd`。
4. 确认线上 `backend.env`、`nginx`、`systemd` 已准备完成。

## 推荐发布命令

```powershell
py -3 .\scripts\deploy-workspace-to-production.py `
  --host 10.10.3.241 `
  --user root `
  --verify
```

说明：

- `--verify` 会在本地先执行 `pnpm test` 和 `pnpm build`。
- 不在命令行直接传密码时，脚本会交互提示输入，避免密码进入终端历史。

## 脚本实际做的事

1. 排除 `.git`、`node_modules`、`dist`、`output`、本地 `.env.*.local` 等不应上线内容。
2. 将当前工作区打成一个临时 `tar.gz`。
3. 通过 SSH/SFTP 上传到服务器 `/tmp`。
4. 在服务器创建新的 release 目录并解包源码。
5. 读取 `/srv/crm-intelligent-analytics/shared/backend.env` 作为正式环境变量来源。
6. 执行 `pnpm install --frozen-lockfile`。
7. 即使线上 `NODE_ENV=production`，也会显式安装构建所需的 `devDependencies`。
8. 执行 `pnpm build` 构建前后端产物。
9. 将 `.runtime` 软链接到共享目录。
10. 切换 `current` 软链接到新版本。
11. 重启 `crm-intelligent-analytics`，并 reload `nginx`。
12. 验证 `10.10.3.241` 入口、后端 3001 端口和企业微信回调路由。

## 日常更新流程

1. 在本地完成代码修改。
2. 先执行：

```powershell
pnpm test
pnpm build
```

3. 执行发布脚本：

```powershell
py -3 .\scripts\deploy-workspace-to-production.py --host 10.10.3.241 --user root --verify
```

4. 发布完成后，检查：

```bash
systemctl status crm-intelligent-analytics --no-pager
journalctl -u crm-intelligent-analytics -n 100 --no-pager
curl -I http://10.10.3.241/
```

## 回滚方式

如新版本异常，可在服务器上把 `current` 指回上一版并重启：

```bash
cd /srv/crm-intelligent-analytics
ls -lah releases
ln -sfn /srv/crm-intelligent-analytics/releases/<上一版本目录> current
systemctl restart crm-intelligent-analytics
systemctl reload nginx
```

## 注意事项

- 正式密钥、数据库密码、企业微信 Secret 只允许放在服务器 `shared/backend.env`，不要放进发布包。
- 当前脚本默认发布“本地工作区现状”，包括未提交但未被排除的源码变更。
- 如需要保留本地临时压缩包排障，可加 `--keep-archive`。
