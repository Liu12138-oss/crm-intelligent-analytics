# 线上接口性能稳定治理方案

## 1. 文档目标

本文档用于指导 CRM 智能分析系统的长期性能稳定治理，重点解决线上接口在登录、权限解析、能力快照、审计查询和分析工作台初始化时出现的秒级等待问题。

方案目标不是改变现有功能边界，而是在保持一期功能、权限判断、审计留痕和运行态恢复能力不变的前提下，降低 Node.js 主线程阻塞、控制运行态文件膨胀、隔离慢 SQL 审计写入，并提升线上排障可观测性。

## 2. 当前现象与初步结论

### 2.1 线上现象

- 浏览器网络面板显示登录接口约 `27` 秒，能力、当前策略、模板、历史查询等接口出现 `2` 到 `4` 秒等待。
- 静态资源和页面首屏资源加载较快，慢点主要集中在后端接口。
- 服务器 CPU、内存和磁盘容量未出现明显瓶颈。
- Nginx 和后端进程均处于运行状态。

### 2.2 排查证据

- 后端日志在慢请求时间窗口出现 `CRM 身份查询超时，准备重建只读连接池`，标签为 `buildLiveUserContext:departments`，默认超时阈值为 `8000ms`。
- 从应用机使用同一套后端环境变量和 `mysql2` 连接 CRM 只读库时，`SELECT DISTINCT department_id FROM users_departments WHERE user_id = ?` 查询可在毫秒级完成，并命中 `index_users_departments_on_user_id`。
- 线上运行态文件 `/srv/crm-intelligent-analytics/shared/.runtime/app-storage.json` 约 `148MB`。
- 运行态文件内 `sqlAuditRecords` 约 `66932` 条，是文件体积膨胀的主要来源。
- 当前 `SqlAuditRepository.create()` 每新增一条 SQL 审计记录都会调用 `AppStorageService.persist()`，而 `persist()` 会同步序列化并重写完整 `app-storage.json`。

### 2.3 根因判断

线上慢请求由两个问题叠加导致：

1. **运行态审计记录无限增长**：SQL 审计长期追加到 `app-storage.json`，导致单文件持续膨胀。
2. **同步整文件持久化阻塞事件循环**：每条 SQL 审计都同步 `JSON.stringify` 并写入完整文件，文件越大，Node.js 主线程阻塞越明显。鉴权、能力快照、模板列表等轻量接口也会被同进程阻塞拖慢。
3. **CRM 身份查询连接池偶发卡住后放大登录耗时**：身份解析查询本身有索引，但当连接池内旧查询或网络抖动导致等待超过 `8000ms` 时，登录链路会触发连接池重建和重试，一次登录可能被放大到二十秒以上。

## 3. 治理原则

- **不影响现有功能**：保留登录、权限判断、分析问数、治理审计、合同审核、企业微信能力和运行态恢复能力。
- **审计不丢主链路**：审计写入失败或延迟不得阻塞用户接口成功返回；审计数据可降级落盘、异步补偿或按保留策略归档。
- **权限实时性优先**：会话、权限、数据范围和模板可见性仍按当前实时权限判断，不能为了性能复用长期陈旧授权快照。
- **可灰度、可回滚**：每个阶段通过环境变量或独立开关启用，异常时可以回退到上一稳定策略。
- **可观测先行**：上线优化前先补齐接口耗时、上游耗时、运行态体积和审计队列指标，避免后续再次靠猜测排障。

## 4. 分阶段方案

## 4.1 阶段一：生产止血与运行态控量

### 目标

在不改业务逻辑的前提下，快速降低 `app-storage.json` 体积和同步写盘阻塞，恢复登录和页面初始化接口的稳定响应。

### 措施

1. **备份运行态文件**
   - 在生产机备份 `/srv/crm-intelligent-analytics/shared/.runtime/app-storage.json`。
   - 备份文件放入受控目录，权限保持仅运维和应用用户可读。
   - 备份文件不得外传，不得包含在发布包或文档附件中。

2. **归档历史 SQL 审计记录**
   - 将 `sqlAuditRecords` 中较早记录导出为按日期分片的归档文件。
   - 线上运行态仅保留近期记录，例如最近 `7` 天或最近 `5000` 条。
   - 归档脚本必须只处理 `sqlAuditRecords`，不得修改会话、模板、权限、企业微信上下文和合同审核产物。

3. **增加运行态体积巡检**
   - 每日检查 `app-storage.json` 文件大小、`sqlAuditRecords` 数量和最近一次修改时间。
   - 文件超过 `50MB` 或 SQL 审计超过 `10000` 条时告警。

4. **限制 SQL 审计内存保留量**
   - 在 `SqlAuditRepository.create()` 写入后执行保留策略。
   - 默认只在运行态内保留最近 `N` 条，建议初始值为 `5000`。
   - 保留上限通过环境变量控制，例如 `SQL_AUDIT_RUNTIME_MAX_RECORDS`。

### 验收标准

- `app-storage.json` 体积降到可控范围，建议小于 `30MB`。
- 登录、能力快照、模板列表、最近查询接口在正常网络下稳定低于 `1` 秒。
- 审计中心仍可查询近期 SQL 审计记录。
- 历史审计可通过归档文件追溯，不影响近期治理审计功能。

### 回滚方式

- 如归档后发现运行态缺失关键数据，停止应用后用备份文件恢复。
- 如保留上限导致审计中心近期记录不足，可调大 `SQL_AUDIT_RUNTIME_MAX_RECORDS` 并重启服务。

## 4.2 阶段二：审计写入异步化

### 目标

让 SQL 审计写入不再阻塞用户请求主链路，避免轻量接口因审计落盘同步等待而变慢。

### 措施

1. **引入内存队列**
   - `SqlAuditService.execute()` 生成审计记录后先放入轻量内存队列。
   - 用户接口不等待完整持久化完成，只等待审计记录入队成功。
   - 队列长度达到阈值时触发告警或进入降级策略。

2. **批量刷盘**
   - 后台定时批量持久化审计记录，例如每 `1` 到 `3` 秒或每 `100` 条刷盘一次。
   - 单次刷盘只写必要变更，避免每条 SQL 都重写完整运行态文件。
   - 应用退出前尝试刷新队列，减少正常重启时的审计丢失窗口。

3. **审计降级策略**
   - 队列过长或持久化失败时，接口继续返回业务结果。
   - 系统记录降级告警，包括队列长度、失败原因、最近失败时间。
   - 降级期间可暂停低风险读查询审计，只保留失败、阻断、高风险和写回相关审计。

4. **保留关键审计同步路径**
   - 权限阻断、写回、导出、敏感 SQL reveal 等高风险事件可继续同步确认写入。
   - 普通只读 SQL 查询审计使用异步路径。

### 验收标准

- 普通只读 SQL 审计写入不再显著增加接口响应时间。
- 高风险审计事件仍能可靠落盘。
- 服务重启后近期审计记录不丢失或丢失窗口在业务可接受范围内。
- 审计队列状态可在日志或治理后台排障入口查看。

### 回滚方式

- 通过环境变量关闭异步审计，例如 `SQL_AUDIT_ASYNC_ENABLED=false`。
- 关闭后回到阶段一的保留上限策略，避免再次无限膨胀。

## 4.3 阶段三：审计存储独立化

### 目标

将 SQL 审计从 `app-storage.json` 中拆出，避免运行态快照承载高频流水数据。

### 推荐方案

优先将 SQL 审计迁移到独立数据库表，独立于 CRM 只读库和 CRM 写回库。建议使用应用侧存储库或独立 MySQL schema，避免污染 CRM 业务库。

### 数据表建议

建议新增 `sql_audit_records` 表，字段保留现有审计实体的核心信息：

- `id`
- `created_at`
- `stage`
- `status`
- `risk_level`
- `actor_id`
- `actor_role_ids_json`
- `channel`
- `session_id`
- `request_id`
- `module_key`
- `program_name`
- `database_role`
- `operation_type`
- `tables_json`
- `sql_summary`
- `normalized_sql`
- `sql_fingerprint`
- `params_json`
- `param_summary`
- `row_count`
- `affected_rows`
- `duration_ms`
- `timeout_ms`
- `blocked_reason`
- `error_message`

索引建议：

- `created_at`
- `actor_id, created_at`
- `module_key, created_at`
- `status, created_at`
- `sql_fingerprint, created_at`
- `request_id`
- `session_id`

### 兼容策略

1. **双读兼容**
   - 审计中心列表优先读取新表。
   - 若新表未启用，则回退读取 `app-storage.json` 中的 `sqlAuditRecords`。

2. **双写灰度**
   - 灰度期可同时写入新表和旧运行态保留区。
   - 对比两边记录数量、失败率和字段完整性。

3. **切换只写新表**
   - 新表稳定后，`app-storage.json` 仅保留必要运行态数据。
   - 旧文件中的历史 `sqlAuditRecords` 进入归档，不再参与高频接口读写。

4. **迁移脚本**
   - 提供一次性迁移脚本，把旧 `sqlAuditRecords` 分批写入新表。
   - 迁移过程中按批次提交，避免一次性加载和写入过大数据。
   - 迁移脚本必须支持断点续跑。

### 验收标准

- `app-storage.json` 不再随 SQL 审计量增长。
- SQL 审计列表、详情、reveal、统计功能保持可用。
- 审计查询分页在 `10万` 条记录规模下仍能稳定返回。
- 新表不可用时，业务主链路不受影响，并有明确降级日志。

## 4.4 阶段四：身份解析与权限快照稳定化

### 目标

减少登录后页面初始化期间的重复身份查询，降低 CRM 只读连接池抖动对用户请求的影响。

### 措施

1. **延长短时身份缓存**
   - 当前会话用户解析已有短缓存，默认约 `30` 秒。
   - 可通过 `CRM_AUTH_SESSION_USER_CACHE_TTL_MS` 调整到 `60` 到 `120` 秒。
   - 缓存只用于减少同一会话短时间内重复解析，不改变实时权限策略；权限配置变更时仍应主动失效相关缓存。

2. **身份上下文合并查询**
   - 将 `getUserById`、角色查询、部门查询尽量合并为更少 SQL。
   - 保留当前字段语义：用户、角色、部门、组织和企业微信映射不变。
   - 合并前后必须使用同一批真实用户样本对比输出。

3. **连接池健康探测**
   - 身份查询连续超时时记录连接池恢复事件。
   - 增加只读库轻量健康检查：连接耗时、`SELECT 1` 耗时、连接池可用性。
   - 不在用户请求内频繁执行健康检查，避免额外放大压力。

4. **连接池参数显式化**
   - 在生产环境显式配置 `CRM_READONLY_DB_CONNECT_TIMEOUT_MS`、`CRM_READONLY_IDENTITY_QUERY_TIMEOUT_MS`、`CRM_READONLY_IDENTITY_QUERY_RETRY_COUNT`。
   - 默认建议：
     - `CRM_READONLY_DB_CONNECT_TIMEOUT_MS=8000`
     - `CRM_READONLY_IDENTITY_QUERY_TIMEOUT_MS=5000`
     - `CRM_READONLY_IDENTITY_QUERY_RETRY_COUNT=1`
   - 调整前需结合生产网络和 CRM 数据库稳定性验证，不建议盲目缩短到过低。

### 验收标准

- 登录后同一页面初始化期间，身份解析 SQL 次数明显下降。
- 页面切换和刷新不再连续触发多次角色、部门查询。
- CRM 只读库偶发超时时，后端能降级提示或快速恢复，不再拖慢所有轻量接口。

## 4.5 阶段五：可观测性与容量基线

### 目标

建立长期运行指标，让性能问题在用户感知前暴露。

### Nginx 指标

Nginx access log 应增加以下字段：

- `$request_time`
- `$upstream_response_time`
- `$upstream_connect_time`
- `$upstream_status`
- `$request_id`

用途：

- 判断慢请求发生在浏览器到 Nginx、Nginx 到后端，还是后端内部。
- 对比同一接口在不同时间段的上游耗时。

### 后端指标

后端至少记录：

- 请求路径、状态码、总耗时。
- CRM Open API 调用耗时。
- CRM 只读 SQL 执行耗时。
- SQL 审计队列长度和刷盘耗时。
- `app-storage.json` 文件大小。
- `sqlAuditRecords` 运行态保留数量。
- 身份连接池重建次数。
- 事件循环延迟，可通过 Node.js `perf_hooks.monitorEventLoopDelay()` 采样。

### 告警建议

- 登录接口 `P95 > 3s`，触发告警。
- 常规查询接口 `P95 > 2s`，触发告警。
- `app-storage.json > 50MB`，触发告警。
- SQL 审计队列积压超过 `5000` 条，触发告警。
- 身份连接池重建 `10分钟内 >= 3次`，触发告警。
- Node.js 事件循环延迟 `P95 > 200ms`，触发告警。

## 5. 不影响现有功能的控制点

### 5.1 功能兼容

- 登录、登出、会话续期、企业微信扫码登录保持原接口和响应结构。
- 分析工作台的模板、最近查询、历史记录和能力快照字段保持兼容。
- 治理审计后台仍可查询 SQL 审计列表、详情和敏感参数 reveal。
- 合同审核、日报、企业微信机器人上下文不从 `app-storage.json` 中移除。

### 5.2 数据兼容

- 运行态裁剪仅处理 `sqlAuditRecords`。
- 其他数组和对象不得被归档脚本修改。
- 历史审计归档文件应保留完整字段，后续可迁移到新表。

### 5.3 权限兼容

- 最近查询、导出、模板可见性和治理后台仍按当前实时权限判断。
- 缓存只用于短时间减少重复身份装载，不能绕过权限变更后的失效机制。
- reveal 操作继续记录审计事件，不受普通 SQL 审计异步化影响。

### 5.4 发布兼容

- 所有新能力通过环境变量开关控制。
- 默认上线顺序为：只读观测 -> 保留上限 -> 异步写入 -> 独立存储。
- 每个阶段保留回滚路径，避免一次性大改。

## 6. 建议实施顺序

| 顺序 | 工作项 | 风险 | 是否需要停机 | 回滚难度 |
|------|--------|------|--------------|----------|
| 1 | 备份并归档历史 `sqlAuditRecords` | 低 | 建议短暂停应用 | 低 |
| 2 | 增加运行态审计保留上限 | 低 | 需要重启后端 | 低 |
| 3 | 增加 Nginx 和后端耗时日志 | 低 | 需要 reload 或重启 | 低 |
| 4 | 审计写入异步化 | 中 | 需要重启后端 | 中 |
| 5 | SQL 审计迁移到独立表 | 中 | 可灰度不停机 | 中 |
| 6 | 身份解析 SQL 合并和缓存失效增强 | 中 | 需要重启后端 | 中 |

## 7. 测试与验收清单

### 7.1 回归测试

- 账号密码登录成功。
- 无效账号密码返回友好错误。
- 登录后刷新页面仍能恢复会话。
- 分析工作台能力快照正常返回。
- 模板列表、最近查询、历史查询正常返回。
- 治理审计列表可分页查询。
- SQL 审计详情和 reveal 权限控制正常。
- 合同审核任务列表和详情正常。
- 企业微信机器人会话上下文不丢失。

### 7.2 性能测试

- 连续登录 `20` 次，记录登录接口平均值、`P95` 和最大值。
- 登录后同时请求能力、策略、模板、历史查询，验证页面初始化不出现长时间串行等待。
- 批量触发 `1000` 条普通 SQL 审计，确认用户接口不随审计数量线性变慢。
- 将 `app-storage.json` 控制在目标体积后，重启后端并验证启动时间。

### 7.3 灰度验证

- 先在测试环境用生产脱敏后的运行态文件验证归档脚本。
- 再在生产低峰期只启用保留上限。
- 观察至少一个工作日后，再启用异步审计。
- 新审计表灰度期间开启双写，并每日对账记录数量和失败数量。

## 8. 运维操作建议

### 8.1 日常巡检

每日巡检以下项目：

- `systemctl status crm-intelligent-analytics --no-pager`
- `journalctl -u crm-intelligent-analytics --since "1 hour ago" --no-pager`
- `du -h /srv/crm-intelligent-analytics/shared/.runtime/app-storage.json`
- SQL 审计运行态保留数量。
- 登录接口、能力快照接口、模板接口的 `P95`。

### 8.2 生产参数建议

建议在 `/srv/crm-intelligent-analytics/shared/backend.env` 中逐步补充非敏感性能参数：

```env
SQL_AUDIT_RUNTIME_MAX_RECORDS=5000
SQL_AUDIT_ASYNC_ENABLED=false
CRM_AUTH_SESSION_USER_CACHE_TTL_MS=60000
CRM_CAPABILITY_SNAPSHOT_TTL_MS=30000
CRM_READONLY_DB_CONNECT_TIMEOUT_MS=8000
CRM_READONLY_IDENTITY_QUERY_TIMEOUT_MS=5000
CRM_READONLY_IDENTITY_QUERY_RETRY_COUNT=1
```

说明：

- `SQL_AUDIT_ASYNC_ENABLED` 初始保持 `false`，待异步审计实现和灰度验证完成后再开启。
- 身份查询超时参数需要结合 CRM 数据库网络稳定性逐步调整。
- 环境变量只写参数名和值，不在文档中记录数据库密码、密钥或网关凭证。

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 归档脚本误删运行态数据 | 会话、模板或企业微信上下文异常 | 脚本只允许修改 `sqlAuditRecords`，执行前备份，执行后校验关键数组数量 |
| 审计异步化导致短时间内记录未落盘 | 审计中心查询有延迟 | 页面提示近期审计可能有秒级延迟，高风险事件保留同步写入 |
| 新审计表不可用 | 审计写入失败或积压 | 队列告警、降级记录、业务主链路不阻塞 |
| 身份缓存过长 | 权限变更生效延迟 | 缓存 TTL 控制在分钟级，权限策略变更主动失效 |
| Nginx 日志字段变更影响日志采集 | 运维报表解析失败 | 先新增独立 access log 格式或灰度日志文件，再切换采集规则 |

## 10. 最终目标状态

治理完成后，系统应达到以下状态：

- `app-storage.json` 只保存低频运行态，不承载高频 SQL 审计流水。
- 普通 SQL 审计写入异步化，高风险审计仍可靠记录。
- SQL 审计具备独立存储、分页查询、归档和清理能力。
- 登录和页面初始化接口不再受审计文件体积线性影响。
- CRM 身份查询具备短缓存、连接池恢复和健康指标。
- 运维可以通过日志和指标快速判断慢请求发生在 Nginx、后端、CRM Open API、CRM MySQL 还是审计持久化链路。

### 11.4 回滚说明

- 异步审计异常：设置 `SQL_AUDIT_ASYNC_ENABLED=false` 并重启后端，回退到同步写入加运行态保留上限。
- 独立存储异常：设置 `SQL_AUDIT_INDEPENDENT_STORE_ENABLED=false` 并重启后端，审计中心回退读取运行态保留区。
- 归档异常：停止应用后恢复归档脚本生成的 `app-storage-*.bak.json`。
- 身份缓存异常：降低 `CRM_AUTH_SESSION_USER_CACHE_TTL_MS`，或通过治理配置变更触发缓存清理。
