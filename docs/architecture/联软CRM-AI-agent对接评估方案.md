## 2. 先给结论

当前资料已经足够我们完成“对象建模评估”和“第一阶段只读联调方案设计”。与上一轮相比，新增的标准 API 契约显著降低了直接适配 SQLite 的必要性。

本次评估结论如下：

1. 对方给出的对象范围、字段字典、权限思路和标准 API 契约已经形成了较完整的第一阶段只读对接输入。
2. 对方当前 CRM 的核心业务对象更接近“渠道型 CRM”，主线是 `用户 -> 渠道 -> 报备 -> 商机 -> 报价 -> 订单`，与本项目现有的一期主线“用户 / 部门 / 客户 / 商机 / 合同 / 企业微信映射 / 审计治理”仍存在结构差异。
3. 由于现在已经提供了正式 API 契约，本项目不应再优先考虑“直接连 SQLite”，而应优先走“API-first 对接”。
4. 第一阶段已经可以启动“只读查询联调”，重点覆盖身份上下文、权限范围、六类对象查询和问数主题映射。
5. 综合当前项目代码现状与后续维护成本，最推荐方案调整为：“第一阶段走标准 API，只读联调先落地；第二阶段视复杂分析需求再补 MySQL 分析镜像库；第三阶段再补写回与企微深度映射”。

## 3. 当前资料能支撑什么

从已提供资料来看，对方已经明确了以下信息：

### 3.1 已明确的核心对象

1. 用户 `users`
2. 渠道 `partners`
3. 客户报备 / 客户档案入口 `registrations`
4. 商机 `opportunities`
5. 报价 `quotes`
6. 订单 `orders`

### 3.2 已明确的权限主轴

1. 用户角色：`superadmin`、`admin`、`partner_admin`、`staff`
2. 权限过滤主字段：`role`、`region`、`bigRegion`、`partnerId`、`partnerName`
3. 业务对象归属主字段：`assignedStaffId`、`assignedPartnerId`、`createdBy`

### 3.3 已明确的业务链路

1. 报备可以转商机
2. 商机可以关联合同前置对象中的报价
3. 报价可以进一步关联订单
4. 渠道存在一级 / 二级层级关系

这些内容说明：对方已经不是“完全没准备”，而是已经把 AI-agent 第一阶段最需要的对象和口径初步梳理出来了。

## 4. 与本项目现有能力的匹配度评估

## 4.1 高匹配部分

以下能力可以较自然地映射到本项目：

1. Web 智能分析问数
2. 企业微信机器人问数
3. 最近查询
4. 常用查询模板
5. 基于角色、区域、渠道、人员的权限过滤
6. 商机、报价、订单、客户报备类经营分析

原因是这部分能力主要依赖：

1. 统一的用户上下文
2. 可读业务对象
3. 明确的数据范围字段
4. 可归一化的分析主题

这些条件对方文档已经提供了较好的基础。

## 4.2 中等匹配部分

以下能力可做，但需要补一层统一语义映射：

1. 客户对象
   当前对方的 `registrations` 同时承担“客户报备”和“客户档案入口”职责，不能直接等同于本项目中的“稳定客户主数据”。
2. 组织权限
   当前对方更偏“区域 + 渠道 + 员工”的权限模型，而不是标准的“部门树 + 人员树”模型。
3. 负责人归属
   当前有 `owner`、`ownerId`、`assignedStaffId`、`createdBy` 等多个字段，需要统一界定哪个字段是“权限主归属”。
4. 企业微信身份映射
   当前资料未提供现成企微用户映射结构，需要补映射规则或补映射表。

## 4.3 低匹配或当前缺失部分

以下能力按现有资料暂时无法稳定复用：

1. 合同审核主链路
   本项目现有一期包含“CRM 待 1 级审批合同列表 + 合同审核详情 + 审批历史”，而对方目前提供的是 `quotes` 和 `orders`，未提供明确的“合同对象”和“审批对象”标准模型。
2. 企业微信目录同步
   本项目现有实现优先复用 CRM 内已有企业微信映射链，如 `wx_user_maps`、`wx_users` 等；对方资料中没有对应结构。
3. 企业微信扫码登录后的 CRM 用户落位
   当前只看到 SSO 字段，没有看到企微 `userid`、手机号匹配规则、唯一绑定策略。
4. 受控写回链路
   目前文档主要说明了查询对象，没有给出独立的跟进记录对象、写回接口契约、幂等规则和审计要求。

## 5. 当前最大风险点

## 5.1 风险一：直接适配 SQLite 会把项目改造成“联软专版”

本项目当前后端真实 CRM 数据链路是按 MySQL 读写模型设计的，若直接适配对方 SQLite：

1. 需要重做底层数据库驱动抽象
2. 需要重做 SQL 方言适配
3. 需要把大量对象字段映射写死到联软口径
4. 后续每次对方改表结构，我们都要同步跟着改

这会让项目从“通用 CRM 智能分析中台”变成“针对单一 SQLite CRM 的深度定制版本”，长期维护风险较大。

## 5.2 风险二：对象语义存在明显不对齐

本项目当前最核心的对象语义包括：

1. 用户
2. 部门 / 组织
3. 客户
4. 商机
5. 合同
6. 企业微信映射
7. 跟进写回

而对方当前最稳定的对象语义是：

1. 用户
2. 渠道
3. 报备
4. 商机
5. 报价
6. 订单

其中：

1. `registrations` 不能简单当作客户主表
2. `quotes` 不能简单当作合同
3. `orders` 不能直接替代合同审批流
4. 渠道树不能直接等同于部门树

如果不先做统一语义层，后面问数口径会漂。

## 5.3 风险三：企微对接条件尚不充分

企业微信接入稳定落地至少需要以下条件：

1. 企微用户如何映射到 CRM 用户
2. 企微部门如何映射到 CRM 组织
3. 企微扫码登录回来后如何唯一定位 CRM 用户
4. 当映射失败时如何降级

当前三份资料没有覆盖这部分核心契约，因此企微接入不能直接开工。

## 6. 方案选型评估

## 6.1 方案 A：AI-agent 直接连 SQLite

### 方案说明

由本项目直接读取对方 SQLite 数据库，按现有对象结构做适配。

### 优点

1. 前期不需要对方开发接口
2. 理论上最直接

### 缺点

1. 与本项目现有 MySQL 链路冲突较大
2. 代码改造面大
3. 后续结构变更维护成本高
4. 不利于权限、审计、写回能力长期收敛

### 结论

在已提供正式 API 契约的前提下，不再推荐该方案。该方案仅在 API 不可用或 PoC 极短期验证数据库字段时才有讨论价值。

## 6.2 方案 B：对方提供标准化适配 API

### 方案说明

由对方在 SQLite 之上补一层轻量 API / 适配服务，对外只暴露统一对象与统一动作。

目前对方已经在契约中提供或承诺了以下第一阶段接口：

1. `POST /auth/token`
2. `GET /auth/me`
3. `GET /meta/permission-scope`
4. `GET /meta/dictionaries`
5. `GET /users`、`GET /users/{id}`
6. `GET /partners`、`GET /partners/{id}`
7. `GET /registrations`、`GET /registrations/{id}`
8. `GET /opportunities`、`GET /opportunities/{id}`
9. `GET /quotes`、`GET /quotes/{id}`
10. `GET /orders`、`GET /orders/{id}`

如果后续需要写回，再补：

1. 跟进写回接口
2. 新增客户接口
3. 新增商机接口
4. 审批或状态推进接口

### 优点

1. 符合本项目“CRM API-first”的治理方向
2. 由对方封装自己的 SQLite 复杂度，我们不直接绑死在其内部表结构上
3. 后续升级 CRM 内部实现时，只要 API 契约不变，我们这边改动很小
4. 更适合承接企微登录、写回和权限判定

### 缺点

1. 需要对方把契约真正实现并开放联调环境
2. 六类对象目前仍偏业务原型口径，问数主题需要我方再做一层统一语义映射

### 结论

推荐作为当前正式主方案。

## 6.3 方案 C：SQLite 同步到 MySQL 镜像库

### 方案说明

由对方或双方联合把 SQLite 核心业务对象同步到 MySQL 标准镜像库，本项目继续走成熟的 MySQL 分析链路。

### 优点

1. 与本项目当前分析链路最匹配
2. 适合复杂聚合、问数、报表、模板、审计
3. 便于后续沉淀标准分析模型

### 缺点

1. 需要额外同步链路
2. 需要处理增量同步、一致性、延迟问题
3. 如果将来有写回，仍需要再补 API 或回写通道

### 结论

推荐作为“复杂分析问数增强方案”或“正式生产增强方案”，不必作为第一阶段前置条件。

## 6.4 方案 D：API + MySQL 镜像双层方案

### 方案说明

把查询与执行拆成两层：

1. 事务性动作、身份、权限、写回走 API
2. 分析问数、复杂聚合、模板、报表走 MySQL 镜像库

### 优点

1. 最符合本项目现有架构
2. 既保证分析性能，又保证写回和身份控制稳定
3. 最利于企业微信、Web、治理后台共用一套语义层

### 缺点

1. 方案最完整，实施工作量也最大
2. 需要双方技术配合程度较高

### 结论

这是最推荐的中长期目标架构。

## 7. 推荐落地方案

综合评估，推荐采用：

### 第一阶段：标准化适配 API

目标：先打通登录用户识别、对象查询、权限过滤和基础问数。

对方当前已经给出契约或应立即落地的接口：

1. `auth/token`
2. `auth/me`
3. `meta/permission-scope`
4. `meta/dictionaries`
5. 六类对象的列表与详情接口

对方仍需补充的最小项：

1. 联调环境 Base URL
2. `appKey`、`appSecret`
3. API 调用频率限制说明
4. `accessToken` 失效与刷新策略说明
5. 字典返回中的完整状态枚举
6. `opportunities` 的 `stage` 全量字典
7. 报价、订单的完整详情字段样例
8. 外部调用审计字段说明
9. 企微用户到 CRM 用户的映射规则
10. 企微部门到渠道 / 区域 / 组织的映射规则

我方 AI-agent 先承接：

1. Web 智能分析
2. 企业微信问数
3. 模板与最近查询
4. 治理审计

### 第二阶段：分析镜像库

目标：承接复杂聚合和稳定问数。

建议由对方补：

1. SQLite -> MySQL 同步任务
2. 统一标准化分析表
3. 增量同步策略
4. 时间字段和状态字段标准化

我方继续复用现有问数、模板、导出和审计链路。

### 第三阶段：受控写回与企微深度集成

目标：补齐跟进写回、企微扫码登录落位、组织映射与主动通知。

建议由对方补：

1. 用户与企微映射规则
2. 受控写回接口
3. 幂等规则与审计规则
4. 若存在审批流程，再补审批对象标准模型

## 8. 建议的目标架构

```text
企业微信机器人 / 企业微信扫码 / Web 工作台
                |
                v
        AI-agent 统一智能中台
                |
   +------------+-------------+
   |                          |
   v                          v
联软标准 OpenAPI            MySQL 分析镜像库
   |                          |
   +------------+-------------+
                |
                v
          联软 CRM SQLite
```

说明：

1. 用户识别、权限判定、对象详情查询优先走 API
2. 问数聚合、统计分析、复杂报表在第二阶段优先走 MySQL 镜像库
3. 写回、审批、企微身份辅助接口在第三阶段继续补 API
4. 这样既避免把 AI-agent 绑死在 SQLite 上，也能保留后续扩展空间

## 9. 对方下一步还需要补的资料

虽然当前四份资料已经足够进入第一阶段接口联调设计，但要真正进入稳定开发，还至少缺以下内容：

1. 联调环境与正式环境的 API 地址
2. `appKey`、`appSecret` 发放方式
3. 限流与超时策略
4. 完整字典值清单，尤其是商机阶段、报价状态、订单状态
5. 删除标记 / 失效标记规则
6. 用户与企微用户映射规则
7. 渠道 / 区域与企微组织映射规则
8. 跟进记录对象结构
9. 审批对象结构
10. 写回动作白名单与幂等规则
11. API 审计日志样例
12. 是否支持按更新时间增量拉取

## 10. 对本项目一期能力的影响判断

基于当前联软 CRM 资料，建议对一期能力按以下方式处理：

### 可优先保留

1. Web 智能分析工作台
2. 企业微信问数入口
3. 最近查询
4. 常用查询模板
5. 治理审计

### 需要按联软口径重映射

1. 客户对象
2. 组织 / 权限模型
3. 企业微信身份映射

### 建议延后或单独立项

1. 合同审核
2. 合同审批列表
3. 与合同审批强绑定的审查链路
4. 依赖企业微信身份强绑定的扫码直登落位

原因是当前对方尚未提供与本项目合同审核模型等价的“合同 + 审批 + 审批历史”结构，也尚未提供企微身份落位契约。

## 11. 最终建议

最终建议如下：

1. 不建议本项目直接适配对方 SQLite 作为正式方案。
2. 现在可以明确以对方已给出的“标准 OpenAPI 契约”作为第一阶段主路径。
3. 第一阶段可先落地只读能力：身份上下文、权限范围、六类对象查询、基础问数。
4. 复杂问数和统计分析第二阶段再评估是否补 MySQL 镜像库。
5. 企业微信接入要等“用户映射、组织映射、扫码登录落位规则”补齐后再正式实施。
6. 合同审核能力从当前联软 CRM 对接范围中暂时拆出，单独评估。

这条路径最稳，也最符合本项目现有代码结构、治理约束和后续维护成本控制目标。
