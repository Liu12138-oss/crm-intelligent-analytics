# 联软CRM-真实登录页面直测操作清单

## 1. 文档目的

本文用于指导我方在本地页面上直接验证“第二阶段真实登录链路”是否可用。

本轮页面直测的目标不是验证第一阶段标准 OpenAPI 六类对象查询，而是验证以下链路是否打通：

1. 页面登录表单提交到我方系统登录接口
2. 我方后端改走对方远端 mock 真实登录接口
3. 登录成功后按 `user_id` 完成身份落位
4. 页面成功进入系统并保持会话

说明：

1. 截至 2026-06-04，对方正式真实登录接口 `http://10.18.16.114:3000/api/v2/auth/login` 实测仍返回 `404 Cannot POST /api/v2/auth/login`
2. 因此当前页面直测，应基于“对方远端 mock 真实登录 + 远端 mock 身份查询 API”开展
3. 这一步通过后，可证明我方真实登录适配层已经具备切正式接口的落地条件

## 2. 启动前配置切换

页面直测前，请先调整 `backend/.env.development.local`。

### 2.1 必须确认的配置

以下配置应保持为当前联调值：

```env
CRM_OPEN_API_BASE_URL=http://10.18.16.114:3000
CRM_OPEN_API_LOGIN_PATH=/api/mock/crm-open-api/v2/auth/login
CRM_OPEN_API_VERSION_CODE=9.9.9
CRM_OPEN_API_DEVICE=open_api
CRM_AUTH_IDENTITY_API_BASE_URL=http://10.18.16.114:3000
CRM_AUTH_IDENTITY_API_USER_PATH=/api/mock/crm-open-api/v1/identity/users/{userId}
CRM_AUTH_IDENTITY_API_AUTH_MODE=none
```

### 2.2 页面直测前必须切换的开关

把：

```env
CRM_AUTH_MOCK_ENABLED=true
```

临时改成：

```env
CRM_AUTH_MOCK_ENABLED=false
```

原因说明：

1. `true` 代表仍优先走我方本地 mock 登录兜底
2. `false` 才会真正进入“真实登录适配层”
3. 如果不切这个值，页面上即使能登录，也不能证明真实 CRM 登录链路已生效

## 3. 启动方式

推荐在项目根目录执行：

```bash
pnpm dev
```

如果需要分别观察前后端日志，可分开启动：

```bash
pnpm dev:backend
pnpm dev:frontend
```

说明：

1. 修改 `.env.development.local` 后，至少需要重启后端
2. 如果前端登录页缓存了旧状态，建议刷新页面或重新打开浏览器标签页

---

## 4. 页面访问入口

本地开发环境请以 `pnpm dev` 实际输出地址为准，常见入口如下：

1. `http://127.0.0.1:5173/login`
2. `http://localhost:5173/login`
3. 若前端控制台显示的是局域网地址，也可以直接访问对应 `/login`

如果你本机已有其他服务占用了默认前端端口，以终端实际打印地址为准。

## 6. 页面直测操作步骤

### 6.1 登录成功链路验证

对每个成功账号，按以下步骤执行：

1. 打开 `/login`
2. 输入用户名和密码
3. 点击登录
4. 观察是否成功进入系统首页或默认落地页
5. 观察页面右上角或个人信息区的用户名是否与预期一致
6. 按 `F5` 或浏览器刷新
7. 观察刷新后是否仍保持登录状态
8. 再打开一个受保护页面，确认不是“刚登录成功但实际没有会话”

### 6.2 登录失败链路验证

对失败账号，按以下步骤执行：

1. 打开 `/login`
2. 输入 `15192888211 / 123456`
3. 点击登录
4. 预期页面提示登录失败
5. 后端应返回与“待审批”一致的错误语义，而不是本地 mock 账号错误提示

## 7. 每个账号建议核验点

页面直测时，不要只看“能不能进首页”，建议至少核验以下内容：

1. 登录是否成功
2. 页面展示的用户姓名是否正确
3. 刷新后是否保持登录
4. 退出后是否能正常回到登录页
5. 重新登录另一个角色时，页面身份是否切换干净

如果你还要顺带做角色权限视角验证，可继续补查：

1. 高权限账号能看到的页面，低权限账号是否被收敛
2. 同一列表页中，不同角色看到的数据量是否有差异
3. 是否出现“页面能进，但接口无权限”的半成功状态

---

## 8. 后端日志关注点

页面直测时，建议同步盯后端日志，重点看以下两段：

1. `CRM Open API 登录请求完成`
2. `真实登录身份查询 API 请求完成`

当前首轮验证通过时，预期会看到：

1. 登录请求 endpoint 为 `/api/mock/crm-open-api/v2/auth/login`
2. 身份查询请求地址为 `http://10.18.16.114:3000/api/mock/crm-open-api/v1/identity/users/{userId}`

如果日志仍显示本地 mock 逻辑，说明配置还没真正切过去。

## 9. 常见问题排查

### 9.1 页面提示“账号或密码错误”

优先排查：

1. `CRM_AUTH_MOCK_ENABLED` 是否仍是 `true`
2. 后端是否重启
3. 当前是否仍走我方本地 mock 登录

这类现象通常说明：

1. 还没切到真实登录适配层
2. 页面测到的并不是对方远端 mock 真实登录链

### 9.2 页面提示“CRM 登录失败”

优先排查：

1. 当前登录路径是不是被切到了正式 `/api/v2/auth/login`
2. 对方该路径当前是否仍返回 `404`
3. `CRM_OPEN_API_LOGIN_PATH` 是否被误改

### 9.3 登录成功但页面异常或白屏

优先排查：

1. 后端是否成功调用了身份查询 API
2. 身份查询返回里是否带回 `roleIds`、`organizationIds`、`departmentIds`
3. 前端是否因为某个页面接口失败而误以为登录失败

### 9.4 登录后刷新掉线

优先排查：

1. 后端会话是否已建立
2. `/api/v1/auth/session` 是否可正常续会话
3. 浏览器是否拦截了 Cookie

## 10. 页面直测通过标准

本轮页面直测可判定“通过”的最低标准如下：

1. 至少 2 个成功账号可通过页面正常登录
2. 至少 1 个失败账号返回符合预期的失败提示
3. 登录成功后刷新页面不掉线
4. 后端日志确认已走远端 mock 真实登录接口，而不是本地 mock 登录
5. 身份查询已成功按 `user_id` 落位

满足以上条件，即可认为：

```text
我方第二阶段真实登录页面链路已具备切正式接口前的联调基础
```

## 11. 测试完成后的恢复动作

页面直测结束后，如你还要继续使用本地默认账号 `admin / admin123`，请恢复以下配置：

把：

```env
CRM_AUTH_MOCK_ENABLED=false
```

改回：

```env
CRM_AUTH_MOCK_ENABLED=true
```

然后重启后端。

恢复后说明：

1. `admin / admin123` 会重新可用
2. 页面登录将再次回到本地 mock 模式
3. 这不会影响第一阶段标准 OpenAPI 六类对象联调能力
