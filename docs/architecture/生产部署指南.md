# 生产部署指南

## 1. 文档目标

本文档面向运维、实施和项目交付人员，说明 CRM 智能分析系统在生产环境中的推荐部署方式、资源要求、目录规划、环境变量、发布步骤、验收方法、回滚策略和常见排障点。

本文档聚焦当前仓库已落地的实现形态：

- 前端：`Vue 3 + Vite` 打包后的静态资源
- 后端：`NestJS` 编译后的 Node.js 服务
- 运行时状态：`.runtime/app-storage.json` 与 `.runtime/contract-review/`
- 外部依赖：`CRM Open API`、`CRM MySQL`、`AI 网关`、`企业微信 HTTP / WebSocket`

配套示例文件位于：

- `docs/architecture/deploy-examples/backend.production.env.example`
- `docs/architecture/deploy-examples/crm-intelligent-analytics.service`
- `docs/architecture/deploy-examples/crm-intelligent-analytics.nginx.conf`
- `docs/architecture/deploy-examples/install-production-host.sh`
- `docs/architecture/deploy-examples/release-production.sh`
- `docs/architecture/deploy-examples/生产部署参数清单.md`

## 2. 适用范围

本文档适用于以下场景：

- 首次在生产环境部署本系统
- 从测试环境迁移到生产环境
- 在生产环境执行版本升级、回滚或例行巡检
- 向运维团队交接运行要求

如仅需本地开发启动，请优先查看 `README.md` 与 `specs/001-crm-intelligent-analytics/quickstart.md`。

## 3. 部署总览

当前推荐采用单机或双机的经典前后端分离部署方式：

1. 通过 `pnpm build` 生成前后端产物。
2. 使用 `Nginx` 承载前端静态文件，并反向代理后端接口。
3. 使用 `systemd` 或 `pm2` 托管后端 Node.js 进程。
4. 使用独立环境变量文件为后端注入生产配置。
5. 将 `.runtime/` 挂载到持久化磁盘，避免进程重启后会话和合同审核产物丢失。

### 3.1 推荐拓扑

```text
公网用户 / 企业微信
        |
        v
   [ Nginx : 80/443 ]
      |          |
      |          +--> frontend/dist 静态资源
      |
      +--> 127.0.0.1:3001
               |
               v
     [ Node.js / NestJS 后端 ]
               |
               +--> CRM Open API
               +--> CRM MySQL
               +--> AI 网关
               +--> 企业微信 HTTP / WebSocket
               +--> .runtime/app-storage.json
               +--> .runtime/contract-review/
```

### 3.2 当前实现限制

- 当前仓库没有内置 `Dockerfile`、`docker-compose` 或 Kubernetes 清单。
- 当前仓库没有独立 `/health` 健康检查接口。
- 前端 `VITE_API_BASE_URL` 在构建时写入产物，发布后不能像后端环境变量那样热修改。
- 后端会优先读取进程环境变量；若未提供，会继续尝试读取仓库根目录或 `backend/` 下的 `.env.local`。生产环境允许这样工作，但不建议把敏感配置长期散落在代码目录内。

## 4. 服务器要求

## 4.1 推荐硬件

### 标准生产配置

- CPU：`8 核`
- 内存：`16 GB`
- 磁盘：`300 GB SSD`

该配置适用于以下组合负载：

- Web 智能分析工作台
- 企业微信机器人问数与跟进写回
- 合同审核上传、规则快审与 AI 补充审核
- 治理后台、审计中心、日报与通知

### 起步配置

- CPU：`4 核`
- 内存：`8 GB`
- 磁盘：`80 GB SSD`

仅建议用于低并发、演示或试点环境，不建议承接正式的合同审核与企业微信高峰流量。

### 扩展建议

- 如果企业微信并发较高、AI 网关响应较慢、合同审核任务多，建议拆分为两台应用机，并在前面挂负载均衡。
- 如果合同审核产物和原始合同文件保留周期较长，建议额外挂载数据盘，不要把 `.runtime/contract-review/` 与系统盘混用到告警阈值附近。

## 4.2 推荐操作系统

- 架构：`Linux x86_64`
- 现网生产环境：`Rocky Linux 9.5`
- 生产部署建议：继续与现网保持一致，优先使用 `Rocky Linux 9.5`

本文档中的命令示例以 `Rocky Linux 9.5` 为例，若后续新增生产服务器，建议继续沿用同版本或同大版本 Rocky Linux，降低系统运维差异。

## 4.3 必备软件

| 软件 | 建议版本 | 用途 |
|------|----------|------|
| Node.js | `20 LTS` | 运行后端、执行前后端构建 |
| pnpm | `8.15.9` | 与仓库 `packageManager` 保持一致 |
| Nginx | 稳定版 | 承载前端静态资源并反向代理后端 |
| systemd | 系统自带 | 托管后端进程 |
| Git | 稳定版 | 拉取代码、切换版本 |

## 5. 网络与安全组要求

## 5.1 入站端口

公网或负载均衡层建议只开放：

- `80`
- `443`

后端默认监听：

- `3001`

但 `3001` 不建议对公网开放，建议仅供：

- 本机 `Nginx` 回源
- 内网负载均衡
- 受控运维入口

## 5.2 出站访问

生产服务器必须能够访问：

- `CRM Open API`
- `CRM MySQL`
- `AI 网关`
- `企业微信 HTTP 接口`
- `企业微信 WebSocket 接口`
- `DNS`
- `NTP` 或企业内部时间同步服务
- 企业内部软件源或公共软件源

## 5.3 证书与域名

- Web 访问建议使用正式域名，例如 `analytics.company.com`
- 生产环境必须配置 HTTPS 证书
- 企业微信扫码登录回调地址必须与正式域名、正式协议保持一致

## 6. 目录规划

以下为推荐的发布目录：

```text
/srv/crm-intelligent-analytics/
├── current/                      # 当前线上版本软链接
├── releases/
│   ├── 20260427-120000/
│   │   ├── backend/
│   │   ├── frontend/
│   │   ├── docs/
│   │   ├── package.json
│   │   └── ...
│   └── 20260428-090000/
├── shared/
│   ├── .runtime/
│   │   ├── app-storage.json
│   │   └── contract-review/
│   ├── logs/
│   └── backend.env
└── backups/
```

### 6.1 目录说明

- `releases/`：保存每次发布的完整代码与构建产物，便于回滚。
- `current/`：指向当前线上版本。
- `shared/.runtime/`：保存运行态文件，切版本时不能丢。
- `shared/backend.env`：保存后端生产环境变量，避免把明文配置写进代码仓库。
- `shared/logs/`：如果使用文件日志，可集中落盘到这里。

### 6.2 权限建议

- 应用用户建议单独创建，如 `crmapp`
- `shared/backend.env` 权限建议为 `600`
- `shared/.runtime/` 需要对应用用户可读写
- Nginx 静态资源目录需要对 Nginx 用户可读

## 8.5 前端构建变量

前端最关键的是：

- `VITE_API_BASE_URL`

该变量会在构建时写入 `frontend/dist/`，因此：

- 如果后端正式入口是 `https://analytics.company.com`
- 且后端 API 由同域名 Nginx 代理到 `/api/v1`

则推荐在构建前设置：

```bash
export VITE_API_BASE_URL=https://analytics.company.com
```

不要让前端使用默认回退值 `http://127.0.0.1:3001`，否则浏览器会错误请求用户自己的本机地址。

## 10. 后端进程托管

## 10.1 systemd 服务示例

创建 `/etc/systemd/system/crm-intelligent-analytics.service`：

```ini
[Unit]
Description=CRM Intelligent Analytics Backend
After=network.target

[Service]
Type=simple
User=crmapp
Group=crmapp
WorkingDirectory=/srv/crm-intelligent-analytics/current/backend
Environment=NODE_ENV=production
EnvironmentFile=/srv/crm-intelligent-analytics/shared/backend.env

# 当前实现会从仓库根定位 .runtime，因此工作目录只放在 backend/ 不影响运行态定位。
ExecStart=/usr/bin/node /srv/crm-intelligent-analytics/current/backend/dist/src/main.js
Restart=always
RestartSec=5
TimeoutStartSec=60

# 统一使用 journal，避免默认把日志写散到代码目录。
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

启用并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable crm-intelligent-analytics
sudo systemctl start crm-intelligent-analytics
sudo systemctl status crm-intelligent-analytics
```

仓库中已提供同内容的示例文件：

脚本支持以下动作：

- `sync-all`
- `preview`
- `reminders`
- `close`
- `summaries`
- `night-batch`
- `morning-batch`

说明：

- `sync-all` 调用 `/api/v1/internal/wecom-directory-sync/run`
- `preview` 调用 `/api/v1/governance/daily-report-delivery/preview`
- `reminders` / `close` / `summaries` 调用 `/api/v1/daily-reports/cron/*`
- `daily-reports/cron/*` 当前仍依赖登录态，脚本会先通过 `/api/v1/auth/login` 换取会话 Cookie
- 因此生产环境需要在 `backend.env` 中额外提供 `SCHEDULER_BASE_URL`、`SCHEDULER_LOGIN`、`SCHEDULER_PASSWORD`；这些变量只供脚本使用，不被后端应用本身直接消费

推荐联调或正式调度顺序：

```cron
55 21 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh sync-all
00 22 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh reminders
59 23 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh close
50 07 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh sync-all
00 08 * * * /srv/crm-intelligent-analytics/shared/scripts/run-daily-report-jobs.sh summaries
```

## 11. Nginx 配置

## 11.1 站点配置示例

创建 `/etc/nginx/sites-available/crm-intelligent-analytics.conf`：

```nginx
upstream crm_analytics_api {
    server 127.0.0.1:3001;
    keepalive 32;
}

server {
    listen 80;
    server_name analytics.company.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name analytics.company.com;

    ssl_certificate     /etc/nginx/ssl/analytics.company.com.crt;
    ssl_certificate_key /etc/nginx/ssl/analytics.company.com.key;

    root /srv/crm-intelligent-analytics/current/frontend/dist;
    index index.html;

    client_max_body_size 20m;

    location /api/ {
        proxy_pass http://crm_analytics_api;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;
    }

    location = /index.html {
        add_header Cache-Control "no-cache, no-store, must-revalidate" always;
        add_header Pragma "no-cache" always;
        add_header Expires "0" always;
    }

    location /assets/ {
        add_header Cache-Control "public, max-age=31536000, immutable" always;
        try_files $uri =404;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

启用配置：

```bash
sudo ln -s /etc/nginx/sites-available/crm-intelligent-analytics.conf /etc/nginx/sites-enabled/crm-intelligent-analytics.conf
sudo nginx -t
sudo systemctl reload nginx
```

仓库中已提供同内容的示例文件：

- `docs/architecture/deploy-examples/crm-intelligent-analytics.nginx.conf`

补充说明：

- `index.html` 必须显式关闭缓存，否则浏览器可能继续使用旧入口页，进而加载旧的前端 bundle。
- `assets` 目录下的构建文件名自带 hash，适合开启长缓存，不会影响发布后拿到新版本。

## 11.2 上传大小说明

合同审核支持 `.docx` 上传，默认最大文件大小受：

- 后端 `CONTRACT_REVIEW_MAX_FILE_SIZE_MB`
- Nginx `client_max_body_size`

共同约束。

如果后端允许 `10 MB`，Nginx 不应配置成更小的值，否则请求会在到达应用前被拦截。

## 16. 常见问题与排障

## 16.1 页面能打开，但接口全是 `127.0.0.1:3001`

原因通常是：

- 前端构建前没有正确设置 `VITE_API_BASE_URL`

处理方式：

1. 重新设置正式域名
2. 重新执行前端构建
3. 重新发布 `frontend/dist/`

## 16.2 后端启动成功，但扫码登录回跳错误

已知生产门户注意事项：

- 门户平台可能隐藏外层代理上下文，扫码初始化请求中只能看到内层 IP 形态的 `webBaseUrl` / `returnUrl`，且没有 `GratuitousProxy`。
- 如果企业微信换票成功、共享 Cookie 已写入，但浏览器停在门户 `302 Found` 页面，通常不是登录失败，而是最终落点缺少门户代理上下文。
- 当前代码要求：共享 Cookie 直登成功后，带 `GratuitousProxy` 的成功回跳继续直接 `302` 到业务页；缺少该参数时返回自动 `window.history.back()` 的登录完成页，回到扫码前的门户上下文。排障和改造时不得把这个兜底删掉。

排查顺序：

1. 检查 `APP_WEB_BASE_URL`
2. 检查 `WECOM_WEB_LOGIN_CALLBACK_URL`
3. 检查企业微信后台登记的回调域名与路径
4. 检查 `APP_WEB_ALLOWED_BASE_URLS` 是否包含门户 Web 基址和内网 Web 基址
5. 检查 `APP_WEB_SHARED_COOKIE_DOMAIN` 是否允许企业微信回调域与门户域共享登录 Cookie
6. 检查 Nginx 是否正确代理 `/api/`
7. 查看 `/api/v1/auth/wecom/initiate` 日志中的 `webBaseUrl` / `returnUrl` 是否带 `GratuitousProxy`；如果线上真实请求拿不到该参数，应确认后端返回自动回退页，而不是继续直接跳门户业务页

## 16.3 企业微信机器人无法建立稳定连接

排查顺序：

1. 检查出站网络是否可访问企业微信 HTTP 与 WebSocket
2. 检查 `WECOM_BOT_ID`、`WECOM_BOT_SECRET`、`WECOM_BOT_SIGNATURE`
3. 检查服务器时间是否同步
4. 查看后端日志中的重连与投递告警

## 16.4 企业微信主动通知频率限制

排查顺序：

1. 在审计中心查看失败任务的最终通道、外部错误码和中文失败原因。
2. 如果外部错误码为 `846607`，按“企业微信机器人发送频率超限”处理，确认固定周期日报通知是否误走机器人通道。
3. 检查 `WECOM_BOT_PROACTIVE_MIN_INTERVAL_MS`、`WECOM_BOT_RATE_LIMIT_RETRY_DELAYS_MS`、`WECOM_APP_MESSAGE_MIN_INTERVAL_MS` 和 `WECOM_APP_MESSAGE_RATE_LIMIT_RETRY_DELAYS_MS` 是否使用上线默认值。
4. 手工补发失败通知时必须逐条发送，每条间隔 1 到 2 分钟；不得批量并发补发。

## 16.5 合同上传失败

排查顺序：

1. 检查 Nginx `client_max_body_size`
2. 检查后端 `CONTRACT_REVIEW_MAX_FILE_SIZE_MB`
3. 检查 `.runtime/contract-review/` 目录权限
4. 检查磁盘容量是否充足

## 16.6 AI Profile 可保存，但无法激活

排查顺序：

1. 检查 `AI_PROFILE_MASTER_KEY` 是否已配置
2. 检查 AI 网关地址、模型名、密钥是否正确
3. 检查生产机是否需要代理访问 AI 网关
4. 查看治理后台健康检查返回的脱敏失败原因

## 16.7 服务重启后企业微信上下文丢失

排查顺序：

1. 检查共享 `.runtime/app-storage.json` 是否存在
2. 检查应用用户是否有写权限
3. 检查版本切换时是否误删或覆盖 `.runtime`
