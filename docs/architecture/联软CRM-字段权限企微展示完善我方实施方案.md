# 联软CRM-字段权限企微展示完善我方实施方案

## 1. 文档目的

本文档用于明确 AI-agent 侧针对联软渠道 CRM 接入的后续实施方案，重点覆盖三块：

1. 联软 API 字段完整性治理。
2. 不同角色权限下的数据口径验证。
3. 企业微信端更稳定、美观、可信的报告展示。

本方案只列 AI-agent 我方需要做的工作。联软侧需要补充的字段、权限和样例资料，见《联软CRM-需补充完善字段与资料清单》。

## 2. 总体原则

1. 继续优先走联软标准 OpenAPI，不直接依赖联软 SQLite 表结构。
2. 继续复用现有 `analysis` 主链路，不新增一套联软专用问数入口。
3. AI 负责自然语言理解和报告表达，程序负责权限、字段白名单、接口调用、聚合和审计。
4. Web 与企业微信共享同一份结果包，避免两个入口口径不一致。
5. 所有新增能力必须兼容现有商机、合同、客户、企微、AI 配置和审计能力。

## 3. 当前运行链路

```text
Web / 企业微信输入自然语言
  -> 登录态或企微身份映射为 CRM 用户
  -> 装载当前权限范围
  -> AI 解析意图
  -> 查询规划器生成受控任务
  -> 字段白名单和安全校验
  -> 联软标准 OpenAPI 取数
  -> 本地安全聚合
  -> 结果归一化
  -> AI 富报告生成
  -> Web / 企业微信展示
  -> 审计记录
```

## 4. 模块改造范围

| 模块 | 改造目标 |
|---|---|
| `crm-standard-api` | 补字段归一、字段完整性诊断、字典适配、错误码翻译 |
| `analysis` | 补联软语义层、服务商画像、转化链路、字段缺失降级 |
| `governance` | 补联软连接配置、权限视角诊断、字段完整性页面 |
| `wecom` | 优化企微报告结构、图片表格、追问复用和失败降级 |
| `ai-models` | 复用已有 AI Profile，不新增模型调用方式 |
| `audit` | 记录用户、角色、OpenAPI client、权限范围、字段缺失和 requestId |
| `frontend` | 补诊断页、权限视角验证页、字段完整性状态展示 |

## 5. 阶段一：字段完整性治理

### 5.1 目标

让系统明确知道联软标准 API 每个对象有哪些字段可用、哪些字段缺失、哪些字段可筛选、哪些字段可聚合，并在缺字段时给出可信提示。

### 5.2 我方任务

1. 建立联软字段能力矩阵。
2. 在 `crm-standard-api` 中维护六类对象字段定义。
3. 补充字段中文名、业务含义、是否敏感、是否可展示。
4. 对字段缺失情况做统一降级。
5. 在 AI 提示词和语义层中明确可用字段边界。
6. 在结果报告中展示“字段缺失说明”，避免 AI 编造。
7. 在诊断接口中输出字段完整性状态。

### 5.3 建议新增字段能力结构

```ts
interface LianruanFieldCapability {
  resource: string;
  field: string;
  label: string;
  requiredLevel: 'P0' | 'P1' | 'P2';
  available: boolean;
  filterable: boolean;
  sortable: boolean;
  aggregatable: boolean;
  sensitive: boolean;
  dictionaryKey?: string;
  missingBehavior?: string;
}
```

### 5.4 首批重点修复

| 问题 | 我方处理 |
|---|---|
| `是否技术服务商` 字段缺失 | 标准字段未补前，报告明确提示“不具备独立字段”，不编造 |
| 服务商加入时间 | 明确按 `partners.createdAt` 作为加入时间筛选 |
| 服务商等级 | 用 `partnerLevel` + `partnerLevels` 字典中文化 |
| 区域 / 大区 | 优先使用 `region`、`bigRegion`，缺失时报告说明 |
| 金额单位 | 在 adapter 层统一为元，展示层按万元格式化 |
| 状态字段 | 统一字典中文化，避免直接显示 `active` 等内部值 |

### 5.5 验收用例

1. “最近一年加入的服务商有多少家，合作级别、等级、是否技术服务商维度一起给我。”
2. “山东区域最近三个月有商机的服务商，商机数量和金额是多少？”
3. “本年度订单金额按服务商排名前十。”
4. “最近半年报价转订单情况。”
5. “按服务商等级看商机金额分布。”

验收要求：

1. 不再误报“当前查询访问了未授权字段”。
2. 字段缺失时明确说明缺什么，不返回假结论。
3. 所有字段展示为中文业务含义。
4. Web 和企业微信口径一致。

### 5.6 当前落地状态

截至当前版本，阶段一已完成第一批基础能力：

1. 已新增联软字段能力矩阵，覆盖 `users`、`partners`、`registrations`、`opportunities`、`quotes`、`orders` 六类对象。
2. 已在字段矩阵中标记字段中文名、优先级、是否可筛选、是否可排序、是否可聚合、是否敏感和字典依赖。
3. 已把 `partners.isTechnicalServiceProvider` 标记为 P0 缺失风险字段，并明确缺失时不生成“是否技术服务商”维度结论。
4. 已把字段完整性诊断接入联软标准 OpenAPI 诊断接口，可基于远端样例返回字段完整度、P0 缺失字段和资源样例状态。
5. 已在“连接策略”页面新增“联软 OpenAPI 可分析性检查”，展示字段完整度、P0 缺失字段、字典完整度、绑定用户和六类资源样例数量。
6. 已补充后端单测和前端构建验证。

## 6. 阶段二：不同角色权限口径验证

### 6.1 目标

确保不同角色账号在 Web 和企业微信中看到的数据范围，与联软 CRM 权限口径一致。

### 6.2 我方任务

1. 建立角色权限测试矩阵。
2. 新增或完善“权限视角诊断”接口。
3. 登录后保存 CRM 用户身份快照。
4. 标准 OpenAPI 调用时记录绑定 client、当前用户、权限范围。
5. 对同一问题使用不同账号跑对比测试。
6. 在结果中展示“当前数据口径”。
7. 在审计中记录权限范围、OpenAPI `requestId` 和筛选条件。

### 6.3 权限视角诊断输出

```json
{
  "crmUserId": "A030",
  "userName": "刘龙海",
  "role": "superadmin",
  "scopeType": "all",
  "regions": [],
  "partnerIds": [],
  "clientMode": "bound-client",
  "boundClientUserId": "A030",
  "resources": {
    "partners": { "sampleCount": 20 },
    "opportunities": { "sampleCount": 15 },
    "quotes": { "sampleCount": 8 },
    "orders": { "sampleCount": 5 }
  }
}
```

### 6.4 角色测试矩阵

| 角色 | 测试重点 |
|---|---|
| 超管 | 是否全量可见，是否不被本地白名单误挡 |
| 区域负责人 | 是否只看所属区域 / 大区 |
| 渠道管理员 | 是否只看绑定服务商及允许下级 |
| 普通员工 | 是否只看自己负责、创建、分配或授权的数据 |
| 禁用账号 | 是否无法登录或查询 |

### 6.5 核心验证问题

1. “最近三个月山东区域有商机的服务商数量和金额。”
2. “最近一年加入的服务商按等级统计。”
3. “本年度订单金额按服务商排名。”
4. “本月报价转订单情况。”
5. “服务商画像按是否技术服务商统计。”

同一问题分别用不同角色执行，记录：

1. 返回行数。
2. 金额总计。
3. 可见服务商。
4. 可见区域。
5. 结果是否与联软侧预期一致。

### 6.6 多用户 OpenAPI 策略

短期仍可使用多 client 映射：

```text
CRM 用户 / 角色 / 权限范围
  -> appKey / appSecret
  -> 标准 OpenAPI token
  -> 对应权限数据
```

正式建议切换用户态 token：

```text
真实登录 user_token
  -> 标准 OpenAPI
  -> 联软侧按真实用户权限裁剪
```

我方需要同时保持兼容：

1. 有用户态 token 时优先使用用户态 token。
2. 没有用户态 token 时使用配置的多 client 映射。
3. 都没有时保持当前单 client 绑定用户一致性校验，不允许普通用户借用超管 client 扩权。

### 6.7 当前落地状态

截至当前版本，阶段二已完成第一批诊断基础能力：

1. 联软标准 OpenAPI 诊断接口已输出 `permissionView`。
2. `permissionView` 包含当前绑定 CRM 用户、角色、权限范围、绑定 client 用户、当前登录用户和两者是否一致。
3. 诊断接口会对六类资源分别读取最小样例，输出 `AVAILABLE`、`EMPTY`、`FAILED` 状态、样例数、总数和远端 `requestId`。
4. 前端连接策略页已展示权限视角和六类资源样例状态。

待继续完成：

1. 四类角色真实账号的对比测试记录。
2. 多 client 映射或用户态 token 正式策略。
3. 审计中补充联软 `requestId` 与字段缺失诊断快照。

## 7. 阶段三：企业微信报告展示优化

### 7.1 目标

企业微信端不只是回复一段文字，而是稳定输出移动端可读的经营报告，包含摘要、指标、关键发现、重点汇总、建议和必要的表格图片。

### 7.2 我方任务

1. 统一企微报告结构。
2. 优化表格图片布局，避免文字遮挡。
3. 长表格自动截断为 Top 10。
4. 列过多时自动选择关键列。
5. 图片生成失败时降级为 Markdown 表格，不再声称已生成图片。
6. 支持“看明细”“看分布图”“看趋势图”“分析原因”等追问复用上一轮结果。
7. 默认把 AI 分析报告、关键发现、重点汇总、经营建议发送到企业微信。

### 7.3 企微报告结构

```text
标题

查询口径：
- 时间范围
- 数据范围
- 当前角色视角

核心指标：
- 服务商数量
- 商机数量
- 商机金额

关键发现：
1. ...
2. ...
3. ...

重点汇总：
- ...

经营建议：
1. ...
2. ...

结果预览：
- Markdown 表格或图片表格

可继续回复：
- 看明细
- 看分布图
- 分析原因
```

### 7.4 图片表格规则

| 场景 | 规则 |
|---|---|
| 行数小于等于 10 | 直接展示 |
| 行数大于 10 | 展示 Top 10，并提示可回复“看明细” |
| 列数大于 6 | 自动选择关键列 |
| 服务商名称过长 | 优先用简称；没有简称时自动换行或截断 |
| 金额为空 | 显示“未填金额”，不显示 0 元误导 |
| 图片生成失败 | 降级 Markdown 表格 |

### 7.5 企微追问能力

| 用户追问 | 处理方式 |
|---|---|
| 看明细 | 复用上一轮 `queryId` 展示明细 |
| 看前 10 | 复用上一轮结果取 Top 10 |
| 看分布图 | 复用上一轮结果重新发送图片 |
| 看趋势图 | 复用上一轮趋势切片 |
| 分析原因 | 基于上一轮结果生成解释，不重新查数 |
| 导出报告 | 若用户有权限，返回 Web 详情或导出入口 |

## 10. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 联软字段未补齐 | 分析维度缺失 | 明确缺字段说明，不编造 |
| 权限返回空数组但无错误 | 容易误判无数据 | 要求联软提供无权限错误码或诊断字段 |
| 单 client 绑定超管 | 普通用户可能扩权 | 保持绑定用户一致性校验 |
| 企微图片遮挡 | 用户无法阅读 | 自动截断、换行、分页 |
| AI 跑题 | 结果不可信 | 程序固定执行边界，AI 只做理解和表达 |
| 字段改名 | 查询失效 | 字段能力诊断和回归测试 |

## 12. 推荐实施顺序

```text
第 1 步：字段能力矩阵和缺失字段降级
第 2 步：权限视角诊断和四类账号验证
第 3 步：服务商画像、商机贡献、报价、订单典型问题回归
```
