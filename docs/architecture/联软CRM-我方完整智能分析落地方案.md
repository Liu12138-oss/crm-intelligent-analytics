# 联软CRM-我方完整智能分析落地方案

## 1. 目标

我方目标是把联软渠道 CRM 接入 AI-agent 后，形成“真实登录、权限识别、自然语言理解、标准 API 取数、渠道 CRM 语义映射、分析结果展示、审计排障”的完整智能分析闭环。

本方案只列我方需要做的工作，不包含对方开发清单。对方需要配合提供的接口、字段和样例，已在《联软CRM-渠道CRM完整智能分析接入分工清单》中单独说明。

---

## 2. 总体技术路线

我方采用以下路线：

```text
真实登录 / 企业微信身份
  -> CRM 用户身份落位
  -> 权限范围装载
  -> AI 自然语言意图解析
  -> 联软渠道 CRM 语义层
  -> 标准 OpenAPI 数据查询
  -> 本地安全聚合 / 可选分析镜像库
  -> 结果归一化
  -> AI 经营分析报告
  -> 页面展示 / 企业微信回复
  -> 审计与诊断
```

核心原则：

1. 优先复用当前 `analysis` 主链路，不另做一套联软专用分析入口。
2. 优先走联软标准 OpenAPI，不直接绑定 SQLite 表结构。
3. 登录、权限、查询、展示、审计全部保持可回退、可诊断。
4. 先做首批可验收问题，再扩展到更多主题。
5. 不影响现有 MySQL / mock / 企业微信 / 合同审核等功能。

## 4. 阶段二：真实登录与身份落位

## 4.1 正式真实登录接入

当前已有真实登录兼容能力，但正式路径仍需接入。我们要做：

1. 保留当前 `CRM_AUTH_MOCK_ENABLED` 灰度开关。
2. 支持正式 `CRM_OPEN_API_LOGIN_PATH`。
3. 登录请求支持：
   - `login`
   - `password`
   - `corp_id`
   - `device`
   - `version_code`
4. 登录成功后读取：
   - `data.user_id`
   - `data.user_token`
5. 登录失败时把对方错误翻译为友好中文。

## 4.2 身份查询 API 接入

真实登录后必须把 `user_id` 转成我方 `CrmUser`：

1. 调用 `CRM_AUTH_IDENTITY_API_USER_PATH`。
2. 支持 `crm-token` 和 `none` 两种鉴权模式。
3. 映射字段：
   - 用户 ID
   - 用户姓名
   - 角色
   - 区域
   - 大区
   - 渠道
   - 负责人范围
   - 是否管理员
   - 企业微信用户 ID
4. 身份缺失时禁止建立会话，避免无权限用户进入系统。

验收标准：

1. 超管、区管、渠道管理员、员工四类账号均可登录。
2. 待审批、禁用、密码错误能正确失败。
3. 登录后页面显示正确用户和权限范围。
4. 刷新页面后会话仍能恢复。

## 5. 阶段三：标准 OpenAPI 多用户鉴权

当前标准 OpenAPI 主要是单 client 绑定用户模式。要支持完整智能分析，我们需要做一层鉴权策略。

## 5.1 短期方案：多 client 映射

适合联调和灰度：

1. 增加配置结构：`crmUserId / role / scopeType -> appKey/appSecret`。
2. 当前登录用户进入分析时，按身份选择对应 client。
3. 标准 API token 缓存按 client 维度隔离。
4. 日志记录使用的 clientId，不记录 appSecret。

## 5.2 正式方案：用户态 token

适合生产：

1. 标准 API 客户端支持真实登录 `user_token`。
2. 每次分析按当前登录用户访问数据。
3. 移除“标准 API 绑定用户必须等于当前用户”的临时限制。
4. API 返回权限由联软 CRM 统一裁剪，我方只做二次安全收窄。

验收标准：

1. 同一个问题，四类账号看到的数据范围不同。
2. 员工账号无法看到超出个人范围的数据。
3. 页面结果显示当前数据来源和权限范围。

## 7. 阶段五：多对象智能分析执行器

当前已有商机首批执行器，需要扩展成多对象执行能力。

当前落地状态：

1. 标准 OpenAPI 分析执行器已由商机专用聚合扩展为可按资源分发的多对象聚合层。
2. 已覆盖 `opportunities` 商机、`registrations` 客户报备、`partners` 渠道、`quotes` 报价、`orders` 订单的首批列表聚合能力。
3. 已支持总览、负责人排名、时间趋势、状态/阶段/等级分布、区域贡献等首批结果形态。
4. 当前仍保留“标准 API 绑定用户必须等于当前登录用户”的安全限制；正式多用户分析需要继续落地多 client 映射或用户态 token。

## 7.1 商机分析

支持问题：

1. 最近一个月山东区商机情况。
2. 本季度各渠道商机金额排名。
3. 本月商机阶段分布。
4. 各负责人商机金额排名。
5. 高风险商机概览。

实现动作：

1. 继续扩展 `opportunities` 执行器。
2. 补金额、数量、阶段、负责人、渠道、区域聚合。
3. 加强时间和区域过滤。

## 7.2 报备分析

支持问题：

1. 本月各渠道报备数量。
2. 报备状态分布。
3. 报备转商机情况。
4. 报备未转商机清单。

实现动作：

1. 新增 `registrations` 执行器。
2. 支持状态分布、渠道排行、负责人排行。
3. 通过 `regId` 关联商机计算转化。

## 7.3 渠道分析

支持问题：

1. 各渠道商机贡献排名。
2. 各渠道订单贡献排名。
3. 一级 / 二级渠道表现对比。
4. 某渠道最近三个月经营情况。

实现动作：

1. 新增 `partners` 聚合逻辑。
2. 结合 `opportunities / registrations / orders` 计算贡献。
3. 支持渠道层级和父子渠道汇总。

## 7.4 报价分析

支持问题：

1. 本季度报价状态分布。
2. 报价金额趋势。
3. 报价关联商机情况。
4. 报价未转订单清单。

实现动作：

1. 新增 `quotes` 执行器。
2. 支持报价状态、金额、渠道、负责人维度。
3. 关联 `oppId / oppIds` 和订单字段。

## 7.5 订单分析

支持问题：

1. 本季度订单金额趋势。
2. 各渠道订单贡献排名。
3. 各区域订单金额对比。
4. 员工可见范围内订单明细。

实现动作：

1. 新增 `orders` 执行器。
2. 支持订单金额、订单数量、状态分布、时间趋势。
3. 关联报价、商机、渠道。

## 7.6 转化漏斗分析

支持问题：

1. 报备到订单的转化漏斗。
2. 某渠道报备 -> 商机 -> 报价 -> 订单转化情况。
3. 最近三个月各阶段流失情况。

实现动作：

1. 新增跨对象漏斗执行器。
2. 支持 `registrations -> opportunities -> quotes -> orders` 关联。
3. 输出每一层数量、金额和转化率。

验收标准：

1. 首批 10 个问题均能返回真实数据。
2. 结果至少包含指标卡、图表、明细表、AI 摘要。
3. 数据为空时给出清晰提示，不展示假数据。

## 16. 风险与兜底

| 风险 | 影响 | 兜底 |
|---|---|---|
| 正式登录未开放 | 无法真实账号进入 | 继续使用 mock 链路灰度 |
| 标准 API 只支持单 client | 多角色分析受限 | 先做多 client 映射 |
| 列表接口分页太小 | 聚合不准或慢 | 增加聚合接口或镜像库 |
| 字段缺金额或关联 ID | 报价、订单、漏斗无法算 | 先降级为数量统计 |
| 字典不完整 | 页面显示枚举值 | 本地临时字典兜底 |
| 企微映射缺失 | 企微入口无法上线 | 先只上线 Web |
| AI 输出不稳定 | 意图解析失败 | 使用 `json_object` + 本地校验 + fallback |

---

## 17. 结论

我方要做的核心不是单独写一个“联软 CRM 查询页面”，而是把联软渠道 CRM 接入现有 AI-agent 主链路。

推荐按以下顺序开工：

1. 先打通正式登录和身份权限。
2. 再解决标准 API 多用户鉴权。
3. 然后扩展渠道 CRM 语义层和多对象分析执行器。
4. 同步补页面诊断、测试和部署配置。
5. 最后接企业微信和生产性能优化。

这样可以保证每一步都有可验收结果，也能避免一次性大改影响现有功能。
