# 联软 CRM 对方待补充项清单

本文档用于整理联软 CRM 在第一阶段接口联调前后，仍需继续补充给 `AI-agent` 团队的资料、规则和契约项，便于双方快速收口缺口，避免边联调边反复口头确认。

## 2. 使用说明

建议对方按优先级分批补充：

1. `P0`
   第一阶段联调前必须补齐
2. `P1`
   第一阶段开发中必须补齐
3. `P2`
   第二阶段分析增强或第三阶段企微 / 写回前补齐

## 3. P0：联调前必须补齐

## 3.1 环境与认证信息

请对方提供：

1. 联调环境 Base URL
2. 正式环境 Base URL 规划
3. `appKey`
4. `appSecret`
5. 白名单 IP 配置方式
6. Token 过期时长说明
7. Token 刷新方式说明
8. 接口超时时间说明
9. 限流规则说明

交付形式建议：

1. 一份环境参数表
2. 一组联调 client 凭证
3. 一份错误返回样例

## 3.2 联调测试账号

请对方至少提供以下测试账号：

1. 超级管理员账号
2. 区域管理员账号
3. 渠道管理员账号
4. 员工账号

每个账号建议同时提供：

1. 用户 ID
2. 用户名
3. 角色
4. 区域
5. 渠道
6. 该账号在 CRM 页面上的实际可见范围说明

## 3.3 六类对象的联调样例数据

请对方为以下对象准备可查样例：

1. `users`
2. `partners`
3. `registrations`
4. `opportunities`
5. `quotes`
6. `orders`

每类建议至少准备：

1. 3 条正常数据
2. 1 条边界数据
3. 1 条跨权限不可见数据

这样便于验证：

1. 列表检索
2. 详情查询
3. 权限裁剪
4. 越权阻断

## 4. P1：第一阶段开发中必须补齐

## 4.1 完整字典与枚举值

当前契约里已有部分字典，但仍不够支撑稳定问数和展示转换。

请对方补齐：

1. `opportunities.stage` 全量阶段字典
2. `quotes.status` 全量状态字典
3. `orders.status` 全量状态字典
4. 用户状态全量字典
5. 渠道状态全量字典
6. 渠道等级全量字典
7. 是否存在已删除、作废、归档状态

建议交付形式：

1. 统一字典接口补充
2. 或单独枚举字典文档

## 4.2 时间字段标准说明

请对方明确：

1. 所有时间字段是否都为 ISO 8601
2. 时区是否固定为北京时间
3. 是否存在只到日期、不含时间的字段
4. `createdAt`、`updatedAt`、`approvedAt`、`expectedClose` 的业务含义
5. 是否支持按更新时间做稳定增量拉取

## 4.3 删除与失效规则

请对方明确：

1. 数据删除是物理删除还是逻辑删除
2. 如果逻辑删除，使用哪个字段表示
3. 失效、禁用、归档是否共用同一字段
4. 列表接口默认是否过滤已删除 / 已归档数据

## 4.4 权限裁剪口径说明

请对方进一步明确：

1. 区域管理员是否只能看本区域
2. 渠道管理员是否可看下属二级渠道
3. 员工是否只能看本人，还是可看本渠道部分数据
4. `partnerId`、`assignedPartnerId`、`assignedStaffId`、`createdBy` 的优先级
5. `region` 和 `partnerId` 冲突时以谁为准

建议交付形式：

1. 一份权限样例矩阵
2. 不少于 8 个典型账号的权限可见范围样例

## 5. P2：第二阶段分析增强前补齐

## 5.1 复杂分析支持项

如果第二阶段要做更复杂的智能分析，请对方补齐：

1. 是否支持按更新时间增量同步
2. 是否允许建设 MySQL 分析镜像库
3. 字段变更通知机制
4. 数据量级说明
5. 高峰期 QPS 与接口调用限制

## 5.2 分析主题补充资料

请对方补：

1. 常用业务分析口径 10 到 20 条
2. 商机漏斗口径
3. 报价转订单口径
4. 区域 / 渠道 / 员工三级统计口径
5. 时间范围常用定义
6. 金额单位说明

## 6. P2：企业微信接入前补齐

## 6.1 用户映射规则

请对方明确：

1. 企业微信用户唯一标识是什么
2. CRM 用户唯一标识是什么
3. 二者如何映射
4. 是靠手机号、邮箱、工号还是独立映射表
5. 一人多企微账号是否允许
6. 映射失败时如何处理

## 6.2 组织映射规则

请对方明确：

1. 企业微信部门如何映射到 CRM 区域 / 渠道 / 组织
2. 一个 CRM 渠道是否对应一个企微部门
3. 是否存在跨区域或跨渠道的企微组织关系
4. 是否存在虚拟部门、临时部门

## 6.3 扫码登录落位规则

请对方明确：

1. 企业微信扫码后如何唯一定位 CRM 用户
2. 首次扫码未绑定时如何处理
3. 已停用或待审批用户扫码时如何处理
4. 是否允许自动创建用户

## 7. P2：写回与审批前补齐

## 7.1 跟进记录对象

请对方补充：

1. 跟进记录对象结构
2. 关联客户 / 商机 / 订单的字段
3. 跟进内容字段
4. 跟进时间字段
5. 跟进人字段
6. 是否支持附件

## 7.2 写回契约

请对方补充：

1. 哪些对象允许写回
2. 每类写回的必填字段
3. 状态限制
4. 幂等规则
5. 审计字段
6. 错误码约定

## 7.3 审批对象与审批记录

如果未来要承接审批 / 合同审查，请对方补充：

1. 审批对象主表
2. 审批记录表
3. 审批状态流转规则
4. 当前节点字段
5. 审批意见字段
6. 审批时间字段

## 8. 建议交付格式

建议对方按以下格式补齐资料：

1. 环境参数表
2. 字典与枚举清单
3. 权限矩阵
4. 企微映射规则说明
5. 写回规则说明
6. 审批对象结构说明

如果走接口形式补齐，建议优先提供：

1. 扩展字典接口
2. 企微映射查询接口
3. 写回接口草案
4. 审批接口草案

## 9. 建议的补齐顺序

建议对方按以下顺序补齐：

1. 先补联调环境、凭证、白名单和测试账号
2. 再补完整字典、时间字段、删除规则和权限矩阵
3. 再补企微映射规则
4. 最后补写回和审批契约

## 10. 当前最小可开工边界

如果对方暂时只能先补最小项，那么至少需要先补齐以下内容，第一阶段就可以开工：

1. 联调环境 Base URL
2. `appKey`
3. `appSecret`
4. 白名单 IP
5. 4 类测试账号
6. 六类对象样例数据
7. `opportunities.stage` 字典
8. `quotes.status` 字典
9. `orders.status` 字典
10. 权限矩阵样例

## 11. 当前不补会直接影响什么

如果以下内容不补，会直接影响对应能力：

1. 不补联调环境与凭证
   无法开始接口联调
2. 不补完整字典
   无法稳定做中文展示和问数语义映射
3. 不补权限矩阵
   无法验证越权阻断是否正确
4. 不补企微映射规则
   无法做企微登录和企微问数落位
5. 不补写回契约
   无法做跟进写回和新增对象
6. 不补审批模型
   无法做合同审核和审批相关能力
