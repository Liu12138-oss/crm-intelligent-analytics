# CRM 官方 API 优先接入治理说明

## 1. 文档目的

本文档用于落实 `enforce-crm-api-first-integration` 变更，统一说明三件事：

- 当前存量非认证能力分别通过什么路径接入 CRM。
- 哪些能力应该继续优先走 CRM 官方 API，哪些能力可以保留受控数据库或受控内部接口路径。
- 后续新增或改造 CRM 能力时，方案评审必须补齐哪些内容。

## 2. 适用范围

- 适用于所有新增 CRM 相关功能。
- 适用于既有非认证能力的需求扩展、重构和缺陷修复。
- 不适用于当前稳定的用户认证、登录、会话建立和统一鉴权链路；`auth` 模块若要调整，必须单独立项。

## 3. 存量非认证能力盘点

| 模块 / 能力 | 当前接入方式 | 是否优先依赖官方 API | 当前结论 | 后续收敛建议 | 优先级 |
| --- | --- | --- | --- | --- | --- |
| `analysis` 智能问数编译与执行 | 受控 CRM 数据库查询与受控执行器 | 否 | 当前 CRM 官方 API 不覆盖自然语言问数所需的多维聚合、权限注入、受控 SQL 预检和统一报表生成 | 继续保留受控数据库或内部接口路径，但只允许走白名单编译、AST 安全校验、预检、确认与审计链路，禁止新增自由查库或自由写库入口 | P1 |
| `wecom-directory-sync` 企业微信目录同步 | 企业微信官方 API + CRM 原生 `wx_*` 表受控写回 | 是 | 已有官方 API，可作为标准做法 | 继续优先走官方 API；若后续字段补齐不足，只允许在同步链路内补受控写回，不新增旁路脚本 | P1 |
| `wecom-auth` 企业微信发送者识别 | 读取 CRM `wx_user_maps`、`wx_users` 等原生映射表 | 否 | 当前身份识别依赖 CRM 已落库映射，官方 API 不能直接替代 CRM 内部映射关系 | 保持受控数据库识别，但要确保前置同步链路稳定；异常时优先给出明确降级提示 | P1 |
| `wecom-followup-writeback` 企业微信跟进写回 | CRM 官方详情接口 + CRM 官方协作人接口 + 受控权限判定 | 是 | 当前对象级权限需要同时读取负责人和协作人集合，优先依赖官方接口确认对象事实 | 后续继续优先走官方详情与协作人接口；仅在官方接口缺失对象、字段或权限语义时，才允许受控兜底 `entities_assist_users` 等关系表，并补齐缺口说明 | P1 |
| `query-assets` 常用查询与最近查询 | 应用库自有数据 | 不涉及 | 当前不直接访问 CRM | 继续保持与 CRM 解耦，禁止为了模板展示再引入新的 CRM 直连查询 | P2 |
| `governance` 治理策略与模板管理 | 应用库自有数据 | 不涉及 | 当前不直接访问 CRM | 继续保持治理配置在应用库存储；若后续需要引用 CRM 对象，先评估是否可通过官方 API 拉取元数据 | P2 |
| `audit` 审计检索 | 应用库自有数据 | 不涉及 | 当前不直接访问 CRM | 继续以应用库审计记录为准，不要把审计检索扩展成对 CRM 业务库的旁路查询 | P2 |
| `export` 结果导出 | 复用分析结果与应用库导出记录 | 间接依赖 `analysis` | 当前不新增独立 CRM 查询 | 后续若导出需求需要补查明细，必须优先复用 `analysis` 已批准的受控执行结果，不得单独新增导出查库或导出写回逻辑 | P2 |

## 4. 首批优先整改名单

以下能力在后续进入改造时，必须优先执行“官方 API 可用性检查 + 兜底说明”：

1. `analysis`
   现状最依赖 CRM 复杂表结构，也是最容易因为字段误用导致口径漂移的链路。
2. `wecom-directory-sync`
   已经具备官方 API 路径，后续要防止回退到脚本直查或旁路同步。
3. `wecom-auth`
   身份识别依赖同步结果，后续应重点治理“同步是否完成”和“映射是否可用”，而不是在认证时临时拼更多数据库兜底。
4. `export`
   当前风险不高，但一旦新增“导出更多明细字段”，最容易绕开 `analysis` 直接查库，需要提前卡住。

明确不纳入本轮整改的模块：

- `auth`
- `crm-auth.service`
- `wecom-web-login.service`
- `session-auth.guard`
- 本地登录会话、扫码登录回调和 Cookie 装载链路

## 6. 方案评审检查清单

- 是否已经先查询 `https://apidoc.weiwenjia.com/`，而不是先假设只能查库。
- 若官方 API 可用，方案是否明确以官方 API 为主路径。
- 若官方 API 不可用，是否已经写明缺口原因，而不是只写“开发更快”。
- 是否明确列出涉及表、字段白名单和权限继承来源。
- 是否说明审计留痕、失败降级和回退方案。
- 是否确认不触碰 `auth`、登录、会话和统一鉴权装载链路。
- 是否说明该能力在后续应继续收敛到官方 API，还是长期保留受控数据库路径。

## 7. 使用方式

- 开发前：先按第 5 节补齐可用性检查与兜底说明。
- 评审时：逐项核对第 6 节检查清单。
- 实施后：若能力接入方式发生变化，回写第 3 节盘点表。
