# 联软 CRM 第二阶段分析库完整落地记录

## 1. 本阶段目标

第二阶段目标是把“AI-agent 自建 MySQL 分析库 + 语义层 + 受控 Text-to-SQL”从超管验证扩展到真实角色视角，并优先解决企业微信问数中高频、口径明确的问题。

本阶段不改变登录、企业微信机器人入口、OpenAPI 同步入口和原有 OpenAPI 查询兜底链路。分析库路径不可用、权限范围无法安全注入、AI 生成 SQL 过复杂或 SQL Guard 校验失败时，仍回退到既有问数链路。

## 3. 当前运行逻辑

1. 用户从 Web 智能分析或企业微信机器人发起自然语言问题。
2. 系统完成登录态、当前用户和 `ScopeSnapshot` 解析。
3. 智能分析编排器优先调用分析库执行器。
4. 分析库执行器读取联软分析库中的用户权限提示。
5. 如果是全量权限，允许 AI 生成 SQL 进入 SQL Guard 校验。
6. 如果不是全量权限，系统先把用户范围归一为区域、服务商或负责人范围。
7. 命中固定高频业务口径时，执行固定受控 SQL，并注入当前用户范围。
8. 未命中固定口径时，AI 生成 SQL 后，程序尝试把行级权限条件注入 SQL。
9. SQL Guard 再次校验表、字段、函数、只读语句和 LIMIT。
10. 只读查询执行并审计，结果转换为原有分析结果切片。
11. 任一步失败都回退到既有 OpenAPI / 旧问数链路，避免影响已有功能。

## 5. 权限边界

当前分析库行级权限支持以下映射：

| 权限模式 | 主要来源 | 注入字段 |
| --- | --- | --- |
| 全量权限 | `user.isAdmin`、`ScopeSnapshot.isFullAccess`、联软 `scope_type=all`、`superadmin` 角色 | 不加行级条件。 |
| 区域权限 | 联软权限桥表 `regions`、用户维度 `region/big_region`、区域管理员角色 | `region`、`big_region`。 |
| 服务商权限 | 联软权限桥表 `partnerIds`、用户维度 `partner_id` | `partner_id`、`parent_partner_id`、`assigned_partner_id`。 |
| 负责人权限 | 联软权限桥表 `userIds`、当前 `ScopeSnapshot.ownerIds`、用户维度 `user_id` | `owner_id`、`assigned_staff_id`、`created_by`。 |

如果某张事实表没有对应权限字段，例如区域角色查询没有区域字段的报价或订单事实表，系统不会硬查，而是回退旧链路或等待后续语义层补字段。

## 6. 当前限制

1. AI 生成 SQL 仍拒绝子查询、`UNION` 等复杂结构的自动权限注入，避免字符串改写误伤权限边界。
2. 订单、报价等事实表目前缺少区域字段时，区域角色的部分分析会受限。
3. 当前 `bridge_lianruan_user_permission` 本地同步只有当前 client 权限记录，更多角色主要依赖 `dim_lianruan_user` 的角色和区域字段推断。
4. 固定高频口径目前优先覆盖“客户报备未关联商机”。后续可继续把经营看板、超两周未更新商机、客户报备转化等 P0 问法固化为受控模板。
5. AI SQL 的语义准确性仍依赖字段目录、指标目录和提示词质量；程序只负责安全边界、权限注入和执行兜底。
