# 发给联软 CRM：只读库 Text-to-SQL 接入资料清单与我方实施方案

## 1. 背景与目标

当前我方 CRM 智能分析系统已经接入联软渠道 CRM 标准 OpenAPI，并支持企业微信机器人、Web 智能分析工作台和部分联软对象分析能力。

为了进一步提升自然语言问数覆盖面，下一阶段拟增加“联软只读库 + AI 受控 SQL”模式：

```text
企微用户提问
  -> AI 理解自然语言语义
  -> AI 基于联软只读库语义层生成查询计划或 SQL 草稿
  -> 我方程序做只读校验、字段白名单、权限注入、参数化和审计
  -> 使用联软只读库账号查询数据
  -> AI 基于真实查询结果生成分析报告
  -> 返回企业微信机器人和 Web 工作台
```

本文包含两部分：

- 联软渠道 CRM 需要提供的只读库、表结构、字段语义、权限口径和样例材料。
- 我方 CRM 智能分析系统的实施方案、阶段计划、安全边界和验收方式。

## 2. 总体原则

- AI 负责理解用户问题、识别对象、指标、维度、筛选条件和展示形态。
- AI 可以生成查询计划或候选 SQL，但不能直接绕过程序执行。
- 程序负责最终 SQL 安全校验、字段白名单、角色权限条件注入、查询超时、行数限制和审计留痕。
- 联软只读库必须使用只读账号，不允许写入、删改、建表、跨库访问或执行危险函数。
- 当前标准 OpenAPI 继续保留，作为稳定只读接口路径和 AI 离线兜底路径。
- 若 AI 不可用、超时或低置信度，我方只执行高置信固定意图；无法确认时应追问，不再默认返回同一类数据。

## 5. 我方实施方案

### 5.1 目标架构

```text
企业微信机器人 / Web 智能分析
  -> 会话与身份识别
  -> CRM 用户与角色权限解析
  -> AI 语义理解层
  -> 只读库语义层
  -> 查询计划生成
  -> SQL 安全编译与校验
  -> 权限条件强制注入
  -> 只读库查询执行
  -> 结果标准化
  -> AI 经营分析报告生成
  -> 企微卡片 / 表格 / 图片 / 文件 / Web 报告展示
```

### 5.2 核心模块改造

| 模块 | 改造内容 |
| --- | --- |
| 语义资产层 | 导入联软表结构、字段中文名、字段别名、指标口径、枚举字典 |
| AI 理解层 | 输出结构化查询计划，包含对象、指标、维度、筛选、时间、结果形态 |
| SQL 生成层 | 基于语义层生成 SQL 草稿或 AST，不允许自由猜表字段 |
| SQL 安全层 | 只允许 `SELECT`，禁止写入、DDL、危险函数、跨库、无限制查询 |
| 权限注入层 | 根据当前企微用户对应 CRM 角色强制追加权限条件 |
| 执行层 | 使用只读库连接池、超时、分页、行数限制和失败降级 |
| 结果标准化 | 把查询结果转为指标卡、表格、图表、明细和报告素材 |
| 报告生成层 | AI 基于真实数据生成摘要、关键发现、经营建议 |
| 企微展示层 | 默认发送分析报告、关键发现、重点汇总和经营建议 |
| 审计治理层 | 记录用户、问题、SQL、参数、权限快照、行数、耗时、结果摘要 |

### 5.3 AI 输出查询计划格式

AI 第一阶段不直接输出最终可执行 SQL，而是优先输出结构化查询计划。

```json
{
  "intent": "READONLY_ANALYSIS",
  "subject": "service_provider",
  "metrics": ["opportunity_count", "opportunity_amount"],
  "dimensions": ["service_provider"],
  "filters": {
    "region": "山东",
    "timeRange": "最近三个月",
    "hasOpportunity": true
  },
  "resultShape": "summary_table",
  "needsDetailRows": true,
  "confidence": "HIGH",
  "missingSlots": []
}
```

程序根据该计划选择：

- 标准 OpenAPI 统计接口。
- 标准 OpenAPI 列表接口 + 本地聚合。
- 只读库受控 SQL。

### 5.4 SQL 安全策略

SQL 执行前必须通过以下校验：

- 只能执行单条 `SELECT`。
- 禁止 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`。
- 禁止访问未登记表。
- 禁止访问未纳入白名单字段。
- 禁止 `information_schema`、系统表、跨库查询。
- 禁止危险函数和文件函数。
- 必须参数化条件，禁止字符串拼接用户输入。
- 必须注入当前用户权限条件。
- 必须设置 `LIMIT`。
- 必须设置查询超时。
- 大结果集只能返回摘要，明细需要二次确认或导出权限。

### 5.5 权限控制方案

权限不能由 AI 自己生成，必须由程序统一注入。

```text
当前企微 userid
  -> 映射 CRM 用户 ID
  -> 读取 CRM 角色、部门、区域、服务商权限
  -> 生成权限条件
  -> 注入 SQL WHERE 条件
  -> 审计记录权限快照
```

示例：

```sql
WHERE
  opportunity.region IN (:allowedRegions)
  AND opportunity.owner_user_id IN (:allowedOwnerUserIds)
```

超管用户也要记录权限快照，不能省略审计。

### 5.6 AI 离线或低置信兜底

| 场景 | 处理方式 |
| --- | --- |
| AI 正常且高置信 | 走 AI 查询计划 + 受控 SQL |
| AI 超时 | 走固定高置信意图 / 标准 OpenAPI 兜底 |
| AI 低置信 | 向用户追问，不默认查同一份数据 |
| 字段不存在 | 返回“当前只读库未提供该字段”，并记录缺口 |
| 权限无法确定 | 阻断查询，提示检查企微用户与 CRM 用户映射 |
| 查询超时 | 提示缩小时间范围或维度 |
| 结果为空 | 明确说明查询条件、权限范围和空结果原因 |

### 5.7 分阶段计划

#### P0：资料接收与语义层建模

目标：让系统知道联软只读库里有什么、字段是什么意思、哪些可以查。

我方工作：

- 建立联软只读库数据源配置。
- 建立表结构导入格式。
- 建立字段白名单和敏感字段清单。
- 建立字段中文映射、别名和枚举字典。
- 建立基础权限映射模型。
- 建立样例问题回归集。

验收：

- 能在治理后台看到联软只读库表和字段语义。
- 能识别字段是否允许分析、是否敏感。
- 能跑通只读连接测试。

#### P1：AI 查询计划生成

目标：让 AI 先理解问题，不再把大量问题落到同一个默认查询。

我方工作：

- 扩展 AI 结构化输出协议。
- 提供联软语义目录给 AI。
- 让 AI 输出对象、指标、维度、筛选、时间、结果形态。
- 低置信时追问，不硬查。
- 建立语义回归测试。

验收：

- 常见问题能映射到不同对象和结果形态。
- “服务商商机金额”“客户未报备”“订单金额”“区域对比”等不再返回同一类数据。

#### P2：受控 SQL 编译与执行

目标：让查询计划安全落到只读库。

我方工作：

- 实现查询计划到 SQL 的编译器。
- 实现 SQL AST 校验。
- 实现权限条件注入。
- 实现只读库执行器。
- 实现查询超时、分页、行数限制。
- 实现查询审计。

验收：

- 真实只读库查询可用。
- 写入、删改、跨库、危险 SQL 均被阻断。
- 不同角色查询结果符合联软权限口径。

#### P3：企微报告化展示

目标：让企微端拿到更直观的报告，而不是只返回一段文字。

我方工作：

- 统一指标卡、表格、图表、明细和报告结构。
- 默认向企微发送 AI 分析报告、关键发现、重点汇总、经营建议。
- 支持表格图片、摘要图片和文件附件。
- 支持“看明细”“看图表”“继续分析”等追问。

验收：

- 企微端可直接查看摘要、表格、图表和建议。
- Web 端可查看完整报告和审计链路。

#### P4：治理与持续优化

目标：把“问不懂、查错、字段缺失”变成可运营闭环。

我方工作：

- 记录未理解问题。
- 记录字段缺口。
- 记录错误 SQL 和阻断原因。
- 提供语义资产维护入口。
- 建立回归测试和版本发布机制。

验收：

- 每次语义规则或字段变更都有回归记录。
- 高频失败问题可沉淀为样例和语义资产。

## 6. 典型问题执行示例

### 6.1 服务商商机金额

用户问：

```text
最近三个月山东区域，有商机的服务商，对应的商机数量和商机金额、以及总金额
```

AI 理解：

- 对象：服务商、商机。
- 指标：商机数量、商机金额、总金额。
- 维度：服务商。
- 条件：山东区域、最近三个月、有商机。
- 展示：汇总表 + 总计 + 经营建议。

程序执行：

- 注入当前用户 CRM 权限。
- 查询服务商和商机表。
- 按服务商聚合数量和金额。
- 返回企微表格和分析报告。

### 6.2 客户未报备商机

用户问：

```text
有多少客户是没有报备商机的，分别创建了多长时间
```

AI 理解：

- 对象：客户、报备、商机。
- 指标：未报备客户数、未建商机客户数。
- 维度：客户创建时长分桶。
- 条件：当前用户权限范围内全部可见客户。
- 展示：指标卡 + 创建时长分布表。

程序执行：

- 查询客户表。
- 左关联报备和商机。
- 按创建时间计算创建时长。
- 按权限范围裁剪。
- 返回生命周期分析。

### 6.3 服务商多维画像

用户问：

```text
最近一年加入的服务商有多少家，合作级别、等级、是否技术服务商维度一起给我
```

AI 理解：

- 对象：服务商。
- 指标：服务商数量。
- 维度：合作级别、等级、是否技术服务商。
- 条件：最近一年加入。
- 展示：多维分布 + 汇总建议。

程序执行：

- 使用服务商加入时间字段。
- 按三个维度聚合。
- 枚举值转中文。
- 返回表格和分布说明。

## 7. 风险与控制

| 风险 | 控制措施 |
| --- | --- |
| AI 生成错误字段 | 只能基于语义层白名单生成，执行前校验 |
| AI 漏加权限 | 权限由程序强制注入，AI 不负责权限 SQL |
| SQL 越权访问 | 表白名单、字段白名单、权限条件和审计 |
| 查询过慢 | 超时、LIMIT、分页、必要时要求缩小范围 |
| 结果编造 | AI 只能基于程序返回的数据生成报告 |
| 字段口径不一致 | 依赖联软字段口径文档和回归样例 |
| 只读库延迟 | 展示数据更新时间和同步延迟说明 |
| 敏感字段泄露 | 敏感字段默认不进入 AI 上下文 |

## 9. 首轮联调验收清单

- 查询 SQL 全部通过只读校验。

- Web 端可查看完整查询结果和审计链路。

## 10. 需要联软优先确认的问题

1. 当前 SQLite 只读库是直接访问文件、定时同步副本，还是由联软提供只读服务端口？
2. 只读库是否包含完整历史数据，还是只包含当前业务快照？
3. 是否可以提供全量 DDL、字段中文含义和表关系图？
4. 用户角色权限是否可以通过表结构计算，还是需要额外接口提供？
5. 企业微信 userid 与 CRM 用户 ID 是否已有稳定映射？
6. 超管、区域经理、普通销售的可见范围分别如何计算？
7. 服务商等级、合作级别、是否技术服务商分别对应哪些字段？
8. 商机金额、订单金额、有效收入分别采用哪些字段？
9. 客户创建时间、服务商加入时间分别采用哪些字段？
10. 哪些字段禁止进入 AI 上下文或企微展示？

## 11. 我方推荐落地顺序

建议不要一次性全量放开所有自然语言 SQL，而是按以下顺序推进：

1. 先接入只读库连接和表结构导入。
2. 先开放 6 类核心对象：客户、服务商、报备、商机、报价、订单。
3. 先覆盖 20 到 30 条真实高频问题。
4. 先验证超管、区域经理、普通销售三类角色。
5. 再扩展复杂组合查询、明细追问和多图表报告。
6. 最后开放语义资产治理页面，让业务人员持续维护别名、口径和样例。
