# 联软 CRM：数据仓库语义层 AI 智能分析总体方案与双方分工

## 1. 方案目标

本文用于明确联软渠道 CRM 与我方 CRM-Agent 智能分析系统下一阶段的完整接入方案。

目标不是让 AI 直接访问联软生产库，也不是简单把联软只读库暴露给 AI，而是建设一套更稳定、更安全、更适合长期运营的分析链路：

```text
联软渠道 CRM
  -> 数据同步
  -> 我方分析数据仓库
  -> 标准数据模型
  -> 业务语义层
  -> AI 生成查询计划 / 候选 SQL
  -> 程序安全校验、权限注入、参数化执行
  -> 查询真实数据
  -> AI 基于结果生成分析报告
  -> 企业微信机器人 / Web 智能分析工作台展示
```

该方案要解决的问题：

- 用户在企业微信里用自然语言提问，系统能理解复杂组合问题。
- 不同问题走不同数据模型和查询链路，不再大量返回同一类默认数据。
- 支持客户、合作伙伴、报备、商机、报价、订单、下单、风险、趋势、分布、明细等综合分析。
- 支持当前企微用户对应联软 CRM 角色权限下的数据裁剪。
- 支持 AI 离线、低置信、字段缺失、权限不明时的安全兜底。
- 支持后续持续扩展字段、指标、语义资产和回归测试。

## 2. 适用场景

本方案适合以下问题类型：

- “帮我统计今年山东区域合作伙伴客户、商机的报备情况和下单情况。”
- “有多少客户是没有报备商机的，分别创建了多长时间。”
- “有哪些商机超过两个星期没有更新进展。”
- “最近三个月山东区域有商机的服务商，对应商机数量和商机金额、以及总金额。”
- “最近一年加入的服务商有多少家，按合作级别、等级、是否技术服务商维度一起分析。”
- “广州办渠道下单汇总分析，输出经营报告、关键发现、趋势和建议。”
- “按当前账号权限范围，列出高风险商机和重点跟进对象。”

这类问题通常不是单一 SQL 能完整表达，而是需要 AI 先拆解为多个分析子任务，再由程序分别执行受控查询，最后汇总成一份经营分析报告。

## 3. 推荐总体架构

### 3.1 总链路

```text
数据侧：
联软 CRM 生产数据
  -> 增量同步 / 定时同步 / 文件快照
  -> ODS 原始层
  -> DWD 标准明细层
  -> DWS 汇总服务层
  -> ADS 分析应用层

问数侧：
企微 / Web 问题
  -> 身份识别
  -> 权限装载
  -> AI 语义理解
  -> 查询计划生成
  -> SQL 编译与校验
  -> 权限条件注入
  -> 数仓查询
  -> 结果标准化
  -> AI 报告生成
  -> 企微 / Web 展示
```

### 3.2 分层说明

| 层级 | 名称 | 作用 | 是否面向 AI |
| --- | --- | --- | --- |
| ODS | 原始数据层 | 保存联软同步过来的原始表、原始字段和同步批次 | 否 |
| DWD | 标准明细层 | 清洗成客户、伙伴、报备、商机、报价、订单等标准明细模型 | 部分 |
| DWS | 汇总服务层 | 预聚合常用指标，如伙伴贡献、客户生命周期、商机风险 | 是 |
| ADS | 应用分析层 | 面向企微和 Web 的报告、看板和专题分析结果 | 是 |
| 语义层 | 业务语义目录 | 定义对象、字段、指标、维度、筛选、字典、同义词和口径 | 是 |
| 安全层 | SQL 与权限控制 | 白名单、只读校验、权限注入、超时、审计 | 否 |

### 3.3 当前第一版已落地能力

截至 2026-06-09，AI-agent 后端已先落地“智能分析数据仓库同步基础能力”，用于把联软标准 OpenAPI 数据安全地进入我方运行态仓库快照，为后续正式 MySQL 分析库、语义层和受控 Text-to-SQL 打底。

已落地内容：

| 能力 | 当前状态 | 说明 |
| --- | --- | --- |
| 同步来源枚举 | 已支持 | `OPENAPI`、`SQLITE_SNAPSHOT`、`FILE_EXPORT` 三类来源已统一建模 |
| OpenAPI 手动同步 | 已支持 | 可按资源同步字典、角色权限、用户、伙伴、客户、报备、商机、报价、订单、当前权限范围 |
| SQLite 快照兼容入口 | 已预留 | 当前不直接读取生产 SQLite；未启用时返回跳过状态，避免误伤联软生产库 |
| 文件导入兼容入口 | 已预留 | 当前先建模来源状态，后续接 CSV / JSON / Parquet / SQLite 快照导入 |
| 同步运行记录 | 已支持 | 记录来源、资源、状态、耗时、失败原因和每类资源结果 |
| 原始快照缓存 | 已支持 | 第一版写入 AppStorage 运行态，后续可替换为正式 ODS 表 |
| 同步检查点 | 已支持 | 记录资源最后同步批次、时间和 `updatedAt` 游标，用于后续增量同步 |
| 敏感字段脱敏 | 已支持 | `token`、`secret`、`password`、手机号、邮箱等字段写入前脱敏 |
| 治理接口 | 已支持 | 提供同步概览和手动同步接口，复用治理权限校验 |

当前后端接口：

| 接口 | 方法 | 用途 |
| --- | --- | --- |
| `/governance/analysis-warehouse/overview` | `GET` | 查看同步来源状态、最近同步记录、资源快照数量和检查点 |
| `/governance/analysis-warehouse/sync-runs` | `POST` | 手动触发一次同步 |

手动同步请求示例：

```json
{
  "sourceType": "OPENAPI",
  "resources": ["dictionaries", "rolePermissions", "users", "partners", "customers", "registrations", "opportunities", "quotes", "orders", "permissions"],
  "mode": "INCREMENTAL",
  "dryRun": false,
  "pageSize": 200,
  "maxPages": 1
}
```

联软 2026-06-09 二次确认后，当前同步实现已按以下 P0 规则对齐：

| 项目 | 当前处理 |
| --- | --- |
| 推荐同步顺序 | 字典、角色权限、用户、伙伴、客户、报备、商机、报价、订单、当前权限范围 |
| 分页参数 | 使用 `pageNo`、`pageSize`，单页上限按联软确认调整为 `200` |
| 增量参数 | 增量模式使用 `updatedAfter`，并按 `updatedAt asc` 排序 |
| 角色权限矩阵 | 已支持同步 `/meta/role-permissions`，内部资源名为 `rolePermissions` |
| 当前权限范围 | 已支持同步 `/meta/permission-scope`，内部资源名为 `permissions` |
| SQLite / 文件快照 | 继续作为补充来源预留，不直连生产 SQLite |

### 3.4 MySQL 分析库第一版落地说明

截至 2026-06-09，AI-agent 后端已新增 MySQL / MySQL 兼容分析库落地能力。该能力与联软 CRM 业务库隔离，使用独立配置项：

| 配置项 | 说明 |
| --- | --- |
| `ANALYSIS_WAREHOUSE_DB_HOST` | AI-agent 自建分析库地址 |
| `ANALYSIS_WAREHOUSE_DB_PORT` | 分析库端口，默认按 MySQL `3306` |
| `ANALYSIS_WAREHOUSE_DB_NAME` | 分析库库名 |
| `ANALYSIS_WAREHOUSE_DB_USER` | 分析库账号 |
| `ANALYSIS_WAREHOUSE_DB_PASSWORD` | 分析库密码 |
| `ANALYSIS_WAREHOUSE_DB_CONNECT_TIMEOUT_MS` | 分析库连接超时，默认 `8000` 毫秒 |

当前自动建表范围：

| 层级 | 表 |
| --- | --- |
| ODS | `ods_lianruan_sync_runs`、`ods_lianruan_raw_records`、`ods_lianruan_sync_checkpoints` |
| DWD 维度 | `dim_lianruan_user`、`dim_lianruan_partner`、`dim_lianruan_customer` |
| DWD 事实 | `fact_lianruan_registration`、`fact_lianruan_opportunity`、`fact_lianruan_quote`、`fact_lianruan_order` |
| 权限桥表 | `bridge_lianruan_user_permission` |
| 语义层 | `semantic_field_catalog`、`semantic_metric_catalog` |

当前同步行为：

- 如果未配置 `ANALYSIS_WAREHOUSE_DB_*`，同步仍写入 AppStorage 运行态快照，不影响现有联调。
- 如果配置完整，首次同步时自动执行 `CREATE TABLE IF NOT EXISTS`，不会删除已有数据。
- OpenAPI 原始数据先写入 `ods_lianruan_raw_records`。
- P0 对象会同步上卷到对应 `dim_*` / `fact_*` 标准表。
- 当前权限范围会同步到 `bridge_lianruan_user_permission`。
- 字段和指标种子会写入语义层目录，联软回传最终字段口径后继续补齐。

当前受控 SQL 能力：

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/governance/analysis-warehouse/controlled-query` | `POST` | 治理联调用，默认 dry-run 校验；显式 `dryRun=false` 时执行 |

受控 SQL 当前边界：

- 只允许单条 `SELECT`。
- 只允许访问 `dim_lianruan_*`、`fact_lianruan_*`、`semantic_*` 等语义层登记表。
- 禁止访问 ODS 原始 JSON 表，避免绕过敏感字段治理。
- 禁止 `SELECT *`。
- 禁止写入、DDL、系统库探测、外部文件调用。
- 默认补 `LIMIT 100`，最大 `LIMIT 1000`。
- 执行时写入 SQL 审计，数据库角色标记为 `ANALYSIS_WAREHOUSE`。

受控 SQL dry-run 请求示例：

```json
{
  "sql": "SELECT p.partner_name, COUNT(o.opportunity_id) AS opportunity_count FROM dim_lianruan_partner p LEFT JOIN fact_lianruan_opportunity o ON o.partner_id = p.partner_id WHERE p.region = ? GROUP BY p.partner_name ORDER BY opportunity_count DESC",
  "params": ["山东"],
  "dryRun": true
}
```

执行请求示例：

```json
{
  "sql": "SELECT p.partner_name, COUNT(o.opportunity_id) AS opportunity_count FROM dim_lianruan_partner p LEFT JOIN fact_lianruan_opportunity o ON o.partner_id = p.partner_id WHERE p.region = ? GROUP BY p.partner_name ORDER BY opportunity_count DESC LIMIT 20",
  "params": ["山东"],
  "dryRun": false,
  "timeoutMs": 5000
}
```

当前尚未开放的能力：

- 企业微信自然语言直接进入自由 Text-to-SQL。
- AI 绕过程序校验直接执行 SQL。
- 对联软生产 SQLite 或生产业务库执行 SQL。
- 明细导出绕过权限、脱敏和审计。

阶段边界：

- 当前第一版只负责同步底座和诊断闭环，不开放 AI 直接自由执行 SQL。
- 当前快照缓存用于联调验证和数仓落地前过渡，不作为长期分析库。
- 正式 Text-to-SQL 仍必须等待 MySQL / MySQL 兼容分析库、语义层字段目录、权限注入和 SQL 审计全部补齐后再开放。
- 联软侧如果仍是 SQLite，可以继续提供 SQLite 只读副本或快照文件；AI-agent 侧负责导入自己的分析库，不要求联软改造成 MySQL。

## 4. 方案可行性评估

### 4.1 可行性结论

该方案可行，并且比“直接查联软只读库”更适合作为长期架构。

原因：

- 查询压力从联软 CRM 转移到我方分析数仓，不影响对方生产系统。
- 可把联软历史字段、枚举、脏数据、口径差异清洗成统一数据模型。
- AI 面向语义层和标准模型理解问题，准确性高于直接面对原始库表。
- 可建立预聚合表和索引，复杂统计性能更可控。
- 可在数仓层保留历史快照，支持同比、环比、趋势、漏斗、生命周期分析。
- 权限、审计、脱敏、导出限制可以在我方统一治理。

### 4.2 不建议的做法

不建议采用以下方式：

- AI 直接连接联软生产库。
- AI 自由生成 SQL 后不经校验直接执行。
- 只同步业务数据，不同步用户、角色、部门、区域和权限数据。
- 只做字段同步，不做指标口径和字典枚举说明。
- AI 离线时默认返回固定负责人排行或同一类兜底数据。
- 为了短期可用，把所有字段完全开放给 AI。

## 5. 联软渠道 CRM 需要做什么

### 5.1 提供数据同步源

联软需要确认并提供至少一种同步方式。

| 同步方式 | 说明 | 适用性 |
| --- | --- | --- |
| 标准 OpenAPI 增量同步 | 通过接口按更新时间分页拉取 | 推荐，安全可控 |
| 只读库同步 | 提供只读 SQLite / MySQL / PostgreSQL 数据源 | 可行，需明确表结构 |
| 文件快照同步 | 定时导出 CSV / Parquet / SQLite 文件 | 适合初期联调 |
| 消息队列 / CDC | 数据变更实时推送 | 后续增强 |

首阶段建议优先选择：

- 如果联软标准 OpenAPI 已经稳定，优先用 OpenAPI 做增量同步。
- 如果 OpenAPI 覆盖不足，可补充只读库或文件快照。
- 如果当前仍是 SQLite，建议提供“只读副本文件 + 同步时间戳 + 表结构说明”，不要直接暴露生产运行中的 SQLite 文件。

需要特别区分：

| 概念 | 可选形态 | 说明 |
| --- | --- | --- |
| 联软 CRM 数据源 | 标准 OpenAPI、SQLite 只读副本、SQLite 快照、CSV / JSON / Parquet 导出、后续 CDC | 联软可继续保持现有 SQLite，不要求为第一阶段改造为 MySQL |
| AI-agent 分析库 | MySQL / MySQL 兼容库优先，后续可升级 ClickHouse、Doris、StarRocks 等 | 由 AI-agent 侧承建，用于数仓分层、语义层和受控 SQL 查询 |

因此，联软侧如果当前只能提供 SQLite，也完全可行。推荐做法是联软生成稳定只读副本或批次快照，AI-agent 负责把 SQLite 数据同步进入自己的分析库，再基于标准模型和语义层执行 Text-to-SQL。

联软需要提供以下对象的全量和增量数据。

| 对象 | 用途 | 必要性 |
| --- | --- | --- |
| 客户 | 客户数量、创建时长、未报备、生命周期 | P0 |
| 合作伙伴 / 服务商 / 渠道商 | 伙伴画像、伙伴贡献、等级分布 | P0 |
| 客户报备 | 报备数量、报备状态、客户到商机链路 | P0 |
| 商机 | 商机数量、金额、阶段、风险、更新时间 | P0 |
| 报价 | 报价数量、报价金额、商机转报价 | P1 |
| 订单 / 下单 / 合同 | 下单数量、订单金额、成交转化 | P0 |
| 用户 | 负责人、创建人、企微用户映射 | P0 |
| 部门 / 组织 | 部门权限、团队统计、组织树 | P0 |
| 角色 | 超管、区域经理、销售等权限身份 | P0 |
| 区域 / 大区 | 山东、广州办、华东等区域口径 | P0 |
| 字典枚举 | 状态、阶段、等级、合作级别中文翻译 | P0 |
| 跟进记录 | 最后跟进时间、超期未更新商机 | P1 |

### 5.3 提供字段清单和中文语义

联软需要为每张表提供字段级说明。

| 信息 | 说明 | 必要性 |
| --- | --- | --- |
| 表名 | 真实表名或接口资源名 | 必须 |
| 表中文名 | 面向业务用户的中文名称 | 必须 |
| 字段名 | 真实字段名 | 必须 |
| 字段中文名 | 字段业务含义 | 必须 |
| 字段类型 | 字符串、数字、日期、枚举、布尔等 | 必须 |
| 是否主键 | 是否唯一标识 | 必须 |
| 是否关联键 | 与其它对象如何关联 | 必须 |
| 是否允许同步 | 敏感字段可不同步 | 必须 |
| 是否允许分析 | 是否进入 AI 语义层 | 必须 |
| 是否允许企微展示 | 是否可展示给用户 | 必须 |
| 是否需要脱敏 | 手机、证件、地址、联系人等 | 必须 |
| 示例值 | 脱敏后的样例值 | 建议 |

### 5.4 提供表关系和业务链路

联软需要明确以下关系，不能让 AI 或我方根据名称猜测。

| 关系 | 需要说明 |
| --- | --- |
| 客户到报备 | 客户 ID、报备 ID、是否一对多 |
| 报备到商机 | 报备转商机字段、状态变化 |
| 商机到报价 | 商机 ID 与报价 ID 关系 |
| 报价到订单 | 报价转订单或合同关系 |
| 合作伙伴到客户 | 伙伴负责、推荐、归属或关联客户口径 |
| 合作伙伴到商机 | 伙伴参与商机的字段 |
| 合作伙伴到订单 | 伙伴下单统计字段 |
| 商机到跟进记录 | 最后跟进时间、更新时间来源 |
| 用户到部门 | 用户所属部门、上级组织 |
| 用户到角色 | 角色、权限范围、超管标识 |
| 对象到区域 | 客户、伙伴、商机、订单分别按哪个字段归属区域 |

### 5.5 提供关键指标口径

联软需要确认所有关键统计指标的计算方式。

| 指标 | 必须确认的问题 |
| --- | --- |
| 客户数 | 是否排除无效客户、删除客户、停用客户 |
| 合作伙伴数 | 服务商、渠道商、代理商是否同一对象 |
| 报备数 | 哪些报备状态计入有效报备 |
| 未报备客户 | 无任何报备，还是无有效报备 |
| 商机数 | 哪些阶段或状态计入有效商机 |
| 商机金额 | 使用预计金额、有效收入、预算金额还是其它金额 |
| 报价金额 | 使用报价总额、最终报价、有效报价 |
| 订单数量 | 草稿、取消、作废订单是否排除 |
| 订单金额 | 下单金额、合同金额、有效收入、回款金额的区别 |
| 下单情况 | 订单创建时间、审批通过时间、签约时间以哪个为准 |
| 客户创建时长 | 使用客户创建时间还是首次报备时间 |
| 商机未更新 | 使用商机更新时间、最后跟进时间还是阶段更新时间 |

## 9. 安全与治理要求

### 9.1 数据安全

- 不同步禁止分析的敏感字段。
- 敏感字段默认脱敏。
- 明细展示受角色和导出权限控制。
- AI 上下文只放必要字段和聚合结果。
- 不把数据库密码、token、密钥写入源码或文档。

### 9.2 SQL 安全

- AI 不能直接执行 SQL。
- SQL 必须经程序编译或校验。
- 权限条件必须由程序注入。
- 禁止跨库、系统表、危险函数。
- 默认限制返回行数。
- 所有查询必须审计。

### 9.3 权限治理

- 每次查询记录当前用户权限快照。
- 当前权限实时生效，不复用旧授权快照直接放行。
- 超管也需要记录权限口径。
- 权限缺失时阻断，不返回全量。

### 9.4 语义治理

- 高频失败问题进入语义资产池。
- 字段变更后必须跑回归测试。
- 指标口径变更必须有版本记录。
- AI 离线时只允许高置信固定意图，不允许默认返回同一数据。

## 11. 首批建议覆盖问题

建议首批固定 20 到 30 条真实业务问题作为回归集。第一批至少包括：

1. 今年山东区域合作伙伴客户、报备、商机和下单整体情况。
2. 最近三个月山东区域有商机合作伙伴的商机数量和金额。
3. 最近三个月山东区域有下单合作伙伴的订单数量和金额。
4. 有多少客户没有报备商机，分别创建了多长时间。
5. 有哪些商机超过两个星期没有更新进展。
6. 最近一年加入的服务商按合作级别、等级、是否技术服务商分析。
7. 本月各区域商机数量和金额对比。
8. 本月各负责人商机金额排行。
9. 报备到商机、报价、订单的转化漏斗。
10. 当前账号权限范围内高风险商机清单。

为了尽快落地，建议最小可行版本先做以下范围：

### 数据范围

- 客户。
- 合作伙伴 / 服务商。
- 报备。
- 商机。
- 订单。
- 用户。
- 部门。
- 角色。
- 区域。
- 字典。

### 分析能力

- 客户数量。
- 伙伴数量。
- 报备数量。
- 商机数量和金额。
- 订单数量和金额。
- 未报备客户。
- 客户创建时长分布。
- 超两周未更新商机。
- 按区域、伙伴、负责人、月份聚合。

### 展示能力

- 企微汇总表。

- 字段白名单。
- 权限注入。
- 查询超时。
- 审计日志。

## 13. 关键决策点

双方需要优先确认：

1. 数据同步采用 OpenAPI、只读库、文件快照还是组合模式。
2. 数据同步频率是实时、分钟级、小时级还是每日。
3. 是否同步历史全量数据。
4. 是否有统一客户 ID、伙伴 ID、商机 ID 和订单 ID。
5. 合作伙伴、服务商、渠道商、代理商是否同一业务对象。
6. 山东区域按哪个对象字段裁剪。
7. 下单金额采用哪个字段。
8. 商机超过两周未更新以哪个时间字段判断。
9. 当前企微用户权限如何映射到 CRM 权限范围。
10. 哪些字段禁止进入 AI 分析和企微展示。

## 14. 结论

“数据同步 + 数据仓库 + 数据模型 + 语义层 + AI 受控 SQL”的方案可行，并且是实现稳定企业微信智能问数的推荐长期方案。

该方案的核心不是让 AI 自由查询数据库，而是：

- 用数据仓库承接联软 CRM 数据。
- 用标准模型消化业务口径差异。
- 用语义层提升 AI 理解能力。
- 用程序控制 SQL、权限和审计。
- 用 AI 基于真实结果生成经营分析报告。

这样既能提升自然语言理解覆盖面，也能保证数据权限、安全边界和结果可信度。
