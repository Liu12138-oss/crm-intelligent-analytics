# AI 与企业微信机器人核心模块收敛清理方案

## 1. 文档目的

本文档基于《AI企微核心收敛边界说明》进一步拆解每个模块和业务能力的收敛方式。

目标是说明：

- 某个模块是否保留、封存、拆分或后续清理。
- 如果要收敛该模块，应按什么步骤做。
- 如何保证收敛该模块时不影响其它模块。
- 每个模块收敛后需要验证什么。

本文档只定义收敛方案，不直接要求立即删除代码。后续真正改代码时，应按本方案拆成独立变更逐步执行。

## 2. 总体收敛策略

### 2.1 四阶段推进

| 阶段 | 名称 | 目标 | 关键动作 |
| --- | --- | --- | --- |
| 阶段 0 | 边界冻结 | 先停止继续扩张业务范围 | 更新文档、README 和用户可见能力清单。 |
| 阶段 1 | 入口收敛 | 先让用户和调度入口不再触达暂缓能力 | 隐藏导航、关闭意图、关闭调度、关闭真实发送。 |
| 阶段 2 | 依赖拆分 | 把核心模块从业务模块依赖中拆出来 | 拆企业微信通道层、AI 核心层、业务动作层。 |
| 阶段 3 | 代码清理 | 在测试稳定后再物理移除或归档 | 删除未引用注册、归档测试、清理配置和文档。 |

### 2.2 收敛优先级

优先收敛会主动触发外部动作或业务写入的模块：

1. 主动通知、日报调度、CRM 写回。
2. 企业微信机器人业务动作入口。
3. Web 导航和公开接口入口。
4. CRM、渠道 CRM、合同评审等重依赖业务链路。
5. 最后再清理治理、审计、模板和导出等支撑能力。

原因是主动动作和写入链路最容易造成真实业务影响，必须先关住。

### 2.3 不影响其它模块的通用做法

每个模块收敛时默认采用以下保护措施：

| 保护措施 | 说明 |
| --- | --- |
| 先加能力开关 | 新开关默认关闭，保持回滚能力。 |
| 先隐藏入口 | 前端导航、企微意图、定时任务入口先关闭。 |
| 保留数据结构 | 不删除 `.runtime`、应用存储、历史审计和任务记录。 |
| 保留只读接口兼容 | 对仍被页面或测试引用的接口返回“能力未启用”，避免启动失败。 |
| 禁止跨模块顺手重构 | 收敛一个模块时只改该模块入口和依赖，不改登录、会话、AI 核心等稳定模块。 |
| 分模块测试 | 每次只验证被收敛模块和核心 AI/企微烟测，不要求一次跑完整旧系统。 |

### 2.4 方案审查结论与当前异常

审查结论：收敛到“对接 AI + 对接企业微信机器人”这个方向成立，但现有实现还不是纯粹的 AI 核心和企微通道结构。如果按目标边界直接删除 provider、关闭 `analysis` 或移除 CRM 只读相关服务，会影响企业微信机器人启动、身份解析或普通 AI 回复。

| 当前异常 | 现状证据 | 可能影响 | 实施保护线 |
| --- | --- | --- | --- |
| 企业微信核心依赖 `analysis` | `WecomAiConversationOrchestrationService` 直接依赖 `AiGatewayService` 和 `ClarificationService`。 | 先关闭或删除 `analysis` 会导致机器人编排无法构造。 | 先抽轻量 `AiChatService` 或保留 `AiGatewayService` 兼容门面，完成迁移后再清理。 |
| `WecomBotService` 仍是业务聚合服务 | 构造函数注入 `AnalysisService`、`DailyReportService`、CRM 创建、商机查询、跟进写回、治理权限、看板编排等服务。 | 直接反注册日报、写回、看板或治理 provider 会导致应用启动失败。 | 先拆 `WecomBotCoreService` 和 `WecomBusinessActionRouter`，或保留 provider 并统一返回未启用。 |
| `AppModule` 仍是大模块注册 | 分析、企业微信、合同、日报、渠道 CRM、治理、导出等 controller/provider 在同一模块集中注册。 | 只靠模块开关很难保证构造期依赖完整。 | 禁止第一步从 `AppModule` 大量删除 provider；先做入口开关、stub 和职责拆分。 |
| 企业微信身份仍依赖 CRM 映射 | `WecomAuthService`、`CrmWecomIdentityRepository`、维护降级服务仍使用 CRM 只读或渠道 CRM 绑定身份。 | 无 CRM 数据源时，普通 AI 对话可能在身份校验阶段被阻断。 | 若要支持无 CRM 普通对话，先定义最小身份模式：验签、来源、senderId、会话和 allowlist。 |
| AI 配置依赖应用存储 | `AiModelsModule` 通过 `DatabaseModule` 获取应用存储、审计仓储和本地运行配置。 | 盲目拆 `DatabaseModule` 会影响 AI Profile、健康检查和激活配置。 | 保留 `DatabaseModule` 或先拆出 `CoreStorageModule`，保证 AI 配置读写可用。 |
| 已关闭能力仍可能出现在提示词 | 企微 idle intent、任务切换和帮助提示仍枚举日报、CRM 创建、实体查询等 action。 | 用户会被引导到已关闭能力，形成半执行链路。 | 能力关闭时同步收敛 prompt、schema、帮助文案和路由映射。 |
| 数据源检查可能过早阻断 | 维护降级服务会检查 CRM 运行时数据源是否可用。 | 机器人收到普通消息后仍可能提示 CRM 身份或数据不可用。 | 普通 AI 对话不应执行实时 CRM 数据源检查；该检查只保留给 CRM 问数等业务动作。 |

因此，本方案的实施顺序必须是：先收入口和用户可见能力清单，再保留 provider 做未启用兜底，然后拆出 AI 对话和企微通道核心，最后才做物理清理。

## 4. `ai-models` 模块收敛方案

### 当前职责

`ai-models` 当前负责：

- AI Profile 创建、编辑、复制、启停。
- 密钥加密。
- 健康检查。
- 激活和回滚。
- 运行时配置解析。
- OpenAI 兼容 HTTP、历史 Codex / Claude adapter。
- 统一 AI 执行门面。

### 收敛目标

保留为核心模块，但收敛为“OpenAI 兼容 HTTP 优先”的服务端 AI 调用底座。

### 具体做法

1. 保留 `AiRuntimeConfigResolver`、`UnifiedAiExecutionService`、`OpenAiCompatibleHttpAdapter`。
2. 保留 Profile 密钥加密、健康检查和激活回滚。
3. 将历史 SDK / CLI / provider adapter 标记为迁移兼容，不在前端提供新增入口。
4. 新增或复用 AI 核心开关：`AI_CORE_ENABLED`。
5. AI 核心关闭时，所有调用返回“AI 服务未启用”的业务异常，不向下游 provider 发请求。
6. AI 核心开启但配置不可用时，企业微信机器人返回友好降级提示。

### 不影响其它模块的措施

- 不改 `UnifiedAiExecutionService` 对外接口，避免影响 `analysis`、`wecom` 和 `contract-review` 的编译。
- 不删除历史 Profile 记录，只在页面和运行态限制可选类型。
- 不在收敛阶段改变密钥加密格式，避免旧配置无法回读。
- 不立即移除 `DatabaseModule`、`AppStorageService`、`AuditEventRepository` 和本地运行配置服务；AI Profile 仍依赖这些存储能力。
- 如果后续要拆存储模块，先提供等价 `CoreStorageModule`，再迁移 `AiModelsModule` 的导入关系。

### 验证点

- 没有激活 Profile 时，可读取环境变量默认 AI 配置。
- 激活 Profile 后，企业微信机器人使用激活 Profile。
- 健康检查失败时不会激活新配置。
- 错误提示不包含密钥、内部代理、堆栈或完整网关地址。

## 5. `wecom` 模块收敛方案

### 当前职责

`wecom` 当前同时承担：

- 企业微信消息接入、适配、回传和 receipt。
- 身份映射和企业微信目录同步。
- AI 入口分流。
- 分析结果展示。
- 日报入口。
- 客户/商机创建入口。
- 跟进模板和写回确认。
- 实体查询。
- 主动推送。

### 收敛目标

拆成“通道层”和“业务动作层”：

```text
wecom-channel
  - 消息接入
  - 验签
  - 幂等
  - 会话
  - 回复发送

wecom-ai-chat
  - 帮助提示
  - 普通 AI 对话
  - 未启用能力拒答

wecom-business-actions
  - CRM 问数
  - 客户/商机创建
  - 跟进写回
  - 日报
  - 主动通知
```

本轮默认只启用前两层。

### 当前异常与保护线

当前 `wecom` 不是单纯通道模块，尤其要注意：

- `WecomBotService` 同时注入分析、日报、CRM 创建、商机查询、跟进写回、权限治理、看板编排和审计服务。只关闭入口不等于可以删除这些 provider。
- `WecomAiConversationOrchestrationService` 直接依赖 `analysis/AiGatewayService` 和 `ClarificationService`。未迁移前，`analysis` 中的这两个服务仍属于机器人启动保护依赖。
- `WecomMaintenanceDegradationService` 当前会检查 CRM 身份源和实时数据源。普通 AI 对话如果仍走这一步，就会被 CRM 数据源状态误伤。
- 帮助文案、idle intent、任务切换和 active task 仍包含日报、CRM 创建、实体查询等已暂缓能力。收敛时必须同步删减或统一映射到未启用提示。

安全实施顺序：

1. 先保留现有 provider 注册，通过开关让已暂缓 action 返回未启用提示。
2. 再抽 `WecomBotCoreService`，只保留验签、来源校验、幂等、会话、普通 AI 回复和错误回传。
3. 再抽 `WecomBusinessActionRouter`，把 CRM 问数、日报、创建、写回、实体查询和看板桥接集中挂到 `WECOM_BUSINESS_ACTIONS_ENABLED` 后面。
4. 再把普通 AI 对话迁移到轻量 `AiChatService` 或等价门面。
5. 最后确认启动和烟测稳定后，才清理不再引用的业务 provider。

### 具体做法

1. 将 `WecomBotController` 保留为唯一机器人消息入口。
2. `WecomBotService` 只保留消息校验、会话创建、幂等和回复调度。
3. `WecomAiConversationOrchestrationService` 的默认 action 收敛为：
   - `HELP_GUIDANCE`
   - `AI_CHAT`
   - `UNSUPPORTED_BUSINESS_CAPABILITY`
4. 从默认 idle intent 中移除或禁用以下 action：
   - `DAILY_REPORT`
   - `DAILY_REPORT_QUERY`
   - `TEAM_DAILY_REPORT_QUERY`
   - `CRM_CREATE_CUSTOMER`
   - `CRM_CREATE_OPPORTUNITY`
   - `OPPORTUNITY_LOOKUP`
   - `ENTITY_LOOKUP`
   - `FOLLOW_UP_ANALYZE`
5. 如果历史 action 暂时不能删除，则在总开关 `WECOM_BUSINESS_ACTIONS_ENABLED=false` 时统一映射为未启用提示。
6. 保留 `WecomMessageAdapterService`、`WecomMessageReceiptRepository`、`WecomConversationContextRepository` 这类通道基础能力。
7. 将 `WecomDirectorySyncController` 和目录同步服务从默认核心里移出，改为后续身份/权限能力。

### 不影响其它模块的措施

- 不立即删除跟进、日报、创建相关 helper，先断入口。
- 不改变企业微信消息接入 URL，避免外部机器人配置失效。
- 不改企业微信 Web 登录回跳逻辑；该逻辑属于 `auth` 稳定能力。
- 对被关闭的业务动作返回明确中文提示，不抛出未处理异常。

### 验证点

- 企业微信普通消息可以得到 AI 回复。
- 重复 messageId 不重复调用 AI。
- CRM 问数请求返回“CRM 数据分析能力未启用”。
- 新增客户/商机请求不会调用 `opportunities` 模块。
- 日报请求不会进入 `daily-report` 模块。
- AI 配置缺失时有降级提示。

## 6. `analysis` 模块收敛方案

### 当前职责

`analysis` 当前负责：

- 自然语言意图解析。
- 补问。
- CRM 受控查询。
- SQLite / MySQL / OpenAPI / Markdown 快照等多路径执行。
- 结果归一化、报告生成、趋势预测、建议。
- Web 和企业微信结果适配。
- 查询历史、保存模板和追问。

### 收敛目标

本轮不再把 `analysis` 作为核心业务模块。只抽取普通 AI 对话需要的轻量编排能力，其余 CRM 分析链路全部暂缓。

### 当前异常与保护线

`analysis` 当前既包含 CRM 问数业务，也包含企业微信仍在使用的 AI 意图识别和补问服务。关闭 CRM 分析不等于可以立即删除 `AiGatewayService`、`ClarificationService`、能力包 runtime 或相关 provider。实施时必须先把普通 AI 对话、帮助意图和未启用能力识别迁移到新的轻量门面，再把 CRM 受控查询、报表生成、趋势预测和执行器链路关闭或归档。

### 具体做法

1. 新增轻量 `AiChatService` 或等价门面，专门服务企业微信普通 AI 对话。
2. 企业微信默认不再调用 `AnalysisService.createQuery` 处理普通消息。
3. `AiGatewayService` 中与 CRM 问数、日报、创建、实体查询相关的 prompt 和 schema 改为能力包，默认不加载。
4. 关闭以下执行器默认入口：
   - `LianruanCrmAnalysisExecutorService`
   - `AnalysisWarehouseAnalysisExecutorService`
   - `CrmSqliteReadonlyAnalysisExecutorService`
   - OpenAPI Markdown 快照查询主链
5. 保留 `UnifiedAiExecutionService` 作为 AI 调用源，不在 `analysis` 内另起模型调用方式。
6. 对 Web `/analysis` 相关接口按功能开关返回能力未启用，或从前端隐藏入口。

### 不影响其它模块的措施

- 不删除 `analysis` 数据模型和历史结果记录。
- 不改变已有结果详情公共链接逻辑，除非确认无人使用。
- 先从企业微信路由断开，再清理内部执行器注册。
- 如果合同审核暂时仍引用 `AiGatewayService`，不要直接删除该服务；先迁移到统一 AI 门面。

### 验证点

- 关闭 CRM 分析后，企业微信普通 AI 对话仍可用。
- Web `/analysis` 页面隐藏或提示未启用。
- OpenAPI、SQLite、MySQL 未配置时应用仍能启动。
- 用户请求经营数据时不会生成模拟数据。

## 7. 渠道 CRM 与 `crm-standard-api` 收敛方案

### 当前职责

该模块负责：

- 联软渠道 CRM 标准 OpenAPI token、列表、统计和诊断。
- 看板统计接口。
- Dashboard 报告生成。
- 渠道商、报备、商机、报价、订单等对象分析。

### 收敛目标

完整暂缓，不进入 AI + 企微核心。

### 具体做法

1. 默认关闭 `CHANNEL_CRM_ANALYSIS_ENABLED`。
2. 不在应用启动后自动调用渠道 CRM OpenAPI。
3. 隐藏治理页中的联软连接策略、诊断和看板模板入口。
4. 企业微信请求渠道商、渠道排名、渠道贡献时，返回渠道 CRM 分析未启用。
5. 保留配置项和文档，后续需要时可按“渠道 CRM 分析能力”独立恢复。

### 不影响其它模块的措施

- 不删除 OpenAPI client 代码，避免影响已有联调资料。
- 不删除 `.runtime` 或快照目录。
- 不修改 AI 配置模块。

### 验证点

- 未配置 `CRM_STANDARD_OPEN_API_*` 时，机器人核心可启动。
- 看板和渠道分析入口不可见或明确提示未启用。
- 不产生真实 OpenAPI 请求。

## 8. `analysis-warehouse`、SQLite 和 Text-to-SQL 收敛方案

### 当前职责

该链路负责分析库同步、SQL Guard、SQLite 快照导入、受控 SQL 查询等能力。

### 收敛目标

暂缓，不进入核心。

### 具体做法

1. 默认关闭 `ANALYSIS_WAREHOUSE_ENABLED` 或等价开关。
2. 关闭启动时同步、导入和 SQL 执行入口。
3. 将 Text-to-SQL prompt、SQL guard 和数据仓库执行器从企业微信默认路径移除。
4. 文档中明确：当前机器人不支持实时或离线数据库问数。

### 不影响其它模块的措施

- 不删除 SQL Guard 代码，避免未来恢复时丢失安全边界。
- 不删除数据库梳理文档和字段字典。
- 不修改 CRM 认证和登录链路。

### 验证点

- 不配置分析库时应用启动正常。
- 企业微信普通 AI 对话不访问数据库。
- 数据库问数请求返回未启用提示。

## 9. `contract-review` 合同评审收敛方案

### 当前职责

合同评审模块已经包含：

- CRM 待 1 级审批合同列表。
- 合同源数据详情。
- 按合同 ID 创建审核任务。
- 本地 `.docx` 兼容上传。

## 21. 推荐执行顺序

### 第一批：立刻降低业务风险

1. 更新 README 和机器人帮助文案，只列 AI 对话和基础帮助。
2. 关闭企业微信业务动作意图。
3. 关闭日报调度和主动通知真实发送。
4. 隐藏合同、分析、管理报表、看板和审计导航。

### 第二批：拆核心依赖

1. 从 `wecom` 中拆出通道层和普通 AI 对话层。
2. 从 `analysis` 中拆出轻量 AI 对话服务。
3. 让机器人普通回复只依赖 `ai-models` 和会话存储。
4. 用能力未启用提示承接所有暂缓业务请求。

### 第三批：接口和启动注册收敛

1. 按开关控制 `contract-review`、`opportunities`、`daily-report`、`notifications`、`crm-standard-api` 的注册或执行。
2. 删除 AppModule 中不再需要默认启动的 controller/provider 注册，或改为模块级条件注册。
3. 保持历史接口返回兼容提示，避免前端旧路由直接白屏。

### 第四批：物理清理和归档

1. 确认至少一个版本周期内未重新启用暂缓能力。
2. 将不再维护的页面、测试和文档移入 archive 或删除。
3. 同步更新规格、部署参数和测试清单。

| 要求审核合同 | 提示合同评审未启用。 |
| 要求新增客户 | 提示 CRM 创建未启用。 |
| 要求写跟进 | 提示 CRM 跟进写回未启用。 |
| 要求生成日报 | 提示日报未启用。 |
| 要求导出数据 | 拒绝导出并说明当前不支持。 |

### 启动烟测

| 缺失配置 | 期望 |
| --- | --- |
| 无 CRM 只读库 | 应用可启动，普通 AI 对话可用。 |
| 无 CRM 写回账号 | 应用可启动，写回请求提示未启用。 |
| 无联软 OpenAPI | 应用可启动，渠道分析提示未启用。 |
| 无合同标准包 | 应用可启动，合同评审提示未启用。 |
| 无日报配置 | 应用可启动，不发送日报。 |
| 仅关闭 `CRM_ANALYSIS_ENABLED` 和 `WECOM_BUSINESS_ACTIONS_ENABLED` | 应用可启动，企业微信“你好”仍返回普通 AI 回复或 AI 未配置提示。 |

## 23. 文档同步要求

真正执行收敛时，需要同步检查并更新：

- `README.md` 的项目定位、能力总览、企业微信机器人当前支持能力、环境变量。
- `AGENTS.md` 中关于合同、CRM、日报、治理审计的默认约束，避免与新边界冲突。
- `specs/001-crm-intelligent-analytics/` 下主规格、计划、数据模型、快速开始和任务。
- `docs/需求文档/企业微信机器人.md`，使其成为新的机器人核心边界来源。
- `docs/architecture/ai-model-profile-governance.md`，确认 AI 配置仍是核心能力。
- 部署文档中的 CRM、合同、日报和通知配置项，将其标为后续业务能力。

## 24. 最终交付判定

当以下条件都满足时，可以认为模块收敛完成：

1. 默认部署只需要企业微信机器人凭证、AI 配置和轻量应用状态。
2. 不配置 CRM、渠道 CRM、合同、日报、通知相关环境变量时，系统仍可启动。
3. 企业微信机器人可以稳定完成普通 AI 对话。
4. 所有暂缓业务请求都有明确中文提示，不进入真实业务链路。
5. 前端不再展示暂缓能力入口。
6. 主动通知和日报不会自动发送。
7. 历史数据、标准包、配置和审计记录未被误删。
8. 后续任一业务能力都可以按独立开关和独立变更重新启用。
