## Requirements
### Requirement: 查询模板必须支持自由 SQL 编写与系统级范围治理解耦

系统 MUST 允许具备模板治理权限的用户编写只读业务 SQL，但模板作者不需要也不得被要求记忆内部权限参数。系统 MUST 在模板保存、校验、预览和执行阶段自动生成范围治理快照，并基于该快照决定自动注入、兼容放行、审核后放行或阻断。

#### Scenario: 自由 SQL 模板自动生成范围治理快照
- **WHEN** 模板作者保存一条未包含 `:scopeDepartmentIds` 或 `:scopeOwnerIds` 的只读 SELECT 模板
- **THEN** 系统必须分析该 SQL 的主业务表、范围条件、展示维度和静态维度来源
- **THEN** 系统必须生成中文范围治理快照，而不是要求作者手动补内部权限占位符

#### Scenario: 模板作者看到中文修复建议
- **WHEN** 模板 SQL 因范围不明确、展示口径不一致或静态团队清单而无法直接发布
- **THEN** 系统必须返回中文风险说明和修复建议
- **THEN** 提示内容不得只暴露内部参数名或数据库错误

#### Scenario: 治理灰度模式记录风险但不扩大权限
- **WHEN** `QUERY_TEMPLATE_SCOPE_GOVERNANCE_MODE` 配置为 `observe`
- **THEN** 系统必须生成治理快照、风险项和审计记录
- **THEN** 系统不得因为观察模式扩大用户 `scopeSnapshot`
- **THEN** 普通用户执行复杂模板时可以保留兼容放行，但结果说明必须标明当前模板需要治理复核

### Requirement: 简单模板必须按当前用户范围自动收口执行

当模板被识别为可自动收口模板时，系统 MUST 在执行 SQL 时基于当前用户 `scopeSnapshot` 自动注入组织、部门和负责人范围。自动注入后的执行 SQL MUST 与原始 SQL、执行参数、范围快照和审计记录建立关联。命中应用超级管理员授权的用户 MUST 使用统一范围解析返回的全量范围执行，不得继续依赖旧的分析全量查询名单。

#### Scenario: 部门负责人执行默认全量模板时自动收口
- **WHEN** 部门负责人执行一条没有显式部门条件的全量业务模板
- **THEN** 系统必须在执行前按该负责人的最终 `scopeSnapshot` 自动收口
- **THEN** 结果不得返回不属于当前范围且未被白名单授权的部门、负责人或团队数据

#### Scenario: 应用超级管理员执行同一模板时允许全量
- **WHEN** 应用超级管理员执行同一条可自动收口模板
- **THEN** 系统必须按其应用超级管理员全量范围执行
- **THEN** 审计记录必须明确本次为应用超级管理员全量范围

#### Scenario: 历史全量分析名单不再作为独立放行条件
- **WHEN** 用户只存在于历史全量分析名单但未完成迁移或未命中新应用超级管理员授权
- **THEN** 系统不得仅凭旧名单放行模板全量执行
- **THEN** 系统必须要求其命中新应用超级管理员授权或按普通范围自动收口

### Requirement: 固定范围模板必须通过权限兼容性校验

当模板 SQL 写死部门 ID、部门名称、负责人 ID、团队名称或其它固定范围条件时，系统 MUST 将其识别为固定范围模板。固定范围模板执行前 MUST 校验当前用户范围覆盖该固定范围；无法解析固定范围或当前用户无权覆盖时，系统 MUST 阻断执行。

#### Scenario: 有权限用户执行固定山东区模板
- **WHEN** 模板固定查询山东区，且当前用户的最终范围覆盖山东区
- **THEN** 系统可以执行该模板
- **THEN** 审计记录必须保存固定范围说明和兼容性校验结果

#### Scenario: 无权限用户执行固定山东区模板被阻断
- **WHEN** 模板固定查询山东区，但当前用户最终范围不覆盖山东区
- **THEN** 系统必须拒绝执行
- **THEN** 系统必须提示当前模板限定了特定范围，用户无权查看该范围

### Requirement: 复杂模板必须审核后发布或阻断

系统 MUST 将静态目标表、静态团队清单、`UNION ALL` 团队枚举、多主表聚合、复杂 CTE、展示口径与过滤口径不一致的模板识别为复杂模板。复杂模板 MUST 由治理人员补充权限主表、展示部门口径、目标表收口策略和审核结论后才能对普通用户发布；无法证明范围安全的模板 MUST 阻断。

#### Scenario: 静态团队目标模板需要审核
- **WHEN** 模板通过 `UNION ALL` 或硬编码列表内置全公司团队目标
- **THEN** 系统必须要求治理人员声明目标团队如何按当前用户范围收口
- **THEN** 未完成审核前普通用户不得执行该模板

#### Scenario: 无法证明安全的复杂模板被阻断
- **WHEN** 模板存在多个主业务表且系统无法判断应按哪个归属字段收口
- **THEN** 系统必须阻断普通用户执行
- **THEN** 系统必须记录阻断原因和待补充的治理元数据

### Requirement: 结果展示口径必须与权限过滤口径一致

系统 MUST 校验模板结果中的团队、部门、负责人、区域等展示维度来源是否与权限过滤口径一致，或已被治理审核明确声明为合法展示口径。系统 MUST 对表格、图表、矩阵表、转置表和导出结果使用同一展示口径校验结果。

#### Scenario: 按商机归属过滤却按客户部门展示时被识别为风险
- **WHEN** 模板按 `opportunities.department_id` 或 `opportunities.user_id` 收口，但结果团队名称来自 `customers.department_id`
- **THEN** 系统必须识别为展示口径与过滤口径不一致
- **THEN** 未经审核确认前不得向普通用户交付该结果

#### Scenario: 表达式别名必须追踪到基础字段来源
- **WHEN** 模板通过 `CASE`、函数、`GROUP BY` 或 CTE 输出 `team_name`、`department_name`、`owner_name`
- **THEN** 系统必须追踪这些展示列引用的基础表别名和字段
- **THEN** 如果来源字段经过 `customers.department_id -> departments.name` 关联，而权限过滤口径来自 `opportunities.department_id` 或 `contracts.department_id`，系统必须识别为展示口径不一致

#### Scenario: 已审核客户归属口径可以展示
- **WHEN** 模板明确声明客户归属部门是该报表的业务展示口径，且治理人员已审核通过
- **THEN** 系统可以按该展示口径交付结果
- **THEN** 结果范围说明和审计记录必须标明该模板使用客户归属部门展示

### Requirement: 模板治理结果必须覆盖预览、执行、重跑和导出

系统 MUST 保证模板预览、正式执行、最近查询重跑、历史结果补查和导出补查使用同一套范围治理结论。任何入口不得复用历史授权快照或历史模板结果绕过当前范围治理。

#### Scenario: 最近查询重跑重新执行范围治理
- **WHEN** 用户从最近查询中重跑某条模板结果
- **THEN** 系统必须基于当前模板定义和当前用户范围重新执行范围治理
- **THEN** 系统不得沿用历史执行时的范围结论直接放行

#### Scenario: 导出补查沿用同一治理结论
- **WHEN** 用户对模板结果发起导出
- **THEN** 系统必须确认导出数据仍满足当前模板范围治理结论和当前用户范围
- **THEN** 导出内容不得超过页面结果被允许展示的范围

#### Scenario: 历史模板结果当前不可执行时阻断导出
- **WHEN** 用户导出历史模板结果，但当前模板治理结论已变为不可对该用户执行
- **THEN** 强制模式下系统必须阻断导出并提示重新执行查询
- **THEN** 观察模式下系统可以导出当前页面已有结果，但必须在导出审计中记录治理风险

### Requirement: 固定范围模板必须支持确认后改写为当前可见范围

当模板 SQL、默认条件或问题文本写死部门、负责人、团队、区域或静态团队清单时，系统 MUST 先校验当前用户范围是否覆盖该固定范围。若当前用户不覆盖该固定范围，系统 MUST 阻断原模板执行。系统 MAY 生成“改为当前可见范围”的 AI 辅助改写建议，但必须在用户确认后，才允许将固定范围条件替换为当前 `scopeSnapshot` 覆盖范围，并重新执行 SQL 白名单、AST 安全校验、范围注入和审计。

#### Scenario: 无权限用户执行固定山东模板被提示改写
- **WHEN** 模板固定查询山东区
- **AND** 当前北京领导用户没有山东区数据权限
- **THEN** 系统必须阻断原山东范围执行
- **THEN** 系统可以提示“该模板原范围为山东区，你当前无权查看山东数据。是否改为查询你当前可见范围”
- **THEN** 系统不得在用户确认前静默改写并执行

#### Scenario: 用户确认改写后重新校验
- **WHEN** 用户确认将固定范围模板改为当前可见范围
- **THEN** 系统必须使用当前用户 `scopeSnapshot` 替换固定范围条件
- **THEN** 系统必须重新执行只读 SQL、字段白名单、AST 安全、范围注入和预检校验
- **THEN** 校验通过后才可以正式执行模板

#### Scenario: 固定范围无法安全替换时阻断
- **WHEN** 模板包含无法可靠解析或替换的固定部门条件、静态团队清单或复杂展示口径
- **THEN** 系统必须阻断改写执行
- **THEN** 系统必须提示联系管理员将模板治理为通用模板

### Requirement: 固定范围改写不得修改来源模板定义

系统 MUST 将固定范围改写限定为本次执行参数、当前用户个人模板副本或个人配置。系统 MUST NOT 将执行人的当前部门、负责人或范围条件自动写回来源模板定义。若用户将改写结果保存为模板，系统 MUST 创建新的个人模板副本并保留来源模板追溯。

#### Scenario: 改写执行不改变来源模板
- **WHEN** 北京领导确认把山东模板改为当前可见范围并执行
- **THEN** 系统不得修改来源模板中的山东固定条件
- **THEN** 后续其他有山东权限的用户执行来源模板时仍可按原模板治理结果处理

#### Scenario: 用户保存改写结果为我的模板
- **WHEN** 用户在改写执行成功后选择保存为模板
- **THEN** 系统必须创建当前用户自己的模板副本
- **THEN** 新模板必须记录来源模板 ID 和改写范围摘要

### Requirement: 固定范围识别与改写必须完整审计

系统 MUST 为固定范围模板的识别、原范围阻断、用户确认改写、改写后校验、改写执行成功和改写失败写入审计。审计记录 MUST 至少包含来源模板 ID、原固定范围摘要、执行人当前范围摘要、用户确认状态、改写结果和失败原因。

#### Scenario: 原范围执行被阻断时写审计
- **WHEN** 用户无权覆盖固定范围模板并被阻断
- **THEN** 系统必须记录固定范围阻断审计
- **THEN** 审计必须包含原固定范围和当前用户范围摘要

#### Scenario: 改写执行成功时写审计
- **WHEN** 用户确认改写并成功执行模板
- **THEN** 系统必须记录固定范围改写执行审计
- **THEN** 审计必须包含改写前后的范围摘要和用户确认时间
