## Requirements
### Requirement: 系统必须建立“CRM 权限权威源 + 应用能力治理层”的双层权限模型

系统 MUST 将 CRM 组织、部门、负责人、角色和数据范围权限作为常规数据访问的权威来源，并将应用层的菜单、动作、导出资格、合同审核扩展授权、企业微信入口资格和应用超级管理员授权作为独立的能力治理层。常规应用层能力配置 MUST 只能收紧入口和动作，不得扩大 CRM 已定义的数据范围、对象范围或字段范围；应用超级管理员授权是受审计的显式例外，命中后可以获得本系统内全量功能、全量操作和全量数据范围，但仍不得绕过登录认证、安全前置检查、字段白名单、SQL 白名单、危险动作确认和审计约束。

#### Scenario: 常规应用层能力配置不得扩大 CRM 数据范围
- **WHEN** 平台管理员为某个 CRM 角色开通 `analysis.use`、`analysis.export` 或 `wecom.analysis.use`
- **THEN** 系统必须仍按该角色当前 CRM 组织、部门、负责人和字段权限生成最终 `scopeSnapshot`
- **THEN** 不得因为常规应用层开通了动作就扩大其 CRM 数据范围

#### Scenario: 应用超级管理员授权显式扩大系统内权限和数据范围
- **WHEN** 平台管理员为某个 CRM 用户或 CRM 角色开通应用超级管理员授权
- **THEN** 系统必须通过统一权限决策授予其本系统内全部菜单和全部动作
- **THEN** 系统必须通过统一范围解析授予其全量数据范围，并在范围快照中标明授权来源

#### Scenario: 同一用户的 Web 与企业微信共享相同 CRM 数据范围
- **WHEN** 同一 CRM 用户分别从 Web 工作台和企业微信机器人发起分析请求
- **THEN** 系统必须对两个入口注入同一份当前实时 CRM 数据范围或应用超级管理员全量范围
- **THEN** 企业微信灰度或通道资格只可影响入口是否可用，不得改变最终数据范围

### Requirement: 系统必须按 CRM 角色维护菜单与动作能力矩阵

系统 MUST 提供按 CRM `roleId` 维护的角色能力矩阵。每个角色 MUST 至少可配置 `visibleMenus`、`actionKeys`、`webConsoleEnabled`、`wecomBotEligible`、`exportAllowed`、`templateManageAllowed`、`contractReviewUploadAllowed`、`contractReviewCrossViewAllowed` 和 `contractReviewCrossDownloadAllowed`。系统 MUST 支持在不新增专门“治理审计管理员”组织角色的前提下，将模板治理、权限治理、审计检索和 AI 模型治理能力分配给平台管理员或被授权的业务管理角色。

#### Scenario: 平台管理员更新某个 CRM 角色的菜单与动作权限
- **WHEN** 平台管理员在权限中心编辑某个 CRM 角色的菜单、动作或渠道资格并保存
- **THEN** 系统必须持久化该角色的能力矩阵
- **THEN** 后续命中该角色的用户在下一次能力快照请求或入口鉴权时必须读取更新后的权限

#### Scenario: 业务管理角色只被授予模板治理而非系统级治理
- **WHEN** 某个 CRM 角色仅被授予 `template.manage`，但未被授予 `governance.policy.manage`、`audit.view` 和 `ai_profile.manage`
- **THEN** 命中该角色的用户必须只能维护查询模板
- **THEN** 该用户不得访问权限中心、审计检索或 AI 模型治理系统级入口

### Requirement: 系统必须提供权限中心治理页面并限制访问范围

系统 MUST 在 Web 治理区提供 `权限中心` 页面，作为 `查询模板管理`、`审计检索` 和 `AI 模型治理` 之外的独立治理入口。权限中心 MUST 至少包含顶部摘要卡、角色权限分区、企业微信灰度分区、数据与导出基线分区、身份映射诊断分区和变更记录分区。系统 MUST 仅允许具备 `governance.policy.manage` 或等价系统级治理动作的用户访问该页面和相关接口。

#### Scenario: 平台管理员进入权限中心
- **WHEN** 具备系统级治理动作的管理员进入治理区并点击 `权限中心`
- **THEN** 页面必须展示权限治理摘要、角色矩阵、企业微信灰度和身份映射诊断等分区
- **THEN** 页面不得跳转到独立的新壳层，而应继续沿用现有治理后台骨架

#### Scenario: 未被授权的用户访问权限中心被阻断
- **WHEN** 不具备 `governance.policy.manage` 的用户尝试访问权限中心页面或调用其接口
- **THEN** 系统必须拒绝访问
- **THEN** 该用户不得读取角色矩阵、灰度策略或身份诊断明细

### Requirement: 系统必须返回统一的有效权限快照供前端消费

系统 MUST 提供统一的有效权限快照接口，返回当前登录用户的 `scopeSummary`、`roleNames`、`visibleMenus`、`actionKeys`、`wecomBotAccessState`、`wecomBotAccessReason`、合同审核权限摘要、导出资格与剩余次数、模板可见数量、数据新鲜度和服务状态。前端 MUST 基于该快照决定导航显隐、动作可用态和无权限提示，不得再通过散落的角色名关键字硬编码判断菜单可见性。

#### Scenario: Web 工作台根据有效权限快照决定菜单显隐
- **WHEN** 登录用户请求工作台初始化能力快照
- **THEN** 系统必须返回该用户当前可见的菜单键集合和动作键集合
- **THEN** 前端必须据此决定 `智能分析`、`智能合同审核`、`查询模板管理`、`权限中心`、`审计检索` 和 `AI 模型治理` 的可见性

#### Scenario: 权限变更后下一次能力快照立即生效
- **WHEN** 管理员修改某角色的菜单或动作权限后，命中该角色的用户再次请求能力快照
- **THEN** 系统必须返回新的有效权限快照
- **THEN** 不得继续返回旧的菜单、动作或企业微信入口状态

### Requirement: 系统必须提供用户级有效权限预览能力

系统 MUST 提供用户级有效权限预览接口，允许平台管理员基于 `crmUserId` 或 `wecomUserId` 预览某个用户在当前时刻命中的角色权限、菜单、动作、企业微信入口状态、合同审核权限和最终拒绝原因。预览结果 MUST 复用与正式运行相同的权限决策服务，不得另写一套仅供页面展示的伪逻辑。

#### Scenario: 管理员按 CRM 用户预览最终有效权限
- **WHEN** 平台管理员输入某个 `crmUserId` 发起权限预览
- **THEN** 系统必须返回该用户命中的角色、菜单、动作、导出资格、合同审核资格和企业微信入口状态
- **THEN** 返回结果必须与该用户真实运行时会命中的权限判断保持一致

#### Scenario: 管理员按企业微信用户预览映射后的权限
- **WHEN** 平台管理员输入某个 `wecomUserId` 发起权限预览
- **THEN** 系统必须先完成企业微信到 CRM 用户的映射解析
- **THEN** 若映射成功，系统返回映射后的最终有效权限
- **THEN** 若映射失败，系统返回 `UNMAPPED_CRM_IDENTITY` 或等价状态，而不是伪造权限结果

### Requirement: 系统必须提供身份映射诊断与排障链路

系统 MUST 提供身份映射诊断能力，能够基于企业微信用户或 CRM 用户展示 `wx_user_maps`、`wx_users`、`wx_organization_maps`、`wx_departments`、`wx_user_department_maps` 与 CRM 用户/角色/部门之间的映射关系、当前映射状态、当前权限状态和失败原因。系统 MUST 将身份映射失败、映射冲突和映射成功但入口未开通区分为不同诊断结果。

#### Scenario: 企业微信用户已正确映射并命中角色权限
- **WHEN** 管理员查询一个已完成映射且已具备角色权限的企业微信用户
- **THEN** 系统必须展示其对应 CRM 用户、角色、部门、分析资格和企业微信入口状态
- **THEN** 诊断结果中不得将该用户误标记为未映射或无权限

#### Scenario: 企业微信用户未映射时返回专属失败类型
- **WHEN** 管理员查询一个尚未命中 `wx_user_maps` 或无法解析到有效 CRM 用户的企业微信账号
- **THEN** 系统必须将该结果标记为未映射或映射异常
- **THEN** 不得把该结果混同为普通角色未开通或灰度未命中

### Requirement: 系统必须继续执行全局基线策略与角色动作的叠加校验

系统 MUST 在角色动作允许的前提下继续执行全局基线策略，包括允许访问的数据域、字段白名单、敏感字段脱敏、单次导出行数上限和每日导出次数上限。角色动作只决定用户是否具备发起资格，最终对象、字段和阈值仍 MUST 受全局基线策略控制。

#### Scenario: 角色允许导出但全局导出阈值仍然生效
- **WHEN** 某个角色已被授予 `analysis.export` 且当前用户具备导出资格
- **THEN** 系统仍必须校验单次导出行数和每日导出次数上限
- **THEN** 若超过全局阈值，系统必须阻断导出并返回明确原因

#### Scenario: 角色允许分析但敏感字段仍必须脱敏
- **WHEN** 某个角色已被授予 `analysis.use` 并成功发起分析请求
- **THEN** 系统必须继续按 `maskedFields` 和字段白名单处理结果
- **THEN** 不得因为该角色可以分析就直接放开敏感字段返回

### Requirement: 系统必须为权限治理与排障行为记录审计事件

系统 MUST 对角色权限变更、角色权限发布、用户权限预览、身份映射诊断查询、企业微信灰度策略变更和关键权限拒绝行为记录审计事件。审计事件 MUST 至少包含操作者、目标角色或目标用户、变更前后摘要、命中的权限状态、失败原因和发生时间。

#### Scenario: 平台管理员修改角色权限时产生审计事件
- **WHEN** 平台管理员保存某个角色的菜单、动作或渠道权限变更
- **THEN** 系统必须写入一条角色权限变更审计事件
- **THEN** 审计事件中必须包含变更角色、变更前摘要和变更后摘要

#### Scenario: 管理员执行权限预览时产生审计事件
- **WHEN** 平台管理员对某个 CRM 用户或企业微信用户执行权限预览
- **THEN** 系统必须记录该次预览行为
- **THEN** 审计事件中必须包含目标用户标识、命中的最终权限状态和预览时间

### Requirement: 审计中心必须提供独立的 SQL 审计分区

系统 MUST 在现有审计中心内新增独立的 `SQL 审计` 分区，用于检索本系统后端服务访问 CRM 数据库时的 SQL 审计记录。该分区 MUST 与现有 AI 审计、用户行为审计并列展示，而不是要求治理用户跳转到新的一级菜单或旁路排障页面。

#### Scenario: 被授权治理用户在审计中心看到 SQL 审计 Tab

- **WHEN** 具备 SQL 审计查看权限的治理用户进入审计中心
- **THEN** 页面必须展示 `SQL 审计` Tab
- **THEN** 该用户不得需要离开现有审计中心骨架才能查看 SQL 审计

#### Scenario: 未被授权用户看不到 SQL 审计分区

- **WHEN** 用户进入审计中心但未具备 SQL 审计查看权限
- **THEN** 页面不得展示 `SQL 审计` Tab
- **THEN** 该用户不得通过前端切换或直接接口调用读取 SQL 审计数据

### Requirement: SQL 审计分区必须提供摘要、筛选、明细与详情能力

SQL 审计分区 MUST 至少提供顶部摘要卡、筛选栏、分页明细表和详情抽屉。摘要卡 MUST 至少展示 SQL 总量、写 SQL 数、失败数、阻断数、高风险数和平均耗时；筛选栏 MUST 至少支持按用户、模块、数据库角色、执行阶段、操作类型、状态、表名、请求 ID、会话 ID 和时间范围检索。

#### Scenario: 治理用户按表名和时间范围检索 SQL 审计

- **WHEN** 治理用户在 SQL 审计分区输入表名并限定时间范围
- **THEN** 系统必须返回命中条件的 SQL 审计列表
- **THEN** 列表结果必须能够展示每条记录的时间、用户、模块、操作类型、状态和耗时摘要

#### Scenario: 治理用户查看单条 SQL 审计详情

- **WHEN** 治理用户点击某条 SQL 审计记录查看详情
- **THEN** 页面必须展示该 SQL 的业务上下文、技术上下文、执行来源、命中适配器、fallback 原因和脱敏 SQL 摘要
- **THEN** 若当前用户没有敏感明细权限，页面不得自动展示完整 SQL 与完整参数

### Requirement: 权限中心必须提供超级管理员授权治理入口

权限中心 MUST 将历史“全量查询授权”升级为“超级管理员授权”治理入口。该入口 MUST 支持按 CRM 用户和 CRM 角色配置应用超级管理员授权，并以中文明确提示该授权等同本系统超级管理员，会开放全部功能、全部操作和全量数据范围。

#### Scenario: 管理员查看超级管理员授权分区

- **WHEN** 具备系统级治理动作的管理员进入权限中心
- **THEN** 页面必须展示“超级管理员授权”分区
- **THEN** 页面必须展示当前已授权用户、已授权角色、更新时间、更新人和变更原因

#### Scenario: 保存超级管理员授权必须填写原因

- **WHEN** 管理员新增、禁用或恢复应用超级管理员授权
- **THEN** 系统必须要求填写中文变更原因
- **THEN** 未填写原因时不得保存授权变更

### Requirement: 权限中心人员和部门选择入口必须统一复用企业微信组织架构选择器

权限中心中所有选择人员或部门的入口 MUST 统一复用企业微信组织架构选择器，包括企业微信准入、超级管理员授权、白名单设置、日报治理、身份映射诊断和用户权限预览。人员和部门选择入口不得继续各自维护平铺下拉搜索逻辑；角色、状态、模式等非组织对象选择 MAY 继续使用既有选择器。

#### Scenario: 权限中心所有人员和部门入口使用统一选择器

- **WHEN** 管理员分别在权限中心不同标签页选择人员或部门
- **THEN** 系统必须打开同一套企业微信组织架构选择体验
- **THEN** 不同标签页不得出现搜索行为、清空行为或未映射处理口径不一致的问题

#### Scenario: 非组织对象入口不强制改造

- **WHEN** 管理员在权限中心选择角色、状态、模式或布尔配置
- **THEN** 系统可以继续使用现有下拉或分段控件
- **THEN** 本次改造不得要求非组织对象保存字段发生变化

#### Scenario: 权限中心逐页审查人员与部门入口

- **WHEN** 实施本变更时审查权限中心所有标签页
- **THEN** 角色权限页若不包含人员或部门选择则无需替换
- **THEN** 企业微信准入、超级管理员授权、白名单设置、日报治理、身份映射诊断和用户权限预览中的人员或部门入口必须全部替换为统一组织架构选择器

### Requirement: 权限中心人员和部门选择改造不得改变既有治理保存语义

权限中心人员和部门选择器替换后，企业微信灰度策略、超级管理员授权、数据范围白名单、日报团队、身份映射诊断和权限预览的保存或查询 payload MUST 保持原有业务字段语义。前端组件只负责选择体验和 ID 映射，不得在提交时新增绕过后端校验的授权字段。

#### Scenario: 企业微信灰度用户保存字段保持不变

- **WHEN** 管理员通过组织架构选择器调整企业微信灰度白名单用户和停用用户并保存
- **THEN** 请求仍必须提交 CRM 用户 ID 形式的 `allowUserIds` 与 `denyUserIds`
- **THEN** 后端必须继续执行原有灰度策略校验和审计

#### Scenario: 企业微信灰度部门保存字段保持不变

- **WHEN** 管理员通过组织架构选择器调整企业微信灰度白名单部门并保存
- **THEN** 请求仍必须提交 CRM 部门 ID 形式的 `allowDepartmentIds`
- **THEN** 后端必须继续执行原有灰度策略校验和审计

#### Scenario: 数据范围白名单部门保存字段保持不变

- **WHEN** 管理员通过组织架构选择器调整数据范围白名单授权部门并保存
- **THEN** 请求仍必须提交 CRM 部门 ID 形式的 `departmentIds`
- **THEN** 后端必须继续执行原有数据范围白名单校验和审计

#### Scenario: 权限预览查询字段保持不变

- **WHEN** 管理员通过组织架构选择器选择 CRM 用户或企业微信账号执行权限预览
- **THEN** 请求仍必须按场景提交 `crmUserId` 或 `wecomUserId`
- **THEN** 预览结果必须继续复用正式运行时权限决策服务
