## Purpose
规范 CRM 读写能力的 API-first 接入顺序与受控兜底边界，确保新增能力、存量改造、受控分析读取和企业微信受控创建都优先复用官方 API，并在官方能力不足时进入可审计的受控兜底路径。

## Requirements
### Requirement: CRM业务能力必须优先复用官方 API

涉及 CRM 业务对象读写的新增能力，以及对既有非认证能力的功能改造，系统 MUST 先评估并优先复用 CRM 官方 API。对于受控分析读取，系统 MUST 对每个读取步骤先匹配官方 API 适配器，再决定是否进入受控直查 SQL 或受控内部只读接口。只要官方 API 已满足目标对象、动作、字段、权限语义、聚合形态与稳定性要求，就不得因数据库更熟悉、实现更快或已有旧代码而绕过官方 API。

#### Scenario: 新能力命中可用官方 API

- **WHEN** 团队要实现一个新增 CRM 业务能力，且官方 API 已覆盖所需对象与动作
- **THEN** 方案必须以官方 API 作为主接入路径，并将数据库直连排除为默认实现

#### Scenario: 存量非认证能力发生改造

- **WHEN** 团队对既有的非认证功能进行需求扩展、重构或缺陷修复
- **THEN** 该变更必须先复核是否已有可用官方 API，并在可用时优先收敛到官方 API 路径

#### Scenario: 分析读取步骤命中官方 API 适配器

- **WHEN** 受控分析读取中的某个步骤已被确认可以通过官方 API 满足对象、字段、聚合与权限语义
- **THEN** 系统必须为该步骤选择官方 API 适配器作为执行来源，并在快照中记录该选择结果

### Requirement: 官方 API 缺失时必须进入受控兜底路径

当 CRM 官方 API 不存在、权限语义不足、返回字段不满足、聚合形态不满足、时效无法满足或稳定性无法支撑业务目标时，系统 MUST 先形成缺口说明，再选择受控内部接口、受控只读 SQL 或受控数据库路径。兜底方案 MUST 明确对象范围、表字段白名单、权限继承方式、审计留痕、失败降级与回退方案。对于受控分析编排，兜底步骤 MUST 额外记录 `gapReason`、对象与字段白名单快照、权限继承快照、审计引用和回退策略；受控直查方案 MUST 明确结果规模限制、危险动作阻断和失败降级方案。

#### Scenario: 官方 API 不覆盖目标能力

- **WHEN** 需求所需的 CRM 对象或动作在官方 API 中不存在，或无法满足经过确认的业务约束
- **THEN** 设计文档必须记录 API 缺口、选用的兜底路径和对应的权限边界与审计方案后，系统才可以进入开发

#### Scenario: 官方 API 不覆盖目标分析形态

- **WHEN** 当前分析步骤需要的对象、字段、聚合形态或返回结构无法由官方 API 满足
- **THEN** 设计文档必须记录 API 缺口与受控只读兜底路径后，系统才可以进入开发

#### Scenario: 禁止无说明直连数据库

- **WHEN** 实现方案准备直接访问 CRM 数据库或新增内部直连接口
- **THEN** 若未提供官方 API 不可用证明和受控边界说明，该方案必须被视为不合规并停止推进

#### Scenario: 分析读取步骤进入受控直查路径

- **WHEN** 受控分析中的某个读取步骤无法由官方 API 满足并被路由到只读 SQL 或内部只读接口
- **THEN** 系统必须为该步骤记录缺口原因、白名单快照、权限快照和审计引用后才允许执行

### Requirement: 存量非认证能力必须按同一原则逐步收敛

系统 MUST 对现有非认证 CRM 能力建立接入方式台账，标记其当前使用官方 API、自建接口或数据库路径的现状，并在后续功能变更时按“能用官方 API 就先收敛官方 API”的原则逐步整改，不要求一次性整体重写。

#### Scenario: 盘点现有非认证能力

- **WHEN** 团队梳理现有的问数、治理、审计、导出、目录同步或其他非认证能力
- **THEN** 每项能力都必须记录当前接入方式以及后续是否需要向官方 API 收敛

#### Scenario: 存量能力继续扩展

- **WHEN** 某个已有非认证能力进入下一轮需求开发
- **THEN** 新增改动部分必须遵循相同的 API 优先顺序，且不得继续扩大无约束数据库直连范围

### Requirement: 用户认证链路在本次约束梳理中保持冻结

当前已经稳定的用户认证、登录、会话建立和统一鉴权装载链路 MUST 保持现状，不得在本次“API 优先约束”落地中顺带重构；若后续确需调整认证能力，必须另起独立变更进行评估和实施。

#### Scenario: API 优先梳理影响认证模块

- **WHEN** 研发团队在落实 CRM API 优先约束时发现认证模块也可以被顺手调整
- **THEN** 本次变更必须明确排除该调整，保持认证行为、接口和会话边界不变

#### Scenario: 认证能力需要后续独立调整

- **WHEN** 团队确实需要修改 CRM 登录、扫码登录、会话 Cookie 或统一鉴权装载
- **THEN** 必须通过独立规格变更单独评审，而不能并入当前 API 优先改造

### Requirement: 企业微信受控创建必须复用内部创建服务与 CRM 官方创建接口

当企业微信机器人执行受控新建客户或受控新建商机时，系统 MUST 复用既有内部创建服务和 CRM 官方创建接口，不得绕过既有字段校验、自定义字段映射、默认值补齐和错误处理逻辑。系统 MUST 明确禁止把机器人创建请求降级为数据库直写或未受控的临时接口。

#### Scenario: 机器人发起新建客户

- **WHEN** 企业微信机器人进入受控新建客户确认并执行创建
- **THEN** 系统复用内部新增客户服务和 CRM 官方 `/api/v2/customers`
- **THEN** 系统不得新增数据库直写分支绕过现有字段映射与校验

#### Scenario: 机器人发起新建商机

- **WHEN** 企业微信机器人进入受控新建商机确认并执行创建
- **THEN** 系统复用内部新增商机服务和 CRM 官方 `/api/v2/opportunities`
- **THEN** 当产品映射、自定义字段映射或默认值补齐失败时，系统必须直接拒绝创建，而不是切换到其它非受控路径

### Requirement: 分析读取执行模式必须由服务端统一 AI 理解结果决定

系统 MUST 基于统一 AI 理解层输出的结构化结果，由服务端决定当前请求进入解释结果复用、结构化计划执行还是受控 AI 直查执行。前端、企业微信客户端或外部调用方 MAY 提供偏好参数，但系统不得仅依赖客户端直接传入的执行模式标签绕过服务端理解和校验。

#### Scenario: Web 问数请求由服务端决定执行模式
- **WHEN** Web 用户提交一条新的自然语言分析请求
- **THEN** 系统必须先基于统一 AI 理解结果决定进入计划执行、受控直查或阻断路径
- **THEN** 系统不得仅根据客户端传入的 `executionMode` 字段直接进入某种执行模式

#### Scenario: 解释型追问不因客户端标签而误触发重取数
- **WHEN** 用户在已有结果后发起“这说明什么”一类解释型追问
- **THEN** 系统必须优先根据统一 AI 理解结果识别为解释型追问并复用当前结果包
- **THEN** 系统不得因为客户端错误传入模式标签而直接重新执行读取

### Requirement: AI 读取意图必须继续命中 API-first 与受控只读边界

即使分析请求已经由统一 AI 理解层识别为需要受控读取，系统 MUST 继续按官方 API 优先、受控只读 SQL 或受控内部接口兜底的顺序执行。统一 AI 理解层只能决定“需要什么读取意图”，不得绕过既有 API-first 和只读安全边界。

#### Scenario: AI 理解结果命中可用官方 API
- **WHEN** 统一 AI 理解层输出的读取意图可由 CRM 官方 API 满足
- **THEN** 系统必须优先选择官方 API 适配器执行
- **THEN** 系统不得因为 AI 已经完成理解就跳过 API-first 约束

#### Scenario: AI 理解结果进入受控只读兜底
- **WHEN** 官方 API 无法满足统一 AI 理解层输出的读取意图
- **THEN** 系统可以进入受控只读 SQL 或内部只读接口兜底
- **THEN** 系统必须继续记录 `preferredSource`、`matchedAdapter`、`gapReason` 和权限快照

### Requirement: 所有进入受控 SQL / 数据库兜底路径的 CRM 动作必须生成 SQL 审计记录

系统 MUST 为所有进入受控 SQL / 数据库兜底路径的 CRM 读取或写入动作生成 SQL 审计记录。该要求同时适用于分析只读兜底、经营报表查询、身份映射与权限补查、企业微信目录同步写库、登录兜底修复和后续受控数据库动作。系统不得仅记录普通业务审计事件而缺失底层 SQL 审计明细。

#### Scenario: 只读兜底路径同时留下业务审计与 SQL 审计

- **WHEN** 某个 CRM 读取步骤因官方 API 不满足要求而进入受控只读 SQL 兜底路径
- **THEN** 系统必须同时保留业务级审计事件和 SQL 审计记录
- **THEN** SQL 审计中必须能追溯 `databaseRole`、命中表和执行状态

#### Scenario: 数据库写入兜底路径留下 SQL 审计

- **WHEN** 企业微信目录同步或登录兜底修复触发受控数据库写入路径
- **THEN** 系统必须为该写入路径生成 SQL 审计记录
- **THEN** 审计记录必须能区分写入表、操作类型、影响行数和失败原因
