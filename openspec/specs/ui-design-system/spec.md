## Purpose

定义 CRM 智能分析系统的根级 UI 设计系统、综合参考策略、旧原型优先级、可信 AI 结果展示、交互反馈和无障碍要求，确保后续 Web 界面设计与实现既具备企业 AI 工作台气质，又满足分析、合同审核、治理审计和权限边界的业务约束。
## Requirements
### Requirement: 系统必须提供根级设计系统入口

仓库 MUST 在根目录提供 `DESIGN.md`，作为 CRM 智能分析系统后续 UI 视觉、组件、动效、文案语气和设计评审的统一入口。`DESIGN.md` MUST 使用中文正文，并明确功能边界、权限规则、接口字段和验收口径仍以 `specs/001-crm-intelligent-analytics/` 下的主规格为准。

#### Scenario: 协作者查找 UI 设计规范

- **WHEN** 协作者需要实现或评审 Web UI
- **THEN** 仓库必须提供根级 `DESIGN.md` 作为优先读取入口
- **THEN** `DESIGN.md` 必须说明它与一期主规格之间的职责分工

### Requirement: 新视觉方向必须采用企业 AI 工作台定位

`DESIGN.md` MUST 将 CRM 智能分析系统定义为企业 AI 经营指挥舱，而不是传统后台管理界面。设计系统 MUST 综合参考多个成熟产品设计方向，并明确每个方向在本系统中的吸收范围。设计系统 MUST 覆盖自然语言问数、阶段反馈、结构化结果、AI grounded 洞察、合同风险识别、治理审计和导出限制等核心场景的视觉与交互原则。

#### Scenario: 后续新增分析工作台界面

- **WHEN** 后续变更新增或重构 Web 智能分析工作台
- **THEN** 设计输入必须优先体现 AI 问数、执行过程、可信解释和继续追问能力
- **THEN** 不得只按传统菜单、筛选表单和静态表格组织首屏体验

#### Scenario: 后续协作者查看设计参考来源

- **WHEN** 协作者需要理解本系统设计为何不是单一 Cohere 风格
- **THEN** `DESIGN.md` 必须说明 Cohere、Sentry、ClickHouse、Linear、Airtable、Coinbase、IBM、Intercom、Apple 或 Vercel 等参考方向的取舍
- **THEN** `DESIGN.md` 必须给出适合本系统的综合设计结论

### Requirement: 旧原型不得继续作为新视觉强约束

仓库 MUST 明确 `docs/prototype-ui/crm-intelligent-analytics-ui.pen` 和 `docs/prototype-ui/UI规范.md` 是旧版历史参考，不再作为新视觉风格、布局结构、卡片形态、尺寸或一级导航的强约束。后续需要可读尺寸原型时，MUST 基于 `DESIGN.md` 重新生成新的 `.pen`，不得以旧原型小修小补替代新版设计。

#### Scenario: 后续 UI 方案与旧 pen 不一致

- **WHEN** 后续 UI 方案基于 `DESIGN.md` 重设导航、首屏结构、核心卡片或结果详情
- **THEN** 不得因旧 `.pen` 原型不一致而判定方案无效
- **THEN** 若需要原型交付，必须输出与新设计系统一致的新 `.pen`

### Requirement: 设计系统必须约束可信 AI 结果呈现

`DESIGN.md` MUST 要求分析结果、合同审核结果和治理审计结果展示事实来源、权限范围、筛选条件、数据更新时间、口径说明、执行状态和审计状态。AI 洞察 MUST 与当前受控结果绑定，不得在视觉或文案上暗示未验证事实。

#### Scenario: 展示一次分析结果

- **WHEN** Web 页面展示一次 AI 分析结果
- **THEN** 页面必须能够呈现摘要、关键指标、主视图、明细、口径说明、权限范围和审计状态
- **THEN** AI 洞察必须被标识为基于当前结果事实生成

### Requirement: 设计系统必须覆盖交互反馈与无障碍要求

`DESIGN.md` MUST 定义关键交互状态和无障碍要求，至少覆盖悬浮、点击、聚焦、禁用、加载、成功、失败、空状态、无权限、超时、降级和风险提示。颜色不得作为唯一状态区分方式，文字对比度 MUST 满足可读性要求。

#### Scenario: 用户触发导出但被限制

- **WHEN** 用户触发导出但因权限、次数或行数限制被阻断
- **THEN** 界面必须展示清晰的阻断原因和下一步建议
- **THEN** 阻断状态必须通过颜色以外的文本或图标同步表达

### Requirement: AppShell 内容页必须取消重复内容标题栏

系统 MUST 在保留 `AppShell` 全局顶部品牌栏和左侧导航的前提下，取消业务页面主内容区顶部的独立内容标题栏。任何进入 `AppShell` 主内容区的业务页面，首屏 MUST 直接进入首个业务面板或首个结果模块，不得再在其上方渲染只承载页面标题、简短说明或页级操作的独立 `hero` / `PageHeaderCompact` 区域。

#### Scenario: 用户进入分析工作台时直接看到首个业务面板

- **WHEN** 用户进入 Web 智能分析工作台
- **THEN** 页面首屏必须直接显示问数输入面板，而不是先显示一条独立的内容标题栏
- **THEN** 全局顶部品牌栏和左侧导航必须继续保留

#### Scenario: 用户进入治理或审计页面时不再出现重复页头

- **WHEN** 用户进入治理策略、查询模板管理、连接策略管理或审计中心
- **THEN** 页面首屏必须直接显示首个表单面板、筛选面板或标签页业务区
- **THEN** 不得在首个业务区上方再渲染一层单独的页面标题区域

### Requirement: 页头上下文信息不得以迁移版形式保留在内容区

系统 MUST 不再把原本位于内容标题栏中的页面标题、说明文案、状态标签、查询 ID 和同类上下文元素迁移到内容区作为替身继续展示。只有确有业务必要的操作按钮 MAY 保留，并且 MUST 附着到对应业务模块本身，不得重新组合成新的页头区。

#### Scenario: 分析工作台删除页头说明与页头状态标签

- **WHEN** 用户进入 Web 智能分析工作台
- **THEN** 页面不得在问数输入区顶部继续显示原页头标题、副文案、`Web Console` 或服务状态胶囊
- **THEN** 工作台只保留业务区本身原有的模块标题和必要操作

#### Scenario: 治理页保留必要按钮但不保留页头替身

- **WHEN** 用户进入治理策略、查询模板管理、连接策略管理或审计中心
- **THEN** 页面可以保留保存、创建、刷新等必要按钮
- **THEN** 页面不得继续显示“治理策略”“查询模板管理”“连接策略管理”“审计中心”等页头式标题、副文案或摘要胶囊的迁移版

### Requirement: Web 页面必须以中文业务标签展示内部代码

系统 MUST 在面向用户的 Web 页面中将后端返回的状态码、风险等级、审计事件类型、入口场景、目标工作流、fallback 原因、执行模式、数据来源、动作类型、模板状态、服务状态和视图类型转换为中文业务标签后展示。页面 MUST NOT 直接把全大写枚举、英文状态等级、kebab-case 原因码或内部工作流标识作为主要展示文案。

#### Scenario: 审计中心展示 AI 入口与 fallback 统计

- **WHEN** 审计接口返回 `WECOM_IDLE_MESSAGE`、`WECOM_DAILY_REPORT_ENTRY`、`active-conversation-flow-continue`、`LOW`、`critical` 或 `info`
- **THEN** 审计中心必须展示对应中文业务标签
- **THEN** 页面不得把这些内部代码作为主要可见文案直接展示给用户

#### Scenario: 分析结果详情展示执行状态与可执行动作

- **WHEN** 分析结果详情接口返回 `RETURNED`、`BLOCKED`、`PLAN_EXECUTION`、`GUARDED_READONLY_SQL` 或内部动作类型
- **THEN** 分析结果详情页必须展示中文状态、中文执行模式、中文执行来源和中文动作名称
- **THEN** 页面必须保留原有导出、追问和详情查看行为不变

#### Scenario: 治理页面展示策略与模板状态

- **WHEN** 治理策略或查询模板接口返回 `ACTIVE`、`INACTIVE`、`RANKING_TABLE` 或其他模板视图类型
- **THEN** 治理策略页和查询模板页必须展示中文状态和中文视图类型
- **THEN** 页面必须保留原有保存策略、创建模板和刷新数据行为不变

### Requirement: 前端必须提供统一业务代码展示出口

系统 MUST 在前端提供统一业务代码展示模块或同等能力，供页面和组件复用。该模块 MUST 按业务分类提供格式化函数，避免不同上下文的同名代码被错误翻译。页面 MUST 优先调用该统一出口，不得在多个页面重复维护同一枚举的中文映射。

#### Scenario: 页面需要显示风险等级

- **WHEN** 任一页面需要显示 `LOW`、`MEDIUM` 或 `HIGH`
- **THEN** 页面必须通过统一风险等级格式化出口获取“低风险”“中风险”“高风险”等中文标签
- **THEN** 页面不得在局部 template 中直接输出原始风险等级代码

#### Scenario: 页面需要显示审计入口或工作流

- **WHEN** 审计中心、分析详情或其他页面需要显示入口场景、目标工作流或 fallback 原因
- **THEN** 页面必须通过统一分类格式化出口获取中文标签
- **THEN** 同一代码在相同分类下的中文标签必须保持一致

### Requirement: 未知业务代码必须保留可诊断中文兜底

系统 MUST 对未知、未登记或新出现的业务代码提供中文兜底文案，并在兜底文案中保留原始代码。系统 MUST NOT 因未知代码而空白展示、静默隐藏或只显示英文原值。

#### Scenario: 后端新增尚未登记的审计事件类型

- **WHEN** 审计接口返回前端字典尚未登记的事件类型 `NEW_INTERNAL_EVENT`
- **THEN** 页面必须显示“未知事件类型（NEW_INTERNAL_EVENT）”或同等中文兜底
- **THEN** 页面不得只显示 `NEW_INTERNAL_EVENT`

#### Scenario: 可选字段为空

- **WHEN** 入口场景、工作流、fallback 原因或状态字段为空
- **THEN** 页面可以继续显示 `--` 或当前页面既有空状态中文文案
- **THEN** 页面不得把空值误判为未知代码

### Requirement: 页面级回归必须覆盖全部已注册路由页面

系统 MUST 以 `frontend/src/router/index.ts` 中注册的页面为检查清单，验证登录页、智能分析工作台、分析结果详情、合同审核工作台、合同审核详情、治理策略、查询模板、连接策略和审计中心不会裸露典型内部代码。相关组件测试和 E2E 测试 MUST 使用后端原始代码作为输入，并断言浏览器最终展示中文标签。

#### Scenario: 执行页面级本地化巡检

- **WHEN** 开发者完成业务代码中文化修复
- **THEN** 必须逐页检查所有已注册路由页面的用户可见文案
- **THEN** 必须确认截图样例中的 raw code 不再裸露在页面主体、表格、标签、卡片或提示中

#### Scenario: 后续新增页面或新增枚举

- **WHEN** 后续新增 Web 页面或新增需要展示的业务枚举
- **THEN** 必须同步扩展统一业务代码展示模块和测试样例
- **THEN** 不得通过页面内临时 `switch` 或直接插值绕过统一展示出口

### Requirement: 前端必须统一采用 Element Plus 作为基础 UI 框架

系统 MUST 在后续 Web 前端重构中统一采用 `Element Plus` 作为 Vue 3 基础 UI 组件库，并配套使用 `@element-plus/icons-vue` 作为默认图标来源。Element Plus MUST 用于承接按钮、表单、输入框、选择器、表格、标签、提示、抽屉、上传、加载、空状态、消息反馈等通用交互组件，避免继续在各页面重复手写基础控件。

#### Scenario: 协作者开始前端 UI 框架重构

- **WHEN** 协作者开始实施前端 UI 框架重构
- **THEN** 必须安装并接入 `element-plus`
- **THEN** 必须安装并接入 `@element-plus/icons-vue`
- **THEN** 不得再引入第二套桌面端 Vue UI 组件库承担同类基础组件职责

#### Scenario: 页面需要新增通用交互控件

- **WHEN** 页面需要新增按钮、表单、输入框、表格、标签、抽屉、上传、加载或空状态
- **THEN** 应优先使用 Element Plus 对应组件或项目封装后的 Element Plus 组件
- **THEN** 不得无理由新增与 Element Plus 能力重复的手写基础控件

### Requirement: Element Plus 主题必须服从 DESIGN.md

系统 MUST 将 Element Plus 作为基础组件能力来源，而不是视觉风格来源。Element Plus 的颜色、圆角、字体、边框、焦点、禁用、加载和状态表达 MUST 通过主题变量、项目级样式覆盖或项目封装组件与根级 `DESIGN.md` 对齐。页面 MUST 保持浅色主导、克制可信、统一 `12px` 控件圆角、标签不换行、清晰加载反馈和响应式优先的视觉方向。

#### Scenario: 协作者接入 Element Plus 样式

- **WHEN** 协作者导入 Element Plus 基础样式
- **THEN** 必须同时提供项目级主题覆盖
- **THEN** 主色、状态色、圆角、边框、字体和焦点外环必须映射到 `DESIGN.md` 的设计令牌
- **THEN** 页面不得呈现 Element Plus 默认蓝白后台风格作为最终交付效果

#### Scenario: 协作者重构已有页面

- **WHEN** 协作者使用 Element Plus 重构登录、分析、合同、治理或审计页面
- **THEN** 页面骨架、信息层级、权限边界、风险表达和交互反馈仍必须遵循 `DESIGN.md`
- **THEN** 不得因为使用 UI 框架而弱化可信结果说明、审计留痕、权限范围或响应式可读性

### Requirement: 图标语义必须由 Element Plus 图标体系补齐

系统 MUST 使用 `@element-plus/icons-vue` 补齐当前缺失的图标语义。关键导航、主操作、状态、风险、上传、导出、刷新、筛选、搜索、返回、关闭、展开收起、审计事件类型和合同风险等级 MUST 有稳定图标表达，并与中文文案共同传达状态；颜色不得作为唯一状态信号。

#### Scenario: 协作者重构应用壳层导航

- **WHEN** 协作者使用 Element Plus 重构左侧导航或顶部栏
- **THEN** 导航项必须使用稳定图标表达业务入口
- **THEN** 不得继续使用文字首字、临时字符或无语义装饰作为导航图标
- **THEN** 当前激活项必须同时通过图标、文字和视觉状态表达

#### Scenario: 页面展示状态或风险

- **WHEN** 页面展示成功、失败、警告、风险、阻断、加载、上传、导出或审计事件状态
- **THEN** 状态必须同时包含图标和中文说明
- **THEN** 图标语义必须与状态含义一致
- **THEN** 不得只依赖颜色区分状态等级

### Requirement: Element Plus 重构必须保持现有业务行为不变

系统 MUST 将 Element Plus 重构限定在前端 UI 表现层、组件标签、样式、图标和局部展示状态范围内。实施时 MUST 保留现有路由、`v-model`、事件绑定、store/service 调用、登录流程、分析请求、结果导出、追问、合同上传、风险展开、治理保存、模板创建、审计刷新和筛选行为。除非另立功能变更，系统 MUST NOT 修改后端 API、数据库、OpenAPI 契约、权限模型或企业微信能力边界。

#### Scenario: 协作者替换页面按钮与表单

- **WHEN** 协作者将原生按钮、输入框或表单替换为 Element Plus 组件
- **THEN** 原有 `@click`、`v-model`、`:disabled`、加载状态和错误提示必须保持等价
- **THEN** 已有单元测试和 E2E 覆盖的用户行为不得发生回归

#### Scenario: 协作者替换上传、抽屉、表格或消息组件

- **WHEN** 协作者将上传、抽屉、表格或消息反馈替换为 Element Plus 组件
- **THEN** 文件选择、拖拽上传、导出、分页、空状态、错误状态和关闭操作必须保持原有业务调用
- **THEN** 如果 Element Plus 组件事件模型不同，必须在页面层适配，不得改变 service 或后端契约

### Requirement: Element Plus 重构必须采用渐进式迁移与验证

系统 MUST 按阶段实施 Element Plus 重构，先建立依赖、主题覆盖和图标映射，再逐页替换登录、应用壳层、分析链路、合同审核、治理配置和审计中心。每个阶段 MUST 运行对应验证，最终 MUST 运行前端 lint、单元测试、E2E 测试和构建。

#### Scenario: 协作者完成基础接入阶段

- **WHEN** Element Plus 依赖、主题覆盖和图标映射完成
- **THEN** 必须运行 `pnpm --dir frontend lint`
- **THEN** 必须运行 `pnpm --dir frontend build`
- **THEN** 页面不得出现全局样式污染导致的登录页、工作台或表格布局明显异常

#### Scenario: 协作者完成页面替换阶段

- **WHEN** 任一页面完成 Element Plus 替换
- **THEN** 必须核对该页面所有请求型按钮都有加载反馈
- **THEN** 必须核对该页面关键图标、状态说明、空状态和失败反馈完整
- **THEN** 必须运行覆盖该页面的单元测试或 E2E 测试

#### Scenario: 协作者完成全部前端重构

- **WHEN** 所有目标页面完成 Element Plus 重构
- **THEN** 必须运行 `pnpm --dir frontend lint`
- **THEN** 必须运行 `pnpm --dir frontend test:unit`
- **THEN** 必须运行 `pnpm --dir frontend test:e2e`
- **THEN** 必须运行 `pnpm --dir frontend build`

### Requirement: 设计系统必须定义业务视觉丰富度底线

`DESIGN.md` MUST 明确 CRM 智能分析系统的页面不能只依赖白色卡片、灰色文字和单一品牌色完成视觉表达。每个一级 Web 页面 MUST 至少具备一个业务视觉锚点，例如模块徽标、页面状态条、核心指标图形、业务插画、进度图形或数据可视化，并且该视觉锚点 MUST 服务业务识别、状态理解或数据比较。

#### Scenario: 协作者设计一级页面首屏

- **WHEN** 协作者为智能分析、经营报表、合同审核、治理审计、查询模板、连接策略或 AI 配置设计一级页面
- **THEN** 页面首屏必须能通过主内容区识别当前业务模块
- **THEN** 页面不得只由普通白色卡片、表单、表格和灰色文字组成
- **THEN** 页面必须至少包含一个业务视觉锚点

#### Scenario: 页面展示空状态或无权限状态

- **WHEN** 页面展示空状态、加载态、失败态、无权限态或降级态
- **THEN** 状态必须包含中文原因说明和下一步动作
- **THEN** 状态必须配合状态图标、模块徽标或轻量业务插画
- **THEN** 页面不得只显示一行“暂无数据”或“失败”

### Requirement: 设计系统必须提供受控业务辅助色和图表色板

`DESIGN.md` MUST 定义业务辅助色和图表色板，用于模块识别、图表序列、等级标签和局部业务图形。业务辅助色 MUST 不替代品牌主色，且同一页面默认 MUST 受控使用模块辅助色、品牌主色和必要状态色，避免单色粗糙或多色杂乱。

#### Scenario: 协作者为模块页面选择颜色

- **WHEN** 协作者为某个业务页面选择主视觉色
- **THEN** 必须优先从 `DESIGN.md` 的业务辅助色中选择模块辅助色
- **THEN** 同一页面默认不得同时大面积使用多个模块辅助色
- **THEN** 风险、成功、降级、离线等状态仍必须使用统一语义状态色

#### Scenario: 协作者设计图表

- **WHEN** 页面展示趋势、排行、占比、风险等级或经营异常图表
- **THEN** 图表必须优先使用 `DESIGN.md` 定义的图表色板
- **THEN** 图表状态色必须与标签、摘要卡和表格状态保持一致
- **THEN** 图表不得长期只使用单一品牌渐变表达所有数据系列

### Requirement: 设计系统必须定义图标图形与业务插画规则

`DESIGN.md` MUST 定义图标使用分层、图标尺寸、业务图形类型和插画使用规则。前端 MUST 使用 `@element-plus/icons-vue` 和项目级封装表达导航、操作、状态、业务对象和等级，不得使用 emoji、首字母、无语义字符或来源不明图片替代关键图标。

#### Scenario: 协作者补齐业务对象图标

- **WHEN** 页面展示客户、负责人、合同、模板、策略、连接、审计事件、数据表、图表或企业微信来源
- **THEN** 页面必须通过图标、状态点、等级色或对象标签中的至少一种方式提供非文字识别线索
- **THEN** 图标语义必须与业务对象一致
- **THEN** 页面不得使用临时字符或首字母作为业务对象图标

#### Scenario: 协作者添加业务插画

- **WHEN** 页面需要为空状态、加载态、失败态或引导态添加插画
- **THEN** 插画必须使用低饱和模块色和中性色
- **THEN** 插画必须与当前业务模块相关
- **THEN** 插画不得替代真实数据展示

### Requirement: 设计系统必须定义页面级视觉改造基线

`DESIGN.md` MUST 为智能分析工作台、经营报表、查询模板、权限中心、连接策略、合同审核、审计中心和 AI 配置提供页面级视觉改造基线。每个页面基线 MUST 说明主信息结构、应强化的图形元素、必须保留的业务边界和禁止退化的粗糙形态。

#### Scenario: 协作者改造智能分析工作台

- **WHEN** 协作者改造智能分析工作台
- **THEN** 页面必须形成 `AI 问数 + 经营概览 + 可复用资产 + 可信边界` 的组合
- **THEN** 未提问时不得留下大面积空白
- **THEN** 结果出现后必须形成摘要、关键发现、图表、表格、口径边界和后续动作的稳定阅读路径

#### Scenario: 协作者改造审计中心

- **WHEN** 协作者改造审计中心
- **THEN** 页面必须表现为事件复核工作台
- **THEN** 首屏必须展示查询量、拦截量、导出量、敏感命中或待复核高风险等摘要信息
- **THEN** 审计事件列表必须展示事件类型图标、风险等级、行为人、对象、时间、处理状态和复核入口

### Requirement: 设计系统必须提供设计评审检查清单

`DESIGN.md` MUST 提供设计稿、原型或前端页面提交评审前的检查清单。检查清单 MUST 覆盖业务识别、主次关系、色彩控制、图标图形、数据表达、文字层级、交互状态、响应式稳定、低频降噪和业务边界。

#### Scenario: 协作者提交页面视觉改造评审

- **WHEN** 协作者提交任一页面视觉改造方案
- **THEN** 必须按 `DESIGN.md` 的设计评审检查清单自查
- **THEN** 必须确认页面具备业务识别、可读文字层级、交互状态和响应式稳定性
- **THEN** 不得把仅通过单张桌面截图看起来可用的页面视为完成

### Requirement: 分析工作台必须提供左侧可调常用查询资产区

Web 智能分析工作台 MUST 在桌面端提供左侧常驻常用查询资产区。该区域 MUST 遵循 `DESIGN.md` 的浅色主导、可信结果呈现、统一控件圆角、标签不换行和响应式要求。该区域 MUST 支持默认宽度、点击展开、点击收缩和拖拽调整宽度；宽度变化不得破坏右侧查询、结果表格、图表和操作按钮布局。

#### Scenario: 桌面端展示左侧常用查询资产区
- **WHEN** 用户在桌面端打开智能分析工作台
- **THEN** 页面必须展示左侧常用查询资产区
- **THEN** 右侧必须继续展示查询输入、结果和报告区域
- **THEN** 页面不得将常用查询默认隐藏在抽屉中

#### Scenario: 用户拖拽左侧资产区宽度
- **WHEN** 用户拖拽左侧常用查询资产区边界
- **THEN** 页面必须在受控最小宽度和最大宽度内调整布局
- **THEN** 右侧内容不得出现文字重叠、按钮挤压或表格不可读

### Requirement: 常用查询资产区必须支持标签页、筛选和大列表反馈

常用查询资产区 MUST 提供“我的模板”和“其它模板”标签页。两个标签页 MUST 支持关键词、标签和必要用户筛选。列表 MUST 支持虚拟滚动或等效大列表方案，并为加载中、加载失败、空状态、无权限、复制成功和执行失败提供中文反馈。

#### Scenario: 用户筛选模板
- **WHEN** 用户在左侧资产区输入关键词、选择分类或选择标签
- **THEN** “查询”按钮必须紧跟筛选控件
- **THEN** 用户按 Enter 必须执行同一次查询
- **THEN** 当前分析结果不得因筛选模板而被清空

#### Scenario: 模板列表为空
- **WHEN** 当前筛选条件下没有模板
- **THEN** 页面必须展示中文空状态
- **THEN** 空状态必须说明当前条件下没有匹配模板
- **THEN** 页面不得只显示“暂无数据”

#### Scenario: 模板列表加载失败
- **WHEN** 模板列表加载失败
- **THEN** 页面必须展示中文失败原因和重试入口
- **THEN** 页面不得只在控制台输出错误

### Requirement: 保存模板与复制模板必须提供明确交互反馈

保存为模板弹窗 MUST 包含模板名称、模板说明、标签、AI 推荐标签提示和保存 / 取消动作。标签控件 MUST 支持选择已有标签和自由输入新标签。保存、复制、校验失败和权限阻断 MUST 通过中文消息反馈，右上角瞬时提示默认 3 秒后自动消失，除非需要持续确认。

#### Scenario: 用户保存模板成功
- **WHEN** 用户填写保存模板表单并保存成功
- **THEN** 页面必须提示“模板已保存到我的模板”或等价中文文案
- **THEN** 提示必须自动消失或提供关闭入口

#### Scenario: 用户自由输入新标签
- **WHEN** 用户在标签控件中输入新的中文标签
- **THEN** 页面必须允许该标签作为模板标签保存
- **THEN** 标签不得换行挤压按钮或遮挡表单字段

#### Scenario: 保存模板失败
- **WHEN** 模板因无法复现 SQL、校验失败或权限不足保存失败
- **THEN** 页面必须先说明发生了什么
- **THEN** 页面必须给出下一步建议
- **THEN** 页面不得直接展示内部异常、SQL 片段或堆栈

### Requirement: 最近查询必须继续作为抽屉能力保留

智能分析工作台 MUST 保留最近查询抽屉入口。常用查询左侧常驻后，最近查询 MUST NOT 被强行合并进左侧常驻区。最近查询抽屉的再次运行、加载、失败、空状态和权限反馈 MUST 继续保持可用。

#### Scenario: 用户打开最近查询
- **WHEN** 用户点击最近查询入口
- **THEN** 页面必须打开最近查询抽屉
- **THEN** 最近查询列表必须展示个人历史记录
- **THEN** 左侧常用查询资产区不得被替换为最近查询

#### Scenario: 用户从最近查询再次运行
- **WHEN** 用户在最近查询抽屉中点击再次运行
- **THEN** 页面必须关闭或保持抽屉状态不影响主结果展示
- **THEN** 系统必须给出运行中、成功或失败反馈
