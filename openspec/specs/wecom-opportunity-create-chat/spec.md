# wecom-opportunity-create-chat Specification

## Purpose
定义企业微信中受控新建商机能力的入口识别、对象唯一化、字段补齐、产品解析与确认执行要求，确保商机创建在受控边界内完成。
## Requirements
### Requirement: 企业微信机器人必须识别受控新建商机意图

系统 MUST 将“新增商机”“新建商机”“帮我建一个商机”等企业微信消息识别为受控新建商机意图，并进入专用的商机创建链路，而不是继续走问数分析或统一写入拒绝分支。

#### Scenario: 命中新建商机意图后进入商机创建链路

- **WHEN** 已完成企业微信身份映射的用户在机器人会话中发送“帮我新建一个商机”
- **THEN** 系统进入受控新建商机状态并返回商机字段采集提示
- **THEN** 系统不得把该消息继续当作分析问数处理

### Requirement: 商机创建前必须唯一确认客户并补齐必填字段

系统 MUST 在商机创建链路中补齐“项目名称、最终客户、线索编号、关联产品、预计有效收入、预计签单日期、被续签合同号、代理商全称、项目现状及关键点、售前”等必填字段。若用户提供的是客户名称而非客户 ID，系统 MUST 先通过受控客户查询能力将客户唯一化；未唯一命中前不得进入最终确认。

#### Scenario: 最终客户唯一命中后继续补齐商机字段

- **WHEN** 用户在新建商机会话中提供客户名称且系统唯一命中一个现有客户
- **THEN** 系统将该客户 ID 写入当前商机草稿
- **THEN** 系统继续追问尚未补齐的商机必填字段

#### Scenario: 最终客户存在多个候选

- **WHEN** 用户提供的客户名称命中多个候选客户
- **THEN** 系统返回候选列表并要求用户回复准确客户名称或候选序号
- **THEN** 在客户未唯一确认前，系统不得发起商机创建请求

#### Scenario: 初始消息缺少部分商机必填字段

- **WHEN** 用户只提供了项目名称和客户名称，未提供线索编号、关联产品或预计签单日期等必填项
- **THEN** 系统保留已识别的项目名称和客户信息
- **THEN** 系统继续按缺失字段列表逐项追问，不得丢失已采集内容

### Requirement: 商机创建链路必须把关联产品解析为至少一条 productId

系统 MUST 在执行商机创建前，将用户提供的关联产品信息解析成至少一条有效的 `productId`。系统 MUST 支持配置化产品别名映射，也 MUST 支持用户显式提供产品 ID 作为兜底。若无法解析出至少一条 `productId`，系统不得继续创建。

#### Scenario: 通过产品别名映射拿到 productId

- **WHEN** 用户提供的关联产品名称命中受控配置中的产品别名映射
- **THEN** 系统将该映射结果写入商机草稿中的 `productAssets`
- **THEN** 在其它字段齐全后允许进入摘要确认

#### Scenario: 关联产品无法解析

- **WHEN** 用户提供的关联产品名称未命中任何受控别名映射，且也未提供可直接使用的产品 ID
- **THEN** 系统返回无法识别关联产品的明确提示
- **THEN** 系统不得向 CRM 发起商机创建请求

### Requirement: 新建商机执行必须支持确认、取消和失败重试

系统 MUST 在客户、产品和其它必填字段都齐全后生成商机摘要，并等待用户确认、修改或取消。创建失败时 MUST 保留草稿与失败原因，允许用户继续修改字段或直接重试。

#### Scenario: 字段齐全后进入商机摘要确认

- **WHEN** 商机草稿已经补齐全部必填字段且客户、产品都已解析完成
- **THEN** 系统返回包含项目名称、最终客户、关联产品、预计有效收入、预计签单日期、负责人和所属部门的摘要
- **THEN** 系统提示用户回复“确认”“修改”或“取消”

#### Scenario: 用户确认后创建成功

- **WHEN** 用户已确认商机摘要且 CRM 官方创建接口返回成功
- **THEN** 系统返回商机 ID、商机标题、客户信息、负责人、创建时间和成功提示
- **THEN** 系统记录商机创建成功审计事件并清理当前待创建状态

#### Scenario: 创建失败后保留草稿

- **WHEN** 用户已确认商机摘要但 CRM 官方创建接口返回失败
- **THEN** 系统将当前商机草稿标记为失败并保留已采集字段与失败原因
- **THEN** 系统提示用户可以回复“修改”继续编辑，或回复“重试”再次发起创建

### Requirement: 新建商机入口与续填回复必须统一先经过 AI 理解层
系统 MUST 将企业微信中的新增商机入口识别、字段续填、客户确认、产品补充、确认、修改、取消和重试回复统一接入 `ai-entry-intent-orchestration`。统一 AI 理解层 MUST 先识别该输入属于创建入口、字段补充、对象澄清、确认提交、修改草稿、取消创建还是继续澄清；旧的入口关键词、对象查询正则和确认词表仅可作为 fallback，不得继续承担主理解职责。

#### Scenario: 非固定关键词表达仍可进入新增商机链路
- **WHEN** 用户发送“帮我录一个新项目机会”或语义等价的自然语言表达
- **THEN** 系统必须先通过统一 AI 理解层识别该输入属于新增商机意图
- **THEN** 系统不得因为未命中固定关键词就误入分析问数或帮助链路

#### Scenario: 续填回复先由 AI 识别为商机草稿更新
- **WHEN** 用户在商机创建流程中继续发送客户、产品、预计金额、日期或项目现状等补充信息
- **THEN** 系统必须先通过统一 AI 理解层识别该回复属于草稿字段更新或对象澄清
- **THEN** 系统不得因局部关键词命中而误切到其它任务

#### Scenario: 确认与取消回复先由 AI 识别
- **WHEN** 用户在商机摘要确认阶段回复“ok create it”或“先停一下”
- **THEN** 系统必须先通过统一 AI 理解层识别该回复属于确认还是取消
- **THEN** 系统不得继续仅依赖固定确认词或取消词表推进状态

### Requirement: 新增商机入口必须消费 capability pack 的 idle / active task 结果
企业微信新增商机入口 MUST 消费统一 AI capability pack 输出的空闲态入口结果与活跃任务回复结果进入既有商机创建链路。只要 idle entry pack 已经将消息识别为 `CRM_CREATE_OPPORTUNITY`，系统 MUST 直接进入受控新建商机流程，不得因为无关可选字段缺失而回落帮助兜底；进入商机创建流程后的取消、切换、继续和确认语义 MUST 优先由 active task reply pack 识别。

#### Scenario: 空闲态新增商机入口直接进入创建流程
- **WHEN** 企业微信空闲会话收到 `新增商机`、`帮我建一个商机` 或语义等价表达，且 idle entry pack 返回 `CRM_CREATE_OPPORTUNITY`
- **THEN** 系统必须进入既有受控新建商机字段采集链路
- **THEN** 系统不得因为 capability pack 未返回 `helpScene` 或其它无关字段而进入帮助兜底

#### Scenario: 商机创建中的取消与切换优先由 active task pack 识别
- **WHEN** 用户已经处于商机创建流程中，并回复“先停一下”或“切到生成日报”
- **THEN** 系统必须优先消费 active task reply pack 的语义结果决定取消当前创建或切换任务
- **THEN** 系统不得继续仅依赖旧确认词表或局部关键词推进状态

### Requirement: 企业微信新增商机链路必须校验 wecom.opportunity.create
系统 MUST 在企业微信新增商机入口、字段补齐后的确认执行、失败重试以及对应受控商机创建 HTTP 入口上校验 `wecom.opportunity.create`。用户即使已被允许使用企业微信机器人，也不得因为拥有其它 `wecom.*` 动作而获得新增商机资格。

#### Scenario: 缺少 wecom.opportunity.create 的用户不能进入新增商机链路
- **WHEN** 某用户已通过企业微信通道准入，但未被授予 `wecom.opportunity.create`，并发送“新增商机”
- **THEN** 系统必须拒绝进入新增商机字段采集流程
- **THEN** 不得为其创建商机草稿或进入确认阶段

#### Scenario: 直接调用商机创建接口也必须命中 wecom.opportunity.create
- **WHEN** 某个已登录用户未被授予 `wecom.opportunity.create`，但直接调用受控商机创建接口
- **THEN** 系统必须拒绝该请求
- **THEN** 不得因为该请求绕过了企业微信聊天入口就继续执行

### Requirement: 新增商机确认执行前必须再次校验实时权限
系统 MUST 在商机草稿已经补齐并等待确认时，于真正调用 CRM 创建接口前再次校验当前用户的 `wecom.opportunity.create`。若权限已被撤销，系统 MUST 中止创建，并保留草稿和失败原因。

#### Scenario: 草稿阶段权限撤销后最终创建被阻断
- **WHEN** 某用户已经生成商机草稿，但在确认前被撤销 `wecom.opportunity.create`
- **THEN** 系统必须拒绝最终商机创建
- **THEN** 系统必须保留草稿状态和权限不足原因，供后续重试或排障
