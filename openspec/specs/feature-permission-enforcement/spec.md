## Requirements
### Requirement: 权限中心中的 30 个权限点必须映射到真实执行点
系统 MUST 为权限中心中所有基础开关、菜单可见性和动作权限建立唯一的运行时语义和真实执行点映射。系统 MUST 保证每个权限点至少命中一个真实页面入口、后端接口或企业微信工作流门槛，不得继续保留“页面可配置但系统不消费”的死配置。

#### Scenario: 管理员查看权限覆盖清单时不存在未映射权限点
- **WHEN** 管理员基于当前权限矩阵核对系统权限点与执行点的对应关系
- **THEN** 系统必须能说明每个权限点对应的前端入口、后端执行点或企业微信工作流
- **THEN** 不得存在长期未绑定任何真实执行点的权限配置项

#### Scenario: 权限中心预览结果与真实执行点保持一致
- **WHEN** 管理员在权限中心预览某个用户的菜单、动作和入口资格
- **THEN** 同一用户访问对应页面、接口或企业微信工作流时必须命中相同的允许或拒绝结果
- **THEN** 不得出现“预览可用但真实接口仍按旧逻辑拒绝”或“预览拒绝但真实接口仍放行”

### Requirement: 系统必须以统一权限矩阵作为功能权限真源
系统 MUST 以角色权限矩阵和统一权限决策结果作为运行时功能权限真源。旧的 `isAdmin`、`policy.enabledRoleIds`、`policy.exportRoleIds`、`user.exportAllowed` 和其它散落判断 MAY 仅在迁移期作为兼容条件存在，但功能是否最终放行 MUST 通过统一权限决策服务判定，不得继续由业务模块各自发明第二套准入逻辑。

#### Scenario: 历史管理员角色未配置动作时不得仅凭管理员标识放行
- **WHEN** 某个用户仅因为历史角色名称包含“管理员”而命中 `isAdmin`
- **THEN** 系统不得仅凭该标识放行审计中心、AI 模型治理或治理策略接口
- **THEN** 运行时最终放行仍必须由统一动作权限判定

#### Scenario: 迁移期兼容条件不能覆盖统一权限拒绝结果
- **WHEN** 某个接口同时存在旧的全局策略兼容判断和新的统一权限矩阵判断
- **THEN** 一旦统一权限结果明确拒绝，该接口必须拒绝执行
- **THEN** 旧兼容条件不得反向覆盖统一权限矩阵的拒绝结果

### Requirement: 系统必须区分通道准入与工作流动作准入
系统 MUST 将 Web / 企业微信通道准入与具体工作流动作准入分开判断。`webConsoleEnabled` 和 `wecomBotEligible` 只负责决定用户能否进入对应通道；在通道已准入后，系统 MUST 再按具体动作权限决定用户能否执行分析、创建、写回、日报预览等工作流。系统 MUST 不得把“能进入企业微信机器人”错误等同于“企业微信所有功能都可用”。

#### Scenario: 企业微信用户只具备日报预览动作时不能执行问数
- **WHEN** 某用户通过企业微信通道准入且拥有 `wecom.daily_report.preview`，但未拥有 `wecom.analysis.use`
- **THEN** 系统可以允许该用户进入企业微信机器人并查看日报预览
- **THEN** 系统必须拒绝其发起企业微信问数工作流

#### Scenario: 企业微信用户只具备问数动作时不能查看日报预览
- **WHEN** 某用户通过企业微信通道准入且拥有 `wecom.analysis.use`，但未拥有 `wecom.daily_report.preview`
- **THEN** 系统可以允许该用户发起企业微信问数工作流
- **THEN** 系统必须拒绝其访问个人或团队日报预览工作流

### Requirement: 系统必须对系统级治理、审计检索和模板可见执行后端权限校验
系统 MUST 对 `governance.policy.manage`、`audit.view`、`ai_profile.manage`、`template.view` 和 `template.manage` 执行后端真实校验，而不是只依赖前端菜单显隐、路由守卫或管理员硬编码。审计中心、AI 模型治理、治理策略和模板可见列表必须按统一权限矩阵执行放行或拒绝。

#### Scenario: 缺少 audit.view 的用户直接调用审计接口被拒绝
- **WHEN** 某个已登录用户未被授予 `audit.view`，但直接调用审计中心接口
- **THEN** 系统必须返回明确的权限不足响应
- **THEN** 不得因为该用户已登录或具有其它治理权限而放行审计数据读取

#### Scenario: 缺少 template.view 的用户不能获取模板可见列表
- **WHEN** 某个用户未被授予 `template.view`，但请求分析页模板列表接口
- **THEN** 系统必须返回空模板结果或明确拒绝
- **THEN** 不得仅依据模板资源本身的 `visibleRoleIds` 放行模板列表

### Requirement: 系统必须对页面级基础访问资格执行前后端一致校验
对于没有独立动作点、但仍代表页面级基础访问资格的菜单权限，系统 MUST 同时控制前端入口和后端页面读接口。`analysis-workbench`、`contract-review`、`permission-center`、`audit-center`、`template-governance` 和 `ai-model-governance` 不得只停留在导航显隐层；对应页面的关键只读接口必须依据统一权限结果放行。

#### Scenario: 合同审核菜单被撤销后本人任务列表也被阻断
- **WHEN** 某个用户被撤销 `contract-review` 菜单资格，且未保留其它合同审核基础访问兼容权限
- **THEN** 前端不得继续展示合同审核导航入口
- **THEN** 后端合同审核工作台列表或本人任务详情接口也必须拒绝该用户访问

#### Scenario: 权限中心菜单被撤销后相关接口也被阻断
- **WHEN** 某个用户失去 `permission-center` 菜单和 `governance.policy.manage` 动作
- **THEN** 前端不得继续展示权限中心入口
- **THEN** 后端权限中心接口必须拒绝该用户读取摘要、矩阵、灰度策略和预览结果

### Requirement: 系统必须为权限拒绝写入可排障审计
系统 MUST 为所有新增接入的权限拒绝写入审计事件，至少记录操作者、角色、权限点、资源类型、资源 ID、通道、拒绝原因和发生时间。权限拒绝审计 MUST 足以解释“为什么该用户看不到页面”“为什么该按钮不可点”“为什么接口被 403”“为什么企业微信工作流被阻断”。

#### Scenario: 企业微信工作流被动作权限拒绝时写入审计
- **WHEN** 企业微信用户已通过通道准入，但在执行具体工作流时因缺少动作权限被拒绝
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 该事件必须记录被拒绝的动作键、目标工作流和拒绝原因

#### Scenario: 系统级治理接口被拒绝时写入审计
- **WHEN** 某个用户调用审计中心、治理策略或 AI 模型治理接口时因缺少动作权限被拒绝
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 审计事件必须能够区分是 `audit.view`、`governance.policy.manage` 还是 `ai_profile.manage` 拒绝

### Requirement: 权限快照缓存必须受统一权限决策和失效事件约束

系统 MAY 在 Web 导航与菜单显隐层复用短时权限快照，以避免每次菜单切换都重复执行完整能力构建；但该快照 MUST 仍然受统一权限矩阵、组织范围和显式失效事件约束。系统 MUST 在登录、退出、401、权限矩阵保存、治理策略保存、组织范围变更等事件发生后立即使旧快照失效，不得仅依赖自然过期继续沿用旧结果。

#### Scenario: 治理保存后下一次导航立即使用新权限结果

- **WHEN** 管理员修改角色权限、治理策略或组织范围并保存成功
- **THEN** 当前页签中的旧权限快照必须立即失效
- **THEN** 用户下一次导航、菜单显隐或页面初始化必须使用新的统一权限结果

#### Scenario: 登录态失效后不得继续复用旧权限快照

- **WHEN** 当前页签收到 401、用户显式退出登录或会话被判定失效
- **THEN** 系统必须立即清除本地权限快照
- **THEN** 不得继续使用旧菜单和旧动作可见性结果

### Requirement: 前端权限快照不得替代后端真实执行校验

系统 MAY 使用前端权限快照优化菜单显隐、路由入口和页面恢复速度，但系统 MUST NOT 使用该快照直接放行敏感动作或真实数据接口。导出、审计检索、合同审核发起、经营报表读取和其它真实业务接口 MUST 继续由后端统一权限决策与范围校验最终放行或拒绝。

#### Scenario: 前端快照允许但后端实时权限拒绝时以后端为准

- **WHEN** 某个用户在页签中仍持有短时允许快照，但其真实权限已被后台撤销
- **AND** 该用户尝试执行导出、审计读取或其它敏感动作
- **THEN** 后端必须依据实时统一权限结果拒绝该请求
- **THEN** 前端不得因为旧快照仍显示按钮就绕过真实阻断

#### Scenario: 菜单秒开不等于真实接口无校验

- **WHEN** 系统为了热切换性能而复用前端权限快照完成菜单跳转
- **THEN** 后续页面真实接口仍必须经过统一权限决策
- **THEN** 系统不得把“页面跳转成功”错误等同于“所有页面数据都应被放行”

### Requirement: 系统必须对 SQL 审计与敏感明细查看执行独立动作校验

系统 MUST 为 SQL 审计查看与完整 SQL / 参数 reveal 建立独立动作权限。`audit.sql.view` MUST 负责控制 SQL 审计 Tab、SQL 摘要列表和详情摘要视图；`audit.sql.view_sensitive` MUST 负责控制完整 SQL、完整参数和必要错误原文的 reveal 行为。系统 MUST 不得因为用户已经具备 `audit.view` 就自动放开 SQL 审计或敏感明细查看能力。

#### Scenario: 仅具备 audit.view 的用户不能查看 SQL 审计

- **WHEN** 某个用户具备 `audit.view`，但未被授予 `audit.sql.view`
- **THEN** 该用户可以进入普通审计中心
- **THEN** 系统必须拒绝其访问 SQL 审计接口或查看 SQL 审计 Tab

#### Scenario: 仅具备 audit.sql.view 的用户不能 reveal 完整 SQL

- **WHEN** 某个用户具备 `audit.sql.view`，但未被授予 `audit.sql.view_sensitive`
- **THEN** 系统必须允许其查看 SQL 摘要列表和详情摘要
- **THEN** 系统必须拒绝其 reveal 完整 SQL 或完整参数

#### Scenario: 具备敏感权限的管理员可以 reveal 完整 SQL

- **WHEN** 最高权限管理员同时具备 `audit.sql.view` 和 `audit.sql.view_sensitive`
- **THEN** 系统必须允许其通过受控 reveal 行为查看完整 SQL 与完整参数
- **THEN** reveal 行为本身必须留下新的审计记录

### Requirement: 统一权限决策必须消费应用超级管理员授权

系统 MUST 在统一权限决策服务中消费应用超级管理员授权。命中应用超级管理员授权的用户 MUST 被视为拥有权限目录中登记的全部菜单和全部动作，权限预览、能力快照、前端导航、后端 `ensureAction` 和真实执行点 MUST 使用同一结果。

#### Scenario: 超级管理员权限预览与真实接口一致

- **WHEN** 管理员在权限中心预览某个应用超级管理员用户
- **THEN** 预览结果必须展示全部菜单、全部动作和全部高风险能力
- **THEN** 该用户真实访问对应页面、接口或企业微信工作流时必须得到相同的允许结果

#### Scenario: 新增权限点自动纳入超级管理员

- **WHEN** 系统在统一权限目录中新增一个菜单或动作权限点
- **THEN** 应用超级管理员必须自动获得该权限点
- **THEN** 不得要求新增功能再单独维护超级管理员名单

### Requirement: 超级管理员授权不得替代安全前置检查

系统 MUST 将应用超级管理员授权限定为功能权限和数据范围覆盖，不得用它替代登录、通道、签名、来源、文件安全、SQL 安全和审计等固定前置检查。

#### Scenario: 企业微信通道关闭时超级管理员不能绕过通道关闭

- **WHEN** 全局企业微信机器人通道被关闭
- **AND** 应用超级管理员通过企业微信发起业务工作流
- **THEN** 系统必须按通道关闭规则拒绝请求
- **THEN** 不得因为其拥有全部动作权限而绕过通道关闭

#### Scenario: SQL 敏感 reveal 仍写入审计

- **WHEN** 应用超级管理员 reveal SQL 审计敏感明细
- **THEN** 系统必须允许其执行 reveal
- **THEN** reveal 行为本身必须继续写入审计事件

### Requirement: 系统必须提供受控查询 SQL 编写权限

系统 MUST 为具备 SQL 能力但不应拥有完整模板治理能力的用户提供独立的最小动作权限。该权限建议命名为 `template.sql.write` 或等价中文权限“查询模板 SQL 编写”。该权限 MUST 只允许用户进入受控 SQL 模板新增、校验、预览和保存流程，不得自动授予完整模板治理、权限治理、审计治理或敏感 SQL reveal 能力。

#### Scenario: 具备 SQL 编写权限的用户新增查询模板
- **WHEN** 用户具备 `template.sql.write` 但不具备 `template.manage`
- **THEN** 系统必须允许其进入受控 SQL 模板新增流程
- **THEN** 系统必须允许其提交只读 SQL 进行校验、预览和保存
- **THEN** 系统不得允许其管理其它用户模板、修改全局治理策略或查看审计敏感明细

#### Scenario: 缺少 SQL 编写权限的用户被阻断
- **WHEN** 用户不具备 `template.sql.write` 且尝试直接调用 SQL 模板新增或保存接口
- **THEN** 系统必须拒绝该请求
- **THEN** 系统必须返回中文权限不足提示
- **THEN** 系统必须记录权限拒绝审计

### Requirement: SQL 编写权限不得绕过模板安全治理

系统 MUST 将 `template.sql.write` 限定为功能入口资格。具备该权限的用户保存、预览或执行模板时，仍 MUST 通过只读 SQL、字段白名单、AST 安全、危险语句拦截、范围治理、结果规模限制和审计留痕。系统 MUST NOT 因用户具备 SQL 编写权限而扩大数据范围或跳过执行期权限校验。

#### Scenario: SQL 用户提交危险语句
- **WHEN** 具备 `template.sql.write` 的用户提交包含更新、删除、建表、改表或其它危险动作的 SQL
- **THEN** 系统必须阻断保存和预览
- **THEN** 系统必须提示该模板 SQL 未通过安全校验

#### Scenario: SQL 用户预览模板仍按当前权限收口
- **WHEN** 具备 `template.sql.write` 的用户预览一条合法只读模板
- **THEN** 系统必须按该用户当前实时权限范围执行预览
- **THEN** 预览结果不得返回当前用户范围外的数据

### Requirement: 超级管理员必须自动获得新增 SQL 编写权限

系统 MUST 将 `template.sql.write` 或等价权限纳入统一权限目录。命中应用超级管理员授权的用户 MUST 自动获得该权限，但仍不得绕过 SQL 安全、范围治理和审计留痕。

#### Scenario: 应用超级管理员预览 SQL 编写权限
- **WHEN** 管理员在权限中心预览应用超级管理员用户
- **THEN** 预览结果必须包含查询模板 SQL 编写能力
- **THEN** 真实接口调用必须与预览结果保持一致

#### Scenario: 超级管理员提交 SQL 仍需校验
- **WHEN** 应用超级管理员提交查询模板 SQL
- **THEN** 系统必须继续执行只读、白名单、AST 和范围治理校验
- **THEN** 系统不得因为超级管理员身份跳过固定安全前置检查
