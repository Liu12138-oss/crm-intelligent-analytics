## Purpose
定义统一 AI 理解层路由后的受控分析执行边界，确保系统可以在不突破权限、白名单、API-first 与审计约束的前提下，支持计划执行与受控 AI 直查两类只读分析模式。
## Requirements
### Requirement: 系统必须支持受控 AI 直查执行模式

系统 MUST 在受控分析编排中支持“结构化计划执行”和“受控 AI 直查执行”两类只读执行模式，并将不固定的自然语言问数默认收敛到受控 AI 直查主链。自由问数文本 MUST 先经过统一 AI 理解层，再由服务端结合当前场景、知识层命中情况和风险边界决定进入 `GUARDED_DIRECT_QUERY`、`PLAN_EXECUTION`、解释复用、补问或阻断路径；固定功能主题入口一旦被统一 AI 理解层明确命中，系统 MUST 继续走固定程序流，不得误降级为自由问数直查。受控 AI 直查执行中，模型 MUST 先生成 1 到 3 个受控查询任务，任务 MAY 表达只读 SQL、已注册只读 API / 查询工具或受控内部读取意图；系统 MUST 在实际执行前完成语句类型校验、权限范围注入、表字段白名单、结果规模限制、超时限制和审计快照记录。系统 MUST 阻止 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`、schema 探测、未声明写入工具和未声明外部请求。

#### Scenario: 自由自然语言问数默认进入受控直查主链
- **WHEN** Web 或企业微信用户提交一条不属于固定功能主题入口的自然语言问数
- **THEN** 系统必须优先尝试进入受控 AI 直查执行模式
- **THEN** 系统不得默认先把该问题压缩成窄枚举结构化计划后再决定是否允许直查

#### Scenario: 固定功能主题入口不得误降级为自由问数直查
- **WHEN** 企业微信或 Web 输入已经被统一 AI 理解层明确识别为跟进、创建、日报、项目查询或其它固定功能主题入口
- **THEN** 系统必须继续进入对应固定程序流
- **THEN** 系统不得因为该输入同时包含查询词或数据词就把它当作自由问数直查执行

#### Scenario: AI 直接生成受控查询任务后仍需通过安全栈执行
- **WHEN** 用户提交一个允许通过只读 SQL 或已注册只读工具回答的经营分析问题
- **THEN** 系统可以接受模型生成的受控查询任务并从中提取单条或多条只读读取步骤
- **THEN** 系统必须在执行前完成语句类型校验、权限范围注入、白名单校验和行数 / 超时限制
- **THEN** 系统只可将受控结果交给后续 grounded 总结与交付链路

#### Scenario: AI 尝试执行危险语句时被阻断
- **WHEN** 模型生成包含删除、更新、建表、改表或其它未声明危险动作的 SQL 或工具调用
- **THEN** 系统必须立即阻断执行
- **THEN** 系统必须记录阻断原因与对应执行快照

### Requirement: 系统必须通过统一受控工具边界执行读取

系统 MUST 将分析编排中的知识推理与实际读取操作解耦。模型可以直接生成可执行查询，但真正执行 MUST 仍经过已注册的受控工具边界，由系统统一完成权限注入、白名单校验、API-first 路由、结果规模限制和审计。系统 MUST 阻止任何绕过该工具边界的自由数据库连接、自由 schema 枚举、未声明写入或未声明外部调用。

#### Scenario: AI 直查也必须命中过受控工具边界
- **WHEN** 模型决定直接读取 CRM 数据而不是先产出结构化计划
- **THEN** 系统必须将该读取请求映射到已注册的受控工具
- **THEN** 系统不得让模型绕过统一执行边界直接访问数据库连接

#### Scenario: 非法直接执行路径被阻断
- **WHEN** 任一编排步骤试图直接执行未经过受控工具边界的数据库或外部调用
- **THEN** 系统必须立即阻断该步骤并记录安全审计事件

### Requirement: 系统必须对每个读取步骤执行 API-first 与只读兜底路由

系统 MUST 对每个读取步骤先评估 CRM 官方 API 适配器，再决定是否进入受控 AI 直查 SQL 或受控内部只读接口。只要官方 API 已满足对象、字段、权限语义、聚合形态和稳定性要求，系统 MUST 优先选择官方 API；只有在不满足时，系统才 MAY 进入只读 SQL / 内部接口兜底，并记录 `preferredSource`、`matchedAdapter`、`gapReason` 和当前权限快照。

#### Scenario: 读取步骤优先命中官方 API
- **WHEN** 某个读取步骤能够由 CRM 官方 API 满足
- **THEN** 系统必须优先选择官方 API 适配器
- **THEN** 系统不得因为 SQL 更方便就默认绕过该 API

#### Scenario: 读取步骤进入受控 SQL 兜底
- **WHEN** 官方 API 无法满足当前分析步骤的对象、字段、聚合或稳定性要求
- **THEN** 系统可以进入受控只读 SQL 或内部只读接口兜底
- **THEN** 系统必须同时记录 API 缺口原因、命中适配器和权限快照

### Requirement: 系统必须在只读边界内执行最小纠错闭环

当只读执行步骤失败时，系统 MUST 先对错误进行分类，并仅对可判定、可收敛且不会扩大权限范围或读取范围的错误执行最小修正。自动纠错总尝试次数 MUST 不超过 3 次（含首次执行），每次修正都 MUST 保留前一版执行步骤、错误分类、修正说明和重试结果；对未知、高风险、越权或危险动作错误，系统 MUST 直接阻断并结束自动纠错。

#### Scenario: 可修复的只读错误触发有限次纠错
- **WHEN** 执行步骤因字段别名、聚合映射或已知适配差异而失败
- **THEN** 系统必须在不扩大权限和读取范围的前提下执行最小修正并有限次重试

#### Scenario: 高风险错误禁止自动重试
- **WHEN** 执行步骤命中越权访问、敏感字段探测、危险语句、注入风险或未知错误类型
- **THEN** 系统必须停止自动纠错并向用户返回明确阻断原因

### Requirement: 系统必须根据追问类型决定是否重新取数

系统 MUST 区分解释型追问和改条件追问。该分流 MUST 优先由统一 AI 理解层判定，而不是继续由多个局部规则和客户端标签共同决定。解释型追问 MUST 基于最近一次已验证结果包和执行快照生成说明，不得重新发起新的数据访问；改条件追问、缩小范围追问或新增维度追问 MUST 在当前权限下生成新的执行请求，并产出新的结果包与历史记录。

#### Scenario: 解释型追问复用已验证结果
- **WHEN** 用户在成功结果后继续追问“这说明什么”或“为什么会这样”
- **THEN** 系统必须基于最近一次已验证结果包生成解释
- **THEN** 系统不得为了回答解释型追问重新取数

#### Scenario: 改条件追问重新执行读取
- **WHEN** 用户在成功结果后追问“近三个月也看看”或“按区域重新比较”
- **THEN** 系统必须在当前权限下重新执行读取并返回新的分析结果

### Requirement: Web 分析入口与追问必须统一消费 AI 入口路由结果
系统 MUST 将 Web 智能分析工作台中的首次问数、解释型追问和改条件追问统一接入 `ai-entry-intent-orchestration`。服务端 MUST 先消费统一 AI 理解结果，再决定进入计划执行、结果复用解释、受控直查、补问澄清或阻断路径；本地中文词表和语义规则仅可保留为安全阻断与最小澄清 fallback，不得继续承担主理解链路。

#### Scenario: 首次问数先由统一 AI 理解决定执行路径
- **WHEN** Web 用户提交一条新的经营分析问题
- **THEN** 系统必须先基于统一 AI 理解结果决定进入计划执行、受控直查、补问或阻断
- **THEN** 系统不得仅依赖本地规则直接推断主题、指标、维度并进入主执行链路

#### Scenario: 解释型追问先由统一 AI 理解决定是否复用结果
- **WHEN** 用户在已有分析结果后继续输入“这说明什么”或语义等价的解释型追问
- **THEN** 系统必须先通过统一 AI 理解层判断该输入属于解释型追问
- **THEN** 只有在理解结果明确命中解释路径时，系统才可复用最近一次已验证结果包

#### Scenario: AI 主链不可用时只允许最小安全 fallback
- **WHEN** Web 分析入口的统一 AI 理解层不可用、超时或返回非法结构
- **THEN** 系统可以退回到安全阻断或最小澄清 fallback
- **THEN** 系统不得继续以旧的本地语义规则充当长期主理解链路

### Requirement: 受控分析编排必须消费 analysis intent 与 follow-up capability pack 结果
Web 智能分析工作台和企业微信分析入口 MUST 通过 `analysis-intent-pack` 与 `analysis-follow-up-pack` 消费统一 AI capability pack 的归一化结果，再决定进入计划执行、受控直查、解释复用、补问澄清或阻断路径。业务服务 MUST 不再直接内联问数入口 prompt、few-shot 和 provider 差异。

#### Scenario: 首次问数通过 analysis intent pack 决定执行路径
- **WHEN** 用户提交新的经营分析问题
- **THEN** 系统必须先通过 `analysis-intent-pack` 输出的归一化结果决定进入计划执行、受控直查、补问或阻断
- **THEN** 系统不得继续在分析入口服务中直接拼装完整 prompt

#### Scenario: 解释追问与改条件追问通过 follow-up pack 分流
- **WHEN** 用户在已有结果后继续追问“这说明什么”或“近三个月也看看”
- **THEN** 系统必须先通过 `analysis-follow-up-pack` 识别该输入属于解释型追问还是改条件追问
- **THEN** 系统不得继续分别在多个调用点硬编码这两类追问的分类逻辑

### Requirement: 系统必须对首次问数与继续追问执行独立动作校验
系统 MUST 对分析首次问数和继续追问执行独立动作校验。首次问数 MUST 以 `analysis.use` 作为最终放行条件；解释型追问、条件改写追问和基于既有结果包继续发起的新分析请求 MUST 以 `analysis.follow_up` 作为继续追问放行条件。系统 MUST 不得因为用户已经拥有首次问数资格就自动放开继续追问能力。

#### Scenario: 用户无首次问数权限时被阻断
- **WHEN** 某用户访问 Web 分析入口并提交新的经营分析问题，但未被授予 `analysis.use`
- **THEN** 系统必须拒绝本次分析执行
- **THEN** 不得仅因为其已登录或拥有分析页菜单资格而继续进入受控分析链路

#### Scenario: 用户无继续追问权限时不能基于已有结果继续分析
- **WHEN** 某用户拥有 `analysis.use`，但未被授予 `analysis.follow_up`，并在已有结果后继续输入解释型或改条件追问
- **THEN** 系统必须拒绝该追问执行
- **THEN** 不得因为其已经拿到上一轮结果而默认放开继续追问

### Requirement: 系统必须对导出结果执行统一动作与阈值双重校验
系统 MUST 在导出接口中先校验 `analysis.export` 与统一角色矩阵导出资格，再校验全局导出行数和次数阈值。旧的 `policy.exportRoleIds` 与 `user.exportAllowed` 若在迁移期继续存在，只可作为兼容附加条件，不得绕过统一动作权限结果。

#### Scenario: 用户缺少 analysis.export 时导出被阻断
- **WHEN** 某用户已完成分析，但未被授予 `analysis.export`
- **THEN** 系统必须拒绝其发起导出
- **THEN** 不得因为旧的导出角色配置或历史导出资格残留而继续放行

#### Scenario: 用户拥有 analysis.export 但超过阈值时仍被阻断
- **WHEN** 某用户已被授予 `analysis.export`，但本次导出超过单次行数上限或每日次数上限
- **THEN** 系统必须拒绝导出
- **THEN** 拒绝原因必须明确区分“权限不足”和“阈值超限”

### Requirement: 系统必须以 template.view 作为分析模板列表入口资格
系统 MUST 将分析页模板列表的可见性分成两层：用户是否有资格请求模板列表由 `template.view` 判定；某个模板资源是否对该用户可见仍由模板资源自身的 `visibleRoleIds`、状态等资源级条件判定。系统 MUST 不得继续仅凭旧的全局启用角色集合放行模板列表。

#### Scenario: 缺少 template.view 的用户模板列表为空
- **WHEN** 某用户未被授予 `template.view`，但请求分析页模板列表
- **THEN** 系统必须返回空模板结果或明确拒绝
- **THEN** 不得因为模板资源自身可见角色命中就返回模板列表

#### Scenario: 用户具备 template.view 但只看到资源级可见模板
- **WHEN** 某用户被授予 `template.view`，但部分模板资源并未将其角色包含在 `visibleRoleIds` 中
- **THEN** 系统必须只返回该用户在资源级也可见的模板
- **THEN** 不得因为用户具备模板查看动作就越过模板资源可见性边界

### Requirement: 受控分析执行必须消费统一时间槽
系统 MUST 在计划执行和受控 AI 直查两类执行模式中消费同一份标准时间槽。查询计划、受控 SQL、只读 API 适配器、查询参数、执行快照和审计记录 MUST 使用同一时间边界；系统不得在不同执行模式或不同渠道中重新解析原始问题文本生成不同时间口径。

#### Scenario: 计划执行使用时间槽编译查询
- **WHEN** 自由问数入口输出 `temporalSlot` 且执行模式为 `PLAN_EXECUTION`
- **THEN** 查询计划和 SQL 编译必须使用该时间槽中的起止边界
- **THEN** 编译结果必须在执行快照中记录该时间槽来源和边界

#### Scenario: 受控直查使用时间槽生成查询任务
- **WHEN** 自由问数入口输出 `temporalSlot` 且执行模式为 `GUARDED_DIRECT_QUERY`
- **THEN** 受控查询任务生成器必须将该时间槽传入 AI 任务上下文
- **THEN** 生成的查询任务必须在元数据中保留同一时间槽，不得只保留自由 SQL 文本

### Requirement: 查询执行前必须校验时间字段和时间边界
系统 MUST 在执行前校验 AI 生成或计划编译得到的时间过滤条件。校验内容 MUST 至少覆盖时间字段是否在对象白名单内、时间边界是否存在、时间窗口是否超过系统允许范围、相对时间是否已绑定基准时间和时区；校验失败时 MUST 阻断执行或发起补问。

#### Scenario: 商机新增默认使用允许的新增时间字段
- **WHEN** 用户查询“最近四个月新增商机金额趋势”
- **THEN** 系统必须确认查询使用的商机时间字段属于允许字段集合
- **THEN** 若 AI 生成了不在白名单内的时间字段，系统必须阻断或要求最小修正

#### Scenario: 缺少时间边界时不得全量取数
- **WHEN** 用户问题需要时间过滤但生成的查询任务缺少 `startAt` 或等价时间条件
- **THEN** 系统必须拒绝执行该读取任务或进入补问
- **THEN** 系统不得因为时间解析失败而默认查询全量历史数据

### Requirement: 系统必须通过统一问数知识层生成受控查询任务

系统 MUST 在不固定自然语言问数场景下引入统一问数知识层，作为统一 AI 理解层和受控查询任务生成器的共享上下文。问数知识层 MUST 至少覆盖字段别名、业务口径别名、组织 / 团队别名、码表翻译、高频查询模板、已验证查询示例和高风险字段边界。AI 在生成受控查询任务时 MUST 优先消费这些已批准知识资产，而不是每次从零假设字段语义或自由探测 schema；系统 MAY 在模板命中时直接复用已验证模板，但模板复用后的执行仍必须经过 API-first 路由、权限注入、白名单、AST 和预检链路。

#### Scenario: 自由问法通过知识层别名稳定命中受控查询任务
- **WHEN** 用户使用口语化、错别字、中英混写或业务别名描述同一类经营分析问题
- **THEN** 系统必须优先通过统一问数知识层中的别名和已验证样例稳定生成受控查询任务
- **THEN** 系统不得仅因未命中固定中文关键词就直接阻断或退回旧的关键词主链

#### Scenario: 高频问题优先复用已验证模板
- **WHEN** 用户问题命中已经批准的高频查询模板或已验证查询示例
- **THEN** 系统必须优先复用对应模板或等价模板资产生成受控查询任务
- **THEN** 系统不得在可复用模板已存在时仍强制每次从零生成新 SQL

#### Scenario: 知识层缺失时系统进入受控澄清或兜底
- **WHEN** 当前问数涉及的字段语义、口径或组织映射不在统一问数知识层中，且无法安全推断
- **THEN** 系统必须进入补问、计划执行兜底或显式阻断中的受控路径
- **THEN** 系统不得让模型继续以自由 schema 探测或自由扩大读取范围的方式补齐缺失知识

### Requirement: 受控分析必须消费统一组织范围快照

系统 MUST 在 Web 工作台和企业微信机器人问数进入受控分析执行前，基于当前企业微信组织事实、CRM 身份映射、默认团队范围和数据范围白名单生成统一组织范围快照。受控分析执行链路 MUST 仅消费该快照进行范围注入、审计和结果展示，不得直接读取历史静态权限表、企业微信灰度白名单或前端传入的部门条件来扩大数据范围。常用查询模板、模板预览、最近查询重跑和模板导出补查同样 MUST 消费该统一组织范围快照；当模板 SQL、展示口径或静态维度无法证明在当前范围内安全收口时，系统 MUST 阻断执行或要求治理审核，不得因为模板是预置模板或常用模板就绕过范围治理。

#### Scenario: Web 与企业微信问数使用同一范围解析
- **WHEN** 同一个 CRM 用户分别从 Web 工作台和企业微信机器人发起相同经营分析问题
- **THEN** 系统必须基于同一组织范围解析服务生成权限范围
- **THEN** 两个入口不得因为通道不同返回不同的数据范围

#### Scenario: 普通成员问数只返回本人范围
- **WHEN** 普通成员发起“本月新增商机金额排名”等问数请求
- **THEN** 系统必须将范围限制为该成员本人数据
- **THEN** 不得返回其同部门其他成员或团队管理者的数据

#### Scenario: 团队管理者问数返回其团队范围
- **WHEN** 团队管理者发起团队经营分析请求
- **THEN** 系统必须基于直属上级链和递归下属关系注入其团队成员范围
- **THEN** 不得返回不属于其默认团队且未被白名单授权的部门数据

#### Scenario: 数据白名单扩展问数范围
- **WHEN** 当前用户或其角色命中有效数据范围白名单
- **THEN** 系统必须在默认组织范围之外叠加白名单授权部门
- **THEN** 结果中的范围说明和审计快照必须标明白名单授权来源

#### Scenario: 请求超出组织范围时被阻断
- **WHEN** 用户明确请求查看未被默认范围或白名单覆盖的部门、团队或人员数据
- **THEN** 系统必须阻断或收敛到当前有效范围并明确提示
- **THEN** 不得按用户自然语言中的目标部门直接扩大查询范围

#### Scenario: 常用查询模板按当前范围重新收口
- **WHEN** 部门负责人执行一个默认全量的常用查询模板
- **THEN** 系统必须在执行前消费当前 `scopeSnapshot` 并完成模板范围治理
- **THEN** 查询结果、图表、转置表、矩阵表和导出补查都不得展示当前范围外的数据

#### Scenario: 多部门负责人分析范围先按默认业务部门裁剪
- **WHEN** 部门负责人在企业微信直属链中拥有跨区域、跨技术线或跨服务线下属
- **THEN** 系统必须先按该负责人默认业务部门及其子部门裁剪默认负责人范围
- **THEN** 未被默认业务部门或数据范围白名单覆盖的负责人不得进入常用查询模板、自由问数或导出补查范围

#### Scenario: 模板范围无法证明安全时阻断执行
- **WHEN** 常用查询模板存在静态团队清单、展示口径与过滤口径不一致或无法解析的固定范围
- **THEN** 系统必须阻断普通用户执行或要求治理审核
- **THEN** 系统不得因为该模板已预置、已收藏或来自最近查询而继续放行

### Requirement: 受控分析编排必须消费发布态语义资产快照
系统 MUST 在自由问数进入统一 AI 理解层、受控查询任务生成器和结果编排前，先装载当前已发布的语义资产快照。该快照 MUST 至少覆盖别名提示、时间口径提示、组织归一提示、已验证问法和负例样例，并作为统一共享上下文输入后续编排步骤；系统 MUST 不得继续让编排层只依赖代码内静态常量作为唯一问数知识源。

#### Scenario: 自由问数优先消费已发布语义资产
- **WHEN** Web 或企业微信收到一条新的自由问数请求
- **THEN** 系统必须先装载当前已发布的语义资产快照，再进入统一 AI 理解层和后续受控编排
- **THEN** 系统不得跳过已发布资产直接只依赖本地静态提示生成查询任务

#### Scenario: 语义资产缺失时进入受控澄清或阻断
- **WHEN** 当前问数依赖的字段语义、组织归一或时间口径不在已发布语义资产中，且系统无法安全推断
- **THEN** 系统必须进入补问、计划执行兜底或显式阻断中的受控路径
- **THEN** 系统不得因为知识缺失而放宽为自由 schema 探测或扩大读取范围

### Requirement: 受控分析编排必须生成统一执行轨迹摘要
系统 MUST 为每次受控分析执行生成统一执行轨迹摘要，并将其绑定到同一份执行快照与结果包。执行轨迹摘要 MUST 至少包含入口问题、标准化问题、命中的语义资产摘要、任务级 `resultKind`、任务级 `executionSource`、`preferredSource`、`matchedAdapter`、最终 `fallbackReason`、统一 `consistencyToken` 和结果区块引用摘要。系统 MUST 允许结果页、审计详情和后续回归验证消费同一份轨迹摘要，而不是分别从多个局部快照字段自行拼装。

#### Scenario: 执行轨迹记录命中的语义资产与执行来源
- **WHEN** 一次问数成功进入计划执行或受控直查执行
- **THEN** 系统必须在统一执行轨迹摘要中记录本次命中的语义资产摘要以及每个任务的执行来源
- **THEN** 后续结果页与审计详情必须能够基于同一轨迹解释“为什么走了这条执行路径”

#### Scenario: fallback 场景保留统一轨迹摘要
- **WHEN** 受控分析在统一 AI 理解层、任务生成或结果交付前命中 `fallbackReason`
- **THEN** 系统必须在统一执行轨迹摘要中保留该 `fallbackReason` 及对应阶段
- **THEN** 审计端不得只看到最终失败结果而无法解释 fallback 发生在哪个阶段

### Requirement: 受控分析轨迹必须与审计和结果包共享同一事实边界
系统 MUST 保证统一执行轨迹摘要与结果包、审计快照共享同一权限快照、同一时间口径和同一数据集引用边界。若轨迹摘要、结果包和审计记录之间出现来源不一致、时间口径不一致或数据集引用缺失，系统 MUST 阻止正式结果交付或标记该轨迹无效，不得继续向用户展示看似可信但无法追溯的结果。

#### Scenario: 轨迹摘要与结果包共享同一时间口径
- **WHEN** 一次分析执行生成了结果包与执行轨迹摘要
- **THEN** 二者必须引用同一份时间口径与权限快照
- **THEN** 系统不得允许轨迹摘要解释一个时间范围而结果包展示另一时间范围

#### Scenario: 轨迹摘要缺少数据集引用时阻止正式交付
- **WHEN** 系统无法将某个结果区块或结论回溯到执行轨迹中的任务或数据集引用
- **THEN** 系统必须阻止该次正式结果交付或将其标记为无效
- **THEN** 不得继续把缺乏引用链路的结果当作可信正式结果返回给用户

### Requirement: 受控分析命中 SQL 路径时必须生成关联 SQL 审计记录

系统 MUST 在受控分析进入 SQL 路径时，为任务 SQL 与预检 SQL 生成可回放的 SQL 审计记录。SQL 审计记录 MUST 与分析请求、会话、执行模式、执行来源、命中适配器、fallback 原因和统一执行轨迹建立关联；若当前分析步骤未命中 SQL 路径而改走官方 API 或内部只读接口，系统 MAY 不生成 SQL 审计记录，但不得伪造“已执行 SQL”。

#### Scenario: 受控直查 SQL 在分析执行中留下阶段化审计

- **WHEN** 自由问数最终命中 `GUARDED_READONLY_SQL` 路径并生成正式 SQL 与预检 SQL
- **THEN** 系统必须分别为预检与正式执行留下 SQL 审计记录
- **THEN** 审计记录必须能关联到对应 `requestId`、`executionMode` 和 `matchedAdapter`

#### Scenario: SQL 候选在到库前被阻断时留下阻断审计

- **WHEN** AI 生成的候选 SQL 在白名单、AST、时间边界或风险校验阶段被阻断
- **THEN** 系统必须生成 `BLOCKED` 状态的 SQL 审计记录
- **THEN** 治理用户必须能够从 SQL 审计直接看到阻断原因，而不是只能从普通业务审计猜测

### Requirement: 自由问数结果保存模板必须复用受控执行事实

系统 MUST 支持将成功自由问数结果保存为查询模板。保存时，系统 MUST 通过 `queryId` 读取已验证的受控执行事实，优先复用实际执行的受控 SQL、参数、范围治理结论、结果展示配置和一致性标识；若只有执行计划，系统 MUST 在服务端将其编译为参数化 SQL 模板并重新校验。系统 MUST NOT 接受前端直接提交自由拼接 SQL 作为自由问数保存模板的执行来源。

#### Scenario: 基于 queryId 保存成功分析结果
- **WHEN** 用户对已成功返回的自由问数结果发起保存模板
- **THEN** 系统必须根据 `queryId` 读取该结果的执行快照
- **THEN** 系统必须基于服务端受控 SQL 或可编译执行计划生成模板
- **THEN** 系统不得使用前端提交的 SQL 作为可信执行来源

#### Scenario: 执行计划编译后重新校验
- **WHEN** 分析结果没有直接保存受控 SQL，但存在可复现执行计划
- **THEN** 系统必须在服务端将执行计划编译为参数化 SQL 模板
- **THEN** 编译后的模板必须重新通过只读、白名单、AST、安全范围治理和预检校验

### Requirement: 不可复现分析结果不得保存为模板

当自由问数结果缺少可复现 SQL，且执行计划无法安全编译为参数化 SQL 模板时，系统 MUST 拒绝保存模板，并返回面向业务用户的中文原因。系统 MUST 记录保存失败审计，不得创建不可执行、不可校验或无法追溯执行来源的模板记录。

#### Scenario: 缺少可复现执行事实时拒绝保存
- **WHEN** 用户尝试保存一条缺少受控 SQL 且执行计划不可编译的分析结果
- **THEN** 系统必须拒绝保存模板
- **THEN** 系统必须提示当前查询暂不支持保存为模板
- **THEN** 系统不得创建模板记录

#### Scenario: 保存失败写入审计
- **WHEN** 自由问数保存模板因执行事实不足或安全校验失败被拒绝
- **THEN** 系统必须写入保存失败审计
- **THEN** 审计必须包含 `queryId`、操作者、失败阶段和中文失败原因

### Requirement: 保存生成的模板执行时必须重新装载当前权限

由自由问数保存生成的模板 MUST 只保存查询口径、参数结构和展示配置，不得保存为后续执行放行依据的历史授权快照。任何用户执行该模板时，系统 MUST 基于执行人当前实时权限、当前模板定义和当前范围治理结论重新执行。

#### Scenario: 其它用户执行保存模板
- **WHEN** 用户 A 保存模板后，用户 B 执行该模板
- **THEN** 系统必须按用户 B 当前权限范围重新注入组织、部门和负责人范围
- **THEN** 系统不得复用用户 A 保存模板时的权限快照

#### Scenario: 保存人权限变化后再次执行
- **WHEN** 模板保存人的权限在保存后发生变化
- **THEN** 保存人再次执行该模板时必须使用最新权限范围
- **THEN** 系统不得沿用保存时的历史权限放行
