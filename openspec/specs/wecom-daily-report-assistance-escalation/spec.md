## Requirements
### Requirement: 日报正式对上发送时必须检查协助升级事项

系统 MUST 在日报正式发送给主管或负责人时，检查该日报中的 `HELP_REQUIRED` 片段和结构化 `structuredHelpNeeded` 内容。若条目包含明确的问题描述且明确请求内部某人协助，系统 MUST 生成协助升级任务；若条目为“无”“暂无困难或协助需求”“无需协助”等否定值，系统 MUST 不得生成协助升级任务。系统 MUST 不得在 `22:00` 发给本人确认的草稿、未确认日报或仅预览场景中提前触发协助升级。

#### Scenario: 正式发给主管时为明确协助请求创建升级任务

- **WHEN** 某份日报已经进入正式对上发送给主管的出站流程，且其中一条 `HELP_REQUIRED` 内容明确写有“需要李四协助补充实施排期”之类的协助请求
- **THEN** 系统必须为该协助事项生成协助升级任务
- **THEN** 系统必须继续执行主管侧日报送达，而不是要求人工再转发协助事项

#### Scenario: 本人确认草稿和无协助日报不得触发升级

- **WHEN** 系统只是在 `22:00` 向日报提报人发送本人确认草稿，或 `HELP_REQUIRED` 内容为“无”“暂无协助”
- **THEN** 系统不得生成任何协助升级任务
- **THEN** 系统不得把草稿中的协助信息提前发送给协助人或协助人的主管

### Requirement: 协助人和协助人主管必须基于当前身份与组织关系唯一解析

系统 MUST 仅在协助请求中出现可唯一映射到内部用户的显式人名或稳定身份标识时，才把该用户视为协助人。解析 MUST 复用当前 CRM 用户、企业微信身份映射和当前组织/负责人关系，并进一步解析该协助人的当前主管。若协助人未命中、命中多个用户、只出现角色描述或主管关系缺失，系统 MUST 阻断该条协助通知的直发并留下审计记录，但 MUST 不得阻断主管侧日报送达。

#### Scenario: 显式协助人可唯一命中时解析协助人及其主管

- **WHEN** 日报中的协助事项写明“需要产品同事李四协助补充实施排期”，且 `李四` 能通过当前身份映射唯一命中到内部用户
- **THEN** 系统必须把该内部用户解析为协助人
- **THEN** 系统必须继续解析该协助人的当前主管作为第二接收对象

#### Scenario: 角色描述或歧义姓名不得被直接发送

- **WHEN** 日报中的协助事项只写“需要产品同事协助”，或写明的人名命中多个内部用户
- **THEN** 系统不得向任何协助人候选发送直达通知
- **THEN** 系统必须记录“协助人无法唯一识别”或等效原因的审计信息，同时保留主管侧日报正常送达

### Requirement: 系统必须向协助人及其主管发送聚合后的协助通知

对于每个已成功解析的协助人，系统 MUST 在同一日报正式出站批次中生成至少一条协助人通知和一条协助人主管通知。系统 MUST 按“同一份日报 + 同一协助人”聚合同批次协助事项；若最终接收目标因角色重叠而重复，例如协助人主管与日报主管相同，系统 MUST 按最终目标用户去重，只保留一条实际发送记录，同时保留关联来源审计。

#### Scenario: 协助人和协助人主管均收到同批次通知

- **WHEN** 某份正式日报中有一条事项请求 `李四` 协助，且 `李四` 及其主管 `王主管` 都已被成功解析
- **THEN** 系统必须在该日报正式发送给主管的同一批次里向 `李四` 发送协助请求通知
- **THEN** 系统必须在同一批次里向 `王主管` 发送“你的成员李四被请求协助”的通知

#### Scenario: 角色重叠时按最终目标去重

- **WHEN** 某条协助事项中，协助人主管与日报主管是同一个最终接收人，或同一份日报里多条事项都指向同一个协助人
- **THEN** 系统不得对同一个最终目标重复发送内容等价的多条消息
- **THEN** 系统必须在审计中保留该接收人同时承担多个通知角色或多条事项来源的关联关系

### Requirement: 协助通知内容必须最小化描述问题与请求

系统 MUST 在协助通知中至少包含提报人、业务日期、相关客户或商机上下文、问题摘要和请求协助事项。系统 MUST 仅发送与协助相关的最小必要信息，不得把整份日报或无关工作总结原样转发给协助人。协助人主管版本 MUST 明确说明是哪位成员被请求协助、由谁发起以及具体问题摘要。

#### Scenario: 同一协助人的多条事项被聚合成一条最小必要通知

- **WHEN** 同一份日报里有两条不同客户或商机事项都请求同一个协助人支持
- **THEN** 系统必须把这两条事项聚合成一条协助通知，并按条目列出对应问题摘要
- **THEN** 系统不得把该日报中无关的“信息共享”或“计划”段落一并转发给协助人

#### Scenario: 协助人主管通知必须说明协助链路

- **WHEN** 系统准备向某协助人的主管发送该协助事项通知
- **THEN** 消息内容必须说明“谁在什么日期的日报中请求你的成员提供什么协助”
- **THEN** 消息不得只转发一段脱离上下文的原始日报文本

### Requirement: 协助升级通知必须复用主动通知底座并与主日报送达失败隔离

系统 MUST 将协助升级通知作为正式主动通知任务接入现有 `ProactiveNotificationService`，并复用既有幂等、有限重试、测试改投和审计留痕机制。若某一条协助通知在接收人解析、权限校验或发送阶段失败，系统 MUST 只将对应协助子任务标记为失败，而 MUST 不得回滚已经成功送达给主管的日报，也不得阻断同批次其他协助通知继续发送。

#### Scenario: 协助子任务失败不阻断主管侧日报送达

- **WHEN** 某份日报已经成功送达主管，但其中一条协助人通知因权限校验失败或发送重试耗尽而最终失败
- **THEN** 系统必须保留主管侧日报的已送达状态
- **THEN** 系统必须把该协助子任务的失败原因、重试次数和原始日报关联写入审计记录

#### Scenario: 同一日报批次重试时不得重复发送已成功的协助通知

- **WHEN** 同一份日报的正式出站批次因局部失败被再次重试，且某个协助人的消息版本和目标用户都未变化
- **THEN** 系统必须命中幂等保护，不得向该协助人或其主管重复发送已成功的通知
- **THEN** 系统必须只重试尚未成功的协助子任务
