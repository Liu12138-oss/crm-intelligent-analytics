## MODIFIED Requirements

### Requirement: 系统必须创建可追踪的合同审核任务

系统 MUST 支持基于 CRM 待1级审批合同创建独立的合同审核任务，并为任务保存请求人、来源类型、源合同 ID、源合同关键字段快照、任务状态、规则集标识和时间戳等基础信息。任务状态 MUST 至少覆盖 `UPLOADED`、`PARSING`、`REVIEWING`、`GENERATING_REPORT`、`COMPLETED`、`FAILED` 和 `BLOCKED`，且每次状态变更都必须可追踪。

#### Scenario: 从待1级审批合同成功创建审核任务

- **WHEN** 用户在待1级审批合同表格中选择一条合同并发起审核
- **THEN** 系统必须创建一条绑定该源合同快照的审核任务，并按顺序推进到解析、审核和产物生成等后续阶段

#### Scenario: 源合同缺少前置审核条件时任务被阻断

- **WHEN** 系统发现当前源合同缺少可用审核文本、关键字段缺失、无可用规则集或其他执行前置条件不满足
- **THEN** 系统必须将任务标记为 `BLOCKED` 或 `FAILED`，并记录用户可读的原因说明

### Requirement: 系统必须在正式审核前提取合同文本并固化规则快照

系统 MUST 在进入 AI 审核前完成源合同审核快照整理和基础结构识别，并为该任务固化一份规则集快照。对于 CRM 来源任务，系统 MUST 基于合同详情、关键字段、特殊条款和可用审批备注构造统一审核文本快照，而不是要求必须存在本地上传文件；规则快照 MUST 能标识本次任务使用的规则版本或快照哈希，以供后续问题定位和报告生成使用。

#### Scenario: CRM 来源任务成功保存规则快照与合同快照

- **WHEN** 审核任务从待1级审批合同进入正式解析流程
- **THEN** 系统必须为该任务绑定当前生效规则快照，并保存当次源合同的字段与文本摘要

#### Scenario: CRM 源合同无法整理出可审核快照时终止正式审核

- **WHEN** 系统无法从源合同详情、特殊条款或审批备注中整理出可审核的文本快照
- **THEN** 系统必须终止正式审核流程，并将任务标记为失败或阻断，而不是继续生成不可信的审核结果

## ADDED Requirements

### Requirement: 系统必须按 CRM 官方 API 优先、受控只读兜底的方式获取待1级审批合同源数据

系统 MUST 先评估 CRM 官方 API 是否已覆盖“待1级审批合同列表”和“合同详情审核所需字段”能力。若官方 API 已满足对象、字段、审批步骤筛选和权限语义要求，系统 MUST 优先复用官方 API；若官方 API 无法直接提供待1级审批列表或无法满足审批步骤口径，系统 MUST 记录缺口原因后，才允许走受控只读数据路径读取 `contracts` 及相关白名单表字段。

#### Scenario: 官方 API 可直接满足待1级审批合同源数据

- **WHEN** 已确认官方 API 可以直接返回待1级审批合同列表和合同详情审核所需字段
- **THEN** 系统必须以官方 API 作为主接入路径，而不是默认直查数据库

#### Scenario: 官方 API 无法直接满足待1级审批列表口径

- **WHEN** 官方 API 仅能提供通用合同列表或详情，但不能直接满足“待1级审批”筛选口径
- **THEN** 系统必须记录 API 缺口原因、字段白名单、权限继承和审计边界后，才允许进入受控只读兜底路径

### Requirement: 系统必须让 CRM 来源任务保留来源语义并与审核结果关联

系统 MUST 让基于 CRM 合同创建的审核任务保留来源类型、源合同 ID、审核时刻的源合同字段快照和审批步骤信息。任务详情、结构化结果和审计事件 MUST 能追溯这些来源信息，以便后续复盘本次审核到底基于哪一份 CRM 合同数据。

#### Scenario: 任务详情可追溯源合同

- **WHEN** 用户或审计角色查看某个基于 CRM 合同创建的审核任务
- **THEN** 系统必须能够追溯该任务对应的源合同 ID、源合同名称和审核时刻的审批步骤快照

#### Scenario: 源合同数据发生后续变化时历史任务仍保留原始快照

- **WHEN** 源合同在 CRM 中被继续编辑或审批状态发生变化
- **THEN** 历史审核任务仍必须保留创建任务时固化的源合同快照，而不是被最新合同数据覆盖
