## Context

当前合同审核模块已经具备任务编排、AI 审核、确定性校验、结果详情页和产物下载能力，但工作台仍以“上传本地 `.docx` 文件”作为唯一入口。用户这次要求改为直接接入 CRM 中“待1级审批”的合同数据，先在页面表格中浏览合同，再按行发起审核。

这次改动同时涉及前端工作台入口、后端合同源数据接入、审核任务建模和审计留痕，属于跨模块变更。它还受到以下约束：

- 必须遵循现有“CRM 官方 API 优先、受控只读兜底”的接入策略。
- 不得把用户提供的内网地址、数据库账号或密码写入代码、文档、测试夹具或日志。
- 必须复用现有 `contract-review` 审核引擎、`AppShell` 路由、权限矩阵和审计中心，而不是再起一条平行链路。
- 工作台应优先服务商务人员当前的审批动作，不再让上传区成为主入口。

结合已检查的 CRM 官方文档，可以确认官方提供了合同通用列表与详情接口，例如 `/api/v2/contracts` 与 `/api/v2/contracts/:id`，也提供了审批流详情接口；但当前未确认存在一个可以直接返回“待1级审批合同列表”的合同模块官方接口。为了避免在工作台列表阶段引入跨域审批中心 N+1 查询，本次设计先采用受控只读 DB 兜底读取“待1级审批”合同，再保留后续切回官方 API 的适配边界。

## Goals / Non-Goals

**Goals:**

- 把合同审核工作台改成“待1级审批合同表格 + 合同数据查看 + 发起审核”流程。
- 复用现有审核引擎，让 CRM 合同记录也能进入相同的 AI 审核与结果详情页。
- 为每个审核任务保存源合同快照，确保结果、审计和回溯都能定位到审核时的 CRM 数据。
- 补齐源合同列表、详情和发起审核接口，以及对应前端表格交互和回归测试。

**Non-Goals:**

- 不在本次变更中接入合同审批通过/驳回写回 CRM。
- 不在本次变更中开放批量审核、筛选器编排或新的审批管理后台。
- 不在本次变更中把 CRM 附件自动还原成原始 `.docx` 再生成批注稿。
- 不调整现有登录、会话、统一鉴权或权限矩阵基础模型。

## Decisions

### 决策 1：工作台改为表格驱动，并用抽屉查看源合同数据

工作台首页改为加载“待1级审批合同”表格，合同名称可点击查看源合同详情，操作列提供“开始审核”按钮。审核完成后仍然进入现有任务详情页展示问题结果。

原因：

- 用户的主动作已经从“上传文件”变成“在 CRM 待审批合同里挑一份审核”。
- 合同源数据查看属于审核前确认，不需要再额外新起一个详情路由，抽屉比二次跳页更轻。
- 审核结果页已经存在，继续复用能减少无关改动。

备选方案：

- 仍保留上传区为主入口，仅额外增加 CRM 表格。问题是会让主流程并行，造成用户认知冲突。
- 点击合同名称跳到新的 CRM 合同详情页。问题是页面链路更长，且与审核结果页的职责容易混淆。

### 决策 2：新增独立的 CRM 合同源读取服务，但继续复用现有审核服务主链路

新增 `ContractReviewCrmSourceService` 负责待1级审批合同列表、合同源数据详情和权限内可见性判断；`ContractReviewService` 只新增“基于 CRM 合同快照创建审核任务”的入口，并继续复用既有审核编排、问题生成和详情查询逻辑。

原因：

- 这样可以把“源数据接入”和“审核任务处理”两个职责拆开，避免把 CRM 读取逻辑塞进已经很长的审核服务里。
- 现有单元测试对 `ContractReviewService` 构造依赖很多，若直接把 CRM 读取能力硬注入进去，会显著放大回归面。
- 读取服务后续也更容易切换到 CRM 官方 API 实现。

备选方案：

- 让 `ContractReviewService` 直接负责 CRM 列表和详情。问题是职责继续膨胀。
- 单独再起一个新的合同审核模块。问题是会重复现有结果详情、权限和审计逻辑。

### 决策 3：源合同列表先走受控只读 DB 兜底，并保留官方 API 切换边界

本次“待1级审批合同列表”先通过 `CrmReadonlyService` 受控读取 `contracts`、`customers`、`opportunities`、`users`、`departments` 与 `contract_multistep_approves` 等白名单表字段实现。设计上保留“官方 API 适配器”边界，后续一旦确认存在可直接覆盖“待1级审批列表”的官方接口，即可替换读取实现。

原因：

- 已确认官方 API 提供合同通用列表 `/api/v2/contracts` 和详情 `/api/v2/contracts/:id`，但当前未确认合同模块有直接的“待1级审批”过滤参数；审批流详情接口存在于独立审批域，若逐条补查会造成列表阶段额外复杂度。
- 现有项目已经有 `CrmReadonlyService` 和受控数据库配置，可在不暴露凭据的情况下承接这次列表接入。
- 只读 DB 路径更容易精确限制字段白名单和权限过滤。

备选方案：

- 直接在工作台串联合同官方列表 API 和审批流详情 API。问题是实现复杂度高，且依赖更多在线令牌与跨域审批接口。
- 直接读取前端页面 URL 的 HTML。问题是不可控且不满足稳定接入要求。

### 决策 4：为 CRM 来源任务补充源合同快照与来源类型，而不是复用上传文件语义

合同审核任务新增 `sourceType`、`sourceContractId` 和 `sourceContractSnapshot`。对于 CRM 来源任务，`originalFileName` 继续承载合同名称，`storedFilePath` 使用内部来源标识，审核过程中直接基于源合同快照构造 `ContractReviewDocumentSnapshot`，而不是要求必须存在本地文件。

原因：

- 任务详情、审计事件和结构化结果都需要知道这次审核针对的是哪一份 CRM 合同。
- 如果仍强行假装是上传文件，会让后续排障和审计丢失来源语义。
- 用统一的 `ContractReviewDocumentSnapshot` 作为审核输入，可以最大限度复用现有 AI 审核与确定性校验链路。

备选方案：

- 继续沿用“文件上传任务”模型，不保存源合同快照。问题是审计不可追溯。
- 为 CRM 来源单独起一套审核结果模型。问题是会重复大量详情和产物逻辑。

### 决策 5：源合同任务默认只稳定交付报告与结构化结果，批注稿按不可用处理

基于 CRM 合同数据发起的审核任务默认生成审核报告和结构化结果文件；由于当前没有稳定的原始 `.docx` 输入，带批注合同产物默认标记为不可生成或失败原因明确的不可用状态，前端继续按产物状态展示。

原因：

- 用户当前核心诉求是“看合同数据并发起审核”，不是恢复原始文档批注。
- 强行尝试把 CRM 字段拼装成伪 `.docx` 会降低结果可信度，也会显著增加实现复杂度。
- 现有详情页已经具备按产物状态展示失败原因的能力，复用成本低。

备选方案：

- 完全隐藏产物区。问题是会让已有详情页行为分叉太大。
- 从合同附件里自动找原始 docx 后生成批注稿。问题是附件字段映射和文件可用性当前未明确。

### 决策 6：源合同查看与发起审核都进入合同审核既有权限和审计边界

源合同列表与详情继续命中合同审核工作台基础访问资格；发起审核继续复用现有 `contract.review.upload` 动作权限作为“发起审核”许可；新增源合同查看审计事件，并让任务审计记录带上源合同 ID 与来源类型。

原因：

- 这次只是入口从上传切到 CRM 合同，不需要同时引入新的权限模型。
- 复用现有动作权限可以降低权限中心、能力快照和测试矩阵的改造成本。
- 新增源合同查看审计事件能补齐“谁看过哪份待审批合同并基于它发起了审核”的排障链路。

备选方案：

- 新增独立 `contract.review.start` 权限点。问题是会额外扩大权限矩阵改造面。
- 不记录源合同查看审计。问题是缺少审核前查看行为留痕。

## Risks / Trade-offs

- [待1级审批判定可能与线上 CRM 页面口径有偏差] → 首版以 `pending_step = 1` 等明确字段为准，并把口径写入 OpenSpec 与主规格，便于后续对齐。
- [官方 API 后续若补齐直接过滤能力，本次只读实现会成为临时路径] → 通过独立源数据服务隔离读取实现，后续可平滑切换。
- [CRM 源合同缺少完整正文时，AI 审核命中质量可能弱于 docx] → 用特殊条款、签署信息、审批备注和核心合同字段构造统一审核快照，并在缺少关键内容时明确阻断。
- [没有原始 docx 时批注稿不可生成] → 详情页按产物状态明确提示，不伪造可下载结果。
- [复用 `contract.review.upload` 作为发起审核权限语义不够直观] → 本次先保证兼容，后续若权限中心统一重命名，再单独发起权限口径变更。

## Migration Plan

1. 先补 OpenSpec proposal、design、specs 和 tasks，明确工作台、编排和审计变更。
2. 后端增加 CRM 合同源读取服务和新接口，同时让审核服务支持基于 CRM 合同快照建任务。
3. 前端把工作台替换为待1级审批合同表格，并接入合同数据查看与发起审核。
4. 更新 OpenAPI、快速开始和主规格中的合同审核流程描述。
5. 回归验证前端工作台、任务详情和后端单元/集成测试。
6. 若线上联调发现待1级审批筛选口径不一致，可先回滚工作台入口到旧版上传模式，同时保留后台代码分支与 OpenSpec 变更记录。

## Open Questions

- 业务侧最终确认的“待1级审批”数据库口径，是否仅以 `pending_step = 1` 为准，还是还要叠加 `approve_status` 过滤。
- CRM 合同详情中是否存在稳定的原始合同正文或正文类自定义字段，可用于提升 AI 审核命中质量。
- 后续是否需要把“已审过但仍待审批”的最新审核摘要也回写到工作台表格中，方便直接复查。
