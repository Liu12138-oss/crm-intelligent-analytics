## ADDED Requirements

### Requirement: 系统必须记录每轮企业微信目录同步任务状态
系统必须为每轮企业微信目录同步任务记录任务类型、开始时间、结束时间、成功状态、游标、分页统计和失败原因。该记录必须覆盖部门同步、用户同步和用户部门变更同步三类任务。

#### Scenario: 部门增量同步成功
- **WHEN** 一轮部门增量同步全部分页成功完成
- **THEN** 系统必须记录成功状态、推进后的游标、同步条数和分页数

#### Scenario: 用户增量同步失败
- **WHEN** 用户增量同步在某一页调用外部 API 失败
- **THEN** 系统必须记录失败状态、失败页码、失败原因和最后成功游标

### Requirement: 系统必须支持查询最近一次同步状态和游标
系统必须提供内部状态查询能力，至少能回答：

- 最近一次部门同步是否成功
- 最近一次用户同步是否成功
- 最近一次用户部门变更同步是否成功
- 当前各自的最后成功时间和游标值

#### Scenario: 查询同步状态
- **WHEN** 运维或开发人员查询企业微信目录同步状态
- **THEN** 系统必须返回最近成功时间、最近失败原因和当前游标

#### Scenario: 区分“无增量”与“同步失败”
- **WHEN** 最近一次同步调用成功但未返回任何增量记录
- **THEN** 系统必须将其标记为“成功但无增量”，而不是失败

### Requirement: 身份识别结果必须记录命中来源
当机器人身份识别成功或失败时，系统必须记录命中来源或失败原因，至少区分：

- `crm-wx-map`
- `mirror-userid`
- `mirror-phone`
- `mirror-email`
- `unresolved`
- `ambiguous`

#### Scenario: 通过邮箱兜底识别成功
- **WHEN** 企业微信用户最终通过邮箱匹配到 CRM 用户
- **THEN** 系统必须记录命中来源为 `mirror-email`

#### Scenario: 未识别用户
- **WHEN** 企业微信发送者经过原生映射和镜像兜底仍无法识别到 CRM 用户
- **THEN** 系统必须记录命中来源为 `unresolved`，并保留失败原因

#### Scenario: 多命中用户
- **WHEN** 企业微信发送者通过手机号或邮箱匹配到多个 CRM 用户
- **THEN** 系统必须记录命中来源为 `ambiguous`，并保留多命中原因供人工处理
