## ADDED Requirements

### Requirement: 系统必须支持从外部 EMM API 增量同步企业微信部门
系统必须使用外部 EMM API 的应用访问令牌调用部门增量接口 `POST /organizational/listDepartmentByTime`，按 `updatetimestamp` 和分页参数拉取新增、修改、删除的部门信息，并写入应用侧镜像表。同步结果不得直接写回 CRM 主库。

#### Scenario: 首次同步部门目录
- **WHEN** 系统尚不存在部门同步游标
- **THEN** 系统必须按配置的初始时间启动部门分页同步，并在全部分页成功后初始化游标

#### Scenario: 按时间戳拉取增量部门
- **WHEN** 系统已存在最近一次部门同步游标
- **THEN** 系统必须以该游标为起点调用增量接口，仅同步后续变化的部门数据

#### Scenario: 部门同步分页中途失败
- **WHEN** 部门同步在某一分页调用外部接口失败
- **THEN** 系统必须保留最后成功游标，不得把失败页之后的时间戳提前写成已成功同步

### Requirement: 系统必须支持从外部 EMM API 增量同步企业微信用户
系统必须使用外部 EMM API 的应用访问令牌调用用户增量接口 `POST /organizational/listUserByTime`，按 `updatetimestamp` 和分页参数拉取用户详情快照，并保存 `uiduserid`、姓名、手机号、邮箱、部门等关键字段到应用侧镜像表。

#### Scenario: 用户增量同步保存手机号与邮箱
- **WHEN** 外部用户增量接口返回 `uiduserid`、`phone`、`email` 等用户字段
- **THEN** 系统必须把这些字段保存到应用侧用户镜像表，供后续身份识别兜底使用

#### Scenario: 用户分页同步中断后可重试
- **WHEN** 用户增量同步在分页过程中失败
- **THEN** 系统必须保留最后成功游标和失败页信息，以便后续重试继续同步

#### Scenario: 用户增量接口返回空页
- **WHEN** 用户增量接口成功返回但 `rows` 为空
- **THEN** 系统必须将其识别为“当前无增量数据”，而不是同步失败

### Requirement: 系统必须支持同步用户部门变更记录
系统必须调用 `POST /organizational/listUserDeptChangeByTime` 拉取用户部门变更事件，并将其与用户详情镜像分开存储。该事件流至少应支持 `changeDept`、`addDept`、`removeDept` 等操作类型的记录。

#### Scenario: 用户部门变更被记录到事件表
- **WHEN** 外部接口返回用户部门变更记录
- **THEN** 系统必须将变更事件落入应用侧镜像事件表，而不是覆盖掉用户详情快照

#### Scenario: 用户部门变更与用户详情同步并存
- **WHEN** 同步任务同一轮既拉取用户详情，又拉取用户部门变更
- **THEN** 系统必须分别保存两类数据，并保证后续排障能区分“当前快照”和“历史变更”

### Requirement: 系统必须为每类资源独立维护同步游标
系统必须分别为部门、用户和用户部门变更三类资源维护独立同步游标。任何一类资源同步失败时，不得影响其他资源继续按自己的游标推进。

#### Scenario: 用户同步失败但部门同步成功
- **WHEN** 同一轮任务中部门同步成功、用户同步失败
- **THEN** 系统必须允许部门游标推进，而用户游标保持在最后成功位置
