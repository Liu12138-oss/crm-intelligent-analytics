## Context

根级 `DESIGN.md` 已经完成从旧版白底蓝灰后台到浅色企业 AI 工作台的方向重设，并且前端已经接入 Vue 3、Element Plus、`@element-plus/icons-vue`、统一应用壳层、基础主题变量和若干业务代码中文化能力。

当前主要问题不在功能链路，而在视觉语言和页面完成度：业务模块缺少稳定识别锚点，图表色彩长期依赖单一品牌渐变，表格和列表缺少对象图标 / 等级 / 状态线索，空状态和加载态偏文字化，页面标题、数字、正文和标签的层级差距不足。领导反馈“颜色太单一、页面太粗糙、图标图形元素太少、字体间距不够”后，必须把这些问题沉淀为设计系统和页面级改造要求，而不是靠单页临时补色。

约束条件：

- 所有新增或修改文档正文使用中文。
- 前端技术栈保持 Vue 3 + Element Plus + `@element-plus/icons-vue`。
- Element Plus 只作为基础组件能力来源，视觉必须服从 `DESIGN.md`。
- 旧 `.pen` 与旧 UI 规范只作为历史页面范围参考，不作为视觉强约束。
- 本变更不修改业务能力边界、接口字段、CRM 白名单、权限模型、审计链路或企业微信能力清单。
- 登录、会话、企业微信扫码回跳兜底为稳定链路，视觉改造不得破坏相关逻辑和回归用例。

## Goals / Non-Goals

**Goals:**

- 将 `DESIGN.md` 补强为可指导具体页面落地的设计系统，而不是只描述“浅色、克制、可信”的方向。
- 建立业务辅助色、图表色板、图标图形、空状态、数据可视化和设计评审检查清单。
- 将智能分析、经营报表、查询模板、权限连接、合同审核、审计中心、AI 配置拆成独立页面改造基线。
- 形成前端实现路径：设计 token -> 图标映射 -> 图表组件 -> 页面模块 -> 响应式与截图验证。
- 保持现有业务行为、路由、store、service 和 API 契约稳定。

**Non-Goals:**

- 不重写后端接口、数据库、OpenAPI 契约或 CRM 接入方式。
- 不改变企业微信机器人支持能力、提示目录或自然语言理解主链路。
- 不引入第二套 Vue UI 组件库替代 Element Plus。
- 不把系统改造成营销页、深色大屏或高饱和炫技风格。
- 不在本提案阶段直接完成所有页面 UI 实现；实现由后续 apply 阶段分批完成。

## Decisions

### Decision 1: 以 `DESIGN.md` 作为唯一视觉基线，OpenSpec 固化验收口径

继续保留根级 `DESIGN.md` 作为 UI 设计入口，并在 `ui-design-system` 与 `web-ui-ux-redesign` delta spec 中补充强约束。这样后续协作者先读规格即可知道“为什么要改、改到什么程度、哪些不能动”。

替代方案：只在 `DESIGN.md` 中补充说明，不创建 OpenSpec 变更。
拒绝原因：文档容易被当成建议而不是实施口径，后续上下文丢失时无法指导任务拆解和验收。

### Decision 2: 使用“模块辅助色 + 品牌主色 + 状态色”的受控色彩模型

新增业务辅助色用于模块识别和图表序列，但限制同一页面默认只使用一个模块辅助色、一个品牌主色和必要状态色。这样可以解决“颜色单一”，同时避免演变成多品牌拼接。

替代方案：直接将页面背景、按钮和卡片改成多彩样式。
拒绝原因：会削弱企业级可信感，也容易破坏风险、成功、降级等语义状态的一致性。

### Decision 3: 图标图形服务业务识别，不作为泛装饰

统一通过 `@element-plus/icons-vue` 和项目级封装扩展导航图标、操作图标、状态图标、业务对象图标和等级图标。空状态、加载态、风险态和无权限态使用轻量图形或模块徽标，但每个图形必须表达业务对象、状态或优先级。

替代方案：引入随机插画库或大量装饰性 SVG。
拒绝原因：会增加视觉噪声和维护成本，也可能与 `DESIGN.md` 的克制可信方向冲突。

### Decision 4: 图表组件先升级语法，再决定是否引入新依赖

第一阶段优先在现有组件基础上补齐图表色板、横向排名条、迷你趋势线、圆环进度、文字摘要和空状态。若现有自绘组件无法满足复杂趋势、漏斗、矩阵或组合图，再单独评估引入 ECharts 等图表库，并另立依赖变更说明。

替代方案：立即引入大型图表库并替换所有图表。
拒绝原因：会扩大变更面，增加包体和迁移风险；当前问题首先是视觉语法和数据表达不完整。

### Decision 5: 页面按业务价值分批改造

实施顺序建议为：智能分析工作台 -> 查询模板 / 模板治理 -> 审计中心 -> 合同审核 -> 经营报表 -> 权限连接 -> AI 配置。先改最能体现系统门面的页面，再沉淀组件给其他页面复用。

替代方案：一次性同时重构所有页面。
拒绝原因：风险高，容易破坏既有权限、审计、导出、上传和登录链路，也不利于截图验收。

### Decision 6: 详情下钻优先使用抽屉，复杂长文保留独立页面

表格行详情、审计事件、模板对象、策略对象、合同风险项等上下文内下钻优先使用右侧抽屉。需要长时间阅读、复杂编辑、多对象对比或独立 URL 恢复的场景继续保留独立页面。

替代方案：所有详情都改成抽屉或所有详情都保留独立页面。
拒绝原因：前者会压缩复杂长文阅读，后者会让列表工作流跳转过重。

## Risks / Trade-offs

- **视觉补强演变成过度装饰** → 使用设计评审检查清单约束每个新增颜色、图标和图形必须服务业务识别、状态理解或数据比较。
- **页面改造破坏业务行为** → 任务拆分要求保留现有 `v-model`、事件绑定、store/service 调用、路由和权限判断；每阶段运行对应测试。
- **图标语义不足或重复** → 先扩展 `UiIcons` 的业务分类出口，再在页面中复用，不允许局部随意导入不一致图标。
- **图表升级带来响应式溢出** → 图表和表格必须在 `360px`、`768px`、`1200px`、`1440px` 下截图巡检，宽表只在局部容器滚动。
- **业务辅助色与状态色冲突** → 模块色只用于识别，风险 / 成功 / 降级 / 离线等状态必须优先使用语义状态色。
- **旧页面局部样式过多导致统一困难** → 先建立 token 和共享组件，再逐页替换，避免在每个页面复制样式。

## Migration Plan

1. **文档和 token 基线**
   - 完成 `DESIGN.md` 业务视觉语言补强。
   - 在 OpenSpec 中固化 `ui-design-system` 和 `web-ui-ux-redesign` 要求。
   - 梳理 `main.css` 和 `element-plus-theme.css` 中现有 token 与新增业务辅助色、图表色板的差距。

2. **基础组件与图标**
   - 扩展 `frontend/src/ui/icons.ts`，按导航、操作、状态、业务对象、等级建立稳定出口。
   - 建立模块徽标、对象图标标签、状态摘要条、业务空状态等可复用组件或样式。
   - 补齐按钮、图标按钮、tooltip、`aria-label`、状态标签的统一规则。

3. **数据可视化与表格**
   - 升级分析图表与经营报表图表的色板和展示语法。
   - 增加横向排名条、迷你趋势线、圆环 / 进度表达和文字摘要。
   - 优化表格主列对象识别、状态点、等级标签、固定操作列和更多菜单。

4. **页面分批实施**
   - 第一批：智能分析工作台、分析结果区。
   - 第二批：查询模板 / 模板治理、审计中心。
   - 第三批：合同审核、经营报表。
   - 第四批：权限中心、连接策略、AI 配置。

5. **验证与回归**
   - 每批页面至少运行 `pnpm --dir frontend lint`、相关单元测试和目标页面 E2E / 截图巡检。
   - 全量完成后运行 `pnpm --dir frontend test:unit`、`pnpm --dir frontend test:e2e`、`pnpm --dir frontend build`。
   - 对关键页面做桌面与窄屏截图检查，确认文字不重叠、按钮不溢出、标签不换行、表格不撑开页面。

## Open Questions

- 是否需要在后续实现阶段引入 ECharts 或其它图表库，取决于现有自绘图表升级后的表现。
- 是否需要为业务图形产出新的 `.pen` 原型或图形资产包，取决于页面改造评审要求。
- 每个业务模块的最终辅助色是否需要和公司品牌色或现有产品品牌进一步对齐，目前先按 `DESIGN.md` 新增色板执行。
