## Context

当前仓库里已经存在两套企业微信触达基础能力：

- 智能机器人长连接能力，当前用于企业微信问数、日报和受控写回等会话型场景。
- 企业微信自建应用配置能力，当前已提供可用的 `agentId`，并通过现有运行时配置装载企业微信应用凭证。

现有判断难点并不是“机器人能不能主动推送”。按照当前企业微信官方能力，智能机器人长连接已经支持主动推送，也支持把单聊用户 `userid` 当作单发目标；但它本质上仍是会话型产品形态，更适合回推到已有机器人上下文。相对地，自建应用消息天然更贴近“正式通知”模型，接收人支持成员、部门、标签，消息类型也更丰富，更容易形成稳定的运维与审计边界。

但现阶段仓库还没有一层统一的“主动通知交付”设计。结果是后续任何通知需求都会面临同一组问题：到底走机器人还是走自建应用、如何把 CRM 用户映射成企业微信接收人、是否要按当前权限再校验一次、失败后如何重试、以及怎么在审计里解释“为什么这条消息发给了这个组长”。

本次变更要解决的是“主动通知底座”本身，而不是某一个业务通知模板。像日报催报、次日团队汇总、治理复核等业务规则复杂、字段口径多、后续变化频繁，应该放到独立的业务提案中复用这套底座，而不应继续堆在基础能力提案里。

## Goals / Non-Goals

**Goals:**

- 为企业微信主动通知建立统一任务模型，避免后续每类通知各自接一条发送路径。
- 明确“企业微信自建应用消息”和“智能机器人主动推送”的分工与优先级。
- 明确通道选型依据是产品形态与治理边界，而不是单纯的“机器人是否支持单人定向”。
- 提供业务场景可复用的通知投递接口、接收人解析接口和发送治理能力。
- 在发送前补做当前权限校验，并把发送、失败、重试和拦截全部纳入审计。
- 首版允许复用当前已提供的企业微信自建应用配置，降低落地门槛，同时为后续独立通知应用预留切换空间。
- 让真实通知代码可以直接按正式逻辑落地，但默认受配置开关控制，测试阶段只向 `王亮2（WangLiang02）` 投递。

**Non-Goals:**

- 不在本次变更中直接实现所有通知模板，只定义统一底座和首批基础场景。
- 不在本次变更中定义日报、周报、告警等具体业务口径，这些业务规则应通过独立 OpenSpec change 承接。
- 不把机器人扩展成泛办公助手，不新增审批、待办、日历等与当前 CRM 智能分析无关的通用办公能力。
- 不新起一套独立的人员与组织权限模型，接收人、组长、组织关系仍优先复用 CRM 用户、角色、部门和企业微信映射链。
- 不承诺首版支持所有企业微信消息类型，首版只要求支持足够覆盖正式通知的 `markdown`/模板卡片能力。

## Decisions

1. 采用“通知任务中心 + 渠道适配器”结构，而不是把主动通知直接写进现有机器人服务。

   通知需求的共同点不是“来自哪个业务功能”，而是“都需要接收人解析、通道选择、去重、审计和失败重试”。因此应新增统一通知任务层，对上接受业务侧投递请求，对下路由到企业微信自建应用消息或智能机器人主动推送。这样后续日报、告警、异步结果、审计复核提醒都能复用同一套治理能力。备选方案是把每个场景直接写在 `wecom-bot.service.ts` 或某个业务服务里，但那样会快速形成分叉逻辑，难以维护。

2. 固定周期类、管理汇总类、指定成员正式通知默认优先走企业微信自建应用消息。

   这里的选型理由不是“机器人做不到单发”，而是自建应用消息更适合正式通知：接收人模型天然支持成员、部门、标签，消息类型上也更适合承载文本通知卡片、模板卡片、小程序通知等标准化消息。它更适合“日报汇总发组长”“异常提醒发责任人”“治理复核提醒发管理员”这类正式通知。首版直接复用当前已经提供的企业微信自建应用配置，其中 `agentId=1000271` 可作为官方应用消息发送通道的初始落点；后续如需从 Web 登录应用拆分为独立通知应用，只需要切换通知发送配置，不改业务任务模型。备选方案是所有通知一律走智能机器人主动推送，但这样会把“正式通知”和“会话内回推”混在一起，后续权限和运维口径会越来越模糊。

3. 智能机器人主动推送只承担“与机器人上下文强相关”的通知。

   机器人主动推送更适合异步分析完成提醒、长任务结果回推、用户刚在机器人里触发过的延迟结果通知，以及明确已有会话上下文的群聊提醒。它虽然支持单聊主动推送，但接收目标仍是“某个机器人单聊或群聊上下文”，不是正式通知体系里的组织化触达模型。它不应成为所有正式通知的默认通道，否则用户会在机器人会话里同时接收问数回复、日报补问、任务回推和固定周期提醒，阅读体验和治理口径都会变差。备选方案是继续把机器人当唯一企业微信通道，但不满足后续大量通知类需求的可治理性。

4. 接收人解析与发送前校验必须基于当前实时权限，而不是复用历史快照。

   仓库已有明确约束：常用查询、最近查询、导出和审计行为都必须遵循当前实时权限。主动通知同样不能绕过这条规则。对于任何管理汇总类或正式通知，发送前都必须重新校验目标接收人当前是否仍拥有查看该通知内容的权限；如果接收人映射失效、角色变化或组织归属已调整，系统必须阻断发送并记录原因。备选方案是生成通知时就把授权快照固化，但这会在权限变更后带来错发风险。

5. 首版配置策略采用“通知专用配置可选覆盖，未配置时回落到当前自建应用配置”，并增加真实发送开关。

   为了让方案能尽快落地，首版允许通知通道直接复用当前已经提供的企业微信自建应用 `corpId/secret/agentId` 组合；同时在运行时配置层预留 `WECOM_NOTIFY_*` 这一类专用配置位，并增加“消息通知真实发送开关”。真实发送开关关闭时，系统仍执行完整的通知生成、接收人解析和发送链路，但真实消息只允许改投测试接收人 `王亮2（WangLiang02）` 或直接走测试模式；只有显式开启开关后，才允许向真实业务对象发送。备选方案是开发期直接注释掉真实发送代码，但那会导致正式逻辑长期缺少真实校验。

6. 主动通知必须具备独立的幂等、重试、结果记录和审计事件。

   主动通知与会话内即时回复不同，它既可能由定时任务触发，也可能由异步事件触发，一旦缺少独立台账，就很难判断“这条提醒是否发过、为什么重发、为什么部分接收人失败”。因此通知任务、接收人快照、发送结果、重试次数和最终状态必须单独落库或单独留痕，不得仅依赖业务日志。备选方案是复用现有机器人下发记录，但那只能覆盖机器人链路，无法解释自建应用消息和聚合型通知。

## Risks / Trade-offs

- [复用当前自建应用会让登录与通知共用同一套应用凭证] -> 通过“通知专用配置可选覆盖”保留后续拆分空间，并在设计上避免把共享配置写死进业务逻辑。
- [组长关系口径如果不稳定，会导致日报汇总接收人不一致] -> 首版只允许复用 CRM 现有组织、部门和负责人归属关系，不额外引入自定义组长表。
- [权限实时校验会让部分原本可发的消息在发送时被阻断] -> 把阻断原因明确写入发送结果和审计，优先保证不越权、不误发。
- [同时维护两条企业微信发送通道会增加实现复杂度] -> 用统一通知任务中心封装差异，业务侧只面向单一投递接口。
- [机器人与自建应用都能单发，团队后续可能再次混淆选型] -> 在规格中显式写明“正式通知优先自建应用、强会话通知优先机器人”的判定原则，并把通道路由做成统一策略而不是人工约定。
- [开发联调时真实消息可能误发给业务人员] -> 增加配置目录内的真实发送开关，默认关闭，并在测试模式下固定投递到 `王亮2（WangLiang02）`。

## Migration Plan

1. 先补齐 OpenSpec，明确主动通知已从“排除范围”进入受控能力范围，但仅限批准场景。
2. 在后端增加通知任务中心、发送通道适配和运行时配置覆盖位，默认复用当前已提供的企业微信自建应用配置。
3. 后续业务场景如日报、告警、治理复核等，必须在独立提案中定义业务口径后再接入该通知底座。
4. 再逐步接入异步结果提醒、异常告警和治理复核提醒等其他通知场景。

回滚时只需要关闭主动通知任务入口和定时触发器，保留已存在的机器人问数、日报会话和 Web 登录链路不变。

## Open Questions

- 后续业务场景的接收人映射，是统一通过组织映射服务抽象，还是允许各业务提案引入自己的规则提供器。
- 真实发送开关与测试收件人配置，是仅应用于企业微信主动通知，还是应扩展为更通用的消息安全开关。
