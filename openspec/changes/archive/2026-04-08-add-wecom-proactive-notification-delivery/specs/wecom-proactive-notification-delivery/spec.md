## ADDED Requirements

### Requirement: 系统必须提供统一的企业微信主动通知任务与通道路由能力

系统 MUST 提供统一的企业微信主动通知任务模型，由业务场景先生成标准化通知任务，再由通知中心选择具体发送通道。对于固定周期类、管理汇总类和指定成员正式通知，系统 MUST 默认优先选择企业微信自建应用消息；对于与机器人会话强耦合的异步结果回推、继续追问入口和已有群聊上下文提醒，系统 MUST 选择智能机器人主动推送。业务侧不得直接绕过通知中心分别调用两条发送链路。

#### Scenario: 团队汇总类正式通知默认走自建应用消息
- **WHEN** 系统生成某条面向指定负责人的团队汇总类正式通知任务
- **THEN** 通知中心必须将该任务路由到企业微信自建应用消息通道
- **THEN** 业务服务不得直接调用机器人发送接口完成该场景的正式通知

#### Scenario: 机器人异步结果回推走智能机器人主动推送
- **WHEN** 用户在企业微信机器人中发起了需要延迟返回结果的任务，且结果完成后需要回推到原有机器人上下文
- **THEN** 通知中心必须将该任务路由到智能机器人主动推送通道
- **THEN** 系统必须保留原始机器人上下文信息，避免把该类通知误发为正式应用消息

#### Scenario: 机器人虽可单发但正式通知仍优先自建应用
- **WHEN** 某条正式通知同时满足“机器人可单聊主动推送”和“自建应用可定向发送”的技术条件
- **THEN** 系统仍必须按正式通知规则优先选择企业微信自建应用消息通道
- **THEN** 系统不得仅因机器人也能单发就把正式通知改走机器人链路

### Requirement: 企业微信自建应用消息必须支持按成员定向发送正式通知

系统 MUST 支持通过企业微信自建应用按成员定向发送正式通知。首版 MUST 支持使用当前已提供的企业微信自建应用配置完成发送；若后续配置了通知专用应用参数，系统 MUST 优先使用通知专用配置。系统 MUST 能够记录无效成员、未授权成员和未开通成员等发送结果，不得把部分失败静默吞掉。

系统 MUST 将自建应用消息作为正式通知通道使用，其理由包括但不限于：接收人模型支持成员、部门、标签三类组织化对象，且消息形态能够承载正式通知所需的标准卡片和后续扩展入口。系统不得把自建应用通道简化为“只是机器人单发的备选方案”。

#### Scenario: 单个负责人收到正式汇总通知
- **WHEN** 某位负责人被解析为某条正式汇总通知的唯一接收人
- **THEN** 系统必须使用企业微信自建应用向该负责人定向发送单条正式汇总通知
- **THEN** 发送结果中必须记录该负责人的企业微信接收标识与最终发送状态

#### Scenario: 部分接收人无效时保留部分失败结果
- **WHEN** 某个正式通知任务包含多个成员接收人，且其中部分成员在企业微信返回为无效或未授权
- **THEN** 系统必须记录成功接收人与失败接收人的分项结果
- **THEN** 系统不得把整批任务误记为全部成功

#### Scenario: 管理汇总类通知需要部门或标签扩展能力
- **WHEN** 后续某类正式通知需要从“单人定向”扩展为“按部门”或“按标签”发送
- **THEN** 系统必须继续沿用企业微信自建应用消息通道承接该类扩展
- **THEN** 系统不得要求业务方改造为机器人会话模型后才能发送

### Requirement: 智能机器人主动推送必须支持单聊与既有群聊上下文通知

系统 MUST 支持通过企业微信智能机器人主动推送消息到单聊用户或既有群聊会话。单聊主动推送时，系统 MUST 使用用户 `userid` 作为单聊目标标识；群聊主动推送时，系统 MUST 使用既有群聊回调事件中获得的 `chatid`。系统不得在没有可靠群聊上下文标识的情况下尝试群发通知。

#### Scenario: 单聊用户收到机器人主动结果提醒
- **WHEN** 某条机器人异步任务已经完成，且目标接收人为单聊用户
- **THEN** 系统必须使用该用户的企业微信 `userid` 作为单聊主动推送目标
- **THEN** 推送内容必须与原任务关联，便于用户继续回到机器人上下文

#### Scenario: 缺少群聊上下文时不得冒险发送
- **WHEN** 某条机器人相关通知需要发往群聊，但当前没有可靠的 `chatid`
- **THEN** 系统必须阻断该次群聊主动推送
- **THEN** 系统必须记录“缺少群聊上下文标识”的失败原因，而不是改发给错误对象

### Requirement: 正式通知接收人解析必须基于当前权限与组织映射发送

系统 MUST 基于当前实时组织关系、负责人归属和企业微信身份映射来解析正式通知的接收人。发送前，系统 MUST 重新校验目标接收人当前是否仍具备查看该通知内容的权限；若接收人映射失效、组织归属变化或权限已撤销，系统 MUST 阻断发送。业务提案可以定义各自的接收人规则，但不得绕开该统一解析与校验过程。

#### Scenario: 业务通知通过统一接收人解析流程投递
- **WHEN** 某条正式通知已经生成候选接收人，需要进入发送流程
- **THEN** 系统必须先通过统一接收人解析与权限校验流程确认最终接收人
- **THEN** 业务模块不得直接跳过该流程调用底层发送适配器

#### Scenario: 发送时发现接收人权限已失效
- **WHEN** 某条正式通知准备发送给目标接收人，但发送前校验发现该接收人当前已不再拥有查看对应通知内容的权限
- **THEN** 系统必须阻断该次发送
- **THEN** 系统必须记录权限阻断原因并进入审计，而不是沿用旧授权快照继续放行

### Requirement: 真实消息通知必须受配置开关控制并支持测试收件人覆盖

系统 MUST 在真实消息发送前检查配置目录中的“消息通知真实发送开关”。当开关未开启时，系统 MUST 不得向真实业务对象发送主动通知；对于需要联调真实消息内容的测试场景，系统 MUST 允许把消息统一改投到测试收件人 `王亮2（WangLiang02）`。真实功能代码 MUST 直接按正式逻辑实现，不得以“测试环境”为理由删除或注释真实通知发送分支。

#### Scenario: 真实发送开关关闭时改投测试收件人
- **WHEN** 开发或测试环境触发了真实消息联调，但配置中的真实消息发送开关仍为关闭状态
- **THEN** 系统必须阻止向真实业务对象发送
- **THEN** 系统必须按测试策略将消息改投给 `王亮2（WangLiang02）` 或进入等效测试模式

#### Scenario: 开关开启后才允许真实业务发送
- **WHEN** 配置目录中的真实消息发送开关被显式开启，且接收人映射、权限校验均通过
- **THEN** 系统才允许向真实业务对象发送主动通知
- **THEN** 系统必须在发送记录中标记该次投递属于真实业务发送

### Requirement: 主动通知必须具备独立的幂等、重试与审计留痕

系统 MUST 为主动通知提供独立的幂等键、发送记录、有限重试和审计留痕能力。对于同一业务任务在同一目标、同一时间窗内重复触发的通知，系统 MUST 能识别并阻止重复发送。对于短暂网络抖动或企业微信接口临时失败，系统 MUST 执行有限次重试；连续重试后仍失败时，系统 MUST 保留最终失败状态和失败原因，不得只在控制台输出错误。

#### Scenario: 重复任务不重复发消息
- **WHEN** 同一条日报汇总任务因调度重试被重复触发，且目标组长和消息版本均未变化
- **THEN** 系统必须命中幂等保护并阻止重复发送
- **THEN** 系统必须在发送记录中标记该次请求为幂等命中

#### Scenario: 连续重试后仍失败需保留失败审计
- **WHEN** 主动通知在有限次重试后仍然发送失败
- **THEN** 系统必须保留最终失败状态、失败原因和重试次数
- **THEN** 系统必须生成对应的审计事件，便于后续排查和补发
