## ADDED Requirements

### Requirement: 系统必须将 CRM 自然语言问题编排为受控分析工作流
系统 MUST 先将用户输入的 CRM 自然语言问题解析为标准化分析工作流计划，再把工作流拆分为一条或多条受控查询任务。每条查询任务 MUST 明确其用途、使用的数据对象、指标、维度、过滤条件和预期结果形态，且不得直接把模型自由生成的原始 SQL 作为可执行输入。

#### Scenario: 简单问题被拆分为单条查询任务
- **WHEN** 用户输入“本月各销售负责人新增商机金额排名”
- **THEN** 系统必须生成一条明确的分析工作流和一条只读查询任务，用于返回该问题所需的核心结果

#### Scenario: 复合问题被拆分为多条查询任务
- **WHEN** 用户输入“本季度签单金额趋势、区域排名和重点客户明细”
- **THEN** 系统必须将问题拆分为多条用途明确的查询任务，并在统一工作流中按顺序或并行执行

### Requirement: 系统必须将每条查询任务编译为受控只读 SQL
系统 MUST 对每条查询任务执行受控 SQL 编译，并且生成结果 MUST 仅访问一期批准的数据对象、字段、JOIN 路径和函数白名单。系统 MUST 自动注入当前用户的组织、部门、负责人等权限范围，并 MUST 拒绝任何写入、更新、删除、DDL 或绕过白名单的语句。

#### Scenario: 查询任务被编译为带权限范围的只读 SQL
- **WHEN** 已授权用户发起一条允许执行的 CRM 分析请求
- **THEN** 系统必须为每个查询任务生成只读 SQL，并自动附加当前权限范围条件

#### Scenario: 非法 SQL 在编译或校验阶段被阻断
- **WHEN** 任一查询任务试图引用未批准字段、非法 JOIN、写入语句或高风险片段
- **THEN** 系统必须停止该任务执行，并向用户返回明确的中文阻断原因

### Requirement: 系统必须对多次查询结果组装统一数据集
系统 MUST 将单次分析流程中全部查询任务的执行结果组装为同一份统一数据集，并为每个结果块记录来源任务、数据集标识和用途说明。摘要、关键指标、图表、表格和导出内容 MUST 基于这份统一数据集生成，不得混用不同批次、不同口径或未校验的数据来源。

#### Scenario: 多条查询结果被合并为统一数据集
- **WHEN** 一次分析包含趋势查询、排名查询和明细查询
- **THEN** 系统必须将这些结果合并为一份统一数据集，并保留各结果块与来源任务的对应关系

#### Scenario: 数据集来源不一致时阻止正式结果返回
- **WHEN** 系统发现摘要、图表或表格引用了不同批次或无法校验一致性的查询结果
- **THEN** 系统必须阻止正式分析结果返回，并提示当前结果需要重新确认条件或重新执行

### Requirement: 系统必须基于统一数据集生成 AI 分析报告
系统 MUST 在统一数据集构建完成后生成 AI 分析报告，并且报告中的结论、关键发现、数值引用和建议 MUST 仅使用已确认的数据集事实。系统 MUST 在报告返回前执行关键指标回算、结果一致性检查和低置信度拦截，无法保证准确性时必须优先补问或拒答。

#### Scenario: 统一数据集生成结构化分析报告
- **WHEN** 查询任务全部执行成功且统一数据集通过一致性校验
- **THEN** 系统必须返回包含标题、摘要、关键发现、关键指标、图表或表格块的分析报告

#### Scenario: 结果准确性无法保证时拒绝生成报告
- **WHEN** 系统发现关键指标无法回算、重复分组、主指标为空或 AI 报告引用了未确认事实
- **THEN** 系统必须阻止正式报告输出，并给出明确的失败或补问提示

### Requirement: 系统必须让 Web 与企业微信复用同一套分析编排结果
系统 MUST 对 Web 工作台与后续企业微信机器人复用同一套分析工作流、查询任务执行结果和统一数据集，不得为不同渠道维护分叉的查询逻辑。渠道差异 MUST 仅体现在输入适配、结果裁剪和展示形式上，而不得改变分析结论的数据来源。

#### Scenario: 不同渠道共享同一分析主链路
- **WHEN** Web 工作台与企业微信机器人分别提交相同的授权问题
- **THEN** 系统必须使用同一套分析编排逻辑与数据集来源，并保证核心结论口径一致

#### Scenario: 渠道差异仅影响展示层
- **WHEN** 企业微信因展示限制无法完整承载图表或表格
- **THEN** 系统必须仅裁剪展示形式，而不得重新生成另一套不一致的业务分析逻辑
