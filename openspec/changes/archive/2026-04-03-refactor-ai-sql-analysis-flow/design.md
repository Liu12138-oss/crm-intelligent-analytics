## Context

当前实现已经具备 `AnalysisIntentService`、`QueryCompilerService`、`QueryAstValidatorService`、`QueryPreflightService`、`ResultStreamerService` 和 `AnalysisWorkbenchPage.vue` 等基础模块，但核心问题仍然明显：

- `backend/src/modules/analysis/analysis.service.ts` 仍将语义识别、查询编译、执行、样例数据兜底、结果拼装和审计揉在同一个服务中，难以支撑“多次查询 -> 汇总分析 -> 报告输出”的新主流程。
- `QueryPlanAst` 与 `CompiledQuery` 目前默认只表达单条 SQL，无法清晰描述“主查询 + 对比查询 + 明细查询”这类多步分析链路。
- `AnalysisIntentService` 仍然偏向规则补全与单次结果类型推断，AI 只参与了有限的结构化识别，没有形成“受控查询规划 + 最终报告生成”的完整作用边界。
- `AnalysisResultRecord` 与前端 `AnalysisQueryResult` 仍以单主图、单表格、说明文本为中心，不足以表达 AI 分析报告、关键发现、图表/表格组合以及数据引用关系。
- `frontend/src/pages/analysis/AnalysisWorkbenchPage.vue` 和 `analysis-query.store.ts` 目前还能工作，但状态仍偏散，且页面同时承载较多辅助卡片，和“报告优先、反馈优先”的目标不完全一致。

本次设计必须同时满足以下约束：

- 一期仍然只能访问 CRM 只读白名单数据，不能出现写入、更新、删除、DDL 或绕过权限模型的路径。
- Web 与企业微信后续必须复用同一套后端分析编排结果，不能在两个渠道各写一套查询与报告逻辑。
- 页面结构必须遵循既有 `.pen` 原型的一级分区，不允许脱离主信息架构自行新增核心分区。
- 按仓库协作规范，文档、用户提示和代码注释都应优先使用中文；关键代码与关键判断必须补充中文注释。

## Goals / Non-Goals

**Goals：**

- 将分析链路重构为“自然语言理解 -> 查询任务规划 -> 多次只读 SQL 执行 -> 统一数据集 -> AI 报告生成”的稳定主流程。
- 明确 AI 的职责边界：负责语义理解、报告总结和表达增强，不负责自由生成可直接执行的 SQL。
- 让 Web 工作台与企业微信机器人共享同一份分析结果模型，由渠道适配层决定如何渲染。
- 让最终结果以报告为中心输出，同时补齐加载中、成功、失败、空数据、重试建议等反馈体验。
- 为清理无关能力提供落地路径，逐步淘汰当前分析模块中与真实 SQL 报告流程不一致的冗余代码分支。

**Non-Goals：**

- 不在本次设计中扩展新的业务主题，仍然限定在一期批准的商机、合同、客户等 CRM 分析域。
- 不在本次设计中开放写入型能力、自动创建业务动作或自由表设计器。
- 不要求企业微信在一期就完整复刻 Web 图表体验，企业微信可只返回摘要化报告和跳转入口。
- 不把原始完整 SQL 默认直接展示给普通用户，避免暴露不必要的底层结构细节。

## Decisions

### 决策一：引入统一分析编排器，拆开“理解、查询、汇总、报告、呈现适配”五层职责

新增后端核心编排层，建议落位为 `backend/src/modules/analysis/analysis-workflow.orchestrator.ts` 或等价服务，并将当前 `AnalysisService` 从“大而全服务”收敛为接口协调层。新编排器按固定阶段推进：

1. 语义理解：`AnalysisIntentService` 输出标准化问题语义。
2. 查询任务规划：新建 `AnalysisQueryPlannerService`，把问题拆成一到多条查询任务。
3. 查询执行：`QueryCompilerService`、白名单校验、AST 校验、预检与执行器协作完成只读 SQL 执行。
4. 数据汇总：新建 `AnalysisDatasetAssemblerService`，把多次查询结果整理成统一数据集。
5. 报告生成：新建 `AnalysisReportComposerService`，基于统一数据集生成 AI 报告与结构化展示块。

这样做的原因是：当前单个 `AnalysisService` 里混杂太多职责，后续只要增加多次查询、企业微信适配或报告引用校验，就会持续恶化可维护性。

备选方案是继续在 `AnalysisService` 内堆叠逻辑，仅增加几个辅助函数。该方案短期改动少，但会让多次查询和报告生成更加难以测试，因此不采用。

### 决策二：把单条 `QueryPlanAst` 扩展为“工作流计划 + 查询任务列表”，并限制每次分析的查询步数

保留现有 `QueryPlanAst` 作为“单个查询任务”的底层结构，但在其外层新增 `AnalysisWorkflowPlan` 与 `AnalysisQueryTask`：

- `AnalysisWorkflowPlan`：描述问题目标、报告目标、渠道、风险等级、补问状态、任务列表。
- `AnalysisQueryTask`：描述单次查询用途，如 `primary-metric`、`comparison`、`detail-sample`、`trend-series`。
- 每个任务单独经过编译、白名单校验、AST 校验和预检，执行结果带 `datasetId` 与用途标签返回。

为了控制成本和复杂度，一期默认将单次分析的查询任务数限制在 `3` 条以内；若超出，需要先补问缩小问题或拒绝执行。这样能覆盖“总览 + 趋势 + 明细”这类常见场景，同时避免无边界扩张。

备选方案一是继续只支持单条 SQL，并强迫把所有问题塞进一次查询。该方案对复合问题支持不足。备选方案二是允许模型自由决定无限条查询。该方案不可控，也会显著增加时延和资源风险，因此不采用。

### 决策三：AI 只负责语义理解与报告表达，SQL 与结果视图骨架继续保持确定性生成

本次链路中，AI 允许参与两处：

- 将自然语言收敛为受控语义、查询目标和任务规划建议。
- 基于统一数据集生成中文分析摘要、关键发现、风险提示和建议动作。

以下内容仍由确定性逻辑生成：

- SQL 结构与实际执行文本
- 白名单、权限、只读、JOIN、聚合、排序等校验逻辑
- 指标卡、图表数据、表格数据和引用关系
- 报告中涉及的具体数值来源与一致性校验

报告生成阶段采用“结构化数据 + AI 文案增强”的混合模式：系统先生成结构化指标、图表、表格和事实摘要，再把这些已确认事实喂给 AI 输出自然语言结论。这样可以降低 AI 在数值层面幻觉的风险。

备选方案是让 AI 直接输出 SQL 和整份 JSON 报告。该方案实现看似简单，但会削弱可校验性，与本项目的只读、安全和准确性要求冲突，因此不采用。

### 决策四：定义统一报告结果模型，Web 与企业微信通过渠道适配器消费同一份结果

新增统一结果模型 `AnalysisReportPayload`，用于逐步取代当前前后端以 `primaryView`、`secondaryViews`、`tableRows` 为主的单层结构。建议统一模型至少包含：

- `reportTitle`
- `executiveSummary`
- `keyFindings[]`
- `metricCards[]`
- `chartBlocks[]`
- `tableBlocks[]`
- `datasetReferences[]`
- `streamBlocks[]`
- `emptyState` / `errorState`
- `availableActions[]`

Web 工作台直接渲染完整报告结构；企业微信渠道通过 `AnalysisChannelPresenterService` 将同一份报告压缩成摘要块、关键指标块和跳转提示，不重新生成业务逻辑。

为降低切换成本，后端可以在过渡期由 `AnalysisResponseMapper` 把新报告模型映射成当前接口字段，再逐步让前端切到新字段。这样能避免一次性同时重写全部组件。

备选方案是继续维护 Web 与企业微信两套响应结构。该方案会快速导致行为分叉，因此不采用。

### 决策五：前端改为“输入区 + 报告区 + 过程反馈区 + 轻量辅助区”的状态机页面

前端工作台保留原型一级分区，但交互上切换为更明确的状态机：

- `idle`：未开始分析，输入框与查询资产优先。
- `clarifying`：存在补问，突出待确认条件与继续提交流程。
- `running`：展示阶段进度、局部骨架屏、按钮禁用、处理中提示。
- `reported`：展示完整分析报告、流式过程和导出/追问动作。
- `blocked` / `failed`：展示失败原因、下一步建议和重试入口。

`analysis-query.store.ts` 将围绕该状态机收敛状态，避免继续堆叠多个松散布尔值。页面交互要求补齐：

- 鼠标悬浮、点击、禁用态、加载态
- 接口成功/失败即时反馈
- 空结果与无权限文案
- 轻量动效，如报告区切换、流式块追加、提交按钮反馈

备选方案是继续沿用当前“结果有无 + 若干布尔值”的方式。该方案勉强能工作，但不利于后续叠加补问、多次查询和统一报告呈现，因此不采用。

### 决策六：清理冗余代码采用“同接口替换 + 观察期 + 最终删除”的方式推进

用户明确要求删除不需要的前后端功能或代码，但当前仓库仍需要一定的开发与联调可用性，因此采用分阶段清理：

1. 新流程先通过统一接口接入，不立即删除旧代码。
2. 对现有 `executeMockAnalysis`、结果拼装辅助函数、页面次要卡片逻辑等进行归类。
3. 能被新流程完全覆盖的逻辑，在回归通过后删除。
4. 开发态仍需保留的演示或样例能力，移入显式的适配层或测试夹具，禁止继续驻留在正式业务服务中。

这样可以满足“砍掉无关代码”的目标，同时避免一次性硬删导致本地联调和回归验证失控。

备选方案是立即删除所有旧分支。该方案风险过高，也不利于定位问题，因此不采用。

## Risks / Trade-offs

- [多次查询会带来更高时延与调度复杂度] → 通过限制任务步数、并行执行无依赖查询、为每个任务记录用途和耗时来控制复杂度。
- [AI 报告生成仍可能出现措辞过度推断] → 只向 AI 提供已确认事实摘要，并对关键数值做引用与回算校验，数值冲突时直接阻断结果。
- [前后端结果模型升级可能带来一次兼容期] → 先通过 `AnalysisResponseMapper` 做兼容映射，再逐步切换前端组件消费新结构。
- [清理旧代码时容易误删仍在使用的能力] → 先补测试和调用链盘点，再按模块删除，并在删除前确认无企业微信或治理端依赖。
- [企业微信与 Web 复用同一结果结构后，移动端承载能力有限] → 通过渠道适配器裁剪展示层，而不是裁剪业务结果本身。
- [工作台简化后，部分辅助信息可能不如现在显眼] → 保留轻量摘要入口与详情展开方式，保证信息可达但不抢占核心注意力。

## Migration Plan

1. 先扩展后端类型与服务边界，引入工作流计划、查询任务、统一数据集和报告模型，但暂时保持当前接口可用。
2. 在 `AnalysisService` 中接入新的编排器，并通过兼容映射返回现有接口字段；同时补齐多次查询、报告一致性和失败反馈测试。
3. 重构前端 store 与工作台页面，让其优先消费新报告模型或兼容映射结果，并完成输入区、反馈区和报告区的布局调整。
4. 为企业微信预留渠道适配器，确保其未来直接复用同一报告结果结构。
5. 在联调与回归通过后，删除已被新流程完全覆盖的旧逻辑和无关 UI 模块。

回退策略：

- 若新流程上线后出现问题，可先在后端入口层切回旧编排路径，同时保留新服务代码不删除，便于后续继续修复。
- 前端工作台若存在严重展示异常，可先回退到旧页面布局，但保留新结果模型适配层，避免后端再次分叉。

## Open Questions

- 最终是否向普通用户展示“查询步骤摘要”或“SQL 摘要说明”，需要在 UI 联调时确认展示深度。
- 单次分析允许的最大查询任务数是否固定为 `3`，还是对管理员开放更高上限，需要结合性能验证再定。
- AI 报告中的图表类型选择是完全由规则引擎决定，还是允许 AI 在受限集合中推荐，需要在实现前再确认。
- 企业微信一期是否只返回摘要 + 关键指标 + Web 跳转入口，还是需要同时返回简化表格，需要产品与业务一起确认。
