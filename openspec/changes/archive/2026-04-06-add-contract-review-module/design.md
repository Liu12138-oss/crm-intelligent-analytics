## Context

当前 CRM-Agent 已经具备统一登录、左侧导航框架、Web 智能分析工作台、AI 网关配置装载、应用库存储层和审计事件检索能力，但这些能力仍主要围绕“自然语言问数”构建。合同审核场景与问数场景存在明显差异：

- 输入介质从文本问题变为文件上传；
- 主链路从受控查询变为“文件解析 + 规则快照 + AI 审核 + 产物生成”；
- 结果不只是页面展示，还包括审核报告、带批注合同和结构化结果文件；
- 权限控制粒度从“当前查询结果”转为“任务、问题明细和产物下载”。

仓库中已经存在合同审核参考 skill 与公司规则基线，但尚未形成适配当前项目技术栈的正式模块方案。本次设计需要在不打破现有 `Vue 3 + NestJS + AppShell + auth + audit` 基线的前提下，引入一个新的业务模块，并保证以下约束成立：

- 不新起独立系统，不另造一套登录、导航、审计或配置体系；
- 一期仅做 Web 端 `.docx` 合同审核，不扩展到企业微信文件审核；
- 合同审核模块的数据全部落入应用库存储层，不写入 CRM 业务库；
- 敏感合同正文只能在授权上下文中访问，不能进入普通日志或开放下载地址；
- 新能力后续需要能够补齐 OpenAPI、`.pen` 原型和实现任务，而不是停留在说明文档层面。

本次设计涉及的主要干系人包括商务人员、销售协作人员、法务复核人员、审计管理员，以及负责前端工作台、后端模块和存储层改造的研发团队。

## Goals / Non-Goals

**Goals:**

- 在现有 Web 系统内新增 `智能合同审核` 一级入口，并形成独立但复用基础设施的业务模块。
- 建立“上传合同 -> 创建审核任务 -> 文本提取 -> 规则快照 -> AI 审核 -> 产物生成 -> 结果查看 -> 审计留痕”的稳定主链路。
- 将前端主路径收敛为“首页上传 -> 自动进入审核详情 -> 返回首页查看最近任务”，减少与风险判断无关的页面噪音。
- 为审核结果提供结构化问题模型，明确风险等级、一票否决项、规则依据和原文定位信息。
- 对任务详情、产物下载和历史记录施加清晰的访问控制与审计规则。
- 以最小可用方式落地一期 MVP，避免在首版同时引入审批流、OCR、批量审核和复杂规则管理后台。

**Non-Goals:**

- 不在本次设计中引入企业微信上传合同、扫码发起审核或移动端审核流程。
- 不建设合同审批、盖章、回款、签署回写或任何 CRM 写入型流程。
- 不在首版中支持 PDF、图片扫描件、多文件批量审核或 RAG 检索。
- 不将本地 `SKILL.md` 直接作为生产运行时主入口，只将其视为规则整理参考。
- 不修改现有问数 `analysis` 模块的核心职责，也不把文件审核逻辑塞入其中。
- 首版工作台不单独建设统计卡、处理中任务专区或历史任务筛选搜索面板。

## Decisions

### 决策 1：新增独立 `contract-review` 模块，而不是复用 `analysis` 模块承载文件审核

后端将新增独立 `contract-review` 模块，负责合同上传、任务编排、规则快照、AI 审核、结果聚合和产物元数据管理。该模块复用 `auth`、`audit`、`shared/config` 和应用库存储层，但不复用 `analysis` 的核心业务流。

原因：

- `analysis` 面向自然语言问数，其状态机、数据访问方式和结果结构都围绕受控查询设计，不适合直接承载文件处理链路。
- 将文件审核单独建模更利于后续扩展文件解析器、产物生成器和规则集版本管理。
- 这能避免后续维护时出现“问数逻辑”和“合同审核逻辑”强耦合的问题。

备选方案：

- 直接在 `analysis` 模块中新增文件入口。问题是会让问数与文件审核共享不适配的实体和状态。
- 做成独立微服务。问题是一期改动和部署复杂度过高，不符合当前单仓 MVP 节奏。

### 决策 2：采用异步审核任务模型，上传请求只负责创建任务并驱动详情页主链路

首版合同审核采用“快速创建任务 + 后续查询状态/详情”的异步模型。上传接口在完成文件校验和任务落库后立即返回任务 ID，由前端立即跳转到该任务的结果详情页，并在详情页轮询任务状态或刷新页面获取进度与结果。

原因：

- 文件解析、AI 审核、报告生成和批注文档生成都属于长耗时操作，不适合绑在单个 HTTP 请求生命周期内。
- 异步任务模型更容易向前端展示 `PARSING`、`REVIEWING`、`GENERATING_REPORT` 等阶段状态，同时避免首页堆叠处理中模块。
- 它也更适合失败重试、断点排查和后续接入真正的后台队列。

备选方案：

- 同步等待整个审核流程完成。问题是请求超时风险高，用户感知差。
- 首版直接上独立消息队列与 worker。问题是基础设施投入过重，不利于快速落地 MVP。

### 决策 3：规则集以“版本化快照”方式进入任务，而不是运行时直接消费 skill 文本

系统会把当前生效的合同审核规则整理为规则集对象，并在任务创建时固化 `rule_set_code`、版本号和规则快照摘要。审核执行只读取该快照，不在生产链路中动态解析本地 `SKILL.md`。

原因：

- 规则快照是结果可解释、可复现、可审计的前提。
- 直接读取 skill 文本会让运行时依赖不可控，也难以说明本次任务到底依据了哪一版规则。
- 版本化快照为后续引入多规则集、规则变更记录和审核复盘留出了空间。

备选方案：

- 每次任务直接读取本地 markdown 规则文件。问题是无法清晰追踪规则版本。
- 首版直接建设在线规则编辑器。问题是超出 MVP 范围。

### 决策 4：文件与产物采用“文件存储 + 元数据落库”双层模型

首版采用后端管理的文件存储目录保存原始合同与生成产物，在应用库存储层落任务、问题、产物元数据和审计关联。下载必须经过后端鉴权接口，不提供裸文件直链。

原因：

- 当前仓库还没有成熟的对象存储抽象，采用本地或挂载目录最符合一期节奏。
- 元数据落库可以支撑任务列表、详情页、产物状态显示和审计追踪。
- 鉴权下载可以避免产物地址泄露后被绕过权限直接访问。

备选方案：

- 全量文件内容直接落数据库。问题是体积大、读取和迁移成本高。
- 直接生成静态可访问下载链接。问题是安全边界过宽。

### 决策 5：把“结构化问题结果”和“可下载产物”分离建模

审核任务完成后，系统会首先产出结构化问题集合，再由独立产物生成阶段输出审核报告、带批注合同和结构化结果文件。即使某个产物生成失败，只要问题集合已经稳定生成，任务详情仍可展示结构化审核结果，并单独标记产物失败状态。

原因：

- 问题集合是页面展示、搜索、统计和复核的基础事实，不能与文档导出过程强绑定。
- 带批注文档生成技术风险高于纯结构化结果，分离后可以降低单点失败影响。
- 这样更容易对不同产物类型进行重试或替换实现。

备选方案：

- 只有全部产物都生成成功才算任务成功。问题是会放大批注生成失败对核心结果展示的影响。
- 只生成页面结果，不生成文档产物。问题是不满足业务目标。

### 决策 6：任务可见性默认按“上传者本人可见”，通过角色扩展审计查看权限

首版访问控制采用保守策略：

- 审核任务和问题详情默认仅上传者本人可见；
- 审计管理员或后续明确授权角色可查看审计事件和必要的任务元数据；
- 产物下载必须再次校验当前用户与任务的访问关系；
- 列表接口和普通日志默认不返回完整合同正文。

原因：

- 合同内容敏感度明显高于普通问数结果，默认按本人可见更安全。
- 审计与复核角色往往只需要可追踪性，不一定需要所有正文内容默认外露。
- 这为后续是否开放部门管理员查看权限留下了增量空间。

备选方案：

- 默认同部门全员可见。问题是敏感边界过宽。
- 首版引入复杂 ACL。问题是实现和验收成本偏高。

### 决策 7：结果页采用“风险优先、信息降噪”的单列展示策略

合同审核结果页将优先服务商务人员的快速风险判断，而不是完整暴露所有审核上下文。页面采用风险优先的单列结构，默认只突出以下内容：

- 总体结论与一票否决提示；
- 高、中、低风险数量；
- 每条风险的简短标题、风险等级、问题描述与建议动作；
- 必要的原文片段。

规则编码、模板来源、技术状态、内部标签、长路径依据等低频信息默认折叠到“查看依据”或“展开详情”中，不在首屏大面积铺开。

原因：

- 商务人员首先要判断“能不能签、哪里有风险、下一步改什么”，而不是阅读模型命中路径和规则技术标识。
- 当前示意图的主要问题不是信息不全，而是把风险信息与元数据、标签、编码混在同一视觉层级，导致注意力分散。
- 单列风险卡片更适合快速扫描，也更利于后续按“必须修改 / 建议修改 / 可接受”分组。

备选方案：

- 保留多标签、多分栏、全信息展开的卡片形式。问题是商务用户阅读负担高，风险主次不清。
- 只展示摘要，不提供依据入口。问题是后续复核与争议解释能力不足。

### 决策 8：工作台首页采用极简结构，只保留上传入口与最近审核任务

合同审核工作台首页将不再承载统计卡、处理中任务专区、高风险待处理专区或复杂筛选区域。首页默认只保留以下两块内容：

- 上传合同入口；
- 最近审核任务列表。

上传成功后，前端必须自动进入对应任务的审核详情页；用户在详情页点击“返回首页”后，再回到极简首页查看最近审核任务。最近审核任务列表需要把刚完成或刚创建的审核记录回写到顶部，确保用户回首页时可以立刻再次进入该任务。

原因：

- 商务人员进入合同审核模块的第一目的不是浏览运营状态，而是尽快提交合同并看到风险结果。
- 在首页同时展示统计、处理中、高风险待处理等信息，会把注意力从“当前要审哪份合同”转移到次要信息上。
- 将处理中状态收拢到任务详情页，可以让首页信息架构稳定，也更符合“上传后自动进入详情”的单线流程。

备选方案：

- 在首页保留统计卡、处理中任务和高风险待处理专区。问题是会增加视觉噪音，削弱上传入口和最近审核记录的主导性。
- 上传后停留在首页，由用户自行进入详情页。问题是多了一次跳转判断，不利于快速查看风险结果。

## Risks / Trade-offs

- [带批注 Word 生成实现复杂] → 先把问题结果与产物生成分离，确保结构化结果可先稳定交付，批注文档以独立产物状态管理。
- [异步任务会增加前端状态处理复杂度] → 用明确的任务状态枚举、详情页轮询机制和上传后自动跳转降低前端分支混乱。
- [过度简化结果页可能影响复核信息获取] → 将规则依据、编码和来源信息收敛到可展开区域，而不是彻底删除。
- [工作台过度极简可能影响批量跟踪效率] → 首版先优先保障上传与查看风险主链路，后续若业务确认有需要，再补充筛选和批量管理能力。
- [本地文件存储不利于后续横向扩展] → 在设计上保留存储适配器边界，后续可替换为对象存储。
- [规则集版本管理过晚引入会影响追溯] → 即使首版只有一套规则，也必须保存规则快照哈希和版本号。
- [本人可见策略可能与真实组织协作流程不完全吻合] → 首版先保守落地，并在审计数据和业务反馈稳定后再扩展角色可见性。

## Migration Plan

1. 新增 OpenSpec 能力定义与设计文档，形成正式实现输入。
2. 在应用库存储层增加合同审核任务、问题、产物和规则快照相关表。
3. 在后端新增 `contract-review` 模块和受保护下载接口，并通过特性开关控制是否暴露入口。
4. 在前端路由与 `AppShell` 中新增 `智能合同审核` 导航项、极简首页和结果详情页，并打通上传后自动进入详情、详情返回首页的交互。
5. 灰度阶段先向少量商务人员开放上传、查看和下载能力，观察任务状态、失败率和产物可用性。
6. 回滚时关闭导航入口与后端开关，已生成任务和产物保留用于审计，不影响现有智能分析链路。

## Open Questions

- 带批注 `.docx` 的具体技术实现选型，是采用现成 Node 库还是自定义 OpenXML 操作层。
- 规则集版本是否首版只做内置单版本，还是需要在创建任务时允许选择多套规则。
- 历史任务的查看范围后续是否需要扩展到部门管理员或法务协作角色。
- 产物与原始文件的保留周期、清理策略和磁盘容量告警是否要在首版一并落地。
