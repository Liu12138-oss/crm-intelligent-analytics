## Context

当前方案有两个明显短板。第一，后端分析链路虽然已经有 `AnalysisIntentService`、`QueryCompilerService`、白名单校验和只读校验，但自然语言到 SQL 的链路仍然过浅：缺少稳定的语义中间层、缺少更细粒度的 SQL 结构校验，也没有把“结果准确性优先于回答完整度”写成硬规则。第二，前端 `AnalysisWorkbenchPage` 把权限、状态、模板、最近查询、结果、流式块均匀铺开，导致页面主次失衡，查询完成后结果区域并没有成为视觉中心，AI 流式输出也过于粗略。

本次变更需要遵守以下约束：

- 不脱离 `docs/prototype-ui/crm-intelligent-analytics-ui.pen` 定义的固定一级分区，但允许在区内重排主次和模块相对位置。
- 一期仍只允许只读访问 CRM 业务数据，不允许生成写入、更新、删除、DDL 或绕过白名单的访问路径。
- 可参考 CRM Open API 文档做业务对象、字段和术语对照，但实际执行仍以当前项目批准的数据白名单、权限模型和只读查询链路为准。

## Goals / Non-Goals

**Goals：**

- 让 CRM 域内自然语言问数从“能识别部分固定问题”提升为“支持更自然的 CRM 业务表达，并稳定转成受控只读查询”。
- 把准确性保护放在首位，明确低置信度、缺失关键条件、结果不一致时优先补问或拒答，而不是返回看似完整但不可靠的数据。
- 明确 SQL 只能从标准化查询计划确定性生成，并经过语义、结构、权限和执行前多层校验。
- 明确结果必须可以回证到同一份查询结果集，避免摘要、图表、表格和导出之间口径漂移。
- 把 AI 查询页改造成结果优先页面，突出结果摘要、图表、表格和流式输出，并把常用查询、最近查询移动到输入区附近。

**Non-Goals：**

- 不支持 CRM 以外的通用问答，不扩展到天气、新闻、股票等域外知识查询。
- 不开放任何写入型能力，不允许通过自然语言触发新增、修改、删除、提醒创建或其他业务写回。
- 不在本次变更中新增新的权限模型或扩大一期白名单数据范围。

## Decisions

### 决策一：采用“语义标准化 + 查询计划 AST + 受控编译器”而不是让模型直接自由生成 SQL

系统将把“自然语言 -> 查询”拆成四层：

- CRM 域识别与语义标准化
- 查询意图与术语映射
- 标准化查询计划 AST 生成
- 确定性只读 SQL 编译与执行

查询计划 AST 至少包含：主题域、基础对象、指标、维度、过滤条件、时间范围、排序、限制、比较关系、结果视图类型、置信度与缺失条件。模型或规则层负责理解“用户想查什么”，编译器只负责把已批准的 AST 映射为 SQL 模板，不接受模型直接产出的原始 SQL。

这样做的意义是把“语义理解”和“SQL 生成”切开：前者允许更灵活，后者必须确定、可校验、可测试。

备选方案是让 LLM 端到端直接生成 SQL。该方案覆盖面看似更大，但极易出现越权字段、未授权对象、写入语句或口径漂移，无法满足“不能给老板看假数据”的目标，因此不采用。

### 决策二：建立 CRM 术语映射层，语义依据来自项目白名单和 CRM Open API 对照

意图识别层将维护一份 CRM 术语映射字典，来源包括：

- 当前项目一期白名单对象、字段、指标与维度
- `specs/001-crm-intelligent-analytics/` 中已批准的业务口径
- CRM Open API 文档中的对象命名和接口术语对照

该映射层负责把“签单/合同/成交金额”“客户/客资”“负责人/销售/业务员”等常见业务说法收敛到批准语义，降低模型自由猜测数据库结构的概率。

备选方案是完全依赖模型自行理解数据库表结构。该方案在短期内开发更快，但长期不可控，也与项目要求“优先以数据库梳理和主规格为准”冲突，因此不采用。

### 决策三：SQL 校验采用“语义校验 + 编译校验 + AST 校验 + 执行前预检”四层栈

SQL 生成不是编译完成就直接执行，而是按四层校验推进：

- 语义校验：检查主题域、对象、指标、维度、过滤条件、排序和比较关系是否是批准组合，缺条件就补问，歧义大就拒答
- 编译校验：只允许从批准的查询计划 AST 编译到固定 SQL 模板，禁止自由拼接未批准片段
- AST 校验：解析 SQL，校验只读性、表字段白名单、JOIN 路径、聚合函数白名单、`GROUP BY` 与 `SELECT` 一致性、`ORDER BY` 合法性和行数限制
- 执行前预检：对真实数据库连接场景执行 `EXPLAIN`/预编译检查或等价预检，提前发现不存在字段、非法别名和高风险扫描计划

这样做的目标不是“多一道保险”，而是把 SQL 错误尽量消灭在执行前。

备选方案是允许模型在字段缺失时自动补默认值继续回答。该方案会提高“有答复”的比例，但会显著增加错误回答风险，因此不采用。

### 决策四：结果准确性采用“单数据源生成 + 关键指标回算 + 异常即拒绝”

结果返回前增加三类准确性检查：

- 单数据源生成：摘要、指标卡、图表、表格、导出和流式说明都必须来自同一份标准化结果集
- 关键指标回算：对金额汇总、数量统计、占比和排名等关键结果执行回算检查，确保派生结果能从底层结果集重新计算出来
- 异常拦截：当出现空数据、重复分组、主指标缺失、汇总值不守恒、图表与表格行数明显不一致等异常时，不直接返回“看似正常”的结果，而是转为错误提示或重新补问

这样可以把“结果准确性”从主观要求变成可执行规则。

### 决策五：工作台改为“结果优先 + 查询资产近输入区 + 流式输出独立强化”

前端工作台将保留现有固定区域，但重新分配视觉权重：

- 顶部主区域：查询输入框，旁侧或下方紧邻常用查询与最近查询
- 中部核心区域：结果摘要、关键指标、主图表、结果表格
- 中部并列重点区域：详细 AI 流式输出面板
- 次级区域：权限摘要、AI 状态、说明信息，压缩为更轻量的摘要卡

这样既满足原型固定分区约束，也更符合用户的真实浏览顺序。

备选方案是继续保留当前“所有卡片权重接近”的平铺结构。该方案实现最省事，但正是当前页面“杂乱无章、不分主次”的根源，因此不采用。

### 决策六：流式输出展示“处理阶段 + 校验结论”，但不暴露不必要的底层细节

流式输出将重点展示：

- 问题识别结果
- 缺失条件与补问
- SQL 编译与校验通过状态
- 查询执行状态
- 结果整理状态
- 最终结论摘要

在普通用户视图下，不直接暴露可能泄露底层库结构的原始 prompt 或未经处理的完整 SQL 文本；但应展示“已完成只读校验/权限注入/结果一致性校验”等业务可读状态，帮助用户理解结果为何可信。

## Risks / Trade-offs

- [自然语言覆盖面越大，错误映射风险越高] → 通过术语字典、查询计划 AST、补问机制、低置信度拒答和白名单编译限制来降低误识别风险。
- [更深的 SQL 校验会降低部分复杂问题的通过率] → 明确以准确性优先于覆盖率，对超范围问题优先友好拒绝而不是勉强生成 SQL。
- [执行前预检会增加一点查询延迟] → 只对真实数据库路径启用必要预检，并优先保证管理层看到的是可信结果而不是最快结果。
- [结果优先布局会压缩部分辅助卡片空间] → 保留摘要式权限与状态卡，不删除必要信息，只调整视觉层级。
- [用户可能希望看到完整 SQL] → 当前先展示安全的处理阶段与结果，不对普通用户暴露底层库细节；若有审计需要，后续单独扩展管理员视图。
- [参考 CRM Open API 文档可能与本地库字段不完全一致] → 文档只作为对象术语与业务含义对照，实际执行仍以项目白名单和数据库梳理为准。

## Migration Plan

本次变更不引入数据库结构迁移。建议按“语义识别与查询计划 AST -> SQL 多层校验与执行前预检 -> 结果准确性校验与流式输出 -> 工作台结果优先改版 -> 联调与回归”顺序实施；若上线后出现问题，可分别回退前端工作台和后端语义编排改动，不影响现有只读数据库连接方式。

## Open Questions

- CRM Open API 文档中的对象命名与当前库表命名存在差异时，最终以哪一套对象命名作为前端展示口径，需要在实现前统一一次映射表。
- 普通用户是否需要看到“SQL 编译摘要”这一层流式信息，还是只展示更业务化的阶段描述，需要在 UI 联调时确认文案深度。
- 对真实数据库路径的执行前预检采用 `EXPLAIN`、预编译还是轻量试跑，需要结合现网库权限再做一次实现确认。
