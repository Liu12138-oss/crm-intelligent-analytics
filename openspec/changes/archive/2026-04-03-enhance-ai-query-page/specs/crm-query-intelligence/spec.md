## ADDED Requirements

### Requirement: 系统必须优先识别 CRM 域内自然语言并对域外请求给出友好提示
系统 MUST 在用户使用口语、简称、业务俗称或不完整业务表述提问时，优先判断该问题是否属于 CRM 一期支持范围。对属于 CRM 范围的问题，系统 MUST 继续进行语义解析；对明显不属于 CRM 范围或超出一期主题范围的问题，系统 MUST 返回友好提示而不是执行查询。

#### Scenario: 口语化 CRM 问题被识别
- **WHEN** 用户输入“帮我看看这季度谁签单最多，哪个团队机会最差”
- **THEN** 系统必须将其识别为 CRM 域内分析请求，并继续进入条件补全或查询编译流程

#### Scenario: 非 CRM 问题被友好拒绝
- **WHEN** 用户输入与天气、股票、外部新闻或其他非 CRM 场景相关的问题
- **THEN** 系统必须明确提示当前仅支持 CRM 智能分析相关问题，并且不得执行数据库查询

### Requirement: 系统必须将已识别的 CRM 问题编译为受控只读查询
对于已识别且允许执行的 CRM 分析请求，系统 MUST 先生成标准化查询意图和查询计划 AST，再由受控编译器生成只读 SQL。生成结果 MUST 仅使用项目白名单允许的数据表、字段、指标和维度，并自动注入当前用户权限范围。系统 MUST 拒绝任何写入、更新、删除、DDL 或绕过白名单的查询路径。

#### Scenario: CRM 问题被编译为带权限范围的只读查询
- **WHEN** 销售负责人输入“本月各负责人新增商机金额排名”
- **THEN** 系统必须生成仅访问允许表字段的只读查询，并自动附加当前用户组织、部门或负责人权限范围

#### Scenario: 写入型或高风险意图被阻断
- **WHEN** 用户输入“把这个商机改成已成交”或其他写入型自然语言
- **THEN** 系统必须明确拒绝，并提示当前一期仅支持受控问数

### Requirement: 系统必须对 SQL 生成执行多层深度校验
系统 MUST 在 SQL 执行前依次完成语义校验、编译校验、SQL AST 只读校验和执行前预检。校验内容 MUST 至少覆盖主题域是否合法、对象与指标组合是否合法、字段白名单、JOIN 路径白名单、聚合函数白名单、`GROUP BY` 与聚合匹配、排序字段合法性、结果行数限制以及 SQL 只读性。任何一层校验失败时，系统 MUST 阻止执行。

#### Scenario: 非法 JOIN 或字段组合被阻断
- **WHEN** 生成的查询计划或 SQL 引用了未批准的关联路径、字段组合或聚合方式
- **THEN** 系统必须在执行前阻止该 SQL，并返回明确的中文原因

#### Scenario: 执行前预检发现 SQL 结构问题
- **WHEN** SQL 在执行前预检阶段发现字段不存在、别名错误或高风险扫描问题
- **THEN** 系统必须停止执行并返回失败提示，而不是继续返回分析结果

### Requirement: 系统必须在结果返回前执行准确性保护
系统 MUST 在返回结果前执行必要补问、白名单校验、AST 只读校验、权限范围注入、结果一致性校验和低置信度拦截。对于缺少关键条件、语义映射不明确、结果结构不一致或无法保证准确性的请求，系统 MUST 优先补问或拒答，而不得返回可能误导管理层的分析结果。

#### Scenario: 缺少关键条件时先补问
- **WHEN** 用户输入“看一下商机转化怎么样”但未提供时间范围或比较对象
- **THEN** 系统必须先发起补问，并在补齐条件前不返回正式分析结果

#### Scenario: 结果一致性无法保证时拒绝返回
- **WHEN** 系统发现摘要、图表和表格无法基于同一批查询结果保持一致
- **THEN** 系统必须阻止结果返回，并提示当前问题需要重新确认条件或缩小范围

### Requirement: 系统必须对结果执行关键指标回算与异常校验
系统 MUST 使用同一份标准化结果集生成摘要、指标卡、图表、表格和导出数据，并对关键指标执行回算校验。若出现汇总值不守恒、排名与明细不一致、占比无法回算、重复分组、主指标为空或其他足以影响管理判断的异常，系统 MUST 阻止正式结果返回。

#### Scenario: 金额汇总与表格明细无法回算一致
- **WHEN** 指标卡中的汇总金额与结果表格明细累计值不一致
- **THEN** 系统必须阻止该结果进入正式展示，并返回结果校验失败提示

#### Scenario: 结果集中出现重复分组或主指标为空
- **WHEN** 结果集中存在重复业务分组、关键指标为空值或排名顺序与明细不一致
- **THEN** 系统必须将该次请求标记为不可靠结果，并阻止返回正式结论

### Requirement: 系统必须基于 CRM 对象语义对齐解析结果
系统 MUST 以一期批准的数据白名单、数据模型和 CRM Open API 对象术语对照为基础，维护 CRM 业务对象、指标、维度和常见同义词映射，避免模型自由猜测数据库结构。若请求引用的对象或字段不在批准映射范围内，系统 MUST 拒绝编译为查询。

#### Scenario: CRM 同义词被映射到批准对象
- **WHEN** 用户交替使用“签单”“合同”“成交金额”“赢单率”等 CRM 常见业务说法提问
- **THEN** 系统必须将这些术语映射到批准的对象和指标口径后再生成查询计划

#### Scenario: 未批准字段不参与查询
- **WHEN** 用户请求查询一期未纳入白名单的对象、字段或日志数据
- **THEN** 系统必须提示当前问题超出支持范围，并且不得生成对应 SQL
