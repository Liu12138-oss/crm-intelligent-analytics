## ADDED Requirements

### Requirement: 受保护菜单热切换必须脱离同步会话往返

系统 MUST 将同一页签内已登录用户的一级菜单热切换设计为“默认不阻塞实时会话请求”的交互模型。对于已经完成过一次有效会话水合且当前登录态未失效的用户，系统 MUST 允许路由守卫直接复用页签内会话与能力快照完成菜单跳转；系统 MUST NOT 在每次点击受保护菜单时都同步等待 `session`、`capabilities` 等鉴权接口返回后才确认路由。

#### Scenario: 已登录用户在同一页签内切换一级菜单

- **WHEN** 用户已经在当前页签完成登录并进入任一受保护页面
- **AND** 用户随后点击另一个一级菜单
- **THEN** 系统必须先完成路由确认和页面骨架切换
- **THEN** 菜单热路径不得再次同步等待 `session` 或 `capabilities` 请求完成

#### Scenario: 首次进入受保护页面仍允许实时会话校验

- **WHEN** 用户首次打开站点、硬刷新页面或本地尚未建立有效会话快照
- **THEN** 系统可以执行一次实时会话校验
- **THEN** 该实时校验不得被误扩展为后续每次菜单点击都重复执行

#### Scenario: 会话失效后立即退出热切换缓存

- **WHEN** 当前页签发生 401、显式登出或其它会话失效事件
- **THEN** 系统必须立即丢弃本地会话与能力快照
- **THEN** 后续再次进入受保护页面时必须重新执行实时会话校验

### Requirement: 重页面必须采用分层加载并优先复用最近状态

系统 MUST 将智能分析、经营报表、权限中心、审计中心等重页面设计为“路由即时生效 + 页面骨架先出 + 区块按需加载”的模型。重页面再次进入时，系统 SHOULD 优先复用当前页签内最近一次成功状态；若需要刷新数据，系统 MUST 在页面主体已展示后再静默执行，而不是把重请求放回菜单切换热路径。

#### Scenario: 用户切换到权限中心时先看到页面骨架

- **WHEN** 用户从其它一级页面切换到权限中心
- **THEN** 系统必须先展示权限中心页面骨架与首屏摘要区
- **THEN** 身份映射、日报策略或其它重区块可以在页面显示后再按需加载

#### Scenario: 用户返回智能分析页时复用最近状态

- **WHEN** 用户已经加载过智能分析页的模板、最近查询或结果态
- **AND** 用户在同一页签内切换到其它一级页面后再次返回
- **THEN** 系统应优先复用最近一次成功状态
- **THEN** 若需要刷新，必须在页面已展示后再后台补齐

### Requirement: 系统必须记录菜单导航性能并建立回归门槛

系统 MUST 为一级菜单点击到页面可交互之间的关键阶段记录性能指标，至少覆盖 `menu_click`、`route_confirmed`、`shell_visible` 和 `page_data_ready`。系统 MUST 能区分“导航本身变慢”和“页面数据加载变慢”，并为热切换建立独立回归门槛。

#### Scenario: 菜单热切换性能被持续观测

- **WHEN** 已登录用户在同一页签内切换一级菜单
- **THEN** 系统必须记录从点击到路由确认、从路由确认到壳层显示、从壳层显示到数据就绪的分段耗时
- **THEN** 这些指标必须可用于回归比较

#### Scenario: 性能回归时能够定位到具体阶段

- **WHEN** 某次版本迭代后菜单切换再次出现明显卡顿
- **THEN** 系统必须能够区分问题出在路由守卫、壳层渲染还是页面数据加载
- **THEN** 不得只留下一个无法定位的总耗时数字
