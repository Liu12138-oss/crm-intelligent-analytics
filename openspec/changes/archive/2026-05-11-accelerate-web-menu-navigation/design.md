## Context

当前系统的工作台菜单切换存在三个串联问题：

- 路由守卫把受保护路由切换等同于强制会话水合，导致每次点菜单都重复请求 `session` 和 `capabilities`
- 智能分析、权限中心、经营报表等页面在进入时还会继续触发多个初始化接口
- 后端每个受保护接口都会经过会话解析与数据库身份装载，导致即使网络层出现 `304`，整体切换仍需等待数秒

因此，这次变更不是要“让某一个接口更快”，而是要重构菜单切换的责任边界：

- 路由切换负责即时响应和正确落页
- 页面负责按需加载自己的数据
- 后端负责统一权限决策和最终阻断
- 缓存负责把短时间内的重复解析从热路径上拿掉

## Goals / Non-Goals

**Goals:**

- 让已登录用户在同一页签内切换一级菜单时达到毫秒级响应
- 让重页面采用“先切页、后补数”的模型，避免菜单切换等待页面首屏接口完成
- 在不破坏统一权限矩阵、实时组织范围和后端敏感动作校验的前提下，引入短时快照缓存
- 建立清晰的失效事件和性能验收口径，避免后续回退到“每点一次菜单就重鉴权”的模式

**Non-Goals:**

- 不改变现有业务权限边界和动作定义
- 不因为前端缓存而放宽导出、审计、合同审核、真实数据读取等后端权限校验
- 不在本次变更中引入新的长期物化表或跨进程分布式缓存
- 不要求首次硬刷新进入任意重页面时所有数据都毫秒返回；首版目标是“页面先秒开，重数据后补”

## Decisions

### 决策 1：菜单热切换默认不得触发同步会话重拉

前端一级菜单热切换改为优先复用当前页签已水合的登录态和能力快照。路由守卫只负责以下几类场景的实时会话刷新：

- 首次进入受保护页面
- 企业微信扫码回流
- 本地会话尚未建立
- 401 失效回退
- 显式强制刷新权限

普通业务菜单切换不得再默认同步请求：

- `GET /auth/session`
- `GET /analysis/capabilities`

原因：

- 一级菜单点击是高频行为，必须被视为纯交互热路径
- 对于几秒内已完成过一次统一权限决策的页签，没有必要每次再做一遍完整鉴权往返

### 决策 2：前端使用“页签级会话 + 短 TTL 能力快照 + Promise 去重”模型

前端会话层拆成两类缓存：

1. `session`
   - 页签内常驻
   - 登录成功后立即建立
   - 直到登出、401、扫码换绑或显式失效才清除
2. `capabilities`
   - 使用短 TTL 复用，建议 `10s - 30s`
   - 使用 `Promise` 去重，避免同一轮初始化并发触发多次相同请求
   - 登录、权限保存、范围治理保存、401、登出后主动失效

这样可以保证：

- 菜单切换不再同步等待能力请求
- 同一轮页面初始化不会重复打相同接口
- 权限变更后又不会长时间使用旧快照

### 决策 3：重页面统一采用“路由即时生效 + 页面骨架 + 分区懒加载”

工作台一级页面分成两类：

- 轻页面：进入后直接展示，初始化请求较少
- 重页面：权限中心、经营报表、审计中心、部分分析场景

重页面统一采用以下结构：

1. 路由确认后立刻渲染页面骨架与标题区
2. 若存在最近状态，则先复用最近状态
3. 首屏只加载必要摘要区块
4. 其它区块在激活、滚动进入或显式刷新时再按需加载

页面级要求：

- `analysis` 页面优先复用上次模板、最近查询和结果态，模板与历史查询改为并行补齐
- `management-report` 页面优先复用上次快照和已加载专题，切回后后台静默刷新
- `permission-center` 首屏只加载概览和必要候选项，角色矩阵、身份映射、日报策略、预览结果改成分区加载

### 决策 4：后端使用短时会话用户快照与能力快照，但最终执行仍以实时校验为准

后端引入两层短时缓存：

1. 会话用户快照
   - 基于 `sessionId`
   - 复用最近一次数据库身份解析结果
   - TTL 建议 `5s - 30s`
2. 能力快照
   - 基于 `sessionId + 权限版本 + 范围版本`
   - 复用统一权限矩阵计算结果
   - TTL 建议 `10s - 30s`

但以下接口不得只依赖前端缓存放行：

- 导出
- 审计读取
- 合同审核任务创建
- 任何真实数据读取接口

这些接口仍然必须经过后端统一权限决策与范围校验。缓存只负责减少重复身份解析和重复能力构建，不负责绕过执行校验。

### 决策 5：缓存失效必须由显式事件驱动，不得只依赖自然过期

以下事件必须触发前后端快照立即失效：

- 登录成功
- 退出登录
- 企业微信换绑或扫码重新登录
- 任意 401 鉴权失效
- 权限中心保存角色矩阵
- 治理策略保存
- 组织范围治理保存
- 其它会影响菜单或动作可见性的治理写操作

原因：

- 如果只靠 TTL，短时间内可能继续沿用旧菜单与旧权限结果
- 权限治理类页面的核心价值之一，就是“保存后立即生效”

### 决策 6：性能验收使用分段指标，而不是只看单个接口时间

本次变更采用四段指标：

- `menu_click -> route_confirmed`
- `route_confirmed -> shell_visible`
- `shell_visible -> primary_content_visible`
- `primary_content_visible -> page_data_ready`

验收目标：

- 已登录页签内一级菜单热切换时，前两段必须维持在毫秒级
- 重页面允许后两段继续异步完成，但不得阻塞前两段

同时补充后端分段日志：

- 会话解析耗时
- 能力快照构建耗时
- 页面首屏摘要接口耗时
- 懒加载区块接口耗时

## Risks / Trade-offs

- [短 TTL 缓存导致菜单短时间显示旧状态] → 通过治理保存、401、登录态变化等显式事件立即失效
- [前端页面秒开但内容迟到引发误解] → 所有重页面必须提供明确的骨架、刷新状态和区块级加载反馈
- [缓存只做前端不做后端，仍然在真实接口上反复慢] → 同步补后端短时会话解析缓存
- [重页面拆成分区加载后出现更多状态分支] → 统一抽象页面级“摘要成功 / 分区失败 / 可重试”状态模型
- [KeepAlive 滥用导致内存上升] → 仅缓存一级页面和必要区块，不对明细详情和低频页面无限保留

## Migration Plan

1. 先新增 OpenSpec 变更，明确热切换、快照缓存和分区加载边界
2. 第一阶段只改前端路由守卫和 `auth store`，把 `session` / `capabilities` 从菜单热路径上拿掉
3. 第二阶段改重页面初始化模型，补 `KeepAlive`、最近状态复用和分区懒加载
4. 第三阶段补后端短时会话解析缓存、能力快照缓存和失效机制
5. 第四阶段补性能埋点、日志和回归验收

回滚策略：

- 如果前端快照复用产生权限显示异常，可先关闭能力快照复用，只保留“菜单切换不强制 rehydrate session”的最小收益版本
- 如果后端短时缓存与治理生效冲突，可先缩短 TTL，并保留显式失效链路

## Open Questions

- 审计中心与权限中心的首屏摘要最小集，需要在实现前进一步确认接口拆分优先级
- 是否需要在首版就把性能指标纳入自动化门禁，还是先以人工采样和日志看板观测
