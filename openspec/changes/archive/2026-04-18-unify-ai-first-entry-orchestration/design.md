## Context

当前一期已经形成多入口并行格局：

- 企业微信机器人入口负责问数、跟进、日报、创建和部分帮助提示；
- Web 智能分析入口负责自然语言问数、追问、最近查询重跑和结果解释；
- Web 合同审核入口负责文件上传、任务创建、详情展示和下载。

这些入口都已经有各自稳定的固定程序边界，例如：

- 企业微信签名校验、来源校验和消息结构校验；
- Web 登录态与会话鉴权；
- 合同文件 MIME、大小与上传校验；
- CRM 权限继承、白名单校验、写回确认、主动通知和审计留痕。

真正不统一的是“用户业务输入理解”这一层。当前企业微信大量依赖关键词、状态机和规则 helper；Web 分析入口虽然有 AI 解析，但仍未形成统一的入口理解层；合同审核入口则尚未定义“文件前置校验”和“后续自然语言解释/补充说明”之间的统一边界。这直接导致：

- 同一类话术在不同入口命中不同流程；
- 帮助、取消、继续、改条件、直接提交等回复语义不断追加规则；
- 新增功能时容易继续堆规则，而不是复用统一理解层；
- 当前活跃变更中存在“提案写 AI 主链，代码仍是规则过渡”的口径冲突。

本次设计的核心不是让 AI 直接替代全部程序，而是把“所有业务自然语言输入”统一收敛成：

`固定安全前置检查 -> 统一 AI 理解层 -> 固定受控程序执行`

## Goals / Non-Goals

**Goals：**

- 为企业微信机器人、Web 智能分析和合同审核相关自然语言交互建立统一 AI 入口理解层。
- 明确哪些步骤必须固定程序先执行，哪些步骤必须先经 AI 理解后再路由。
- 统一 AI 输出 schema，使跟进、日报、问数、帮助、取消、切换任务、解释追问和合同审核说明等都能走同一类结构化路由。
- 将现有规则状态机降级为 fallback，而不是继续作为默认主链路。
- 统一审计快照，保留原始输入、AI 理解结果、路由结果、fallback 原因和最终执行程序。

**Non-Goals：**

- 不把企业微信签名校验、Web 鉴权、文件上传校验或权限判断交给 AI。
- 不放开任何绕过白名单、确认、权限和审计的直接执行路径。
- 不在本次设计中重做现有登录、扫码登录、Cookie、Session 或统一鉴权链路。
- 不要求一次性替换所有规则 helper；允许保留最小安全 fallback，但不得继续把 fallback 描述成 AI 主链。

## Decisions

### 决策 1：统一入口按“固定前置检查”和“AI 理解前置”两段式拆分

所有入口先执行不会依赖业务语义的固定程序检查，再决定是否进入统一 AI 理解层。

固定程序前置检查包括：

- 企业微信签名、来源和消息结构校验；
- Web 登录态、会话、权限装载；
- 合同文件类型、大小和上传完整性校验；
- 明确的黑白名单、安全拦截和资源配额检查。

只有通过这些检查后，用户的业务自然语言输入才进入 AI 理解层。

原因：

- 这能满足“所有业务输入优先经过 AI”的诉求，同时不破坏安全边界。
- 如果把签名、鉴权、文件校验也交给 AI，本质上是在移除安全前置条件。

备选方案：

- 所有输入无差别先送 AI。问题是签名无效、未登录、非法文件也会进入模型，既浪费成本又弱化安全。
- 保持现有各入口各自解释。问题是规则持续分裂，无法形成统一能力。

### 决策 2：建立统一 AI 入口 schema，而不是各链路各自定义一套返回结构

统一 AI 理解层必须输出稳定 schema，至少包含：

- `scene`
- `targetWorkflow`
- `requestedAction`
- `replyIntent`
- `confirmationIntent`
- `cancelIntent`
- `switchTaskIntent`
- `helpIntent`
- `structuredSlots`
- `confidence`
- `fallbackReason`
- `blockReason`

其中：

- `scene` 用于区分企业微信问数、跟进、日报、创建、帮助、合同审核解释等场景；
- `targetWorkflow` 指向固定受控程序，而不是让模型直接执行；
- `structuredSlots` 承载跟进四段草稿、分析维度时间范围、回复确认对象等结构化信息。

原因：

- 当前问题的根本不是“有没有 AI”，而是缺少统一路由 contract。
- 一旦 schema 统一，企业微信和 Web 可共享理解层和审计模型，只在后续展示与执行上分流。

备选方案：

- 企业微信、Web、合同审核各自维护不同 schema。问题是无法共享测试、审计和 fallback 机制。

### 决策 3：AI 只负责理解和生成受控意图，固定程序继续负责执行

统一 AI 理解层输出的不是最终执行结果，而是“应进入哪个固定受控程序、携带哪些结构化槽位”。例如：

- 企业微信跟进自由文本 -> 路由到跟进草稿确认程序；
- 企业微信“可以”“先这样吧” -> 路由到草稿确认程序；
- Web “最近三个月也看看” -> 路由到受控分析重执行程序；
- Web “这说明什么” -> 路由到结果包解释程序；
- 合同审核后续说明或解释请求 -> 路由到合同审核解释程序。

原因：

- 这样可以把 AI 放在最擅长的理解层，同时保留程序在权限、确认、写回、查询和审计上的确定性。
- 这也是把现有规则状态机渐进迁移到 AI-first 的唯一可控方式。

备选方案：

- 让 AI 直接完成端到端流程。问题是会穿透确认、审计和权限边界。
- 继续让程序主导理解，AI 只做个别补充。问题是仍然回到“规则加到无穷无尽”。

### 决策 4：规则逻辑统一退为 fallback，并必须显式记录 fallback 原因

现有关键词、短句匹配、候选字符打分、固定模板 explanation 和日报摘要模板都允许保留，但只在以下条件下使用：

- AI 网关不可用；
- AI 响应超时；
- AI 输出结构不合法；
- AI 置信度低于当前场景阈值；
- 当前安全策略要求不进入 AI 主链。

一旦发生 fallback，系统必须在审计快照里记录：

- 原始输入；
- 预期目标场景；
- fallback 触发原因；
- 实际启用的规则模块；
- 后续执行程序。

原因：

- 当前最大的治理问题之一，是 fallback 已经实际上成了主链，却没有被显式承认。
- 只有把 fallback 显式化，后续才能真正量化 AI 覆盖率和规则退化率。

备选方案：

- 保持现状，规则和 AI 混杂。问题是文档、实现和验收持续失真。

### 决策 5：统一审计快照按“理解、路由、执行”三层落库

本次新增或收敛的快照建议统一为三层：

- `entryInterpretationSnapshot`
  - 原始输入、入口、AI schema 输出、置信度、fallback 原因
- `workflowRoutingSnapshot`
  - 命中的固定程序、是否需要确认、是否需要补问、是否被阻断
- `executionSnapshot`
  - 现有分析执行、写回确认、日报生成、合同审核执行等执行侧快照

原因：

- 现有快照多集中在执行后半段，入口理解过程不可见。
- 这次改造的核心资产恰恰是“AI 是如何理解并决定路由的”。

备选方案：

- 继续只保留执行快照。问题是无法定位 AI 误判、低置信和 fallback 的根因。

### 决策 6：现有三个相关变更不废弃，但必须回收口径并建立依赖

对当前未归档变更的处理原则：

- `enhance-wecom-ai-structured-assistant` 保留，但改为“企业微信场景在统一 AI 入口层上的专用落地”；
- `enhance-controlled-analysis-orchestration` 保留，但改为“统一 AI 入口层路由到受控分析执行后的专用落地”；
- `fix-wecom-unrecognized-input-help-guidance` 保留，但缩成“低置信 / 未识别 / fallback 帮助策略”的子问题，而不是继续单独堆规则。

原因：

- 现有变更已经积累了不少设计、测试和快照工作，不应全部推翻。
- 但它们必须服从新的总边界，否则会继续把规则过渡实现误标为 AI 完成。

## Risks / Trade-offs

- [统一 AI 理解层成为单点风险] → 保留显式 fallback，按入口分别允许最小安全降级，但不得伪装成 AI 主链成功。
- [统一 schema 过大，导致单轮 prompt 复杂度上升] → 采用“共享顶层 schema + 场景子 schema”方式，避免所有入口共用一个臃肿 prompt。
- [企业微信与 Web 的交互语义差异较大] → 共享理解层 contract，但保留渠道级 prompt、置信阈值和展示裁剪。
- [合同审核上传本身并非自然语言输入，容易被误纳入 AI-first] → 在规格中明确“文件上传属于固定程序前置，不进入 AI 理解层”。
- [现有任务清单与真实实现冲突，短期内看起来像回退] → 通过总提案统一解释为“回收错误完成状态，重新建立主链与 fallback 边界”。

## Migration Plan

1. 先创建统一 AI 入口提案与 spec，冻结“什么必须 AI-first、什么必须固定前置”的边界。
2. 再修订企业微信和受控分析两个现有主提案，把规则主链回收为 fallback，恢复未完成任务。
3. 第一期实现优先落企业微信和 Web 分析入口，因为这两类自然语言输入最集中、收益最大。
4. 合同审核入口先只收敛边界说明和后续自然语言交互方案，不强制改动当前文件上传任务创建链路。
5. 若统一 AI 入口层效果不稳定，可按入口关闭对应场景特性开关，回退到最小安全 fallback。

## Open Questions

- 企业微信和 Web 是否共用同一模型与同一 prompt pack，还是按入口维持不同 prompt 配置但共享 schema。
- 合同审核是否在本轮就提供结果解释型自然语言交互，还是先只冻结边界并在后续单独立项。
- 统一 AI 理解层的结果快照是否直接进入现有 audit event，还是新增独立存储表以便后续分析误判率。
