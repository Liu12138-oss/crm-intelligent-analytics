## Context

当前通知底座的主规格已经把企业微信主动通知拆成两类：

- 正式通知默认走企业微信自建应用消息
- 会话型通知默认走企业微信机器人主动推送

这个模型从技术能力视角没有问题，但从用户视角存在明显缺陷。用户不会天然区分“应用消息”和“机器人消息”背后的技术实现，只会感知到自己在企业微信里同时看到两个 CRM 相关入口：一个能聊天，一个能推送。只要两个入口都和 CRM 智能分析有关，用户就很容易把应用误当作机器人入口。

由于企业微信机器人本身已经支持主动推送、单聊定向和群聊上下文通知，因此没有必要为了“正式通知”再暴露第二个用户入口。更稳妥的方案是把所有用户可见通知统一收敛到机器人中，自建应用仅保留底层授权和非聊天用途。

## Goals / Non-Goals

**Goals:**

- 把消息通知基础模块改成“机器人单入口”模型，避免应用与机器人双入口混淆。
- 明确企业微信应用不再作为用户可见通知入口，只保留底层授权用途。
- 为后续业务提案提供统一的单入口通知底座，让日报、告警、治理复核等场景都默认走机器人。
- 通过消息形态和交互约束，降低主动通知对正常问答上下文的干扰。

**Non-Goals:**

- 不在本提案中定义日报、告警、治理复核等具体业务口径。
- 不在本提案中修改企业微信登录、通讯录同步或现有 OAuth 方案。
- 不要求删除企业微信自建应用配置；它仍可作为底层能力存在，只是不再承接用户可见通知。

## Decisions

1. 用户可见的正式通知与会话型通知统一通过企业微信机器人推送。

   这次重构的核心不是“机器人能不能发”，而是“用户到底记住几个入口”。既然机器人已经具备主动推送与单聊定向能力，那么所有用户可见通知都统一由机器人承接，能最大程度避免“去应用里聊天”的误用。备选方案是继续保留“双入口”，但这会持续带来认知混淆。

2. 企业微信应用只保留底层用途，不再承担通知入口职责。

   企业微信应用仍可用于 Web 登录授权、通讯录同步、官方接口授权载体等底层能力，但在用户心智上必须从“通知入口”降级为“后台支撑能力”。备选方案是继续让应用承接正式通知，但这和“机器人单入口”目标冲突。

3. 机器人中的主动通知需要使用明确的消息形态与交互边界。

   既然正式通知也回到机器人里，就必须避免机器人会话变成杂乱消息流。建议所有主动通知都带固定标签，例如 `【日报提醒】`、`【团队汇总】`、`【异步结果】`，并默认不直接污染当前问答上下文；只有用户点击动作或明确回复后，才进入下一步业务流程。备选方案是把主动通知当作普通聊天消息直接插入上下文，但会破坏问答体验。

4. 业务提案不再决定“用应用还是机器人”，只决定“发什么、发给谁、何时发”。

   路由权应从业务提案下沉到通知底座。后续日报、告警、治理复核等提案只描述业务规则，不再自行决定使用企业微信应用还是机器人，避免再次分叉。备选方案是每个业务提案各自决定通道，但这样会再次把单入口原则打散。

5. 真实消息开关与测试收件人覆盖继续保留，但发送目标固定为机器人。

   安全控制本身仍然有效，只是把“真实/测试发送”都固定在机器人通道内完成。备选方案是继续保留“测试走机器人、正式走应用”，但那会把单入口原则破坏掉。

## Risks / Trade-offs

- [所有通知都进入机器人，消息密度会增加] -> 通过固定标签、摘要化输出和动作按钮隔离问答与通知两类消息。
- [企业微信应用不再发通知，现有“正式通知走应用”的实现需要重构] -> 用独立提案修改基础模块，而不是在业务提案里零散调整。
- [后续业务方可能再次提出“正式通知更适合应用”的诉求] -> 在基础模块规格中显式固化机器人单入口原则，除非重新立提案评审，否则不允许偏离。

## Migration Plan

1. 先冻结新需求中的“应用推送”入口扩张。
   从本提案生效开始，所有新业务提案不再新增“企业微信应用通知入口”描述；如果确有历史方案仍写着“正式通知走应用”，必须先补文档修正，再进入实现。

2. 再把通知底座的默认路由改为机器人单入口。
   原有“正式通知 -> 应用、会话通知 -> 机器人”的默认策略，需要统一改成“用户可见通知 -> 机器人”。这一步是底座切换，不要求业务提案逐个先改完代码，但要求底座先不再新增任何新的应用通知路径。

3. 存量业务按“先兼容、后收敛”的顺序迁移。
   对已经存在的正式通知调用点，不要求一次性全部删除，而是先通过底座兼容层把它们重定向到机器人推送，业务侧调用参数尽量保持不变。这样可以把迁移成本集中在底座，而不是分散到每个业务模块。

4. 企业微信应用保留配置，但降级为后台能力。
   `WECOM_WEB_LOGIN_*`、通讯录同步相关配置、以及企业微信应用本身都继续保留，用于登录授权、目录同步和官方接口凭证承载；但在产品说明、培训材料和联调口径中，不再把应用描述为“提醒入口”或“聊天入口”。

5. 机器人消息形态必须先于大规模迁移落地。
   在把更多正式通知切回机器人之前，先统一通知标签、摘要结构、动作入口和上下文隔离策略。否则只是把应用里的消息搬到机器人里，仍会形成新的体验噪音。

6. 联调阶段统一走机器人测试收件人。
   真实发送开关关闭时，所有原本会发给真实业务对象的通知都统一改投机器人测试收件人；不再存在“应用侧测试、机器人侧正式”这种双口径验证。这样可以在迁移期保持单入口体验，同时降低误发风险。

7. 回滚策略保留在底座层，而不是业务层。
   如果机器人单入口方案出现严重问题，回滚应优先发生在通知底座的路由策略上，而不是让各业务提案自行恢复“应用推送”逻辑。这样能避免回滚时多个业务各自恢复旧代码，造成更大分叉。

8. 最终验收以“用户只记住机器人入口”为准。
   迁移完成的判断标准不只是技术上“能发消息”，而是：培训材料、联调文档、业务说明、测试场景和真实交互中，都不再要求用户进入企业微信应用处理 CRM 提醒。

## Open Questions

- 正式通知回到机器人后，是否需要统一定义一套模板卡片或标签前缀规范。
- 机器人主动通知默认是否进入会话历史，还是只在用户交互后写入上下文，需要在实现前再收敛一次。
