## ADDED Requirements

### Requirement: 系统必须为企业微信机器人入口提供独立灰度准入模式

系统 MUST 为 `wecom-bot` 提供独立的灰度准入策略，并至少支持 `DISABLED`、`PILOT_ONLY` 和 `FULL` 三种模式。`DISABLED` 表示企业微信机器人入口整体关闭；`PILOT_ONLY` 表示只有命中灰度白名单的用户可进入；`FULL` 表示所有已具备企业微信角色资格的用户都可进入。该灰度模式 MUST 独立于 Web 入口存在，不得因为企业微信机器人关闭就同时关闭 Web 工作台。

#### Scenario: 企业微信机器人处于关闭模式
- **WHEN** 企业微信机器人灰度模式被设置为 `DISABLED`
- **THEN** 系统必须拒绝所有企业微信机器人入口请求
- **THEN** 同一用户若具备 Web 工作台权限，仍可继续使用 Web 入口

#### Scenario: 企业微信机器人处于全量开放模式
- **WHEN** 企业微信机器人灰度模式被设置为 `FULL`
- **THEN** 系统必须允许所有已具备 `wecomBotEligible` 与 `wecom.analysis.use` 的用户进入机器人入口
- **THEN** 不得再要求这些用户额外命中白名单

### Requirement: 系统必须支持按 CRM 用户、角色和部门进行企业微信灰度放量

在 `PILOT_ONLY` 模式下，系统 MUST 支持按 CRM 用户、CRM 角色和 CRM 部门三个维度配置灰度白名单。系统 MUST 以 CRM 维度做判断，而不是直接按企业微信组织树做正式授权。若同一用户命中任一白名单维度，则可以通过灰度准入；若未命中任何白名单维度，则必须被阻断。

#### Scenario: 用户命中个人白名单后可进入企业微信机器人
- **WHEN** 某个 CRM 用户被加入企业微信灰度 `allowUserIds`
- **THEN** 该用户在完成企业微信身份映射后必须可以进入机器人入口
- **THEN** 即使同角色其它用户未开通，也不得阻断该用户

#### Scenario: 用户命中角色或部门白名单后可进入企业微信机器人
- **WHEN** 某个用户未命中个人白名单，但其 CRM 角色或 CRM 部门命中 `allowRoleIds` 或 `allowDepartmentIds`
- **THEN** 系统必须允许该用户进入企业微信机器人入口
- **THEN** 诊断结果中必须说明其是通过角色或部门白名单命中的

### Requirement: 系统必须支持用户级企业微信入口强制停用

无论企业微信机器人处于 `PILOT_ONLY` 还是 `FULL` 模式，系统 MUST 支持通过 `denyUserIds` 对单个 CRM 用户执行强制停用。被强制停用的用户 MUST 在完成身份映射后被企业微信入口拒绝，并返回明确的停用提示；该停用仅影响企业微信机器人入口，不得隐式撤销其 Web 侧能力。

#### Scenario: 用户在全量开放模式下被单独停用
- **WHEN** 企业微信机器人处于 `FULL` 模式，且某个 CRM 用户被加入 `denyUserIds`
- **THEN** 系统必须阻断该用户的企业微信机器人入口请求
- **THEN** 该阻断结果不得影响其已具备的 Web 工作台访问资格

#### Scenario: 白名单用户同时命中停用名单时必须以停用为准
- **WHEN** 某个 CRM 用户同时命中白名单和 `denyUserIds`
- **THEN** 系统必须以停用名单优先，阻断该用户进入企业微信机器人
- **THEN** 诊断结果中必须显示是由于用户级停用而非白名单未命中被拒绝

### Requirement: 系统必须在企业微信正式业务编排前完成灰度准入判断

系统 MUST 在企业微信入口完成签名校验、来源校验、消息结构校验和身份映射后、进入正式业务编排前执行灰度准入判断。未通过灰度准入的消息 MUST 被直接拒绝，并记录统一的准入拒绝事件；系统不得让未命中灰度的消息继续进入分析、受控新增、跟进写回、日报预览或其它正式业务执行链路。

#### Scenario: 灰度未命中用户在进入正式分析链路前被拒绝
- **WHEN** 企业微信发送者完成身份映射，但未命中当前灰度白名单
- **THEN** 系统必须在进入正式分析链路前阻断该消息
- **THEN** 不得创建正式分析请求、合同任务或业务写回草稿

#### Scenario: 命中灰度的用户可继续进入后续动作判断
- **WHEN** 企业微信发送者已命中灰度准入
- **THEN** 系统才可以继续判断其是否具备 `wecom.analysis.use`、`wecom.customer.create`、`wecom.opportunity.create`、`wecom.followup.writeback` 或 `wecom.daily_report.preview` 等动作资格
- **THEN** 灰度命中本身不得替代具体动作权限判断

### Requirement: 系统必须返回统一的企业微信入口状态码与中文提示

系统 MUST 为企业微信入口准入返回统一状态码，至少覆盖 `ALLOWED`、`CHANNEL_DISABLED`、`ROLE_NOT_ENABLED`、`PILOT_REQUIRED`、`EXPLICITLY_DENIED` 和 `UNMAPPED_CRM_IDENTITY`。每个状态码 MUST 对应明确的中文提示文案，避免将“灰度未开通”“用户停用”“未映射 CRM 身份”“角色未开通”混为同一类普通无权限提示。

#### Scenario: 灰度未开通用户收到试点提示
- **WHEN** 用户因为未命中白名单而被拒绝
- **THEN** 系统必须返回 `PILOT_REQUIRED` 或等价状态
- **THEN** 中文提示必须明确说明“当前企业微信机器人仍在灰度开放中，你的账号暂未开通体验资格”或等价语义

#### Scenario: 未映射用户收到身份映射提示
- **WHEN** 用户未命中有效 CRM 身份映射
- **THEN** 系统必须返回 `UNMAPPED_CRM_IDENTITY` 或等价状态
- **THEN** 中文提示必须说明是身份映射缺失，而不是普通角色未开通

### Requirement: 系统必须提供企业微信灰度策略治理与命中预览能力

系统 MUST 在权限中心中提供企业微信灰度策略治理界面和命中预览能力。管理员 MUST 能查看当前模式、白名单用户/角色/部门、停用名单、最近更新时间和变更备注，并能对指定用户执行“当前是否可进入企业微信机器人、命中了哪条规则、为何被拒绝”的预览。

#### Scenario: 管理员在权限中心预览某个用户的企业微信入口状态
- **WHEN** 管理员在企业微信灰度分区输入某个 CRM 用户或企业微信用户进行预览
- **THEN** 系统必须返回该用户当前的企业微信入口状态、命中的白名单维度或停用原因
- **THEN** 该预览结果必须与该用户真实运行时会命中的准入判断保持一致

#### Scenario: 管理员切换灰度模式后页面立即刷新
- **WHEN** 管理员将灰度模式从 `PILOT_ONLY` 切换为 `FULL` 或 `DISABLED`
- **THEN** 系统必须持久化新模式并返回最新策略快照
- **THEN** 权限中心页面必须展示新的模式、更新时间和更新人

### Requirement: 系统必须为企业微信灰度策略变更与拒绝行为记录审计

系统 MUST 对企业微信灰度模式变更、白名单调整、停用名单调整、命中预览和准入拒绝行为记录审计事件。审计事件 MUST 至少包含操作者、目标用户或目标策略、变更前后摘要、命中的准入状态和发生时间。

#### Scenario: 管理员调整灰度白名单时生成审计事件
- **WHEN** 管理员新增、删除或修改企业微信灰度白名单
- **THEN** 系统必须记录该次灰度策略变更审计事件
- **THEN** 审计中必须包含修改前后的白名单摘要和当前灰度模式

#### Scenario: 企业微信用户被灰度门闩阻断时生成审计事件
- **WHEN** 某个企业微信用户因 `DISABLED`、`PILOT_REQUIRED` 或 `EXPLICITLY_DENIED` 被阻断
- **THEN** 系统必须记录一条企业微信准入拒绝审计事件
- **THEN** 审计中必须包含目标用户、拒绝状态和拒绝原因摘要
