## MODIFIED Requirements

### Requirement: 系统必须提供统一的企业微信主动通知任务与通道路由能力

系统 MUST 提供统一的企业微信主动通知任务模型，由业务场景先生成标准化通知任务，再由通知中心选择具体发送通道。通知中心 MUST 根据通知交互属性选择企业微信机器人主动推送或企业微信自建应用消息：与机器人上下文强相关的异步结果回推 MUST 走企业微信机器人主动推送；日报提醒、日报确认、团队汇总等固定周期系统通知 MUST 默认走企业微信自建应用消息。业务侧不得直接绕过通知中心分别调用企业微信应用消息和机器人消息两条用户可见链路。

#### Scenario: 固定周期系统通知默认走企业微信自建应用消息
- **WHEN** 系统生成某条日报提醒、日报确认、团队汇总或其它固定周期系统通知任务
- **THEN** 通知中心必须将该任务路由到企业微信自建应用消息通道
- **THEN** 通知中心必须按企业微信自建应用消息通道的发送队列和最小间隔逐条投递
- **THEN** 业务服务不得直接绕过通知中心调用企业微信应用消息接口

#### Scenario: 机器人异步结果回推仍走机器人主动推送
- **WHEN** 用户在企业微信机器人中发起了需要延迟返回结果的任务，且结果完成后需要回推到原有机器人上下文
- **THEN** 通知中心必须将该任务路由到企业微信机器人主动推送通道
- **THEN** 系统必须保留原始机器人上下文信息，避免把该类通知误发到其他入口

#### Scenario: 正式通知按场景分类选择通道
- **WHEN** 某条正式通知同时满足“机器人可单聊主动推送”和“自建应用也具备发送能力”的技术条件
- **THEN** 通知中心必须根据通知场景分类选择最终通道
- **THEN** 系统不得再因为通知是用户可见正式通知就无条件优先选择企业微信机器人主动推送

### Requirement: 用户可见的主动通知必须统一通过通知中心治理

系统 MUST 将用户可见的主动通知统一收敛到通知中心治理，不得让业务模块自行决定绕过接收人解析、权限校验、真实发送开关、测试改投、幂等、重试和审计。企业微信自建应用 MAY 承接固定周期系统通知投递，但不得被设计成聊天入口；用户后续交互、补录、确认或追问 MUST 继续回到企业微信机器人或 Web 工作台完成。

#### Scenario: 用户收到系统通知后仍回到机器人处理
- **WHEN** 用户收到一条通过企业微信自建应用消息投递的 CRM 日报提醒或团队汇总
- **THEN** 该消息必须明确说明后续处理入口是 CRM 智能分析机器人或 Web 工作台
- **THEN** 系统不得要求用户在企业微信自建应用中继续聊天处理该通知

#### Scenario: 业务模块不得绕过通知中心直接发送
- **WHEN** 业务模块需要向用户发送 CRM 相关提醒、汇总、催办或异步结果消息
- **THEN** 业务模块必须提交统一主动通知任务
- **THEN** 业务模块不得直接调用企业微信机器人 SDK 或企业微信应用消息接口完成用户可见通知

### Requirement: 真实消息联调与真实发送必须遵循通知中心通道路由

系统 MUST 保留真实消息发送开关和测试收件人覆盖能力，并由通知中心在最终通道路由后统一执行。真实发送开关关闭时，系统 MUST 阻止向真实业务对象发送，并按配置改投测试接收人；真实发送开关开启时，系统 MUST 使用通知中心解析出的最终通道完成真实发送。系统不得在日报模块或其它业务模块内写死测试接收人，也不得绕过通知中心切换通道。

#### Scenario: 测试联调按最终通道改投测试收件人
- **WHEN** 开发或测试环境触发了真实消息联调，但真实发送开关仍为关闭状态
- **THEN** 系统必须阻止向真实业务对象发送
- **THEN** 系统必须按通知中心解析出的最终通道将消息改投到测试收件人
- **THEN** 非自动化测试运行环境必须真实发送给测试接收人，而不是只返回模拟成功

#### Scenario: 开关开启后按场景路由真实发送
- **WHEN** 真实消息发送开关已显式开启，且接收人映射、权限校验均通过
- **THEN** 系统必须通过通知中心解析出的最终通道完成真实发送
- **THEN** 系统不得因为正式发送或测试发送而绕过通知中心路由策略

### Requirement: 主动通知必须具备独立的幂等、重试与审计留痕

系统 MUST 为主动通知提供独立的幂等键、发送记录、有限重试和审计留痕能力。对于同一业务任务在同一目标、同一时间窗内重复触发的通知，系统 MUST 能识别并阻止重复发送。对于短暂网络抖动或企业微信接口临时失败，系统 MUST 执行有限次重试；连续重试后仍失败时，系统 MUST 保留最终失败状态、失败原因、外部错误码、外部错误信息、重试次数、最终通道和最后尝试时间，不得只在控制台输出错误。

#### Scenario: 重复任务不重复发消息
- **WHEN** 同一条日报汇总任务因调度重试被重复触发，且目标组长和消息版本均未变化
- **THEN** 系统必须命中幂等保护并阻止重复发送
- **THEN** 系统必须在发送记录中标记该次请求为幂等命中

#### Scenario: 连续重试后仍失败需保留失败审计
- **WHEN** 主动通知在有限次重试后仍然发送失败
- **THEN** 系统必须保留最终失败状态、失败原因、外部错误码、外部错误信息、最终通道和重试次数
- **THEN** 系统必须生成对应的审计事件，便于后续排查和补发

#### Scenario: 企业微信频率限制失败需归因到可读原因
- **WHEN** 企业微信机器人主动推送返回 `846607` 或等效频率限制错误
- **THEN** 系统必须将失败原因归一为“企业微信机器人发送频率超限”
- **THEN** 系统必须在通知任务、发送尝试和审计快照中记录外部错误码、外部错误信息和下一步重试策略

## ADDED Requirements

### Requirement: 企业微信机器人主动通知必须具备限流队列与退避重试

系统 MUST 为企业微信机器人主动推送通道提供全局串行发送队列、可配置最小发送间隔和频率限制退避重试。机器人通道不得在同一服务进程内并发发送多条主动通知，也不得在识别到企业微信频率限制后继续按普通短间隔重试。

#### Scenario: 机器人主动通知串行发送
- **WHEN** 多条主动通知同时被路由到企业微信机器人通道
- **THEN** 系统必须按全局队列串行发送
- **THEN** 相邻两次机器人主动发送之间必须满足配置的最小发送间隔

#### Scenario: 频率限制错误按退避策略重试
- **WHEN** 企业微信机器人返回 `846607` 或等效频率限制错误
- **THEN** 系统必须按频率限制退避配置等待后再重试
- **THEN** 系统不得继续使用毫秒级普通重试间隔立即重复发送

#### Scenario: 退避后仍失败保留下一次可补发信息
- **WHEN** 机器人通道按退避策略完成所有有限重试后仍失败
- **THEN** 系统必须将通知任务标记为失败或部分失败
- **THEN** 系统必须保留最后失败原因和建议的下一次人工补发时间

### Requirement: 企业微信自建应用消息必须具备批量发送间隔与退避重试

系统 MUST 为企业微信自建应用消息通道提供全局串行发送队列、可配置最小发送间隔和频率限制退避重试。任何批量发送、调度发送、补发或测试改投都不得并发打满企业微信接口；系统必须逐条发送并在相邻两条消息之间等待配置的最小间隔。

#### Scenario: 自建应用批量通知逐条间隔发送
- **WHEN** 日报提醒、个人确认、团队汇总或补发任务一次产生多条企业微信自建应用消息
- **THEN** 系统必须按自建应用消息队列逐条发送
- **THEN** 相邻两条自建应用消息之间必须满足配置的最小发送间隔

#### Scenario: 自建应用频率限制错误按退避策略重试
- **WHEN** 企业微信自建应用消息接口返回频率限制、系统繁忙、临时不可用或等效错误
- **THEN** 系统必须按自建应用消息退避配置等待后再重试
- **THEN** 系统不得继续使用毫秒级普通重试间隔立即重复发送

#### Scenario: 测试改投也必须遵守应用消息间隔
- **WHEN** 真实发送开关关闭，系统将多条通知改投给测试接收人
- **THEN** 改投后的自建应用消息也必须进入同一个发送队列
- **THEN** 系统不得因为目标都是测试接收人就并发发送或跳过最小间隔

### Requirement: 主动通知审计必须区分计划时间、实际尝试时间与审计创建时间

系统 MUST 在主动通知相关审计中区分计划触发时间、实际发送尝试时间和审计事件创建时间。审计事件的 `createdAt` MUST 表示该审计记录实际创建时间；计划触发时间、业务日期、通知任务 ID、发送开始时间、最后尝试时间和最终通道 MUST 进入审计快照。

#### Scenario: 调度任务延迟发送时审计时间可解释
- **WHEN** 某条日报提醒原计划在 22:00 触发，但实际发送尝试在之后才完成
- **THEN** 审计事件的 `createdAt` 必须记录实际审计创建时间
- **THEN** 审计快照必须同时包含计划触发时间和实际发送尝试时间

#### Scenario: 审计详情能关联底层通知任务
- **WHEN** 管理员在审计中心查看某条日报提醒发送失败事件
- **THEN** 审计详情必须包含底层主动通知任务 ID 和最终通道
- **THEN** 管理员必须能从审计详情定位到外部错误码、重试次数和最后尝试时间

### Requirement: 企业微信自建应用消息适配器必须归一化失败结果

系统 MUST 保证企业微信自建应用消息适配器对 access_token 获取失败、网络异常、HTTP 非 2xx、企业微信业务错误码、请求体构造失败和无效接收人返回标准失败结果。适配器不得让异常穿透并中断主动通知主流程；通知中心必须能保存失败任务、失败尝试和审计事件。

#### Scenario: 自建应用 access_token 获取失败保留失败任务
- **WHEN** 企业微信自建应用消息发送时 access_token 获取失败
- **THEN** 系统必须将通知尝试记录为失败
- **THEN** 系统必须在通知任务和审计快照中保留中文失败原因

#### Scenario: 自建应用消息网络异常保留失败审计
- **WHEN** 企业微信自建应用消息发送请求发生网络异常或 HTTP 非 2xx
- **THEN** 系统必须返回标准失败结果并进入有限重试
- **THEN** 重试后仍失败时必须生成 `PROACTIVE_NOTIFICATION_FAILED` 审计事件

#### Scenario: 自建应用返回无效接收人时标记失败
- **WHEN** 企业微信自建应用消息接口返回 `invaliduser`、`invalidparty` 或 `invalidtag`
- **THEN** 系统必须将对应无效目标写入发送尝试记录
- **THEN** 系统必须把本次任务标记为失败或部分失败，而不得误判为全量发送成功
