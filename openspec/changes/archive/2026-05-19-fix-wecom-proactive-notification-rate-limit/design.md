## Context

现有主动通知底座已经提供 `ProactiveNotificationService`、`WecomBotNotificationService` 和 `WecomAppMessageService`。业务侧通过标准化任务提交通知，底座负责接收人解析、测试改投、幂等、重试和审计。

历史规格为了避免用户混淆企业微信应用和机器人两个入口，要求所有用户可见主动通知固定走企业微信机器人。2026-05-18 试点首日暴露出该策略的新问题：日报提醒属于定时批量通知，3 条真实提醒均在底层 AI Bot WebSocket 发送时触发 `errcode=846607` 频率限制。系统在约 1 秒内对每条消息连续重试 3 次，仍然失败，审计只显示“企业微信机器人主动通知发送失败”，缺少外部错误码和可执行原因。

这次设计需要同时满足两个约束：

- 用户侧仍只能把“CRM 智能分析机器人”作为交互入口，不新增需要用户理解的新聊天入口。
- 定时、批量、单向系统通知不能继续无条件占用 AI Bot 主动推送通道，必须使用更适合批量投递的企业微信自建应用消息，并保留回到机器人继续处理的入口文案。

## Goals / Non-Goals

**Goals:**

- 将主动通知通道路由从“所有用户可见通知固定走 Bot”调整为“会话型走 Bot，固定周期系统通知走自建应用消息”。
- 让日报提醒、日报确认、团队汇总默认避开 AI Bot 发送频率限制。
- 为所有企业微信主动通知通道增加全局串行队列、最小间隔和频率限制退避，批量发送也必须逐条间隔投递。
- 将企业微信外部错误码、错误信息、限流归因、重试计划和最终失败原因写入任务、尝试记录和审计快照。
- 修正日报调度审计时间口径，让计划时间、实际尝试时间和审计创建时间各自清晰。
- 提供短期运维止血配置和长期代码修复路径。

**Non-Goals:**

- 不改变日报目标人群、日报内容生成、团队汇总口径和主管映射规则。
- 不修改企业微信登录、通讯录同步、CRM 身份映射和权限模型。
- 不把企业微信自建应用设计成新的对话入口；后续交互仍回到机器人或 Web 工作台。
- 不在本变更中实现失败任务自动补发后台；只要求失败原因足够清晰，补发能力可另立变更。

## Decisions

### 决策 1：按通知交互属性选择通道，而不是按“用户可见”一刀切

通道路由规则调整为：

- `kind=CONVERSATION_CONTEXT` 或 `audience.type=WECOM_CONVERSATION`：走 `WECOM_BOT_MESSAGE`，因为结果必须回到原机器人上下文。
- `sceneKey` 以 `daily-report.` 开头，或明确标记 `metadata.deliveryClass=SYSTEM_SCHEDULED`：走 `WECOM_APP_MESSAGE`，因为它是固定周期、单向、可批量的系统通知。
- 其它正式通知默认先走 `WECOM_APP_MESSAGE`，除非业务场景明确需要机器人上下文连续性。
- 当通知要求 Bot 通道但接收人不是机器人支持的目标类型时，必须阻断并写审计，不得静默切到另一个通道。

备选方案是继续全部走 Bot 并只调大重试间隔。该方案能缓解小批量发送，但仍把定时批量流量压在 AI Bot 上，后续团队汇总、治理通知、补发任务一旦增多仍会复发。

### 决策 2：企业微信自建应用只承接系统通知投递，不承接聊天处理

自建应用消息用于“送达通知”，消息正文必须包含清晰的后续处理指引，例如继续到 CRM 智能分析机器人补录、确认或查看详情。系统不得把自建应用变成聊天入口，也不得在应用消息中要求用户与应用对话。

真实发送开关关闭时，通知中心已经把真实目标替换为测试接收人；因此在非测试运行环境中，自建应用适配器也必须真实发送给该测试接收人，而不是返回 mock 成功。只有 `NODE_ENV=test` 的自动化测试环境才允许 mock，避免误触真实企业微信。

这样做保留原“单一交互入口”的产品原则，同时使用企业微信官方更适合主动通知的消息接口完成批量投递。备选方案是新建另一个应用入口承接提醒处理，但会重新造成用户入口混淆。

### 决策 3：通知任务记录要保留业务通道和实际投递通道

现有 `preferredChannel` / `resolvedChannel` 继续保留，但实现时需要明确：

- `preferredChannel`：业务侧偏好或历史调用参数。
- `resolvedChannel`：通知中心最终选择的投递通道。
- `metadata.deliveryClass`：建议新增为轻量分类，取值可包括 `CONVERSATION_CONTEXT`、`SYSTEM_SCHEDULED`、`FORMAL_NOTICE`，用于路由与审计解释。
- `attempts[*].failureReason`：面向业务用户的中文原因。
- `attempts[*].externalErrorCode`、`externalErrorMessage`、`retryStrategy`、`nextRetryAt`：用于治理审计和排障的技术快照。

若当前类型不便立即扩展，可先把外部错误字段写入 `attempts[*].metadata` 或 `sessionSnapshot`，但最终应沉淀为可检索字段。

### 决策 4：Bot 通道增加进程内全局串行队列和可配置最小间隔

Bot 通道发送必须在服务进程内串行执行，同一时刻只允许一个主动发送动作进入 SDK。队列对所有 Bot 主动通知共享，默认配置：

- `WECOM_BOT_PROACTIVE_MIN_INTERVAL_MS=15000`
- `WECOM_BOT_DELIVERY_MAX_RETRIES=2`
- `WECOM_BOT_RATE_LIMIT_RETRY_DELAYS_MS=60000,180000,300000`

当识别到 `846607` 时，不使用普通重试间隔，而按频率限制退避配置等待。普通网络失败仍可使用较短但不低于配置下限的重试间隔。

备选方案是把队列落到数据库或 Redis。当前生产实例是单 Node 进程，进程内队列能以最低复杂度解决当前问题；若后续横向扩容，再引入分布式队列或外部锁。

### 决策 5：自建应用消息通道也必须串行限速，批量发送不得并发打满

日报固定周期通知改走企业微信自建应用消息后，应用通道同样必须具备发送队列和最小间隔。通知中心不能因为换了通道就并发发送批量消息；所有日报催报、个人确认、团队汇总和补发任务都必须逐条排队发送。

默认配置建议：

- `WECOM_APP_MESSAGE_MIN_INTERVAL_MS=3000`
- `WECOM_APP_MESSAGE_MAX_RETRIES=2`
- `WECOM_APP_MESSAGE_RATE_LIMIT_RETRY_DELAYS_MS=60000,180000,300000`
- `WECOM_PROACTIVE_BULK_MAX_CONCURRENCY=1`

如果企业微信自建应用接口返回频率限制、系统繁忙、临时不可用或等效错误，应用通道必须按退避配置等待后再重试。即使当前企业微信应用消息阈值高于 AI Bot，也不能依赖供应商阈值承接本系统批量峰值。

备选方案是只限制 Bot，不限制自建应用消息。该方案会把日报批量发送风险从 Bot 迁移到应用消息通道，无法满足“批量发送也要有时间间隔”的上线要求。

### 决策 6：识别 SDK 错误时必须保留原始错误上下文并输出中文归因

`WecomBotNotificationService` 当前捕获异常后只返回 `error.message` 或泛化文案。实现时需要新增错误解析器：

- 从异常对象、错误消息或 SDK ACK 日志文本中提取 `errcode`、`errmsg`、`hint`。
- `846607` 统一归因为“企业微信机器人发送频率超限”。
- 归因结果必须进入尝试记录、通知任务失败原因和 `PROACTIVE_NOTIFICATION_FAILED` 审计快照。
- 对用户可见页面展示中文原因；技术字段只在治理审计或诊断区域展示。

如果 SDK 没有把 ACK 错误抛给调用方，只在日志中输出，则需要在 `WecomTransportService.sendProactiveMessage()` 的 SDK 响应/异常处理处补充包装，确保 sendMessage Promise 能拿到失败状态；不能只依赖 journal 日志排障。

企业微信自建应用适配器也必须遵守同一失败归一规则。`getAccessToken()`、`fetch()`、请求体构造、HTTP 非 2xx、企业微信业务错误码和无效接收人都必须返回标准 `ProactiveNotificationChannelSendResult`，不得让异常穿透到 `deliverWithRetry()` 外层，避免通知任务停留在 `PENDING` 或缺失失败审计。

### 决策 7：日报审计创建时间使用实际发生时间，计划时间进入快照

`DailyReportService.runReminderSweep()` 当前将调度传入的 `sentAt` 作为日报提醒审计的 `createdAt`。当扫描和实际发送耗时较长时，页面会看到“22:00 发送失败”，但底层任务在 22:41 才失败。

调整为：

- `createdAt`：写入该审计记录实际创建时间。
- `sessionSnapshot.scheduledAt`：写入调度器触发时间。
- `sessionSnapshot.businessDate`：继续写业务日期。
- `sessionSnapshot.actualDispatchStartedAt`：写入调用通知中心的开始时间。
- `sessionSnapshot.notificationTaskId`：关联底层主动通知任务。
- `sessionSnapshot.lastAttemptAt`：底层尝试结束时间。

这样审计列表默认按真实事件发生时间排序，详情里仍能看到原计划 22 点任务。

### 决策 8：短期运维止血与长期代码修复分开

在代码修复上线前，生产可先调整：

- `WECOM_BOT_DELIVERY_RETRY_DELAY_MS=60000`
- 若需要补发失败通知，手工逐条补发，每条间隔 1 到 2 分钟。

这只是降低再次撞限概率，不替代通道路由和队列修复。上线长期修复后，日报通知默认走自建应用消息，Bot 通道限流主要保护会话型异步回推。

## Risks / Trade-offs

- [风险] 自建应用消息让用户误以为需要去应用里处理日报。→ [缓解] 消息文案只把应用作为投递载体，后续动作明确回到 CRM 智能分析机器人或 Web 工作台。
- [风险] 企业微信自建应用凭证或可见范围配置不完整导致日报改道后失败。→ [缓解] 上线前增加健康检查和联调脚本，验证 `WECOM_NOTIFY_CORP_ID`、`WECOM_NOTIFY_AGENT_ID`、`WECOM_NOTIFY_SECRET`、目标成员可见范围和 markdown 发送能力。
- [风险] 频率限制错误不能从 SDK Promise 中拿到，只存在日志。→ [缓解] 先在 `WecomTransportService` 包装 SDK 响应和异常；若 SDK 不暴露 ACK 失败，需要向 SDK 层增加事件监听或维护本地 pending request 结果映射。
- [风险] 进程内队列在多实例部署下不能全局限流。→ [缓解] 当前按单实例修复；若生产扩为多实例，必须升级为共享队列或分布式锁。
- [风险] 修改历史“机器人单入口”规格会影响培训口径。→ [缓解] 对外仍强调机器人是业务交互入口，自建应用只是系统通知投递通道，不作为聊天入口。

## Migration Plan

1. **生产止血**
   - 调整 `WECOM_BOT_DELIVERY_RETRY_DELAY_MS` 到 `60000`。
   - 暂停批量补发；如需补发昨天失败的 3 条提醒，按人逐条间隔发送。

2. **代码实现**
   - 在通知底座引入通道分类与路由策略。
   - 将日报通知场景路由到 `WecomAppMessageService`。
   - 为 Bot 通道增加队列、限流退避和错误解析。
   - 扩展任务尝试记录与审计快照。
   - 修正日报审计时间字段。

3. **验证**
   - 单元测试覆盖通道路由、限流错误解析、退避重试和审计字段。
   - 集成测试覆盖日报 22 点提醒在真实发送开关关闭时改投测试人、开启时走自建应用消息。
   - 生产联调先发送给测试接收人，再开启真实发送。

4. **上线**
   - 上线前检查 `WECOM_NOTIFY_*` 自建应用配置完整性。
   - 上线后观察审计中心 `PROACTIVE_NOTIFICATION_FAILED`、`DAILY_REPORT_REMINDER_SENT` 和 journal 中 `846607` 数量。
   - 若自建应用发送异常，临时回滚到上一版本或关闭日报真实发送，避免批量失败扩散。

5. **回滚**
   - 代码回滚时保留运维止血配置，避免重新触发 Bot 短间隔重试。
   - 若仅自建应用配置异常，可先将日报真实发送关闭并改投测试接收人，修复凭证和应用可见范围后再恢复。

## Open Questions

- 企业微信自建应用消息在当前企业内的频率限制阈值是否需要单独压测确认。
- SDK 是否能直接暴露 `846607` ACK 失败对象；若不能，需要在实现阶段确认可监听的失败事件或升级 SDK。
- 自建应用消息中的“继续处理”入口最终指向机器人、Web 工作台还是二者都提供，需要产品联调时确定。
