## Context

当前后端已经通过 `CrmCustomerApiService` 和 `CrmOpportunityApiService` 封装了 CRM Open API 查询能力，也已经通过 `CrmFollowUpWritebackService` 封装了受控写回能力，但尚未提供“新增客户”的统一后端入口。业务截图中的必填字段不仅包含标准客户字段，还包含 `IT决策权所在地`、`统一社会信用代码` 这类 CRM 自定义字段；而官方“新增客户”文档只直接列出标准字段，这意味着实现时必须在“业务字段名”和“CRM 自定义字段键名”之间做一层受控映射。

当前阶段用户明确要求“先实现接口提供口子，暂不开发机器人交互”，因此本次设计需要优先满足后端可调用、可测试、可配置，且不能破坏现有“机器人写入暂不开放”的边界。

## Goals / Non-Goals

**Goals:**

- 提供受会话鉴权保护的内部新增客户接口，供后续机器人或其它受控调用方复用。
- 对截图中标注的必填字段做后端校验，并把业务字段映射为 CRM 官方接口接受的标准字段和自定义字段。
- 复用当前登录态中的 CRM access token 调用 CRM 官方 `/api/v2/customers`，禁止数据库直写。
- 支持通过环境变量配置默认字段值与自定义字段键名，减少后续机器人调用方对 CRM 内部实现细节的感知。
- 在 `test`/mock 模式下提供稳定响应，便于集成测试和契约测试覆盖。

**Non-Goals:**

- 本轮不改造企业微信机器人意图识别、追问、确认或发送链路。
- 本轮不实现客户创建后的自动联系人补建、自动商机补建、审批流编排或审计扩展。
- 本轮不提供自定义字段元数据查询界面，也不自动探测 CRM 字段映射。

## Decisions

### 决策一：新增内部路由 `/api/v1/crm/customers`，不复用外部 CRM 路径

- 选择：新增内部接口 `/api/v1/crm/customers`。
- 原因：当前服务自身已经有全局前缀 `api/v1`，如果直接复用 CRM 原始路径，容易与已有查询控制器中混入的 `/api/v2/...` 风格继续叠加，造成边界不清。内部路由单独挂在 `crm/` 命名空间下，更能表达“这是本系统对 CRM Open API 的受控封装”。
- 备选方案：直接暴露 `/api/v2/customers` 的透传路由。
- 不采用原因：会让内部 API 与外部 CRM Open API 的职责混淆，也不利于后续继续做字段收敛和错误标准化。

### 决策二：使用“业务友好 DTO + 服务层映射”，不把 CRM 原始字段名暴露给调用方

- 选择：控制器接收 `name`、`phone`、`itDecisionLocation`、`unifiedSocialCreditCode` 等业务化字段，服务层负责映射成 `customer[...]` 及对应自定义字段键名。
- 原因：截图中的字段名称是后续机器人和业务配置更容易理解的表达；如果直接要求调用方传 `customer[text_asset_xxx]`，会把 CRM 内部字段实现泄漏到上层调用方。
- 备选方案：直接透传 `customer[...]` 或 `text_asset_xxx` 键名。
- 不采用原因：调用方难以维护，且一旦 CRM 自定义字段键名变更，所有上游调用都会被迫同步修改。

### 决策三：标准必填字段使用后端强校验，自定义字段键名缺失时快速失败

- 选择：接口强制校验 `name`、`phone`、`itDecisionLocation`、`unifiedSocialCreditCode`；其中后两项通过环境变量映射到 CRM 自定义字段键名，若未配置则直接返回 `400`。
- 原因：截图已把这些字段标为必填，后端必须防止调用方绕过前端校验；而自定义字段键名是环境差异项，缺失时最安全的行为是拒绝写入而不是静默丢字段。
- 备选方案：字段映射缺失时忽略该字段或回退到原始 key。
- 不采用原因：会导致“业务上必填，但 CRM 实际没写进去”的隐性错误。

### 决策四：标准字段默认值从“会话用户 + 环境配置”补齐

- 选择：`user_id` 默认取当前会话 `crmUser.id`，`want_department_id` 默认取当前用户的首个部门；CRM 文档要求的 `category`、`source` 则允许调用方显式传入，否则尝试使用环境变量默认值。
- 原因：截图中的负责人和所属部门都表现为系统默认值，适合由后端兜底；而 `category`、`source` 是 CRM 标准必填字段，但截图未标红，说明业务上可能已有固定默认值或预置选择。
- 备选方案：把所有 CRM 文档必填项都要求调用方显式传入。
- 不采用原因：会让接口口子对上层过于生硬，不符合“后续给机器人调用”的目标。

### 决策五：新增客户复用现有 `CrmCustomerApiService` 扩展创建能力

- 选择：在 `CrmCustomerApiService` 中新增 `createCustomer`，配合独立控制器暴露 HTTP 接口。
- 原因：当前客户相关 CRM Open API 访问已经集中在该服务中，继续在同一服务中维护查询与创建，能够复用日志、鉴权和 mock 策略。
- 备选方案：新建单独的 `CrmCustomerCreateService`。
- 不采用原因：会把同一 CRM 领域的能力拆散，增加后续维护成本。

## Risks / Trade-offs

- [风险] CRM 环境的自定义字段键名与截图字段不一致，导致字段无法成功落库 → 缓解：把键名做成显式环境变量，并在缺失时返回可定位的 400 错误。
- [风险] 当前用户缺少部门信息，无法补齐 `want_department_id` → 缓解：允许调用方显式传入部门；若两者都缺失则返回 400，并提示必须补齐所属部门。
- [风险] CRM 文档未对所有自定义字段写入格式给出示例，生产环境可能出现字段类型不匹配 → 缓解：本轮优先按文本字段写入，保留响应日志与错误透传，后续结合真实字段配置校正。
- [风险] 本轮未接入机器人审计流程，未来接入时可能需要补充更细的审计事件 → 缓解：当前先把控制器、服务和响应结构固定，审计扩展放到后续机器人集成变更中。

## Migration Plan

1. 增加环境变量占位符，声明默认 `category/source` 取值及必填自定义字段键名。
2. 实现控制器、服务映射、mock 响应与错误处理。
3. 更新 OpenAPI 契约和一期规格文档，标明当前仅开放后端接口口子。
4. 通过集成测试验证接口校验、默认值补齐和 mock 创建返回。
5. 上线时先在目标环境补齐字段映射配置，再启用真实 CRM Open API 调用。

## Open Questions

- `客户来源` 在目标 CRM 环境中的默认值是否已经固定，如果没有，需要后续由业务确认枚举值。
- `IT决策权所在地` 与 `统一社会信用代码` 在 CRM 中的真实字段类型是否都为文本；若包含枚举或查重逻辑，后续需要补充更细分的字段格式校验。
