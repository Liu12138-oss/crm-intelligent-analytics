## Context

当前 CRM 项目里的智能问数已经具备统一 AI 入口、受控分析编排、结果包、执行快照和企业微信 / Web 双入口基础，但整体仍然更像“结构化分析平台”而不是“固定功能 + 自由问数双轨机器人 / 工作台”。具体表现为：

- 企业微信固定功能与自然语言问数仍在入口和交付层混杂，固定功能虽然已逐步 AI-first 化，但问数链路没有真正收敛成独立主路。
- Web 问数默认仍围绕 `analysis-intent-pack`、结构化结果块和厚结果页展开，难以承接完全自由的问法。
- `GUARDED_DIRECT_QUERY` 已存在，但仍是可选执行模式，没有成为不固定问数的默认主链。
- 结果交付仍以结构化结果块为主，虽然适合审计和导出，但不利于机器人和 Web 直接共享同一份更自由、更完整的 AI 总结。
- 当前前端仓库没有现成 Markdown 渲染依赖，企业微信侧则天然支持 markdown 消息，这导致两个渠道在“同源结果、不同展示”上还没有统一格式。

同时，本次设计必须遵守以下现实约束：

- 固定功能边界不能退化。跟进商机、跟进客户、创建客户、创建商机、项目查询等流程仍必须走受控程序流。
- 自由自然语言问数必须允许用户完全自由表达，但不能放开自由数据库连接、自由 schema 枚举、自由写库或绕过 API-first / 白名单 / 审计。
- Web 与企业微信必须复用同一套问数后端事实来源，不允许出现两套问数逻辑。
- Markdown 只能作为展示格式，不能替代统一结果后端中的执行快照、结果集、权限摘要和审计记录。

## Goals / Non-Goals

**Goals：**

- 将机器人和 Web 的不固定自然语言问数统一收敛为 Hermes 风格主链：`自然语言问数 -> AI 生成受控查询任务 -> 程序执行 -> AI grounded 总结 -> 渠道裁剪交付`。
- 明确固定功能与自由问数的双轨边界，让固定功能继续走程序流，自由问数走受控直查主链。
- 在统一结果包中增加 Markdown 展示载荷，支持企业微信 markdown 输出与 Web Markdown 预览。
- 将自然语言识别从窄枚举结构化意图，调整为面向自由输入的最小受控语义抽取。
- 引入问数知识层，统一沉淀字段别名、口径别名、组织别名、码表翻译、高频模板和已验证查询示例。
- 保留现有权限、安全、审计、统一结果后端和 API-first 路由，不为自由问数牺牲边界控制。

**Non-Goals：**

- 不改动固定功能的执行确认、写回、创建、登录、会话和统一鉴权链路。
- 不让 AI 直接持有数据库连接、自由执行任意 SQL、自由探测 schema 或自行决定写操作。
- 不把 Markdown 作为唯一结果模型，也不取消现有导出、重跑、审计和最近查询所依赖的结构化结果底座。
- 不要求企业微信完整复刻 Web 的图表与明细体验；企微只消费裁剪后的 Markdown 与必要结构块。

## Decisions

### 决策 1：机器人与 Web 问数统一采用“双轨入口”，固定功能与自由问数分开编排

入口层先将用户输入分为两类：

- 固定功能：跟进、日报、创建、项目查询、帮助等已有主题入口，继续走固定程序流。
- 自由问数：没有固定模板边界、需要自由问法理解和数据分析的问题，进入统一问数后端。

路由规则上，固定功能优先于自由问数。只要入口理解结果明确命中固定功能主题，系统就不得把它降级成自由问数代理。

原因：

- 固定功能本质上是业务动作，要求确认、幂等、对象唯一性和写回边界，不能与问数主链混用。
- 自由问数需要更大表达空间，如果继续和固定功能共用一套窄规则或一套结构化表单，会持续误判。

备选方案：

- 所有机器人输入统一走问数代理。问题是固定功能边界会被冲掉。
- 所有入口继续混合在一个统一意图包里硬分流。问题是问数和固定功能的演进节奏不同，容易继续积累大而全 prompt。

### 决策 2：不固定自然语言问数主链默认切到“受控直查优先”，计划执行退为复杂兜底

自由问数的后端统一采用以下主链：

1. 用户自由输入自然语言问题。
2. AI 基于最小受控语义和问数知识层，生成 1 到 3 个受控查询任务。
3. 程序对每个任务执行 API-first 路由、白名单、AST、预检、权限注入和行数 / 超时限制。
4. 程序执行并汇总结果集，形成统一 `ResultBundle`。
5. AI 只基于 `ResultBundle`、权限摘要和执行快照生成 grounded 总结。
6. Web 与企业微信消费同一份总结源，做不同展示裁剪。

其中：

- `GUARDED_DIRECT_QUERY` 成为自由问数默认主路。
- `PLAN_EXECUTION` 保留给高歧义、多轮补问、复杂分解或主链生成失败后的受控兜底。

原因：

- 这与实际项目“自然语言问数、AI 用 SQL 查库、AI 分析总结”的预期一致。
- 当前已有安全栈可以承接受控直查，不需要再把所有问题都压到窄意图模型里。

备选方案：

- 继续以结构化计划执行为默认主路。问题是会限制自由表达和问数覆盖面。
- 完全 Hermes 式自由 SQL 代理。问题是缺少现有项目需要的安全和审计护栏。

### 决策 3：自然语言识别改成“最小受控语义抽取”，不再把窄枚举意图当问数主 contract

自由问数场景下，AI 入口理解层不再要求先把问题压成固定的少量 `domain / metric / dimension` 枚举组合，而是优先抽取最小受控语义：

- 查询对象：客户、商机、合同、回款、负责人、部门、组织等。
- 时间范围：本月、本季度、最近 30 天、去年同期、近三个月等。
- 范围条件：区域、团队、负责人、部门、组织、类别等。
- 结果意图：排行、趋势、分布、明细、对比、解释复用、重跑改条件。
- 输出偏好：更关注金额、数量、赢单率、转化、贡献、进度等。

该层输出不直接等于最终 SQL，也不要求一次就落到狭窄枚举；它的作用是约束 AI 生成受控查询任务，而不是代替查询任务。

原因：

- 用户输入完全自由时，先做最小语义抽取更稳，更符合 Hermes 的能力组织方式。
- 继续要求严格窄枚举，会导致大量真实问法被误判成补问或阻断。

备选方案：

- 扩大现有枚举集合。问题是扩到一定程度后仍然会被自由表达击穿。
- 取消结构化理解直接做端到端 SQL。问题是失去入口可审计性。

### 决策 4：新增统一问数知识层，承接 Hermes 的可复用资产

统一问数知识层由以下内容组成：

- 字段中文别名与业务别名。
- 部门 / 区域 / 团队归一规则。
- 高风险字段和敏感字段黑白名单。
- 已批准的业务口径说明。
- 自定义字段码表与翻译字典。
- 高频查询模板与已验证查询示例。
- 典型问法、反例问法和回归样例。

问数知识层只用于三件事：

- 帮助入口理解层更稳定地识别自由表达。
- 为 AI 生成受控查询任务提供参考上下文。
- 为程序选择模板 SQL 或官方 API 适配器提供已验证资产。

原因：

- Hermes 的真正价值不在于“自由代理框架”，而在于把数据库知识翻成业务语义知识包。
- 当前仓库虽然有库表梳理文档，但缺少可直接被问数主链消费的统一知识层。

备选方案：

- 继续依赖代码里的本地规则和静态 prompt。问题是难维护，难持续扩充。
- 直接让 AI 即时 schema-search。问题是耗时高、稳定性差，也不符合当前受控边界。

### 决策 5：统一结果后端继续保留，但新增 Markdown 交付载荷

统一结果包继续保留现有结构化底座，包括：

- 执行模式、执行来源、适配器命中、权限快照。
- 统一结果集、数据集引用、一致性标识、导出与重跑能力。
- 审计快照、最近查询、结果复用和解释复用所需的结构化信息。

在此基础上新增三类展示载荷：

- `groundedMarkdown`：完整 Markdown 总结，作为 Web 默认阅读内容。
- `wecomMarkdown`：企微裁剪版 Markdown，限制长度、块数和语法子集。
- `markdownOutline`：结构化分段摘要，便于 Web 做目录、折叠和辅助侧栏。

Markdown 生成必须基于同一份 `ResultBundle`，不得重新读取数据。Markdown 不是唯一事实源，只是统一展示格式。

原因：

- 机器人和 Web 都能消费 Markdown，便于共享一份结果。
- 保留统一结果底座后，导出、审计、重跑和权限校验不需要重做。

备选方案：

- 只保留 Markdown，不保留结构化结果。问题是审计、导出和重跑会失去稳定底座。
- 继续只保留结构化结果块。问题是用户阅读体验和跨渠道复用不够自然。

### 决策 6：企业微信消费 Markdown 子集，Web 增加 Markdown 预览与安全渲染

企业微信展示规则：

- 使用 markdown 消息作为主输出格式。
- 只保留标题、粗体、列表、引用、分隔线、短表格摘要或列表化摘要。
- 超出长度时分块发送，必要时提示到 Web 查看完整结果。

Web 展示规则：

- 工作台和详情页新增 Markdown 预览区域。
- 同页继续保留明细数据、执行信息、重跑、导出和审计摘要。
- Markdown 渲染必须带安全清洗，禁止原始 HTML 注入。

实现上允许新增 Markdown 渲染依赖，但必须在前端层完成渲染和清洗，不把原始不受控 HTML 交给浏览器直接插入。

原因：

- Hermes 已验证企微 markdown 交付可行。
- Web Markdown 预览更符合“AI 总结自由表达”的目标，同时不影响结构化底座。

备选方案：

- Web 只显示纯文本。问题是表达力不足。
- Web 直接显示原始 HTML。问题是安全风险高。

### 决策 7：AI grounded 总结与结构化细节解耦，摘要由 AI 生成，细节仍由程序提供

AI grounded 总结负责：

- 执行摘要。
- 关键发现。
- 风险提示。
- 建议下一步问题。
- Markdown 章节组织与措辞。

程序负责：

- 原始结果集。
- 指标快照。
- Top N / 明细表。
- 执行与审计信息。
- 导出与重跑所需事实。

这意味着 Web 页面既可以展示 AI 总结，也可以在下方展开数据明细；企业微信则优先消费 Markdown 摘要与少量关键数字。

原因：

- 这样可以让 AI 自由表达，但避免它承担数值事实源角色。
- grounded 失败时，系统仍可回退到结构化结果加模板摘要，而不会失去全部交付能力。

备选方案：

- 让 AI 输出整页内容和全部细节。问题是事实边界不清。
- 继续全部用程序模板生成。问题是自然表达能力不足。

## Risks / Trade-offs

- [自由自然语言识别误判固定功能或问数] → 固定功能优先匹配，所有 fallback 与路由结果写入入口审计快照。
- [AI 受控查询任务生成不稳定，导致执行失败率上升] → 保留模板优先、API-first、最小自动纠错和计划执行兜底。
- [Markdown 总结与结构化细节不一致] → Markdown 只能消费统一结果包，交付前执行一致性校验。
- [企业微信 markdown 能力有限，复杂结果难完全承载] → 企微只展示裁剪版摘要，完整阅读回到 Web。
- [Web 引入 Markdown 渲染带来 XSS 风险] → 使用安全渲染依赖并强制做 HTML 清洗，禁止原样渲染模型返回的 HTML。
- [知识层扩张后维护成本增加] → 以模板、别名、码表和已验证样例为最小资产单元，纳入治理与回归测试。

## Migration Plan

1. 先新增 OpenSpec 规格与设计，冻结双轨入口、Hermes 化问数主链和 Markdown 交付边界。
2. 在后端保留现有统一结果底座的前提下，将自由问数默认主路切到 `GUARDED_DIRECT_QUERY`，固定功能保持不动。
3. 引入问数知识层，先沉淀高频模板、别名和已验证样例，再逐步扩充。
4. 增加 grounded Markdown 生成与渠道裁剪，先支持企业微信摘要 markdown，再补 Web Markdown 预览。
5. 在运行期保留 `PLAN_EXECUTION` 与模板摘要作为显式 fallback，并记录命中原因。
6. 当 Hermes 化问数主链稳定后，逐步收缩过窄的问数枚举结构和厚结果页默认展示。

回退策略：

- 若受控直查生成稳定性不足，可临时恢复问数默认走 `PLAN_EXECUTION`，但保留新知识层与 Markdown 交付能力。
- 若 Markdown 渲染或企业微信消息交付异常，可暂时回退到结构化结果摘要与现有阶段块，不影响统一结果后端。

## Open Questions

- Web Markdown 预览首版是直接放在工作台主结果区，还是先放在详情页作为灰度入口。
- 企微 markdown 单条长度和多块分发策略是否需要纳入统一配置中心。
- 问数知识层的首版维护入口放在治理后台、代码资产还是两者并存。
- 对于“完全自由但高歧义”的问数，首版是优先补问还是优先尝试受控直查后再返回澄清建议。
