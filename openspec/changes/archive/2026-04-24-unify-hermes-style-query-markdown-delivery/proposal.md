## Why

当前智能问数链路仍然偏向“结构化意图 + 结构化报表”的分析平台思路，与实际项目中“固定功能走程序流、不固定功能走自由自然语言问数”的使用方式不完全一致。随着 Web 与企业微信问数要统一后端、共享结果并支持更自由的表达与展示，系统需要尽快收敛到 Hermes 风格的受控直查与 grounded 总结主链，同时保留现有权限、安全、审计和统一结果底座。

## What Changes

- 将不固定的自然语言问数主链调整为“AI 生成受控查询任务 -> 程序执行并返回结果集 -> AI 基于结果集和权限摘要生成 grounded 总结”。
- 保留固定功能主题入口与受控业务动作的既有程序流，不让其退化为自由问数代理。
- 统一 Web 与企业微信智能问数后端，两个渠道只在展示层分叉，不再维护两套问数执行逻辑。
- 将问数结果交付格式升级为“统一结果包 + Markdown 展示载荷 + 渠道裁剪”，支持企业微信 markdown 消息和 Web Markdown 预览。
- 将自然语言识别重点从窄枚举结构化意图，调整为“自由问法下的最小受控语义抽取”，优先支持对象、时间、范围、排序、排行/趋势/分布/明细等问数要素。
- 强化问数知识层，沉淀字段别名、口径别名、组织别名、高频模板、码表翻译和已验证查询示例，减少纯规则和纯自由 SQL 的不稳定性。
- 保留现有 API-first、白名单、AST、预检、权限注入、审计快照和统一结果后端，不放开自由写库、自由 schema 探测或绕过受控工具边界的路径。

## Capabilities

### New Capabilities

- 无

### Modified Capabilities

- `controlled-analysis-orchestration`: 将自然语言问数主链调整为 Hermes 化受控直查优先，并明确固定功能与自由问数的双轨编排边界。
- `structured-analysis-result-delivery`: 将统一结果交付扩展为同源 Markdown 展示载荷，支持企业微信 markdown 输出、Web Markdown 预览和渠道裁剪。
- `ai-entry-intent-orchestration`: 将问数入口理解从窄枚举结构化识别调整为面向自由自然语言输入的最小受控语义抽取，并加强问数场景下的自由表达识别与 fallback 留痕。

## Impact

- 影响后端分析主链：`backend/src/modules/analysis/*` 中的入口理解、执行模式建议、受控直查任务生成、结果交付和渠道呈现。
- 影响企业微信问数交付：需要在统一结果包基础上生成符合企微限制的 markdown 载荷，并与固定功能消息流并存。
- 影响 Web 智能分析工作台：需要新增 Markdown 预览结果区域或等价组件，同时保留明细数据、重跑、导出和审计入口。
- 影响查询知识资产：需要把高频模板、业务口径、字段别名、自定义字段码表和已验证查询示例纳入统一问数知识层。
- 影响测试与验收：需要补齐自由自然语言识别、受控直查安全校验、grounded 总结一致性、Markdown 渲染与双渠道裁剪的回归验证。
