## Context

我们已经验证过两件关键事实：

1. 企业微信机器人真实入站消息中的发送者标识是企业微信原生 `userid`，例如 `WangLiang02`。
2. EMM 同步源中的主键是另一套 `uiduserid`，例如 `2507890364643945379`，并不能可靠地通过 `WangLiang02` 直接反查出来。

这意味着：即使 EMM 能同步出用户的手机号和邮箱，机器人身份识别仍然要先跨过“`from.userid` 到 EMM 用户主键”的不稳定映射，才能进入兜底识别流程。相比之下，企业微信官方通讯录 API 天然围绕企业微信原生 `userid` 工作，更适合作为机器人身份识别的目录来源。

我们已经在官方文档中确认的关键接口包括：

- `GET /cgi-bin/gettoken`：获取企业微信应用 `access_token`
- `GET /cgi-bin/department/list`：获取部门列表
- `GET /cgi-bin/user/list`：获取部门成员详情
- `GET /cgi-bin/user/get`：读取单个成员详情

因此，本次设计不再继续增强 EMM 方案，而是直接用企业微信官方 API 替换整个目录同步链路。

## Goals / Non-Goals

**Goals：**

- 使用企业微信官方通讯录 API 同步部门和成员目录。
- 让机器人身份识别直接复用企业微信原生 `userid`。
- 提供生产环境一键同步入口，发现目录未同步时可立即执行。
- 清理 EMM 目录同步相关配置、代码和文档，避免双方案并存。

**Non-Goals：**

- 不保留 EMM 和企业微信官方 API 双活同步。
- 不继续围绕 EMM `uiduserid` 做补丁式映射。
- 不修改 CRM 主库，只更新应用侧镜像和识别链路。
- 不改变机器人 AI 会话编排和结果回传主体逻辑。

## Decisions

### 决策一：以企业微信原生 `userid` 作为同步与识别的统一主键

应用侧镜像表和识别逻辑都围绕企业微信原生 `userid` 建模，而不再使用 EMM 返回的 `uiduserid` 作为主键。

这样做的原因：

- 机器人消息回调直接拿到的就是 `userid`；
- 企业微信官方 `user/get`、`user/list` 等接口也围绕 `userid`；
- 可以把“回调标识”和“目录标识”统一成同一套主键。

### 决策二：同步策略采用“部门列表 + 部门成员详情 + 成员详情补齐”

建议同步链路为：

1. 先调用 `department/list` 获取部门树；
2. 按授权范围筛出需要同步的目标部门；
3. 逐个调用 `user/list` 获取部门成员详情；
4. 必要时调用 `user/get` 补齐单个成员的完整字段。

这样做的原因：

- `department/list` 适合建立组织树；
- `user/list` 适合按部门成批拉取成员；
- `user/get` 可用于对单个成员做字段补齐或重新校验。

### 决策三：当前生产环境只同步“联软科技集团”授权范围

生产环境一键同步脚本和内部同步入口只同步“联软科技集团”及其下属组织，不同步 `ROOT` 全量组织树。

这样做的原因：

- 这是当前现行约束；
- 可以降低同步范围、权限范围和误同步风险；
- 也更符合机器人当前实际服务对象。

### 决策四：移除 EMM 方案，而不是保留兼容层

本次变更要求删除：

- `EMM_API_*` 配置
- EMM 客户端
- EMM 同步服务和相关脚本
- EMM 运维说明

这样做的原因：

- 保留兼容层会让目录同步来源不确定；
- 机器人身份识别会继续面临“两套主键”的混乱；
- 运维也无法判断到底应该执行哪套同步入口。

## Risks / Trade-offs

- [企业微信官方 API 的字段返回受应用权限范围影响] → 明确要求同步应用具备通讯录可见范围，并把当前只同步“联软科技集团”的条件写入脚本与文档。
- [逐部门拉取成员详情的调用量高于 EMM 增量接口] → 通过范围限制、缓存 token 和按部门增量刷新降低调用量。
- [删除 EMM 后短期内失去现有同步记录] → 在切换前保留应用侧镜像表数据，但只让官方 API 继续写入。
- [部门成员详情接口不一定包含全部字段] → 对关键成员可追加 `user/get` 做补齐。

## Migration Plan

1. 先新增企业微信官方通讯录配置和客户端。
2. 将目录同步脚本和内部同步入口改写为调用企业微信官方 API。
3. 改造身份识别逻辑，直接基于企业微信原生 `userid` 命中应用镜像和 CRM 映射。
4. 删除 EMM 相关配置、客户端、文档和脚本。
5. 重新执行一次“联软科技集团”范围的一键同步，并验证目标用户识别。

回退策略：

- 若官方 API 方案在灰度阶段异常，可临时回滚到上一版代码，但不应再继续迭代 EMM 方案。

## Open Questions

- 当前企业微信官方应用是否已经具备读取“联软科技集团”范围内通讯录的权限？
- `user/list` 是否已经足够返回机器人识别所需的手机号 / 邮箱，还是必须统一追加 `user/get`？
- 是否需要同时接入企业微信的成员 / 部门变更回调，以便后续做准实时刷新？
