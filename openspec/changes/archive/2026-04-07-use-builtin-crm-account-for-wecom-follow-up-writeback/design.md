## Context

当前企业微信机器人能基于 CRM 映射识别发送人，并复用现有受控链路完成跟进草稿确认与写回。但真正执行 `POST /api/v2/revisit_logs` 时，`CrmFollowUpWritebackService` 仍从当前用户的本地会话里读取 `crmAccessToken`。企业微信发送人通常不会长期保持 Web 登录态，因此链路会在最终写回阶段因为缺少 token 失败。

这次调整必须遵守两条既有边界：

- 继续优先复用 CRM 官方 Open API，不回退到数据库直写。
- 不改动当前稳定的 Web 账号密码登录、企业微信扫码登录和统一会话装载链路。

## Goals / Non-Goals

**Goals:**

- 让企业微信机器人跟进写回在没有发送人登录态的情况下，仍可通过受控内置 CRM 账号完成 Open API 写回。
- 为机器人写回补充实际企微发送人的内容署名，避免 CRM 中所有跟进都只显示系统账号而无法区分真实跟进人。
- 保持现有确认、失败保留、审计留痕、幂等保护和群分享后续流程不变。

**Non-Goals:**

- 不调整 Web 登录、扫码登录、会话管理或通用 CRM 认证能力。
- 不改变企业微信问数、商机查询、客户查询等其他链路的认证来源。
- 不引入新的数据库写回路径，也不修改跟进写回对象范围。

## Decisions

### 决策一：机器人跟进写回单独使用“内置账号换 token”而不是复用发送人会话

新增机器人写回专用的内置 CRM 账号配置，通过 `POST /api/v2/auth/login` 在服务端换取 `user_token`，再调用 `POST /api/v2/revisit_logs`。这样可以维持官方 API 路径，同时避免机器人链路被发送人的 Web 登录态绑定。

不采用“要求企业微信用户先登录 Web 才能写回”的方案，因为这会让机器人链路失去独立性，且与当前已具备的企微身份映射能力冲突。

不采用“在配置里直接存长效 token”的方案，因为 token 失效后不可自恢复，且泄漏风险更高。账号密码虽然也属于敏感配置，但可通过已有本地配置约定管理，并支持运行时自动换取新 token。

### 决策二：内置账号 token 只在写回服务内部获取与缓存，不扩散到通用查询服务

本次只修改 `CrmFollowUpWritebackService` 的认证来源，查询链路仍保持现状。这样可以把影响面限制在用户明确授权的受控写回路径，避免把“系统账号代写回”误扩展为“系统账号代查询”。

实现上可在写回服务内部新增一个轻量 token 提供器，按需登录并基于过期时间做短时内存缓存；登录失败时直接返回可观测错误，并沿用现有草稿保留逻辑。

### 决策三：实际跟进人署名在草稿阶段就前置写入，并在更新内容时做幂等归一

机器人确认给用户看的草稿内容、最终发往 CRM 的内容以及失败后保留的草稿内容，都统一使用 `实际跟进人：跟进内容` 格式。这样用户在确认时看到的就是最终落库文本，不会出现“确认内容”和“最终写回内容”不一致。

为避免用户手动修改时出现重复前缀，系统在每次生成或更新草稿时都先剥离已有的相同发送人前缀，再重新拼接标准格式。

## Risks / Trade-offs

- [内置账号配置缺失或密码错误] → 写回会继续失败，但必须返回明确的配置/认证失败原因，并保留草稿供后续重试。
- [系统账号作为 CRM 操作人会弱化原生字段中的真实归属] → 通过内容署名、审计事件中的 `requesterId/requesterName` 和企微上下文补齐真实操作者信息。
- [内置账号 token 频繁登录可能触发 CRM 认证限流] → 在服务内做短时缓存，优先复用尚未过期的 token。

## Migration Plan

1. 在部署环境补充 `CRM_OPEN_API_WRITEBACK_LOGIN` 与 `CRM_OPEN_API_WRITEBACK_PASSWORD` 两个配置项。
2. 发布后先在测试环境验证“未建立 Web 登录态的企微账号”仍可完成跟进写回。
3. 若出现异常，可先移除新增配置并回滚代码；回滚后系统恢复为“依赖发送人登录态”的旧行为。
