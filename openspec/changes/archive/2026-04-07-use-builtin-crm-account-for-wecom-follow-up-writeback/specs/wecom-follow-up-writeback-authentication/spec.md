## ADDED Requirements

### Requirement: 企业微信机器人跟进写回必须支持内置 CRM 账号认证

系统 MUST 为企业微信机器人受控跟进写回提供独立的内置 CRM 账号认证能力。进入机器人跟进写回确认流程后，系统 MUST 通过受控配置的内置账号换取 CRM Open API token，再调用官方接口 `POST /api/v2/revisit_logs` 完成写回，而不是依赖当前企微发送人的 Web 登录会话 token。若内置账号未配置、登录失败或 token 换取失败，系统 MUST 保留当前草稿、返回明确失败原因，并允许用户后续修改或重试。

#### Scenario: 发送人没有有效登录态时仍可完成机器人写回

- **WHEN** 企业微信发送人已经完成身份识别和跟进内容确认，但当前没有可用的 `CRM access token`
- **THEN** 系统必须改用已配置的内置 CRM 账号换取 Open API token 并继续执行写回
- **THEN** 系统不得要求发送人先去 Web 端重新登录后才能继续当前机器人写回

#### Scenario: 内置账号认证失败时必须保留草稿并返回明确原因

- **WHEN** 企业微信机器人执行跟进写回时，内置 CRM 账号未配置、密码错误或 CRM 登录接口返回失败
- **THEN** 系统必须将本次写回标记为失败并保留草稿内容
- **THEN** 系统必须向当前会话返回可理解的失败提示，说明是机器人写回内置账号认证失败，而不是笼统提示发送人缺少登录态

### Requirement: 企业微信机器人写回内容必须携带实际跟进人署名

系统 MUST 在企业微信机器人生成、展示、修改和最终写入 CRM 的跟进内容中统一携带实际企微发送人的署名，格式为 `实际跟进人：跟进内容`。当 CRM 最终操作人变为系统内置账号时，系统 MUST 通过该署名保留真实业务跟进人的可识别信息。若用户在修改草稿时已经输入相同发送人的前缀，系统 MUST 避免重复追加。

#### Scenario: 初次生成草稿时展示标准署名格式

- **WHEN** 企业微信机器人根据当前发送人的内容生成跟进写回草稿
- **THEN** 草稿内容必须以 `当前发送人姓名：` 作为前缀
- **THEN** 用户在确认前看到的内容必须与最终写入 CRM 的内容格式保持一致

#### Scenario: 修改草稿后仍保持单一署名

- **WHEN** 用户在待写回阶段重新编辑跟进内容，且文本中已包含相同发送人的署名
- **THEN** 系统必须在保存草稿和执行写回前做归一化处理，避免出现重复的 `当前发送人姓名：当前发送人姓名：内容`
