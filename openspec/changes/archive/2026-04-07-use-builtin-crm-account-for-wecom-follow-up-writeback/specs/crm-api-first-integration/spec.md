## MODIFIED Requirements

### Requirement: CRM业务能力必须优先复用官方 API

涉及 CRM 业务对象读写的新增能力，以及对既有非认证能力的功能改造，系统 MUST 先评估并优先复用 CRM 官方 API。只要官方 API 已满足目标对象、动作、字段、权限语义与稳定性要求，就不得因数据库更熟悉、实现更快或已有旧代码而绕过官方 API。对于企业微信机器人跟进写回这类受控写入场景，系统 MAY 通过受控内置账号换取执行所需的 API token，但仍 MUST 继续调用官方 API 完成写回，不得因此回退到数据库直写或新增未审计的内部写回路径。

#### Scenario: 新能力命中可用官方 API

- **WHEN** 团队要实现一个新增 CRM 业务能力，且官方 API 已覆盖所需对象与动作
- **THEN** 方案必须以官方 API 作为主接入路径，并将数据库直连排除为默认实现

#### Scenario: 企业微信机器人受控写回需要系统级 API 凭据

- **WHEN** 企业微信机器人在既有受控确认与审计链路内执行跟进写回，且发送人本身没有可用的 Web 登录会话 token
- **THEN** 系统可以通过受控配置的内置 CRM 账号换取 API token 继续调用官方写回接口
- **THEN** 系统不得因为缺少发送人会话 token 就改为数据库直写、跳过审计，或新增绕过确认流程的写入路径

#### Scenario: 存量非认证能力发生改造

- **WHEN** 团队对既有的非认证功能进行需求扩展、重构或缺陷修复
- **THEN** 该变更必须先复核是否已有可用官方 API，并在可用时优先收敛到官方 API 路径
