## Context

当前后端已经提供商机名称查询和跟进写回，但没有“新增商机”的统一封装入口。业务截图显示，新增商机不仅需要标准字段 `title`、`customer_id`、`expect_amount`、`expect_sign_date`，还要求 `线索编号`、`被续签合同号`、`代理商全称`、`项目现状及关键点`、`售前` 等业务字段，这些字段并不直接出现在 CRM 官方创建商机文档中，说明它们属于目标环境中的自定义字段或扩展录入规则。

因此，设计的关键不是简单透传 `/api/v2/opportunities`，而是要在“截图业务字段”与“CRM 标准字段 / 产品关联字段 / 自定义字段键名”之间建立一层稳定映射，同时把接口控制在后端受控边界内。

## Goals / Non-Goals

**Goals:**

- 提供受会话鉴权保护的内部新增商机接口，作为后续机器人调用的稳定后端入口。
- 对截图中的必填字段做后端校验，并把标准字段、产品关联字段和自定义字段分别映射到 CRM 官方创建商机请求。
- 复用当前登录会话中的 CRM access token，统一处理错误映射、日志记录和 mock 响应。
- 通过环境变量配置截图必填业务字段所对应的 CRM 自定义字段键名和默认销售阶段/来源/类型。

**Non-Goals:**

- 本轮不实现机器人对话中的建商机意图识别、补问、确认与审计编排。
- 本轮不实现产品、客户、联系人、售前人员等元数据查询接口增强，只提供创建口子。
- 本轮不实现商机审批流和后续合同转换联动。

## Decisions

### 决策一：新增内部路由 `/api/v1/crm/opportunities`

- 选择：新增 `/api/v1/crm/opportunities` 作为内部建商机入口。
- 原因：与新增客户保持同一命名空间，便于后续机器人按“内部 CRM 写入接口”统一接入，也避免与现有查询控制器的历史路径风格相互污染。
- 备选方案：直接在现有查询控制器上增加 POST `/api/v2/opportunities`。
- 不采用原因：查询控制器当前职责已经偏重“按名称检索”，继续叠加创建逻辑会让职责边界更模糊。

### 决策二：对外暴露业务化字段，对内拆分为标准字段、产品字段、自定义字段

- 选择：接口直接接收 `title`、`customerId`、`expectAmount`、`expectSignDate`、`leadCode`、`renewalContractCode`、`agentFullName`、`projectStatusSummary`、`preSalesName`、`productAssets` 等业务字段；服务层负责分别映射到 CRM 标准字段、`product_assets_attributes` 和自定义字段键名。
- 原因：截图字段是后续机器人补问和业务校验的主语义，应该保留在内部接口边界；CRM 原始键名只应存在于服务层。
- 备选方案：让调用方直接构造 `opportunity[...]` 和 `product_assets_attributes`。
- 不采用原因：会让上游调用方与 CRM 内部字段格式强耦合，不利于后续机器人演进。

### 决策三：把截图必填项提升为服务端硬约束

- 选择：把 `项目名称`、`最终客户`、`线索编号`、`关联产品`、`预计有效收入`、`预计签单日期`、`被续签合同号`、`代理商全称`、`项目现状及关键点`、`售前` 作为新增商机接口的后端必填项。
- 原因：这些字段已被业务页面标成必填，如果后端不校验，就无法保证机器人未来绕过页面时仍满足同一套录入规则。
- 备选方案：仅按 CRM 官方文档要求 `title` 与 `customer_id`。
- 不采用原因：会导致后端口子与当前业务录入约束不一致，后续机器人写入的数据质量不可控。

### 决策四：产品关联字段必须至少提交一条产品

- 选择：将“关联产品”映射为 `product_assets_attributes`，要求至少存在一条 `productId`。
- 原因：截图将“关联产品”标为必填，而 CRM 官方创建商机文档支持在同一请求中写入产品关联，这比先建商机再补产品更贴近现有业务录入路径。
- 备选方案：本轮只建商机主体，不处理关联产品。
- 不采用原因：不符合截图必填约束，也会给后续机器人调用增加二次写入复杂度。

### 决策五：默认负责人、部门与阶段允许由会话或环境兜底

- 选择：`user_id` 默认取当前会话用户，`want_department_id` 默认取首个部门；`stage`、`source`、`kind` 允许使用环境默认值。
- 原因：截图中多个字段带预置值，说明该流程存在“默认值先填充，再由用户修改”的业务习惯。后端接口应保留这一层兜底能力。
- 备选方案：所有字段都必须显式传入。
- 不采用原因：会提高后续机器人补问复杂度，并把原本应由后端兜底的 CRM 配置细节泄漏到调用方。

### 决策六：复用 `CrmOpportunityApiService` 扩展创建能力

- 选择：在 `CrmOpportunityApiService` 内新增 `createOpportunity`，配合独立创建控制器暴露 HTTP 接口。
- 原因：商机领域的 CRM Open API 读写可以继续集中在同一服务中，复用当前配置、日志和 mock 策略。
- 备选方案：新建独立写服务。
- 不采用原因：会把同一领域能力进一步拆散，未来查询、创建、更新难以统一维护。

## Risks / Trade-offs

- [风险] 截图中的自定义字段在不同 CRM 环境里对应不同 key，导致写入失败或字段落错位 → 缓解：用环境变量配置每个业务字段对应的 CRM 自定义字段键名，缺失时直接返回 400。
- [风险] “关联产品”需要的产品 ID 依赖外部元数据来源，本轮如果调用方只知道产品名称将无法直接创建 → 缓解：本轮明确接口要求传产品 ID，产品检索能力放到后续补充。
- [风险] 商机截图中的“售前”字段可能在 CRM 中是用户 ID 而不是纯文本 → 缓解：当前先按文本字段映射设计，如目标环境确认是人员引用字段，再在后续变更中改成显式用户选择映射。
- [风险] 默认阶段/来源/类型取值不正确会导致数据进入错误漏斗 → 缓解：支持调用方显式传值覆盖，并把默认值放入环境配置而不是硬编码。

## Migration Plan

1. 增加环境变量占位符，声明默认阶段/来源/类型和各业务自定义字段映射键名。
2. 实现控制器、服务映射、mock 创建响应和标准错误处理。
3. 更新 OpenAPI 契约与一期规格文档，说明当前仅开放后端接口口子。
4. 补充集成测试，覆盖必填校验、产品必填、默认值补齐和 mock 创建。
5. 上线前在目标环境确认产品 ID 来源及自定义字段键名，再切换真实 CRM Open API 调用。

## Open Questions

- “售前”在目标 CRM 环境中是文本字段、用户选择字段还是协作人字段；如果不是文本，需要后续补充字段类型转换。
- “被续签合同号”是否存在固定格式或需要先校验合同是否存在；本轮暂按文本字段处理。
