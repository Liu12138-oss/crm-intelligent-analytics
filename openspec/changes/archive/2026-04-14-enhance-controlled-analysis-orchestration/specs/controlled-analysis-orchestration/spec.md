## ADDED Requirements

### Requirement: 系统必须支持受控 AI 直查执行模式

系统 MUST 在受控分析编排中支持“结构化计划执行”和“受控 AI 直查执行”两类只读执行模式。用户的分析文本 MUST 先经过统一 AI 理解层，再由服务端决定进入哪一种执行模式；前端或外部调用方 MAY 提供模式偏好，但系统不得仅依赖客户端直接传入的模式标签绕过服务端理解和校验。受控 AI 直查执行中，模型 MAY 直接生成可执行 SQL、直接选择已注册的只读 API / 查询工具，或直接请求受控数据库读取；但系统 MUST 在执行前完成语句类型校验、权限范围注入、表字段白名单、结果规模限制、超时限制和审计快照记录。系统 MUST 阻止 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`、schema 探测、未声明写入工具和未声明外部请求。

#### Scenario: AI 直接生成只读 SQL 并通过安全栈执行

- **WHEN** 用户提交一个允许通过只读 SQL 回答的经营分析问题
- **THEN** 系统可以接受模型生成的可执行只读 SQL
- **THEN** 系统必须在执行前完成语句类型校验、权限范围注入、白名单校验和行数 / 超时限制
- **THEN** 系统只可将受控结果交给后续分析与交付链路

#### Scenario: AI 尝试执行危险语句时被阻断

- **WHEN** 模型生成包含删除、更新、建表、改表或其它未声明危险动作的 SQL 或工具调用
- **THEN** 系统必须立即阻断执行
- **THEN** 系统必须记录阻断原因与对应执行快照

#### Scenario: 执行模式由服务端根据统一 AI 理解结果决定

- **WHEN** Web 或企业微信用户提交一条新的分析文本输入
- **THEN** 系统必须先基于统一 AI 理解结果决定进入计划执行、受控 AI 直查、解释复用或阻断路径
- **THEN** 系统不得仅根据客户端传入的 `executionMode` 字段直接进入某种执行模式

### Requirement: 系统必须通过统一受控工具边界执行读取

系统 MUST 将分析编排中的知识推理与实际读取操作解耦。模型可以直接生成可执行查询，但真正执行 MUST 仍经过已注册的受控工具边界，由系统统一完成权限注入、白名单校验、API-first 路由、结果规模限制和审计。系统 MUST 阻止任何绕过该工具边界的自由数据库连接、自由 schema 枚举、未声明写入或未声明外部调用。

#### Scenario: AI 直查也必须命中过受控工具边界

- **WHEN** 模型决定直接读取 CRM 数据而不是先产出结构化计划
- **THEN** 系统必须将该读取请求映射到已注册的受控工具
- **THEN** 系统不得让模型绕过统一执行边界直接访问数据库连接

#### Scenario: 非法直接执行路径被阻断

- **WHEN** 任一编排步骤试图直接执行未经过受控工具边界的数据库或外部调用
- **THEN** 系统必须立即阻断该步骤并记录安全审计事件

### Requirement: 系统必须对每个读取步骤执行 API-first 与只读兜底路由

系统 MUST 对每个读取步骤先评估 CRM 官方 API 适配器，再决定是否进入受控 AI 直查 SQL 或受控内部只读接口。只要官方 API 已满足对象、字段、权限语义、聚合形态和稳定性要求，系统 MUST 优先选择官方 API；只有在不满足时，系统才 MAY 进入只读 SQL / 内部接口兜底，并记录 `preferredSource`、`matchedAdapter`、`gapReason` 和当前权限快照。

#### Scenario: 读取步骤优先命中官方 API

- **WHEN** 某个读取步骤能够由 CRM 官方 API 满足
- **THEN** 系统必须优先选择官方 API 适配器
- **THEN** 系统不得因为 SQL 更方便就默认绕过该 API

#### Scenario: 读取步骤进入受控 SQL 兜底

- **WHEN** 官方 API 无法满足当前分析步骤的对象、字段、聚合或稳定性要求
- **THEN** 系统可以进入受控只读 SQL 或内部只读接口兜底
- **THEN** 系统必须同时记录 API 缺口原因、命中适配器和权限快照

### Requirement: 系统必须在只读边界内执行最小纠错闭环

当只读执行步骤失败时，系统 MUST 先对错误进行分类，并仅对可判定、可收敛且不会扩大权限范围或读取范围的错误执行最小修正。自动纠错总尝试次数 MUST 不超过 3 次（含首次执行），每次修正都 MUST 保留前一版执行步骤、错误分类、修正说明和重试结果；对未知、高风险、越权或危险动作错误，系统 MUST 直接阻断并结束自动纠错。

#### Scenario: 可修复的只读错误触发有限次纠错

- **WHEN** 执行步骤因字段别名、聚合映射或已知适配差异而失败
- **THEN** 系统必须在不扩大权限和读取范围的前提下执行最小修正并有限次重试

#### Scenario: 高风险错误禁止自动重试

- **WHEN** 执行步骤命中越权访问、敏感字段探测、危险语句、注入风险或未知错误类型
- **THEN** 系统必须停止自动纠错并向用户返回明确阻断原因

### Requirement: 系统必须根据追问类型决定是否重新取数

系统 MUST 区分解释型追问和改条件追问。该分流 MUST 优先由统一 AI 理解层判定，而不是继续由多个局部规则和客户端标签共同决定。解释型追问 MUST 基于最近一次已验证结果包和执行快照生成说明，不得重新发起新的数据访问；改条件追问、缩小范围追问或新增维度追问 MUST 在当前权限下生成新的执行请求，并产出新的结果包与历史记录。

#### Scenario: 解释型追问复用已验证结果

- **WHEN** 用户在成功结果后继续追问“这说明什么”或“为什么会这样”
- **THEN** 系统必须基于最近一次已验证结果包生成解释
- **THEN** 系统不得为了回答解释型追问重新取数

#### Scenario: 改条件追问重新执行读取

- **WHEN** 用户在成功结果后追问“近三个月也看看”或“按区域重新比较”
- **THEN** 系统必须在当前权限下重新执行读取并返回新的分析结果
