## ADDED Requirements

### Requirement: 跟进主题入口必须消费 capability pack 的日报/跟进路由结果
企业微信 `跟进商机`、`跟进客户`、`今日跟进` 等主题入口 MUST 消费统一 AI capability pack 输出的 `DAILY_REPORT + dailyReportPrompt` 路由结果进入既有跟进/日报链路。只要 capability pack 已经完成归一化并通过条件校验，系统 MUST 继续复用现有跟进模板、自由文本整理、对象识别和写回确认流程，不得因为能力包未返回无关字段而落入帮助兜底。

#### Scenario: 跟进商机进入跟进模板入口
- **WHEN** 企业微信空闲会话收到 `跟进商机`，且 capability pack 返回 `DAILY_REPORT` 与 `FOLLOW_UP_TEMPLATE_ENTRY`
- **THEN** 系统必须进入既有跟进模板或自由文本整理入口
- **THEN** 系统不得返回帮助能力清单

#### Scenario: 今日跟进继续复用现有写回确认链路
- **WHEN** 企业微信空闲会话收到 `今日跟进` 并被 capability pack 识别为日报/跟进入口
- **THEN** 系统必须继续复用既有结构化草稿、对象识别、确认和写回边界
- **THEN** 引入 capability pack 不得绕过现有受控写回确认流程

### Requirement: 跟进四段草稿抽取与缺项提示必须优先通过 AI 提示词优化贴合上下文
企业微信跟进主题中的四段草稿抽取、可选缺项提示和继续/跳过补充语义 MUST 优先通过 capability pack 的提示词、few-shot、负例和结构化 contract 优化实现。对于用户原文中已经明确表达的问题、阻塞项、信息共享或后续计划，系统 MUST 优先让模型理解并抽到对应字段；程序层仅可在模型缺漏时做最小补齐或安全兜底，不得长期依赖额外规则分支承担主抽取职责。

#### Scenario: 问题与计划语义应优先抽到对应字段
- **WHEN** 用户自由文本中出现“客户不好沟通”“推进缓慢”“明天继续跟进”这类明显问题和后续计划表达
- **THEN** 系统必须优先通过 AI 草稿抽取把这些内容识别到 `helpNeeded` 或 `visitPlan`
- **THEN** 不得只把整段原文全部压进 `followUpContent`

#### Scenario: 缺项提示只围绕真实缺失字段
- **WHEN** 系统已经识别出 `跟进内容`、`遇到与协助`、`拜访计划`，只剩 `信息共享` 缺失
- **THEN** 缺项提示必须只围绕 `信息共享` 提供补充建议
- **THEN** 系统不得继续提示补已识别出的 `遇到与协助` 或 `拜访计划`

#### Scenario: 不补充应按当前草稿继续
- **WHEN** 系统已经明确告知用户“如果现在不想补充，也可以直接提交”，用户随后回复 `不补充` 或语义等价表达
- **THEN** 系统必须按当前草稿继续进入后续确认流程
- **THEN** 系统不得把该回复当成取消或结束任务
