## Context

当前 `AppShell` 已经提供全局品牌栏和左侧导航，但分析、合同审核、治理和审计页面仍在内容区首屏额外使用 `PageHeaderCompact` 渲染页面标题、状态标签和快捷操作，形成重复层级。实际业务信息又都在标题栏下方的首个面板中，导致首屏先看到一层“过渡页头”，再进入真正的输入、结果、审核或治理区。

这次变更只处理 `AppShell` 下业务页面的内容区标题栏收口，不调整登录页，不移除全局顶部品牌栏，不改左侧导航，不改任何业务逻辑、接口或权限判断。

## Goals / Non-Goals

**Goals:**

- 移除 `AppShell` 内容页的冗余 `PageHeaderCompact` 标题栏。
- 不在内容区保留页头文案、状态标签、查询 ID 等页头元素的迁移版。
- 仅保留确有业务必要的操作按钮，并让它们回到对应业务模块的自然操作位置。
- 保持页面首屏更直接，进入页面后优先看到可执行业务区，而不是中间过渡标题层。

**Non-Goals:**

- 不修改 `AppShell` 顶部品牌栏和左侧导航。
- 不调整登录页结构。
- 不改变按钮行为、接口请求、事件函数、路由、权限校验或状态管理。
- 不借这次变更重做页面信息架构或新增页面模块。

## Decisions

### 决策 1：彻底删除页头语义，不在内容区保留迁移版标题块

后续实现中，每个业务页面进入主内容区后，不再通过首个模块继续承接原页头的页面标题、说明文案、状态胶囊或上下文元信息。首屏只保留模块本身原有的业务标题和业务内容。

原因：

- `AppShell` 已经提供全局层级，再把页头语义搬进内容区，本质上还是重复层级。
- 用户已经明确不接受“页头迁移版”继续存在于内容页。

备选方案：

- 保留一条更瘦的页头横条。问题是仍然保留重复层级，只是高度更小。
- 把原页头整体搬到首个业务面板。问题是形式变化但语义未删，仍然是同一个页头。

### 决策 2：页头上下文元素直接删除，只保留必要业务操作

原来位于 `PageHeaderCompact` 的页面标题、状态标签、说明文案、查询 ID 与同类页头上下文元素，后续直接删除，不再迁移。只有确有业务必要的按钮才允许保留，并附着到对应业务模块本身。

原因：

- 这次变更的目标是彻底去掉页头，而不是把页头换个位置继续显示。
- 区分“业务必要操作”和“页头上下文元素”后，既能满足用户要求，也能保住必要功能。

备选方案：

- 保留全部页头元素并重新排版。问题是违背最新要求。
- 为了删页头而重构脚本层。问题是超出本次范围。

### 决策 3：必要操作回归业务模块自然位置，而不是组成新的页头区

不同页面采用统一原则但不同落点：

- 工作台页：只保留模块本身原有标题与主按钮，不额外补页头说明或页头状态标签。
- 结果详情页：保留结果模块与产物操作，但移除查询 ID、返回入口、页面状态标签等页头语义元素。
- 治理与审计页：保留保存、创建、刷新等必要按钮，但不再显示页面级标题、副文案和摘要胶囊。

原因：

- 各页面仍有少量必要操作，完全删掉会损伤可用性。
- 把按钮放回业务模块自然位置，可以避免再次形成“伪页头”。

备选方案：

- 所有页面都统一放在第一个 `panel__header` 右上角并搭配说明文案。问题是仍会形成新的页头观感。

### 决策 4：页面识别依赖业务模块本身，不再补页头说明

页面识别主要依赖首个业务模块原本就存在的区块标题、结果摘要、风险结论和操作区，不再额外补页面说明文案。

备选方案：

- 继续补一条简短说明。问题是仍然构成页头替身。

## Risks / Trade-offs

- [删除页头后部分按钮失去承载位置] → 只保留真正必要的按钮，并分别放回上传区、治理表单区或结果操作区。
- [历史测试仍把迁移后的页头内容当成正确结果] → 先改测试，明确不允许标题、副文案、状态胶囊和上下文元信息迁入内容区。
- [`PageHeaderCompact` 样式清理后影响未迁移页面] → 先完成全部消费者迁移，再决定是否删除组件与样式。

## Migration Plan

1. 盘点所有 `PageHeaderCompact` 消费页面，整理每个页面的标题、按钮、标签和提示信息的迁移目标。
2. 先更新测试与提案口径，明确标题、副文案、状态胶囊和查询 ID 等页头元素不得迁入内容区。
3. 按“分析工作台与详情 -> 合同审核工作台与详情 -> 治理与审计”顺序逐页删掉迁移版页头内容，并只保留必要按钮。
4. 每迁移一页都核对保留下来的按钮行为不变，同时确认页头语义元素已被彻底删除。
5. 全部页面完成后再清理 `PageHeaderCompact` 的无用引用与冗余样式。
