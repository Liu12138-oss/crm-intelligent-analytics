## Why

当前销售团队日报与部分主管关系依赖根目录历史静态权限表，该表是静态快照，企业微信组织架构调整后无法第一时间同步，且不能表达“多个部门管理人员最终归同一直属上级管理”的层级关系。后续企业微信机器人与 Web 端问数都必须先按当前组织架构收敛数据范围，再叠加明确白名单授权，否则会出现团队范围过宽、主管错配或组织调整后继续使用旧授权的问题。

## What Changes

- **BREAKING**：销售团队日报负责人映射不再以历史静态负责人表作为权威来源，系统改为基于企业微信官方 API 同步的组织事实、直属上级链和 CRM 身份映射推导成员归属与负责人范围。
- 新增组织事实与数据范围治理能力：沉淀企业微信部门树、部门负责人、成员主部门、直属上级链、CRM 用户映射、默认可见范围和数据白名单授权。
- 问数范围从“用户快照中的组织 / 部门 / ownerIds 原样注入”升级为“组织事实默认范围 + 数据白名单叠加 + 审计快照”统一计算，Web 与企业微信机器人共享同一范围结果。
- 普通成员默认只能查自己；团队管理默认只能查直属或递归下属团队；大区负责人按直属上级链聚合下级管理人员及其团队，例如严刚、官俊东直属上级均为王冬时，大北区范围归王冬管理。
- 新增独立于企业微信机器人灰度策略的数据范围白名单，支持按用户或角色授权可查某些部门，并记录是否包含子部门、授权原因、有效期、更新人和审计事件。
- 小组日报预览、次日团队汇总、协助事项主管解析等主管/团队场景统一复用组织范围解析，不再各自读取静态名单或用临时字段兜底。

## Capabilities

### New Capabilities

- `organization-scope-governance`: 定义企业微信组织事实同步、默认数据范围推导、数据白名单授权、范围审计和诊断能力。

### Modified Capabilities

- `controlled-analysis-orchestration`: 受控分析执行必须消费统一组织范围快照，并在 Web 与企业微信问数中执行一致的数据范围注入。
- `wecom-daily-report-reminder-summary`: 次日团队汇总必须从企业微信组织事实和直属上级链推导负责人团队，停止依赖历史静态负责人表。
- `wecom-on-demand-team-daily-report-delivery`: 小组日报即时预览必须按统一组织范围校验团队管理权限和白名单授权。

## Impact

- 后端影响 `backend/src/modules/wecom` 的企业微信目录同步、身份映射和组织镜像模型，需要补齐部门负责人、成员主部门、直属上级链和组织事实诊断。
- 后端影响 `backend/src/modules/auth/user-scope.service.ts`、`backend/src/modules/analysis/query-scope.service.ts`、`backend/src/modules/analysis/query-compiler.service.ts` 与分析审计链路，需要从静态 `CrmUser` 快照切换为统一组织范围解析结果。
- 后端影响 `backend/src/modules/daily-report/sales-leader-mapping.service.ts` 与日报汇总/预览链路，需要废弃静态 Excel 权威来源并迁移为组织范围服务。
- 治理后台需要在权限中心增加数据范围白名单配置、授权诊断、过期授权提示和审计记录。
- 规格与文档需同步移除历史静态负责人表作为销售负责人映射权威来源的描述，并补充企业微信目录同步失败时的降级与阻断口径。
