## Context

当前系统中至少存在两套与“谁能看谁的数据”相关的来源：一套是 CRM 登录态与 `CrmUser` 快照里的组织、部门、角色和 `ownerIds`；另一套是日报模块私有的历史静态负责人表。前者没有实时表达企业微信组织架构的直属上级链，后者是静态表格，无法在企业微信组织调整后即时生效，也无法稳定表达“大区负责人通过下级管理人员的直属上级链管理整个区域”的关系。

企业微信官方目录同步已经具备部门树、成员列表、成员详情、手机号和邮箱映射能力，但当前主要用于身份映射和目录诊断，没有产出统一的数据范围快照。后续 Web 和企业微信机器人问数都要求先校验组织架构：普通成员只能查自己，团队管理只能查自己团队，少数用户或角色通过数据范围白名单扩展到指定部门。

## Goals / Non-Goals

**Goals:**

- 建立统一组织事实层，基于企业微信 API 同步部门树、部门负责人、成员主部门、直属上级链，并映射到 CRM 用户。
- 建立统一组织范围解析服务，为 Web 问数、企业微信问数、团队日报预览和次日团队汇总提供同一份默认数据范围。
- 去除日报销售负责人映射对历史静态负责人表的权威依赖，改为从组织事实和直属上级链推导负责人团队。
- 新增独立的数据范围白名单，允许按用户或角色授予指定部门数据范围，支持是否包含子部门、有效期、授权原因和审计。
- 保持企业微信机器人灰度策略只控制入口，不承担数据范围授权。

**Non-Goals:**

- 不重做 CRM 登录、会话建立、企业微信扫码登录或统一鉴权入口。
- 不把企业微信部门树改造成可编辑的 CRM 权限树；企业微信组织事实只作为当前组织关系和范围推导来源。
- 不开放自由 SQL 或绕过现有字段白名单、AST 校验、API-first 路由与审计。
- 不把数据白名单与企业微信机器人灰度白名单合并。

## Decisions

### 决策 1：采用“组织事实层 -> 默认数据范围层 -> 数据白名单层”的三层模型

最终数据范围解析固定为：

```text
企业微信目录同步
-> 组织事实镜像（部门树、部门负责人、成员、主部门、直属上级链、CRM 映射）
-> 默认数据范围解析（本人 / 直属团队 / 递归下属团队）
-> 数据范围白名单叠加（用户 / 角色 -> 部门，是否含子部门）
-> 统一 ScopeSnapshot / 审计快照 / SQL 范围注入
```

这样可以把“当前组织关系是什么”和“额外授权了什么”拆开，避免继续把灰度准入、角色动作、日报负责人映射和数据范围混在一起。

备选方案是继续扩展 `CrmUser.departmentIds / ownerIds`。问题是这些字段来自登录快照，不能表达部门树、下属链、白名单有效期和授权原因，也不适合作为跨入口统一诊断来源。

### 决策 2：部门负责人不是最终团队归属的唯一依据，必须结合直属上级链收敛

企业微信部门可能挂多个管理人员，但业务上的最终管理者需要根据人员直属上级链收敛。例如大北区下严刚和官俊东都挂在部门中，但两人的直属上级都是王冬，则大北区最终管理范围应归王冬。系统必须支持从成员往上追溯直属上级，并按目标管理者聚合递归下属成员。

备选方案是直接使用 `department_leader` 作为部门最终负责人。该方案会把多个挂名管理人员误判为并列最终负责人，导致团队日报错发和问数范围过宽。

### 决策 3：数据白名单独立建模，不复用企业微信灰度策略

企业微信灰度策略只决定“能否进入机器人入口”，数据白名单决定“能额外看哪些部门数据”。两者必须分离。数据白名单建议字段包含：

```ts
interface DataScopeGrantRecord {
  id: string;
  subjectType: 'USER' | 'ROLE';
  subjectId: string;
  departmentIds: string[];
  includeSubDepartments: boolean;
  reason: string;
  expiresAt?: string;
  status: 'ACTIVE' | 'INACTIVE' | 'EXPIRED';
  updatedBy: string;
  updatedAt: string;
}
```

权限中心应提供白名单治理入口和预览能力，预览结果必须展示默认范围、白名单扩展范围、是否过期和命中原因。

### 决策 4：分析查询只消费统一范围快照，不直接读取静态表或灰度白名单

`AnalysisService` 获取用户后，必须调用统一组织范围解析服务生成 `scopeSnapshot`。`QueryScopeService` 只负责把该快照注入结构化意图，`QueryCompilerService` 只负责编译范围过滤。企业微信机器人和 Web 端使用相同服务，避免入口之间范围不一致。

当组织事实不可用、企业微信目录同步失败、CRM 映射缺失或范围解析结果为空时，系统必须拒绝团队范围查询并给出明确原因；普通本人查询可以在能唯一确认本人 CRM 身份时退化为本人范围。

### 决策 5：日报团队汇总改为使用组织范围服务，静态 Excel 仅可作为迁移期诊断对照

`SalesLeaderMappingService` 不再读取历史静态负责人表作为生产权威来源。实现期可保留只读诊断或测试 fixture，用于对照新旧团队范围差异，但正式发送、正式预览和主管解析必须使用组织范围服务。

迁移完成后，规格与文档必须移除“按历史静态负责人表发送”的验收描述，避免后续继续维护静态表格。

## Risks / Trade-offs

- [企业微信 API 不返回或未授权直属上级字段] -> 同步阶段必须记录字段缺口；范围解析不得默默扩大权限，可降级为本人范围或阻断团队范围能力。
- [企业微信部门负责人和 CRM 用户映射不完整] -> 诊断页必须显示未映射、重名、手机号/邮箱冲突和最近同步时间，不能把映射失败伪装成无数据。
- [白名单配置错误导致范围过宽] -> 白名单必须有授权原因、更新人、有效期和审计；预览必须展示默认范围与扩展范围差异。
- [组织树规模较大导致递归解析成本升高] -> 组织事实服务应缓存同步后的部门树和上下级索引，并在同步成功后失效刷新。
- [移除 Excel 后短期结果与旧汇总不一致] -> 首次上线应输出新旧范围对照报告，发现差异时以企业微信当前组织事实为准，并由治理人员修正组织映射或白名单。

## Migration Plan

1. 扩展企业微信目录同步模型，补齐部门负责人、成员直属上级、成员部门归属和 CRM 映射诊断字段。
2. 新增组织范围解析服务和数据白名单存储，先提供只读预览与诊断，不立即替换正式日报发送。
3. 为 Web 与企业微信分析链路接入统一 `scopeSnapshot`，补齐权限范围审计和阻断提示。
4. 将团队日报预览和次日团队汇总切换到组织范围服务，并保留新旧 Excel 对照日志一个迁移周期。
5. 权限中心上线数据范围白名单治理后，停止读取历史静态负责人表，同步更新规格、README 和运维文档。

回滚策略：若组织范围解析异常，可临时关闭团队范围查询和团队日报真实发送；本人范围问数继续使用已映射 CRM 用户本人范围。不得回滚到扩大权限的静态 Excel 生产路径。

## Open Questions

- 企业微信当前授权应用是否能稳定返回成员直属上级字段；若不能，需要确认字段来源是企业微信官方详情、扩展属性还是 CRM 现有主管字段。
- 部门白名单是否默认包含子部门；当前建议配置项显式决定，默认不包含子部门，避免意外扩权。
- 大区负责人是否需要跨多个根部门管理；当前设计默认限定在“联软科技集团”同步根范围内。
