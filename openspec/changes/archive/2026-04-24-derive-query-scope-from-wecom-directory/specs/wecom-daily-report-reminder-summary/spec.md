## MODIFIED Requirements

### Requirement: 次日团队汇总必须按负责人聚合个人摘要与缺报名单

系统 MUST 在次日 `08:00` 按当前组织范围解析结果聚合前一日成员的个人日报摘要，并单独列出“当天没有任何日报源数据”的成员名单。销售团队 MUST 基于企业微信官方目录同步得到的部门树、成员主部门、直属上级链和 CRM 身份映射推导团队负责人及其成员范围；系统 MUST 停止将历史静态负责人表作为生产负责人映射权威来源。售前团队若组织范围来源仍未补齐，系统 MUST 阻断售前真实发送并记录原因。

#### Scenario: 销售团队汇总按当前组织范围聚合发送

- **WHEN** 系统在次日 `08:00` 生成销售团队汇总并解析接收人
- **THEN** 系统必须按企业微信当前组织事实和直属上级链聚合成员日报并发送给对应团队负责人
- **THEN** 系统不得继续依赖历史静态负责人表或把所有团队汇总统一发送给单一管理员

#### Scenario: 大区负责人聚合下级管理人员团队

- **WHEN** 严刚和官俊东均属于大北区管理节点，且两人的直属上级都是王冬
- **THEN** 系统必须将严刚、官俊东及其递归下属成员纳入王冬的大北区团队汇总范围
- **THEN** 系统不得把严刚和官俊东分别作为大北区最终接收人重复发送同一大区汇总

#### Scenario: 团队汇总必须附带无源数据名单

- **WHEN** 某些成员在前一日没有任何日报源数据
- **THEN** 系统必须在团队汇总中单独列出这些成员名单
- **THEN** 系统不得把这些成员与“已有日报摘要”的成员混在同一列表中

#### Scenario: 组织范围缺失时阻断团队真实发送

- **WHEN** 系统需要执行真实团队汇总发送，但当前企业微信组织事实、直属上级链或 CRM 映射不足以唯一确定团队负责人和成员范围
- **THEN** 系统必须阻断该团队真实发送
- **THEN** 系统必须记录“组织范围来源未补齐”及具体缺口，等待同步或映射修复后再开启

#### Scenario: 售前映射未补齐时阻断售前真实发送

- **WHEN** 系统需要为售前团队执行真实汇总发送，但当前缺少可靠的售前负责人映射来源
- **THEN** 系统必须阻断该次售前真实发送
- **THEN** 系统必须记录“售前负责人映射未补齐”的原因，等待映射补齐后再开启
