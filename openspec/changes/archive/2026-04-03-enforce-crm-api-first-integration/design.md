## Context

当前仓库已经通过一期规格明确了两类边界：

- 分析问数链路继续保持受控只读。
- 企业微信目录同步与后续写入能力按各自规格定义受控写入范围。
- CRM 官方 API 的统一参考文档地址为 `https://apidoc.weiwenjia.com/`。

这解决了“能不能写 CRM”这一层问题，但没有解决“应该优先从哪里接 CRM”这一层问题。现状上，不同能力已经出现多种接入方式并存：

- 登录链路已稳定复用 CRM Open API。
- 分析链路主要依赖受控数据库查询。
- 企业微信目录同步等能力存在官方 API 与数据库写回边界的组合路径。

随着后续功能扩展，如果没有统一的“官方 API 优先”原则，团队很容易因为数据库熟悉、开发更快或历史遗留原因，继续扩大自建接口和数据库直连范围。对于 CRM 这种表结构复杂、权限口径强依赖既有系统的场景，这会直接增加 AI 场景下的准确性和可维护性风险。

本次设计是一个跨模块、跨文档的治理型变更，目标是统一未来变更的接入顺序，同时保持当前稳定的认证链路不动。

## Goals / Non-Goals

**Goals：**

- 建立明确的 CRM 接入优先级：官方 API 优先，缺失时才允许进入受控兜底路径。
- 为官方 API 不满足时的自建接口或数据库路径定义统一准入条件，避免“先做再补说明”。
- 给存量非认证能力提供渐进式收敛策略，避免一次性全量重构。
- 明确本轮约束不触碰用户认证、登录和会话链路，降低交叉变更风险。
- 把上述规则同步到仓库主文档、一期规格和 OpenSpec 变更中，形成后续开发可执行的共识。

**Non-Goals：**

- 不在本次变更中直接重写现有模块代码。
- 不要求一次性把所有历史数据库访问改造成官方 API。
- 不调整用户认证、登录、扫码登录、Cookie 会话或统一鉴权守卫。
- 不新增新的产品接口字段或修改现有 OpenAPI 契约。

## Decisions

### 决策一：统一采用“官方 API -> 受控自建接口 -> 受控数据库路径”的接入顺序

后续所有新增 CRM 业务能力，以及对既有非认证能力的改造，先评估 CRM 官方 API 是否覆盖所需对象、动作、字段、权限语义和稳定性；满足则直接采用官方 API。只有官方 API 不存在或无法满足经确认的业务约束时，才允许进入自建接口或数据库兜底路径。

这样设计的原因是：

- 官方 API 天然更贴近 CRM 官方对象语义与演进方向，误读底层表结构的概率更低。
- 能减少团队对复杂 CRM 表关系的重复解读，降低 AI 生成或后续协作者误用数据库字段的风险。
- 后续如果 CRM 官方能力升级，收敛成本也低于散落在各模块里的数据库定制逻辑。

备选方案：

- 继续按模块自行选择接入方式：灵活，但口径不可控，不采用。
- 继续以数据库优先：开发初期快，但长期准确性与维护成本高，不采用。

### 决策二：官方 API 缺失时，必须先补齐“兜底准入说明”再开发

当需求无法由官方 API 覆盖时，不允许直接跳到“那就查库/写库”。必须先记录至少以下内容：

- 哪个对象或动作缺少官方 API；
- 为何现有官方 API 无法满足业务所需字段、权限语义、时效或稳定性；
- 选择的是自建接口还是数据库路径；
- 涉及的表、字段、白名单边界与权限继承来源；
- 审计留痕、失败降级和回退方案。

这样做是为了把“数据库兜底”从默认路径降级为显式例外，避免后续文档与实现脱节。

备选方案：

- 仅由评审口头确认是否能查库：信息容易丢失，不采用。
- 只要求写原因，不要求写边界与回退：无法形成真正可执行的约束，不采用。

### 决策三：存量能力采用“先盘点、后按触点收敛”的渐进策略

对现有非认证能力，不做一次性整体迁移。先建立接入方式台账，标记每项能力当前属于“官方 API / 自建接口 / 数据库路径”中的哪一种；后续一旦该能力进入功能新增、重构或修复，就先重新评估官方 API 是否已经可用，并优先在这次改动范围内收敛。

这样设计的原因是：

- 当前仓库已存在较多实现与规格，立刻全量迁移成本高、风险大。
- “按触点收敛”更符合实际研发节奏，也更容易和业务排期结合。

备选方案：

- 一次性全面改造存量模块：风险过高，不采用。
- 对历史能力不做要求，只约束新功能：会让遗留路径持续扩张，不采用。

### 决策四：认证链路冻结，避免把稳定能力卷入本轮治理变更

本次变更明确排除 `auth` 模块相关能力，包括账号密码登录、扫码登录、本地会话、Cookie、统一鉴权守卫和会话装载。即使在盘点时发现认证链路也能进一步“统一到 API-first 口径”，也不得并入本次工作，而是要独立走新的规格变更。

这样设计的原因是：

- 用户已明确说明认证链路稳定，当前优先级不在这里。
- 认证能力牵涉入口稳定性，和本次“非认证能力接入策略治理”耦合度低。

备选方案：

- 顺带整合认证链路：范围失控，不采用。

## Risks / Trade-offs

- [官方 API 覆盖不足导致部分能力短期无法完全收敛] → 通过兜底准入说明保留受控例外，并持续维护台账。
- [过渡期内同时存在官方 API、自建接口和数据库路径，口径仍会并存] → 用文档、台账和后续变更评审强制收敛，避免新增无说明路径。
- [认证链路冻结可能让部分共享能力无法一起重构] → 明确只冻结认证行为与边界，不阻止其他模块复用现有认证结果。
- [文档先行、代码后跟可能造成执行不一致] → 在任务中加入存量盘点和后续变更准入模板，减少文档成为“口号”的风险。

## Migration Plan

1. 在 OpenSpec 变更中固化 proposal、design、specs 和 tasks，形成正式约束来源。
2. 同步更新 `README.md`、`AGENTS.md`、一期 `spec.md`、`plan.md`、`quickstart.md` 与 `docs/architecture/`，统一仓库口径。
3. 盘点现有非认证模块的 CRM 接入方式，建立首版台账。
4. 从下一批需求开始，新增或改造非认证能力前必须先完成 CRM 官方 API 可用性检查和兜底说明。

回退策略：

- 本次变更以文档和规格为主，没有直接运行时迁移；若后续业务决策否定该原则，只需回退文档与变更约束，不涉及线上数据回滚。
