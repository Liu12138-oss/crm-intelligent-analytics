## Why

CRM 数据库表关系复杂、历史包袱重，如果后续功能继续默认直连数据库或按模块各自决定接入方式，AI 场景下很容易出现字段误用、权限口径漂移和存量能力越改越散的问题。当前仓库虽然已经定义了“分析链路只读、写入链路受控”的边界，但还缺少一条更上游的统一原则：凡是 CRM 业务能力，默认先看官方 API，只有官方 API 不满足时才进入自建接口或数据库兜底路径。

同时，现有用户认证、登录和会话鉴权链路已经稳定，本轮梳理的目标是约束后续功能与存量非认证能力的接入策略，而不是顺带重构认证体系。

## What Changes

- 新增一条仓库级实施约束：所有新增 CRM 业务能力，以及对既有非认证能力的改造，必须优先评估并复用 CRM 官方 API。
- 明确官方 API 不满足时的兜底准入条件：必须先补齐 API 缺口说明，再定义自建接口或数据库路径的白名单、权限继承、审计留痕、失败降级与回退方案。
- 要求存量非认证能力建立接入方式台账，后续按“改到哪里、收敛到哪里”的原则逐步对齐到同一规则，而不是继续扩大无约束数据库直连范围。
- 明确本次约束梳理不改动用户认证、登录、会话建立和统一鉴权装载链路；若后续需要调整认证能力，必须独立立项。
- 同步更新仓库主文档和一期规格文档，使 README、协作规范、一期规格、计划、快速开始与架构说明保持同一口径。

## Capabilities

### New Capabilities

- `crm-api-first-integration`: 定义 CRM 官方 API 优先、数据库兜底受控、存量非认证能力逐步收敛以及认证链路冻结的统一约束。

### Modified Capabilities

- 无。

## Impact

- 影响仓库级协作规范、一期主规格与架构文档。
- 影响后续所有非认证 CRM 相关模块的设计评审与实现顺序，重点包括 `analysis`、`wecom`、`query-assets`、`governance`、`audit`、`export` 等模块。
- 影响存量功能的后续改造方式：需要先盘点当前接入路径，再决定哪些能力优先收敛到 CRM 官方 API。
- 不影响当前稳定的 `auth` 模块行为、登录入口和会话机制。
