## Context

当前仓库里与日报相关的企业微信能力主要分成两类：一类是“查看我今天的日报预览”，面向日报提报人本人；另一类是 `22:00` 个人确认 / 催报和次日 `08:00` 团队汇总，面向固定调度时点的主动通知。`DailyReportService` 已经能基于销售负责人映射生成 `【某某小组日报分析】` 正文，但这一能力只被 `runSummarySweep()` 调用，企业微信聊天入口并没有与之对应的显式识别与权限校验链路。

另一个约束来自现有的“空闲未识别输入帮助兜底”变更。该变更故意把 `王文定小组日报` 这类短句定义为低置信输入，要求系统返回帮助提示，避免误把它当作日报采集或实体直查入口。因此，本次设计不能简单放宽到“只要包含小组日报就直接进入新能力”，而必须给出更严格的显式入口口径，确保新能力不会吞掉既有帮助保护。

## Goals / Non-Goals

**Goals：**

- 让企业微信机器人支持“显式索取指定负责人 / 小组今日日报预览”的新入口，满足管理者移动场景下的临时查看诉求。
- 复用现有销售负责人映射、日报源数据汇总和团队正文拼装逻辑，避免再造一套日报聚合或组员归属规则。
- 为该入口增加实时权限校验，确保查看范围继续继承当前 CRM 的负责人层级关系，而不是引入新的独立权限模型。
- 保持低置信短句兜底不变，避免 `王文定小组日报` 这类输入因为新能力而重新误触发。

**Non-Goals：**

- 本次不扩展“昨天 / 本周 / 指定日期区间”的团队日报查询，只覆盖“今天 / 今日”的当天预览。
- 本次不把该能力扩展为新的定时推送、导出能力或 Web 页面新入口，只处理企业微信聊天中的受控查看。
- 本次不重定义 `22:00` / `08:00` 既有日报生成、催报和正式汇总规则，只复用其底层聚合能力。

## Decisions

### 决策一：仅在“显式索取动作 + 小组 / 团队目标 + 日报主题”同时命中时进入新能力；未写时间默认按今天处理

- 原因：现有帮助兜底已经明确把 `王文定小组日报` 视为低置信输入。如果只看到“小组 + 日报”就进入新能力，会把既有保护直接打穿，重新引入误触发问题。
- 具体口径：消息中至少同时包含以下信号：
  - 日报主题：`日报`
  - 团队目标：`小组`、`团队`
  - 主动索取动作：`发给我`、`给我看`、`查看`、`看看`、`输出`、`发我`
- 若消息显式包含 `今天`、`今日`、`本日`、`当天`，按当天处理；若未写时间，也默认按当天处理；若显式包含 `昨天`、`本周`、`上月` 等当前未支持时间，则不得误判为当天预览。
- 备选方案：把所有 `<人名>小组日报` 都视为支持入口。放弃原因是低置信度太高，会与现有帮助兜底和日报采集保护直接冲突。

### 决策二：小组目标解析优先复用销售负责人映射，按“负责人名 -> 负责人用户 -> 组员集合”链路落地

- 原因：团队汇总当前就是基于 `SalesLeaderMappingService` 从历史静态负责人表中解析负责人及其成员，本次应继续沿用同一来源，避免出现“定时汇总一套组员，聊天索取另一套组员”的双口径问题。
- 具体做法：
  - 显式解析 `<负责人名>小组`、`<负责人名>团队` 里的负责人名称；
  - 名称命中后，使用 `SalesLeaderMappingService.listMappedSalesGroups()` 找到目标负责人和成员集合；
  - 若未命中或命中不唯一，返回中文澄清提示，要求用户补充更完整的人名。
- 备选方案：直接用 `crmReadonlyService.listUsers()` 做全量人名模糊匹配。放弃原因是“负责人”与“普通成员”会混在一起，容易把非组长用户误识别为小组目标。

### 决策三：按当前 CRM 层级关系做最小必要权限校验，不为小组日报新增独立权限模型

- 原因：项目协作规范已经要求权限判断优先复用现有 CRM 用户、角色、组织、部门和负责人归属关系，不能另起一套独立权限模型。本能力是“查看谁的小组日报”，本质上仍应落在当前负责人层级关系里。
- 具体规则：
  - 管理员可以查看任意已映射销售小组；
  - 非管理员仅允许查看两类小组：
    - 当前用户本人负责的小组；
    - 当前用户直属下级负责人负责的小组，即目标负责人 `supervisorId === 当前用户.id`；
  - 若不满足以上条件，返回明确的无权限提示，不得泄露目标小组的汇总正文。
- 备选方案：直接允许所有具备日报能力的用户查询任意小组。放弃原因是这会绕过现有负责人层级约束，形成新的数据暴露面。

### 决策四：新增“按负责人生成小组日报预览”服务出口，复用成员快照与团队正文拼装，但不创建正式汇总批次

- 原因：现有 `runSummarySweep()` 会保存 `DailyReportSummaryBatchRecord` 并触发主动通知；聊天索取只是当前会话里的即时预览，不应伪造一条新的 `08:00` 汇总批次，也不应触发额外推送。
- 具体做法：
  - 在 `DailyReportService` 新增仅面向预览的服务方法；
  - 该方法内部仍复用 `collectSummaryMemberSnapshots()` 和 `buildTeamSummaryMessage()`；
  - 返回值只包含目标负责人、业务日期、摘要正文和空状态信息；
  - 不写入 summary batch，不调用主动通知底座，不影响既有 `08:00` 批次幂等逻辑。
- 备选方案：直接调用 `runSummarySweep()` 并把生成结果截回当前聊天。放弃原因是会产生额外批次、通知和审计副作用，与“即时查看”语义不符。

### 决策五：返回结果统一走聊天预览口径，明确告知“这是今天的小组日报预览”

- 原因：今天的团队日报在白天可能仍处于“待确认 / 未交 / 正在补录”的动态状态，不能让用户误以为这是已经正式送达的固定版汇总。
- 具体做法：
  - 成功时统一使用“这是 `<负责人名>` 小组今天的日报预览”之类的前置话术；
  - 正文继续复用现有 `【某某小组日报分析】` 结构；
  - 若当前组内暂无可汇总内容，也要返回空状态说明，而不是只回一句“没有数据”。
- 备选方案：直接返回团队汇总正文，不加预览说明。放弃原因是用户难以区分它与次日正式汇总的语义差异。

## Risks / Trade-offs

- [负责人姓名在主数据中重名或命名不规范] -> 优先按销售负责人映射解析；解析失败时返回澄清提示，不进入错误小组。
- [白天查询导致部分成员仍未确认或未补录，用户误以为数据不完整] -> 明确标注“今天的日报预览”，并保留已确认 / 待确认 / 未交统计。
- [新入口放得过宽，重新吞掉帮助兜底] -> 强制要求“索取动作 + 今天 / 今日 + 小组 / 团队 + 日报”同时命中，短句继续走帮助。
- [复用团队汇总聚合时引入副作用] -> 只新增预览服务出口，不复用 `runSummarySweep()` 的批次落库与主动发送流程。

## Migration Plan

1. 先补齐企业微信入口识别、负责人解析和权限校验逻辑，并新增相应集成测试。
2. 在 `DailyReportService` 落地按负责人生成当天预览的服务方法，确认不会创建 summary batch 或主动通知记录。
3. 同步更新 README、机器人统一能力目录与主规格验收场景，再打开真实环境联调。
