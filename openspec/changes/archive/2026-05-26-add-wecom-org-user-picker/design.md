## Context

权限中心当前在多个标签页中直接使用 `el-select` 平铺选择人员或部门，包括企业微信准入、超级管理员授权、白名单设置、日报治理、身份映射诊断和用户权限预览。该模式在人员数量增长或部门层级变深后存在三个明显问题：管理员缺少企业微信组织上下文、无法按部门或小组批量选择人员、无法在同一套交互中选择部门本身、搜索选择后关键词残留。部分入口虽然已经通过 `reserve-keyword=false` 缓解搜索残留，但没有形成统一能力。

仓库已有企业微信组织事实与 CRM 映射基础，包括 `wecomSyncedDepartments`、`wecomSyncedUsers`、CRM 原生 `wx_users`、`wx_user_maps`、`wx_departments`、`wx_user_department_maps`，以及权限中心现有 `/governance/access-options` 平铺选项。新能力应复用这些事实来源，不新增独立人员、部门或权限模型。

前端技术栈为 Vue 3、Element Plus 和 `@element-plus/icons-vue`。新增 UI 必须遵循 `DESIGN.md` 和本仓库权限中心现有浅色治理界面，不直接套用默认后台视觉，也不得使用旧 `.pen` 风格作为强约束。

## Goals / Non-Goals

**Goals:**

- 在权限中心沉淀统一的企业微信组织架构人员/部门选择组件，替换所有人员和部门类平铺下拉选择。
- 支持组织树展开、人员搜索、部门搜索、部门本身选择、部门下人员全选、已选对象管理、单选/多选和搜索后关键词自动清空。
- 支持根据业务场景输出 CRM 用户 ID、企业微信 userid、CRM 部门 ID 或企业微信部门 ID，并清晰处理未映射 CRM 的企业微信成员与部门。
- 保持现有保存接口 payload 的业务字段不变，降低后端权限策略和审计逻辑的改造风险。
- 复用现有企业微信组织事实层和 CRM 身份映射，保证选择体验改善不扩大真实权限范围。
- 将该组件写入项目级规约，后续其它页面或新增业务只要涉及选择人员或部门，默认复用该组件。

**Non-Goals:**

- 不重构登录、会话、统一鉴权和企业微信扫码回跳链路。
- 不新增独立人员库、独立部门库、独立角色体系或绕过 CRM 权限权威源的授权模型。
- 不改变企业微信灰度、超级管理员授权、数据范围白名单和日报团队的业务保存语义。
- 不在本变更中重做权限中心整体信息架构或一级导航。

## Decisions

### 1. 新增统一组织架构对象选择组件，而不是继续增强各处 `el-select`

新增共享组件建议命名为 `WecomOrgSubjectPicker.vue`，由权限中心各标签页和后续需要选择人员或部门的业务页面复用。输入框区域只承担“展示已选摘要 + 打开选择器”的职责，复杂交互进入弹窗或抽屉完成。

选择原因：

- 平铺下拉框无法承载组织树、搜索结果、已选列表、未映射提示、部门本身选择和部门下人员全选等复杂交互。
- 统一组件可以一次性修复搜索残留、标签删除、禁用态和空状态，不再在每个 tab 或新增页面内复制逻辑。
- 弹窗或抽屉更适合权限中心的治理场景，管理员可以在大面积区域中确认组织范围和已选对象。

备选方案：

- 继续为每个 `el-select` 增加 `reserve-keyword=false` 与远程搜索。该方案只能修复关键词残留，无法解决组织树、部门选择和部门下人员全选。
- 使用 `ElTreeSelect` 同时展示部门与人员。该方案在人员量较大时可读性和已选管理较弱，也难以同时展示人员映射、部门映射和不同场景的不可选原因。

### 2. 后端提供组织对象选项视图，前端不自行拼接权限事实

新增或扩展治理接口，例如 `GET /governance/wecom-organization-subjects`，返回部门、成员、CRM 用户映射、CRM 部门映射、可选状态和不可选原因。接口必须复用现有治理访问权限校验，只有具备权限中心访问资格或业务页面明确授权资格的用户可以读取。

建议返回结构：

```ts
interface WecomOrgSubjectOptionsView {
  departments: Array<{
    departmentId: string;
    name: string;
    parentDepartmentId?: string;
    displayOrder?: number;
    syncStatus: 'ACTIVE' | 'DELETED';
    crmDepartmentId?: string;
    crmDepartmentName?: string;
    mappingStatus: 'MAPPED' | 'UNMAPPED' | 'CONFLICTED' | 'DELETED';
    disabledReason?: string;
  }>;
  users: Array<{
    wecomUserId: string;
    name: string;
    departmentIds: string[];
    primaryDepartmentId?: string;
    crmUserId?: string;
    crmUserName?: string;
    position?: string;
    avatar?: string;
    mappingStatus: 'MAPPED' | 'UNMAPPED' | 'CONFLICTED' | 'DELETED';
    disabledReason?: string;
  }>;
  lastSyncedAt?: string;
}
```

选择原因：

- 后端更接近企业微信同步镜像、CRM 原生 `wx_*` 表、CRM 用户映射和 CRM 部门映射，能统一处理已删除、未映射、重名和映射冲突。
- 前端只负责展示和选择，不承担权限或身份事实推导，避免出现“页面能选但后端不能保存”的不一致。

备选方案：

- 前端调用现有 `/governance/access-options` 后自行拼树。该接口只有平铺 CRM 用户、部门和企业微信用户，缺少成员所属部门、部门映射状态、CRM 映射状态和部门全选所需关系。

### 3. 组件通过 `subjectTypes` 和 `valueType` 明确选择对象与输出 ID 类型

组件提供 `subjectTypes` 参数控制可选对象类型：

- `['user']`：只选择人员。
- `['department']`：只选择部门。
- `['user', 'department']`：允许混合选择，首版权限中心暂不需要混合保存，但组件能力需要支持后续业务复用。

组件提供 `valueType` 参数控制输出值：

- `crmUserId`：输出 CRM 用户 ID，适用于企业微信灰度白名单/停用名单、超级管理员授权、数据范围白名单授权对象、日报团队组长和组员。
- `wecomUserId`：输出企业微信 userid，适用于身份映射诊断和按企业微信账号预览权限。
- `crmDepartmentId`：输出 CRM 部门 ID，适用于企业微信灰度白名单部门和数据范围白名单授权部门。
- `wecomDepartmentId`：输出企业微信部门 ID，适用于后续只需要企业微信组织对象的业务场景。

当 `valueType='crmUserId'` 时，未映射或映射异常的企业微信成员必须禁用，并展示中文原因，例如“未绑定 CRM，不能保存为授权人员”。当 `valueType='crmDepartmentId'` 时，未映射到 CRM 部门的企业微信部门必须禁用，并展示中文原因。当输出企业微信 ID 时，可允许选择未映射对象，同时展示 CRM 映射状态。

选择原因：

- 现有后端保存接口多数仍以 CRM 用户 ID 或 CRM 部门 ID 为正式授权主体，不能把企业微信 userid 或企业微信部门 ID 错存到 CRM 字段。
- 诊断类场景需要按企业微信 userid 查询，不能强制要求先映射到 CRM 用户。

### 4. 部门节点必须区分“选择部门本身”和“选择部门下人员”

当 `subjectTypes=['department']` 时，管理员勾选部门表示选择该部门本身；当 `subjectTypes=['user']` 且开启部门下人员批量选择时，管理员勾选部门表示选择该部门及子部门下所有当前场景可选人员。若部分成员或部门因未映射、已删除、权限不足或场景限制不可选，应保留禁用状态并在操作反馈中提示“已选择 X 个对象，Y 个对象因未绑定 CRM 未加入”。

选择原因：

- 同一棵组织树同时服务选人和选部门，必须避免“点部门到底是选择部门还是选择人”的歧义。
- 该策略可以同时满足“直接全选某个组内人员”“选择某个部门作为授权范围”和“未映射对象不能保存为 CRM 主体”的要求。

### 5. 搜索状态由组件内部托管并在选择后清空

组件内部维护 `searchKeyword`。无论用户从搜索结果中单选、批量选择，还是从组织树中选择人员或部门，完成一次选择动作后都必须清空搜索关键词。多选场景中清空关键词不应清空已选对象。

选择原因：

- 搜索关键词残留是用户明确反馈的问题，应由统一组件保证，不依赖调用方记得配置。
- 清空搜索不影响已选状态，符合“搜索只是定位手段”的心智。

### 6. 权限中心逐项替换人员和部门入口，保留非组织对象选择器

本次审查后的权限中心接入清单：

- 角色权限：仅包含角色状态筛选和角色权限编辑，不涉及人员或部门选择，本次不替换。
- 企业微信准入：白名单用户、停用用户使用人员模式；白名单部门使用部门模式；灰度模式、白名单角色保留现有选择控件。
- 超级管理员授权：已授权人员使用人员模式；已授权角色保留现有选择控件。
- 白名单设置：授权对象类型为用户时使用人员模式；授权对象类型为角色时保留角色选择；授权部门使用部门模式；包含子部门、授权状态保留现有选择控件。
- 身份与权限诊断：企业微信账号使用人员模式并输出企业微信 userid。
- 用户权限预览：CRM 用户使用人员模式并输出 CRM 用户 ID；企业微信账号使用人员模式并输出企业微信 userid。
- 日报治理：新增/编辑团队中的组长和组员使用人员模式；团队查询框和状态选择不属于组织对象选择。

角色、状态、模式、布尔值等非人员/部门组织对象选择仍使用现有控件。

### 7. 项目级规约要求后续复用该组件

`AGENTS.md` 需要新增前端组织对象选择规约：任何页面或后续新增业务只要涉及选择人员或部门，默认复用 `WecomOrgSubjectPicker.vue` 或其项目级封装；只有明确不是企业微信/CRM 组织对象、或该组件无法表达的特殊业务，才允许单独设计，并需在设计或 OpenSpec 中说明原因。

选择原因：

- 选择人员和部门是跨模块基础能力，继续各写各的会导致搜索残留、映射口径、禁用态和审计字段反复分裂。
- 项目级规约可以防止后续新增页面绕开统一组件。

## Risks / Trade-offs

- [Risk] 企业微信组织事实未同步或同步不完整，选择器可能为空或人员缺部门。  
  Mitigation：接口返回 `lastSyncedAt` 和空状态原因；页面提供中文提示与重试入口，不自动扩大权限或伪造部门关系。

- [Risk] 同一企业微信成员映射到多个 CRM 用户、企业微信部门缺少 CRM 部门映射或 CRM 主体缺失，管理员可能误选。  
  Mitigation：后端统一返回 `mappingStatus` 与 `disabledReason`；CRM ID 场景禁用异常项，诊断场景允许选择但展示状态。

- [Risk] 大组织一次性返回全部部门与人员可能影响页面性能。  
  Mitigation：首版可复用现有组织规模一次性返回；实现时为后续远程搜索和部门懒加载保留接口参数，但不提前引入复杂分页。

- [Risk] 部门节点同时承担部门选择和部门下人员批量选择，用户可能误解当前模式。  
  Mitigation：组件标题、空状态、部门行操作文案和确认反馈必须明确当前是在“选择部门”还是“选择部门下人员”。

- [Risk] 批量部门全选可能一次选择过多人员，造成保存误操作。  
  Mitigation：组件展示已选数量、部门全选反馈和清空入口；高风险保存仍复用现有变更原因、审计和后端校验。

- [Risk] 替换多个 tab 的人员或部门入口可能引入 payload 变化。  
  Mitigation：任务中要求逐项回归现有保存接口字段，测试断言保存 payload 与改造前业务字段一致。

## Migration Plan

1. 更新 `AGENTS.md` 项目规约，明确后续人员/部门选择默认复用统一组织架构组件。
2. 新增组织对象选项接口与类型，不改变现有 `/governance/access-options` 的返回结构。
3. 新增 `WecomOrgSubjectPicker.vue` 并先在单元测试中覆盖人员单选、人员多选、部门选择、部门下人员全选、搜索清空和禁用项。
4. 按权限中心 tab 逐步替换人员与部门选择入口，每替换一类入口即补充页面测试。
5. 完成所有人员和部门入口替换后，保留现有角色、状态、模式等非组织对象选择器，删除权限中心内不再使用的人员/部门选项拼接逻辑。
6. 若上线后出现组织接口异常，前端回退为“无法加载组织架构，请稍后重试”，不得自动降级为无约束平铺全量选人或选部门。

## Open Questions

- 企业微信组织对象接口首版是否需要支持服务端关键词参数，还是先全量返回后由前端本地过滤；建议首版全量返回并保留参数扩展位。
- 企业微信部门到 CRM 部门的映射是否存在完全可靠的直接映射字段；若缺失，首版必须在 CRM 部门 ID 场景禁用该部门，而不是按名称猜测保存。
- 日报团队“新增团队”是否要支持直接选择企业微信部门作为团队来源并自动带出团队名；该能力超出本次“选择人员/部门组件”主目标，建议后续单独评估。
