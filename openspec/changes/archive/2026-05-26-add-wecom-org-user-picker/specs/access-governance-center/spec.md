## ADDED Requirements

### Requirement: 权限中心人员和部门选择入口必须统一复用企业微信组织架构选择器

权限中心中所有选择人员或部门的入口 MUST 统一复用企业微信组织架构选择器，包括企业微信准入、超级管理员授权、白名单设置、日报治理、身份映射诊断和用户权限预览。人员和部门选择入口不得继续各自维护平铺下拉搜索逻辑；角色、状态、模式等非组织对象选择 MAY 继续使用既有选择器。

#### Scenario: 权限中心所有人员和部门入口使用统一选择器

- **WHEN** 管理员分别在权限中心不同标签页选择人员或部门
- **THEN** 系统必须打开同一套企业微信组织架构选择体验
- **THEN** 不同标签页不得出现搜索行为、清空行为或未映射处理口径不一致的问题

#### Scenario: 非组织对象入口不强制改造

- **WHEN** 管理员在权限中心选择角色、状态、模式或布尔配置
- **THEN** 系统可以继续使用现有下拉或分段控件
- **THEN** 本次改造不得要求非组织对象保存字段发生变化

#### Scenario: 权限中心逐页审查人员与部门入口

- **WHEN** 实施本变更时审查权限中心所有标签页
- **THEN** 角色权限页若不包含人员或部门选择则无需替换
- **THEN** 企业微信准入、超级管理员授权、白名单设置、日报治理、身份映射诊断和用户权限预览中的人员或部门入口必须全部替换为统一组织架构选择器

### Requirement: 权限中心人员和部门选择改造不得改变既有治理保存语义

权限中心人员和部门选择器替换后，企业微信灰度策略、超级管理员授权、数据范围白名单、日报团队、身份映射诊断和权限预览的保存或查询 payload MUST 保持原有业务字段语义。前端组件只负责选择体验和 ID 映射，不得在提交时新增绕过后端校验的授权字段。

#### Scenario: 企业微信灰度用户保存字段保持不变

- **WHEN** 管理员通过组织架构选择器调整企业微信灰度白名单用户和停用用户并保存
- **THEN** 请求仍必须提交 CRM 用户 ID 形式的 `allowUserIds` 与 `denyUserIds`
- **THEN** 后端必须继续执行原有灰度策略校验和审计

#### Scenario: 企业微信灰度部门保存字段保持不变

- **WHEN** 管理员通过组织架构选择器调整企业微信灰度白名单部门并保存
- **THEN** 请求仍必须提交 CRM 部门 ID 形式的 `allowDepartmentIds`
- **THEN** 后端必须继续执行原有灰度策略校验和审计

#### Scenario: 数据范围白名单部门保存字段保持不变

- **WHEN** 管理员通过组织架构选择器调整数据范围白名单授权部门并保存
- **THEN** 请求仍必须提交 CRM 部门 ID 形式的 `departmentIds`
- **THEN** 后端必须继续执行原有数据范围白名单校验和审计

#### Scenario: 权限预览查询字段保持不变

- **WHEN** 管理员通过组织架构选择器选择 CRM 用户或企业微信账号执行权限预览
- **THEN** 请求仍必须按场景提交 `crmUserId` 或 `wecomUserId`
- **THEN** 预览结果必须继续复用正式运行时权限决策服务
