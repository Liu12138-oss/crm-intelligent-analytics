## Context

仓库现状已经具备统一 AI 理解层的雏形：Web 问数会先调 `AnalysisIntentService` 与 `AiGatewayService`，企业微信空闲态 / 活跃任务也已引入 `classifyWecomIdleConversationIntent`、`classifyWecomTaskReplyIntent` 等能力，合同审核与日报摘要也存在 AI 调用。然而这些能力尚未形成真正的硬性总线，企业微信入口前仍存在帮助词表、日报词表、项目直查正则、创建入口检测、确认词精确匹配等规则短路；Web 问数仍保留中文语义规则作为主兜底；日报正文采集与部分跟进草稿整理仍以关键词切段和正则提取为主。结果是系统在中文标准表达上“多数可用”，但在多语言、口语变体、歧义短句与混合表达上仍不能满足 AI-first 的统一验收口径。

本次设计的核心不是继续“补词表”，而是把自然语言语义理解、固定程序路由、执行确认门闩和安全 fallback 明确拆层，并让现有企业微信、Web 与后续合同审核自然语言能力都复用同一套约束。

## Goals / Non-Goals

**Goals：**
- 让所有业务自然语言文本在通过固定安全前置检查后，统一先进入 AI 理解层，再由固定程序消费结构化 schema 做路由、补问、阻断或执行。
- 明确“AI 理解”与“确定性执行门闩”分层，避免把高风险提交、写回、创建完全交给模型判断。
- 把企业微信空闲态消息、活跃任务短回复、日报 / 跟进自由文本、客户 / 商机创建续填、Web 问数与追问统一纳入同一套 AI 入口契约。
- 收紧 fallback，只允许帮助、澄清或阻断，不再允许旧规则长期承担主理解链路。
- 为每次理解结果留下可检索审计快照，并补齐多语言、变体表达与歧义短句的验证样例。

**Non-Goals：**
- 不改变固定安全前置检查边界，不把签名校验、登录鉴权、文件校验、权限装载、白名单校验交给 AI。
- 不在本次变更中放开新的写入能力，也不扩大 CRM 数据白名单。
- 不把合同审核现有“上传 -> 固定校验 -> 任务创建 -> 结果查看”流程重构为聊天式审核，仅为后续自然语言说明预留统一入口方式。
- 不要求所有按钮式、结构化表单式或显式候选序号选择都强制走 AI；本次只约束自由文本业务输入。

## Decisions

### 决策一：引入统一 `AiEntryIntent` 结构，所有文本入口都先产出同一份结构化结果

统一 AI 理解层必须输出稳定 schema，而不是每个模块维护各自的“差不多像 JSON”的结构。建议最小字段集至少包含：

- `scene`：当前文本所属场景，例如 `WEB_ANALYSIS_QUERY`、`WEB_ANALYSIS_FOLLOW_UP`、`WECOM_IDLE_MESSAGE`、`WECOM_ACTIVE_TASK_REPLY`、`WECOM_FREEFORM_DRAFT`。
- `targetWorkflow`：固定程序目标，例如 `ANALYSIS_QUERY_EXECUTION`、`ANALYSIS_RESULT_EXPLANATION`、`WECOM_HELP_GUIDANCE`、`WECOM_CRM_CREATE_CUSTOMER`。
- `intent` / `replyIntent`：语义分类结果，例如 `ANALYZE`、`EXPLAIN_RESULT`、`CONFIRM`、`CANCEL`、`MODIFY`、`TASK_SWITCH`。
- `structuredSlots`：由 AI 抽取的结构化槽位，例如指标、维度、过滤条件、草稿字段、候选对象文本、时间范围。
- `confidence`、`language`、`usedFallback`、`fallbackReason`、`blockReason`。

固定程序只允许消费这份 schema，不再直接读取原始关键词判断是否切换日报、是否确认提交、是否继续写回。这样能把“AI 理解错了”与“程序路由错了”分开审计，也能统一灰度和回滚边界。

备选方案是继续保留各模块私有结构，仅在审计时拼接兼容字段。该方案短期改动小，但会继续放大模块间语义不一致、字段命名漂移和测试重复成本，因此不采用。

### 决策二：把“语义理解”与“执行确认门闩”明确分层

本次不追求“所有动作都交给 AI 决定”，而是要求“所有文本先交给 AI 理解”。对高风险动作，AI 只负责将文本识别为 `CONFIRM / CANCEL / MODIFY / CONTINUE / UNKNOWN`，真正执行写回、提交、创建、状态推进仍由固定程序门闩决定。门闩必须结合当前任务状态、候选唯一性、草稿完整性、权限边界和幂等约束，再决定是否真正执行。

这意味着：

- `确认 / 好的 / proceed / 就这么办 / 先这样提交` 这类表达必须先经 AI 理解；
- 但 AI 命中 `CONFIRM` 后，系统仍需验证当前会话是否处于允许确认的状态；
- 对写回、创建、正式提交这类动作，若状态不完整或上下文有歧义，程序必须继续澄清，而不是直接执行。

备选方案一是继续纯规则确认，问题是语言覆盖面太差；备选方案二是让 AI 直接决定执行，问题是高风险误触发代价过高。因此采用“AI 理解 + 确定性门闩”的分层方案。

### 决策三：将 fallback 收缩为“最小安全 fallback”，不再允许规则主链冒充 AI 主链

当前最大问题不是“有 fallback”，而是 fallback 仍承担大量主链职责。本次要求 fallback 只保留三类能力：

- 返回帮助能力清单；
- 请求用户澄清；
- 明确阻断不安全或超范围请求。

旧的帮助关键词、日报入口关键词、项目直查正则、确认词表、字段切段关键词可以保留，但只能在 AI 不可用或结构非法时提供最小安全兜底，且必须显式记录 `usedFallback=true` 与具体 `fallbackReason`。它们不再允许直接承担主入口识别、主任务切换或主确认推进。

备选方案是继续保留“AI 主链 + 规则副主链”。该方案看似稳妥，实则会持续制造口径分裂，并让灰度结果难以解释，因此不采用。

### 决策四：日报、跟进、客户创建、商机创建的自由文本统一改为 AI 抽槽位主链

对于企业微信自由文本，不再让关键词拆段和正则提取承担主理解。统一改为：

- AI 负责提取四段草稿、客户 / 商机创建字段、当前回复语义、候选对象文本；
- 固定程序负责字段校验、候选召回、对象唯一化、权限检查、执行确认与写回。

这样既保留了程序对业务边界的控制，也能支持多语言、变体表达、口语化和错误输入。现有规则 helper 只在 AI 主链失败时，作为最小兜底输出“我还缺哪些字段 / 我需要你澄清哪个对象”。

### 决策五：每次理解都必须形成“理解快照 + 路由快照 + 执行快照”

统一审计必须至少包含三层：

- `entryInterpretationSnapshot`：AI 理解输出、场景、语言、置信度、fallback 情况；
- `workflowRoutingSnapshot`：最终固定程序、目标工作流、是否需要确认；
- `executionSnapshot`：真正执行的读写动作、执行模式、阻断原因、工具边界。

这三层分开后，才能判断问题究竟出在 AI 理解、程序路由还是执行门闩。

### 决策六：以多语言和反例测试作为验收门槛，而不是“补更多中文词表”

本次验收样例必须覆盖：

- 英文、中英混写、韩文 / 日文等非纯中文表达；
- 错别字、缩写、口语化中文、带情绪语气词的确认与取消；
- “像确认但实际是修改”的歧义句；
- “像日报但实际是跟进补充”的跨任务文本；
- AI 不可用时的帮助 / 澄清 / 阻断回退。

若系统只能靠继续补中文关键词才能通过测试，则说明本次设计未达标。

## Risks / Trade-offs

- [AI 理解层覆盖面扩大后，首次接入成本上升] → 先统一 schema 和路由骨架，再逐模块迁移，避免一次性替换全部旧逻辑。
- [高风险动作若完全交给 AI 会带来误执行风险] → 保留确定性执行门闩，把 AI 限定在语义理解层。
- [fallback 收缩后，短期内部分历史表达可能不再“凑巧命中”旧规则] → 通过澄清与帮助兜底保证安全可用，并补齐真实表达测试集。
- [企业微信不同任务链路原本各自稳定，统一收口可能引入回归] → 以“空闲态路由、活跃态回复、自由文本抽槽、确认门闩”四层拆分实施，逐层加集成测试。
- [灰度期间 AI 开关关闭可能导致入口不可用] → 重新定义开关语义：关闭 AI 主链时只退回最小安全 fallback，不再恢复旧规则主链。

## Migration Plan

1. 先新增统一 `AiEntryIntent` schema、校验器和审计快照模型，让现有 Web 与企业微信链路都能输出统一结构。
2. 优先迁移 Web 问数与企业微信空闲态 / 活跃态路由，收回入口前置规则短路。
3. 迁移跟进 / 日报 / 创建类自由文本抽槽，把旧 helper 下沉为 fallback。
4. 最后迁移确认 / 取消 / 修改 / 继续等短回复识别，并把写回 / 创建提交改成“AI 理解 + 确定性门闩”。
5. 灰度期间按入口、场景和语言维度记录 `usedFallback`，若误判上升则先回退到“AI 理解 + 澄清帮助”，而不是恢复旧规则主链。
6. 回滚策略：保留旧 helper 和旧正则作为最小安全 fallback，但禁止它们重新成为主分流逻辑。

## Open Questions

- 企业微信确认类高风险动作，最终是否需要补充按钮式确认，进一步降低自由文本确认误触发风险？
- 多语言支持是否只要求“常见外文和混写表达可安全理解 / 澄清”，还是要明确首期支持语言集合并纳入产品文案？
- 当前日报正文与跟进正文是否需要共用同一套抽槽位 schema，还是仅共享基础字段并保留各自场景字段扩展？
