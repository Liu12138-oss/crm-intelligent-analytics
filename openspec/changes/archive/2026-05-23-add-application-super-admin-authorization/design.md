## Context

系统当前已经有角色权限矩阵、数据范围白名单和分析全量查询名单三类治理能力，但它们的语义不同：

- 角色权限矩阵控制菜单、动作、通道和部分高风险动作。
- 数据范围白名单只扩展部门数据范围。
- 分析全量查询名单只在分析查询链路中返回 `FULL_ANALYSIS_SCOPE`，并没有统一影响经营报表、合同审批、治理入口、审计检索和所有后端动作。

这导致“菜单已开通”与“真实数据/动作可用”之间存在断层。应用超级管理员授权需要成为权限决策和数据范围解析的统一覆盖层，同时仍遵守登录认证、安全前置检查、字段白名单、SQL 白名单、危险动作确认和审计要求。

## Goals / Non-Goals

**Goals:**

- 建立“应用超级管理员授权”策略，支持按 CRM 用户和 CRM 角色授权。
- 让命中主体拥有系统内所有菜单、所有动作、所有治理入口、所有导出/跨看/跨下载等高风险能力。
- 让命中主体获得全量数据范围，并让分析问数、经营报表、合同审批、模板执行、最近查询重跑和导出补查消费同一份范围结果。
- 将现有“全量查询授权”迁移为“超级管理员授权”，避免继续保留两个含义接近但效果不同的入口。
- 所有授权变更、权限预览、范围快照和高风险操作继续可审计、可回溯。

**Non-Goals:**

- 不改变登录认证、会话建立、企业微信扫码登录和共享 Cookie 回跳链路。
- 不绕过企业微信签名、来源校验、文件类型与大小限制、SQL 白名单、字段白名单、导出行数和每日次数等安全基线。
- 不新增一套脱离 CRM 用户、角色、组织和部门关系的独立人员体系。
- 不把 CRM 原生管理员 `isAdmin` 与应用超级管理员混为同一个标记。

## Decisions

### 1. 使用独立的应用超级管理员策略，而不是复用 `isAdmin`

新增或升级策略存储为 `ApplicationSuperAdminPolicy`，结构上至少包含授权主体、更新人、更新时间和变更原因。授权主体支持 `USER` 与 `ROLE` 两类：

```text
subjectType: USER | ROLE
subjectId: CRM 用户 ID 或 CRM 角色 ID
status: ACTIVE | INACTIVE
changeReason: 中文变更原因
```

选择独立策略的原因是 `isAdmin` 来自 CRM 登录身份快照，代表 CRM 原生管理员语义；应用超级管理员是本系统治理层的显式授权，两者来源、审计和回滚路径不同。

备选方案是把命中用户临时改写为 `isAdmin = true`，但这会让审计无法区分 CRM 管理员和应用超级管理员，也容易误触历史兼容逻辑，因此不采用。

### 2. 在统一权限决策层做功能覆盖

`AccessDecisionService` 在解析角色权限后检查当前用户是否命中应用超级管理员策略。命中后，统一返回：

- 全部 `FEATURE_PERMISSION_CATALOG` 中登记的菜单键。
- 全部 `FEATURE_PERMISSION_CATALOG` 中登记的动作键。
- `webConsoleEnabled = true`。
- `wecomBotEligible = true`，但仍必须受全局企业微信通道是否启用控制。
- `exportAllowed = true`、模板治理、合同上传、合同跨看、合同跨下载、SQL 审计敏感查看等高风险能力均为 true。

选择在统一权限决策层覆盖，是为了让前端快照、权限预览、后端 `ensureAction` 和真实执行点得到一致结果，避免每个业务模块单独判断。

### 3. 在统一范围解析层做全量数据范围覆盖

`UserScopeService` 或 `OrganizationScopeService` 在解析范围时消费同一份应用超级管理员策略。命中后返回全量范围快照：

```text
organizationIds: 当前用户组织边界
departmentIds: []
ownerIds: []
scopeSource: application-super-admin
scopeSummary: 当前已开通应用超级管理员授权，可查看全公司数据。
```

空 `departmentIds` 和空 `ownerIds` 在全量模式下必须被明确解释为“不按部门/负责人收口”，不能被业务模块降级成 `['__none__']`。经营报表、合同审批、模板执行等模块必须消费 `scopeSource` 或等价全量标记，不能继续直接取 `user.ownerIds`。

### 4. 旧分析全量名单迁移为超级管理员名单

现有 `analysisScopePolicy.fullAccessUserIds` 迁移到新的应用超级管理员策略中，迁移后的主体类型为 `USER`。迁移完成后，“分析全量查询授权”不再作为独立页面和独立运行时判断存在；分析问数只读取统一范围结果。

如果短期需要兼容旧数据，仓储层可以在读取新策略时合并旧字段，但保存时必须写入新结构，并在审计中标明来自历史迁移。

### 5. 安全边界保持在超级管理员授权之外

超级管理员授权只决定“是否具备系统内业务功能和数据范围”，不代表绕过固定安全基线：

- 未登录或会话失效仍拒绝。
- 企业微信签名、来源校验和通道关闭仍生效。
- 文件上传类型、大小、解析安全限制仍生效。
- SQL 只读白名单、字段白名单、敏感字段处理仍生效。
- 写回、导出、敏感明细 reveal 等行为仍写审计，必要时继续执行确认或阈值限制。

## Risks / Trade-offs

- [Risk] 超级管理员授权过宽，误配会造成全系统数据和操作暴露。
  Mitigation：权限中心增加高风险提示，保存必须填写变更原因；所有变更写入高风险审计；支持禁用/撤销并立即失效。

- [Risk] 部分模块仍直接使用 `user.ownerIds`、`user.departmentIds` 或旧全量名单，导致超级管理员仍被局部收口。
  Mitigation：任务中逐一整改经营报表、合同审批、分析问数、模板执行、导出补查和权限预览，并增加回归测试。

- [Risk] 空范围数组在不同模块中含义不一致。
  Mitigation：在 `ScopeSnapshot` 中新增明确的全量标记或 `scopeSource: application-super-admin`，业务查询必须根据标记构造无收口条件，而不是依赖空数组推断。

- [Risk] 与现有“应用层能力不得扩大 CRM 数据范围”规则冲突。
  Mitigation：将应用超级管理员定义为受审计的显式例外，只能由系统级治理权限用户维护，并且仍保留组织边界、安全基线和审计。

## Migration Plan

1. 新增应用超级管理员策略结构和仓储，读取时兼容旧 `analysisScopePolicy.fullAccessUserIds`。
2. 在权限中心新增“超级管理员授权”视图，保存时写入新策略。
3. 将旧全量查询名单迁移为用户级超级管理员授权，迁移后不再展示旧页签。
4. 接入权限决策和范围解析。
5. 整改直接读取 `user.ownerIds` / `user.departmentIds` 的关键模块。
6. 运行权限、范围、经营报表、合同审批、模板执行和导出回归测试。
7. 部署后通过权限预览确认已授权人员的菜单、动作和范围均为超级管理员口径。

Rollback：保留旧字段读取兼容一段时间；如新策略异常，可禁用所有应用超级管理员授权并回退到原角色权限矩阵和数据范围白名单。

## Open Questions

- 应用超级管理员是否应默认获得企业微信机器人入口资格，还是仍要求全局企业微信入口为 `FULL` 或命中灰度策略后才可使用。
- 导出每日次数和单次行数是否对应用超级管理员继续生效；当前设计为继续生效，除非后续明确调整全局基线。
