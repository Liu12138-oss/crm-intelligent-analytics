## Context

当前前端使用 Vue 3、Vite、Pinia 和手写全局 CSS，运行时依赖尚未引入 UI 组件库。近期 Web UI 已按 `DESIGN.md` 重设为浅色企业工作台，但按钮、输入框、表格、标签、抽屉、上传区、加载状态、空状态和图标仍主要依赖自定义 class。随着分析工作台、结果详情、合同审核、治理配置和审计中心继续扩展，继续手写基础交互组件会放大维护成本，并且当前页面已经暴露出图标语义不足、请求态反馈容易遗漏的问题。

`README.md`、`DESIGN.md` 和 `AGENTS.md` 已明确后续统一选用 `Element Plus`，并配套使用 `@element-plus/icons-vue`。本设计说明定义后续实施如何接入、主题化和分阶段替换，确保 UI 框架提升交互质量，而不是把系统改回默认后台模板。

## Goals / Non-Goals

**Goals:**

- 引入 `element-plus` 与 `@element-plus/icons-vue`，建立 Vue 3 基础 UI 组件与图标基线。
- 通过主题变量和项目样式覆盖，让 Element Plus 与 `DESIGN.md` 的色彩、圆角、间距、状态和响应式规则一致。
- 渐进式替换现有手写控件，优先覆盖按钮、输入框、表单、表格、标签、提示、抽屉、上传、加载、空状态和消息反馈。
- 系统性补齐导航、操作、风险、成功、失败、警告、上传、导出、刷新、筛选、搜索、返回、关闭、展开收起等图标语义。
- 保持现有路由、事件绑定、`v-model`、store/service 调用、权限说明、审计留痕和验收行为不变。
- 为后续 UI 重构建立可验证任务清单，确保 lint、单元测试、E2E 和 build 能覆盖关键链路。

**Non-Goals:**

- 不重写业务状态管理、service API、router、types、后端 API、数据库或 OpenAPI 契约。
- 不把 Element Plus 默认主题作为最终视觉，不迁移到传统蓝白后台风格。
- 不引入新的图表库、图标设计系统或低代码页面框架。
- 不在本提案阶段直接安装依赖或重构页面实现。
- 不改变登录、分析、合同审核、治理审计和企业微信能力的功能边界。

## Decisions

### 决策 1：选择 Element Plus 作为唯一基础 UI 框架

后续前端重构统一采用 `Element Plus`，不同时混用 Ant Design Vue、Naive UI、Arco Design Vue、Vuetify 或其他桌面端组件库。

原因：

- 当前技术栈是 Vue 3 + TypeScript + Vite，Element Plus 与该栈匹配度高。
- CRM 智能分析系统需要大量企业后台基础组件，如表单、表格、上传、抽屉、消息、加载、空状态和分页，Element Plus 覆盖成熟。
- Element Plus 的默认视觉可通过 CSS 变量收敛，不会强迫系统采用明显 Ant 风格或重型 Material 风格。
- 团队后续维护和问题检索成本较低。

备选方案：

- Ant Design Vue：企业后台能力成熟，但默认视觉识别度强，容易覆盖 `DESIGN.md` 的 Stripe 化浅色工作台方向。
- Naive UI：主题能力强，视觉更现代，但国内 CRM 后台协作经验和团队接受度可能不如 Element Plus。
- Arco Design Vue：组件和图标体系完整，但会引入较强的 Arco 视觉语法，重构时需要更多主题对齐工作。

### 决策 2：使用 CSS 变量主题覆盖，不直接定制源码或复制 Element Plus 样式

实施时应导入 Element Plus 基础样式，再在项目全局样式中覆盖 `--el-color-primary`、`--el-border-radius-base`、`--el-text-color-*`、`--el-border-color-*`、`--el-bg-color-*`、`--el-font-family` 等变量，使其映射到 `DESIGN.md` 的令牌。

原因：

- 当前项目已经存在根级 CSS 变量，使用 CSS 变量映射成本最低。
- 避免复制 Element Plus 内部样式导致后续升级困难。
- 能保留项目现有视觉方向，同时获得 Element Plus 的交互和无障碍基础能力。

备选方案：

- Sass 变量深度定制：能力更强，但增加构建复杂度和升级成本。
- 页面局部覆盖：短期快，但容易出现页面风格漂移。

### 决策 3：图标优先使用 `@element-plus/icons-vue` 并建立项目映射

后续应在 `frontend/src/ui/icons.ts` 或等价模块中集中导出常用图标映射，例如分析、合同、治理、审计、上传、下载、刷新、筛选、搜索、警告、成功、失败、返回、关闭、展开收起等语义。页面不得继续用首字、纯文本或临时符号承担关键图标语义。

原因：

- 当前页面存在图标缺口，导航和操作识别主要靠文字，信息密度上升后会降低可扫读性。
- 集中映射可以让图标语义可审查、可替换，避免每个页面随意挑选图标。
- 状态表达必须同时使用文字和图标，符合 `DESIGN.md` 无障碍要求。

备选方案：

- 每个页面自行引入图标：实现简单但会导致语义和风格不一致。
- 自建 SVG 图标库：视觉可控但投入更高，不适合当前阶段。

### 决策 4：采用渐进式页面替换，而不是一次性全量重写

实施顺序应为：

1. 安装依赖并建立 Element Plus 插件、主题覆盖和图标映射。
2. 替换全局基础控件和低风险组件，如按钮、输入框、标签、提示、空状态、加载状态。
3. 改造应用壳层与登录页，验证导航图标、登录请求态和扫码入口。
4. 改造分析链路，覆盖问数输入、查询资产、结果摘要、表格、导出、追问和抽屉。
5. 改造合同审核、治理策略、模板管理、连接策略和审计中心。
6. 逐页运行回归测试并补齐缺失用例。

原因：

- 当前页面已经可用，重构目标是提高一致性和交互质量，不应引入功能回归。
- 渐进式替换能让每个页面的事件绑定、请求态、权限边界和审计语义被单独验证。

备选方案：

- 全量页面重写：视觉统一更快，但回归风险高。
- 只安装 Element Plus 不替换页面：无法解决图标缺失和手写控件维护问题。

### 决策 5：保留现有业务函数和数据流

Element Plus 重构只允许改变模板结构、组件标签、class、图标、局部展示文案和样式，不得改变 `<script setup>` 中的业务调用语义。若某页面必须新增局部 loading 状态或 UI 展示状态，应保持在页面组件内部，不上升为业务模型变更。

原因：

- 本项目功能边界、权限和审计规则已经由主规格和接口契约约束。
- UI 框架替换不应改变系统读写边界。
- 保留原调用有利于测试迁移和人工审查。

## Risks / Trade-offs

- [默认主题污染项目视觉] → 通过 `DESIGN.md` 变量映射、项目级样式覆盖和页面验收阻止默认蓝白后台风格扩散。
- [一次性重构造成事件绑定遗漏] → 按页面分批替换，每批核对 `v-model`、`:disabled`、`@click`、上传、导出、筛选和抽屉绑定。
- [Element Plus 组件 DOM 结构导致测试失效] → 优先按用户可见行为调整测试，避免过度依赖内部 DOM；保留关键文案和角色选择器。
- [图标使用混乱或语义不一致] → 建立项目图标映射表，集中审查图标用途。
- [包体积增加] → 首版接受依赖增加，后续如构建体积超出预期，再评估按需导入和自动导入插件。
- [Element Plus 与现有 CSS 冲突] → 先建立主题覆盖文件和选择器边界，再逐步删除冗余自定义 class。
- [重构范围过大] → 本变更只覆盖前端 UI 层，不触碰后端、OpenAPI、数据库和业务权限模型。

## Migration Plan

1. 安装 `element-plus` 与 `@element-plus/icons-vue`，更新锁文件。
2. 在 `frontend/src/main.ts` 接入 Element Plus 插件或按项目约定的局部导入方式，并导入基础样式。
3. 新增或整理 `frontend/src/styles/element-plus-theme.css`，将 Element Plus CSS 变量映射到 `DESIGN.md` 令牌。
4. 新增项目图标映射模块，先覆盖导航、操作、状态和风险图标。
5. 按登录页、应用壳层、分析链路、合同审核、治理审计的顺序逐步替换。
6. 每个页面替换后运行对应单元测试或 E2E，并核对关键事件绑定不变。
7. 最终运行 `pnpm --dir frontend lint`、`pnpm --dir frontend test:unit`、`pnpm --dir frontend test:e2e`、`pnpm --dir frontend build`。

回退策略：

- 若接入后出现严重回归，先保留依赖但回退页面模板替换；如果基础样式污染范围过大，再移除 Element Plus 全局样式导入并恢复原全局样式。
- 每个页面分批提交或分批验收，确保可以按页面回退。
