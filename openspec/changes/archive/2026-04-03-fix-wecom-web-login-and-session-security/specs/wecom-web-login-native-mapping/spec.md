## ADDED Requirements

### Requirement: 已同步企业微信用户必须可直接扫码登录 Web 后台
系统在处理企业微信 Web 扫码回调时，必须优先复用 CRM 原生 `wx_user_maps -> wx_users` 映射结果解析当前扫码用户；当映射已存在且用户具备 Web 渠道权限时，系统必须直接建立 Web 登录会话，而不能再次要求输入用户名密码。

#### Scenario: CRM 原生映射已存在时扫码直登
- **WHEN** 企业微信扫码回调返回了可解析的 `userid`，且 CRM 原生 `wx_users` 与 `wx_user_maps` 已存在有效映射
- **THEN** 系统必须直接解析出对应 CRM 用户并建立 `wecom-scan` 来源的 Web 会话

#### Scenario: 已同步用户不得被误判为未绑定
- **WHEN** 企业微信扫码回调命中了 CRM 原生映射
- **THEN** 系统必须跳过“请输入用户名密码完成绑定”的提示与待绑定流程

### Requirement: 扫码未命中时的手工绑定必须写回 CRM 原生映射
当企业微信扫码未命中 CRM 原生映射时，系统可以提示用户通过用户名密码完成一次绑定；一旦认证成功，系统必须将企业微信用户与 CRM 用户的正式绑定写回 CRM 原生映射表，并清理待绑定令牌。

#### Scenario: 未命中映射时进入待绑定流程
- **WHEN** 企业微信扫码回调未命中 CRM 原生 `wx_user_maps -> wx_users`
- **THEN** 系统必须创建一次短时有效的待绑定令牌，并引导用户返回登录页输入用户名密码完成绑定

#### Scenario: 手工绑定成功后写回 CRM 原生映射
- **WHEN** 用户携带有效待绑定令牌完成用户名密码认证
- **THEN** 系统必须将当前企业微信用户与 CRM 用户的关系写入 CRM 原生映射表，而不能只写入进程内临时仓库

#### Scenario: 服务重启后绑定仍然有效
- **WHEN** 用户此前已经通过手工绑定成功，且服务已经重启
- **THEN** 系统必须仍可基于 CRM 原生映射完成后续扫码直登

### Requirement: 失效待绑定令牌不得产生正式绑定
系统处理扫码未命中后的手工绑定时，必须验证待绑定令牌是否存在且未过期；令牌失效时不得创建正式绑定，也不得建立登录会话。

#### Scenario: 待绑定令牌已过期
- **WHEN** 用户携带已过期的待绑定令牌提交用户名密码
- **THEN** 系统必须拒绝本次绑定请求，并提示用户重新扫码

#### Scenario: 待绑定令牌不存在
- **WHEN** 用户提交的待绑定令牌不存在或已被使用
- **THEN** 系统必须拒绝本次绑定请求，并且不得向 CRM 原生映射表写入任何新关系
