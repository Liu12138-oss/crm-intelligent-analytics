## Context

当前后端已经具备企业微信目录同步到 CRM 原生 `wx_users`、`wx_user_maps` 的能力，企业微信机器人入口也已经优先使用 CRM 原生映射识别用户。但 Web 扫码登录仍停留在“先查进程内 `wecomLoginBindings`，查不到就要求手工绑定”的旧逻辑，导致“组织架构已同步成功”与“扫码仍要求再次认证”并存。

同时，Web 后台受保护接口当前仍允许在缺少正式登录会话时回退到 `x-crm-user-id` / `crm_session_user` 直接装载用户，这与当前后台管理系统的正式登录要求不一致，也放大了伪造身份的风险。前端又把 `sessionId` 持久化到 `localStorage` 并通过请求头回传，进一步削弱了会话边界。

本次变更需要同时覆盖后端认证、企业微信扫码、CRM 原生映射写回、前端会话承载和测试回归，属于跨模块安全收口。

## Goals / Non-Goals

**Goals:**

- 让已完成企业微信组织架构同步的用户可直接通过企业微信扫码登录 Web 后台。
- 让扫码未命中时的手工用户名密码认证在成功后正式写回 CRM 原生映射，而不是只写入进程内临时仓库。
- 收掉 Web 后台管理系统受保护接口对裸用户标识的放行路径，只保留正式会话鉴权。
- 把浏览器登录态改为以后端 HttpOnly Cookie 为主，前端不再保存可复用的 `sessionId`。
- 用自动化测试覆盖扫码直登、绑定写回和鉴权收口。

**Non-Goals:**

- 不改动企业微信机器人问数主链路的业务编排与结果返回逻辑。
- 不在本次变更中扩展新的 CRM 业务写回能力，绑定写回仅限认证所需的 `wx_*` 映射表。
- 不重新设计登录页视觉结构，只修正其数据流和会话处理方式。

## Decisions

### 决策一：企业微信 Web 扫码登录优先复用 CRM 原生 `wx_user_maps -> wx_users`

`WecomWebLoginService` 在拿到企业微信 `code` 后，先通过企业微信接口解析 `userid / mobile / email`，随后优先按 `userid` 走 CRM 原生映射解析。只要 `wx_users + wx_user_maps` 已存在可用映射，就直接装载 CRM 用户并建立 Web 会话，不再经过本地 `wecomLoginBindings`。

这样做的原因：

- 目录同步已经把正式身份映射沉淀到 CRM 原生表，扫码登录应直接复用同一事实源。
- 企业微信机器人和 Web 扫码登录复用同一映射来源后，排障口径才一致。
- 继续把进程内绑定仓库放在首位，会让“已同步”与“仍需绑定”长期冲突。

备选方案是保留现状，只把扫码命中失败提示文案改掉。该方案无法真正打通登录链路，不采用。

### 决策二：手工绑定成功后必须写回 CRM 原生映射，进程内仓库只保留短时待绑定令牌

扫码未命中时，仍允许用户通过用户名密码完成一次绑定。但绑定成功后不再调用 `WecomLoginBindingRepository.saveBinding()` 作为正式结果，而是通过 CRM 原生映射写入能力把 `wecomUserId -> crmUserId` 写入 `wx_user_maps`。本地仓库仅保留 `pendingWecomBindings` 这类 10 分钟内有效的待绑定令牌，用于串起一次扫码到一次账号认证。

这样做的原因：

- 进程内正式绑定关系会在重启后丢失，无法支撑生产环境。
- CRM 原生映射已经是机器人认证主链路，手工绑定也必须落到同一事实源。
- 将“临时态”和“正式态”拆开后，逻辑边界更清晰。

备选方案是把 `WecomLoginBindingRepository` 改成持久化本地库。该方案会制造第二套长期映射源，不采用。

### 决策三：Web 后台受保护接口只接受正式会话，不接受裸用户标识降级

`SessionAuthGuard` 改为只识别正式认证会话。浏览器生产链路只接受 `crm_auth_session` Cookie；对于测试代码，保留专用测试环境能力，但不能作为运行时默认分支出现在正式接口上。

这样做的原因：

- 裸 `x-crm-user-id` 等同于“知道用户 ID 即可冒充身份”，不适合作为后台正式鉴权方案。
- 当前接口契约本就声明以 Cookie 会话为主，应与实现对齐。
- 后续审计、登出、续期和会话失效都需要真实会话实体支撑。

备选方案是继续保留头部降级，只在网关层约束。该方案在代码层无法自证安全边界，不采用。

### 决策四：前端不再持久化 `sessionId`，统一通过 Cookie + `/auth/session` 水合

前端移除 `localStorage` 中的 `crm-auth-session-id`，不再通过 `x-auth-session-id` 请求头回传会话。应用启动、路由切换和扫码回流后的登录态恢复，统一通过带 Cookie 的 `/auth/session` 完成。

这样做的原因：

- `sessionId` 属于可复用凭证，不应暴露给浏览器脚本长期保存。
- 现有“本地存储哨兵值 + Cookie 并存”的双轨模式会导致状态不一致。
- 只保留 Cookie 主链路后，登录态恢复和失效处理更稳定。

备选方案是继续保留 `localStorage` 作为兜底。该方案无法收敛安全边界，不采用。

## Risks / Trade-offs

- [扫码直登从本地绑定仓库切到 CRM 原生映射后，若线上存在历史只存本地、不在 CRM 的绑定记录] → 首次扫码会重新触发一次手工绑定，但绑定成功后会写回 CRM，之后恢复稳定。
- [收掉裸用户标识降级后，依赖该能力的联调脚本可能失效] → 通过测试专用夹具或环境隔离保留测试能力，不让其进入正式运行分支。
- [前端改为 Cookie-only 后，若跨域 Cookie 配置不正确，会导致登录后无法恢复会话] → 同步检查 `credentials: include`、Cookie `sameSite` / `secure` 与本地联调地址配置，并补充回归测试。

## Migration Plan

1. 新建 OpenSpec 修复变更，补齐 proposal、design、specs、tasks。
2. 先改后端扫码解析与绑定写回链路，再收口 `SessionAuthGuard`。
3. 调整前端登录态承载方式，移除 `localStorage` 会话令牌依赖。
4. 补充后端集成测试与前端单测，覆盖关键回归场景。
5. 验证通过后更新任务状态，作为后续归档输入。

回滚策略：

- 如扫码登录链路出现问题，可临时回退到“扫码后走一次手工绑定”的交互，但正式绑定仍应保留写回 CRM 原生映射。
- 如 Cookie-only 登录态联调受阻，可先回滚前端改动，但不回滚后端去除裸用户标识放行的安全收口。

## Open Questions

- 现有线上是否存在只保存在 `wecomLoginBindings`、尚未落到 CRM 原生映射的历史正式绑定数据；若存在，是否需要一次性迁移脚本。
- 本地联调环境是否需要显式暴露 `CRM_AUTH_COOKIE_SECURE=false` 之类的配置项，以便和生产环境区分 Cookie 策略。
