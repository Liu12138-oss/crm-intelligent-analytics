## Context

一期日报规格解决的是“日报如何存、如何确认、如何汇总”，而当前这批实现解决的是“销售如何在企业微信里把一句自然口述变成可确认、可补录、可继续写回 CRM 的日报输入”。现有代码已经出现新的会话状态、日报入口关键词、候选实体确认、固定五问追问顺序和写回桥接逻辑，但这些行为还没有被正式写进 OpenSpec。

这条变更因此定义为“日报二期增强”：继续复用一期日报域模型和状态机，不另起日报数据结构；重点补齐企业微信入口的自由口述拆解、主数据确认、候选选择、多轮补问和日报内写回桥接。

## Goals / Non-Goals

**Goals:**
- 让销售可以通过一段自然口述进入日报收集，并自动拆成固定五类日报字段。
- 让系统在正式落日报片段前先识别客户/项目候选并给出确认摘要，降低误写风险。
- 让日报补问严格按固定顺序推进，并把“没有”“无需共享”等回复视为有效收口。
- 让日报场景在明确写回意图时可以桥接到受控 CRM 跟进写回能力，写回完成后继续补日报。

**Non-Goals:**
- 不重写一期日报域的数据模型、确认流程、提醒流程和汇总批次逻辑。
- 不把普通问数或普通聊天自动提升为日报输入。
- 不允许日报场景绕过既有写回确认、审计和幂等保护直接写入 CRM。
- 不新增 Web 日报页面或改动当前日报工作台信息架构。

## Decisions

1. 采用“主题入口 + 规则拆解 + 主数据确认”的日报二期入口，而不是直接依赖生成式对话自由发挥。
   日报是高频且结构明确的流程型动作，企业微信入口必须可预测。只有命中“帮我写今天的跟进”“写今日跟进”等主题入口关键词时，系统才进入日报链路；进入后先用规则和主数据匹配拆出候选片段与实体，再让用户确认。备选方案是把所有带“拜访/跟进”的话都自动视为日报，但这会误伤普通聊天或问数描述。

2. 自由口述先进入“待确认识别结果”，确认后再写正式日报片段。
   当前实现已经体现出这一点：系统先把口述拆句、识别 `TODAY_FOLLOW_UP` 等片段类型、匹配 CRM 客户/项目、生成逐条摘要，然后提示用户确认；只有确认后，相关片段才进入正式日报收集顺序。这样可以避免客户名、项目名或口述边界识别错误时直接污染日报内容。备选方案是先落库再让用户修正，但回滚和审计都会更复杂。

3. 五类日报字段继续沿用一期顺序，但补问规则改为“只问下一项缺失字段”。
   固定顺序为：`TODAY_FOLLOW_UP`、`CUSTOMER_OR_OPPORTUNITY_CHANGE`、`INFORMATION_SHARE`、`HELP_REQUIRED`、`TOMORROW_PLAN`。日报二期不再一次性丢给用户完整表单，而是在当前项确认后只追问下一项缺失内容，并允许“没有”“无需共享”等标准负向回答直接收口。备选方案是一次性提示全部字段，但在企业微信里阅读负担更高，补录率也更差。

4. 客户/项目识别采取“主数据优先 + 候选回退 + 明确选择”的确定性策略。
   `WecomDailyReportIntakeService` 先从 CRM 只读主数据里匹配客户、项目、合同，再结合引号内容、公司后缀、项目关键词提取候选；如果查不到或命中多个候选，系统必须要求用户修正名称或回复“候选1/候选2”这类明确选择。备选方案是自动猜测最像的对象，但在日报和写回场景里风险过高。

5. 日报中的“写入 CRM 跟进记录”只定义桥接时机，不重定义写回规范。
   当用户在日报口述里明确提出“并写入 CRM 跟进记录”时，日报链路可以在实体确认后生成待写回草稿，若命中多个唯一对象则可批量查询和批量确认；但真正的内容确认、写入、失败保留、幂等保护和审计要求，仍复用独立的 `wecom-followup-writeback-confirmation` 能力。备选方案是把日报写回做成一套独立写回器，但这会造成两套写回规则分叉。

## Risks / Trade-offs

- [规则识别可能漏判或误判] -> 通过“先预览后确认”的两段式处理降低误写日报和误写 CRM 的风险。
- [客户与项目名称相似，容易串类] -> 增加公司后缀、项目后缀和关键词判别，并要求多候选时明确选择。
- [多对象批量写回会让用户混淆当前进度] -> 用逐对象摘要和批量写入结果列表明确每个对象的处理状态。
- [日报写回桥接容易与独立写回规格重叠] -> 本变更只定义日报如何进入写回，不复制写回内部规则。
- [会话中途重开主题入口会污染旧上下文] -> 重新输入主题入口时重置日报二期收集状态，强制按新一轮口述重新开始。

## Migration Plan

1. 先将日报二期识别、补问和候选确认规则写入规格，明确它是一期日报的企业微信增强层。
2. 再固化 `wecom-daily-report-intake.service.ts`、`wecom-bot.service.ts` 和提示词配置中的现有实现行为。
3. 最后用集成测试和单测把自由口述、修正名称、多候选选择、负向回答和写回桥接场景锁住。

回滚时只需关闭日报二期主题入口和实体确认流程，回退到一期“手工片段式”日报收集即可，不影响既有日报域数据结构。

## Open Questions

- 明日计划是否需要进一步解析为“拜访时间 + 客户/项目 + 动作”的更细粒度结构，而不是继续保留文本片段。
- 多对象批量写回完成后，是否需要把每条写回结果同步回日报确认摘要中长期保留。
- 当前“公司后缀”和“项目关键词”规则是否还需要覆盖更多行业别名，后续可基于真实口述样本继续补充。
