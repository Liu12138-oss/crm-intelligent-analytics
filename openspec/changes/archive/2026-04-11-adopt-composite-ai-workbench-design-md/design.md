## Context

当前系统已经进入前后端实现阶段，Web 工作台、合同审核、治理审计等页面已经存在基础实现，但仓库级 UI 约束仍以旧版 `.pen` 原型和传统白底蓝灰后台规范为主。用户已明确要求不再参考旧 `.pen`，并要求结合 `awesome-design-md` 中多家大公司的设计方向，综合形成最适合本系统的设计方案，因此本次变更需要新增根级 `DESIGN.md`，并同步修正 README 与 AGENTS 中的设计优先级。

本次变更是文档与规范变更，不直接改造 Vue 页面、后端接口、数据库或运行时配置。它的目标是先把后续 UI 实现的设计输入固定下来，避免后续协作者继续被旧原型限制。

## Goals / Non-Goals

**Goals:**

- 新增一份中文根级 `DESIGN.md`，明确 CRM 智能分析系统的新视觉方向、设计原则、组件规范和交互要求。
- 将设计方向从传统后台升级为综合大型产品设计语言后的企业 AI 分析工作台风格。
- 明确各参考方向的取舍：Cohere 用于 AI 平台感，Sentry / ClickHouse 用于数据可信与可观测性，Linear 用于精确低噪声，Airtable 用于结构化数据操作，Coinbase / IBM 用于可信企业感，Intercom 用于对话式补问和追问。
- 明确旧 `.pen` 和旧 `UI规范.md` 只作为历史参考，不再约束新视觉、新布局和新信息架构。
- 更新 README、AGENTS 和旧 UI 规范说明，确保文档优先级一致。
- 通过 OpenSpec 新增 `ui-design-system` 能力，为后续 UI 改造提供可验收规范。

**Non-Goals:**

- 不在本次变更中重构 `frontend/` 页面或样式。
- 不新增、删除或修改后端 API、数据库表、OpenAPI 契约或 TypeScript 类型。
- 不输出新的 `.pen` 原型；后续需要视觉原型时再单独创建与 `DESIGN.md` 一致的新 `.pen`。
- 不调整一期业务范围、权限模型、CRM 接入策略、企业微信能力或合同审核执行链路。

## Decisions

### 决策 1：采用综合型企业 AI 工作台方向，而不是单一品牌复刻

新设计系统不复刻单一网站，而是综合多套成熟设计方向，形成 CRM 智能分析系统专属风格：Cohere 提供 AI 平台外壳，Sentry / ClickHouse 提供数据密集和可观测性，Linear 提供克制精确，Airtable 提供结构化数据操作，Coinbase / IBM 提供可信企业感，Intercom 提供对话体验，Apple / Vercel 提供留白、排版和质感。

原因：

- 本系统核心价值是 AI 问数、解释、追问和治理，不适合继续呈现为通用后台表格系统。
- 单一 Cohere 风格有 AI 感，但可能偏营销或炫技；单一 IBM/Carbon 稳定但偏传统；综合方案更能同时覆盖 AI、数据、治理、合同和审计。
- 旧原型的视觉风格已经被用户明确否定，继续参考会直接干扰后续设计。

备选方案：

- 继续使用旧 `.pen` 和白底蓝灰规范。问题是视觉平庸，无法支撑 AI 分析工作台定位。
- 只采用 Cohere。优点是 AI 平台感强，缺点是对治理审计、表格密度和企业可信约束不足。
- 只采用 IBM/Carbon。优点是稳定可信，缺点是偏传统企业后台，颠覆性不足。

### 决策 2：根级 DESIGN.md 成为视觉最高入口

后续 UI 视觉、组件、动效、文案语气和页面氛围优先读取根级 `DESIGN.md`。主规格仍负责功能边界、接口字段、权限约束和验收口径。

原因：

- `README.md` 面向仓库使用者，`AGENTS.md` 面向协作者，`specs/` 面向功能规格；这些文件都不适合承载完整视觉语言。
- 根级 `DESIGN.md` 是后续代理和工程师最容易发现的设计输入。

备选方案：

- 只更新 `docs/prototype-ui/UI规范.md`。问题是该文件已经代表旧视觉，继续复用容易混淆。
- 把设计规则分散到 README 和 AGENTS。问题是规则会重复且难以维护。

### 决策 3：旧 `.pen` 降级为历史参考

旧 `.pen` 和旧 UI 规范不再作为视觉、尺寸、布局、卡片结构和一级导航的强约束，只能用于追溯历史页面范围。后续若需要尺寸可读的原型，应按新设计系统重新生成 `.pen`。

原因：

- 用户已明确要求不要参考旧 `.pen`，且允许颠覆式修改设计。
- 如果不修正 AGENTS 和 README，后续代理仍会被旧规则强制引回旧原型。

备选方案：

- 保留旧 `.pen` 的信息架构，只替换颜色和细节。问题是仍会继承旧设计的根本问题。
- 删除旧 `.pen`。问题是会丢失历史参考和已有资料，不利于追溯。

### 决策 4：本次只落文档规范，不改页面实现

本次变更只新增和更新文档，不触碰 `frontend/` 实现代码。后续真正重构 Web UI 时，需要单独 OpenSpec 变更覆盖页面结构、组件改造、测试和验收。

原因：

- 设计系统需要先稳定，避免一边写规范一边改页面导致范围失控。
- 当前请求明确是使用 OpenSpec 提案选择并落地 DESIGN.md，不是立即重构页面。

备选方案：

- 直接重构现有 Vue 页面。问题是范围大，会混入设计系统、交互、样式和组件重构多个决策。

## Risks / Trade-offs

- [旧文档仍引用 `.pen` 可能造成后续误用] → 同步更新 README、AGENTS 和旧 UI 规范开头说明。
- [综合参考过多导致风格松散] → 在 `DESIGN.md` 中明确“AI 经营指挥舱”为主线，并定义每个参考方向只吸收哪些能力。
- [AI 平台风格过于偏营销或炫技] → 在 `DESIGN.md` 中明确数据密度、可读性、审计可信和动效克制原则。
- [深色基底影响表格和合同正文阅读] → 要求表格、合同正文和可复制内容优先使用浅色数据面板承载。
- [仅落文档不改页面，短期内界面仍保持旧样式] → 在 tasks 中明确后续 UI 改造需另立 OpenSpec，避免误判本变更已完成视觉重构。

## Migration Plan

1. 新增 `DESIGN.md`，建立新设计系统入口。
2. 新增 `ui-design-system` OpenSpec 能力，固化设计系统的验收要求。
3. 更新 README、AGENTS 和旧 UI 规范说明，统一文档优先级。
4. 验证 OpenSpec 和 Markdown 格式。
5. 后续页面重构时，以新的 OpenSpec 变更消费 `DESIGN.md`，不在本变更中直接迁移现有 UI。
