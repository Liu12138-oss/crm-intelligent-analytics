## MODIFIED Requirements

### Requirement: CRM业务能力必须优先复用官方 API

涉及 CRM 业务对象读写的新增能力，以及对既有非认证能力的功能改造，系统 MUST 先评估并优先复用 CRM 官方 API。只要官方 API 已满足目标对象、动作、字段、权限语义与稳定性要求，就不得因数据库更熟悉、实现更快或已有旧代码而绕过官方 API。对于企业微信今日跟进聊天链路，系统 MAY 将多轮日报补问收敛为单次确认直写，但不论交互如何简化，都 MUST 继续通过受控 CRM 官方 API 完成跟进写回，不得转为数据库直写或新增未审计的写入路径。

#### Scenario: 新能力命中可用官方 API

- **WHEN** 团队要实现一个新增 CRM 业务能力，且官方 API 已覆盖所需对象与动作
- **THEN** 方案必须以官方 API 作为主接入路径，并将数据库直连排除为默认实现

#### Scenario: 今日跟进聊天链路改为首次确认直写

- **WHEN** 企业微信“今日跟进”入口从日报补问模式调整为首次确认后直接写入 CRM 跟进
- **THEN** 系统仍必须通过既有受控 CRM 官方写回接口完成执行
- **THEN** 系统不得因为减少聊天步骤而跳过对象校验、审计留痕、失败保留或幂等保护

#### Scenario: 存量非认证能力发生改造

- **WHEN** 团队对既有的非认证功能进行需求扩展、重构或缺陷修复
- **THEN** 该变更必须先复核是否已有可用官方 API，并在可用时优先收敛到官方 API 路径
