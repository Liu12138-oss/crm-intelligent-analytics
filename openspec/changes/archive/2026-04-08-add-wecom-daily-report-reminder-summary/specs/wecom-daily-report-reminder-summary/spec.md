## ADDED Requirements

### Requirement: 晚上22点必须按当日 CRM 日报源数据执行个人确认或催报

系统 MUST 将以下当日 CRM 记录视为日报源数据：

- 当前 CRM 用户当天写入的 `revisit_logs` 跟进记录，且对象类型为 `Customer` 或 `Opportunity`
- 当前 CRM 用户当天新增的 `customers` 记录
- 当前 CRM 用户当天新增的 `opportunities` 记录

系统 MUST 在当日 `22:00` 检查这些源数据，并执行以下分支：

- 若某用户截至该时点已经存在日报源数据，系统 MUST 自动总结出该用户的个人日报摘要，并通过企业微信机器人先发送给该用户本人确认。
- 若某用户截至该时点仍没有任何日报源数据，系统 MUST 通过企业微信机器人向该用户本人发送催报提醒。

#### Scenario: 22点已有 CRM 数据时先发送个人日报确认

- **WHEN** 系统在当日 `22:00` 检查某销售用户的日报源数据，发现其当天已经有跟进记录，或有新增客户 / 新增商机记录
- **THEN** 系统必须自动汇总该用户当天的个人日报摘要
- **THEN** 系统必须先把这份日报摘要发送给该用户本人确认，而不是直接跳过个人确认

#### Scenario: 当天没有任何日报源数据时触发 22 点催报

- **WHEN** 系统在当日 `22:00` 检查某销售用户的日报源数据，发现其当天没有任何跟进记录、没有新增客户、也没有新增商机
- **THEN** 系统必须向该销售用户本人发送催报提醒
- **THEN** 系统不得把该提醒直接发送给其负责人

#### Scenario: 当天已有任一类源数据时不得再走无数据催报分支

- **WHEN** 某销售用户在 `22:00` 前已经产生至少一条当天跟进记录，或至少一条当天新增客户 / 新增商机记录
- **THEN** 系统不得再把该用户判定为“未报”
- **THEN** 系统必须进入“自动总结个人日报并发本人确认”的分支

### Requirement: 结构化跟进记录必须被解析为日报字段并保留更新人信息

系统 MUST 优先解析企业微信结构化跟进写回内容中的 `跟进内容`、`遇到与协助`、`信息共享`、`拜访计划` 四类字段，并将更新人姓名一并纳入日报摘要。若某条跟进记录不是结构化模板，但仍属于当天有效跟进记录，系统 MUST 至少保留该条记录的全文摘要，不得直接丢弃。

#### Scenario: 结构化模板跟进内容被拆回日报字段

- **WHEN** 系统读取到一条由企业微信机器人写入 CRM 的当天跟进记录，且内容中包含 `跟进内容`、`遇到与协助`、`信息共享`、`拜访计划` 等结构标签
- **THEN** 系统必须按字段拆解该记录并纳入个人日报摘要
- **THEN** 系统必须保留更新人姓名，避免团队汇总时丢失“谁更新了这条记录”

#### Scenario: 非模板化历史跟进记录仍需保留摘要

- **WHEN** 系统读取到一条有效的当天跟进记录，但其内容未命中结构化模板标签
- **THEN** 系统必须至少保留该条跟进记录的全文摘要
- **THEN** 系统不得因为格式不标准而直接把该条记录排除在日报之外

### Requirement: 当天新增客户与当天新增商机必须纳入日报汇总

系统 MUST 将当前 CRM 用户当天新增的客户和当天新增的商机纳入个人日报与团队汇总。系统 MUST 在“客户 / 商机变化”维度展示新增情况，不得只依赖跟进记录生成日报。

#### Scenario: 当天新增客户必须出现在日报变化摘要中

- **WHEN** 某销售用户当天通过受控流程新增了一条或多条客户记录
- **THEN** 系统必须把这些新增客户记录纳入该用户的日报摘要
- **THEN** 系统不得因为该用户没有新增跟进记录就忽略这些新增客户动作

#### Scenario: 当天新增商机必须出现在日报变化摘要中

- **WHEN** 某销售用户当天通过受控流程新增了一条或多条商机记录
- **THEN** 系统必须把这些新增商机记录纳入该用户的日报摘要
- **THEN** 系统不得把新增商机排除在团队汇总之外

### Requirement: 个人日报必须由系统基于 CRM 源数据自动生成

系统 MUST 基于同一用户同一业务日期下的当天跟进记录、当天新增客户、当天新增商机自动生成个人日报摘要，而不是继续依赖“手工日报片段已确认”作为主路径。系统 MUST 支持将同一天的多条跟进、多条新增客户、多条新增商机合并成一份个人日报摘要。

#### Scenario: 同一天多条业务记录合并为一份个人日报

- **WHEN** 某用户在同一天内产生多条跟进记录、并新增多条客户或商机
- **THEN** 系统必须把这些记录合并为一份个人日报摘要
- **THEN** 系统不得把同一天的每条业务记录拆成多份彼此独立的日报

#### Scenario: 日报生成不再依赖人工确认状态

- **WHEN** 系统准备判断某用户当天是否已具备日报内容，或准备生成次日团队汇总
- **THEN** 系统必须以 CRM 源数据与自动摘要结果为准
- **THEN** 系统不得继续把“日报草稿是否确认”作为主判断条件

#### Scenario: 晚上22点生成的个人日报必须可供本人确认

- **WHEN** 系统在 `22:00` 基于当天 CRM 源数据生成某用户的个人日报摘要
- **THEN** 该摘要必须作为一条待本人确认的日报消息发送给该用户
- **THEN** 系统不得只在后台生成摘要而不给用户可见确认入口

### Requirement: 次日团队汇总必须按负责人聚合个人摘要与缺报名单

系统 MUST 在次日 `08:00` 按负责人聚合前一日成员的个人日报摘要，并单独列出“当天没有任何日报源数据”的成员名单。销售团队 MUST 优先按历史静态负责人表中“所属区域 -> 销售负责人 -> 销售成员”的映射发送；售前团队若映射来源仍未补齐，系统 MUST 阻断售前真实发送并记录原因。

#### Scenario: 销售团队汇总按负责人聚合发送

- **WHEN** 系统在次日 `08:00` 生成销售团队汇总并解析接收人
- **THEN** 系统必须按历史静态负责人表中的既有负责人映射聚合成员日报并发送给对应销售负责人
- **THEN** 系统不得绕过该映射把所有团队汇总统一发送给单一管理员

#### Scenario: 团队汇总必须附带无源数据名单

- **WHEN** 某些成员在前一日没有任何日报源数据
- **THEN** 系统必须在团队汇总中单独列出这些成员名单
- **THEN** 系统不得把这些成员与“已有日报摘要”的成员混在同一列表中

#### Scenario: 售前映射未补齐时阻断售前真实发送

- **WHEN** 系统需要为售前团队执行真实汇总发送，但当前缺少可靠的售前负责人映射来源
- **THEN** 系统必须阻断该次售前真实发送
- **THEN** 系统必须记录“售前负责人映射未补齐”的原因，等待映射补齐后再开启

### Requirement: 日报通知必须复用主动通知底座并固定走机器人通道

系统 MUST 复用 `ProactiveNotificationService` 发送日报催报与团队汇总，并固定路由到企业微信机器人通道。系统 MUST 复用 `WECOM_NOTIFY_REAL_MESSAGE_ENABLED` 与 `WECOM_NOTIFY_TEST_USER_ID` 控制真实发送与测试改投，不得在日报模块内写死测试接收人，也不得改回企业微信应用消息通道。

#### Scenario: 真实发送开关关闭时改投配置中的测试接收人

- **WHEN** 日报催报或团队汇总准备发送，但 `WECOM_NOTIFY_REAL_MESSAGE_ENABLED=false`
- **THEN** 系统必须阻止向真实业务对象发送
- **THEN** 系统必须把消息改投到 `WECOM_NOTIFY_TEST_USER_ID` 指定的测试接收人，而不是在日报模块内写死单独名单

#### Scenario: 即使传入应用通道偏好也必须固定走机器人

- **WHEN** 某条日报通知任务传入了企业微信应用消息通道偏好，或历史实现仍保留应用消息分支
- **THEN** 主动通知底座必须忽略该偏好并固定选择企业微信机器人通道
- **THEN** 日报实现不得再把企业微信应用作为用户可见的日报通知入口

### Requirement: 晚上22点发送给个人的日报消息和催报消息必须使用友好亲切鼓励的中文文案

系统 MUST 对 `22:00` 发送给个人的日报确认消息和无数据催报消息使用友好、亲切、鼓励式中文文案。系统 MUST 让用户感知到“系统在帮助其完成当日收口”，而不是只给出生硬、命令式催促。系统 MAY 明确下一步动作，但不得使用冷硬责备语气。

#### Scenario: 已有数据时的个人确认消息使用友好总结语气

- **WHEN** 系统在 `22:00` 向某用户发送当天自动总结出的个人日报摘要
- **THEN** 消息文案必须以友好、亲切的方式说明“我先帮你整理好了今天的日报，请你确认一下是否准确”之类的含义
- **THEN** 系统不得只发送生硬的系统通知标题或机械式字段堆砌

#### Scenario: 无数据时的催报消息使用鼓励式提醒语气

- **WHEN** 系统在 `22:00` 发现某用户当天仍没有任何日报源数据，需要发送催报提醒
- **THEN** 消息文案必须以友好、鼓励的方式提醒用户尽量在当天补齐 CRM 数据
- **THEN** 系统不得使用带责备、施压或不友好的催促语气
