## Context

当前仓库里与日报相关的实现已经出现明显分层变化：

- `daily-report` 模块里还保留着“日报片段 / 草稿 / 确认 / 汇总”的早期模型。
- 主动通知底座已经完成“统一任务模型 + 固定机器人路由 + 配置驱动测试改投”的收敛。
- 企业微信跟进写回已经升级为结构化模板，生成的 CRM 跟进内容统一携带更新人姓名，并可稳定拆回 `跟进内容`、`遇到与协助`、`信息共享`、`拜访计划` 四类字段。
- 企业微信新增客户、新增商机链路已经可以在 CRM 中留下当天新增记录。

因此，这条变更不应再把“日报是否存在”理解为“用户有没有在日报模块里补完五段草稿”，而应理解为“系统能否从当天 CRM 业务记录中自动还原出这名用户的日报”。

## Goals / Non-Goals

**Goals:**

- 明确日报源数据来自当天 CRM 跟进记录、当天新增客户、当天新增商机。
- 明确如何从结构化跟进内容中提取日报字段并保留更新人信息。
- 明确 `22:00` 需要做分支处理：有源数据则自动生成个人日报并发本人确认，无源数据则发送友好催报；次日 `08:00` 再生成团队汇总。
- 明确日报相关通知必须复用当前主动通知底座，不在日报模块内重复实现通知安全控制。
- 保留销售负责人映射与团队汇总约束，并为售前映射缺失时的阻断行为留出位置。

**Non-Goals:**

- 不在本提案中继续扩展“手工日报片段录入”作为主路径。
- 不重新定义客户 / 商机 / 跟进写回本身的 CRM API 规则。
- 不在日报模块内重复实现企业微信应用消息、测试改投或真实发送开关。
- 不在当前提案中新增新的用户通知入口或新的一级导航结构。

## Decisions

1. 日报以“CRM 当日源数据”而不是“日报草稿状态”作为唯一业务事实来源。

   22:00 是否生成个人摘要确认、是否催报，以及次日 08:00 是否汇总，都必须依据当天 CRM 里是否已经存在可归属到当前用户的日报源数据。这里的目标是避免日报模块与 CRM 实际动作脱节，也避免出现“草稿填了但业务没落库”或“业务已落库但日报仍被判未提交”的双口径问题。备选方案是继续沿用“日报草稿未确认”作为催报条件，但这已经与现有写回 / 创建能力不一致。

2. 日报源数据固定为三类：跟进记录、当天新增客户、当天新增商机。

   首版自动日报只消费以下来源：
   - 当天由当前 CRM 用户写入的 `revisit_logs` 跟进记录，且对象类型为 `Customer` 或 `Opportunity`
   - 当天由当前 CRM 用户新增的 `customers` 记录
   - 当天由当前 CRM 用户新增的 `opportunities` 记录

   这样可以与现有机器人能力直接闭环，不需要再额外发明新的“日报输入表”。备选方案是再引入独立日报落库入口，但那会导致同一业务动作在 CRM 与日报域出现重复写入。

3. 跟进记录优先按结构化模板提取四类日报字段，并保留更新人姓名。

   企业微信跟进写回内容已经统一为带签名、带结构标签的文本。日报生成时必须优先提取：
   - `跟进内容`
   - `遇到与协助`
   - `信息共享`
   - `拜访计划`

   同时保留更新人姓名，用于个人日报和团队汇总展示“谁更新了什么”。若遇到历史或非模板化跟进内容，允许回退为“整段跟进摘要”，但不得直接丢弃该记录。备选方案是只按全文摘要处理，不解析结构字段，但这会损失现有优化后的业务价值。

4. 当天新增客户 / 新增商机必须进入个人日报摘要。

   企业微信新增客户、企业微信新增商机已经是当天业务动作的一部分，日报生成必须把它们纳入“客户 / 商机变化”维度，而不是只依赖跟进记录。团队汇总也必须能看见成员当天新增了哪些客户或商机、分别有多少条。备选方案是把新增记录排除在日报外，但这会让“当天有建档动作却被提醒未报”变成错误结论。

5. `22:00` 改为“有数据发个人摘要确认，无数据发友好催报”的双分支处理。

   当系统在 `22:00` 执行检查时，如果某用户截至该时点已经存在任一类当天日报源数据，则系统必须立即自动总结个人日报摘要，并先发给该用户本人确认；如果截至该时点仍没有任何当天日报源数据，则才向该用户本人发送友好催报。这样既能让“已有业务动作的人”先看到系统归纳结果，也能把“尚无业务动作的人”单独拉出来提醒。备选方案是只做无数据催报、不主动发个人摘要，但那样会错过当天收口窗口。

6. `22:00` 个人确认消息与催报消息都必须使用友好、亲切、鼓励式文案。

   日报提醒是高频动作，如果文案过于生硬，容易让用户感到被追责而不是被协助。无论是“你今天已有 CRM 数据，我先帮你整理好了日报，请你看一下是否准确”还是“今天还没同步到 CRM 记录，我先提醒你一下，辛苦抽空补一下”，都应保持友好、亲切、鼓励式语气。备选方案是只输出系统化硬提示，但这与当前业务期望相反。

7. 次日 `08:00` 团队汇总按“个人摘要 + 缺报名单”生成，而不是按日报确认数汇总。

   团队汇总的主体内容应当是：
   - 每位成员的自动生成日报摘要
   - 当天跟进、当天新增客户、当天新增商机的重点概览
   - 没有任何日报源数据的成员名单
   - 如后续仍保留“摘要生成失败”场景，则单独列出异常名单

   不再以“已确认 / 迟交 / 未交”作为唯一指标。备选方案是继续按确认数做统计，但那已经不能反映自动日报真实状态。

8. 日报通知必须直接复用主动通知底座的机器人通道与配置驱动测试改投。

   日报场景不得再自己维护“是否真实发送”“默认改投给谁”的分支逻辑。实现上必须复用：
   - `ProactiveNotificationService`
   - 固定机器人路由 `WECOM_BOT_MESSAGE`
   - `WECOM_NOTIFY_REAL_MESSAGE_ENABLED`
   - `WECOM_NOTIFY_TEST_USER_ID`

   `WangLiang02` 只应被视为当前运行时默认回退值，而不是日报提案中的写死业务规则。备选方案是在日报模块里单独写一套测试改投逻辑，但这会和现有底座治理冲突。

9. 销售团队仍按既有负责人映射汇总，售前映射未补齐时继续阻断真实发送。

   销售团队仍优先按历史静态负责人表中“所属区域 -> 销售负责人 -> 销售成员”的映射聚合日报。售前如需进入团队汇总，也应遵循同等模式；若缺少可靠映射来源，则必须阻断售前真实发送并记录原因，而不能手工写死负责人。该约束与旧提案一致，但它现在服务的是“自动日报汇总”而不是“草稿确认汇总”。

## Risks / Trade-offs

- [CRM 表字段与创建 / 更新归属字段可能不统一] -> 在落地阶段先明确 `revisit_logs`、`customers`、`opportunities` 对应的“当前用户归属字段”和时间字段，避免查询误归属。
- [部分历史跟进记录不是结构化模板] -> 优先解析模板字段，解析失败时保留整段摘要作为降级方案，不因格式不标准而整条丢失。
- [自动日报生成后，旧的“已确认 / 未确认”口径会与新逻辑冲突] -> 本提案显式把 `22:00` 口径切换为“有数据先发个人摘要确认、无数据再催报”，并把团队汇总口径切换为“源数据是否存在、摘要是否生成”，不再把旧确认状态作为主判断条件。
- [测试改投如果仍写死到日报模块，会与通知底座配置不一致] -> 提案中明确要求只复用通知底座配置，不接受日报模块私有测试名单。
- [售前负责人映射尚未补齐] -> 继续保留“规则先固化、真实发送先阻断”的策略，避免临时手工名单污染后续实现。

## Migration Plan

1. 先更新本变更的 proposal / design / spec / tasks，把日报口径改为“CRM 源数据自动汇总”。
2. 在实现层把日报相关查询能力补齐，确保能按用户和业务日期读取当天跟进、当天新增客户、当天新增商机。
3. 将提醒与汇总逻辑切换到主动通知底座，移除日报模块中关于旧通道和写死测试接收人的假设。
4. 在测试与运维文档中补齐“结构化跟进内容提取、自动日报摘要生成、配置驱动测试改投”的回归场景。

## Open Questions

- 次日 `08:00` 团队汇总展示到什么粒度最合适：只展示摘要，还是同时透出关键对象清单与数量。
- 如果用户当天只有新增客户 / 新增商机、没有跟进记录，个人日报是否需要生成固定文案的“今日跟进为空”段落。
- 售前负责人映射的最终来源是继续使用 Excel，还是切换为 CRM / 主数据关系表。
