## Context

当前仓库的 AI Profile 治理后端已经保留了 `reasoningEffort` 字段，运行时解析对象和部分业务链路也会读取该字段，但治理后台的新增/编辑抽屉没有把它作为正式配置项暴露出来，导致管理员无法在页面中显式设定推理等级。现状还存在三个具体问题：

- 新建 Profile 时无法声明默认推理档位，历史空值会继续依赖各调用链自己的兜底逻辑；
- 环境默认档案引导时，若本地未配置显式推理等级，会继续落成空值，默认行为分散在不同运行时实现中；
- Codex 主链已有明确的推理档位映射，Claude 侧当前没有等价的显式透传口径，治理字段与 Provider 实际消费能力不透明。

这次变更需要在不改动全局治理模型结构的前提下，把推理等级补齐为正式可治理字段，并把“缺省即最低档”的策略收敛到统一入口，避免继续由各处 fallback 自行决定默认值。

## Goals / Non-Goals

**Goals:**

- 让管理员可以在 AI 模型治理页新增、编辑和查看 Profile 的推理等级。
- 为新建 Profile、历史空值 Profile、复制 Profile 的缺失字段、环境默认档案引导和环境兜底解析统一补齐最低推理等级。
- 保持现有 `reasoningEffort` 数据结构不变，优先通过前后端默认值和运行时归一化实现一致行为，而不是引入额外字段。
- 明确不同 Provider 对推理等级的实际消费边界，避免治理字段存在但行为不透明。

**Non-Goals:**

- 不在本次提案中扩展新的 AI Provider 类型。
- 不引入按业务线分别配置不同推理等级的多活路由能力。
- 不把 prompt 优化、超时策略或 service tier 变更并入本次改造。
- 不要求一次性批量迁移全部历史 Profile 数据，只要求运行时和编辑流程对空值给出一致默认。

## Decisions

### 决策 1：沿用现有 `reasoningEffort` 通用字段，不新增 SDK 专属推理字段

`AiModelProfileRecord`、治理 schema、运行时解析对象里已经存在 `reasoningEffort`。本次不再新增 `codexReasoningEffort`、`claudeReasoningEffort` 这类 SDK 专属字段，而是继续把推理等级维持为 Profile 顶层通用字段。

原因：

- 现有后端领域模型已经具备该字段，继续拆分只会制造冗余映射；
- 推理等级属于管理员理解成本较低的通用治理概念，放在 Profile 顶层比藏进 `sdkOptions` 更清晰；
- 这能把本次改造集中在“前端暴露 + 默认值统一 + Provider 消费说明”，避免数据模型震荡。

备选方案：

- 把推理等级拆进 `sdkOptions`。问题是会和现有顶层字段重复，并让前端表单与运行时解析出现双来源。

### 决策 2：统一使用 `minimal` 作为缺省推理等级，不依赖各调用链自行回退

本次提案将“默认值都为最低等级”定义为统一治理规则：

- 新增 Profile 且管理员未显式选择时，保存值为 `minimal`；
- 编辑历史空值 Profile 时，表单回显默认为 `minimal`；
- 复制 Profile 时若源值为空，则复制结果写入 `minimal`；
- 环境默认档案 bootstrap 时若缺少显式推理等级，则落表为 `minimal`；
- 当后台没有激活 Profile、系统回退到环境默认运行时时，若环境侧未提供推理等级，运行时也按 `minimal` 解析。

原因：

- 用户已经明确要求默认最低等级以优先换取速度；
- 当前 `AiGatewayService` 的兜底是 `medium`，而其它调用链未必一致，默认值分散会导致治理页面显示与真实运行不一致；
- 用统一缺省值取代链路内部分散 fallback，能让后续测试、审计和操作说明都更稳定。

备选方案：

- 继续保留链路内各自 fallback。问题是治理页面显示为空，实际运行却由代码偷偷回退，管理员难以判断真实生效值。

### 决策 3：前端治理抽屉增加显式选择器，并对历史空值做表单级兼容

`AiProfileFormDrawer.vue` 新增“推理等级”选择器，使用现有枚举集合：

- `minimal`
- `low`
- `medium`
- `high`
- `xhigh`

默认选中 `minimal`。对已有 Profile：

- 若记录里已有 `reasoningEffort`，抽屉回显该值；
- 若记录为空，抽屉回显 `minimal`，保存后写回实体。

同时，列表、摘要卡和测试/详情反馈中应展示当前推理等级，避免字段只在抽屉中出现。

原因：

- 这是当前用户可见的直接缺口；
- 表单级回显兼容能让历史空值在不做强制迁移的情况下逐步收敛。

备选方案：

- 只在后端偷偷补默认值，不改前端。问题是管理员依然无法编辑，也看不到当前实际档位。

### 决策 4：Provider 消费采用“已支持的真实透传 + 未支持的保留配置”策略

Codex 侧已有 `minimal | low | medium | high | xhigh` 的明确映射与调用路径，本次实现应继续走真实透传；环境兜底和激活 Profile 场景都要统一遵守 `minimal` 缺省规则。

Claude 侧在当前本地 SDK 代码中未发现显式 reasoning 参数入口，因此提案要求：

- 治理层仍允许配置和持久化 `reasoningEffort`；
- 运行时解析对象继续保留该字段；
- 实现阶段先核对当前 Claude SDK 是否存在正式透传入口；
- 若 SDK 不支持显式档位，则在文档和界面提示中说明“当前仅保留治理配置，不改变 Claude SDK 默认行为”。

原因：

- 不能在提案阶段假设不存在的 SDK 能力；
- 但也不能因为 Claude 当前未透传就把整个治理字段只做成 Codex 专属。

备选方案：

- 只给 Codex 暴露推理等级。问题是会把通用治理字段再次做成 Provider 特判，后续扩展更乱。

### 决策 5：不做一次性数据迁移，优先通过服务层与解析层归一化收口

本次不要求立即把应用存储中所有历史空值 Profile 批量改写。统一归一化入口为：

- `AiModelProfileService.create/update/copy`
- `AiModelEnvBootstrapService`
- `LocalRuntimeConfigService` / `AiRuntimeConfigResolver`
- 前端表单初始化

这样可以保证：

- 历史数据无需先迁移也能立即按 `minimal` 生效；
- 管理员一旦编辑或复制旧 Profile，空值会自然被收敛；
- 变更风险低于一次性扫描并重写本地应用存储。

备选方案：

- 启动时批量迁移全部历史 Profile。问题是对本地持久化文件侵入更大，失败恢复也更复杂。

## Risks / Trade-offs

- [Claude SDK 当前可能不支持显式推理等级] → 实现时先验证 SDK 能力；若不支持，则保留治理字段并补充清晰文档与界面说明。
- [历史空值 Profile 仍然存在于存储中] → 通过表单回显、服务层写入和运行时解析统一兜底到 `minimal`，保证行为先一致，再逐步收敛数据。
- [前端新增字段后，未同步列表/摘要展示会造成“可编辑但不可见”] → 任务中明确要求同步更新抽屉、列表、摘要和测试反馈。
- [环境兜底默认值从 `medium` 调整到 `minimal` 可能影响部分既有质量感知] → 在 quickstart 与治理说明中明确“默认优先速度”的产品策略，并保留管理员手工调高档位的能力。

## Migration Plan

1. 先补 OpenSpec 规格增量，明确治理页可编辑、缺省最低等级和环境引导补齐规则。
2. 改造前端治理抽屉与类型定义，新增推理等级选择器和默认值回显。
3. 改造后端保存、复制、bootstrap 和运行时解析逻辑，把空值统一归一化为 `minimal`。
4. 核对 Codex / Claude Provider 对推理等级的实际消费路径，能透传的路径补齐真实透传，不能透传的路径补齐文档与界面说明。
5. 补齐回归测试与文档，覆盖新建、编辑、复制、环境默认档案和环境兜底路径。

回滚策略：

- 若前端表单或后端归一化逻辑出现问题，可回退本次治理页字段改动与运行时默认值修改；
- 因本次不做强制数据迁移，回滚不需要恢复额外的批量数据变更。

## Open Questions

- Claude SDK 当前版本是否存在正式的 reasoning / thinking 控制参数；若存在，实现阶段应优先走正式参数而不是自定义约定。
- 是否需要在治理列表中直接展示“该 Provider 当前是否实际消费推理等级”的说明标签；当前建议至少在表单帮助文案或操作说明中写清。
