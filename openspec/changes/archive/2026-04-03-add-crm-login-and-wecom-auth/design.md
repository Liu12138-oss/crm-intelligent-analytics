## Context

当前 Web 端没有正式登录页，路由默认直接进入 `/analysis`，后端 `SessionAuthGuard` 主要依赖 `x-crm-user-id` 头或简单的用户标识 Cookie 直接装载用户，上线后既不安全，也无法表达真实的登录态生命周期。企业微信入口目前只做测试签名与发送者映射，尚未把“消息来源可信、用户绑定有效、会话可用、审计可追踪”收成一条正式认证链路。

外部 CRM Open API 文档显示，开放接口当前已提供账号密码换取用户 token 的标准方式：`POST /api/v2/auth/login`，请求体包含 `device=open_api`、`login`、`password`、`corp_id`，成功后返回 `user_token`；后续接口调用需要在 `Authorization` 头中附带 `Token token=..., device=open_api, version_code=9.9.9`。文档同时提示企业级调用限制为每企业 `10 次 / 10 秒`，因此认证适配层需要集中封装，避免前端和多个后端模块各自直连造成风控与限流失控。

UI 原型 `docs/prototype-ui/crm-intelligent-analytics-ui.pen` 已包含“登录页”画板，结构是左侧品牌/能力说明区 + 右侧 `560px` 登录卡片，并在卡片内提供“账号登录”和“企业微信登录”两个切换入口。这意味着本次设计不能自行扩展新信息架构，而要以现有原型为骨架补齐真实认证流程。

## Goals / Non-Goals

**Goals：**

- 建立 Web 正式登录入口，支持账号密码登录和企业微信扫码登录。
- 通过后端统一对接 CRM 认证接口，隐藏外部 `user_token`、`corp_id` 和请求头细节。
- 让 Web 登录态、企业微信消息发送者映射和现有分析能力共享同一套用户上下文装载逻辑。
- 在企业微信机器人进入分析编排前完成验签、来源校验、用户映射和失败审计。
- 保持现有工作台、治理页和审计页的业务能力不变，只替换认证入口和会话载体。

**Non-Goals：**

- 不在本次设计中扩展 CRM 业务写入、注册、找回密码或账号绑定后台。
- 不重新设计工作台主框架、治理页结构或企业微信会话展示样式。
- 不把外部 CRM 的全部用户管理接口镜像到本项目，只引入登录、会话校验与最小必要的用户信息装载。
- 不承诺在本次变更中实现 SSO 到其他外部系统。

## Decisions

### 决策一：采用“后端认证适配器 + 本地会话”模式，而不是前端直连 CRM Open API

系统新增独立的 CRM 认证适配服务，由后端统一调用 CRM 登录接口、持有 `user_token`、换取当前用户上下文，并为浏览器下发本地受控会话 Cookie。浏览器只感知本系统登录态，不直接保存 CRM Open API 的 token。

这样做的原因：

- 可以把 `corp_id`、`device=open_api`、`version_code=9.9.9` 等外部协议细节收敛在后端，避免前端泄漏实现细节。
- 可以在本地会话层增加审计、续期、失效、登出和风控策略，而不受前端实现方式影响。
- 可以让 Web 登录、企业微信扫码登录、企业微信机器人消息映射最终都落到同一套 `crmUser` 解析流程。

备选方案是前端直接请求 CRM `/api/v2/auth/login` 并把返回 token 存在浏览器。该方案会把敏感 token 暴露到浏览器、增加跨域和泄漏风险，也无法自然统一企业微信入口，因此不采用。

### 决策二：把现有 `SessionAuthGuard` 升级为“会话解析器优先，测试头为降级能力”

当前 Guard 直接把 `x-crm-user-id` 或 Cookie 中的用户标识当作真实身份，这只适合联调。变更后应新增会话仓储或签名会话载体，Guard 先解析本地会话，再装载 CRM 用户；`x-crm-user-id` 仅保留给测试或本地开发环境，不作为生产默认入口。

这样做的原因：

- 登录态需要具备会话 ID、创建时间、过期时间、来源渠道、最近活跃时间等元数据，简单用户 ID Cookie 无法表达。
- 后续扫码登录、登出、续期、失效踢下线都需要可管理的会话实体。
- 生产环境应避免“只知道用户 ID 就能伪造身份”的弱校验模式。

备选方案是继续沿用现有 Guard，只把 Cookie 名字改得更像 session。该方案没有真正的会话完整性和失效控制，因此不采用。

### 决策三：账号密码登录与企业微信扫码登录共用同一会话落地结果，但入口流程分离

账号密码登录流程由登录页直接提交账号、密码和企业 `corp_id` 到本地后端，再由后端调用 CRM 登录接口换取用户 token；企业微信扫码登录则采用“扫码发起 -> 外部企微授权回调 -> 本地映射 CRM 用户 -> 建立本地会话”的路径。两条路径最终都必须产出同样的本地会话对象与能力初始化结果。

这样做的原因：

- 登录态统一后，工作台、治理页和审计页无需关心用户通过哪种方式进入。
- 企业微信扫码登录具体协议仍存在外部依赖不确定性，但其最终目标和成功态是明确的，可以先与本地会话解耦。
- 登录失败、无权限、未绑定、已过期等页面反馈可以复用统一状态模型。

备选方案是为账号密码登录和扫码登录分别维护两套用户上下文与 Guard。该方案会导致权限继承、登出、审计和会话刷新逻辑分叉，因此不采用。

### 决策四：企业微信机器人消息认证链路按“四段式”收口

机器人消息进入分析服务前，必须依次完成：

1. 签名有效性校验；
2. 消息来源与消息结构校验；
3. 发送者到 CRM 用户的绑定映射校验；
4. 用户当前可用性与分析能力权限校验。

任一环节失败，都不创建正式分析请求，只返回对应的拒绝结果，并记录审计事件。

这样做的原因：

- 当前 `test-signature` 只适用于本地演示，不能支撑真实环境的来源可信判断。
- 机器人消息天然缺少浏览器式会话上下文，必须把认证结论前置到进入分析链路之前。
- 未绑定企微账号、停用账号、权限被收回等情况都应在入口就被拦截，而不是流入分析模块后再失败。

备选方案是沿用现有轻校验，把失败情况交给分析服务兜底。该方案会污染审计口径，也会放大越权和假请求风险，因此不采用。

### 决策五：登录页严格复用现有原型结构，不新增一级导航或额外步骤页

登录页沿用 `.pen` 中的左右分栏结构：左侧品牌与能力说明区展示价值信息，右侧卡片展示双登录入口。账号密码模式展示账号、密码输入与主按钮；企业微信模式展示扫码引导、登录说明和失败状态。错误提示、加载状态和条款说明均在卡片内完成，不再扩展独立中间页。

这样做的原因：

- 仓库协作规范已明确 UI 必须以 `.pen` 为准，不能脱离原型擅自发散。
- 登录页只是认证入口，不应引入多余页面跳转干扰主链路。
- 原型已经提供了视觉层级与文案骨架，开发只需要补齐状态与交互。

备选方案是新增全屏向导式登录流程。该方案与原型冲突，也会增加前端实现和验收成本，因此不采用。

### 决策六：外部 CRM 认证调用集中限流并分层缓存最小必要信息

由于 CRM Open API 有企业级调用频率限制，后端认证适配层应对高频操作进行保护：

- 登录只在用户主动提交时调用外部接口，不在前端输入过程中预校验；
- 已建立本地会话后，页面访问优先读取本地会话中的最小身份快照；
- 仅在必要时刷新 CRM 用户详情或权限快照，避免每次页面请求都重新登录 CRM。

这样做的原因：

- 登录页和工作台初始化本身都可能触发多次请求，若不集中收口，容易碰到外部限流。
- 对本地会话做最小快照可以减少外部依赖波动对页面访问的影响。

备选方案是每次接口请求都重新调用 CRM 验证 token 是否有效。该方案延迟和调用量都不可接受，因此不采用。

## Risks / Trade-offs

- [企业微信扫码登录的外部协议与回调细节尚未在当前仓库内明确] → 先把扫码登录抽象为独立适配器接口，并把实际授权协议列入开放问题，在实现前对齐。
- [外部 CRM token 失效策略与本地会话生命周期可能不一致] → 本地会话增加失效检测与主动登出处理，遇到 CRM 返回“帐号已在其他地方登录/已下线”时立即清理本地会话并要求重新登录。
- [登录链路接入后，现有联调方式会发生变化] → 保留测试头模式给 `test` / 本地环境使用，但在生产配置下默认关闭。
- [多企业场景下 `corp_id` 选择不明确会影响账号密码登录成功率] → 优先支持固定企业 `corp_id` 配置，并保留多企业选择扩展位。
- [机器人入口认证变严格后，现有测试脚本可能失效] → 为测试环境保留可配置签名与发送者映射桩，并补充新的集成测试夹具。

## Migration Plan

1. 先新增 CRM 认证适配层、会话实体与登录接口，但暂时不替换现有分析接口的鉴权方式。
2. 在前端增加登录页与路由守卫，完成账号密码登录与登录后跳转工作台的主链路。
3. 将 `SessionAuthGuard` 切换到“本地会话优先”模式，同时保留测试环境 `x-crm-user-id` 降级能力。
4. 为企业微信机器人入口补齐新的认证链路与失败审计，再把旧的测试验签逻辑切到配置化。
5. 完成回归与联调后，再移除生产环境对临时身份注入的依赖。

回退策略：

- 若 Web 登录链路出现问题，可先临时恢复仅测试环境可用的用户头注入方式，不影响开发联调。
- 若企业微信新认证链路导致入口不可用，可先回退到旧的发送者映射逻辑，但必须保留失败审计与风险提示。

## Open Questions

- 企业微信扫码登录最终采用哪种授权协议与回调地址：复用现有 CRM 企微版本登录能力，还是走单独的企业微信 OAuth / SSO 接入？
- 多企业用户登录时，`corp_id` 是由系统固定配置，还是需要在登录页显式让用户选择企业？
- 除了登录成功后的用户 ID 之外，是否还需要同步头像、部门展示名、角色展示名等字段到本地会话快照？
- 企业微信机器人消息认证失败时，是否需要回传更细的用户提示文案，还是统一返回“认证失败，请联系管理员”？
