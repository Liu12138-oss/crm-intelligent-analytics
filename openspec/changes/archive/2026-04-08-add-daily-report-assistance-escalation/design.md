## Context

当前仓库已经具备三条与本变更直接相关的基础能力：

- 日报能力正在从“手工片段收口”切换到“基于 CRM 当日数据自动汇总并对上发送”。
- 企业微信跟进写回已经保留 `跟进内容`、`遇到与协助`、`信息共享`、`拜访计划` 等结构化字段，日报可以直接复用这些语义。
- 主动通知底座已经统一收敛到 `ProactiveNotificationService` 和企业微信机器人通道，并要求发送前执行接收人解析、权限校验、幂等、有限重试和审计留痕。

但当前“遇到与协助”仍主要停留在日报阅读层。主管能在日报里看到问题，却还需要自己二次转发给被协助人和被协助人的主管。这个链路既慢，也不稳定，而且协助转发行为没有统一审计。

## Goals / Non-Goals

**Goals:**

- 把日报里的“需要某人协助”从被动阅读信息升级为正式协助通知。
- 将触发点绑定到日报正式对上发送给主管或负责人这一节点，避免在草稿、预览或本人确认阶段误触发。
- 只对可唯一识别的内部协助人发送通知，并同步通知该协助人的当前主管。
- 对同一份日报内同一协助人的多条事项进行聚合和目标去重，减少消息噪音。
- 保证协助通知失败不阻断主管侧日报送达，同时留下可追踪审计记录。

**Non-Goals:**

- 不在本提案中重定义日报的 22:00 本人确认、08:00 团队汇总或日报正文生成规则。
- 不引入新的人工维护协助人别名表、手工主管名单或独立权限模型。
- 不尝试对所有自由文本做高风险的人名猜测；模糊表述优先不直发。
- 不新增企业微信应用消息、邮件或短信等新通道，仍统一走企业微信机器人主动通知。

## Decisions

1. 协助升级只挂在“日报正式对上发送”事件，不挂在 22:00 本人确认草稿。

   当日报只是生成给本人确认的草稿时，内容仍可能继续修改，也可能最终不对上发送。若过早把协助事项通知给协助人，会制造误报和重复沟通。因此本能力只在日报正式发给主管或负责人时触发，无论该正式送达来自“确认后立即推送主管”还是“团队汇总批次对上发送”，都视为同一类出站事件。备选方案是在 22:00 自动摘要生成后立即发协助通知，但会把未确认草稿误当成正式协作请求。

2. 协助事项只从 `HELP_REQUIRED` / `structuredHelpNeeded` 里抽取，并要求出现可唯一识别的明确协助人。

   现有日报与跟进模板已经把“问题与协助”聚焦在单独字段中，这比从整份日报做泛化人名识别更稳。首版只接受“需要李四协助”“请王五支持”等能唯一映射到内部用户的明确表述；像“需要产品同事支持”“需要领导协助”这类角色化或歧义表述只保留在主管日报里，不自动直发。备选方案是对整份日报或任意文本做人名抽取，但误识别和错发风险过高。

3. 协助人及其主管必须复用当前身份映射和组织关系实时解析。

   协助人解析必须优先复用当前 CRM 用户、企业微信身份映射以及组织/负责人关系，协助人的主管也必须按当前实时关系解析，而不是沿用旧快照或人工名单。这样才能满足现有权限和审计约束，也避免组织调整后继续错发。备选方案是维护日报模块私有的“协助人 -> 主管”表，但这会形成第二套组织模型。

4. 协助通知按“同一份日报 + 同一协助人”聚合，并对最终接收人去重。

   同一份日报里可能多次提到同一个协助人。如果每条事项都单独发一次，会产生明显骚扰。因此系统按“同一日报 + 同一协助人”聚合为一条协助人消息和一条协助人主管消息；若最终接收人因为角色重叠而重复，例如协助人的主管刚好就是日报接收主管，或协助人本人已经是该日报的主管接收人，则必须按目标用户去重，只保留一条发送，同时在审计里保留多重来源关系。备选方案是逐条逐人发送，但噪音过高。

5. 协助通知内容只发送最小必要问题摘要，不转发整份日报。

   协助人和协助人主管需要知道的是“谁在什么日期、围绕什么客户或商机、遇到了什么问题、希望你帮什么”，而不是整份日报的全部工作总结。因此通知内容应聚焦协助事项本身，可附带客户/商机、业务日期和提报人，但不得把与协助无关的日报段落原样转发，避免信息过载和不必要扩散。备选方案是把完整日报直接转发给协助人，但会扩大无关信息暴露面。

6. 协助通知失败必须与主日报送达解耦，只记录子任务失败。

   主管看到日报是主链路，协助升级是并行子链路。若协助人身份无法解析、权限校验失败或通知通道重试后仍失败，系统必须只标记对应协助子任务失败并留痕，不得回滚已经成功送达给主管的日报。备选方案是把协助通知作为主日报送达前置条件，但这会放大单点失败影响。

## Risks / Trade-offs

- [协助人姓名可能重名或写法不规范] -> 只接受可唯一映射的明确协助人；解析失败时不直发，只记审计并保留主管侧可见信息。
- [协助通知可能泄露过多日报内容] -> 消息体只保留最小必要问题摘要、相关对象和请求事项，不转发整份日报。
- [同一协助人被同一日报多次点名会造成轰炸] -> 按“日报 + 协助人”聚合，并对最终接收目标去重。
- [协助人主管关系在组织调整后变化] -> 发送前实时解析当前主管，禁止沿用旧授权快照。
- [协助子任务失败可能被误认为主日报失败] -> 在审计和发送记录中分离主日报送达状态与协助升级子任务状态。

## Migration Plan

1. 先补齐本次变更的 proposal、design、tasks 与 capability spec，明确触发点、接收人边界和失败处理规则。
2. 在实现层为“日报正式对上发送”增加协助升级子任务生成逻辑，并把协助事项与原始日报建立可追踪关联。
3. 接入现有身份映射、主管解析、主动通知底座和审计链路，完成协助人通知与协助人主管通知的正式投递。
4. 补齐测试覆盖后，先在测试改投模式下联调；若需要回滚，只需停止生成协助升级子任务，不影响主管侧日报主链路。

## Open Questions

- 首版是否支持在同一条“遇到与协助”文本中解析多个明确协助人，还是先只支持唯一协助人。
- 协助人主管的解析优先级应以“直属负责人”还是“当前部门主管”为准，是否存在需要按角色差异处理的边界。
- 当协助人无法唯一识别时，是否需要在原提报人或原主管侧增加“协助通知未发出”的显式提示，而不是只留审计。
