## Context

当前智能分析链路已经具备三块关键基础能力：

- 受控分析主链已经支持统一 AI 入口、计划执行与受控直查双模式、白名单、AST 校验、预检与统一结果包。
- 问数知识层已经有初步实现，例如别名提示、码表提示、时间字段提示和已验证示例，但仍主要以内置常量方式存在于 `analysis-query-knowledge.service.ts` 中。
- Web 端同时存在“智能分析结果页”和“经营报表页”两套报告表达体系；两者都在走区块化方向，但区块协议、来源说明和执行依据展示仍未完全统一。

这一现状带来三类结构性问题：

- 业务和治理侧无法独立维护问数知识资产，语义质量优化仍高度依赖研发改代码。
- 编排层已经记录了大量执行快照，但结果页与审计页看到的仍多是碎片化字段，不足以解释“为什么命中了这条口径”“为什么这次 fallback”“为什么最终走了这条读取路径”。
- 智能分析与经营报表正在分别演进自己的区块模型，如果继续各做各的，后续会出现组件重复、口径重复和来源说明不一致。

本次变更必须继续满足以下约束：

- 不引入自由 Text-to-SQL，不允许模型绕开既有白名单、权限注入、AST 校验和受控工具边界。
- 不扩新的经营主题，不与现有 `enrich-topic-driven-analysis-reports` 主题扩展范围混做。
- 不新增第二套权限模型；治理入口继续沿用现有后台访问边界。
- 不把 Notebook、离线验证工作台或分析师沙箱纳入正式生产主链。

## Goals / Non-Goals

**Goals:**

- 将问数别名、时间口径、组织归一提示、已验证问法和负例样例升级为可治理、可发布、可审计的语义资产。
- 在受控分析编排中生成统一执行轨迹快照，覆盖知识资产命中、任务级执行来源、fallback 原因和最终执行摘要。
- 将智能分析结果交付协议收敛为可复用的区块化报告协议，并补齐来源说明、口径说明和执行依据摘要。
- 尽量复用现有 management-report 区块组件与口径说明浮层，而不是新建第三套报告组件体系。

**Non-Goals:**

- 不在本次变更中新增更多 `analysisFacetProfile`、主题任务包或派生经营指标。
- 不在本次变更中重构整个治理导航或新增新的后台权限动作。
- 不把语义资产治理做成通用知识库平台，本轮只覆盖受控问数所需最小资产集合。
- 不将审计系统改造成独立日志平台，本轮只增强分析相关轨迹与可解释字段。

## Decisions

### 决策 1：新增独立“语义资产治理”域，而不是继续扩展查询模板或把知识层留在代码里

本次新增独立 capability `analysis-semantic-knowledge-governance`，用于管理以下最小语义资产：

- 指标 / 业务口径别名
- 时间字段与默认时间口径提示
- 组织 / 团队归一提示
- 已验证问法示例
- 负例 / 高风险问法示例

这些资产不并入现有常用查询模板，也不继续只保留在 `AnalysisQueryKnowledgeService` 静态常量中。模板仍负责“用户可见的复用入口”，语义资产负责“统一 AI 理解层与受控查询生成器共享的语义上下文”。

原因：

- 模板的核心职责是入口复用，不适合承载负例、时间字段口径和组织归一提示。
- 继续只放在代码里，会让业务优化依赖研发排期，不具备治理闭环。
- 独立语义资产域后，后续可以版本化、灰度和审计，而不污染模板模型。

备选方案：

- 扩展 `query-template` 模型承载所有知识资产。问题是模板与语义资产的职责边界会混乱。
- 保留现有静态知识层，只追加更多常量。问题是不可治理、不可审计、不可灰度。

### 决策 2：语义资产采用“草稿编辑 + 发布快照”模式，分析主链只消费发布态快照

治理后台允许编辑语义资产，但受控分析主链只读取发布态资产快照。分析执行时记录：

- 资产 ID
- 资产类型
- 资产版本或更新时间
- 命中摘要

不要求本轮引入复杂审批流，但必须区分“后台正在编辑”和“分析主链正在消费”的状态，避免半编辑状态直接污染线上问数。

原因：

- 分析链路需要稳定、可回溯的知识输入，不能直接消费随时变化的草稿内容。
- 发布快照模式更容易与审计、回滚和回归测试结合。

备选方案：

- 每次分析实时读取最新编辑态资产。问题是行为不可复现，排障困难。
- 先不做发布态，只有启停。问题是无法解释“这次结果到底用了哪一版资产”。

### 决策 3：执行轨迹统一收敛为一个中性快照对象，而不是继续散落在 request/result/audit 多处局部字段

当前编排层已经有 `executionSnapshot`、`resultBundleSnapshot`、`insightSnapshot` 等快照，但对用户、治理和审计来说仍然过散。本次新增统一“执行轨迹摘要”视角，收敛以下信息：

- 入口问题与标准化问题
- 命中的语义资产摘要
- 任务级 `resultKind`
- 每个任务的 `executionSource`、`preferredSource`、`matchedAdapter`
- 最终 `fallbackReason`
- 最终 `consistencyToken`
- 结果区块与数据集引用摘要

底层实现可以继续复用现有多个 snapshot 对象，但对上层页面和审计接口暴露统一摘要结构。

原因：

- 页面和审计只需要“能解释这次发生了什么”的统一视图，而不是多个底层对象的原样拼接。
- 统一摘要结构后，Web 结果页、审计详情和后续测试夹具可以共用同一份 contract。

备选方案：

- 继续让页面分别读取多个 snapshot 字段。问题是页面逻辑会越来越碎，也不利于治理端聚合统计。
- 直接把底层所有字段全部暴露到前端。问题是噪声太大，且会放大兼容负担。

### 决策 4：结果交付协议与 management-report 区块协议收敛到“共享 section contract，分开组装服务”

本次不合并 `analysis` 与 `management-report` 两个服务，但要求两者逐步收敛到共享的 section contract。首轮统一以下区块语义：

- `metric-strip`
- `trend`
- `distribution`
- `risk`
- `detail-table`
- `actions`
- `source-notes`

分析链继续由 `AnalysisReportComposerService` 组装结果，经营报表继续由 `management-report` 相关 builder 组装结果，但前端渲染和口径说明层尽量共用已有组件，特别是 `ManagementSectionCanvas` 和 `SectionSourcePopover` 这类能力。

原因：

- 现有经营报表已经沉淀了一批可复用的区块组件和来源说明浮层，直接复用比再造一套成本低。
- 共享 contract、分开组装服务可以避免“一次性合并两个模块”的高风险。

备选方案：

- 让智能分析完全复用经营报表接口。问题是两者入口、查询节奏和上下文能力不同，当前时机过早。
- 为智能分析新建第三套区块协议。问题是后续维护成本最高。

### 决策 5：来源说明、口径说明和执行依据摘要进入统一结果协议，而不是继续作为页面临时拼接文案

结果协议必须显式包含三层解释信息：

- `sourceNotes`：指标来源、维度来源、口径来源
- `footnotes`：边界说明、缺失区块说明、降级说明
- `executionTraceSummary`：本次执行依据摘要

这些字段属于结果交付 contract，而不是页面自行推导的 UI 文案。Web 与审计端都从统一协议中读取，避免同一结果在不同页面显示不同解释。

原因：

- 只有把解释性字段纳入结果协议，才能保证它们与 `consistencyToken`、区块和数据集引用同源。
- 页面临时拼装解释文案很容易与后端真实执行轨迹脱节。

备选方案：

- 继续仅在前端根据已有字段生成说明。问题是难以审计，也难以让企业微信和 Web 保持一致。

### 决策 6：本次治理入口复用现有后台访问边界，不新增新的权限动作

本次不新增新的治理权限 key，默认复用现有治理后台的访问边界承接语义资产治理入口。若后续需要把语义资产治理单独授权，再单开权限变更，不与本次小型 change 混做。

原因：

- 当前目标是先把能力边界和 contract 立起来，避免同时引入新的权限矩阵变更。
- 小型 change 更适合优先解决结构问题，而不是拉长到整套治理授权重构。

备选方案：

- 现在就新增专用权限动作。问题是会扩大变更范围，并触发权限矩阵、README 和测试矩阵同步成本。

## Risks / Trade-offs

- [语义资产治理与主题报告扩展边界不清，后续容易再次膨胀] → 在 proposal/spec 中明确本次只治理资产与协议，不新增更多主题包。
- [共享 section contract 但保留两套组装服务，短期内仍会存在字段映射成本] → 先统一区块协议与说明字段，后续再评估是否值得进一步合并服务层。
- [发布态快照增加了治理成本] → 先采用最小“草稿 / 发布”模型，不引入复杂审批流。
- [执行轨迹字段过多导致页面噪声过大] → 对外只暴露统一摘要视图，底层详细快照继续保留给审计和调试。
- [不新增权限动作可能让语义资产治理入口暂时沿用较宽后台权限] → 在设计中明确这是阶段性取舍，后续若出现治理职责分离需求，再发起独立权限变更。

## Migration Plan

1. 新增 `analysis-semantic-knowledge-governance` spec，并补齐治理模型、发布约束和审计要求。
2. 扩展 `controlled-analysis-orchestration`，定义语义资产命中快照与统一执行轨迹摘要要求。
3. 扩展 `structured-analysis-result-delivery`，定义共享 section contract、来源说明和执行依据摘要要求。
4. 后端先实现语义资产持久化模型与发布态读取，再将 `AnalysisQueryKnowledgeService` 改为“治理资产优先，静态常量兜底”。
5. 后端在编排层补统一轨迹摘要，并将结果协议扩展为带 `sourceNotes / footnotes / executionTraceSummary` 的 section schema。
6. 前端先在分析结果页接入统一 section contract 和轨迹摘要，再逐步复用 management-report 区块组件。

回滚策略：

- 若语义资产治理实现不稳定，可临时关闭治理资产读取，回退到现有静态知识层，但保留新数据模型与文档。
- 若共享 section contract 前端渲染异常，可先保留后端新协议，同时让分析页临时继续走旧布局，避免影响主链执行。
- 若统一轨迹摘要字段引发兼容问题，可先隐藏结果页展示入口，保留审计与后端快照，不回滚底层执行安全栈。

## Open Questions

- 语义资产首版是否需要“批量导入 / 导出”，还是仅支持后台表单维护即可。
- `sourceNotes` 是否直接复用经营报表的指标字典结构，还是只先对齐最小公共字段。
- 结果页中的执行轨迹入口是首版做成折叠区、抽屉还是独立审计跳转入口；实现阶段需要结合现有页面密度再定。
