## Context

当前合同审核模块已经具备独立任务模型、`.docx` 提取、结构化问题、报告生成、带批注合同、权限审计和风险优先结果页。但审核核心仍以代码内置规则为中心：

- 审核标准主要定义在 `backend/src/modules/contract-review/contract-review.rule-set.ts`；
- AI 审核虽然已接入 Codex SDK，但更像“可选增强”，AI 不可用时会快速回退到关键词命中；
- 业务想调整审核标准时，需要改 TypeScript 代码，而不是改文档化要求；
- 审核链路没有显式沉淀“使用了哪套 skill、哪版提示词、当前是完整 AI 审查还是降级执行”。

仓库里已经存在一套可供参考的合同审核 skill 资料：

- `智能合同审核系统相关资料/contract-review/SKILL.md`
- `智能合同审核系统相关资料/contract-review/references/company_requirements.md`
- `智能合同审核系统相关资料/contract-review/references/workflow.md`

这些资料更接近“审核知识包”，但尚未被产品化为当前 NestJS 服务可稳定运行的运行时资产。本次设计的核心，不是把 agent 交互式 skill 直接搬进服务端请求，而是把这些文档整理为可验证、可版本化、可审计的 `skill pack runtime`。

## Goals / Non-Goals

**Goals:**

- 让合同审核标准从代码硬编码升级为外部 `skill pack` 驱动，支持通过文档和配置迭代审核标准。
- 让确定性规则快审成为稳定首轮结果，并将 AI 作为未覆盖检查项的补充审核路径，形成可解释的受控编排链路。
- 为每个审核任务沉淀完整的执行依据，包括 `skill pack` 版本、模型、提示词摘要、执行模式和校验明细。
- 保持现有上传、详情、产物、权限和风险优先结果页结构，避免重新设计整个前端工作台。
- 为后续多套审核标准、合同类型扩展和审核复盘留出清晰边界。

**Non-Goals:**

- 不把本地 `SKILL.md` 直接作为生产运行时执行入口，不在后端请求中启动交互式 agent 会话。
- 不在本轮引入在线规则编辑器、审批流、法务协作工作台或企业微信侧发起审核。
- 不建设独立 Python 微服务来承载主审核流程；若复用 Python 脚本，只作为迁移参考而非正式主链路。
- 不改变现有权限边界、最近任务首页极简结构和风险优先主视觉。

## Decisions

### 决策 1：将参考 skill 资产整理为服务端可运行的 `skill pack`

系统将把现有参考资料抽象为一组版本化运行时资产，而不是继续依赖 `contract-review.rule-set.ts` 单文件：

- `requirements.md`：业务审核要求正文；
- `workflow.md`：审核步骤和提示词约束；
- `profile.yaml`：版本号、适用合同类型、启用模型、阈值和执行模式；
- `prompts/*.md`：规划、分组审查、汇总等提示词模板。

后端新增 `SkillPackLoader` 和 `SkillPackValidator`，在服务启动或热更新时读取并校验激活的 pack。

原因：

- 文档型标准更符合业务“想改标准先改要求文件”的诉求；
- profile 和 prompt 独立后，能区分“业务要求变化”和“模型执行策略变化”；
- pack 校验可以阻止格式损坏或缺字段的标准直接进入生产链路。

备选方案：

- 继续维护 `contract-review.rule-set.ts` 并定期手工同步 markdown。问题是业务无法直接维护标准。
- 直接执行 agent skill。问题是服务端稳定性、并发、超时和审计都不可控。

### 决策 2：在任务创建时编译 `skill pack snapshot`，而不是运行时直接读取原文档

每个审核任务创建时都将把当前激活的 `skill pack` 编译为可执行快照，并固化以下内容：

- pack ID / version / checksum；
- 检查项清单与风险等级；
- prompt profile 摘要；
- 确定性校验器集合；
- 执行模式默认值。

正式审核只消费该快照，不在同一任务中途重新读取原始 pack 文件。

原因：

- 保证同一任务在复查、下载产物和审计检索时具备一致依据；
- 即使后续要求文件更新，历史任务仍可准确追溯；
- 编译态快照可提前完成语义归类、检查项 ID 分配和 validator 绑定。

备选方案：

- 每次阶段执行都动态读取 pack 文档。问题是运行时不稳定，且难以解释历史结果。

### 决策 3：采用“确定性规则快审 + AI 补充审核 + 结果合并”的受控编排

审核主链路拆成三个层次：

1. `Fact Extractor`
   提取合同事实，如金额、税率、付款节点、违约比例、开票名称、知识产权表述、争议解决地等。
2. `Deterministic Validators`
   先对税率、账期、折扣、违约上限、开票前缀等可计算规则做硬校验，产出可立即返回的规则快审结果。
3. `AI Reviewer`
   当存在未被确定性规则覆盖的检查项且 AI 可用时，再基于 `skill pack snapshot` 和合同片段，对按主题分组的检查项进行语义判断，并以 JSON schema 返回补充命中项。

最后由 `Issue Merger` 去重、合并、评级并生成统一结构化问题集合。

原因：

- 数值和格式类问题适合代码硬校验，例如折扣、税率、付款账期，且可以更快形成首轮结果；
- 语义判断类问题适合 AI，例如知识产权归属是否失衡、责任是否明显不对等；
- “规则快审先返回，AI 补充后刷新”既能提高稳定性，又能降低纯 AI 幻觉带来的风险。

备选方案：

- 纯 AI 审合同。问题是可计算规则不稳定，且难以保证一致性。
- 纯规则引擎。问题是复杂语义条款识别能力不足。

### 决策 4：AI 审核使用分阶段、结构化输出，而不是“一次性整份合同自由发挥”

AI 审核将采用受约束的多步模式：

- `planner prompt`：根据合同类型和检查项分组生成审查计划；
- `reviewer prompt`：按主题组审查，并输出严格 JSON；
- `summarizer prompt`：生成摘要、总体结论和用户可读说明。

每次 AI 输出都必须符合预定义 schema，至少包含：

- `checkId`
- `result`
- `riskLevel`
- `isVeto`
- `quote`
- `locator`
- `reason`
- `suggestion`
- `confidence`

原因：

- 分主题审查可降低提示词噪音，避免长合同一次性输入导致质量失控；
- 结构化 schema 便于和确定性校验器结果合并；
- 可直接驱动前端展示、批注落点和报告生成。

备选方案：

- 一次 prompt 返回所有问题。问题是上下文过长、输出不稳定、错误难定位。

### 决策 5：将当前“关键词命中回退”收敛为受控降级模式，并区分规则快审与降级快审

系统保留降级，但执行路径调整为：

- `AI_HYBRID`：规则快审完成后，AI 补充审核也已完成并合并入最终结果；
- `DETERMINISTIC_ONLY`：当前仅返回规则快审结果；若存在 `degradationReason`，表示 AI 补充审核不可用、超时或失败；
- `BLOCKED`：pack 无法加载、prompt 缺失或事实提取失败时阻断。

旧的简单关键词命中不再作为默认正式审核结果来源，只允许在明确标记的兼容模式或迁移期下存在。

原因：

- 关键词命中过于粗糙，容易让业务误以为“AI 已审”；
- 显式执行模式更利于结果解释与灰度观察；
- 对业务来说，“当前先展示规则快审结果”或“本次为降级快审”都比“看起来像正式审核但其实只是关键字”更安全。

备选方案：

- 继续保持关键词兜底为正式结果。问题是结果可信度不足。

### 决策 6：结果页继续保持风险优先，但补充审核依据与降级提示

前端不改变极简首页和风险优先详情主结构，只做最小增量：

- 在结果详情元信息区展示 `skill pack` 名称、版本和执行模式；
- 若任务为 `DETERMINISTIC_ONLY` 且无降级原因，展示显式提示：“当前为规则快审结果，仅覆盖已配置的明确判定项”；
- 若任务为 `DETERMINISTIC_ONLY` 且存在降级原因，展示显式提示：“当前为降级快审结果，并说明降级原因”；
- 若任务存在 AI 补充审核，则在结果页展示补充审核状态；
- 保持规则编码和技术元数据折叠，但允许在“查看依据”中看到 `checkId` 和来源 pack。

原因：

- 商务人员仍应先看风险，不应把首屏变成技术监控页面；
- 但当结果仍停留在规则快审或降级快审时，必须让用户知道覆盖边界；
- pack/version 可帮助业务在标准更新后理解“为什么这一版结果不同”。

## Risks / Trade-offs

- [Skill pack 结构设计过于复杂] → 先限定为 `requirements.md + workflow.md + profile.yaml + prompts/`，避免一上来设计通用 DSL。
- [AI 成本和耗时上升] → 采用分组审查、片段裁剪、结构化 schema 和真正异步任务，避免整份合同一次性长上下文调用。
- [文档标准与代码校验器发生漂移] → 在 pack 编译阶段显式声明 `validatorBindings`，并要求测试校验 pack 与 validator 一致性。
- [降级模式被误当成正式审核] → 前端详情和导出报告都必须显示执行模式，审计中也必须可检索。
- [多版本标准导致历史任务理解成本上升] → 任务详情保留 pack/version/checksum 和摘要，支持复盘。
- [迁移期同时保留旧规则文件和新 pack] → 使用特性开关和双写期，完成灰度后移除硬编码规则路径。

## Migration Plan

1. 以现有 `智能合同审核系统相关资料/contract-review/` 为基础，整理首个可运行 `skill pack`。
2. 新增 pack loader、validator、snapshot compiler 和混合编排服务，但先通过特性开关运行。
3. 在灰度环境同时保留旧 `rule-set.ts` 和新 pack runtime，对比输出差异并修正 pack。
4. 扩展任务、详情、产物和审计模型，记录 pack/version/model/execution metadata。
5. 前端补充审核依据版本、规则快审或降级快审提示，以及 AI 补充审核状态，保持现有风险优先布局不变。
6. 灰度通过后切换默认运行时到 `skill pack runtime`，默认执行策略调整为“规则快审优先 + AI 补充审核”，并将旧关键词兜底收敛为兼容模式或移除。
7. 回滚时恢复旧运行时开关，已生成的任务继续保留并通过执行模式标识区分来源。

## Open Questions

- `skill pack` 是随代码仓库版本发布，还是允许通过挂载目录热更新？
- 是否需要在首版就支持按合同类型选择不同 pack，还是先只保留一套激活 pack？
- AI reviewer 是否直接复用现有 Codex SDK，还是需要抽象出模型提供方接口以便后续替换？
- 导出报告中是否需要显式显示“本次审核依据的 pack 版本与执行模式”？
