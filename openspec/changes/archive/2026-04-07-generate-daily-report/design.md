## Context

当前后端已经有 `daily-report` 模块、日报接口和审计写入能力，日报数据也通过 `AppStorageService` 统一落在 `dailyReports` 与 `dailyReportSummaryBatches` 中。问题不在于“能不能存”，而在于日报生成链路还没有按 `01-web.md` 收敛成统一产品规格：Web 端缺少独立日报工作台，企业微信入口的对话风格也需要更短、更明确，同时日报写入必须始终保持受控数据库边界，不能被误当成 CRM 自由写回。

## Goals / Non-Goals

**Goals:**
- 让日报片段采集、草稿拼装、确认、推送、提醒和汇总走同一条受控链路。
- 让企业微信日报对话保持简短、明确、少轮次，只补问缺失信息。
- 让 Web 端可以查询、查看和确认日报，并提供清晰的加载、空态、失败和无权限反馈。
- 保留来源信息、操作人信息和审计快照，便于追溯日报来自哪里、谁写入、谁确认。

**Non-Goals:**
- 不修改统一登录、会话建立和认证链路。
- 不把日报流程扩展成 CRM 客户、商机或跟进的直接写回。
- 不新增独立的数据库模型层或新的持久化引擎。
- 不重构分析问数编排链路。

## Decisions

1. 复用现有 `daily-report` 模块和仓储层，不单独再建一套日报服务。
   当前接口、状态机、确认流程、提醒流程和审计写入都已经集中在 `backend/src/modules/daily-report/`，继续复用可以避免把日报逻辑拆散到多个模块。备选方案是新建独立日报服务或 BFF 聚合层，但那会增加迁移成本，也会让现有 API 与新实现分叉。

2. 把数据库交互限制在日报服务和仓储层内部。
   片段写入、确认、提醒、封账和汇总都必须通过 `DailyReportService` -> `DailyReportRepository` 完成，外部入口只传业务参数，不直接操作存储状态。这样可以持续保留 `sourceInterface`、`sourceObjectId`、`sourceOperatorId` 和 `sourceCode` 等可追溯字段，也能明确阻断任何把日报当作 CRM 写回入口的做法。备选方案是让控制器或机器人服务直接改存储，但那会让审计和校验分散。

3. 为企业微信新增一个独立的日报意图入口，使用确定性短提示，不复用分析问数的 AI 解释链路。
   `wecom-ai-conversation-orchestration.service.ts` 适合分析问数、补问和解释，但日报是流程型操作，不需要长篇生成式回复。日报入口应先识别是否进入日报收口，再按固定字段追问缺失项，例如主管、片段内容或业务日期，回复必须短、直、明确。备选方案是复用分析 prompt 再做一层提示词改写，但会让日报对话变长、可控性变差。

4. Web 端直接对接现有日报 API，新增独立日报工作台路由。
   当前 `openapi.yaml` 已经定义了日报片段、列表、详情、确认、审计和汇总批次接口，前端没有必要再造新接口。Web 页面应新增独立路由和工作台视图，围绕列表、详情、确认和状态反馈组织页面结构，并复用现有页面反馈模式。备选方案是增加一个聚合 BFF，但这会稀释现有接口契约。

5. 把日报状态推进逻辑保持在服务层，并以现有状态值为准。
   `DRAFT`、`PENDING_CONFIRMATION`、`CONFIRMED`、`LATE_CONFIRMED`、`CLOSED`、`SUMMARIZED` 已经能覆盖当前链路。服务层负责从片段完成度判断是否进入待确认，从确认时间判断是否迟交，从汇总任务判断是否写回批次 ID。备选方案是把状态推导下放到前端，但那会导致口径不一致。

## Risks / Trade-offs

- [AppStorage 仅是当前实现的存储层] -> 现在是内存态，重启会丢失草稿；缓解方式是继续通过仓储抽象封装写入，后续更换持久化实现时不改调用面。
- [企业微信提示语过短可能少解释] -> 通过“只问缺失项、只返回阻断原因”的规则控制长度，避免聊天式回复。
- [Web 和企业微信口径漂移] -> 两个渠道都从同一份日报记录和同一套片段标签生成展示内容。
- [日报与 CRM 写回边界混淆] -> 服务层统一做来源校验和域隔离，禁止写入客户、商机或跟进实体。
- [页面新增会牵涉原型更新] -> 先补 `.pen` 原型再落代码，避免导航和结构和现有规范冲突。

## Migration Plan

1. 先落地规格和设计，确认日报生成只走受控日报域。
2. 先改后端日报服务和企业微信入口，确保写入、确认、审计和提示语都符合新规格。
3. 再加 Web 日报工作台路由和页面，页面只消费现有日报 API。
4. 如果需要新的页面结构或导航，再同步更新 `.pen` 原型与路由入口。

回滚时只需隐藏新的 Web 入口和企业微信日报意图路由，现有日报 API 与存储结构可以保留不动。

## Open Questions

- 日报工作台应作为独立顶层路由，还是放在现有分析工作台内部的一个子页。
- 手工日报片段是否允许在 Web 和企业微信两端都录入，还是只允许 Web 端补录。
- 汇总批次结果是否需要在普通用户视图中展示，还是只给管理员和主管查看。
