## ADDED Requirements

### Requirement: 系统必须按业务日期聚合日报片段并生成草稿

系统 MUST 以 `requesterId + businessDate` 作为日报草稿的聚合键。同一用户同一天提交的多个片段 MUST 合并到同一份日报草稿中，日报 MUST 按预定义片段顺序拼装为草稿摘要；当所有必填片段齐备时，系统 MUST 将日报状态推进到 `PENDING_CONFIRMATION`。

#### Scenario: 首次提交片段生成草稿
- **WHEN** 用户在某个业务日期首次提交日报片段
- **THEN** 系统必须创建一份新的日报草稿并写入该片段

#### Scenario: 同一片段类型再次提交时应覆盖旧内容
- **WHEN** 用户再次提交相同片段类型的日报内容
- **THEN** 系统必须用新内容覆盖旧片段，而不是新增重复片段

#### Scenario: 所有必填片段齐备后进入待确认状态
- **WHEN** 某份日报的所有必填片段都已补齐
- **THEN** 系统必须将该日报状态更新为 `PENDING_CONFIRMATION`

### Requirement: 系统必须校验日报来源并限制数据库写入边界

系统 MUST 只允许通过日报仓储和服务层写入日报数据，不得把日报收口当作 CRM 客户、商机或跟进记录的直接写回入口。系统 MUST 保留来源接口、来源对象 ID、来源操作人和来源返回码；当上游返回失败、来源对象缺失或片段类型与来源不匹配时，系统 MUST 拒绝写入。

#### Scenario: 上游返回失败时拒绝写入
- **WHEN** 来源接口返回码不为 `0`
- **THEN** 系统必须拒绝将该片段写入日报

#### Scenario: 来源类型与片段类型不匹配时拒绝写入
- **WHEN** 商机来源试图写入非客户/商机变化片段，或跟进来源试图写入非今日跟进片段
- **THEN** 系统必须拒绝写入并返回明确错误

#### Scenario: 手工片段可以直接进入日报草稿
- **WHEN** 用户手工录入日报片段且未指定外部来源
- **THEN** 系统必须允许写入，但仍然保留业务日期、提交人和审计信息

#### Scenario: 日报流程不得修改 CRM 写回对象
- **WHEN** 用户提交日报片段或确认日报
- **THEN** 系统必须只更新日报域内数据，不得顺带修改 CRM 客户、商机或跟进实体

### Requirement: 系统必须支持日报确认、迟交标记和主管推送

系统 MUST 仅在日报完成收口后允许确认。确认时系统 MUST 记录确认时间、确认人和是否迟交，并向主管生成推送交付记录；当确认时间超过当日截止点时，系统 MUST 将该日报标记为 `LATE_CONFIRMED`。系统 MUST 支持次日汇总批次生成，并在日报记录中回填汇总批次 ID。

#### Scenario: 日报完成后在截止前确认
- **WHEN** 用户在日报收口完成后并在截止点前确认日报
- **THEN** 系统必须将日报标记为 `CONFIRMED` 并推送给主管

#### Scenario: 截止后确认应标记为迟交
- **WHEN** 用户在截止点之后确认日报
- **THEN** 系统必须将日报标记为 `LATE_CONFIRMED` 并记录迟交信息

#### Scenario: 未完成收口的日报不得确认
- **WHEN** 日报仍处于草稿状态且必填片段未齐备
- **THEN** 系统必须拒绝确认并提示先完成收口

#### Scenario: 次日汇总批次应回填日报引用
- **WHEN** 系统执行次日日报汇总
- **THEN** 系统必须生成汇总批次记录并把批次 ID 写回相关日报

### Requirement: 系统必须提供日报查询与状态反馈

系统 MUST 提供日报列表、单条日报详情、审计快照和汇总批次查询能力，并基于当前权限过滤可见数据。Web 端 MUST 明确展示加载中、空状态、失败、无权限和重试反馈，不得只返回静态内容而缺少结果反馈。

#### Scenario: 普通用户只看到自己的日报
- **WHEN** 非管理员用户查询日报列表或详情
- **THEN** 系统必须只返回当前用户有权访问的日报记录

#### Scenario: 列表为空时展示空状态
- **WHEN** 用户进入日报页面但当前没有可见日报
- **THEN** Web 端必须显示空状态和下一步指引，而不是空白页

#### Scenario: 查询失败时展示可重试反馈
- **WHEN** 日报查询请求超时或失败
- **THEN** Web 端必须显示失败原因并提供重试入口

### Requirement: 系统必须让企业微信日报交互简短明确

系统 MUST 在企业微信日报交互中使用短句、直接、少轮次的提示语。系统 MUST 只追问当前缺失的最少必要信息，不得输出冗长说明、闲聊或无关背景；当日报内容为空或不合法时，系统 MUST 用一句话明确指出问题和下一步动作。

#### Scenario: 首次日报缺少主管信息时只追问主管
- **WHEN** 用户首次提交日报片段但未提供主管接收人
- **THEN** 系统必须只追问主管信息，不得同时展开其他说明

#### Scenario: 日报内容缺失时返回一句话阻断
- **WHEN** 用户提交的日报内容为空或只有空白字符
- **THEN** 系统必须用简短中文提示“日报内容不能为空”并结束当前轮次

### Requirement: 系统必须为日报全链路保留审计快照

系统 MUST 为日报片段写入、确认、提醒、封账和汇总生成可检索审计快照。审计快照 MUST 记录业务日期、操作人、状态变化、来源信息和交付结果；即使流程被阻断或失败，也 MUST 留下对应的审计记录。

#### Scenario: 片段写入后生成审计记录
- **WHEN** 系统成功写入一条日报片段
- **THEN** 系统必须记录该次写入对应的审计快照

#### Scenario: 汇总批次生成后保留审计记录
- **WHEN** 系统成功生成日报汇总批次
- **THEN** 系统必须记录汇总批次编号、确认数、迟交数和未交数等审计信息
