## Context

当前工作台常用查询由 `CommonQueryPanel.vue` 承载，并通过 `AnalysisWorkbenchPage.vue` 中的右侧抽屉打开；最近查询也在抽屉内展示。后端查询模板由 `query-assets` 模块管理，已经具备模板列表、固定 SQL 执行、治理端新增 / 更新 / 删除、SQL 校验和预览能力。模板范围治理已有 `query-template-scope-governance` 约束，要求模板执行、预览、重跑和导出都按当前用户范围重新校验。

本变更会同时影响工作台页面结构、模板数据模型、模板 API、受控分析结果复现、权限点、范围治理和审计，因此需要先明确技术边界。实现必须继续遵守：中文用户文案、当前实时权限、CRM API-first / 受控 SQL 兜底、只读 SQL、安全校验、审计留痕、Element Plus 和根级 `DESIGN.md` 的 UI 约束。

## Goals / Non-Goals

**Goals:**

- 将常用查询改为工作台左侧常驻资产库，并保留最近查询抽屉。
- 支持“我的模板 / 其它模板”、搜索筛选、标签分类、历史点击排序和大列表性能。
- 支持自由问数结果保存为模板，并确保保存的模板可复现、可校验、可审计。
- 支持从其它模板复制到我的模板，生成当前用户自己的模板副本。
- 支持固定部门模板跨部门复用时的权限阻断和用户确认后改写为当前可见范围。
- 支持受控 SQL 编写权限的最小化落点。

**Non-Goals:**

- 不新增用户自定义报表设计器。
- 不允许普通用户绕过 SQL 白名单、AST 校验、范围治理和审计。
- 不改变最近查询的个人历史属性和抽屉入口。
- 不扩大 CRM 数据白名单，不引入跨 CRM 外部数据源。
- 不把 AI 改写结果静默写回来源模板。

## Decisions

### 1. “添加到我的模板”采用个人副本，不采用引用

用户点击“添加到我的模板”时，后端复制模板定义快照并创建一条新的模板记录，设置 `ownerUserId` 为当前用户，记录 `sourceTemplateId`、来源模板名称、来源 SQL 版本和复制时间。

选择原因：

- 用户明确认为“保存到我的模板就是我自己的”，副本模型符合业务直觉。
- 来源模板后续调整不会意外影响用户已经保存的模板。
- 用户具备 SQL 编写权限时，可以在副本基础上继续治理和编辑。

替代方案：

- 个人引用：存储更省，但用户编辑和来源模板变更语义复杂，已不符合当前产品口径。

### 2. 自由问数保存必须由服务端从 `queryId` 复现模板

前端保存模板时只提交名称、说明、分类、标签、可见性和展示配置等元数据。后端根据 `queryId` 读取已成功分析结果中的执行快照，优先使用实际执行的受控 SQL；若只有执行计划，则在服务端编译为参数化 SQL 模板并重新执行只读、白名单、AST、范围治理校验。无法安全复现时拒绝保存。

选择原因：

- 防止前端传入自由 SQL 绕过执行边界。
- 保证保存模板与真实结果来自同一执行事实。
- 允许未来 `PLAN_EXECUTION` 和 `GUARDED_DIRECT_QUERY` 两类执行模式逐步统一到可复现模板资产。

替代方案：

- 直接保存前端传入 SQL：风险过高。
- 所有自由问数都允许保存：会产生不可执行或不可审计模板。

### 3. 模板默认全员可见，但执行按当前用户实时权限收口

普通业务用户保存的新模板默认 `visibilityType=SHARED`。其它用户可以搜索、复制到我的模板和分享，但执行、预览、导出仍按执行人的当前权限范围、模板范围治理和白名单重新校验。

选择原因：

- 产品要求“不用做太多控制”，降低分享和复用门槛。
- 数据安全不依赖模板可见性，而依赖执行期实时权限。

替代方案：

- 默认私有：更安全但不符合共享复用目标。
- 默认公共且复用原授权快照：不可接受，会突破权限边界。

### 4. 标签存储在模板记录上

模板记录新增 `tags`。`tags` 是可选标签数组，统一承担模板分类能力。保存 / 编辑时标签支持选择已有标签和自由输入新标签，AI 推荐标签必须经用户确认后写入。

选择原因：

- 标签能表达部门、时间、指标、用途等复合维度，避免维护两套分类口径。
- 首期直接存模板字段，避免过早引入复杂标签治理表。

替代方案：

- 仅单分类：搜索和复用维度不足。
- 独立标签字典强治理：首期成本偏高，可后续演进。

### 5. 工作台模板列表采用服务端分页 + 前端虚拟滚动

模板列表接口支持 `scope`、`keyword`、`tag`、`ownerUserId`、`page`、`pageSize` 和 `sort=usage_desc`。前端左侧面板使用稳定高度模板卡片和虚拟滚动；筛选条件变更时重置分页，但不清空当前分析结果。

选择原因：

- 模板数量可能达到上千条，仅前端全量过滤会拖慢首屏和交互。
- 服务端分页便于后续迁移到正式应用库表。
- 虚拟滚动能保证左侧常驻区滚动稳定。

替代方案：

- 一次性加载全部模板：实现简单但性能不可控。
- 纯分页无虚拟滚动：可用但滚动体验弱，不适合高频资产区。

### 6. 固定部门模板允许用户确认后改写为当前可见范围

模板范围治理识别出固定部门、固定负责人、硬编码团队枚举或静态区域条件后，先校验当前用户是否覆盖该固定范围。无覆盖权限时，系统阻断原模板执行，并生成“改为当前可见范围”的建议。用户确认后，系统仅替换为当前 `scopeSnapshot` 覆盖范围，再重新走 SQL 安全和范围治理；无法可靠替换时阻断。

选择原因：

- 满足北京领导复用山东模板查询口径的场景。
- 避免静默把“山东”改成“北京”造成用户误解。
- 改写后仍由程序执行安全边界，AI 不直接放行 SQL。

替代方案：

- 直接阻断所有固定部门模板：安全但复用体验差。
- 静默自动改写：体验快但不可解释且容易误用。

### 7. SQL 编写能力使用独立最小权限

新增或复用等价权限点 `template.sql.write`，只控制受控 SQL 模板新增、校验、预览和保存流程。完整模板治理仍由 `template.manage` 控制。

选择原因：

- 产品要求“只能写查询 SQL 控制”。
- 避免把 SQL 用户误授权为全量治理人员。
- 与现有统一权限矩阵和 `ensureAction` 执行点一致。

替代方案：

- 复用 `template.manage`：权限过大。
- 不做后端权限点：无法防止直接调接口。

## Risks / Trade-offs

- 固定部门 SQL 改写误判 → 仅允许替换已识别的部门 / 负责人 / 团队条件；无法证明安全时阻断，并记录审计。
- 自由问数执行快照不足以保存模板 → 保存接口返回中文不可保存原因；后续逐步补齐执行快照字段。
- 个人副本导致模板数量增长更快 → 使用服务端分页、虚拟滚动和历史点击排序；治理后台保留来源模板追溯。
- 默认全员可见可能暴露模板名称中的敏感业务词 → 保存前对模板名称、分类和标签做用户可见文案审查，禁止展示内部字段、权限键和敏感值。
- 历史模板缺少分类、标签和累计点击字段 → 仓储层提供兼容默认值，迁移脚本或运行时规范化补齐。
- 左侧常驻区挤压结果展示 → 设置默认、最小和最大宽度；窄屏折叠为按钮 / 抽屉，桌面端保持常驻。

## Migration Plan

1. 扩展后端类型、仓储规范化和应用存储默认值，给历史模板补齐 `tags`、`ownerUserId`、`visibilityType`、`usageCountTotal` 等字段。
2. 扩展 OpenAPI 和前端类型，保证旧模板列表仍可读取。
3. 新增保存模板、复制到我的模板、分类 / 标签摘要查询接口，并补齐审计。
4. 改造工作台左侧常用查询区，先接入分页列表和搜索筛选，再启用虚拟滚动与拖拽宽度。
5. 扩展治理模板页的标签分类和 SQL 编写权限控制。
6. 增加固定部门模板改写确认流程，并在 observe / enforce 两类治理模式下分别验证。
7. 更新主规格、快速开始、README 相关说明。

回滚策略：

- 保留原模板执行接口与最近查询抽屉行为。
- 若左侧常驻区出现严重问题，可通过前端特性开关临时回退到抽屉展示常用查询。
- 若保存模板或复制副本接口异常，不影响自由问数、模板执行和最近查询重跑主链路。
