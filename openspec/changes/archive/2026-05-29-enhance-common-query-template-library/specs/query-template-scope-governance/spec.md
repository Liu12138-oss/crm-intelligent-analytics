## ADDED Requirements

### Requirement: 固定范围模板必须支持确认后改写为当前可见范围

当模板 SQL、默认条件或问题文本写死部门、负责人、团队、区域或静态团队清单时，系统 MUST 先校验当前用户范围是否覆盖该固定范围。若当前用户不覆盖该固定范围，系统 MUST 阻断原模板执行。系统 MAY 生成“改为当前可见范围”的 AI 辅助改写建议，但必须在用户确认后，才允许将固定范围条件替换为当前 `scopeSnapshot` 覆盖范围，并重新执行 SQL 白名单、AST 安全校验、范围注入和审计。

#### Scenario: 无权限用户执行固定山东模板被提示改写
- **WHEN** 模板固定查询山东区
- **AND** 当前北京领导用户没有山东区数据权限
- **THEN** 系统必须阻断原山东范围执行
- **THEN** 系统可以提示“该模板原范围为山东区，你当前无权查看山东数据。是否改为查询你当前可见范围”
- **THEN** 系统不得在用户确认前静默改写并执行

#### Scenario: 用户确认改写后重新校验
- **WHEN** 用户确认将固定范围模板改为当前可见范围
- **THEN** 系统必须使用当前用户 `scopeSnapshot` 替换固定范围条件
- **THEN** 系统必须重新执行只读 SQL、字段白名单、AST 安全、范围注入和预检校验
- **THEN** 校验通过后才可以正式执行模板

#### Scenario: 固定范围无法安全替换时阻断
- **WHEN** 模板包含无法可靠解析或替换的固定部门条件、静态团队清单或复杂展示口径
- **THEN** 系统必须阻断改写执行
- **THEN** 系统必须提示联系管理员将模板治理为通用模板

### Requirement: 固定范围改写不得修改来源模板定义

系统 MUST 将固定范围改写限定为本次执行参数、当前用户个人模板副本或个人配置。系统 MUST NOT 将执行人的当前部门、负责人或范围条件自动写回来源模板定义。若用户将改写结果保存为模板，系统 MUST 创建新的个人模板副本并保留来源模板追溯。

#### Scenario: 改写执行不改变来源模板
- **WHEN** 北京领导确认把山东模板改为当前可见范围并执行
- **THEN** 系统不得修改来源模板中的山东固定条件
- **THEN** 后续其他有山东权限的用户执行来源模板时仍可按原模板治理结果处理

#### Scenario: 用户保存改写结果为我的模板
- **WHEN** 用户在改写执行成功后选择保存为模板
- **THEN** 系统必须创建当前用户自己的模板副本
- **THEN** 新模板必须记录来源模板 ID 和改写范围摘要

### Requirement: 固定范围识别与改写必须完整审计

系统 MUST 为固定范围模板的识别、原范围阻断、用户确认改写、改写后校验、改写执行成功和改写失败写入审计。审计记录 MUST 至少包含来源模板 ID、原固定范围摘要、执行人当前范围摘要、用户确认状态、改写结果和失败原因。

#### Scenario: 原范围执行被阻断时写审计
- **WHEN** 用户无权覆盖固定范围模板并被阻断
- **THEN** 系统必须记录固定范围阻断审计
- **THEN** 审计必须包含原固定范围和当前用户范围摘要

#### Scenario: 改写执行成功时写审计
- **WHEN** 用户确认改写并成功执行模板
- **THEN** 系统必须记录固定范围改写执行审计
- **THEN** 审计必须包含改写前后的范围摘要和用户确认时间
