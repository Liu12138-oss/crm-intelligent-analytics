## Context

当前仓库已经具备三块与本次需求直接相关的基础能力：

1. 企业微信机器人已有稳定的会话编排、补问、今日跟进写回、失败重试和审计留痕链路；
2. 后端已经提供 `/api/v1/crm/customers` 与 `/api/v1/crm/opportunities` 两个受控创建接口，并收敛了截图必填字段校验、默认值补齐、自定义字段映射和 CRM 官方 Open API 调用；
3. 跟进写回已经落地“企业微信发送人已映射但无 Web 登录态时，改用内置 CRM 账号换取 token”的模式。

真正缺的不是底层 CRM 写入能力，而是“机器人如何把新增客户/新增商机从一句自然语言，安全地编排成一个受控创建动作”。同时，现有主规格仍把这两类请求视为企业微信一期拒绝项，因此本次变更需要同时解决范围边界、会话状态、认证方式、字段补齐、失败提示和审计约束。

## Goals / Non-Goals

**Goals:**

- 为企业微信机器人新增“受控新建客户”和“受控新建商机”两条独立对话链路，仿照现有商机跟进的确认式交互，而不是直接放行自由写入。
- 复用现有客户/商机创建服务与 CRM 官方 Open API，避免再做一套数据库直写或旁路接口。
- 在机器人没有 Web 登录态时，复用内置 CRM 账号认证模式完成官方 API 调用，并把真实企微发起人保留在审计事件中。
- 对客户创建与商机创建都补齐缺失字段追问、摘要确认、取消、失败保留和重试能力。
- 保持除“新增客户”“新增商机”之外的其他写入型请求继续走现有拒绝分支，不扩大机器人写入范围。

**Non-Goals:**

- 本轮不开放机器人修改、删除、批量导入、提醒创建、联系人补建、合同创建或任意字段更新。
- 本轮不改造 Web 登录、扫码登录、统一鉴权装载和会话 Cookie 机制。
- 本轮不自动把“客户不存在”串成“先建客户再建商机”的复合事务；两条创建链路分别独立确认。
- 本轮不补做完整的 CRM 产品主数据检索平台，只在机器人商机创建链路内解决“关联产品”最小可用映射问题。

## Decisions

### 决策一：在企业微信编排层新增独立的“受控创建”分支，而不是修改分析链路里的写入拦截

- 选择：在 `WecomAiConversationOrchestrationService` 与 `wecom-bot.service` 中新增独立的 create action 分支，用专门的状态机处理“新增客户”“新增商机”。
- 原因：现有分析链路中的 BLOCK 语义是“识别到写入就拒绝”，继续在该分支里硬塞创建流程，会把问数补问、日报跟进、写入确认三种语义混在一起，后续更难维护。
- 备选方案：沿用现有 analysis intent，命中“新增客户/新增商机”后在被拒绝前临时跳转。
- 不采用原因：容易造成补问状态、跟进状态与创建状态互相覆盖，也不利于后续为创建链路补独立审计事件。

### 决策二：客户创建与商机创建共用一套“待创建草稿”模型，但保留分开的字段规则

- 选择：新增统一的待创建草稿实体与会话内 work memory 字段，草稿中保存 `entityType`、当前阶段、已采集字段、缺失字段、摘要、失败原因和最终结果；客户与商机共用状态机骨架，但各自维护独立的字段校验与补问顺序。
- 原因：两条链路的交互骨架一致，都是“识别意图 -> 补齐字段 -> 摘要确认 -> 执行创建 -> 成功/失败回传”；统一草稿模型可以减少 `wecom-bot.service` 中重复分支。
- 备选方案：新增两套完全独立的 pending repository 和 work memory 字段。
- 不采用原因：实现成本高，且后续新增“取消/重试/重复确认拦截”时会重复维护两份状态流转逻辑。

### 决策三：机器人创建调用复用现有受控创建服务，不直接回放 HTTP 控制器

- 选择：机器人侧直接复用 `CrmCustomerApiService.createCustomer` 与 `CrmOpportunityApiService.createOpportunity`，由新的机器人创建服务负责把草稿 payload 转成服务入参。
- 原因：控制器层依赖 `SessionAuthGuard` 和 HTTP request 上下文，不适合在机器人内部直接伪造请求；直接复用服务更清晰，也能继续使用现有字段映射与错误标准化逻辑。
- 备选方案：机器人通过内部 HTTP 再调用 `/api/v1/crm/customers` 与 `/api/v1/crm/opportunities`。
- 不采用原因：会引入不必要的内部网络回路，并重复装配会话鉴权与请求对象。

### 决策四：机器人创建沿用内置 CRM 账号模式，但把能力从“跟进写回专用”提升为“企业微信受控创建/写回共用”

- 选择：抽出当前跟进写回里的内置 CRM 账号登录与 token 缓存逻辑，沉淀为可供跟进写回、客户创建、商机创建共用的服务；机器人创建默认不依赖发送人的 Web 登录态。
- 原因：企业微信消息天然没有稳定 Web 会话；如果客户/商机创建继续强依赖发送人 session token，这两条链路在真实移动场景几乎不可用。
- 备选方案：强制发送人先在 Web 端登录，再允许机器人继续创建。
- 不采用原因：会破坏“企业微信移动场景下即刻处理”的目标，也与现有跟进写回已经解决的问题重复倒退。

### 决策五：商机创建必须先唯一确认客户，并通过配置化产品别名映射拿到至少一个 productId

- 选择：商机创建中的 `customerId` 通过现有客户名称查询能力唯一化；`productAssets` 则通过新增的产品别名映射配置解析出 `productId`，同时支持用户显式提供产品 ID 作为兜底。
- 原因：现有仓库没有可直接复用的 CRM 产品检索服务，但商机创建又要求至少一条 `productId`；若不引入最小映射方案，机器人无法完成有效创建。
- 备选方案：在本次变更里同步新建完整的产品查询接口。
- 不采用原因：会显著扩大需求范围，并把本次重点从“机器人受控创建”变成“补齐产品主数据平台”。

### 决策六：创建成功前必须保留“摘要确认”，创建失败后必须保留草稿并允许修改/重试

- 选择：客户与商机创建都要求在执行 CRM 创建前先向用户展示结构化摘要，并支持“确认 / 修改 / 取消”；失败后草稿状态转为 `FAILED`，用户可继续补充字段或直接回复“重试”。
- 原因：这与当前商机跟进写回的使用习惯一致，也能覆盖字段识别不完整、客户命中错误、产品别名映射错误等高风险场景。
- 备选方案：字段齐全后立即执行创建，不再要求确认。
- 不采用原因：新增客户/新增商机一旦误写，后果比普通问数和跟进写回更重，缺少确认会放大误创建风险。

### 决策七：其它写入型请求继续拒绝，只对白名单内的两个意图开口

- 选择：分析风险拦截仍然保留，但在机器人入口前增加“新建客户”“新建商机”意图白名单。除这两类之外，其它新增/修改/删除/提醒/状态变更继续返回拒绝提示。
- 原因：本次需求只放开两个高频动作，不能因为新增状态机后把写入边界整体放松。
- 备选方案：把所有“创建”类意图都送入新的受控创建框架，再逐步补对象。
- 不采用原因：对象白名单、字段映射、权限边界和审计口径都还没有准备好，贸然扩口会破坏一期约束。

## Risks / Trade-offs

- [机器人创建改用内置账号后，实际执行人不再等于企微发送人] → 缓解：所有审计事件、草稿记录和成功回传都必须保留真实发送人、映射 CRM 用户和最终 owner 信息，避免“系统账号代操作”不可追踪。
- [商机创建缺少产品查询能力，用户提供的产品名称可能无法直接落成 `productId`] → 缓解：本次以“配置化产品别名映射 + 显式产品 ID 兜底”收敛范围，未命中时明确提示改走 Web 或补充可识别产品标识。
- [客户名称可能多候选，导致商机创建对象不唯一] → 缓解：复用现有客户查询列表和候选选择交互，未唯一命中前不得进入创建确认。
- [会话状态与现有问数/日报/跟进状态互相污染] → 缓解：受控创建单独维护 work memory 段，进入创建链路时优先消费当前 create state，不复用日报或分析 pendingSlots。
- [当前主规格仍声明机器人直接创建为拒绝项] → 缓解：实现时必须同步更新主规格、quickstart、tasks 与 OpenAPI 说明，避免文档与代码分叉。

## Migration Plan

1. 新建 OpenSpec 规格并评审通过，先统一能力边界、状态机和审计事件。
2. 抽出内置 CRM 账号 token 服务，确保跟进写回链路回归通过后，再接入客户/商机创建。
3. 为机器人创建增加待创建草稿仓储、会话 work memory 和审计事件枚举。
4. 先落客户创建，再落商机创建，最后补充“其它写入继续拒绝”的回归测试。
5. 同步更新 `specs/001-crm-intelligent-analytics/` 与 `contracts/openapi.yaml`，补齐验收场景后再启用真实环境联调。
6. 若上线后发现创建误判率或配置完整度不足，可回滚到“机器人继续拒绝新增客户/新增商机”的原有分支，不影响既有问数和跟进写回。

## Open Questions

- 产品别名映射最终采用环境变量、应用库配置还是治理后台维护；本次设计默认先用环境变量或本地配置文件落地。
- 机器人创建是否需要单独区分“谁有权新建客户/商机”的角色白名单，还是完全复用当前 CRM 角色权限映射；实现前需结合现有角色模型再做一次确认。
