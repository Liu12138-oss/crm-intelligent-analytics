## Context

当前前端基于 Vue 3、Vite、Pinia 和全局 `frontend/src/styles/main.css` 实现，页面已经覆盖登录、应用壳层、智能分析工作台、分析结果详情、智能合同审核、治理配置、模板管理、连接策略和审计检索。现有视觉仍大量使用传统白底蓝灰后台样式、行内样式和普通面板结构，与根级 `DESIGN.md` 当前定义的“浅色主导、Stripe 化企业工作台、统一内容骨架、响应式优先”定位存在明显差距。

本变更只规划 UI/UX 重设，不实施页面改造。后续实施必须保留现有数据流和交互函数，避免把视觉重构扩大为功能重构。

## Goals / Non-Goals

**Goals:**

- 按 `DESIGN.md` 将 Web UI 升级为浅色主导、统一内容骨架、可信结果呈现、风险显性化和响应式优先的企业工作台。
- 覆盖登录、应用壳层、分析链路、合同审核、治理配置和审计检索的页面级体验。
- 建立 CSS 设计令牌、可复用页面区块、交互状态、空状态、加载状态、禁用状态、焦点状态和移动端适配要求。
- 明确后续实施只能修改 Vue `template`、CSS class、静态体验文案和全局样式。

**Non-Goals:**

- 不修改 `<script setup>` 中的业务逻辑、函数签名、computed、watch、生命周期或接口调用。
- 不修改 Pinia store、service、router、types、后端 API、数据库、OpenAPI 契约或环境变量。
- 不新增页面路由，不改权限模型，不改变任何按钮、表单、上传、导出、追问、刷新、保存、筛选的业务行为。
- 不在本提案阶段直接修改 `frontend/` 页面实现。
- 登录页不得脱离现有 `LoginPage.vue` 的信息结构和主要文案口径另起方案。

## Decisions

### 决策 1：实施范围限定为模板与样式层

后续实施只允许调整 Vue `template` 结构、CSS class、全局样式和静态体验文案，不允许修改 `<script setup>` 逻辑。

原因：

- 用户明确要求不要修改功能代码。
- 当前页面已有可用数据流和交互函数，UI/UX 重设不需要改动接口和状态管理。
- 该边界便于验收：可以通过 diff 检查确认没有触碰 service、store、router、types 和后端。

备选方案：

- 仅改 CSS。风险最低，但无法重排首屏、结果详情和合同风险信息层级，体验提升有限。
- 新增纯展示组件。结构更清晰，但范围更大，容易引入组件接口设计和文件拆分争议。

### 决策 2：使用全局设计令牌驱动视觉重设

在 `main.css` 中建立与 `DESIGN.md` 对齐的 CSS 变量，覆盖颜色、渐变、字体层级、圆角、阴影、边框、动画时长和焦点状态。页面 template 只绑定语义化 class，避免继续堆行内样式。

原因：

- 当前多个页面存在大量行内色值和尺寸，难以统一视觉。
- 设计令牌可以让登录、壳层、分析、合同、治理和审计共享同一套视觉语言。

备选方案：

- 在每个页面单独写局部样式。问题是容易出现风格漂移。
- 引入新 UI 主题库。问题是会改变依赖和实现范围，不符合本次边界。

补充约束：

- 交互控件默认统一使用 `12px` 圆角，禁止同一页面混用多个控件圆角值。
- 状态标签、顶部 chip 和轻量提示胶囊必须默认不换行。

### 决策 3：按统一骨架和业务链路重排页面，而不是逐页换肤

后续实施应同时围绕“统一页面骨架”和“业务链路”两条主线执行：桌面端内容页默认采用 `左侧导航 + 顶部吸顶栏 + 主内容区 + 右侧信息区 / 抽屉`；登录页继续沿用现有登录结构，只在左侧增加精简辅助区，并保持右侧登录流程为主。

原因：

- `DESIGN.md` 的核心不是换颜色，而是让骨架、边界、交互反馈和主次层级统一。
- 逐页换肤会保留旧后台的信息架构，无法形成一致的企业工作台体验。

备选方案：

- 只把旧面板改成深色。问题是会变成表面换皮，仍然缺少统一骨架和可信体验。

### 决策 4：登录页保持现有信息基线，只增强左侧辅助区

后续实施中的登录页必须保留当前 `LoginPage.vue` 中的系统名称、副标题、登录标题、账号密码表单、企业微信登录切换、管理员信息和扫码区。允许增强的部分仅为左侧辅助区和整体氛围，且左侧文案必须精简、低饱和，不得生成新的营销文案体系。

原因：

- 当前用户已经明确要求沿用现有系统登录页布局和文案。
- 登录页是低风险、高频入口，不应在视觉重构时改变用户认知。

备选方案：

- 全新设计登录页。问题是会与当前系统口径脱节，并增加实现与验收歧义。

补充约束：

- 左侧辅助区与右侧登录面板必须共享同一视觉画布，不得表现为两个互不相关的漂浮卡片。
- 右侧登录面板仍应是页面主焦点；左侧辅助区只能承担次级识别和语义补充。

### 决策 5：所有交互事件绑定必须原样保留

后续 template 重排时，现有 `@click`、`v-model`、`:disabled`、上传事件、拖拽事件、导航事件和导出事件必须继续绑定到原函数或原状态，不得改名、拆分或迁移。

原因：

- 这是“不修改功能代码”的关键边界。
- 可以降低回归风险，并让验收聚焦视觉和体验。

## Risks / Trade-offs

- [只改 template 和 CSS 可能无法彻底优化组件边界] → 首版接受该限制，后续如需新增纯展示组件必须另行确认。
- [统一骨架可能与个别旧页面结构冲突] → 先以壳层、分析链路、治理页和合同页为主实施，特殊页按骨架原则兼容。
- [页面重排导致事件绑定遗漏] → 任务中要求逐页核对原有 `v-model`、`:disabled`、`@click`、上传和拖拽绑定。
- [大量行内样式迁移可能引入视觉回归] → 先建立全局 token 和语义 class，再按页面逐步替换。
- [不改脚本导致部分状态无法新增] → 只能使用现有状态表达 UI，不能为视觉目的新增业务状态。
- [响应式改造可能引入小屏布局折叠问题] → 后续实施必须将 `360px` 到 `1600px` 视为基本验证区间。

## Migration Plan

1. 新建 OpenSpec 提案产物并通过严格校验。
2. 后续实施阶段先改 `main.css` 设计令牌、统一圆角、标签不换行规则和基础组件 class。
3. 再按壳层、分析链路、治理审计、合同审核、登录页顺序改 template 与 class。
4. 每个页面改造后核对事件绑定、表单绑定和禁用状态未变。
5. 补齐桌面、中小屏和移动端断点验证，确保不出现横向溢出与关键交互丢失。
6. 最后运行前端 lint、单元测试和 build，并进行人工页面验收。
