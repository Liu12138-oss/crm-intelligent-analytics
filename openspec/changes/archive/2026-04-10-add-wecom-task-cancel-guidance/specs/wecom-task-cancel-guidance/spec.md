## ADDED Requirements

### Requirement: 企业微信机器人必须识别进行中任务的取消意图与任务切换意图
当企业微信会话中存在进行中的任务型流程时，系统 MUST 将“取消”“不做了”“先停一下”“先不用了”“不跟进了”等明确结束表达识别为当前任务的取消意图，而不是继续追问原任务内容。若用户在当前任务尚未完成时直接输入另一个已支持任务入口，系统 MUST 将该消息识别为任务切换意图，而不是把它当成旧任务内容。该规则 MUST 至少覆盖跟进模板收集、跟进写回确认、跟进群共享确认、受控新增客户、受控新增商机和日报确认类会话。

#### Scenario: 用户在跟进模板收集中表达不再继续
- **WHEN** 用户已经进入“今日跟进 / 跟进商机 / 跟进客户”的模板收集流程，并回复“先不跟进了”
- **THEN** 系统必须将该回复识别为当前跟进任务的取消意图
- **THEN** 系统不得继续追问“遇到与协助”“信息共享”或“拜访计划”等剩余栏目

#### Scenario: 用户在客户或商机创建确认前表达不再继续
- **WHEN** 用户正处于企业微信客户创建或商机创建的字段收集、摘要确认或失败重试阶段，并回复“先不做了”
- **THEN** 系统必须将该回复识别为当前创建任务的取消意图
- **THEN** 系统不得继续停留在原有字段补齐或确认追问分支

#### Scenario: 用户在跟进任务中直接输入另一个任务入口
- **WHEN** 用户正在企业微信跟进模板收集流程中补充内容，却直接发送“生成日报”或其它已支持的新任务入口
- **THEN** 系统必须将该消息识别为任务切换意图，而不是将其继续归入“跟进内容”等旧任务字段
- **THEN** 系统必须停止当前跟进模板补问，并转入新任务的入口处理

### Requirement: 取消当前任务或切换新任务前必须清理旧任务状态
系统 MUST 在识别到取消意图后结束当前任务，并清理与该任务相关的会话记忆、草稿状态和待确认上下文，使会话回到空闲状态。系统 MUST 在识别到任务切换意图后，先结束旧任务并清理旧任务相关状态，再进入新任务入口。若当前任务已有既定的取消审计、取消状态落库或幂等约束，系统 MUST 继续沿用既有约束，不得因为引入统一取消收口而绕过。

#### Scenario: 用户取消待确认的跟进写回
- **WHEN** 用户已经进入跟进写回确认阶段，并回复“取消”
- **THEN** 系统必须沿用既有跟进写回取消状态与审计逻辑，结束本次写回任务
- **THEN** 系统必须清理当前会话中的跟进写回、群共享和相关日报临时状态，避免后续消息继续命中上一轮写回上下文

#### Scenario: 用户取消创建草稿后发起新任务
- **WHEN** 用户在客户创建或商机创建流程中回复“停止”，随后立即发送一个新的机器人任务请求
- **THEN** 系统必须把后一条消息视为新的任务入口，而不是继续沿用上一轮创建草稿
- **THEN** 系统不得因为上一轮已取消的草稿残留而污染新任务识别

#### Scenario: 用户不先取消而是直接切换任务
- **WHEN** 用户正处于一个活跃任务中，未先回复“取消”，而是直接发送另一个已支持任务入口
- **THEN** 系统必须先清理旧任务相关状态，再按新任务入口继续处理
- **THEN** 系统不得要求用户必须先手动取消旧任务后才能继续新任务

### Requirement: 任务切换后必须优先进入新任务首轮提示
系统 MUST 在识别到任务切换意图后，优先输出新任务的首轮提示、预览或确认入口，而不是继续返回旧任务的缺失项追问。若需要向用户说明旧任务已被放弃，系统 MUST 使用友好中文过渡语气，但不得因为额外提示而阻塞新任务继续。

#### Scenario: 从跟进模板切换到日报任务
- **WHEN** 用户正在补“跟进内容”，但发送“生成日报”
- **THEN** 系统必须放弃当前跟进模板收集，并返回日报任务对应的首轮提示、预览或确认入口
- **THEN** 系统不得继续返回“请在跟进内容里把公司名或客户名写清楚”之类的旧任务提示

#### Scenario: 从创建任务切换到跟进任务
- **WHEN** 用户正在客户创建或商机创建流程中，却直接发送“跟进商机”
- **THEN** 系统必须结束当前创建任务，并返回跟进模板的首轮提示
- **THEN** 系统不得继续追问旧创建任务尚未补齐的字段

### Requirement: 未明确定义为新任务入口的消息不得破坏现有稳定流程
系统 MUST 只在消息明确命中另一个已支持任务入口时才触发任务切换。若用户发送的是当前任务的正常补充文本、字段续填内容、长句描述或未命中明确新任务入口的自然语言，系统 MUST 继续沿用当前稳定任务流程，不得误判为切换。该要求用于保护现有已稳定的跟进模板收集、跟进写回确认、客户创建、商机创建和日报确认链路。

#### Scenario: 跟进模板中的正常续填不得被误切换
- **WHEN** 用户仍在“今日跟进”模板收集中，发送“跟进内容：今天拜访完客户，晚上再整理日报”
- **THEN** 系统必须继续把这条消息视为当前跟进任务的补充内容
- **THEN** 系统不得因为消息中出现“日报”字样就切换到日报任务

#### Scenario: 创建任务中的字段续填不得被误切换
- **WHEN** 用户正在客户创建或商机创建流程中，继续发送“名称：联软科技，备注：后续需要补项目预算”
- **THEN** 系统必须继续按当前创建任务解析字段更新
- **THEN** 系统不得因为文本中出现其它业务词汇就结束当前稳定创建流程

### Requirement: 取消收口提示必须附带当前已支持功能列表
系统 MUST 在取消成功后返回统一的友好收口提示。该提示 MUST 先明确告知用户“当前任务已结束/已取消”，再以亲切中文语气提示机器人当前还能提供的帮助，并列出“当前已支持功能”清单。该清单 MUST 仅包含当前真实可用、面向企业微信用户可见的稳定能力，不得夹带尚未上线、未批准或已下线能力。

#### Scenario: 取消后返回能力引导
- **WHEN** 用户取消一个进行中的任务型会话
- **THEN** 系统返回的提示中必须同时包含“已结束当前任务”的确认语义和“当前已支持功能”的引导语义
- **THEN** 提示中必须列出当前稳定支持的企业微信机器人能力，而不是只返回一句“已取消”

#### Scenario: 当前没有活跃任务时用户发送停止表达
- **WHEN** 当前会话没有进行中的任务型流程，用户单独发送“先不做了”
- **THEN** 系统不得误清理无关上下文或伪造取消审计
- **THEN** 系统可以直接返回当前已支持功能清单，帮助用户重新选择可执行能力

### Requirement: 当前已支持功能清单必须使用统一能力目录维护
系统 MUST 为企业微信机器人维护一份统一的用户可见能力目录，供取消收口提示及后续同类帮助提示复用。新增、下线或调整企业微信机器人用户可见能力时，能力目录 MUST 与实际能力同步更新，避免多个入口分别维护不同版本的功能列表。

#### Scenario: 新增能力后取消提示自动使用新清单
- **WHEN** 系统新增一个已经正式上线的企业微信机器人用户可见能力，并同步更新统一能力目录
- **THEN** 后续取消收口提示必须自动展示更新后的能力清单
- **THEN** 实现方不得再额外修改多处散落的硬编码功能列表才能让提示生效

#### Scenario: 未上线能力不得出现在取消提示中
- **WHEN** 某项机器人能力仍处于提案、开发中或未获批准上线阶段
- **THEN** 统一能力目录不得将该能力标记为当前已支持功能
- **THEN** 取消收口提示不得提前向用户展示该能力
