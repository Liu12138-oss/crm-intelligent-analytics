## Context

当前企业微信机器人已经存在多条任务型会话链路，包括跟进模板收集、跟进写回确认与群共享、受控新增客户、受控新增商机，以及日报确认相关流程。现有实现里，各链路大多已经具备局部“取消”处理，但取消关键词、状态清理和提示文案分散在不同模块中，导致用户在中途回复“先不做了”“停止”“不跟进了”时，往往只能得到一条简单的“已取消”，看不到机器人此刻还支持哪些能力，也无法形成统一的结束体验。与此同时，活跃任务里若用户突然输入另一个已支持任务指令，例如在跟进模板里直接发送“生成日报”，系统也可能继续沿用旧任务的补问逻辑，把新任务入口误当成旧任务内容。

从实现上看，`WecomAiConversationOrchestrationService` 已经提供 `clearDailyReportMemory`、`clearFollowUpTemplateMemory`、`clearFollowUpWritebackMemory`、`clearCrmCreateMemory` 等记忆清理能力；`wecom-bot.service.ts` 中对日报主题入口也已经有“重新开始当前日报轮次”的局部处理。这说明本次变更不需要重做状态机，而是需要在 `wecom` 会话编排层增加统一的“取消收口 + 新任务切换”分支，并补上一份统一维护的用户可见能力清单。

## Goals / Non-Goals

**Goals:**

- 让企业微信机器人在识别到用户明确表达“取消 / 不做了 / 先停一下 / 不跟进了”等意图时，能够统一结束当前进行中的任务型会话。
- 让企业微信机器人在活跃任务中识别另一个已支持任务入口，并优先放弃旧任务、切换到新任务，而不是误把新任务指令当成旧任务内容。
- 复用现有任务记忆清理能力，在取消后把会话恢复到空闲状态，避免旧草稿、旧确认状态污染下一轮任务。
- 复用现有任务记忆清理能力，在任务切换前清理旧任务状态，使新任务从干净上下文开始。
- 在取消成功后，返回一段友好中文提示，并附上“当前已支持功能”清单，帮助用户立即发起其它已支持任务。
- 为“当前已支持功能”建立单一来源，供取消提示和后续其它兜底提示复用，减少文案漂移。
- 在仓库主文档和协作规约中补充同步要求，约束后续新增机器人能力时必须同步更新能力清单。
- 对已稳定的单任务主链路增加明确保护：未命中新任务入口时，系统继续沿用现有补问、确认和写回/创建逻辑，不因本次变更引入额外打断。

**Non-Goals:**

- 不在本次设计中扩展新的 CRM 读写能力，只处理现有已支持任务的取消收口体验。
- 不把“停止”扩展为中断已经提交到底层分析执行器的长查询取消协议；本次仅覆盖会话层的任务型流程。
- 不新增独立的权限模型、数据库表或外部依赖。
- 不改变现有各任务链路的业务确认边界、审计边界和 CRM 官方 API 调用约束。
- 不把所有包含新任务关键词的自然语言都强行解释为切换；仅处理明确、可判定的新任务入口表达。

## Decisions

### 决策一：将取消收口和任务切换都定义为企业微信会话层的统一前置分支，而不是继续分散到各任务实现内部

- 选择：在 `WecomBotService` 中增加统一的“活跃任务中断识别”逻辑，先判断当前会话是否存在活跃的跟进模板、跟进写回/群共享、客户创建、商机创建或日报确认任务；若命中，再优先识别这是“取消当前任务”还是“切换到另一个已支持任务”。
- 原因：当前各任务已经各自能处理局部取消，日报链路也已有局部“重新开始”处理，但关键词集合、切换优先级、提示文案和清理动作并不统一。把中断逻辑前置到会话层，能保证同一类用户表达在不同任务里得到一致结果，也能避免后续每新增一个任务都手写一套打断规则。
- 备选方案：继续在每个任务处理函数中单独扩展“取消后附带能力清单”或“命中新任务后重入”的逻辑。
- 不采用原因：实现容易重复，后续新增任务时也容易漏改，无法满足“以后添加其它功能也能自动带进能力提示”的目标。

### 决策二：复用现有任务记忆清理函数，新增一个统一的“清理当前活跃任务”封装

- 选择：新增一个面向会话层的共享清理方法，内部按活跃任务类型依次复用 `clearDailyReportMemory`、`clearFollowUpTemplateMemory`、`clearFollowUpWritebackMemory`、`clearCrmCreateMemory` 等既有方法；若任务涉及待写回记录或创建草稿状态，也继续沿用各自已有的取消审计和状态落库逻辑。
- 原因：现有记忆清理能力已经覆盖主要任务状态，本次重点是复用而不是重建。通过共享封装统一调用顺序，可以避免不同入口对“哪个状态该先清、哪个状态必须保留审计”理解不一致。
- 备选方案：为取消收口新增一套总清理逻辑，直接手动重置所有 work memory 字段。
- 不采用原因：风险高，容易误清理与当前任务无关的上下文，也会增加未来字段变更时的维护成本。

### 决策三：当一条新消息既像旧任务内容又命中新任务入口时，优先按“任务切换”处理

- 选择：在活跃任务状态下，若一条新消息同时满足“旧任务继续补充”的文本特征和“另一个已支持任务入口”的识别条件，系统优先按“任务切换”处理，先清理旧任务状态，再直接进入新任务首轮提示。
- 原因：截图中的“生成日报”就是典型例子。若仍按旧任务内容优先，会把新任务入口误写进旧任务草稿，用户需要先手动取消才能重试，体验很差。现有日报链路已经证明主题入口重入是合理的用户行为，这次只是把它统一扩展到跨任务场景。
- 备选方案：要求用户必须先明确回复“取消”，再发送新任务。
- 不采用原因：这会把机器人从“自然对话助手”退回成严格命令机，无法满足用户随时切换任务的真实使用习惯。

### 决策四：任务切换只对“明确的新任务入口短句”生效，普通续填文本继续走原稳定链路

- 选择：任务切换识别优先使用现有主题入口与创建入口识别器，只在活跃任务中对“短句、命中已支持任务入口、且不包含明显旧任务字段续填结构”的消息生效；诸如“跟进内容：今天拜访完后我想晚上再生成日报”这类长句或字段续填内容，仍按当前任务文本处理。
- 原因：系统稳定性优先。若把所有含“日报”“客户”“商机”等字样的文本都视为切换信号，很容易把正常业务表达打断，影响已经稳定的收集和确认流程。
- 备选方案：只要消息中包含任一新任务关键词，就立即切换任务。
- 不采用原因：误判率高，会直接破坏现有稳定功能。

### 决策五：将“当前已支持功能”收敛为机器人侧统一能力目录，并由取消提示动态拼装

- 选择：在企业微信提示文案配置层新增统一的能力目录结构，例如分组标题、功能名称、触发短语示例和简短说明；取消收口提示只负责读取并渲染该目录，而不在业务代码里硬编码多份列表。
- 原因：用户明确提出“以后添加的其它功能自动添加进来”，要做到这一点，就不能把能力列表散落在各条提示里。统一目录可以让新增功能时只维护一处配置，同时方便 README 中的能力清单按同一口径同步。
- 备选方案：在取消提示文案里直接写死当前支持的几个能力点。
- 不采用原因：一旦后续再加“查看日报”“新的受控写入能力”之类功能，极易出现代码提示、README、需求文档三处不一致。

### 决策六：取消提示采用“已结束当前任务 + 欢迎继续使用 + 能力清单”的固定结构；任务切换提示采用“已放弃上一任务 + 进入新任务”的固定结构

- 选择：取消后的用户提示固定包含三段信息：第一段确认已结束当前任务；第二段使用亲切、非责备语气表达“如果需要，小助手愿意继续帮忙”；第三段按统一能力目录列出当前支持功能。任务切换时，则先用一句过渡语说明“上一任务已结束/先放下”，随后直接输出新任务的首轮提示，不额外再附能力清单，避免信息过长。
- 原因：取消和切换都是“中断当前任务”，但用户预期不同。取消后用户需要重新选择能力；切换时用户已经明确给了新目标，系统应该尽快进入新任务而不是再给一遍总目录。
- 备选方案：取消和任务切换都统一返回能力清单，再让用户重新点选下一步。
- 不采用原因：会拖慢已明确的新任务入口，造成“我都说了生成日报，你还让我重新选”的反直觉体验。

### 决策七：将能力清单文档同步要求写入 README 与 AGENTS.md，而不是只留在 OpenSpec 提案里

- 选择：在 `README.md` 中增加“企业微信机器人当前支持能力”清单，面向仓库使用者给出当前稳定能力；同时在 `AGENTS.md` 中增加同步规约，要求以后凡是新增、下线或调整企业微信机器人用户可见能力、主题入口或取消提示能力列表时，必须同步更新主文档和统一能力目录。
- 原因：OpenSpec 提案只约束本次 change，无法天然约束后续所有协作者。把规则写进 README/AGENTS，才能形成长期协作约束。
- 备选方案：仅在本次 change 的 tasks.md 里要求同步文档。
- 不采用原因：任务清单在 change 归档后不再是协作者的第一入口，约束力度不足。

## Risks / Trade-offs

- [取消关键词过宽，误伤正常输入] → 仅在存在活跃任务会话时启用统一取消识别；普通问数场景不因为一句“先不看这个”就误触任务取消。
- [任务入口关键词过宽，导致误切换任务] → 仅在活跃任务中对“短句、主题入口型、新任务关键词明确”的消息启用任务切换识别；长段自然语言仍优先按当前任务内容处理，除非明确命中新任务入口。
- [任务切换逻辑破坏现有稳定补问/确认流程] → 为跟进模板续填、跟进写回确认、客户创建字段补齐、商机创建字段补齐、日报确认等现有稳定路径补充“不切换”的回归测试，确保默认行为不变。
- [能力目录与真实能力漂移] → 取消提示改为只读统一目录，并在 README/AGENTS 中补充强制同步要求；测试中增加能力目录渲染校验。
- [一个会话同时挂着多种任务状态，清理顺序不一致] → 明确优先级，先清理最强约束的待写回/群共享，再清理日报与模板草稿，最后清理创建草稿，避免残留悬空状态。
- [取消收口文案过长，影响企业微信阅读体验] → 使用固定分组和短句说明，只列稳定可见能力，不在取消提示里展开长篇帮助说明。

## Migration Plan

1. 在企业微信提示配置中抽取统一的“当前已支持功能”目录和取消收口文案构造函数。
2. 在 `WecomBotService` 中增加活跃任务识别、统一取消分支和新任务切换分支，并复用既有审计、状态落库和记忆清理方法。
3. 为跟进模板、跟进写回、群共享、客户创建、商机创建、日报确认等任务场景补充取消收口和任务切换集成测试，同时补充“不应误切换”的回归用例。
4. 同步更新 `README.md` 与 `AGENTS.md`，补充当前能力清单和后续维护要求。
5. 如上线后发现取消识别误判，可先关闭统一取消分支，回退到各任务原有的局部取消提示，不影响现有业务主链路。

## Open Questions

- “停止/不做了”是否只在任务型会话中生效，还是未来也要扩展到长查询执行中的主动中断？本次设计先不覆盖执行器级取消。
- 能力清单中是否要展示“查看今日日报”“日报提醒/确认”这类系统也会主动触发的能力，还是只展示用户可主动发起的入口？本次建议优先展示用户可直接触发的能力。
