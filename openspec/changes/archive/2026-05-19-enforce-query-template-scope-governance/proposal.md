## Why

当前常用查询模板允许管理员或运营人员编写较自由的业务 SQL，但执行安全仍依赖模板作者主动写入权限占位符、正确选择部门口径和避免静态全量目标清单。牛劲案例已经暴露出风险：查询明细按负责人范围过滤，但展示团队按客户部门聚合，导致山东区负责人看到其它区名称和汇总数据。

需要把“自由编写 SQL”和“受控权限执行”解耦：模板作者可以专注业务查询，系统必须在保存、校验和执行阶段自动识别、注入、校验和审计权限范围，不能把权限边界交给人工记忆。

## What Changes

- 新增查询模板范围治理能力，支持自由 SQL 模板的主业务表识别、自动权限注入、固定范围兼容性校验和复杂模板上线阻断。
- 查询模板执行层统一消费当前用户 `scopeSnapshot`，对未声明范围的模板自动按组织、部门、负责人范围收口。
- 对已经写死部门 ID、部门名称、负责人 ID 或静态团队清单的模板，执行前必须判定为固定范围或复杂范围模板，并确认执行人权限覆盖该范围；无法确认时阻断执行。
- 校验结果展示口径与权限过滤口径是否一致，例如不能用 `opportunities.department_id` 过滤却用 `customers.department_id` 作为团队展示维度，除非模板显式声明并通过治理审核。
- 对静态目标表、`UNION ALL` 团队清单、复杂 CTE、跨主表聚合等模板增加治理元数据：权限主表、展示部门口径、固定范围说明、管理员审核状态和上线风险等级。
- 补齐 Web 治理后台的校验反馈，让模板作者看到中文风险提示和修复建议，而不是要求其记住 `:scopeDepartmentIds`、`:scopeOwnerIds` 等内部参数。
- 补齐牛劲/山东区类场景回归测试，确保部门负责人执行全量模板时只返回其授权部门或团队数据，管理员仍可查看全量。

## Capabilities

### New Capabilities

- `query-template-scope-governance`: 定义自由 SQL 查询模板的范围识别、自动注入、固定范围校验、显示口径一致性检查、复杂模板审核和治理反馈要求。

### Modified Capabilities

- `controlled-analysis-orchestration`: 补充常用查询模板执行前必须消费统一范围快照，并在无法证明结果范围安全时阻断执行。
- `organization-scope-governance`: 补充分析场景下多部门、多岗位用户的默认范围裁剪要求，避免直属上级链把非销售或非默认业务部门成员带入问数范围。

## Impact

- 后端：`query-assets` 模块的模板校验、模板执行、SQL 范围分析、SQL 注入、展示结果组装和最近查询重跑；`analysis` 模块的统一范围快照消费与审计链路。
- 前端：查询模板治理页面的 SQL 校验结果、风险提示、复杂模板审核信息和模板预览反馈。
- 数据与配置：应用存储中的查询模板记录需要新增或迁移治理元数据，例如范围模式、权限主表、展示部门字段、固定范围声明和审核状态。
- 测试：新增模板范围分析单元测试、模板执行集成测试、治理页面交互测试，以及牛劲/山东区权限回归样例。
- 文档：更新主规格、快速验证或治理说明，明确模板作者可自由写业务 SQL，但执行必须由系统统一受控收口。
- 发布策略：新增 `QUERY_TEMPLATE_SCOPE_GOVERNANCE_MODE=observe|enforce` 灰度开关；观察模式记录风险但不阻断普通用户，强制模式按治理结论阻断或放行。
