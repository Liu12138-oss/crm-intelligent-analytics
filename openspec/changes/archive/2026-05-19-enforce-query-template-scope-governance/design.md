## Context

当前常用查询模板通过 `query-assets` 模块保存和执行，模板 SQL 可以在治理后台维护。现有执行链路已经有只读校验、命名参数编译、`scopeSnapshot` 注入和 SQL 审计，但权限收口仍存在三个薄弱点：

1. 模板作者需要理解并主动使用 `:scopeOrganizationIds`、`:scopeDepartmentIds`、`:scopeOwnerIds` 等内部参数，否则只能依赖执行层自动注入。
2. 自动注入主要面向简单主表查询，对静态目标表、复杂 CTE、跨主表聚合和展示口径不一致的模板缺少强校验。
3. 结果展示层直接消费 SQL 返回的行与列，无法判断“展示团队名称”是否来自与权限过滤相同的业务归属字段。

牛劲案例说明，模板即使带有范围条件，也可能因为展示维度来自客户部门、权限过滤来自商机负责人或商机部门，导致用户看到其它区名称。该问题属于权限边界和可信展示问题，必须在模板治理、执行和展示三层共同收口。

## Goals / Non-Goals

**Goals:**

- 允许模板作者继续自由编写业务 SELECT SQL，不要求其记忆内部权限参数。
- 对简单模板自动识别主业务表并注入当前用户范围。
- 对固定部门、固定负责人、静态团队清单和复杂模板进行风险识别、兼容性校验或上线阻断。
- 强制校验展示维度与权限过滤口径一致，避免“按 A 字段过滤、按 B 字段展示”的越界误导。
- 为治理后台提供中文校验结果、风险等级和修复建议。
- 保证模板执行、预览、最近查询重跑和导出补查消费同一份范围判定结果。
- 在分析场景下裁剪多部门、多岗位用户的默认负责人范围，直属上级链结果必须与用户默认业务部门、销售部门或白名单范围相交后才能用于查询。

**Non-Goals:**

- 不在本变更中重写 CRM 登录、会话和基础身份认证链路。
- 不开放任意写入 SQL，不改变只读边界。
- 不把所有历史模板一次性改造成统一业务口径；本变更提供迁移机制和高风险模板阻断策略。
- 不引入新的独立权限模型，仍复用 `scopeSnapshot`、企业微信组织事实、CRM 映射和数据范围白名单。
- 不把运行态治理快照中的完整 SQL 重复长期存储到每条模板记录中；模板记录只保存摘要、分类、风险、审核元数据和快照哈希，原始 SQL 仍由模板自身字段承载。

## Decisions

### 决策 1：将模板范围治理拆成“分析、分类、执行计划”三段

模板保存或执行前先生成 `TemplateScopeGovernanceSnapshot`，包含：

- `scopeClassification`：`AUTO_SCOPABLE`、`DECLARED_DYNAMIC_SCOPE`、`FIXED_SCOPE`、`COMPLEX_REVIEW_REQUIRED`、`UNSAFE_SCOPE`
- `primaryDataSources`：识别出的主业务表及别名，例如 `opportunities o`、`contracts co`
- `scopePredicateSources`：当前 SQL 中已存在的组织、部门、负责人范围条件
- `displayDimensionSources`：团队、部门、负责人等展示维度来源字段
- `staticDimensionSources`：静态 `UNION ALL`、硬编码部门名、硬编码目标团队清单
- `riskFindings`：中文风险项、严重级别和修复建议
- `requiredReviewStatus`：是否需要管理员确认后才能发布

原因：单纯“有没有 `department_id` 字段”不足以判断安全性。必须同时判断范围过滤、展示维度和静态维度来源。

备选方案：继续要求模板作者手写权限占位符。该方案成本低，但无法解决“作者不知道规则”和“显示口径不一致”的根因。

### 决策 2：简单模板优先自动注入，不要求作者写内部参数

当 SQL 满足以下条件时，系统将其判定为 `AUTO_SCOPABLE`：

- 只读 SELECT。
- 存在明确主业务表，优先级为 `opportunities`、`contracts`、`customers`、`users`。
- 没有硬编码部门/负责人范围。
- 没有静态团队清单或无法解释的跨主表部门展示。
- 展示团队或部门字段来自同一个主业务表归属，或来自已批准的等价归属映射。

执行时由系统自动追加范围条件，不要求模板 SQL 出现 `:scopeDepartmentIds` 或 `:scopeOwnerIds`。

原因：模板作者应写业务 SQL，权限边界应由平台强制保障。

备选方案：保存时自动把 SQL 文本改写并持久化。该方案可见性强，但会让作者维护的 SQL 和实际 SQL 混淆；本设计只在执行计划中生成受控 SQL，并在预览中展示“已注入范围”的只读说明。

### 决策 3：固定范围模板允许存在，但必须做权限兼容校验

如果模板显式写死部门 ID、部门名称、负责人 ID 或角色相关固定范围，系统将其判定为 `FIXED_SCOPE`。执行时必须满足：

- 固定组织、部门、负责人范围可被解析为结构化范围。
- 当前执行人的 `scopeSnapshot` 覆盖该固定范围，或当前执行人为全量分析授权用户。
- 审计记录保存固定范围声明和兼容性判断结果。

无法解析固定范围时，模板不得对普通用户执行。

原因：业务上可能确实存在“山东区固定看板”这类模板，不能一刀切禁止，但必须禁止其它部门负责人复用。

### 决策 4：复杂模板进入“审核后发布”，高风险模板直接阻断

以下模板默认进入 `COMPLEX_REVIEW_REQUIRED`：

- 使用静态目标表、静态团队清单或 `UNION ALL` 枚举团队。
- 多个主业务表分别按不同部门字段聚合。
- CTE / 子查询中存在多个可收口入口，系统无法自动判断主口径。
- 展示字段来自客户部门，而过滤字段来自商机或合同归属部门。

治理人员必须补充元数据后才能发布：

- 权限主表。
- 权限主表别名。
- 展示部门口径字段。
- 固定范围说明。
- 目标表收口策略。
- 审核结论和原因。

以下模板直接进入 `UNSAFE_SCOPE` 并阻断：

- 写入或 DDL。
- 动态拼接 SQL。
- 未白名单表字段。
- 运行时无法绑定权限且无法审核解释的跨库/跨源读取。

原因：静态目标和复杂聚合是高价值模板常见形态，允许审核后上线；但无法证明安全的模板必须阻断。

### 决策 5：展示口径必须成为权限校验的一部分

`QueryResultPresentationService` 或其前置服务必须消费治理快照，检查结果行中的团队/部门展示字段是否来自已批准的 `displayDimensionSources`。对转置表、矩阵表、图表 series 也执行同一判断。展示字段来源不能只靠别名字符串判断，必须追踪 `CASE`、函数、`GROUP BY`、CTE 输出列和子查询输出列到基础表字段，例如 `CASE ... d.name ... END AS team_name` 必须识别到 `departments d`，再通过 `d.id = customers.department_id` 判断该展示口径来自客户部门。

对本次暴露的问题，模板如果按 `o.department_id` 或 `co.department_id` 收口，却按 `cu.department_id` 展示团队，必须至少满足一条：

- 模板声明“客户归属部门即业务展示口径”，并经管理员审核通过。
- SQL 额外把展示部门也收口到当前范围。
- SQL 改成按主业务对象归属部门展示。

原因：用户看到的是结果名称和表头，不只是底层明细。展示越界会造成实际业务误解。

### 决策 6：历史模板采取分级迁移

迁移流程：

1. 启动时为旧模板补齐治理快照字段，未校验模板标记为 `PENDING_VALIDATION`。
2. 对低风险模板自动标记为 `AUTO_SCOPABLE`。
3. 对静态目标、硬编码部门和展示口径不一致模板标记为 `COMPLEX_REVIEW_REQUIRED`，普通用户执行时阻断或仅允许管理员预览。
4. 对已知高风险模板建立迁移脚本或治理任务，修正 SQL 口径后重新发布。

原因：生产已有模板不能因字段缺失直接 500，也不能继续静默越权。

### 决策 7：分析范围先按业务部门裁剪，再进入模板治理

`OrganizationScopeService` 生成分析用 `scopeSnapshot` 时，默认负责人范围 MUST 先按当前用户默认部门及其子部门裁剪；当用户存在多个岗位或直属链跨技术、服务、职能线时，只有与用户业务部门、销售部门或数据范围白名单相交的负责人可以进入默认问数范围。白名单授权部门继续叠加，但必须在范围说明和审计中标明来源。

原因：牛劲案例中 `ownerIds` 过宽不是模板 SQL 能完全兜底的问题。如果执行层继续消费过宽 `scopeSnapshot`，即使模板自动注入也会把其它区负责人纳入结果。范围治理必须把“谁能查”与“模板怎么查”同时收紧。

### 决策 8：执行阻断采用灰度开关

新增 `QUERY_TEMPLATE_SCOPE_GOVERNANCE_MODE`：

- `observe`：记录治理快照、风险项和审计说明，但不因复杂模板阻断普通用户，适合测试环境和首次上线观察。
- `enforce`：对 `UNSAFE_SCOPE`、未审核 `COMPLEX_REVIEW_REQUIRED`、固定范围不兼容、展示口径不一致模板执行阻断。

未配置时默认 `observe`，避免新能力上线时对历史高频模板产生不可控阻断；自动化测试需要验证强制阻断时，应在用例中显式设置 `QUERY_TEMPLATE_SCOPE_GOVERNANCE_MODE=enforce`。

### 决策 9：旧结果导出必须重新确认当前治理结论

导出既有 `AnalysisResultRecord` 时，如果记录来自模板执行且保存了模板 ID，导出链路 MUST 重新读取当前模板并计算当前治理摘要。强制模式下，若当前模板已被判定为不可对该用户执行，导出必须阻断并提示重新执行查询；观察模式下允许导出当前页面结果，但审计中必须记录旧结果治理风险。

## Risks / Trade-offs

- 风险：SQL AST 解析无法覆盖所有 MySQL 语法。
  缓解：解析失败默认进入复杂审核或阻断，不默认放行；治理后台展示解析失败原因。

- 风险：自动注入可能改变模板作者预期的统计口径。
  缓解：模板预览展示“当前执行范围”和“已注入范围说明”；审计保留原始 SQL 与执行 SQL。

- 风险：历史高价值模板因复杂审核被临时阻断。
  缓解：按点击量和风险等级排序迁移，先处理常用模板；管理员可在审核后有限发布。

- 风险：展示口径一致性判断过严，阻断合法的客户归属分析。
  缓解：允许模板显式声明客户归属部门为业务展示口径，但必须同时说明适用场景并通过管理员审核。

- 风险：多部门、多岗位用户的默认范围裁剪过严。
  缓解：以用户默认部门及其子部门为默认业务范围，数据范围白名单可显式扩展；审计中记录裁剪前后负责人数量，便于治理人员排查。

## Migration Plan

1. 增加模板治理快照类型和兼容迁移逻辑，确保旧模板读取不报错。
2. 扩展 SQL 范围分析器，输出分类、主表、范围条件、展示字段和静态维度风险。
3. 执行层先以“记录风险但不阻断”模式跑本地和测试环境，确认历史模板分类结果。
4. 对生产高频模板进行治理迁移，优先处理新增商机月度分布、各团队完成预测、客户维度提单等涉及团队展示的模板。
5. 开启普通用户执行阻断：`UNSAFE_SCOPE` 和未审核的 `COMPLEX_REVIEW_REQUIRED` 不允许执行。
6. 开启治理后台强校验：新建或更新模板必须通过校验或完成审核字段后才能发布。
7. 发布后清理分析能力快照、最近查询缓存或强制重跑，避免历史越界结果继续复用。

回滚策略：

- 保留旧模板字段，不破坏已有模板数据。
- 若上线后误阻断过多，可临时把复杂模板改为仅管理员可执行，同时保留自动注入和高风险阻断。
- 不允许回滚到“普通用户可执行无法证明范围安全的模板”。

## Open Questions

- 是否需要在治理后台提供“模拟某个用户执行模板”的预览入口，用于验证不同部门负责人的收口结果。
- 静态目标表是否应长期保存在 SQL 中，还是迁移为受控目标配置表并按部门 ID 关联。
- 客户归属部门、商机归属部门、合同归属部门在不同报表中的默认展示口径是否需要新增统一业务字典。
