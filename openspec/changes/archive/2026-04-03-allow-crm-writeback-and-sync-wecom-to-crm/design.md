## Context

当前系统的规格、README、架构说明和协作约束把“CRM 全局只读”作为系统级前提，但这个前提已经与业务目标脱节。后续系统不仅要支持查询，还要逐步支持写入型能力，如果继续保留“全局只读”口径，会导致每次新增写入能力时都要先推翻总约束。

企业微信机器人当前的认证主链路本身并不复杂：机器人收到 `from.userid` 后，优先通过 CRM 原生 `wx_user_maps + wx_users` 映射到 `users.id`。真正的问题是企业微信目录数据没有正式写回 CRM 原生 `wx_*` 表，导致机器人认证链路长期处于“映射表为空”的状态。

仓库中已有 `get_userid.py`，它证明了在当前企业微信应用凭证下，可以通过 `gettoken -> department/simplelist -> user/list` 拉到成员 `userid / mobile / email`。这说明当前问题不是接口能力不足，而是现有实现把同步结果放到了应用侧镜像与内存仓库，没有把可用数据沉淀到 CRM 原生身份映射源。

## Goals / Non-Goals

**Goals:**

- 从规格层面删除“CRM 全局只读”这一系统级约束。
- 明确新的权限边界：分析链路继续只读，同步链路与未来写入链路按能力单独定义可写范围。
- 复用项目现有企业微信凭证，基于企业微信官方通讯录接口拉取部门成员 `userid / mobile / email`。
- 将企业微信成员与映射正式写回 CRM 原生 `wx_users`、`wx_user_maps`，必要时补齐 `wx_departments`、`wx_user_department_maps`。
- 让企业微信机器人继续复用 CRM 原生 `wx_user_maps -> wx_users` 认证主链路，不再依赖应用侧内存镜像作为长期认证来源。
- 为生产环境提供一键同步与排障口径，能回答“为什么某个企业微信用户仍未绑定 CRM 身份”。

**Non-Goals:**

- 不在本次设计中放开 AI 问数链路对 CRM 业务表的写权限。
- 不在本次设计中定义具体业务写入能力，如商机更新、任务创建、提醒写入等。
- 不保留“应用侧镜像表作为认证主来源”的双轨方案。
- 不新增第二套企业微信凭证配置，默认继续复用项目现有凭证。

## Decisions

### 决策一：删除“CRM 全局只读”，改为“链路级读写边界”

系统级规则不再表述为“CRM 全局只读”，而改成两层边界：

- 分析问数链路：仍然必须只读，继续保留白名单、AST 校验、风险拦截和审计。
- 企业微信组织与身份映射同步链路：允许受控写入 CRM 指定 `wx_*` 表。

这样做的原因：

- 后续写入型功能已经是确定方向，继续保留“全局只读”会不断与真实需求冲突。
- 当前真正需要写 CRM 的不是 AI 结果，而是企业微信身份基础数据。
- 把写权限限定在链路和表范围内，比简单删掉只读口径更可控。

备选方案是直接保留“全局只读”，再把企业微信同步视为临时例外。该方案会让规格长期自相矛盾，不采用。

### 决策二：企业微信同步正式写回 CRM 原生 `wx_*` 表

目录同步的正式落点从应用侧镜像表切换为 CRM 原生表，第一阶段至少覆盖：

- `wx_users`
- `wx_user_maps`

第二阶段视需要补齐：

- `wx_departments`
- `wx_user_department_maps`
- `wx_organization_maps`

这样做的原因：

- 机器人认证当前已经优先依赖 CRM 原生 `wx_user_maps + wx_users`。
- 只要同步把原生映射表填好，机器人认证主链路基本不用推翻。
- 应用侧镜像当前还有内存态实现，重启即丢，不适合作为长期身份源。

备选方案是继续增强 `wecom_sync_users` 作为兜底认证源。该方案会造成“CRM 原生映射”和“应用镜像映射”长期并存，排障复杂，不采用。

### 决策三：同步接口优先对齐 `get_userid.py` 已验证链路

本次同步主链路优先采用以下调用顺序：

1. `gettoken`
2. `department/simplelist`
3. `user/list`

必要时再追加：

4. `user/get`

这样做的原因：

- `get_userid.py` 已经证明这组接口在当前企业微信配置下可拿到 `userid / mobile / email`。
- 先对齐已验证链路，比继续围绕 `user/get` 或 OAuth2 做推测更稳。
- `user/get` 作为补齐接口保留，但不阻塞首版上线。

备选方案是继续以 `department/list + user/get` 为主。该方案并非不可行，但与现有已验证脚本不一致，会增加排障成本，不作为首选。

### 决策四：CRM 用户绑定只允许“唯一命中自动生成映射”

从企业微信成员到 CRM 用户的自动绑定规则采用保守策略：

1. 先按手机号唯一匹配 `users.phone`
2. 再按邮箱唯一匹配 `users.email`
3. 只有唯一命中时才生成或更新 `wx_user_maps`
4. 零命中保留 `wx_users` 记录但不自动绑定
5. 多命中不自动绑定，记录冲突并等待人工处理

这样做的原因：

- 当前目标是恢复机器人认证可用性，不是追求全自动全覆盖。
- 手机号和邮箱是最稳定的结构化字段，且已有业务基础。
- 自动覆盖或多命中强绑的风险远高于“暂不绑定”。

备选方案是增加姓名唯一匹配作为自动绑定规则。该方案短期命中率更高，但误绑风险显著更高，不作为首版默认策略。

### 决策五：生产运维入口从“镜像同步”改成“CRM 原生映射同步”

生产环境的一键同步入口和状态说明都要围绕“CRM 原生映射是否已更新”展开，而不是只看应用侧镜像是否有数据。

状态摘要至少要覆盖：

- 本次拉取成员数
- 写入 `wx_users` 数
- 新增或更新 `wx_user_maps` 数
- 手机号/邮箱缺失数
- 零命中 CRM 数
- 多命中冲突数
- 指定 `userid` 的失败原因

这样做的原因：

- 真实问题已经从“能否拉到通讯录”转变为“为什么 CRM 里还没有可用绑定”。
- 运维和业务排障最终关心的是机器人能否认证，不是应用镜像是否刷新。

## Risks / Trade-offs

- [删除“CRM 全局只读”后，团队可能误解为所有链路都可写 CRM] → 在规格和文档里明确替换为“链路级读写边界”，并列出分析链路仍只读。
- [把同步结果直接写回 CRM 原生表，会触及现有业务系统数据] → 仅允许写 `wx_*` 映射表；不允许写业务表；同步写入必须具备幂等与审计。
- [手机号/邮箱可能为空、重复或历史脏数据] → 自动绑定仅允许唯一命中；零命中和多命中都保留待处理结果。
- [现有企业微信凭证未来可能不再具备完整通讯录字段权限] → 保留同步前字段探测与失败摘要，一旦返回字段变空可快速定位到权限问题。
- [旧的应用镜像与新的 CRM 原生同步会并存一段时间] → 在迁移期保留镜像表仅用于观测，不再作为长期认证主来源。

## Migration Plan

1. 先更新规格、README、架构文档与协作约束，删除“CRM 全局只读”口径，改成链路级读写边界。
2. 在后端同步服务中新增 CRM 原生 `wx_*` 写回能力，保留现有企业微信拉取链路但调整落点。
3. 为 `wx_users` 建立幂等 Upsert 逻辑，为 `wx_user_maps` 建立唯一匹配绑定逻辑。
4. 调整机器人认证服务，继续优先读取 CRM 原生 `wx_user_maps + wx_users`，弱化应用侧镜像兜底。
5. 更新生产一键同步脚本与状态摘要，支持直接验证 CRM 原生映射是否已生成。
6. 在生产环境先对“联软科技集团”授权范围执行一次全量同步，再验证目标账号是否可被机器人识别。

回滚策略：

- 若 CRM 原生写回逻辑异常，可回滚到上一版代码，并暂停一键同步入口。
- 已写入的 `wx_users`、`wx_user_maps` 如需回退，应通过受控脚本按本次同步批次或时间窗恢复，不允许手工散改。

## Open Questions

- `wx_organization_maps` 当前在生产 CRM 中是否已有稳定映射，可否直接复用现有 `wx_organization_id`？
- 首版是否必须同步 `wx_departments`、`wx_user_department_maps`，还是先只做 `wx_users + wx_user_maps` 即可满足机器人认证？
- 生产环境是否需要额外的“未自动绑定用户清单”导出或后台检索入口，供管理员集中处理多命中与零命中问题？
