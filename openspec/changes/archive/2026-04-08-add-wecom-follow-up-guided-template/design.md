## Context

当前企业微信“跟进商机 / 跟进客户”链路已经能够复用既有受控跟进写回能力，完成对象识别、确认、审计和 CRM 官方 API 写回，但入口提示仍偏向“请直接说一句跟进内容”。这对熟悉系统的用户效率高，对大量只会先说“跟进客户”“跟进商机”的用户并不友好，常见问题包括：

1. 公司名或客户名没有写清，后续对象识别需要反复澄清；
2. 用户只写了一句跟进内容，协助诉求、信息共享、拜访计划等容易遗漏；
3. 现有待写回记录只保存 `draftContent`，缺少一个在正式生成草稿前承接结构化填写的轻量层；
4. 若未来要基于企业微信跟进内容自动总结日报，当前自由文本输入过于松散，后续还要再做一次拆分和语义归类。

仓库中已有“新增客户 / 新增商机”的分步采集与摘要确认模式，也已有“跟进写回必须保留姓名前缀并继续走 CRM 官方 API”的明确约束。本次设计的重点，是在不改变现有受控写回边界的前提下，为泛化的跟进入口增加一层模板式引导，同时保留足够稳定的字段语义，便于后续日报能力直接复用。

## Goals / Non-Goals

**Goals:**

- 当用户只提出“跟进商机”“跟进客户”等主题请求时，先返回结构化填写模板，并明确提醒写清公司名或客户名。
- 支持用户按模板逐项补充，或一次性粘贴完整模板内容，再统一整理成最终跟进草稿。
- 聊天模板中不单独展示姓名字段；最终草稿改为 `【姓名】：` 单独起行，再把模板中的补充栏目按固定顺序并入正文。
- 在模板收集阶段保留 `跟进内容`、`遇到与协助`、`信息共享`、`拜访计划` 四类语义边界，方便后续日报能力直接总结与复用。
- 继续复用现有对象识别、确认、失败保留、审计和 CRM 官方 API 写回链路。
- 优化为更亲切、鼓励补充细节的中文提示文案，但不放宽写回权限与执行边界。

**Non-Goals:**

- 不新增任何绕过确认的自由写回入口，也不扩大跟进写回对象范围。
- 不替换用户已经提供完整跟进正文时的快速路径，避免把所有跟进请求都强制改成多轮表单。
- 不引入新的 CRM 数据库直写、旁路接口或独立权限模型。
- 不在本次设计中直接生成日报、重做日报链路、群分享逻辑或 Web 端交互。

## Decisions

### 决策一：仅对“主题型跟进入口”展示空模板，保留已有详细口述的快速路径

- 选择：当用户发送“跟进商机”“跟进客户”“写跟进”等泛化主题请求、且消息中尚未包含足够完整的跟进正文时，系统先返回模板；如果首条消息已经带有明确对象和较完整的跟进内容，则继续沿用现有快速识别与确认流程。
- 原因：这次需求的核心是降低“从空白开始写跟进”的门槛，而不是把熟练用户全部拉回多轮填写。
- 备选方案：所有跟进写回都强制先走模板。
- 不采用原因：会显著增加高频用户的输入成本，也会破坏当前已经可用的单句直达体验。

### 决策二：在会话 work memory 中新增轻量模板草稿，而不是提前落正式待写回记录

- 选择：在企业微信会话上下文中增加 `followUpTemplateDraft` 一类轻量结构，保存 `requesterName`、`followUpContent`、`helpNeeded`、`informationShare`、`visitPlan`、缺失字段和当前阶段；只有当模板已完成并通过对象识别后，才生成现有 `PendingFollowUpWritebackRecord`。
- 原因：现有待写回记录的核心字段是 `draftContent` 和目标对象信息，适合承接“已整理完成、待确认写回”的阶段，不适合承接尚未补齐的半结构化输入。
- 备选方案：一进入模板就创建 `PendingFollowUpWritebackRecord`，边补边改 `draftContent`。
- 不采用原因：会让“模板收集中间态”和“正式待写回态”混在一起，增加失败回滚和状态判定复杂度。

### 决策三：模板字段固定为五项，其中姓名只读自动补齐，正文必须包含明确公司名或客户名

- 选择：聊天模板固定输出以下栏目：
  - `跟进内容`：必填，必须包含公司名、客户名或项目相关明确对象信息；
  - `遇到与协助`：选填，但建议补充，可接受“无 / 暂无 / 需要某某协助”；
  - `信息共享`：选填，可写行业政策、友商 PK、客户反馈等，也接受“无”；
  - `拜访计划`：选填，建议包含时间、同行人、拜访对象、事项和预期结果，也接受“暂无计划”。
- 姓名由系统自动获取，不在聊天模板中单独展示，但在最终跟进草稿中以 `【姓名】：` 单独起行。
- 原因：这五项和用户给出的业务诉求一致，既能提升可读性，也能在不引入复杂表单控件的情况下提高信息完整度。
- 备选方案：只要求“跟进内容”一个字段，其余继续自由发挥。
- 不采用原因：这样无法达到“填写内容精细化”的目标，也不能复用新增客户 / 新增商机的引导思路。

### 决策三补充：首轮模板文案采用“代整理”语气，并明确日报复用价值

- 选择：机器人首轮提示采用“我来帮你整理”的语气，弱化“请按格式填写”的命令感，同时自然说明“这些信息后面也能帮你汇总日报”。推荐文案如下：

```text
收到，我来帮你整理这次跟进。

为了我能更准确识别客户 / 公司，也方便后面帮你自动汇总日报，辛苦尽量按下面模板回复我：

跟进内容：今天拜访了谁 / 跟进了哪家公司，做了什么，现在推进到什么进度
遇到与协助：遇到了什么问题，需要谁协助；没有可写“无”
信息共享：有没有值得同步的干货，比如友商 PK、政策变化、行业信息、客户反馈；没有可写“无”
拜访计划：计划什么时间、和谁一起、拜访谁、做什么、希望达到什么结果；没有可写“暂无”

温馨提醒：公司名或客户名尽量写全，我这边会更容易帮你精准识别。
```

- 原因：跟“请按模板填写”相比，“我来帮你整理”更贴近企业微信对话风格，也更容易让一线销售接受。
- 备选方案：使用更短、更硬的指令式提示。
- 不采用原因：虽然更省字，但鼓励性不足，容易让用户继续只回一句模糊的“今天跟进了一下”。

### 决策四：最终跟进草稿按固定顺序组装，显式填写的“无 / 暂无”保留，姓名前缀继续复用现有归一化逻辑

- 选择：模板完成后，系统按“跟进内容 -> 遇到与协助 -> 信息共享 -> 拜访计划”的顺序把内容组装成单一 `draftContent`，再调用现有姓名前缀归一化逻辑，确保最终结果保持 `姓名：正文` 格式且不会重复署名。对于用户显式填写的“无 / 暂无”，保留在最终草稿中，避免系统擅自省略用户明确表达的结论。同时在模板收集阶段继续保留上述四类字段的独立语义，不在收集早期就混成不可拆分的一段话。
- 原因：统一输出结构能让用户在确认时清楚看到最终将写入 CRM 的内容，也便于失败后直接修改同一份草稿；保留字段语义则能为后续日报自动总结减少二次拆解成本。
- 备选方案：只把有内容的栏目拼进正文，空值全部省略。
- 不采用原因：若用户明确写了“无 / 暂无”，直接省略会改变原意，也会让模板填写的完整性在最终确认时丢失。

推荐成稿样式如下：

```text
【张三】：
跟进内容：今日拜访了XXX科技，和王总沟通年度合作推进，已确认预算范围，当前进入方案细化阶段。
遇到与协助：客户希望下周看到报价测算，需产品同事李四协助补充实施排期。
信息共享：友商本周主推打包折扣方案，客户对交付周期比价格更敏感。
拜访计划：计划4月10日上午和李四一起再次拜访XXX科技，确认报价口径和试点范围，争取推进到方案评审。
```

如果用户明确填写“无 / 暂无”，则成稿中保留对应行，例如：

```text
【张三】：
跟进内容：今日电话跟进了XXX客户，确认了续签意向，客户计划本周内部评估。
遇到与协助：无。
信息共享：无。
拜访计划：暂无。
```

### 决策五：模板字段的结构化语义先保留在会话草稿中，日报复用作为后续能力消费

- 选择：当前版本不直接生成日报，但模板草稿中要清晰区分 `followUpContent`、`helpNeeded`、`informationShare`、`visitPlan` 等字段，供后续日报功能直接读取或总结；`draftContent` 只是当前写回 CRM 的展示与执行载体，不是唯一语义来源。
- 原因：如果现在只保留最终拼接后的长文本，后续日报开发还要重新做字段拆分、关键词识别和容错解析，成本高且不稳定。
- 备选方案：本次只关心 CRM 写回，后续日报再从自由文本中反推结构。
- 不采用原因：这会把本次明明可以顺手保留的结构化信息丢掉，后续开发成本更高。

### 决策六：模板只是前置引导层，后续仍完全复用既有受控跟进写回链路

- 选择：模板完成后继续走现有的对象识别、摘要确认、失败保留、重试、群分享和 `POST /api/v2/revisit_logs` 写回逻辑，不新增新的执行状态枚举或 API 路径。
- 原因：本次需求的价值在于更好地收集和整理输入，而不是重做写回主链路。
- 备选方案：为模板链路新增一套独立的确认状态和写回方法。
- 不采用原因：会重复建设现有审计与失败处理机制，也增加规格和实现分叉风险。

## Risks / Trade-offs

- [模板步骤增加了首次输入长度] → 仅在主题型入口触发空模板；用户一次性贴完整模板时直接解析，不再重复追问。
- [模板解析对字段标签格式较敏感] → 同时支持全角 / 半角冒号、逐行填写和一次性粘贴，并在无法解析时回退为“请按模板补充”。
- [文案润色过度可能引入系统臆造] → 只对用户已填写内容做顺序整理和轻度衔接，不补写未经用户提供的业务事实。
- [显式保留“无 / 暂无”会让部分跟进记录更长] → 通过固定顺序保持可读性，确保用户在确认前明确看到最终落库文本。
- [过早把字段压平成自由文本，会削弱后续日报复用价值] → 在模板草稿阶段保留独立字段结构，`draftContent` 仅作为当前 CRM 写回文本输出。

## Migration Plan

1. 在提示词配置中新增跟进模板入口文案、字段说明和缺项提示。
2. 在企业微信会话上下文中加入模板草稿结构和字段解析逻辑，完成模板收集与校验。
3. 将模板完成后的结果组装为既有 `draftContent`，同时保留可供日报复用的结构化字段，接入当前跟进对象识别与确认流程。
4. 补充主题入口、缺少公司名 / 客户名、一次性粘贴模板、姓名前缀归一化、失败重试以及结构化字段保留等测试用例。
5. 若上线后需要回滚，可关闭模板入口分支，让主题型跟进入口恢复为当前的自由描述提示，不影响既有写回主链路。

## Open Questions

- “今日跟进”这类更泛化的主题词是否也统一返回该模板，还是先只覆盖“跟进客户 / 跟进商机”及其近义表达。
- 模板中的 `拜访计划` 是否需要在后续实现中进一步拆成时间、同行人、对象、目标四个子字段；本提案阶段先按单段文本处理。
- 后续日报功能是直接消费模板草稿结构，还是在此基础上再加一层日报领域映射；本提案阶段先确保结构不丢。
