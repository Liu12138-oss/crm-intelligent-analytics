## Context

当前企业微信跟进写回链路分为三段：跟进入口识别与草稿整理、对象识别与待写回草稿创建、最终确认后调用 CRM 官方 `POST /api/v2/revisit_logs`。运行时已经通过 `wecom.followup.writeback` 控制“这个功能整体是否可用”，但还没有校验“当前用户是否对这个客户或商机具备允许的组织关系”，因此对象唯一命中后仍可能出现越权写回。

现有实现还存在第二个结构性问题：代码与测试已经支持 `Customer` 和 `Opportunity` 两类跟进对象，但主规格、OpenAPI 和部分工作记忆 / 待写回对象命名仍然偏向 `Opportunity-only`。如果继续在这个状态上叠加权限逻辑，后续会同时放大三类风险：对象权限解释不清、审计语义不一致、接口契约与真实行为不一致。

本次设计必须同时满足以下约束：

- 不调整登录、统一鉴权、会话建立与企业微信验签链路。
- 继续优先复用 CRM 官方 `POST /api/v2/revisit_logs`，不新增数据库直写。
- 继续保留 `wecom.followup.writeback` 作为全局动作开关。
- 对象级授权必须基于企业微信组织事实与 CRM 映射，不得回退成页面部门名的粗粒度放权。

## Goals / Non-Goals

**Goals:**

- 为企业微信跟进写回增加“动作权限 + 对象关系权限”双门闩。
- 明确负责人口径：`跟进客户` 按客户负责人，`跟进商机` 按商机负责人，`今日跟进` 按最终选中对象类型继承。
- 将“负责人所在小组”定义为企业微信直属上级事实下的最小直属团队，并允许负责人递归上级写回。
- 在草稿创建前和最终写回前都重校验对象级关系，避免负责人变化后的旧草稿误放行。
- 正式收口 `Customer` / `Opportunity` 双对象写回口径，同步对象模型、接口契约和审计快照。

**Non-Goals:**

- 不扩展 Web 工作台为新的跟进写回入口。
- 不改动 `wecom.followup.writeback` 在权限中心中的键名与基础治理方式。
- 不引入部门级人工覆盖规则替代企业微信直属上级事实。
- 不调整除跟进写回外的分析问数、日报预览、客户创建、商机创建权限边界。

## Decisions

### 决策一：保留 `wecom.followup.writeback` 全局动作门闩，再叠加对象级授权门闩

采用两层校验：

1. `wecom.followup.writeback` 决定该用户是否整体可用企业微信跟进写回。
2. 新增对象级授权决定该用户是否可跟当前命中的客户或商机。

推荐原因：

- 全局动作权限继续由权限中心统一治理，便于整体停开或分角色发放。
- 对象级授权专门表达“能不能跟这个对象”，不会把业务关系边界硬塞进角色矩阵。

备选方案：

- 仅保留对象级授权，不保留全局动作权限。
  不采纳，原因是会弱化后台治理能力，并让停开 / 灰度 / 排障手段明显变差。

### 决策二：对象负责人口径按最终对象类型确定，不做跨对象混算

- `Customer` 跟进按客户当前负责人判权。
- `Opportunity` 跟进按商机当前负责人判权。
- `今日跟进` / 自由文本在对象识别完成后，按最终确认的对象类型取对应负责人。

推荐原因：

- 用户理解成本最低，规则可直接解释为“你跟的是谁，就继承谁的负责人边界”。
- 能自然处理“客户负责人”和“商机负责人”不同的真实业务场景。

备选方案：

- 统一只认商机负责人。
  不采纳，原因是会让 `跟进客户` 变成语义和权限口径都不一致的入口。

### 决策三：“负责人所在小组”按企业微信直属上级事实下的最小直属团队定义

对象级授权的“小组”不按 CRM 页面显示部门名定义，而按企业微信同步事实定义：

- 先通过 CRM 用户与企业微信成员映射找到目标负责人对应的 `wxUserid`。
- 读取其当前 `directLeaderUserids`。
- 以共享同一直属上级的 CRM 成员集合作为“负责人所在小组”。
- 同时把直属上级本人纳入允许范围；递归向上的上级链单独计算。

推荐原因：

- 比部门路径更贴近真实汇报线。
- 可以避免大部门下多个销售小组被整体放行。

备选方案：

- 按 CRM 页面所属部门或企业微信部门整体放权。
  不采纳，原因是范围过粗，容易把“同部门但不同小组”错误放行。

### 决策四：对象级授权在草稿创建前与最终写回前都执行一次

执行点分为两处：

- **草稿创建前**：阻断不具备对象关系权限的用户进入待写回确认链路。
- **最终写回前**：再次读取当前目标对象负责人和组织关系，阻断负责人变化或关系失效后的旧草稿。

推荐原因：

- 只在草稿创建前校验，无法覆盖负责人临时转移、组织关系同步变化等情况。
- 只在最终写回前校验，会让越权用户拿到不该看到的确认草稿与审计上下文。

备选方案：

- 仅在草稿创建前做一次校验。
  不采纳，原因是最终执行边界不够稳。

### 决策五：新增独立 `FollowUpAuthorizationService`，不把关系判断继续堆进 `WecomBotService`

新增专用授权服务，负责：

- 解析目标对象当前负责人；
- 基于组织事实计算“本人 / 同组 / 上级 / 管理员”；
- 输出命中的关系类型、拒绝原因和审计快照字段。

`WecomBotService` 只负责在草稿创建和最终写回时调用授权服务并响应结果。

推荐原因：

- 当前 `WecomBotService` 已承担入口识别、活跃任务、候选确认、写回执行等多重职责，再把对象级授权堆进去会继续膨胀。
- 单独服务更适合做单元测试，也更利于后续让 Web 路径复用。

### 决策六：对象模型采用中性字段增量兼容，不做一次性硬切

待写回记录、工作记忆和响应载荷统一以 `objectType/objectId/objectTitle` 为准；历史 `opportunityId/opportunityTitle` 先保留一版兼容读取，避免当前本地运行态或测试夹具中的旧记录直接失效。

推荐原因：

- 当前已经存在 `Customer` 写回测试与数据，继续使用 `opportunity*` 命名表达客户对象会让代码和审计越来越难维护。
- 增量兼容比一次性删旧字段更稳，回滚也更容易。

## Risks / Trade-offs

- [风险] 企业微信映射或直属上级事实缺失时，原本“看起来能写”的用户会被阻断。  
  → 缓解：统一返回中文提示，补齐权限拒绝审计，并在治理诊断里暴露负责人映射与上级链缺失原因。

- [风险] `followUpWriteback.objectType` 从事实层面正式承认 `Customer`，可能与少量依赖旧契约的前端或测试桩不一致。  
  → 缓解：同步更新 OpenAPI、前端类型、集成测试，并在一版兼容期内保留旧对象字段别名。

- [风险] 最终写回前增加一次负责人重查，会引入额外一次对象读取与组织关系计算。  
  → 缓解：仅在确认执行前触发，不影响普通分析链路；组织关系计算尽量复用企业微信组织事实内存态，避免重复外部调用。

- [风险] “直属团队”定义可能与部分业务口头上的“部门小组”理解不完全一致。  
  → 缓解：在 proposal/spec 中明确以企业微信直属上级事实为准，并通过治理预览和审计快照解释命中关系。

- [风险] 本地运行态中已有的 `PendingFollowUpWritebackRecord` 历史字段命名混杂，若处理不慎可能导致旧草稿恢复异常。  
  → 缓解：对象模型改为增量兼容，读取时优先新字段，缺失时回退旧字段。

## Migration Plan

1. 在 OpenSpec 中先落提案、设计、规格增量和任务，明确 `Customer` / `Opportunity` 双对象口径与对象级授权边界。
2. 实现 `FollowUpAuthorizationService` 和组织关系辅助能力，先接入草稿创建阶段。
3. 再接入最终写回阶段的实时重校验，并补齐统一拒绝审计。
4. 更新待写回对象模型、工作记忆、OpenAPI 和前端类型，保持一版兼容读取旧字段。
5. 补齐单元测试、企业微信集成测试和审计断言后上线。

回滚策略：

- 若上线后对象级授权误伤过多，可先回滚 `FollowUpAuthorizationService` 接线，恢复为仅校验 `wecom.followup.writeback`。
- 保留新增中性对象字段不会影响旧逻辑；回滚时只需停止消费新判权服务，不需要数据层回退。

## Open Questions

- 当前无新的业务口径阻塞问题；负责人对象类型、小组定义、最终写回实时重校验和全局动作总开关均已在提案阶段确认。
- 实施阶段需继续验证真实环境下企业微信组织同步是否稳定提供 `directLeaderUserids`；若个别历史用户缺失该字段，是否需要补同步而不是做代码旁路放权。
