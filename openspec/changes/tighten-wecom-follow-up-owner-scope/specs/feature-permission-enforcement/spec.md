## MODIFIED Requirements

### Requirement: 系统必须区分通道准入与工作流动作准入
系统 MUST 将 Web / 企业微信通道准入与具体工作流动作准入分开判断。`webConsoleEnabled` 和 `wecomBotEligible` 只负责决定用户能否进入对应通道；在通道已准入后，系统 MUST 再按具体动作权限决定用户能否执行分析、创建、写回、日报预览等工作流。对于企业微信跟进写回这类高风险工作流，系统 MUST 在动作权限通过后继续执行对象级授权，确认当前用户是否满足目标对象负责人的允许关系。系统 MUST 不得把“能进入企业微信机器人”错误等同于“企业微信所有功能都可用”，也不得把“具备跟进写回动作权限”错误等同于“可以跟任意对象”。

#### Scenario: 企业微信用户只具备日报预览动作时不能执行问数
- **WHEN** 某用户通过企业微信通道准入且拥有 `wecom.daily_report.preview`，但未拥有 `wecom.analysis.use`
- **THEN** 系统可以允许该用户进入企业微信机器人并查看日报预览
- **THEN** 系统必须拒绝其发起企业微信问数工作流

#### Scenario: 企业微信用户只具备问数动作时不能查看日报预览
- **WHEN** 某用户通过企业微信通道准入且拥有 `wecom.analysis.use`，但未拥有 `wecom.daily_report.preview`
- **THEN** 系统可以允许该用户发起企业微信问数工作流
- **THEN** 系统必须拒绝其访问个人或团队日报预览工作流

#### Scenario: 企业微信用户具备跟进写回动作但不满足对象关系时仍被拒绝
- **WHEN** 某用户通过企业微信通道准入且拥有 `wecom.followup.writeback`，但目标对象负责人关系不满足对象级授权规则
- **THEN** 系统必须拒绝其创建或执行该对象的跟进写回工作流
- **THEN** 不得因为该用户已命中动作权限而跳过对象级授权

### Requirement: 系统必须为权限拒绝写入可排障审计
系统 MUST 为所有新增接入的权限拒绝写入审计事件，至少记录操作者、角色、权限点、资源类型、资源 ID、通道、拒绝原因和发生时间。权限拒绝审计 MUST 足以解释“为什么该用户看不到页面”“为什么该按钮不可点”“为什么接口被 403”“为什么企业微信工作流被阻断”。对于企业微信跟进写回，审计 MUST 能区分“缺少动作权限”与“目标对象关系不满足 / 组织事实缺失”两类阻断。

#### Scenario: 企业微信工作流被动作权限拒绝时写入审计
- **WHEN** 企业微信用户已通过通道准入，但在执行具体工作流时因缺少动作权限被拒绝
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 该事件必须记录被拒绝的动作键、目标工作流和拒绝原因

#### Scenario: 系统级治理接口被拒绝时写入审计
- **WHEN** 某个用户调用审计中心、治理策略或 AI 模型治理接口时因缺少动作权限被拒绝
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 审计事件必须能够区分是 `audit.view`、`governance.policy.manage` 还是 `ai_profile.manage` 拒绝

#### Scenario: 企业微信跟进写回因对象关系不满足被拒绝时写入审计
- **WHEN** 某用户具备 `wecom.followup.writeback`，但在目标对象唯一命中后因不满足负责人关系或组织事实缺失被阻断
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 审计中必须能够区分该次阻断来自对象级授权，而不是普通动作权限缺失
