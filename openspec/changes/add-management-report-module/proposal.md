## Why

当前 CRM 智能分析系统已经具备自由问数、合同审核、治理审计等能力，但缺少一个面向管理人员的固定经营报表入口。现有 HTML 报告已经验证了“线索、客户、商机、验收、收款、风险”这一类多专题驾驶舱的业务价值，而当前自由问数结果页是“单问题 -> 单结果”的交付模型，不适合承载多专题、统一区域范围、统一时间范围、统一口径说明和统一导出的管理视图。

这次变更的目标不是继续扩展自由问数的主题边界，而是在现有系统内新增一个独立的 Web `经营报表` 模块。该模块服务于管理层、销售总监、区域经理和其它已授权管理角色，用固定专题快照的方式提供经营看板；同时严格继承当前 CRM 权限范围，允许用户按团队/部门切换、按时间范围筛选，并在收款专题中只使用 CRM 内部应收/回款数据口径，不再接入周报 Excel、外部资讯或招投标数据源。

从业务上看，这项变更能把“临时报表”和“离线 HTML 文件”收敛到正式产品能力中，降低管理报表制作的重复劳动，同时让管理人员在系统内直接查看统一口径的经营摘要、线索漏斗、客户池、商机池、验收进度和收款状态，减少因多份表、多人维护和口径不一致带来的沟通成本。更关键的是，这个模块必须把“统计数字做对”放在第一优先级；如果统计不准，再漂亮的报表都没有意义。

## What Changes

### 业务能力范围

- 新增独立的 Web `经营报表` 模块，而不是继续复用自由问数结果页。
- 新增经营报表菜单入口、独立路由和独立权限点。
- 新增统一筛选栏，支持：
  - 团队/部门切换（以企业微信组织架构树形选择为准）
  - 时间范围筛选
- 新增经营报表上下文与专题按需加载机制：
  - 首次加载只返回核心摘要、当前筛选上下文和专题可用性
  - 具体专题数据在激活页签、展开模块或显式请求时再按需加载
- 新增固定专题结果，覆盖以下专题块：
  - 总览
  - 经营摘要
  - 区域经营
  - 线索
  - 线索转化
  - 线索机会
  - 商机
  - 客户
  - 代理商/生态
  - 产品方案
  - 验收进度
  - 收款情况
  - 经营风险和建议

### 明确不做的范围

- 不接入 `区域热点` 专题。
- 不接入 `招投标信息` 专题。
- 不接入 `任务完成` 专题。
- 不接入外部资讯源、周报 Excel 或任意手工上传口径。
- 不把收款、应收、回款能力开放给自由问数链路。
- 不把经营报表做成通用 BI 设计器或用户自定义多页报表系统。

### 筛选与加载要求

- 经营报表必须在统一筛选上下文下工作，筛选条件至少包括：
  - `departmentId`
  - `startDate`
  - `endDate`
  - `presetKey`
- 部门/团队筛选必须以企业微信组织架构树形选择为准，默认选中 `全公司`。
- 部门/团队筛选默认始终包含所选节点的全部下级部门，不再向用户暴露“是否包含子部门”开关。
- 时间范围必须同时支持：
  - 常用时间快捷项，例如 `Q1`、`Q2`、`Q3`、`Q4`、月、年、近30天、近90天
  - 自定义时间范围
- 默认时间范围必须为当前日历年第一季度，即当年的 `Q1`。
- 页面必须持续展示当前筛选上下文：
  - 当前部门范围
  - 当前时间范围
  - 数据生成时间
- 页面首屏只允许加载：
  - 当前范围说明
  - 当前时间范围
  - 总览摘要
  - 经营摘要
  - 专题导航与专题可用性
- 重专题数据不得在首次进入页面时一次性全部查询，必须按需加载，避免一进页面就把 CRM 查询打爆。

### 统计正确性要求

经营报表的首要目标不是“把页面做出来”，而是“把每个统计数字做对”。因此本次变更必须把统计正确性作为第一优先级要求，而不是交由前端拼装或 AI 文案兜底。至少需要满足以下约束：

- 每个摘要指标、图表数值、排行结果和专题表格都必须有明确的：
  - 来源表
  - 来源字段
  - 过滤条件
  - 分组维度
  - 聚合公式
  - 时间字段
- 派生指标必须显式声明公式，例如：
  - 线索转客户率
  - 客户有商机率
  - 客户激活率
  - 回款率
  - 逾期占比
- 同一指标在页面不同位置引用时，必须来自同一份后端聚合结果，不允许前端二次计算出另一个版本。
- 各专题必须支持“专题级口径说明”和“关键指标来源说明”，便于业务复核。
- 当核心指标之间出现对不上的情况，例如：
  - 摘要值与表格汇总不一致
  - 分项求和不等于总计
  - 截面口径与期间口径被误混
  - 按合同归属汇总与按回款记录归属汇总不一致
  系统必须优先阻断错误交付或标记结果不可用，而不是继续输出一个“看起来完整”的报表。
- 在实现阶段必须为每个核心专题补充抽样对账样例和自动化回归测试。

### 时间筛选的业务规则

时间筛选必须按专题语义区分“期间发生”和“期末截面”，不能把所有专题都硬套到同一个时间字段上：

- 线索新增、来源、趋势：
  - 使用 `leads.created_at`
- 线索转化：
  - 以筛选期内创建的线索作为 cohort
  - 再看这些线索截至 `endDate` 的转客户、转商机、成交结果
- 客户新增：
  - 使用 `customers.created_at`
- 客户池、激活率、无商机客户：
  - 按 `customers.created_at <= endDate` 的期末客户池计算
- 商机新增、金额趋势、区域/行业分布：
  - 优先使用 `COALESCE(opportunities.get_time, opportunities.created_at)`
- 风险商机：
  - 对象池按 `<= endDate`
  - 风险判断结合 `expect_sign_date`、`revisit_at`、`stage_updated_at`
- 合同签约、验收进度：
  - 优先使用 `contracts.sign_date`
  - 无值时回退 `contracts.created_at`
- 实际回款：
  - 使用 `received_payments.receive_date`
- 应收计划：
  - 使用 `received_payment_plans.receive_date`
- 逾期应收：
  - 以 `receive_date < endDate` 且未完成作为基础口径

### 收款专题口径

收款专题只保留 CRM 口径，明确不再保留周报管理口径。页面和接口必须基于以下表生成结果：

- `contracts`
- `received_payment_plans`
- `received_payments`

收款专题至少包含：

- 收款摘要
- 月度回款趋势
- 应收状态
- 收款负责人拆解
- 逾期应收重点项目
- 合同验收与回款联动

页面上的摘要指标建议至少包括：

- CRM 已收款累计
- CRM 应收账单累计
- CRM 逾期应收金额
- 筛选期内已回款金额
- 筛选期内应收计划金额
- 筛选期内回款率

收款按团队或部门聚合时，必须通过 `contract_id -> contracts.department_id / contracts.user_id` 做归属，不得直接以 `received_payments.organization_id` 作为经营归属口径。

### 性能与稳定性要求

- 页面首次加载不得同时触发全部专题重量查询。
- 系统必须提供统一报表上下文标识或等价机制，保证“核心摘要”和“按需专题详情”仍然属于同一组筛选条件。
- 系统必须对专题查询做缓存，至少按“用户、部门范围、时间范围、专题标识”组合缓存。
- 系统必须限制同一报表上下文下的专题并发查询数量，并为单个专题设置超时与降级策略。
- 当单个专题查询失败或超时时，系统应优先降级该专题，而不是拖垮整页。
- 当当前筛选条件命中缓存时，系统必须优先返回缓存结果，避免重复压测 CRM。

### 模块化与组件化开发要求

- 经营报表页面必须按专题拆分为独立组件开发，避免单个页面文件承载全部业务块。
- 建议拆分为：
  - 顶部筛选栏组件
  - 总览组件
  - 经营摘要组件
  - 区域经营组件
  - 线索组件
  - 线索转化组件
  - 线索机会组件
  - 商机组件
  - 客户组件
  - 代理商/生态组件
  - 产品方案组件
  - 验收进度组件
  - 收款组件
  - 风险与建议组件
- 组件必须共享统一的数据类型、状态模型和口径说明展示方式。
- 后端也必须按“控制器 / 服务 / 查询层 / 组装层 / 类型定义”分层实现，避免把全部报表逻辑写进一个文件。

### 页面视觉与 UI 设计要求

- 在不牺牲可读性、对比度和状态可解释性的前提下，经营报表模块允许采用比现有工作台更大胆、更鲜艳的视觉表达。
- 经营报表首版视觉方向允许：
  - 更高饱和度的品牌渐变
  - 更有层次的彩色专题卡
  - 更强的首屏信息舞台感
  - 更鲜明的状态色和专题配色区分
- 但必须继续满足：
  - 数据可读性优先
  - 颜色不是唯一状态信号
  - 关键数字与正文对比度满足可读要求
  - 桌面端与窄屏端都不出现阅读负担过高的炫技布局

### 权限与安全要求

- 经营报表必须新增独立菜单权限：
  - `management-report`
- 经营报表必须新增独立动作权限：
  - `management.report.view`
  - `management.report.export`
- 前端导航、路由守卫、页面初始化接口、核心快照接口、专题详情接口、导出接口必须使用同一权限决策结果。
- 用户可切换的部门集合必须来源于当前最终有效范围，不得在经营报表模块内部另起“经理可看全部部门”的快捷规则。
- 用户选择某个部门节点后，系统必须默认把该节点的全部子部门纳入统计范围。
- 当用户提交超出默认范围或白名单范围的部门参数时，系统必须阻断请求并记录审计，不得静默回退为默认范围，也不得放行越权查询。

### 审计与导出要求

- 系统必须为经营报表查看和导出行为新增独立审计事件。
- 审计至少记录：
  - 操作者
  - 当前部门范围
  - 时间范围
  - 数据生成时间
  - 导出格式
- 导出内容必须与当前报表上下文保持一致，不得导出未授权范围或与页面不一致的数据。

### 技术实现要求

后端建议新增独立模块：

- `backend/src/modules/management-report/management-report.module.ts`
- `backend/src/modules/management-report/management-report.controller.ts`
- `backend/src/modules/management-report/management-report.service.ts`
- `backend/src/modules/management-report/management-report-query.service.ts`
- `backend/src/modules/management-report/management-report-composer.service.ts`
- `backend/src/modules/management-report/management-report.types.ts`

前端建议新增独立页面和服务：

- `frontend/src/pages/management-report/ManagementReportPage.vue`
- `frontend/src/services/management-report.service.ts`
- `frontend/src/types/management-report.ts`
- `frontend/src/components/management-report/*`

接口建议至少包括：

- `GET /management-report/options`
- `POST /management-report/snapshot`
- `POST /management-report/sections/:sectionKey`
- `POST /management-report/export`

## Capabilities

### New Capabilities

- `management-report-workbench`: 提供独立经营报表模块的页面入口、统一筛选上下文、核心摘要、专题按需加载、CRM 收款专题、统一导出与页面级审计能力。

### Modified Capabilities

- `feature-permission-enforcement`: 新增经营报表菜单与查看/导出动作权限，并要求前后端执行点、导航显隐、路由守卫和接口放行保持一致。
- `organization-scope-governance`: 新增经营报表场景下的部门切换与“包含子部门”范围解析约束，确保页面筛选始终落在当前有效数据范围内，并对越权部门筛选请求进行阻断和留痕。
- `structured-analysis-result-delivery`: 新增经营报表统计结果的指标来源说明、统一聚合结果复用、跨区块一致性校验和错误交付阻断要求。
- `web-ui-ux-redesign`: 新增经营报表页面的模块化专题组件、专题按需加载交互和更大胆鲜艳的独立视觉表达约束。

## Impact

### 前端影响

- 新增经营报表菜单入口和独立页面路由。
- 新增经营报表页面骨架、统一筛选栏和专题组件。
- 页面需要拆分成专题级组件，而不是把全部报表逻辑压进单页大组件。
- 需要在页面中持续展示当前部门范围、时间范围和数据生成时间。
- 需要为无权限导出、无数据、加载中、专题按需加载、专题重试和筛选刷新提供明确反馈。

### 后端影响

- 新增 `management-report` 模块。
- 新增核心上下文接口、专题详情接口和导出接口。
- 新增经营报表查看与导出审计事件。
- 新增对 `leads`、`customers`、`opportunities`、`contracts`、`received_payment_plans`、`received_payments` 的固定报表读取逻辑。
- 需要新增指标口径字典、专题聚合定义和一致性校验逻辑，确保每个统计数字都可追溯、可复核。
- 需要新增缓存、并发控制、查询超时和专题降级逻辑，避免首屏把 CRM 查询打爆。

### 权限与治理影响

- 新增 `management-report` 菜单。
- 新增 `management.report.view` 和 `management.report.export` 动作。
- 需要在权限中心、前端菜单和后端执行点中同时接入。
- 需要复用现有组织范围治理能力，确保经营报表筛选行为与统一范围解析一致。

### API 与契约影响

- 新增经营报表筛选项、核心上下文、专题详情和导出接口。
- 需要同步更新 OpenAPI。
- 需要补充经营报表上下文结构、专题加载结构、时间口径说明和导出契约。

### 文档与实施影响

- 需要同步更新主规格、实施计划、数据模型和任务清单。
- 需要把独立经营报表模块纳入 README 和后续实施说明。
- 需要在实现阶段补一轮 `代理商/生态` 与 `产品方案` 字段核对，确保专题字段来源可验证。
- 需要维护独立的 `metric-dictionary.md`，作为经营报表第一阶段核心指标、公式和对账规则的实现依据。
