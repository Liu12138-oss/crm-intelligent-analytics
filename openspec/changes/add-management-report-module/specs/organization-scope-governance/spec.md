## ADDED Requirements

### Requirement: 系统必须在经营报表场景下按最终有效范围解析部门切换
系统 MUST 在经营报表场景下复用组织事实、默认团队范围和数据范围白名单的最终有效范围解析结果。经营报表可切换的部门集合 MUST 来源于当前用户的有效数据范围，不得在经营报表模块内部再发明一套“经理即可切全部部门”的快捷规则。用户在经营报表中选中某个部门节点后，系统 MUST 默认递归包含该节点的全部子部门；该行为必须在当前有效范围允许的部门树内生效。

#### Scenario: 管理人员只能切换到自己有权查看的部门
- **WHEN** 某个管理人员打开经营报表筛选栏查看可选部门
- **THEN** 系统只允许展示其默认团队范围或白名单授权范围内的部门
- **THEN** 不得展示其当前无权查看的其它团队或部门

#### Scenario: 部门选择默认递归子部门
- **WHEN** 用户在经营报表中选择某个部门节点
- **THEN** 系统必须按现有组织范围治理规则递归展开允许的子部门
- **THEN** 不得因为经营报表是固定页面而绕过原有的子部门授权边界

#### Scenario: 初始化接口返回的部门集合已经按最终有效范围裁剪
- **WHEN** 前端请求经营报表筛选项初始化数据
- **THEN** 系统返回的部门集合必须已经基于最终有效范围完成裁剪
- **THEN** 前端不得再额外承担权限裁剪逻辑

### Requirement: 系统必须阻断经营报表中的越权部门筛选请求
系统 MUST 在经营报表快照请求和导出请求中校验目标部门范围是否属于当前用户的最终有效范围。当用户提交超出默认范围和白名单范围的部门参数时，系统 MUST 阻断请求并记录阻断审计，不得默默回退为默认范围，也不得放行越权查询。

#### Scenario: 用户请求无权限部门时报表被阻断
- **WHEN** 用户提交一个不在其有效范围内的 `departmentId`
- **THEN** 系统必须拒绝本次经营报表快照请求并返回明确无权限提示
- **THEN** 系统必须记录目标部门、当前有效范围和阻断原因

#### Scenario: 导出请求复用同一部门范围校验
- **WHEN** 用户在经营报表页触发导出
- **THEN** 系统必须使用与快照查询相同的部门范围校验规则
- **THEN** 不得出现页面查看被限制但导出接口仍能读取超范围部门数据的情况

### Requirement: 系统必须在经营报表场景下保留范围说明与来源说明
系统 MUST 在经营报表快照和初始化结果中返回当前范围说明，明确说明该范围来自默认团队范围、角色或用户级白名单扩展，或管理员组织级口径。经营报表页面必须能够持续展示该说明，帮助管理人员理解当前看到的是哪个团队或部门的结果。

#### Scenario: 白名单扩展范围在经营报表中可解释
- **WHEN** 某个用户的数据范围来自默认团队范围叠加白名单授权
- **THEN** 经营报表返回的范围说明必须明确指出存在白名单扩展
- **THEN** 用户不应误以为当前部门集合全部来自默认团队关系

#### Scenario: 管理员查看组织级结果时范围说明清晰
- **WHEN** 管理员以组织级口径查看经营报表
- **THEN** 系统必须返回明确的组织级范围说明
- **THEN** 不得把管理员口径误表述为某个单一部门或单一负责人团队
