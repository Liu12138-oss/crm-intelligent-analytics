## ADDED Requirements

### Requirement: 经营报表页面必须采用模块化专题组件结构
系统 MUST 将经营报表页面实现为模块化专题组件集合，而不是单个超大页面组件。经营报表至少 MUST 将筛选栏、总览、经营摘要、区域经营、线索、客户、商机、验收、收款和风险专题拆分为独立组件，并通过统一类型与统一状态模型组合成页面。

#### Scenario: 协作者实现经营报表页面
- **WHEN** 协作者开始实现经营报表页面
- **THEN** 页面必须按专题拆分为独立组件
- **THEN** 不得把全部专题逻辑、全部样式和全部状态堆叠在单个 Vue 页面文件中

#### Scenario: 专题组件共享统一状态模型
- **WHEN** 经营报表页面渲染多个专题块
- **THEN** 各专题组件必须共享统一的数据类型、加载状态、异常状态和口径说明展示方式
- **THEN** 不得让每个专题独立发明自己的状态语法

### Requirement: 经营报表页面必须支持核心摘要优先与专题按需加载交互
系统 MUST 在前端为经营报表提供“核心摘要先展示、专题详情按需加载”的交互结构。页面首次进入时 MUST 优先展示总览和经营摘要；用户切换页签、展开专题或主动重试时，系统 MAY 再加载对应专题详情。前端 MUST 为未加载、加载中、加载失败和已降级专题提供明确反馈。

#### Scenario: 首屏优先展示核心摘要
- **WHEN** 用户首次打开经营报表页面
- **THEN** 页面必须优先展示总览和经营摘要
- **THEN** 未激活专题不得全部同时进入重查询加载状态

#### Scenario: 专题失败时局部反馈而非整页白屏
- **WHEN** 某个专题详情加载失败或被后端降级
- **THEN** 页面必须仅在该专题区域展示失败或降级反馈
- **THEN** 不得因为单个专题失败而导致整页内容一起不可见

### Requirement: 经营报表页面允许采用更大胆鲜艳的视觉方向
系统 MUST 允许经营报表页面在现有设计系统基础上采用更大胆、更鲜艳的视觉表达，以突出经营驾驶舱的首屏聚焦感和专题层次。该页面 MAY 使用更高饱和度的渐变、更鲜明的专题配色和更有舞台感的指标卡，但 MUST 继续保证数据可读性、可访问性和状态可解释性。

#### Scenario: 经营报表首屏使用更鲜明的视觉层次
- **WHEN** 协作者实现经营报表首屏视觉
- **THEN** 页面可以使用比普通工作台更鲜明的品牌渐变和专题配色
- **THEN** 关键数字、表格和正文仍必须保持高可读性

#### Scenario: 颜色不能成为唯一状态信号
- **WHEN** 页面展示专题状态、风险等级、加载失败或降级提示
- **THEN** 页面必须同时使用文字、标签或图标表达状态
- **THEN** 不得仅靠更鲜艳的颜色差异传达状态含义
