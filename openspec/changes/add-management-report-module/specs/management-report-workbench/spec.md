## ADDED Requirements

### Requirement: 系统必须提供独立的经营报表工作台
系统 MUST 在 Web 端提供独立于自由问数工作台的 `经营报表` 模块。该模块 MUST 通过独立菜单和独立路由进入，并以固定经营专题快照的方式向管理人员展示结果，而不是继续复用单次问数结果页。经营报表 MUST 至少覆盖总览、经营摘要、区域经营、线索、线索转化、线索机会、商机、客户、代理商/生态、产品方案、验收进度、收款情况和经营风险与建议等专题。

#### Scenario: 已授权管理人员进入经营报表页面
- **WHEN** 已获得经营报表查看权限的用户登录 Web 应用并点击 `经营报表` 菜单
- **THEN** 系统必须进入独立的经营报表页面
- **THEN** 页面不得复用单个 `queryId` 的分析结果详情作为主展示模型

#### Scenario: 经营报表不包含已排除专题
- **WHEN** 用户查看经营报表专题导航或页面结构
- **THEN** 系统不得继续展示 `区域热点`、`招投标信息` 或 `任务完成` 专题
- **THEN** 页面内容必须仅使用本次变更批准范围内的固定经营专题

### Requirement: 系统必须提供经营报表筛选项初始化能力
系统 MUST 提供经营报表筛选项初始化接口，用于返回当前用户可切换的部门树、时间范围快捷项和默认筛选值。部门树 MUST 以企业微信组织架构为准，并已按当前用户最终有效数据范围做裁剪。初始化结果中的默认部门 MUST 为 `全公司`；默认时间范围 MUST 为当前日历年的第一季度 `Q1`。前端不得自行拼装或猜测可用部门。

#### Scenario: 页面加载时获取可用筛选项
- **WHEN** 前端首次打开经营报表页面并请求初始化接口
- **THEN** 系统必须返回当前用户可选的部门树、预设时间范围和默认筛选值
- **THEN** 前端不得在没有初始化结果的情况下自行构造部门下拉数据

#### Scenario: 默认筛选值为全公司和当年 Q1
- **WHEN** 用户首次打开经营报表页面且未主动修改筛选条件
- **THEN** 系统必须默认以 `全公司` 作为部门范围
- **THEN** 系统必须默认以当前日历年第一季度作为时间范围

#### Scenario: 用户没有可扩展部门时仍可打开报表
- **WHEN** 某个用户只能查看本人或单一团队范围，没有额外可切换部门
- **THEN** 系统仍必须返回可用的默认筛选值
- **THEN** 页面不得因为部门列表过短而进入不可用状态

### Requirement: 系统必须在首屏只加载经营报表核心摘要
系统 MUST 在经营报表首次打开或筛选条件变更后，优先只加载当前筛选上下文、总览摘要、经营摘要和专题可用性信息。系统 MUST NOT 在首屏阶段同时触发全部专题重量查询。区域经营、线索、客户、商机、验收、收款等专题详情 MUST 在用户激活对应专题或显式请求时按需加载。

#### Scenario: 首次进入经营报表页面时只返回核心摘要
- **WHEN** 用户首次进入经营报表页面
- **THEN** 系统必须先返回核心摘要与专题清单
- **THEN** 系统不得在首次加载阶段把所有专题数据一次性全部查询完成

#### Scenario: 用户切换到某专题时按需加载详情
- **WHEN** 用户切换到某个尚未加载的专题页签
- **THEN** 系统必须按当前统一筛选上下文加载该专题详情
- **THEN** 该专题不得使用另一组筛选条件重新初始化上下文

### Requirement: 系统必须为经营报表提供统一的部门和时间筛选
系统 MUST 在经营报表页提供统一筛选栏，至少支持团队/部门切换和时间范围筛选。部门选择 MUST 以企业微信组织架构树形选择为准，并在用户选中某个节点后默认递归包含该节点全部子部门；系统 MUST NOT 再向用户暴露“是否包含子部门”开关。所有专题块 MUST 消费同一组筛选条件，并在页面上持续显示当前部门范围、时间范围和数据生成时间。系统 MUST 支持预设时间范围和自定义起止日期。

#### Scenario: 用户按部门和时间范围刷新经营报表
- **WHEN** 用户选择某个允许访问的部门节点并设置时间范围后触发查询
- **THEN** 系统必须按该组统一筛选条件重新生成整页经营报表快照
- **THEN** 同一页内所有专题块必须共享相同的部门范围和时间范围

#### Scenario: 部门选择默认包含下级部门
- **WHEN** 用户在经营报表中选择某个部门节点
- **THEN** 系统必须默认把该节点的全部子部门纳入统计范围
- **THEN** 页面不得再要求用户额外勾选“包含子部门”

#### Scenario: 页面持续显示当前筛选上下文
- **WHEN** 经营报表快照成功返回
- **THEN** 页面必须展示当前生效的部门范围、时间范围和数据生成时间
- **THEN** 用户不应通过阅读正文去猜测当前报表口径

#### Scenario: 无效时间范围请求被拒绝
- **WHEN** 用户提交结束时间早于开始时间或时间参数非法的请求
- **THEN** 系统必须拒绝本次快照请求并返回明确错误
- **THEN** 不得静默改写用户提交的时间范围

### Requirement: 系统必须在统一报表上下文下按需交付经营报表专题
系统 MUST 先生成统一的经营报表上下文，再在该上下文下按需交付专题块。核心摘要与专题详情可以由不同接口返回，但它们 MUST 共享同一组筛选条件、同一时间口径解释和同一次报表生成上下文。前端不得为每个专题自由拼接彼此无关的请求结果。

#### Scenario: 核心摘要与专题详情共享同一报表上下文
- **WHEN** 前端先获取核心摘要，再请求某个专题详情
- **THEN** 系统必须保证这两次返回属于同一组筛选条件和同一份报表上下文
- **THEN** 前端不得把不同上下文的专题详情拼接到同一页

#### Scenario: 同一快照内口径保持一致
- **WHEN** 用户查看总览、经营摘要和任一专题块中的指标
- **THEN** 同一指标在不同区块中的数值和筛选上下文必须保持一致
- **THEN** 不得出现摘要、图表、表格和导出使用不同时间范围或不同部门范围的情况

#### Scenario: 空专题块仍需保留结构和说明
- **WHEN** 某个专题在当前筛选范围内没有匹配数据
- **THEN** 系统必须返回该专题块的空状态说明或空数据结构
- **THEN** 前端不得因为单个专题无数据而移除整个专题定义

### Requirement: 系统必须为经营报表建立指标来源和公式定义
系统 MUST 为经营报表中的每个核心统计指标建立明确的来源定义。每个指标至少 MUST 具备业务名称、来源表、来源字段、时间字段、聚合方式、适用筛选条件和口径说明。派生指标 MUST 额外声明公式，不得在不同专题中由开发者各自猜测实现。

#### Scenario: 核心指标可追溯到来源表和字段
- **WHEN** 开发者、测试人员或业务复核某个经营报表指标
- **THEN** 系统必须能够说明该指标来自哪些表、哪些字段和哪类聚合方式
- **THEN** 不得存在“页面上有值但无法说明从哪张表算出来”的指标

#### Scenario: 派生指标具备统一公式定义
- **WHEN** 系统生成线索匹配客户率、客户有商机率、激活率、回款率或逾期占比等派生指标
- **THEN** 系统必须使用统一声明的公式进行计算
- **THEN** 不得在不同专题中使用不同分母、不同时间范围或不同对象池口径

### Requirement: 系统必须在经营报表中区分期间发生口径与期末截面口径
系统 MUST 为经营报表中的不同专题显式定义时间语义。期间发生类专题 MUST 按筛选时间范围内发生的业务事件计算；期末截面类专题 MUST 按截至筛选结束日的对象池或状态池计算。系统 MUST 在接口或页面元信息中保留各专题所使用的时间依据说明，不得把所有专题都简单套用同一种时间字段。

#### Scenario: 线索新增趋势按创建时间统计
- **WHEN** 用户查看线索新增、来源或趋势专题
- **THEN** 系统必须按 `leads.created_at` 落在筛选时间范围内的线索计算结果
- **THEN** 不得把线索跟进时间或转化时间误当作新增时间

#### Scenario: 线索转化按 cohort 统计
- **WHEN** 用户查看线索转化专题
- **THEN** 系统必须以筛选时间范围内创建的线索作为基准 cohort
- **THEN** 转化结果必须按截至 `endDate` 的状态进行统计，而不是简单按 `turned_at` 过滤

#### Scenario: 客户池与风险池按期末截面统计
- **WHEN** 用户查看客户池、无商机客户、风险商机或未验收合同等截面专题
- **THEN** 系统必须按截至筛选结束日的对象状态计算结果
- **THEN** 不得仅因为对象创建时间不在筛选期内就把它从期末截面专题中排除

### Requirement: 系统必须在正式交付前完成经营报表统计对账
系统 MUST 在经营报表快照正式返回前完成核心指标对账。至少 MUST 校验摘要指标与专题表格汇总、分项求和与总计、期间发生口径与截面口径边界是否一致。若关键指标对账失败，系统 MUST 阻断错误快照交付或将对应专题标记为不可用并返回明确错误原因。

#### Scenario: 摘要值与分项求和不一致时阻断交付
- **WHEN** 某个核心专题的摘要指标与分项表格求和结果不一致
- **THEN** 系统必须阻断该经营报表快照的正式交付或阻断该专题交付
- **THEN** 系统不得继续返回一个“看起来完整”的错误报表

#### Scenario: 时间口径不一致时专题被标记异常
- **WHEN** 某个专题中的总计、图表和明细引用了不同时间字段或不同时间边界
- **THEN** 系统必须将该专题标记为口径异常
- **THEN** 前端必须能看到明确的异常提示，而不是继续展示不可信数字

### Requirement: 系统必须限制经营报表专题查询对 CRM 的冲击
系统 MUST 对经营报表专题查询提供性能保护，至少包括缓存、并发限制、单专题超时和局部降级策略。系统 MUST 优先保证总览和经营摘要可用，并在重专题慢查询或失败时优先降级该专题，而不是让整页长时间无响应或直接打爆 CRM 读库。

#### Scenario: 多个专题同时请求时命中并发限制
- **WHEN** 同一经营报表上下文下同时触发多个重专题详情请求
- **THEN** 系统必须限制并发查询数量
- **THEN** 不得让无上限专题并发直接冲击 CRM 数据源

#### Scenario: 单个专题超时时只降级该专题
- **WHEN** 某个专题详情查询超过系统允许的超时时间
- **THEN** 系统必须优先将该专题标记为超时或降级
- **THEN** 不得因为单个专题超时而拖垮整个经营报表页面

### Requirement: 系统必须在经营报表中仅使用 CRM 回款与应收口径
系统 MUST 在经营报表的收款专题中仅使用 CRM 内部 `contracts`、`received_payment_plans` 和 `received_payments` 相关数据。系统 MUST 提供至少以下视图：收款摘要、月度回款趋势、应收状态、收款负责人拆解，以及与验收或合同归属相关的回款风险视图。系统 MUST NOT 引入周报管理口径、Excel 上传快照或其它外部财务来源。

#### Scenario: 收款摘要只展示 CRM 口径
- **WHEN** 用户查看收款摘要
- **THEN** 系统必须基于 CRM 应收计划与实际回款数据返回摘要指标
- **THEN** 页面不得再展示“周报口径”“管理口径”或 Excel 来源说明

#### Scenario: 回款按团队汇总时以合同归属为准
- **WHEN** 用户按团队或部门查看收款情况
- **THEN** 系统必须通过 `contract_id` 关联合同归属的部门或负责人进行汇总
- **THEN** 不得直接使用 `received_payments.organization_id` 作为团队经营归属口径

#### Scenario: 收款趋势按实际回款日期统计
- **WHEN** 用户查看月度回款趋势
- **THEN** 系统必须按 `received_payments.receive_date` 进行时间聚合
- **THEN** 不得把应收计划日期或合同创建日期误当作实际回款时间

#### Scenario: 收款摘要与状态拆分能够对账
- **WHEN** 系统返回收款摘要、应收状态和收款负责人专题
- **THEN** 核心应收总额和逾期总额必须能够与状态拆分或负责人拆分结果对账
- **THEN** 若无法对账，系统不得继续把该收款专题作为可信结果交付

### Requirement: 系统必须提供经营报表的统一页面状态反馈
系统 MUST 为经营报表提供可感知的页面状态反馈，至少覆盖：首次加载、筛选刷新、无数据、无权限、导出中、导出失败和数据延迟说明。状态反馈 MUST 同时包含中文说明和下一步指引，不得只通过颜色差异表达状态。

#### Scenario: 首次加载经营报表时展示统一加载态
- **WHEN** 用户首次进入经营报表页面且快照尚未返回
- **THEN** 页面必须展示统一的加载状态
- **THEN** 不得出现空白页面或只有局部零散加载动画

#### Scenario: 当前范围无数据时展示空状态
- **WHEN** 用户在当前部门范围和时间范围下没有匹配数据
- **THEN** 页面必须展示“当前范围无数据”的明确提示
- **THEN** 页面必须继续保留当前筛选条件与重试或调整筛选的入口

#### Scenario: 核心统计未通过校验时展示阻断状态
- **WHEN** 后端判定经营报表快照中的核心统计结果未通过一致性校验
- **THEN** 页面必须展示明确的“结果校验未通过”状态
- **THEN** 页面不得继续把该批数字当作可信经营结果展示给用户

### Requirement: 系统必须支持经营报表导出与审计留痕
系统 MUST 为经营报表提供受控导出能力，并为查看和导出行为记录独立审计。审计至少 MUST 记录操作者、当前部门范围、时间范围、生成时间和导出格式。导出内容 MUST 与当前快照保持一致，不得导出未授权范围或与页面不一致的数据。

#### Scenario: 用户导出当前经营报表结果
- **WHEN** 已获得经营报表导出权限的用户触发导出
- **THEN** 系统必须导出与当前筛选条件一致的经营报表结果
- **THEN** 导出行为必须写入独立审计记录

#### Scenario: 用户查看经营报表时记录访问审计
- **WHEN** 用户成功生成或打开经营报表快照
- **THEN** 系统必须记录一次经营报表访问审计
- **THEN** 审计中必须包含本次部门范围和时间范围快照
