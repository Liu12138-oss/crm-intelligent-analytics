## Context

线上排查显示，接口慢并非由服务器 CPU、内存、磁盘容量或 `users_departments` 查询索引缺失直接造成，而是由应用运行态持久化和身份链路抖动叠加放大。当前 `sqlAuditRecords` 长期追加在 `/srv/crm-intelligent-analytics/shared/.runtime/app-storage.json` 中，文件已增长到百兆级；每次 SQL 审计写入都会同步序列化并重写完整运行态文件，导致 Node.js 事件循环被阻塞。登录和受保护接口又会触发 CRM 实时身份解析，连接池偶发等待超过身份查询超时阈值时，会把登录耗时放大。

现有约束：

- 登录、会话、权限、模板可见性、最近查询、导出、合同审核和企业微信机器人上下文不能因治理方案改变用户可见行为。
- SQL 审计仍是治理后台和排障的重要能力，不能简单关闭。
- 敏感配置不得写入源码、文档、测试夹具或日志。
- 生产环境当前采用 systemd + Nginx + 共享 `.runtime/` 目录部署。

## Goals / Non-Goals

**Goals:**

- 将 SQL 审计从高频同步整文件写入中解耦，避免审计量增长线性拖慢普通接口。
- 为运行态文件建立保留上限、归档、巡检和告警机制。
- 为 SQL 审计引入可灰度的异步写入和独立持久化路径。
- 降低登录后页面初始化期间的重复身份解析和连接池抖动影响。
- 补齐 Nginx、后端请求、CRM 上游、SQL 审计队列、运行态体积和事件循环延迟观测。
- 每个阶段都能通过开关灰度和回滚。

**Non-Goals:**

- 不重做登录认证、会话模型或统一鉴权架构。
- 不改变现有 SQL 审计字段的用户可见含义。
- 不把本系统升级为数据库级全局 SQL 审计系统。
- 不把 SQL 审计写入 CRM 业务库。
- 不在本变更内改造前端视觉或页面信息架构。

## Decisions

### 1. 先控量止血，再迁移存储

先为 `sqlAuditRecords` 增加运行态保留上限和历史归档脚本，再推动独立审计存储。这样可以快速降低线上文件体积和同步写盘成本，同时保留后续迁移的完整数据来源。

替代方案是立即迁移到新表。该方案长期正确，但需要数据模型、迁移脚本、双写对账和查询改造，无法作为低风险第一步。

### 2. 普通只读 SQL 审计走异步路径，高风险审计保留强可靠路径

普通只读 SQL 审计记录先入队，再由后台批量持久化。权限阻断、写回、导出、敏感 reveal 和失败类高风险事件保留同步确认或更高优先级队列，避免治理证据缺失。

替代方案是所有审计全异步。该方案接口性能最好，但在异常退出时可能扩大关键审计丢失窗口，不适合治理场景。

### 3. SQL 审计独立存储使用应用侧数据库或等价持久化后端

新增 `sql_audit_records` 或等价独立存储，承载高频流水。`app-storage.json` 仅保留低频运行态、会话、模板、企业微信上下文和合同审核必要状态。

替代方案是继续使用 JSON 分片文件。分片文件可以作为过渡归档方案，但分页检索、索引、按用户和时间过滤能力弱，长期维护成本更高。

### 4. 身份链路只做短缓存和合并查询，不放弃实时权限

会话用户解析和能力快照可以复用短 TTL 缓存，并在权限策略变更时失效。身份上下文查询可以合并 SQL 或减少重复查询，但不得以长期授权快照替代实时权限。

替代方案是延长缓存到小时级。该方案能降低数据库压力，但会扩大权限变更生效延迟，不符合当前治理要求。

### 5. 可观测性作为上线门槛

在实施异步审计和独立存储前，先补齐 request time、upstream time、CRM SQL 耗时、审计队列长度、运行态文件大小和事件循环延迟。没有这些指标，就无法判断优化是否真正降低了用户感知延迟。

## Risks / Trade-offs

- 归档脚本误处理运行态数据 → 脚本只允许修改 `sqlAuditRecords`，执行前强制备份，执行后校验关键数组数量。
- 异步审计在进程异常退出时存在短窗口丢失 → 高风险审计保留强可靠路径，普通只读审计设置队列刷新间隔和退出前 flush。
- 独立审计表不可用导致写入积压 → 业务主链路不阻塞，记录降级告警，队列超过阈值后暂停低风险只读审计。
- 身份短缓存导致权限变更短暂延迟 → TTL 控制在分钟级以内，治理策略变更主动清理相关缓存。
- Nginx 日志格式变更影响现有采集 → 先新增独立格式或新日志文件灰度，再切换采集规则。

## Migration Plan

1. 在测试环境使用脱敏运行态文件验证 SQL 审计归档脚本，确认只修改 `sqlAuditRecords`。
2. 生产低峰期备份 `app-storage.json`，执行一次性归档，保留最近记录。
3. 发布运行态保留上限配置，默认 `SQL_AUDIT_RUNTIME_MAX_RECORDS=5000`。
4. 发布 Nginx 和后端性能观测，确认登录、能力快照、模板、历史查询具备耗时分解。
5. 灰度启用普通只读 SQL 审计异步写入，观察队列长度、失败率和接口 P95。
6. 新增独立 SQL 审计存储，开启双写对账。
7. 审计中心读取切到新存储，保留旧运行态读取作为兼容回退。
8. 关闭旧运行态高频审计写入，仅保留低频运行态。
9. 优化身份查询短缓存、合并查询和连接池健康指标。

回滚策略：

- 归档异常时停止应用并恢复备份的 `app-storage.json`。
- 异步审计异常时设置 `SQL_AUDIT_ASYNC_ENABLED=false`，回退到同步写入和保留上限。
- 独立审计存储异常时关闭双写或切回运行态读取。
- 身份缓存异常时将 TTL 调回默认值或关闭新增缓存。

## Open Questions

- 独立审计存储最终使用现有应用存储 MySQL、独立 schema，还是后续引入专用日志存储，需要结合生产资源确认。
- 历史 SQL 审计归档保留周期和审计查询入口是否需要支持跨归档文件检索，需要由治理负责人确认。
- 高风险审计的最小强可靠集合是否仅包含写回、阻断、导出和 reveal，还是还要包含所有失败 SQL，需要实现前再做一次审计口径确认。
