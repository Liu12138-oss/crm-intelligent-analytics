## Why

线上登录、能力快照、模板列表、最近查询等接口已经出现秒级等待，排查显示瓶颈主要来自 SQL 审计记录无限追加到 `app-storage.json` 后同步整文件持久化，以及 CRM 身份解析连接池偶发超时被放大到登录主链路。该问题会随审计量持续恶化，需要在不改变现有业务功能、权限边界和审计留痕的前提下，建立长期性能稳定机制。

## What Changes

- 新增生产 API 性能稳定能力，约束运行态文件体积、关键接口耗时观测、CRM 身份查询稳定性、告警门槛和灰度回滚要求。
- 调整 SQL 审计能力：普通只读 SQL 审计不得持续阻塞用户请求主链路；审计记录必须支持运行态保留上限、历史归档和独立存储迁移。
- 引入审计写入降级语义：审计持久化异常时，普通业务接口不得因低风险只读审计落盘失败而失败；高风险、阻断、写回、导出和 reveal 审计仍需可靠记录。
- 增加生产止血和长期治理路径：先备份归档并限制运行态审计数量，再补齐异步写入、独立审计表、身份短缓存和性能指标。
- 不改变登录、会话、权限判断、模板可见性、分析问数、合同审核、企业微信机器人和治理审计的现有用户可见接口结构。

## Capabilities

### New Capabilities

- `production-api-performance-stability`: 定义生产接口性能稳定、运行态体积控制、身份链路稳定、观测指标、告警门槛、灰度与回滚要求。

### Modified Capabilities

- `sql-execution-auditability`: 增加 SQL 审计非阻塞写入、运行态保留上限、历史归档、独立存储、降级和高风险审计可靠性要求。

## Impact

- 后端模块：`AppStorageService`、`SqlAuditService`、`SqlAuditRepository`、SQL 审计控制器、CRM 只读身份解析服务、会话鉴权服务、运行态配置服务。
- 运行态数据：`/srv/crm-intelligent-analytics/shared/.runtime/app-storage.json` 中的 `sqlAuditRecords` 需要支持备份、归档、保留上限和迁移。
- 数据库：新增应用侧 SQL 审计独立存储表或等价持久化后端，不写入 CRM 业务库。
- 运维配置：新增非敏感性能参数、Nginx upstream 耗时日志字段、后端慢请求和事件循环延迟指标。
- 测试与验证：需要覆盖登录、会话恢复、模板列表、最近查询、SQL 审计列表/详情/reveal、合同审核、企业微信上下文恢复和性能回归。
