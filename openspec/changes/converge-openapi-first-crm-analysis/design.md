## Context

当前分析请求的实际链路包含多层历史兼容路径：先尝试宽业务意图解析，再尝试窄业务意图解析，最后在 AI 不可用时通过本地词表构造 CRM 只读兜底意图；执行阶段又会先尝试 SQLite / MySQL 分析库固定模板，再回到计划执行、OpenAPI 或受控 SQL 路由。这种设计在联软 OpenAPI 不完整时有过联调价值，但现在会造成两个问题：

- 同一句自然语言可能在不同时间走不同路径，标题、指标、明细和建议口径漂移。
- SQLite 脱敏样例和分析库固定模板容易混入正式结果，导致客户、渠道商、商机、订单明细不是真实业务对象。

联软已经提供 OpenAPI，当前目标是将正式分析主链收敛到 OpenAPI 真实取数。分析知识库和 SQLite 脱敏资料仍保留为“理解业务对象、字段、关系、枚举和模板”的知识资产，但不得作为生产结果来源。

## Goals / Non-Goals

**Goals:**

- 自由问数主链只保留统一业务语义入口，输出对象、指标、维度、时间、分析主题和明细需求。
- 正式 CRM 分析只通过联软 OpenAPI 获取真实业务明细和统计基础数据。
- OpenAPI 无法满足时返回明确能力缺口，便于要求联软补接口、字段、筛选或分页能力。
- 四类默认主题和综合经营问题稳定落到 OpenAPI 真实业务数据：商机、渠道商、客户报备、订单。
- 保留现有结果包、校验、企微模板卡片和 Web 详情能力，避免交付层重写。

**Non-Goals:**

- 不删除 SQLite 导入、分析库治理、历史测试用例和诊断代码；本变更只把它们从正式分析主链默认执行路径移除。
- 不新增联软未提供的写接口，不改变客户/商机创建、日报、跟进写回等其它企业微信能力。
- 不让 AI 自由执行 SQL、自由探测 schema 或直接访问 CRM 数据库。
- 不在本变更中解决联软 OpenAPI 本身缺失字段或缺失接口的问题；本变更会形成缺口清单。

## Decisions

### 决策 1：统一业务语义解析作为唯一自由问数主入口

自由问数入口只调用 `business-analysis-intent-pack` 获取业务对象、指标、维度、时间、分析模式和输出偏好，再由 `BusinessAnalysisIntentMapperService` 映射为现有 `AnalysisIntent`。

原因：

- 宽业务意图能表达商机、客户报备、渠道商、订单等多对象组合，适合联软渠道 CRM 分析。
- 窄业务意图只覆盖早期三类域和少量指标，容易把综合经营问题压成单一报告。
- 本地词表兜底会绕开 AI 语义理解，继续扩大历史规则分支。

备选方案：

- 保留宽窄双链。问题是仍会因为宽链校验失败进入旧口径。
- 只用本地模板识别。问题是无法覆盖自然语言组合问题和新业务表达。

### 决策 2：正式执行默认 OpenAPI-only

`AnalysisWorkflowOrchestrator` 不再默认调用 `AnalysisWarehouseAnalysisExecutorService.tryExecute`，`AnalysisReadToolRegistryService` 在未命中 OpenAPI 适配器时直接返回能力缺口，不再自动路由到 `GUARDED_READONLY_SQL`。

原因：

- 用户要求明细必须是真实客户、渠道商、商机和订单名，只有 OpenAPI 能承接当前授权范围内真实数据。
- SQLite 快照是脱敏测试资料，MySQL 分析库当前不是用户要使用的生产主链。
- 自动 SQL 兜底会让结果来源不可控，且不符合“联软已提供 OpenAPI 后优先走 OpenAPI”的目标。

备选方案：

- 用环境变量继续默认启用分析库。问题是重启后仍可能走旧路径。
- OpenAPI 失败后自动用 SQLite 模板。问题是会再次出现脱敏明细或标题口径漂移。

### 决策 3：OpenAPI 缺口转成可行动清单

当业务语义已明确但 OpenAPI 适配器、字段、筛选、分页或关联能力不足时，系统返回业务可读的缺口，不展示伪结果。

缺口至少包括：

- 缺哪个对象接口，例如客户报备、订单、商机阶段历史。
- 缺哪个字段，例如渠道商名称、客户名称、阶段中文名、订单金额。
- 缺哪个筛选或排序能力，例如按更新时间、按渠道商、按区域。
- 缺哪个关联关系，例如客户报备到商机、商机到订单。

原因：

- 这能把问题从“系统乱分析”变成“联软需要补什么能力”。
- 不伪造数据比给出错误经营建议更重要。

### 决策 4：SQLite 和分析库保留为知识资产与离线验证

SQLite DDL、字段字典、关系说明、状态枚举和样例数据继续用于：

- 帮助提示词理解联软业务对象和关系。
- 编写模板和回归测试。
- 验证状态枚举中文释义和业务链口径。

但正式请求不得从 SQLite 或 MySQL 分析库返回明细或聚合结果。

## Risks / Trade-offs

- [OpenAPI 当前字段不足导致部分问题不能回答] → 返回明确缺口清单，先完成系统侧收敛，再向联软要求补接口/字段。
- [禁用分析库后可回答范围短期变小] → 优先保证真实准确，避免脱敏样例和历史模板误导业务。
- [旧测试依赖 SQL/SQLite 路径] → 调整主链测试，保留分析库模块自身单测作为离线能力测试。
- [业务综合问题需要多次 OpenAPI 拉取] → 通过任务包和分页上限控制规模，结果包中记录真实来源和缺失区块。

## Migration Plan

1. 新增 OpenSpec 规格，明确 OpenAPI-only 主链和历史路径退出条件。
2. 改造 `AnalysisIntentService`，只以宽业务意图作为自由问数主解析；AI 不可用或结构非法时阻断，不再转窄意图或本地只读兜底。
3. 改造 `AnalysisWorkflowOrchestrator`，默认跳过分析库 / SQLite 执行器。
4. 改造 `AnalysisReadToolRegistryService`，未命中 OpenAPI 适配器时返回受控缺口，不自动进入 SQL。
5. 补回归测试覆盖商机、渠道商、客户报备、订单、综合经营和 AI 失败场景。
6. 运行后端单测、构建和 OpenSpec 严格校验。

回滚策略：

- 如需临时诊断旧分析库能力，可单独通过治理/测试入口调用分析库模块，不恢复到正式问数主链。
- 如某个 OpenAPI 适配器存在缺陷，可仅禁用对应适配器并返回缺口，不启用 SQLite 或 SQL 伪结果兜底。
