## Context

当前仓库的智能问数已经具备统一 AI 入口、受控查询执行、统一结果包、Markdown 交付和企业微信 / Web 双通道能力，但整体仍停留在“单查询 -> 单报告”的分析平台模型中：

- 入口层虽然能识别对象、时间、排行 / 趋势等基本问数要素，但还不能把“负责人经营分析、区域经营分析、客户经营分析、商机风险分析、线索漏斗分析”映射成受控的主题报告档案。
- 计划器当前主要围绕 `owner-ranking / time-trend / stage-distribution / category-distribution` 等少量 `resultKind` 生成最多 3 个任务，不足以支撑管理分析报告所需的多维展开。
- 结果交付当前能输出摘要、指标卡、图表、表格和 Markdown，但缺少“结构分布、风险观察、重点对象、动作建议”等主题化区块。
- 你提供的广州实验特区 CRM HTML 报告展示了大量可迁移的统计维度，例如漏斗、集中度、区域作战地图、风险观察、重点对象池和动作建议；Hermes 的 CRM 技能则强调“对象、过滤条件、输出字段、模板优先、自然语言结论组织”的顺序，这些都说明我们需要把“问数结果页”升级成“主题驱动报告包”。

同时，本次设计必须继续遵守以下约束：

- 所有补充维度都必须继续命中现有受控安全栈，不得绕开 API-first、白名单、AST、预检、超时和审计。
- 主题报告包不是自由 BI 引擎，一期只围绕高频经营主题提供固定查询包和固定区块模板。
- 统一结果包仍然是事实真源，AI 总结、Markdown 和渠道裁剪只能消费结果包，不能再发起额外取数。
- 管理员与普通角色的权限边界必须继续由统一权限快照控制，不允许因为报告包增加维度就扩大数据范围。

## Goals / Non-Goals

**Goals：**

- 为自由问数引入“主题报告包”层，使系统能按主题返回更丰富的多维统计结果。
- 让入口理解层输出主题档案、报告深度和分析侧重点，驱动后续任务编排与展示层级。
- 让受控分析编排支持主题级多任务查询包，并保持统一时间口径与统一权限快照。
- 让结果交付层支持区块化经营分析报告，覆盖执行摘要、核心指标、结构分布、趋势、风险、重点对象和行动建议。
- 吸收 Hermes 的模板优先、维度有序展开和自然语言结论组织方式，同时吸收 HTML 报告中的高价值统计维度。

**Non-Goals：**

- 不把系统改造成完全自由配置的 BI 平台或多页报表设计器。
- 不在本次变更中一次性覆盖 HTML 报告里的全部专题页，优先聚焦高频问数可落地的主题维度。
- 不改变固定功能入口、写操作链路、统一鉴权和企业微信非问数程序流。
- 不让 AI 直接决定查询范围、自由探索 schema 或绕过受控工具边界。

## Decisions

### 决策 1：新增“主题报告档案”作为入口理解与后续编排之间的桥梁

统一 AI 入口理解层在现有 `domain / metrics / dimensions / temporalSlot / resultIntent` 之外，新增以下最小主题语义：

- `analysisFacetProfile`：主题报告档案，例如 `owner-performance-ranking`、`region-operations`、`customer-operations`、`opportunity-risk`、`lead-funnel`、`contract-conversion`。
- `analysisDepth`：报告深度，例如 `snapshot`、`standard`、`deep-dive`。
- `analysisFocus`：本次问数明确要求展开的重点维度，例如 `trend`、`risk`、`industry`、`region`、`customer-contribution`。

原因：

- 仅靠现有 `resultKind` 无法决定应补哪些维度。
- 主题档案可以把“用户问的是什么经营专题”和“系统要展开哪些区块”稳定连接起来。

备选方案：

- 继续把所有主题都压到 `resultKind` 上。问题是表达力不足，且会把不同主题的统计要求混在一起。
- 让前端根据问题文本再猜专题。问题是前后端事实容易漂移。

### 决策 2：新增“查询任务包注册表”，按主题生成固定任务组合

在编排层新增主题任务包注册表，每个 `analysisFacetProfile` 对应一个固定查询包，包含：

- 必选任务：必须成功，否则主题报告不能成立。
- 可选任务：补充结构、趋势、风险、重点对象、贡献来源等区块，可在失败时降级隐藏。
- 任务排序：决定报告中区块的默认阅读顺序。

例如 `owner-performance-ranking` 首版默认包：

1. 负责人金额排名
2. 月度趋势
3. 商机阶段分布
4. 区域 / 部门贡献
5. 重点客户贡献
6. 风险商机摘要

原因：

- 这与 Hermes 的“先明确对象与过滤条件，再按固定维度输出结论”的方式一致。
- 任务包能让问数输出稳定，不再完全依赖模型临时拼装。

备选方案：

- 让 AI 每次自由决定补哪些查询。问题是不可控、难回归。
- 为每种主题写死页面，不走统一结果包。问题是复用性差。

### 决策 3：将单报告结构升级为“区块化主题报告”

统一结果交付层新增区块化报告结构，建议固定区块类型：

- `summary`
- `metric-strip`
- `distribution`
- `trend`
- `risk`
- `focus-list`
- `detail-table`
- `actions`

不同主题只是在区块组合和区块文案上不同，不再每次临时拼接。

负责人排名主题首版建议的核心派生指标：

- `TOP1贡献占比`
- `TOP3贡献占比`
- `平均单笔商机金额`
- `第一名领先第二名差距`
- `命中商机数`
- `负责人分组数`
- `高风险商机数`

原因：

- HTML 报告本质上也是“主题 + 区块”组织，而不是单张大表。
- 统一区块模型后，Web 和企业微信可以共享同源结果，只裁剪层级。

备选方案：

- 继续只保留 `metricCards/chartBlocks/tableBlocks`。问题是表达层太弱。
- 直接把 HTML 报告页签模型搬进结果包。问题是过重。

### 决策 4：主题报告包支持 3 到 8 个任务，并区分必选 / 可选 slice

当前最多 3 个任务的限制不足以承载 richer report。新设计中：

- Web `standard/deep-dive` 报告允许 3 到 8 个受控任务。
- 企业微信默认允许 2 到 4 个任务，但仍复用同一主结果包。
- 可选 slice 失败时可以降级，只要必选 slice 成功，报告仍可交付；但系统必须记录缺失区块和失败原因。

原因：

- richer report 需要更多 supporting slices。
- 不是每个维度都应该阻断整个结果交付。

备选方案：

- 所有任务都强制成功。问题是稳定性差。
- 继续最多 3 个任务。问题是表达力不够。

### 决策 5：主题报告生成继续以统一权限快照与统一时间口径为唯一事实边界

每个主题任务包中的所有查询都必须复用：

- 同一 `scopeSnapshot`
- 同一 `temporalSlot / temporalScope`
- 同一执行模式选择结果

特别是管理员口径：

- 管理员只保留组织级边界，不得继承登录快照里的单部门或单负责人限制。
- 主题报告新增维度时，也不得引入新的隐式范围收窄。

原因：

- richer report 一旦多个 slice 用了不同范围，管理层看到的就会是错的。
- 你当前反馈的“明明有很多人却只出两个人”就是这类问题。

备选方案：

- 不同区块各自补充过滤条件。问题是口径漂移。

### 决策 6：把 HTML 报告和 Hermes 资产吸收到“主题知识层”，而不是直接模板化页面

将两个参考来源拆解为知识资产：

- HTML 报告提供“可展示维度清单”和“区块组织顺序”。
- Hermes 提供“对象、过滤条件、输出字段、模板优先、自然语言结论组织方式”。

首版吸收方式：

- 在问数知识层中为每个主题维护 `recommendedDimensions`、`recommendedSections`、`validatedExamples`。
- 在主题任务包注册表中引用这些资产，决定默认任务组合。

原因：

- 直接照搬 HTML 页面不利于系统化落地。
- Hermes 真正可复用的是工作流与口径组织，不是页面本身。

### 决策 7：前端结果页升级为“区块化报告页”，但保持统一结果底座不变

前端首版不做复杂可视化驾驶舱，而是把详情页和工作台结果区升级为多区块报告：

- 执行摘要
- 关键指标
- 结构 / 趋势图
- 风险观察
- 重点对象列表
- 明细表
- 行动建议

企业微信只消费其中的摘要、关键指标、风险提示和少量明细摘要。

原因：

- 这样能快速提升报告信息密度，又不需要一次性重做整套前端信息架构。

## Risks / Trade-offs

- [主题任务包过多导致执行变慢] → 通过 `analysisDepth` 和渠道级任务上限控制默认任务数。
- [可选区块失败导致报告不完整] → 在结果包里显式记录缺失区块和失败原因，不伪造内容。
- [维度越多越容易口径漂移] → 所有 slice 继续复用同一时间口径与同一权限快照，并执行一致性校验。
- [前端一次性展示太多信息会过载] → Web 采用分区折叠和主题块顺序，企业微信只展示裁剪版。
- [HTML 报告维度过多，首版实现压力大] → 分主题分批落地，先做高频问数主题。

## Migration Plan

1. 新增 `analysisFacetProfile / analysisDepth / analysisFocus` contract，并更新 AI 入口识别。
2. 新增主题任务包注册表，让 `AnalysisQueryPlannerService` 按主题生成多任务包。
3. 扩展结果包结构和报告生成器，支持 richer ranking/report sections。
4. 先落地第一个主题 `owner-performance-ranking`，验证 richer report 输出。
5. 再扩展 `region-operations`、`opportunity-risk`、`customer-operations` 等主题。
6. 最后再考虑吸收线索、代理商、产品方案等需要更多数据表支持的专题。

回滚策略：

- 若 richer report 任务包稳定性不足，可按主题关闭增强任务包，回退到旧的单结果报告生成。
- 若某个主题的可选区块频繁失败，可仅关闭该区块，不回退整个主题能力。
- 若前端区块化报告显示异常，可继续复用统一结果包，仅回退到旧详情页布局。

## Open Questions

- `analysisDepth` 首版是否完全由 AI 决定，还是由 Web / 企业微信分别给默认值再让 AI 覆写。
- 负责人排名主题里的“重点客户贡献”首版是按客户金额 TopN，还是按客户数 TopN。
- 风险商机口径首版采用“超 30/60/90 天未跟进”还是只做单一高风险集合。
- 线索、客户、商机综合驾驶舱是否作为后续单独主题能力，而不是直接混入当前问数结果页。
