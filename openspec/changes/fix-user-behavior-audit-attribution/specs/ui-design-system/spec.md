## MODIFIED Requirements

### Requirement: 页面必须把内部业务代码转换为中文标签

系统 MUST 在面向用户的 Web 页面中将后端返回的状态码、风险等级、审计事件类型、入口场景、目标工作流、fallback 原因、执行模式、数据来源、动作类型、模板状态、服务状态和视图类型转换为中文业务标签后展示。页面 MUST NOT 直接把全大写枚举、英文状态等级、kebab-case 原因码或内部工作流标识作为主要展示文案。对于后端当前已经定义在 `AuditEventType` 中的审计事件，前端 MUST 提供明确中文标签；只有后端未来新增且前端尚未登记的事件类型，才允许显示“未知事件类型（事件码）”或同等中文兜底。

#### Scenario: 已定义审计事件类型被展示

- **WHEN** 审计接口返回后端当前 `AuditEventType` 中已经定义的事件类型
- **THEN** 页面必须显示对应中文事件名称
- **THEN** 页面不得显示“未知事件类型”

#### Scenario: 后端新增尚未登记的审计事件类型

- **WHEN** 审计接口返回前端字典尚未登记的未来事件类型 `NEW_INTERNAL_EVENT`
- **THEN** 页面必须显示“未知事件类型（NEW_INTERNAL_EVENT）”或同等中文兜底
- **THEN** 页面不得直接只显示 `NEW_INTERNAL_EVENT`

## ADDED Requirements

### Requirement: 用户行为审计列表必须展示通用行为字段

系统 MUST 在审计中心的用户行为审计分区展示通用行为字段，而不是默认展示 AI 专属字段。用户行为审计主列表 MUST 至少展示时间、用户、绑定状态、入口、事件类型、业务对象、操作摘要、结果和风险等级。AI 入口场景、目标工作流、fallback、fallback 原因和执行轨迹摘要 SHOULD 保留在 AI 审计分区或详情视图中。

#### Scenario: 普通业务审计事件没有 AI 字段

- **WHEN** 登录、日报、通知、治理、合同审核或跟进写回事件没有 AI 入口场景和 fallback 原因
- **THEN** 用户行为审计主列表仍必须展示完整的通用行为信息
- **THEN** 页面不得在主表中堆叠多个无意义空列

#### Scenario: AI 审计事件查看执行轨迹

- **WHEN** 审计人员查看 AI 问数或 AI 入口治理事件
- **THEN** 页面必须仍能查看入口场景、目标工作流、AI 兜底原因和执行轨迹摘要

### Requirement: 企业微信未绑定用户必须以业务可懂方式展示

系统 MUST 在前端审计中心以业务可懂方式展示未绑定企业微信用户。若后端返回未绑定企业微信用户主体，页面 MUST 显示“未绑定 CRM 用户（企业微信：xxx）”或同等中文表达；页面 MUST NOT 将机器人、应用、`unknown_wecom_sender` 或内部占位值作为用户姓名直接展示。

#### Scenario: 未绑定企业微信用户触发拒绝审计

- **WHEN** 审计列表返回 `actorBindingStatus = UNBOUND_WECOM` 且包含企业微信 senderId
- **THEN** 用户列必须显示该企业微信用户未绑定 CRM
- **THEN** 入口或通道列可以显示“企业微信机器人”
- **THEN** 用户列不得显示为机器人

### Requirement: 审计事件类型筛选必须支持中文选择

系统 MUST 在审计中心为事件类型筛选提供可搜索中文选择体验。筛选控件 MUST 显示中文事件名，支持按中文名或事件码搜索，并在提交查询时继续向后端传递事件码。

#### Scenario: 管理员按日报事件筛选

- **WHEN** 管理员在事件类型筛选中搜索“日报”
- **THEN** 页面必须展示日报相关事件中文选项
- **THEN** 选择后请求参数必须使用对应事件码
