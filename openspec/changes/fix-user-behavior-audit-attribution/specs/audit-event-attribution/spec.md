## ADDED Requirements

### Requirement: 审计归因修复不得破坏现有业务功能

系统 MUST 在修复用户行为审计归因、展示字段和企业微信真实发送人解析时保持现有业务功能兼容。任何新增审计字段、前端展示调整、企业微信入站解析调整或历史数据修复能力都不得改变既有 Web 问数、企业微信机器人问数、日报提醒与汇总、跟进写回、主动通知、权限治理、合同审核、导出和 SQL 审计的业务执行结果。新增字段 MUST 保持可选，旧审计记录 MUST 继续可读、可筛选、可展示。

#### Scenario: 旧审计记录缺少新增字段

- **WHEN** 审计列表读取历史记录，且该记录没有行为人类型、绑定状态、通道代理、动作摘要或目标摘要等新增字段
- **THEN** 审计列表接口必须正常返回该记录
- **THEN** 前端必须使用现有字段或占位展示该记录
- **THEN** 系统不得因为新增字段缺失导致列表加载失败

#### Scenario: 审计归因修复后既有问数链路仍可用

- **WHEN** 用户通过 Web 工作台或企业微信机器人发起原本已支持的问数请求
- **THEN** 系统必须保持原有权限校验、查询执行、结果返回、消息回传和审计写入链路可用
- **THEN** 本次变更只能补充或修正审计归因字段，不得改变查询结果口径

#### Scenario: 审计展示调整不影响其它审计分区

- **WHEN** 审计中心调整用户行为审计主列表字段
- **THEN** AI 审计分区必须仍能查看入口场景、目标工作流、AI 兜底原因和执行轨迹
- **THEN** SQL 审计分区必须仍能查看 SQL 摘要、详情和敏感 reveal 流程

### Requirement: 业务审计事件必须区分真实行为人与通道代理

系统 MUST 在业务审计事件中区分真实行为主体和通道代理。真实行为主体 MUST 写入 `actorId`；企业微信机器人、企业微信应用、SDK botId、网关 source、定时调度器等只允许作为通道代理或系统入口信息记录，不得覆盖真实行为主体。已绑定 CRM 用户通过企业微信机器人发起查询、跟进写回、日报回复或 CRM 创建动作时，审计事件的 `actorId` MUST 使用该 CRM 用户 ID。

#### Scenario: 已绑定 CRM 用户通过企业微信机器人查询

- **WHEN** 企业微信用户已通过 `wx_user_maps` 和 `wx_users` 映射到 CRM 用户，并通过机器人发起查询
- **THEN** 查询相关业务审计事件必须使用 CRM 用户 ID 作为 `actorId`
- **THEN** 机器人 ID 或应用 ID 只能记录在通道代理字段中

#### Scenario: 机器人代理转发用户消息

- **WHEN** 企业微信回调同时包含真实发送人和机器人标识
- **THEN** 系统必须将真实发送人作为身份解析输入
- **THEN** 系统不得把机器人标识当作查询人或动作执行人

### Requirement: 未绑定企业微信用户必须形成可识别审计主体

系统 MUST 对未绑定 CRM 的企业微信用户形成稳定、可检索、可展示的审计主体。若企业微信 senderId 无法映射 CRM 用户，但消息已经到达机器人入口并触发认证失败、拒绝、维护降级或其它受控阻断，系统 MUST 使用 `wecom:<senderId>` 或等价稳定格式作为 `actorId`，并标记该主体为未绑定企业微信用户。

#### Scenario: 未绑定企业微信用户被入口认证拒绝

- **WHEN** 企业微信 senderId 无法映射 CRM 用户
- **THEN** 系统必须记录一条认证失败或拒绝审计事件
- **THEN** 审计事件必须能展示“未绑定 CRM 用户”和该企业微信 senderId
- **THEN** 审计事件不得把机器人显示为行为人

#### Scenario: 缺少可靠发送者标识

- **WHEN** 企业微信回调无法解析出可靠真实发送者
- **THEN** 系统必须拒绝该消息进入正式业务链路
- **THEN** 审计事件必须使用明确的系统入口主体记录该接入失败
- **THEN** 审计事件必须保留可排查的请求摘要，但不得包含敏感密钥或完整原始凭证

### Requirement: 企业微信入站消息必须优先解析真实发送者

系统 MUST 在企业微信入站消息适配层优先解析真实发送者。解析优先级 MUST 优先使用官方 SDK 或 webhook 中表示用户的字段，例如 `from.userid`、`sender.id`、`userid`；只有在这些字段不存在时，才允许使用顶层 `senderId`。当顶层 `senderId` 与机器人 ID 或应用 ID 相同，系统 MUST 将其视为通道代理候选，而不是直接视为真实用户。

#### Scenario: SDK 回调同时包含机器人 senderId 和真实 from.userid

- **WHEN** 企业微信 SDK 回调中顶层 `senderId` 是机器人或应用标识，且 `from.userid` 是真实发送人
- **THEN** 归一化入站消息必须将 `from.userid` 作为真实 `senderId`
- **THEN** 顶层机器人或应用标识必须保留为原始 sender 或通道代理信息

#### Scenario: 群聊消息包含真实发送者

- **WHEN** 企业微信群聊消息包含 `chatid` 和真实发送者标识
- **THEN** 系统必须以“外部会话 + 真实发送者”隔离会话和审计主体

### Requirement: 业务审计事件必须提供通用展示摘要

系统 MUST 为业务审计事件提供适合审计列表展示的通用摘要字段。通用摘要至少 SHOULD 包含行为人展示名、行为人类型、外部身份、绑定状态、入口通道、通道代理、动作摘要、目标类型、目标 ID、目标摘要和结果摘要。AI 入口理解、目标工作流、fallback 原因等 AI 专属字段不得作为用户行为审计主列表判断事件完整性的必要字段。

#### Scenario: 日报事件没有 AI 入口字段

- **WHEN** 日报草稿保存、确认、提醒、汇总或关闭事件没有 AI 入口场景和 fallback 原因
- **THEN** 用户行为审计列表仍必须能展示用户、入口、事件类型、业务对象、操作摘要、结果和时间
- **THEN** 页面不得因为 AI 专属字段为空而让该事件看起来缺失审计信息

#### Scenario: 跟进写回事件展示目标对象

- **WHEN** 企业微信跟进写回草稿、确认、成功、失败或取消事件被记录
- **THEN** 审计事件必须尽量关联草稿 ID、目标客户或商机，并提供业务对象摘要

### Requirement: 用户行为审计汇总必须基于真实事件字段计算

系统 MUST 基于真实审计事件字段计算用户行为审计汇总。企业微信查询占比不得使用固定占位值，必须按具有查询语义的事件和 `channel = wecom-bot` 或等价通道字段计算。

#### Scenario: 审计列表返回企业微信查询占比

- **WHEN** 审计列表同时包含 Web 查询事件和企业微信查询事件
- **THEN** `wecomQueryRatioPercent` 必须等于企业微信查询事件数除以全部查询事件数后的百分比
- **THEN** 当没有查询事件时，该占比必须返回 0

### Requirement: 历史错误归因审计必须支持受控修复

系统 MUST 提供受控维护能力修复历史错误归因审计。维护能力 MUST 默认以 dry-run 方式运行，并输出可修复数量、可映射 CRM 用户数量、未绑定企业微信用户数量和无法判断数量。只有显式 apply 时才允许写入修复结果。

#### Scenario: 历史事件可从快照恢复企业微信 senderId

- **WHEN** 历史审计事件 actor 疑似机器人，但快照中可恢复真实企业微信 senderId
- **THEN** 修复工具必须尝试通过 CRM 企业微信映射解析真实 CRM 用户
- **THEN** 解析成功时必须将 actor 修复为 CRM 用户
- **THEN** 原机器人或应用标识必须保留在通道代理或迁移备注中

#### Scenario: 历史事件无法映射 CRM 用户

- **WHEN** 历史审计事件可恢复企业微信 senderId 但无法映射 CRM 用户
- **THEN** 修复工具必须将 actor 标记为未绑定企业微信用户
- **THEN** 修复工具不得把该事件改写为机器人行为
