## Context

当前仓库已经提供 `AI 模型` 治理入口，核心能力聚焦于 AI Profile 的新增、编辑、测试连接和全局激活；前端页面为 [AiModelProfilePage.vue](/d:/code/CRM/frontend/src/pages/governance/AiModelProfilePage.vue)，后端治理接口由 [ai-model-governance.controller.ts](/d:/code/CRM/backend/src/modules/governance/ai-model-governance.controller.ts) 提供。与此同时，企业微信上下文裁剪与追问复用仍有几处固定实现：

- 企业微信会话轮次压缩在 `compressTurns()` 中写死为“超过 8 轮后压缩，只保留最近 6 轮”；
- `latestQuestion`、`latestSummary` 当前没有统一长度治理；
- Web 解释型追问复用上一轮结果时，没有后台可调的失活时间策略；
- 现有 `连接策略` 和 `治理策略` 主要负责权限、会话并发、心跳和保留天数，不适合继续承载 AI 运行时专属的上下文收敛策略。

这次变更需要把“模型接入参数治理”和“AI 上下文治理”统一收敛到一个更准确的“AI配置”中心，同时尽量保持现有权限动作、路由地址和页面入口稳定，避免对已接入的权限矩阵、审计查询和前端路由造成不必要的兼容成本。

## Goals / Non-Goals

**Goals:**

- 将治理导航、页面标题、权限展示文案中的“AI 模型”统一调整为“AI配置”，但尽量不改变既有路由和动作 key。
- 在 AI 配置中心内新增一组可持久化的 AI 上下文策略，至少覆盖：
  - 问答上下文保留轮次上限
  - 历史摘要保留上限
  - 上一轮问题保留上限
  - 上一轮结果摘要保留上限
  - 普通分析会话失活时长
  - 任务态会话失活时长
- 将企业微信上下文裁剪、Web 解释追问复用与 AI 相关会话失活统一改为消费同一份后台策略。
- 保证后台治理文案仅显示中文业务名称，不向用户暴露 `latestQuestion`、`latestSummary` 等内部字段名。
- 为上下文策略补齐接口、默认值、审计与回归验证要求，保证运维可观测。

**Non-Goals:**

- 不在本次提案中调整 AI Profile 的协议字段、健康检查机制或 Provider 适配逻辑。
- 不迁移 `连接策略` 页面中的在线会话数、并发上限、心跳周期等通用连接治理项。
- 不在本次提案中引入按渠道分别配置多套不同 AI Profile 的多活路由能力。
- 不把所有企业微信业务状态机超时都并入本次治理，只处理 AI 上下文相关的保留和失活策略。

## Decisions

### 决策 1：新增独立 `AiContextPolicyRecord`，归属 AI 配置中心，而不是继续扩展 `AccessPolicyRecord`

本次不把新的上下文长度 / 时效配置继续塞进 `AccessPolicyRecord`。改为在 AI 配置治理域下新增独立的 `AiContextPolicyRecord`、仓储与服务，并挂入 `AppStorageState` 与 AI 配置治理接口。

原因：

- `AccessPolicyRecord` 语义上属于权限、渠道、导出和连接基线，继续叠加 AI 上下文细项会让治理边界变得混杂；
- 这组配置与 AI 入口理解、追问复用和上下文裁剪强绑定，和 AI Profile 一起排查更符合管理员心智；
- 独立对象便于后续扩展更多 AI 运行时治理项，例如解释长度、上下文模板、追问复用开关，而不污染权限策略模型。

备选方案：

- 继续扩展 `AccessPolicyRecord` 并在“连接策略”页面维护。问题是配置归属和用户心智不一致，也不符合本次“收敛到 AI配置 菜单”的目标。

### 决策 2：保留现有路由与权限 key，统一替换用户可见名称为“AI配置”

本次建议继续复用现有路由 `/governance/ai-models`、页面组件 `AiModelProfilePage.vue`、菜单 key `ai-model-governance` 和动作 `ai_profile.manage`，只调整用户可见标签、页面标题、权限目录文案和提示消息。

原因：

- 现有路由、权限矩阵、前端守卫、审计映射和测试矩阵都已经围绕这些 key 建好，直接改 key 会放大改动面；
- 用户实际关注的是“菜单名称”和“页面能力”，不关心内部路由或权限码；
- 保持内部 key 稳定可以把本次变更聚焦在治理能力扩展，而不是权限兼容改造。

备选方案：

- 同时重命名路由、菜单 key 和动作 key。问题是会连带影响权限目录、前端守卫、测试矩阵和审计文案，收益明显低于成本。

### 决策 3：在现有 AI 配置页内增加“上下文策略”分区，而不是再新增一级菜单

前端继续使用现有 AI Profile 治理页作为入口，在页面内新增“上下文策略”区域。推荐布局为：

- 顶部保留当前生效 AI Profile 摘要；
- 中部保留 AI Profile 列表和配置抽屉；
- 同页新增“上下文策略”卡片或二级面板，用于编辑长度与失活阈值。

原因：

- 用户明确要求“放到 AI 模型菜单”，更符合一个入口内集中治理；
- 这组策略与当前激活模型一起排查最常见，例如“某模型 + 某摘要长度”导致追问质量下降；
- 新增独立菜单会进一步稀释治理入口，不利于发现和定位。

备选方案：

- 新增 `AI 上下文策略` 子菜单。问题是会造成入口分裂，也需要新增菜单权限和导航心智。

### 决策 4：为上下文策略新增专用治理接口与服务，不把 `AiModelGovernanceController` 继续做胖

虽然前端入口仍是同一页，但后端建议新增专用的上下文策略治理接口与服务，例如：

- `GET /governance/ai-models/context-policy`
- `PUT /governance/ai-models/context-policy`
- `AiContextPolicyService`
- `AiContextPolicyRepository`
- `ai-context-governance.schema.ts`

并继续复用 `ai_profile.manage` 作为访问动作。

原因：

- 当前 `AiModelGovernanceController` 已经聚焦 Profile 生命周期和健康检查，再把上下文策略直接塞进去会继续膨胀控制器职责；
- 上下文策略的校验规则、默认值、审计快照和返回模型与 Profile 并不相同，拆服务更利于测试；
- 同一路由前缀下拆专用资源，不影响前端入口，又能保持模块清晰。

备选方案：

- 继续在 `AiModelGovernanceController` 中增加上下文策略读写接口。问题是控制器职责继续变大，后续扩展更难维护。

### 决策 5：将长度裁剪与失活判断统一收口到 `AiContextPolicyService`

新增统一的上下文治理服务，对外提供：

- 默认策略读取
- 后台持久化策略读取
- 文本裁剪与摘要截断
- 轮次压缩阈值判断
- 普通分析追问失活判断
- 任务态会话失活判断

企业微信和 Web 不再各自写死阈值，而是只消费该服务给出的结果。

原因：

- 当前长度治理分散在 `compressTurns()`、`updateWorkMemoryAfterResponse()`、Web 解释追问复用等多个位置，后续改阈值容易遗漏；
- 统一服务可以确保“配置值解释、截断策略、过期判断”全链路一致；
- 便于补齐单元测试与回归测试。

备选方案：

- 只在各消费点自行读取配置并裁剪。问题是规则分散，极易出现同名配置在不同链路语义不一致。

### 决策 6：普通分析追问与任务态会话采用不同的失活口径

上下文失活不建议只有一个统一秒数。普通分析问答与活跃任务的业务时效不同，应拆成两类：

- 普通分析会话失活时长：控制 Web 解释追问、改条件追问与企业微信分析追问的上下文复用窗口；
- 任务态会话失活时长：控制日报、跟进、创建客户 / 商机等活跃任务的草稿连续性。

运行态建议：

- 普通分析超过时长后，不再复用旧问题 / 旧摘要，追问按新问题或最小澄清处理；
- 活跃任务超过时长后，系统先清空任务态 memory，再返回新的引导或要求用户重新开始当前任务。

原因：

- 分析追问通常对“上一轮结果仍然可信”敏感，窗口应更短；
- 日报 / 跟进等任务允许用户短暂离开后继续填写，但也不能无限保留旧草稿；
- 两者混用一个值会让管理员难以平衡体验与风险。

备选方案：

- 只提供一个统一失活秒数。问题是无法兼顾分析追问和任务型会话的不同节奏。

### 决策 7：后台全部使用中文治理文案，英文只保留在代码内部

所有页面字段、帮助提示、摘要文案和保存反馈都只显示中文业务名称，例如：

- “上一轮问题保留上限”
- “上一轮结果摘要保留上限”
- “问答上下文保留轮次上限”
- “历史摘要保留上限”
- “普通分析会话失活时长”
- “任务态会话失活时长”

内部字段名可以继续是英文，但不得直接暴露给用户。

原因：

- 用户已明确反馈看不懂 `latestQuestion`、`latestSummary` 这类内部字段名；
- 中文业务名称更适合治理后台，能减少配置误用；
- 有利于后续文档、验收和操作手册统一表达。

备选方案：

- 界面展示“中文 + 英文字段名”。问题是会增加非技术用户理解负担，不符合当前项目文案约束。

## Risks / Trade-offs

- [配置归属从通用治理转向 AI 配置中心后，部分字段看起来和原有连接策略相近] → 明确本次只迁入 AI 上下文相关策略，通用连接与权限阈值继续留在原页面，不做重复治理。
- [旧代码大量依赖硬编码阈值，改为统一服务后可能出现边界行为变化] → 为企业微信上下文裁剪、Web 解释追问复用和任务失活清理分别补单元 / 集成回归。
- [继续保留 `ai-model-governance` 菜单 key 可能导致内部命名与用户可见名称不一致] → 在设计和文档中明确“内部 key 兼容保留，用户界面统一为 AI配置”，避免后续误判为漏改。
- [同一页面承载 Profile 与上下文策略后，页面内容可能变长] → 使用清晰分区和卡片布局，避免新增二级菜单，但也避免所有表单堆在一个抽屉里。
- [普通分析失活后追问体验会更保守] → 在页面和返回文案中明确“上下文已失效，请重新描述或重新发起分析”，降低用户困惑。

## Migration Plan

1. 在 OpenSpec 中新增 `ai-context-governance-policy` 能力，并为 `ai-model-profile-governance`、`ai-entry-intent-orchestration`、`controlled-analysis-orchestration` 补增量 spec。
2. 扩展 `AppStorageState`、默认样例数据和 AI 配置治理接口，加入独立的上下文策略记录。
3. 新增上下文策略治理服务、仓储、schema 和审计事件，保证后端具备持久化与统一读取能力。
4. 改造 `AiModelProfilePage.vue`、导航标签、路由标题和业务文案，将页面统一命名为“AI配置”并新增上下文策略分区。
5. 将企业微信上下文压缩、`workMemory` 写入截断、Web 解释追问复用和任务态失活判断改为统一消费上下文策略服务。
6. 补齐文档、测试和验收场景，覆盖菜单改名、策略保存、超长内容裁剪、会话超时与追问失效说明。

回滚策略：

- 若上下文策略读写或页面结构出现问题，可先回退 AI 配置中心中的上下文策略分区与新接口，保留既有 AI Profile 治理功能；
- 因本次不要求变更现有路由和权限 key，回滚不需要恢复导航结构或权限矩阵；
- 运行态若出现异常，可临时退回原有固定裁剪逻辑，同时保留已写入的数据对象以便后续再次启用。

## Open Questions

- 普通分析会话失活后，Web 结果页追问是直接按“新问题”处理，还是优先返回“上下文已失效，请重新描述”的提示；实现阶段需结合现有交互选一种明确口径。
- 历史摘要保留上限是按字符裁剪还是按句子 + 字符双重裁剪；当前提案建议先按字符生效，后续再按中文句边界优化。
- 是否需要在 AI 配置中心摘要区展示“当前上下文策略版本 / 最近更新时间”；如治理用户经常排查追问问题，建议一并展示。
