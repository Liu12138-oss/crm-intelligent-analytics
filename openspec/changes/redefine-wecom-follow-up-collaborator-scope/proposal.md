## Why

当前企业微信 `跟进客户`、`跟进商机`、`今日跟进` 链路已经具备受控写回能力，但对象级权限仍沿用“负责人本人、负责人所在直属小组成员、负责人递归上级领导、管理员”的旧口径。该规则会把并非负责人或协作人的同组成员错误放行，与当前业务要求“只能由负责人、协作人、管理员及其递归上级链主管写跟进”不一致，因此需要以新的授权主体集合口径重新收口。

同时，现有实现只会在最终确认前重读最新负责人，不会重读最新协作人集合；如果协作关系在草稿确认期间变化，系统仍可能按旧快照误放行。该缺口已经触及正式写回边界，必须通过新的 OpenSpec 变更统一修正规格、设计、测试和审计语义。

## What Changes

- **BREAKING** 废除企业微信跟进写回中“负责人所在直属小组成员可跟进”的旧规则，不再允许仅因同直属团队而写入客户或商机跟进。
- 新增基于“授权主体集合”的对象级权限模型：允许写跟进的主体仅包括负责人本人、协作人本人、负责人递归上级链中的任一主管、协作人递归上级链中的任一主管，以及管理员。
- 明确 `Customer` 与 `Opportunity` 的对象级授权都必须同时读取“当前负责人 + 当前协作人集合”，`今日跟进` 在对象识别完成后按最终对象类型套用同一规则。
- 要求在草稿创建前与最终写回前都按最新对象事实重算权限；最终确认前不仅要重读负责人，还要重读协作人集合，若关系失效必须阻断写回并保留草稿。
- 明确协作人数据源遵循 CRM API-first 约束：优先复用 CRM 官方协作人接口，只有官方 API 无法满足字段或权限语义时，才允许进入受控数据库兜底。
- 更新权限拒绝文案、审计快照和验收场景，确保能够区分“动作权限不足”“对象负责人关系不满足”“对象协作人关系不满足”“组织事实缺失”“协作人事实缺失”等阻断原因。

## Capabilities

### New Capabilities
- `wecom-follow-up-collaborator-scope`: 规范企业微信跟进写回如何基于负责人、协作人及其递归上级链执行对象级授权，并定义草稿创建、最终写回、权限重算与审计边界。

### Modified Capabilities
- `wecom-follow-up-guided-template`: 跟进模板、自由文本整理、对象识别和最终写回确认链路需要接入新的协作人范围对象级授权，并在最终确认前重读最新协作人集合。
- `organization-scope-governance`: 当前企业微信组织事实除用于默认数据范围推导外，还需要支持“负责人 / 协作人递归上级链”关系解析，且事实缺失时不得扩大跟进权限。
- `feature-permission-enforcement`: `wecom.followup.writeback` 需要继续作为全局动作开关，但必须与新的对象级授权门闩叠加执行，并在审计中区分对象关系型拒绝原因。
- `crm-api-first-integration`: 企业微信跟进协作人读取需要纳入 CRM 官方 API 优先策略，明确优先复用官方协作人接口，再考虑受控数据库兜底。

## Impact

- 后端受影响模块：
  - `backend/src/modules/wecom`
  - `backend/src/modules/opportunities`
  - `backend/src/modules/governance`
  - `backend/src/database/crm-readonly`
- 受影响的权限与审计语义：
  - `wecom.followup.writeback` 动作门闩后的对象级授权判定
  - 企业微信跟进写回拒绝原因、关系命中结果与审计快照字段
- 受影响的对象事实来源：
  - CRM 客户 / 商机详情接口
  - CRM 官方协作人接口
  - 企业微信组织同步事实与 `directLeaderUserids`
  - CRM 用户与企业微信映射关系
- 受影响的测试与文档：
  - 企业微信跟进写回单元测试与集成测试
  - 协作人变更后的最终确认阻断回归
  - `spec.md`、`data-model.md`、`quickstart.md` 与相关 OpenSpec delta specs
