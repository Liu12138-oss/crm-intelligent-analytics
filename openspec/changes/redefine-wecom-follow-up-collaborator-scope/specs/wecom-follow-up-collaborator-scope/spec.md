## ADDED Requirements

### Requirement: 企业微信跟进写回必须按负责人、协作人及其递归上级链执行对象级授权
系统 MUST 在 `wecom.followup.writeback` 动作权限通过后，再按目标对象的授权主体集合执行对象级授权。授权主体集合 MUST 仅包括目标对象当前负责人、目标对象当前协作人，以及这些主体各自的递归上级链；系统管理员 MUST 始终可以继续写回。系统 MUST 不得再因为当前用户与负责人处于同一直属小组、同一部门、同一销售体系或其它粗粒度组织关系而直接放行。

#### Scenario: 负责人本人可以继续跟进写回
- **WHEN** 当前用户就是目标客户或目标商机的当前负责人，且已具备 `wecom.followup.writeback`
- **THEN** 系统必须允许该用户进入跟进草稿确认或最终写回流程
- **THEN** 不得再要求其额外命中协作人或主管关系

#### Scenario: 协作人本人可以继续跟进写回
- **WHEN** 当前用户被标记为目标客户或目标商机的当前协作人之一，且已具备 `wecom.followup.writeback`
- **THEN** 系统必须允许该用户进入跟进草稿确认或最终写回流程
- **THEN** 不得因为其不是负责人本人而拒绝写回

#### Scenario: 负责人或协作人的递归上级可以继续跟进写回
- **WHEN** 当前用户在负责人或任一协作人的企业微信直属上级链中属于任一层上级，且已具备 `wecom.followup.writeback`
- **THEN** 系统必须允许该用户继续跟进写回
- **THEN** 审计快照中必须能够说明其命中的是哪一类递归上级链关系

#### Scenario: 同直属小组但不是负责人或协作人的成员不得继续跟进写回
- **WHEN** 当前用户与目标对象负责人共享同一直属上级，但既不是负责人本人，也不是协作人本人，且也不在负责人或协作人的递归上级链中
- **THEN** 系统必须拒绝该次跟进草稿创建或最终写回
- **THEN** 不得仅因同直属团队关系而自动放行

### Requirement: 系统必须按最终对象类型确定授权主体集合
系统 MUST 按最终确认的对象类型决定授权主体集合来源。`Customer` 跟进 MUST 使用客户当前负责人和客户当前协作人；`Opportunity` 跟进 MUST 使用商机当前负责人和商机当前协作人。`今日跟进` 与其它自由文本跟进在对象识别完成前 MUST 不得提前固定授权主体集合，识别完成后 MUST 按最终对象类型套用对应规则。

#### Scenario: 跟进客户按客户负责人和客户协作人判权
- **WHEN** 企业微信主题入口或自由文本最终确认的对象类型为 `Customer`
- **THEN** 系统必须以该客户当前负责人和当前协作人组装授权主体集合
- **THEN** 不得额外依赖该客户关联商机的负责人或协作人覆盖判权结果

#### Scenario: 跟进商机按商机负责人和商机协作人判权
- **WHEN** 企业微信主题入口或自由文本最终确认的对象类型为 `Opportunity`
- **THEN** 系统必须以该商机当前负责人和当前协作人组装授权主体集合
- **THEN** 不得因为关联客户的负责人或协作人不同而替换授权主体集合

#### Scenario: 今日跟进按最终对象类型继承授权主体集合
- **WHEN** `今日跟进` 或自由文本跟进在对象识别后最终选中了一个客户或商机对象
- **THEN** 系统必须按最终对象的类型继承对应的负责人和协作人集合
- **THEN** 在对象未最终确定前，系统不得提前给出“有权写回”结论

### Requirement: 系统必须在草稿创建前与最终写回前按最新负责人和协作人集合重校验
系统 MUST 在两个阶段执行对象级授权：一是在待写回草稿创建前，二是在最终确认写回前。最终确认阶段 MUST 同时重新读取目标对象的最新负责人和最新协作人集合，并基于最新企业微信组织事实重算递归上级链。若负责人、协作人或组织关系在确认期间发生变化，系统 MUST 阻断最终写回并保留草稿与失败原因。

#### Scenario: 草稿创建后协作人被移除导致最终写回被阻断
- **WHEN** 某用户在创建待写回草稿时原本属于目标对象协作人，但在最终确认前该用户已被从协作人集合中移除，且其也不再命中负责人或任一协作人的递归上级链
- **THEN** 系统必须在最终写回前重新计算对象级授权
- **THEN** 系统必须阻断本次写回并保留当前草稿

#### Scenario: 草稿创建后负责人变更导致最终写回被阻断
- **WHEN** 某用户创建待写回草稿时原本具备对象级授权，但在最终确认前该客户或商机负责人已经变化，且当前用户不再命中新授权主体集合
- **THEN** 系统必须在最终写回前重新计算对象级授权
- **THEN** 若当前用户已不再命中允许关系，系统必须阻断本次写回并返回明确中文提示

#### Scenario: 草稿创建前即不满足对象级授权时不得生成草稿
- **WHEN** 目标对象已经唯一命中，但当前用户既不是负责人、也不是协作人、也不在任何授权主体的递归上级链中
- **THEN** 系统必须在草稿创建前直接拒绝
- **THEN** 不得生成待写回草稿或进入内容确认阶段

### Requirement: 授权主体事实缺失时对象级授权必须默认阻断
对象级授权 MUST 建立在稳定的负责人事实、协作人事实、CRM 用户映射与企业微信直属上级链之上。若负责人、协作人、操作者或其上级链所需的关键事实缺失，系统 MUST 默认阻断企业微信跟进写回，不得因为字段缺失而扩大为“可见即允许”“同部门即允许”或“同小组即允许”的宽松授权。

#### Scenario: 协作人事实读取失败时阻断写回
- **WHEN** 系统能够识别出目标客户或商机，但当前无法可靠读取该对象的协作人集合
- **THEN** 系统必须阻断该次跟进写回
- **THEN** 返回文案必须明确说明当前无法确认协作人权限关系

#### Scenario: 操作者缺少企业微信映射时阻断写回
- **WHEN** 当前用户具备 `wecom.followup.writeback`，但系统无法确认其 CRM 用户与企业微信成员映射
- **THEN** 系统必须阻断该次跟进写回
- **THEN** 不得仅凭登录态或角色名继续放权

#### Scenario: 上级链事实缺失时不得放宽到部门级
- **WHEN** 某个负责人或协作人的 `directLeaderUserids` 缺失或相关上级链事实不完整，且当前用户只有“主管关系”这一路径可以命中授权
- **THEN** 系统必须按无法确认对象关系处理并阻断写回
- **THEN** 不得回退为同部门或同直属小组默认放行
