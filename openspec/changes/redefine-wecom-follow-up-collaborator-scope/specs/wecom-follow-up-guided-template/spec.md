## MODIFIED Requirements

### Requirement: 企业微信跟进写回链路必须校验 wecom.followup.writeback
系统 MUST 在企业微信跟进草稿创建、意图确认、内容补充、最终写回执行、失败后重试和重复确认拦截阶段校验 `wecom.followup.writeback`。当目标客户或商机已经被唯一识别后，系统 MUST 在该动作权限之外继续执行对象级授权，确认当前用户是否命中“负责人本人、协作人本人、负责人递归上级链、协作人递归上级链、管理员”任一允许关系。企业微信用户即使具备问数能力，也不得因为已进入同一会话而自动获得跟进写回资格；具备 `wecom.followup.writeback` 但不具备目标对象关系权限的用户，同样不得进入对象确认、内容确认或最终写回阶段。最终确认前，系统 MUST 重读目标对象的最新负责人和最新协作人集合，不得只信任草稿创建时的旧快照。

#### Scenario: 缺少 wecom.followup.writeback 的用户不能创建跟进写回草稿
- **WHEN** 某用户已通过企业微信通道准入，但未被授予 `wecom.followup.writeback`，并尝试发起跟进写回
- **THEN** 系统必须拒绝创建跟进写回草稿
- **THEN** 不得把该用户推进到对象确认、内容确认或最终写回阶段

#### Scenario: 已有动作权限但不满足负责人或协作人关系时不得创建草稿
- **WHEN** 某用户已具备 `wecom.followup.writeback`，但在目标对象唯一命中后既不是负责人、也不是协作人、也不在负责人或协作人的递归上级链中
- **THEN** 系统必须拒绝创建该对象的跟进写回草稿
- **THEN** 不得因为对象已唯一命中或与负责人同组而继续进入跟进确认阶段

#### Scenario: 已有草稿但负责人或协作人关系失效时最终写回被阻断
- **WHEN** 某用户已经拥有待确认的跟进写回草稿，但在最终确认前被撤销 `wecom.followup.writeback`，或目标对象的负责人 / 协作人集合已经变化导致其不再命中对象级授权
- **THEN** 系统必须拒绝本次最终写回
- **THEN** 系统必须保留草稿与失败原因，不得误记为写回成功

### Requirement: 跟进写回拒绝与重试必须写入统一权限拒绝审计
系统 MUST 为企业微信跟进写回中的权限拒绝与重试失败写入统一权限拒绝审计。该审计不仅要覆盖缺少 `wecom.followup.writeback` 的动作权限拒绝，还必须覆盖目标对象负责人关系不满足、目标对象协作人关系不满足、负责人映射缺失、协作人事实缺失、直属上级链缺失或最终确认前关系失效导致的对象级授权拒绝。审计 MUST 记录目标对象、草稿 ID、当前会话、权限点、授权主体集合快照、对象关系结果和拒绝原因。

#### Scenario: 写回确认阶段因动作权限不足失败时写入审计
- **WHEN** 某用户在确认跟进写回时因缺少 `wecom.followup.writeback` 被拒绝
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 该审计事件必须能够关联到当前草稿和目标对象

#### Scenario: 对象关系不满足导致草稿创建失败时写入审计
- **WHEN** 某用户具备 `wecom.followup.writeback`，但在目标对象唯一命中后因不满足负责人或协作人关系被阻断
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 该审计事件必须记录是对象级授权拒绝，而不是普通动作权限缺失

#### Scenario: 最终确认前协作人或负责人关系失效导致重试失败时写入审计
- **WHEN** 某用户已有待确认草稿，但在最终写回或失败后重试时因负责人、协作人或主管链关系变化被阻断
- **THEN** 系统必须写入一条权限拒绝审计事件
- **THEN** 审计中必须能关联到当前草稿、目标对象和最新拒绝原因
