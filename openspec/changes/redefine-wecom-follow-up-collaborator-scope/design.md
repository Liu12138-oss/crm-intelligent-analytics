## Context

当前企业微信跟进写回链路已经具备 `wecom.followup.writeback` 动作权限门闩、对象识别、草稿确认和 CRM 官方 `POST /api/v2/revisit_logs` 写回能力，但对象级授权仍沿用“负责人本人 / 同直属小组成员 / 负责人递归上级 / 管理员”的旧模型。该模型把“负责人团队”当成授权主体，无法表达“只有负责人、协作人及其递归上级链可以写跟进”的新业务口径。

现有实现还有两个结构性缺口：

- `FollowUpAuthorizationService` 只接受 `ownerId/ownerName`，没有协作人输入，也不会区分“负责人主管”与“协作人主管”。
- 最终确认前只会重读最新负责人，不会重读最新协作人集合；协作关系在草稿确认期间变化时，系统仍可能依据旧快照放行。

本次设计需要在不调整登录、统一鉴权、会话装载和企业微信验签链路的前提下，完成权限模型替换，并继续遵守 CRM API-first、受控写回、审计留痕和中文用户提示要求。

## Goals / Non-Goals

**Goals:**

- 将企业微信跟进写回的对象级授权模型替换为“授权主体集合制”。
- 让 `Customer` 和 `Opportunity` 都能基于“负责人 + 协作人”集合完成对象级判权。
- 明确主管口径为“递归上级链”，并同时覆盖负责人和协作人两个维度。
- 在草稿创建前与最终确认前都重读最新对象事实，避免旧草稿基于过期协作关系误放行。
- 为拒绝审计补齐可排障上下文，能明确看出命中或缺失的是哪类关系。

**Non-Goals:**

- 不新增 Web 跟进写回入口。
- 不改动 `wecom.followup.writeback` 在权限中心中的键名、发放方式和基础治理模型。
- 不把企业微信部门路径、CRM 页面部门名或“同销售体系”作为新的放权兜底。
- 不调整跟进写回之外的问数、日报预览、新增客户、新增商机执行边界。

## Decisions

### 决策一：对象级授权改为“授权主体集合 + 递归上级链”模型

最终允许写跟进的主体集合定义为：

1. 管理员；
2. 负责人本人；
3. 协作人本人；
4. 负责人递归上级链中的任一主管；
5. 任一协作人递归上级链中的任一主管。

实现上不再以“负责人直属团队”作为授权主体，也不再使用同直属上级成员集合作为放行条件。

推荐原因：

- 直接贴合业务口径，避免“同组但无业务关系”的成员误放行。
- 授权主体集合天然支持一个对象存在多个协作人，不需要把所有关系压平为一个团队概念。

备选方案：

- 保留“负责人同组成员”作为兼容条件，再叠加协作人关系。
  不采纳，原因是会继续保留当前已被业务确认错误的放权路径。

### 决策二：新增“授权主体解析器”，把对象事实解析与关系判定拆开

新增一个专门的目标解析层，用于返回对象级授权所需的最小事实：

- `objectType/objectId/objectTitle`
- `ownerId/ownerName`
- `assistUserIds/assistUserNames`
- `factSource`
- `missingFacts`

`FollowUpAuthorizationService` 只负责消费这组事实并输出判定结果，不再自己拼接协作人来源或直接耦合具体 API 细节。

推荐原因：

- 把“读取对象授权主体集合”和“基于组织事实判权”拆开后，服务职责更清晰。
- 方便在草稿创建前与最终确认前重复调用同一解析器，避免两处各写一套负责人 / 协作人读取逻辑。

备选方案：

- 继续扩展 `FollowUpAuthorizationService`，让它直接负责调对象详情和协作人接口。
  不采纳，原因是会把授权服务变成“读取 + 计算 + 文案 + 审计”混合体，后续测试和维护成本更高。

### 决策三：协作人数据优先复用 CRM 官方 API，再受控兜底数据库

对象级授权所需的协作人集合遵循 CRM API-first：

- `Customer`：优先调用客户协作人官方接口；
- `Opportunity`：优先调用商机协作人官方接口；
- 仅在官方 API 无法满足字段、权限语义、稳定性或返回结构要求时，才允许进入受控数据库兜底，例如 `entities_assist_users`。

兜底时必须同时记录：

- API 缺口原因；
- 涉及表与字段白名单；
- 权限继承方式；
- 失败降级与回退策略。

推荐原因：

- 协作人关系本身就是 CRM 业务事实，优先复用官方 API 才符合当前项目的接入原则。
- 先走官方接口可以避免在权限模型调整时顺手新增未经治理的查库路径。

备选方案：

- 直接使用 `entities_assist_users` 作为默认数据源。
  不采纳，原因是违背当前 API-first 约束，也不利于后续在环境差异下保持一致行为。

### 决策四：草稿创建前与最终确认前都重算“最新负责人 + 最新协作人集合”

执行门闩分为两次：

- 草稿创建前：对象唯一命中后，立即解析最新负责人和协作人集合并判权；不满足则禁止创建草稿。
- 最终确认前：重新读取当前对象详情与协作人集合，再次重算授权主体集合和递归上级链；只要负责人、协作人或组织关系任一变化导致当前用户不再命中，必须阻断写回并保留草稿。

推荐原因：

- 只重读负责人不够，协作人变化同样会影响放行边界。
- 两次重算能分别保护“草稿可见边界”和“正式执行边界”。

备选方案：

- 草稿阶段读取协作人，确认阶段只校验负责人。
  不采纳，原因是无法覆盖“协作人被移除 / 新增”这类真实业务变更。

### 决策五：递归上级链继续复用企业微信组织事实，不新增部门级兜底

负责人和协作人的主管链都继续基于企业微信 `directLeaderUserids` 递归向上解析；若其中任一授权主体缺少稳定映射或上级链事实缺失，则该主体不产生可放行的主管集合，并把缺失原因纳入整体判定结果。只有管理员与直接命中授权主体本人的情况可以绕过这类上级链缺失。

推荐原因：

- 与现有团队范围、日报汇总、组织排障的事实来源保持一致。
- 防止因为个别字段缺失把规则偷偷回退成“同部门都可以”。

备选方案：

- 上级链缺失时回退到部门负责人、部门路径或角色名。
  不采纳，原因是这些字段无法稳定表达“负责人与协作人的主管链”，会产生新的过度放权。

### 决策六：审计记录区分“动作权限拒绝”和“对象关系拒绝”，并暴露授权主体快照

对象级授权结果需要输出独立的审计上下文，至少包含：

- `objectType/objectId/objectTitle`
- `ownerId`
- `assistUserIds`
- `matchedRelation`
- `matchedSubjectIds`
- `matchedLeaderSubjectIds`
- `missingFacts`
- `denialReason`

动作权限拒绝仍沿用现有 `permissionKey` 语义；对象关系拒绝则在同一 `wecom.followup.writeback` 下补充更细的 `resourceType` 和会话快照。

推荐原因：

- 仅记录“无权跟进”不足以排障，必须能看出到底是负责人、协作人还是组织事实哪一层不满足。
- 后续验收和回归会直接依赖这些上下文字段做断言。

备选方案：

- 仅保留一条中文拒绝文案，不扩展审计快照结构。
  不采纳，原因是无法支撑线上权限争议排查。

## Risks / Trade-offs

- [风险] 真实环境中个别对象可能存在协作人接口为空、对象详情有负责人但协作人同步滞后的情况。  
  → 缓解：把“协作人为空”和“协作人事实读取失败”区分开；前者视为无协作人，后者按事实缺失阻断并记录审计。

- [风险] 新增协作人接口调用会让最终确认阶段比现在多一次或多组外部读取。  
  → 缓解：只在对象唯一命中和最终确认前触发，不影响普通分析链路；读取失败时直接快速失败，不做无边界重试。

- [风险] 旧测试和历史变更都把“同直属小组成员允许”当成正确行为，切换后会有一批断言需要重写。  
  → 缓解：在 OpenSpec 明确标记为 breaking rule replacement，并把测试改造列成单独任务。

- [风险] 若授权主体集合很大，逐个解析递归上级链可能放大组织关系计算成本。  
  → 缓解：先对负责人和协作人 CRM 用户 ID 去重，再按主体去重收集上级链，复用内存态映射，不重复外部请求。

## Migration Plan

1. 先创建新的 OpenSpec 变更，明确该变更替代旧的“同直属小组成员可跟进”规则。
2. 在对象解析层增加“负责人 + 协作人集合”读取能力，并先接入草稿创建阶段。
3. 调整授权服务，只接受授权主体集合并移除同组成员放行逻辑。
4. 在最终确认前接入“对象详情 + 协作人集合”重读与重新判权。
5. 更新权限拒绝审计、用户提示、验收文档和自动化测试。

回滚策略：

- 若上线后协作人事实源存在大面积稳定性问题，可先回滚对象级协作人判权接线，恢复到旧实现，但该回滚仅作为紧急止血，不作为长期方案。
- 新增的授权主体解析器和审计字段可以保留；回滚时只需停止在写回链路消费新判权结果，不需要回退历史草稿数据结构。

## Open Questions

- 当前无阻塞性业务口径问题，递归上级链已经由用户明确拍板。
- 实施阶段仍需在真实环境确认客户协作人与商机协作人的官方 API 返回结构，特别是用户字段是否能稳定映射到当前 CRM 用户 ID；若不能，需要在实现前补一段受控字段映射设计。
