# 预研结论：CRM智能分析系统（一期：企业微信 AI 问数与智能分析工作台）

## 决策 1：采用“企业微信机器人 + Web 智能分析工作台 + Web 治理审计”的统一双入口结构

- **Decision**：使用 `frontend/` + `backend/` 的单仓前后端分离结构交付一期能力，其中企业微信机器人覆盖移动问数场景，Web 前端同时承担 AI 分析工作台、查询复用、治理配置和审计检索。
- **Rationale**：当前需求已经不再只是“企业微信问数”，而是要求用户能在 Web 内获得更完整、更美观的结果展示与复用能力。企业微信适合即时问答，Web 适合深度阅读、图表查看和查询资产管理，两者共享同一后端分析核心可以保证结果一致。
- **Alternatives considered**：
  - 仅保留企业微信入口：无法满足漂亮展示、常用查询和最近查询的主体验。
  - 仅保留 Web 单入口：无法覆盖销售移动办公主场景。
  - 拆成两套独立系统：会造成结果口径、权限边界和审计轨迹分叉。

## 决策 2：AI 接入统一使用 OpenAI 兼容 HTTP Profile，通过内部网关或兼容平台调用

- **Decision**：AI 调用层优先读取后台激活的 OpenAI 兼容 HTTP Profile；未激活 Profile 时回退 `ANALYSIS_AI_*` 与 `OPENAI_API_KEY` 环境默认配置，并通过 `responses` 或 `chat_completions` 协议访问内部网关、Qwen、DeepSeek、GLM 等兼容平台。
- **Rationale**：当前业务只需要文本与结构化输出，不需要本地 Agent CLI、MCP、工具执行或 provider 侧持久线程。统一 HTTP Profile 可以降低部署和排障成本，同时保留密钥治理、健康检查、审计和本地 schema 校验边界。
- **Alternatives considered**：
  - 继续依赖本地 Agent SDK：会保留 CLI、线程、MCP 和工具执行复杂度，不符合当前服务端部署目标。
  - 重新定义另一套模型网关：会增加接入和维护成本。
  - 前端直接调用模型：无法满足权限边界、审计留痕和受控执行控制要求。

## 决策 3：自然语言问数采用“结构化意图 + 模板约束 + 必要补问”而非直接自由生成 SQL

- **Decision**：自然语言请求先转换为结构化分析意图；当缺少关键限定条件时先生成补问提示；对于高频问题优先匹配常用查询模板；最终由后端基于白名单指标、维度和过滤条件编译受控执行计划，并经过 AST 安全校验后执行。
- **Rationale**：直接执行模型生成的自由 SQL 风险过高，也难以保证数据准确性和结果一致性。结构化意图、模板约束和补问机制可以降低越权访问、口径漂移、误解用户意图和编造结论的风险。
- **Alternatives considered**：
  - 直接让模型输出 SQL：安全边界过弱，无法满足一期治理要求。
  - 完全依赖固定模板问答：难以覆盖自然语言表达差异和追问场景。

## 决策 4：结果展示采用“服务端归一化 + 前端多视图渲染”的统一结果模型

- **Decision**：后端返回统一的结果结构，包含摘要、指标卡、主图表、辅助图表、表格、口径说明、筛选条件和数据更新时间；前端根据统一结构渲染 Web 结果页，企业微信则只消费适合会话展示的摘要与流式块。
- **Rationale**：漂亮展示不能依赖模型直接拼 UI 文本，否则会造成不同入口口径不一致。服务端先归一化结果，再由各入口按自身表现形式渲染，才能保证摘要、图表、表格和导出结果一致。
- **Alternatives considered**：
  - 让模型直接输出最终页面片段：可控性差，难以审计和复核。
  - 只返回纯文本：无法满足结构化报表展示要求。

## 决策 5：常用查询与最近查询作为独立查询资产管理

- **Decision**：将常用查询模板和最近查询历史作为独立业务对象管理，并纳入角色可见性、排序、复用、重跑和审计规则。
- **Rationale**：常用查询和最近查询是本期的重要体验能力，不能仅作为前端缓存实现。独立建模后，才能支持按角色下发、配置口径说明、审计模板点击和按当前权限重跑历史问题。
- **Alternatives considered**：
  - 只在前端本地缓存最近查询：无法跨端复用，也无法审计。
  - 把常用查询硬编码在页面中：管理员无法动态维护。

## 决策 6：权限复用 CRM 现有会话与数据范围，模板与历史重跑也必须继承当前最新授权

- **Decision**：复用 CRM 现有登录态、角色、组织、部门、负责人数据范围；分析系统不维护独立用户体系；常用查询执行和最近查询重跑时，均按当前最新权限重新计算。
- **Rationale**：规格已明确不能另起一套权限模型。若模板或历史重跑沿用旧范围，将形成新的越权路径，因此必须在每次执行时重新注入实时授权范围。
- **Alternatives considered**：
  - 新建独立 RBAC：会造成权限口径分叉。
  - 将历史结果直接复用给用户：无法保证当前授权仍然有效。

## 决策 7：数据准确性通过“结果一致性校验 + 数据新鲜度标识 + 拒答策略”三重保护

- **Decision**：在结果返回前执行一致性校验，确保摘要、图表、表格与导出使用相同条件；同时返回数据更新时间；当查询超范围、无数据、数据依赖不足或关联关系不满足时，优先补问、说明或拒答，而不是猜测。
- **Rationale**：用户要求“数据要准确”，这不仅是查询正确，还包括结果解释一致、时间口径明确和在无法可靠回答时拒绝生成误导性答案。
- **Alternatives considered**：
  - 只校验 SQL 可执行：无法发现多视图结果不一致的问题。
  - 无数据新鲜度标识：用户无法判断结果是否可直接用于决策。

## 决策 8：导出策略继续采用“默认展示、不自动导出、超限即阻止”

- **Decision**：查询结果默认在页面或会话内展示；导出必须显式触发；一期对单次导出行数和每日导出次数均做硬限制，超限直接阻止。
- **Rationale**：本期重点是“查得准、看得懂、复用快”，不是开放大规模数据搬运。硬限制导出可以有效降低数据泄露风险，也不需要在一期引入审批流。
- **Alternatives considered**：
  - 超限后进入审批流：可扩展，但会增加流程编排复杂度。
  - 默认允许导出再审计：无法满足“先控制后审计”的安全要求。

## 决策 9：测试策略增加结果展示一致性和查询复用验证

- **Decision**：后端继续使用 Jest + Supertest 覆盖权限、受控查询、导出限制和审计留痕；前端使用 Vitest 覆盖分析工作台、常用查询和最近查询组件；Playwright 覆盖 Web 工作台关键路径；并增加结果一致性与复用验证。
- **Rationale**：当前风险不只在查询安全，还在“结果是否一致”“模板和历史重跑是否正确套用当前权限”“Web 展示是否覆盖关键口径”。
- **Alternatives considered**：
  - 仅做手工测试：难以稳定验证结果一致性和复用逻辑。
  - 只做后端接口测试：无法覆盖工作台展示层与用户交互。

## 决策 10：企业微信会话继续采用受控长连接治理，强调心跳、并发隔离与有序流式返回

- **Decision**：机器人问数链路继续采用受控长连接会话治理，要求会话心跳、超时回收、并发隔离、顺序控制和高峰期排队/降级提示。
- **Rationale**：企业微信机器人是多人共享入口，若缺少连接治理，将直接导致会话失活、消息乱序、结果串发和资源泄漏。即使 Web 工作台补齐后，这条链路仍是移动场景的核心入口。
- **Alternatives considered**：
  - 每次请求独立短连接返回：难以承载补问上下文与流式结果。
  - 仅依赖底层网关默认连接策略：无法确保业务层面的会话隔离和异常可观测性。

## 结论汇总

- 一期统一采用“企业微信机器人 + Web 智能分析工作台 + Web 治理审计”的双入口结构。
- AI 接入统一使用 OpenAI 兼容 HTTP Profile，所有模型调用都由后端集中编排；历史 Agent SDK 配置仅作为迁移参考。
- 受控执行核心原则升级为“结构化意图、模板约束、必要条件补问、白名单编译、AST 校验、结果归一化、一致性校验、全链路审计”。
- 常用查询与最近查询被视为一期核心体验能力，不再是附属功能。
