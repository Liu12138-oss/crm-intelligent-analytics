问题根因定位

用户提问 "渠道商经营分析看板" 后收到帮助文案而非看板数据。追踪完整消息处理后发现：

1
hasExplicitWecomAnalysisOrBlockSignal() 返回 false — 文本不含 "商机|客户|金额" 等 metric 关键词，也不含时间词
2
shouldReturnIdleUnrecognizedHelp() 返回 true — 因上述返回 false
3
进入 unrecognizedHelpResult 帮助分支 → 输出 "你好呀，我是 CRM 智能小助手..."
4
DashboardReportComposer 桥接从未被触发 — 代码死在 ANALYZE 分支之前
修复措施

修复 1：信号层补强看板类关键词（wecom-bot.service.ts）

  • 在 hasExplicitWecomAnalysisOrBlockSignal() 中新增 30+ 看板/运营/分析关键词匹配
  • 覆盖："数据看板|看板分析|运营看板|经营看板|发展运营|运营数据|运营分析|经营分析|数据运营"
  • 覆盖组合模式："渠道商.*看板|代理商.*分析|区域.*看板|订单.*分析|订单.*看板|下单.*汇总|签单.*汇总" 等

修复 2：兜底聚合逻辑（dashboard-report-composer.service.ts）

  • buildCooperationLevelDistributionFromContributions() — 从 partnerContributions 按 cooperationLevel 聚合
  • buildTechServiceTypeDistributionFromContributions() — 从 partnerContributions 按 techServiceType 聚合
  • buildFunnelBlockFromSummariesOrContributions() — 从 summary/contributions 汇总漏斗 + 计算转化率
  • KPI 和执行摘要中的 LEP/金牌/签约/提名/报备/商机均添加 partnerContributions 兜底路径

修复 3：dispatch block 拆分（wecom-bot.service.ts）

  • 从单一 dispatch block 改为按逻辑分组输出 4 个 StreamBlock（标题/KPI / 图表类 / 表格类 / 清单）
  • 末尾追加「看板区块清单」供用户确认哪些 block 生成了

修复 4：诊断日志贯穿全链路

  • compose() 入口/出口记录 bundle 各数据源 OK/NULL 状态和字段数
  • 桥接入口/出口记录问题文本、blockCount、errors、字节数

DashboardReportComposer Block 组装验证

场景 A：所有 OpenAPI 端点正常（完整模式）

生成 13 个 block：

kpi-matrix(核心指标) pie-distribution(合作级别分布) pie-distribution(技术服务商分布) grouped-bar(区域对比) geo-map(省份覆盖) pie-distribution(商机阶段分布) pie-distribution(报价状态分布) pie-distribution(订单状态分布) composite-trend(渠道商新增趋势) funnel(业务转化漏斗) sortable-table(区域排名明细) sortable-table(渠道排名明细) sortable-table(各团队签约明细)

场景 B：funnel 端点失败（兜底聚合）

funnel=NULL 时仍生成 13 个 block（漏斗从 contributions 汇总）

诊断日志：业务转化漏斗：funnel 端点无数据，已从 summary/contributions 兜底聚合。

场景 C：partnerSummary 字段为空（兜底聚合）

byCooperationLevel/byTechServiceType 为空时仍生成 12 个 block（饼图从 contributions 聚合）

诊断日志：合作级别分布：summary 端点无数据，已从 partnerContributions 兜底聚合。

场景 D：channel-order-summary 订单分析模板

生成 5 个 block（KPI + 集中度 + 饼图 + 表格）— 对标参考 HTML 的广州办渠道下单汇总分析模板结构
