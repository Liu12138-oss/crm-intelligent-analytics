## 各阶段详细说明

### 阶段1：加载审核依据

- 使用 `requirements.md` 作为业务审查依据。
- 使用 `checks.json` 中的结构化检查项作为运行时标准快照来源。
- 使用 `prompts/` 中的提示词模板约束 AI 规划、分组审查与汇总。
- 如合同属于公司标准模板衍生合同，应同时按 `requirements.md` 中的模板基线要求理解合同。

### 阶段2：提取合同文本

- 从 `.docx` 中提取段落、标题、条款和摘要。
- 保留定位信息，供问题落点、批注和报告引用。

### 阶段3：识别模板类型

- 判断合同是否明显基于公司标准模板生成。
- 优先识别合同标题、乙方固定信息、章节结构、附件名称、付款分支等模板特征。

### 阶段4：模板基线比对

- 如识别为标准模板合同，先检查固定结构是否完整。
- 检查乙方固定信息、争议解决、知识产权、违约责任、附件标题等基线内容是否被删改。
- 检查合同编号、最终用户、甲方信息、金额、报价清单、签署页等必填槽位是否仍为空白或占位。
- 检查模板中的“付款方式（二选一）”等分支是否已经收敛为实际有效版本。

### 阶段5：主题分组审查

- 识别付款、税率、交付、违约、知识产权、开票、维保等关键主题。
- 将合同片段按主题分组，为后续 AI 审查和确定性校验提供输入。
- 先依据 `checks.json` 生成本次审核的检查项集合。
- 再按主题组执行 AI 语义审查。
- 对付款账期、税率、折扣、违约上限、开票前缀等硬规则执行确定性校验。

### 阶段6：记录问题

- 统一输出结构化问题，至少包含检查项编码、风险等级、定位、原文引用、问题原因和修改建议。
- 模板基线比对发现的问题，可作为补充发现输出，即使当前 `checks.json` 尚未完全覆盖。
- 相同风险点应在 AI 结果和确定性校验结果之间去重。

### 阶段7：添加批注

- 高风险和否决项优先写入批注稿。
- 批注内容应聚焦风险说明、修改建议和审核依据，避免展示多余技术细节。

### 阶段8：生成报告

- 报告首页先给出综合结论和风险数量。
- 风险列表按“否决项 / 高风险 / 中风险 / 低风险”排序。
- 导出内容需要保留所使用的审核依据版本和执行模式。

## 质量检查清单

- [ ] 激活 pack 成功且校验摘要可追溯
- [ ] 所有检查项均有明确风险等级和修改建议
- [ ] 标准模板合同已执行模板基线比对
- [ ] 高风险问题优先展示
- [ ] 批注与结构化结果保持一致
- [ ] 报告能说明所依据的审核标准版本
