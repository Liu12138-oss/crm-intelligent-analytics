# 商务人员维护 `checks.json` 说明

## 1. 文档目的

本说明用于指导商务人员维护当前合同审核标准中的 `checks.json` 文件。

当前系统约定如下：

- `checks.json` 是合同审核项清单的主要维护入口。
- `requirements.md` 用于说明审核背景、业务规则和详细口径。
- `workflow.md` 用于说明审核流程和结果输出原则。
- `购销合同标准模板基线.md` 用于维护标准模板的固定结构、必填槽位和偏离风险。

如需新增、调整或下线某个审核点，优先维护 `checks.json`。

---

## 2. 相关文件位置

- 审核项清单：`checks.json`
- 审核标准说明：`requirements.md`
- 审核流程说明：`workflow.md`
- 标准模板基线：`购销合同标准模板基线.md`
- 版本配置：`profile.yaml`

建议商务人员主要维护前 4 个文件；如需调整版本号或运行方式，请通知研发处理。

---

## 3. 什么情况下需要修改 `checks.json`

出现以下情况时，应新增或调整 `checks.json`：

- 公司新增了明确的合同审核要求
- 原有审核项名称不清晰，商务人员难以理解
- 原有审核项遗漏了关键风险点
- 某个审核项的风险级别需要调整
- 某个审核项的修改建议不够明确，不能直接指导改合同

出现以下情况时，不建议直接拆很多新条目：

- 只是同一审核主题下补充了几个细节字段
- 只是举了更多例子，本质审核口径没有变化

此时优先修改 `description`、`keywords` 或 `requirements.md`。

出现以下情况时，应优先修改 `购销合同标准模板基线.md`：

- 公司更新了购销合同标准模板
- 模板中的固定条款、固定附件、乙方固定信息发生变化
- 模板新增、删除或调整了必填槽位
- 模板新增、删除或调整了付款分支等可选条款

简单判断方式如下：

- 新增一个“审核主题”时，改 `checks.json`
- 调整业务规则解释时，改 `requirements.md`
- 调整审核流程时，改 `workflow.md`
- 调整标准模板的固定结构、固定槽位、固定附件时，改 `购销合同标准模板基线.md`

---

## 4. 维护原则

### 4.1 一条 check 代表一个审核主题

建议一条 check 对应一个商务上可独立理解的审核点，例如：

- 甲方信息完整性
- 付款账期和分期比例合规
- 开票名称符合软件前缀要求
- 知识产权不得归客户独有

不建议拆得过细，例如不要把“甲方信息完整性”拆成下面很多条：

- 甲方单位全称完整
- 甲方电话完整
- 甲方税号完整
- 甲方开户行完整

这类细节应放在 `description` 中描述。
如这些细节本身来自标准模板中的固定槽位，也可以维护在 `购销合同标准模板基线.md` 中，而不是继续把 `checks.json` 拆细。

### 4.2 标题要短，描述要具体

- `title` 用于页面展示，尽量简洁
- `description` 用于解释具体审核内容，应写完整

### 4.3 建议必须可执行

`suggestion` 应直接告诉商务人员怎么改，不要只写“请关注风险”“请完善条款”这类空话。

### 4.4 关键词只保留高相关词

`keywords` 建议控制在 5 到 12 个之间，只保留最有代表性的词，不要整段复制要求原文。

### 4.5 硬规则才使用 `validatorBindings`

只有可稳定程序判断的规则，才配置 `validatorBindings`，例如：

- 税率
- 付款账期
- 折扣阈值
- 违约上限
- 开票前缀

纯语义审核项可保留空数组：

```json
"validatorBindings": []
```

## 5. `checks.json` 字段说明

每个审核项对象包含以下字段：

### `code`

审核项唯一编号，必须唯一，不能重复。

建议格式：

```text
CR-主题缩写-序号
```

示例：

- `CR-IP-001`
- `CR-PAY-001`
- `CR-TAX-001`

### `group`

审核分组，用于结果页分组展示。

当前建议分组示例：

- 主体资格
- 报价与折扣
- 付款与回款
- 票税与结算
- 交付与验收
- 违约与责任
- 知识产权与许可
- 维保与服务周期

### `category`

审核项分类，比 `group` 更细一层，用于帮助理解该审核点属于什么问题。

示例：

- 知识产权
- 付款条件
- 税率
- 开票
- 违约责任

### `title`

审核项名称，要求简洁、明确、面向商务人员。

推荐写法：

- 付款账期和分期比例需符合公司标准
- 软件类开票名称必须包含“*软件*”前缀

### `description`

审核项的完整说明，告诉系统和维护人员“具体要审什么”。

这里可以写得比 `title` 更完整。

示例：

```text
甲方单位全称、地址、联系人、电话、工商注册地、纳税人识别号、开户银行、银行账号、法定代表人应完整，不得为空或保留模板占位。
```

### `riskLevel`

风险级别，目前使用：

- `HIGH`
- `MEDIUM`
- `LOW`

建议口径：

- `HIGH`：必须修改，不改不建议签
- `MEDIUM`：建议修改，不改可能有风险
- `LOW`：提示优化，可按情况调整

### `isVeto`

是否一票否决。

- `true`：命中后原则上不得签署
- `false`：不是一票否决项

### `sourceClause`

审核依据来源，填写 `requirements.md` 中的章节位置，便于回溯。

示例：

- `一、1.3 付款条件`
- `二、2.11 开票；三、否决项 3`

### `keywords`

用于帮助系统识别合同中的相关内容。

示例：

```json
["付款", "首付款", "到货款", "验收款", "账期", "尾款"]
```

### `suggestion`

命中该审核项后，页面展示给商务人员的修改建议。

必须写成可直接操作的话。

不推荐：

```text
请关注风险。
```

推荐：

```text
按公司标准重写付款节点，控制账期和尾款比例；如有特殊商务审批，请补充审批依据。
```

### `applicableContractTypes`

适用合同类型。

当前建议值：

- `购销合同`
- `服务合同`
- `扩容合同`

如某审核项只适用于某一类合同，不要全部都填。

### `validatorBindings`

绑定程序化校验器。

如该审核项主要依赖 AI 语义理解，可写空数组：

```json
"validatorBindings": []
```

## 6. 新增审核项的推荐步骤

### 第一步：先判断是不是一个独立审核主题

如果只是补充同一主题下的细节，不要急着新增一条 check，先考虑修改已有 check 的：

- `description`
- `keywords`
- `suggestion`

如果变化来自标准模板本身，例如空白槽位、附件名称、固定乙方信息、模板分支条款，优先修改 `购销合同标准模板基线.md`。

### 第二步：补充 `requirements.md`

如果这是新的业务规则，请先在 `requirements.md` 里写清楚规则和口径。

### 第三步：新增 `checks.json` 条目

复制一个最接近的审核项，修改字段内容。

### 第四步：自检 JSON 格式

重点检查：

- 最后一项后面不能多逗号
- 所有双引号必须成对
- 中括号、花括号必须闭合
- `code` 不得重复

### 第五步：通知研发做一次验证

新增或修改审核项后，建议通知研发至少验证：

- JSON 能正常加载
- 审核项在页面能正常显示
- 命中后摘要和风险级别是否符合预期

## 7. 推荐模板

新增审核项时，可参考以下模板：

```json
{
  "code": "CR-XXX-001",
  "group": "分组名称",
  "category": "分类名称",
  "title": "审核项标题",
  "description": "详细说明具体要审什么，建议写清楚必须满足的条件、常见风险和不允许出现的内容。",
  "riskLevel": "HIGH",
  "isVeto": false,
  "sourceClause": "requirements.md 对应章节",
  "keywords": ["关键词1", "关键词2", "关键词3"],
  "suggestion": "写成商务人员看完就知道怎么改的建议。",
  "applicableContractTypes": ["购销合同"],
  "validatorBindings": []
}
```

---

## 8. 示例：主体信息完整性

如要新增“甲方信息完整性”审核项，推荐写法如下：

```json
{
  "code": "CR-PARTY-001",
  "group": "主体资格",
  "category": "主体信息",
  "title": "甲方信息完整性",
  "description": "甲方单位全称、地址、联系人、电话、工商注册地、纳税人识别号、开户银行、银行账号、法定代表人应完整，不得为空或保留模板占位。",
  "riskLevel": "HIGH",
  "isVeto": false,
  "sourceClause": "一、1.1 主体资格",
  "keywords": ["甲方", "单位全称", "联系人", "纳税人识别号", "开户银行", "银行账号", "法定代表人"],
  "suggestion": "补齐甲方主体信息后再提交审核；如仍使用模板占位文本，应替换为真实信息。",
  "applicableContractTypes": ["购销合同", "服务合同", "扩容合同"],
  "validatorBindings": []
}
```

## 9. 常见错误

- 同一个审核点拆成过多小条目，导致维护成本变高
- `title` 太长，页面展示不清晰
- `description` 太短，导致审核范围不清楚
- `keywords` 填得过多，反而增加误判
- `sourceClause` 不写，后续无法追溯依据
- `suggestion` 写得太空泛，商务人员无法直接修改
- 修改了业务规则，但没有同步更新 `requirements.md`

---

## 10. 当前维护边界

当前版本的系统已经支持通过 `checks.json` 调整大部分审核主题，但仍有以下边界：

- 某些表格空白字段问题，可能还需要研发补充提取逻辑
- 某些复杂语义判断，仍依赖 AI 审核效果
- 新增大量审核项后，建议研发复核分组、摘要和结果页展示是否仍清晰
- 标准模板相关问题不建议全部拆成大量 check，更适合通过模板基线文档 + AI 补充发现联合处理

因此，商务人员可以优先维护 `checks.json`，但如遇到以下情况，请同步联系研发：

- 需要识别表格空白或模板占位符
- 需要新增新的程序化硬规则
- 需要调整页面展示口径
- 需要修改版本号、执行模式或系统加载逻辑

---

## 11. 建议维护方式

推荐采用以下协作方式：

- 商务人员维护 `checks.json`、`requirements.md`、`购销合同标准模板基线.md`
- 如审核流程本身变化，再同步维护 `workflow.md`
- 研发人员负责格式校验、运行验证、版本发布

这样既能保证商务规则可维护，也能避免因 JSON 格式错误或系统配置变更导致审核功能异常。
