# CRM智能分析系统项目宪章

## Core Principles

### I. 中文一致性交付

- 所有新增或修改的规格、计划、任务、注释、提示文案、测试记录必须使用中文。
- 代码标识符、数据库表名、字段名、协议关键字可以保留原文，但相邻说明必须使用中文。
- 所有文本文件必须使用 UTF-8 编码，出现乱码、混用编码或无法稳定读取视为门禁失败。
- 理由：统一表达和编码规范可以直接降低需求误读、审查歧义和交付返工。

### II. CRM只读与最小授权边界

- 一期功能只允许从 CRM 只读数据源读取业务数据，禁止向 CRM 业务库写入、回填、删改或旁路更新。
- 权限必须继承 CRM 现有会话、组织、部门、角色和负责人范围，不得建立平行授权体系。
- 未命中治理白名单的角色、数据对象、字段、导出行为和访问范围一律默认拒绝。
- 理由：CRM 是权威业务系统，读写边界和授权来源一旦分叉，会直接破坏数据可信度与审计可靠性。

### III. 受控查询与双重校验

- 自然语言请求必须先转换为结构化分析意图，再由白名单编译器生成受控查询。
- 所有待执行查询必须同时通过表字段白名单校验和 AST 只读校验，并阻断多语句、注释绕过、DDL、DML 与危险函数调用。
- 缺少关键限定条件或语义不明确的请求必须先补问，不得以猜测结果替代用户确认。
- 理由：分析能力的主要风险来自语义漂移、越权拼接和危险查询绕过，必须用双重校验缩小攻击面。

### IV. 审计留痕与关键链路验证

- 成功查询、被拦截查询、成功导出、被拦截导出和安全拦截必须形成可检索审计事件。
- 问数主链路、权限隔离、导出限制、治理配置与审计检索必须同时具备人工验收场景和自动化验证任务。
- 凡是影响安全边界、审计完整性或导出限制的变更，没有验证方案不得进入实施。
- 理由：一期系统的可上线性取决于审计可追踪和关键安全链路可验证，而不是单纯功能可用。

### V. 小步交付与复杂度控制

- 一期只交付商机分析主链路及其治理、导出、审计闭环，其他分析主题必须通过新增规格单独立项。
- 默认采用单仓前后端分离和单体 API 方案；新增服务拆分、中间件或外部基础设施必须说明必要性和收益。
- 任务拆分必须支持按用户故事独立交付、独立验证和独立演示，优先保证 MVP 可落地。
- 理由：当前目标是先交付可审计、可控、可演示的一期能力，过早扩展范围只会放大协调和治理成本。

## 附加约束

- 独立应用库只保存系统自有数据，包括访问策略、分析请求快照、导出请求和审计事件。
- 单次导出上限固定为 1000 行，每名用户每日最多导出 3 次，超限必须即时阻断并留痕。
- 结果展示和导出内容默认排除敏感字段，任何例外都必须通过治理配置显式授权。

## 交付流程与质量门禁

- `spec.md`、`plan.md`、`tasks.md` 三份核心文档必须保持术语一致、范围一致、验证策略一致。
- `plan.md` 必须记录 Phase 0 和 Phase 1 的宪章检查结果；`tasks.md` 必须覆盖关键链路的验证任务。
- 所有新增任务必须包含明确文件路径；影响同一文件的顺序性任务不得错误标记为可并行。
- 任何对宪章约束的例外都必须在计划文档中记录原因、影响范围和替代控制措施。

## Governance

- 本宪章高于功能规格、实施计划、任务清单和一般协作约定；出现冲突时必须优先修正文档而不是弱化宪章。
- 任何修订都必须同步评估 `spec.md`、`plan.md`、`tasks.md`、相关模板和运行时指引是否受影响，并在同一轮变更中完成同步或明确记录待办。
- 版本号必须遵循语义化规则：新增原则或实质扩展约束递增 MINOR；原则删除、重定义或不兼容治理变化递增 MAJOR；文字澄清、格式修正和非语义调整递增 PATCH。
- 每次进入 `/speckit.plan`、`/speckit.tasks`、实现前评审或只读分析时，都必须检查本宪章的符合性，重点覆盖只读边界、权限复用、受控查询、审计留痕和验证任务。
- 宪章修订必须由当前功能负责人和实施负责人共同确认，并同步更新版本号、修订日期以及受影响文档。
