# Tasks: [功能名称]

**Input**: 设计文档来自 `/specs/[###-feature-name]/`
**Prerequisites**: `plan.md`（必需）、`spec.md`（必需）；可选参考 `research.md`、`data-model.md`、`contracts/`、`quickstart.md`

**Tests**: 测试任务不是对所有故事一刀切强制，但只要规格或宪章涉及安全边界、权限隔离、导出限制、治理审计或关键业务链路，就必须在任务清单中显式列出自动化验证任务。
**Organization**: 任务必须按用户故事分组，确保每个故事可独立实现、独立验证、独立演示。

## Format: `[ID] [P?] [Story] Description`

- **[P]**：可并行执行（不同文件、无未完成依赖）
- **[Story]**：所属用户故事（例如 `US1`、`US2`、`US3`）
- 每条任务描述必须包含明确文件路径

## 路径约定

- **单项目**：`src/`、`tests/` 位于仓库根目录
- **前后端分离**：`backend/src/`、`frontend/src/`
- **移动端 + API**：`api/src/`、`ios/` 或 `android/`
- 下方示例默认以单项目为例，生成时必须按 `plan.md` 的真实结构调整

## Dependencies & Execution Order

### Phase Dependencies

- **Phase 1: Setup**：无依赖，可立即开始
- **Phase 2: Foundational**：依赖 Setup 完成，阻塞所有用户故事
- **Phase 3+：用户故事阶段**：都依赖 Foundational 完成，可按优先级串行，也可在具备条件时并行
- **Final Phase: Polish**：依赖所有目标用户故事完成

### User Story Dependencies

- **用户故事 1（P1）**：通常作为 MVP，优先实现
- **用户故事 2（P2）**：可以依赖共享底座，也可以在不破坏独立性的前提下复用用户故事 1 的能力
- **用户故事 3（P3）**：如依赖前序结果，必须在说明中写清依赖原因

### Within Each User Story

- 先写验证任务，再做实现任务
- 先实体/协议，再服务/状态，再接口/页面集成
- 先核心路径，再补充异常处理、审计和提示
- 每个故事完成后必须能单独验收

### Parallel Opportunities

- 标记 `[P]` 的 Setup 任务可并行
- 标记 `[P]` 的 Foundational 任务可在同阶段并行
- Foundational 完成后，不同用户故事可以按团队容量并行
- 同一故事中，不同文件且无依赖的验证任务和模型任务可以并行

## Implementation Strategy

### MVP First（只做用户故事 1）

1. 完成 Phase 1：Setup
2. 完成 Phase 2：Foundational
3. 完成 Phase 3：用户故事 1
4. 停下来独立验证用户故事 1
5. 验证通过后再决定是否继续扩展

### Incremental Delivery

1. 完成 Setup + Foundational，形成共享底座
2. 交付用户故事 1，独立验证后形成 MVP
3. 逐步追加用户故事 2、3 及后续故事，每次都独立验证
4. 最后统一做 Polish 与回归

### Parallel Team Strategy

1. 团队先共同完成 Setup 与 Foundational
2. 共享底座稳定后，再按故事拆分人力并行推进
3. 每个故事完成后先独立验收，再进入跨故事整合

## Notes

- `[P]` 仅用于不同文件、且没有未完成依赖的任务
- `[Story]` 用于把任务映射回具体用户故事
- 任务必须具体到可执行，不得出现无文件路径的模糊描述
- 关键链路一旦涉及安全、权限、导出、审计或性能目标，必须显式落入验证任务
