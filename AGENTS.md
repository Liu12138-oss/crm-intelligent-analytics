# 项目协作规范

## 项目背景

- 本仓库当前聚焦 `specs/001-crm-intelligent-analytics/` 这一条主规格线。
- 一期目标是交付“企业微信 AI 问数 + Web 智能分析工作台 + 治理审计后台”的统一 CRM 智能分析系统。
- 仓库当前以需求文档、数据库梳理、根级设计系统、历史 UI 原型、配置资料和实现规格为主，源码目录会在后续实现阶段逐步补齐。

## 关键文档优先级

- 功能边界、验收口径、数据白名单、页面分区和接口字段，优先以 `specs/001-crm-intelligent-analytics/` 下文档为准。
- 具体顺序建议为：`spec.md` -> `plan.md` -> `data-model.md` -> `contracts/openapi.yaml` -> `quickstart.md` -> `tasks.md`。
- 数据库结构与业务关系以 `docs/数据库梳理/` 和 `docs/db/` 为准。
- 涉及 CRM 官方 API 优先接入、存量非认证能力盘点和方案评审模板时，补充参考 `docs/architecture/crm-api-first-governance.md`。
- UI 视觉方向、组件原则、动效、文案语气和后续新界面设计优先以根级 `DESIGN.md` 为准。
- `docs/prototype-ui/crm-intelligent-analytics-ui.pen` 与 `docs/prototype-ui/UI规范.md` 为旧版历史参考，不再作为新视觉风格、布局结构、卡片形态、尺寸或一级导航的强约束。
- 若 README、规格文档、设计系统、历史原型、任务拆解之间出现冲突，默认功能边界和验收口径以最新主规格为准，视觉与交互方向以 `DESIGN.md` 为准，再同步修正文档。

## 语言要求

- 所有新增或修改的文档正文必须使用中文。
- 所有新增或修改的代码注释必须使用中文。
- 面向用户的提示文案、操作说明、测试说明、变更说明优先使用中文。
- 规格文档、检查清单、设计说明、操作手册等项目文档统一使用中文标题和中文正文。

## 允许保留原文的内容

- 代码标识符、数据库表名、字段名、协议关键字、命令行参数可以保留原文。
- 第三方产品名称、标准名称、专有名词在没有准确中文替代时可以保留原文，并尽量补充中文说明。
- 必须引用的外部协议、许可证或标准原文可以保留英文，但应在相邻位置补充中文解释。

## 注释编写原则

- 注释只在必要时添加，重点说明业务意图、边界条件和设计原因。
- 关键代码、关键判断、边界分支、权限校验、异常处理、状态流转等位置必须补充清晰注释，确保后续协作者能够快速理解设计意图与业务约束。
- 涉及白名单过滤、读写边界限制、审计留痕、身份映射、缓存失效、降级兜底等容易误用或改错的逻辑，必须写明”为什么这样处理”。
- 避免写”给变量赋值”这类低价值注释。
- 注释应说明业务背景、触发条件、失败影响和必要限制，避免只重复代码表面含义。
- 修改现有英文注释时，优先改为中文；如涉及外部协议或固定术语，可采用”中文说明 + 原术语”的方式。

## 代码模块化规约

### 新功能复用与回归要求

- **优先复用现有能力**：新增功能前必须先检查仓库中是否已有可复用的服务、查询口子、组件、模块或聚合逻辑；能扩展现有模块时，不允许在其他入口重复实现同一能力。
- **统一能力出口**：同一类业务能力应优先沉淀为统一服务或模块出口，例如日报查询、周报查询、汇总生成、通知发送等应提供可复用口子，再由企业微信、Web 或其他入口调用，避免每个入口各写一套。
- **禁止为赶进度复制逻辑**：如果发现新需求与旧功能高度相似，必须先抽取公共模块或在原模块上扩展，再实现新入口；禁止通过复制粘贴旧逻辑快速交付。
- **新增功能必须验证旧功能**：开发任何新功能时，必须同步评估并验证受影响的旧功能，至少覆盖直接复用的主链路、相邻入口和已有回归样例，避免“新功能可用但旧功能被带坏”。
- **修复缺陷要补回归测试**：凡是因为新功能引入或暴露的回归问题，修复时必须补对应测试用例，优先使用触发问题的真实输入或真实交互文案，确保同类问题不再重复出现。

### 文件拆分原则

- **单一职责原则**：每个文件应只负责一个明确的职责或功能模块，避免将多个不相关的逻辑混杂在同一文件中。
- **文件行数限制**：单个文件代码行数建议不超过 500 行，超过时应考虑按功能维度进行拆分。
- **拆分维度选择**：
  - 按业务功能拆分（如用户管理、订单处理、权限校验各自独立）
  - 按逻辑层次拆分（如数据访问层、业务逻辑层、接口层分离）
  - 按工具/通用能力拆分（如字符串处理、日期格式化、网络请求等工具函数独立成文件）
- **避免过度拆分**：文件拆分的粒度应适中，过度拆分导致文件数量过多、依赖关系复杂化同样不可取。

### 注释密度要求

- **函数/方法级别**：每个函数或方法体上方必须包含注释，说明：
  - 函数的核心功能与业务目的
  - 参数含义、数据类型、取值范围
  - 返回值说明
  - 可能抛出的异常或错误场景
  - 调用注意事项或前置条件
- **核心逻辑注释**：函数内部的核心业务逻辑、算法实现、状态流转等关键代码块，必须在行内或上方添加注释。
- **条件判断注释**：所有的 `if/else`、`switch`、`try/catch` 等条件分支，必须在分支前注释说明判断的业务含义和分支走向的设计原因。
- **循环注释**：`for`、`while` 等循环结构需注释说明循环目的、终止条件和迭代逻辑。
- **复杂表达式注释**：涉及多步运算、链式调用、正则表达式等复杂逻辑，必须拆解说明每一步的作用。
- **边界处理注释**：空值检查、越界处理、默认值设置等边界场景必须注释说明处理原因。

### 模块组织规范

- **导出规范**：模块对外暴露的接口应通过明确的导出声明集中管理，避免直接导出内部实现细节。
- **依赖声明**：文件顶部的依赖引入应按类型分组（如系统库、第三方库、项目内部模块），并添加分组注释。
- **内部函数管理**：仅在模块内部使用的辅助函数应明确标识，并集中放置在文件适当位置，配套完整注释。

## 文档与文件编码

- 文本文档统一使用 UTF-8 编码，避免出现乱码。
- 在脚本或命令中读写中文文件时，应显式使用 UTF-8。

## 敏感信息处理

- `配置/` 目录下包含本地接入资料和敏感配置，禁止把明文密钥、数据库密码、机器人密钥、Token、网关地址凭证写入源码、README、规格文档、测试夹具、日志或截图说明。
- 如需在代码或文档中引用配置，只能使用环境变量名、占位符或“配置来源路径 + 中文说明”的方式表达。
- 涉及 `配置/codex配置/auth.json`、`配置/企业微信/ID和密钥.md`、`配置/数据库/数据库账号密码.txt` 的内容时，只允许提炼结构，不允许转抄敏感值。

## 规格与交付要求

- 功能规格、需求清单、验收标准、测试记录统一使用中文表达。
- 若必须出现英文缩写，应优先给出中文含义，再保留必要缩写。
- 新增示例、占位说明、模板内容时，默认使用中文。
- 若修改了一期能力边界、页面结构、接口字段或数据白名单，至少同步检查 `spec.md`、`plan.md`、`data-model.md`、`quickstart.md`、`contracts/openapi.yaml` 是否需要一起更新。
- 若修改了治理、审计、模板或最近查询相关能力，需同时检查 `tasks.md` 与 `quickstart.md` 中的验证场景是否仍然成立。
- 若新增、下线或调整企业微信机器人用户可见能力、主题入口、取消/阻断提示中的能力清单，必须同步更新 `README.md` 中“企业微信机器人当前支持能力”章节，以及机器人统一能力目录或提示配置，确保用户看到的功能列表与真实能力一致。

## UI输出

- 新增或重构 UI 视觉时，必须优先遵循根级 `DESIGN.md`，不得继续复用旧 `.pen` 的白底蓝灰后台风格作为强约束。
- 需要输出可读尺寸原型时，必须使用 pencil 插件生成新的 `.pen` 文件，且新原型应与 `DESIGN.md` 保持一致。
- 允许围绕“AI 经营指挥舱”重新设计一级导航、核心卡片区块、结果详情、合同审核和治理审计结构；若改动触及功能边界、核心入口或验收口径，必须同步更新主规格或新增 OpenSpec 变更。

## 前端代码样式

- 前端实现必须优先参考 `DESIGN.md` 的设计系统、交互反馈和无障碍要求。
- 当前仓库默认吸收并遵循本地已安装的 `taste-skill` 作为前端审美与实现偏好补充，重点复用其“避免模板化 AI 视觉”“避免紫色霓虹和过饱和渐变”“避免默认三等分卡片布局”“强调单一主强调色、明确层级、完整交互状态、移动端稳定性和性能约束”等原则。
- `taste-skill` 只作为补充约束，若其建议与 `DESIGN.md`、主规格文档、现有信息架构、无障碍要求或本仓库既定技术栈冲突，必须以本仓库规则优先，不得生搬硬套。
- `taste-skill` 中与本仓库不一致的技术默认项，如 React / Next.js、Tailwind CSS、`@phosphor-icons/react`、`@radix-ui/react-icons` 等，只能吸收其设计意图，不能覆盖当前 Vue 3 + `Element Plus` + `@element-plus/icons-vue` 的实现约束。
- 前端重构统一选用 `Element Plus` 作为 Vue 3 基础 UI 组件库，并配套使用 `@element-plus/icons-vue` 补齐图标；不得继续用纯文字、首字母或无语义符号替代导航、操作、状态、风险、上传、导出、刷新、筛选、搜索、返回、关闭、展开收起等关键图标。
- Element Plus 只能作为基础组件和图标能力来源，必须通过主题变量、项目级封装和全局样式保持 `DESIGN.md` 的浅色主导、可信结果呈现、统一控件圆角、标签不换行和响应式要求；不得直接套用默认后台视觉。
- 涉及选择人员或部门的前端入口，默认必须复用统一企业微信组织架构选择组件（如 `WecomOrgSubjectPicker.vue` 或其项目级封装），支持按组织树选择人员、选择部门、按部门批量选择人员和搜索定位；不得在权限中心、治理页、业务配置页或后续新增页面中另写平铺人员/部门下拉作为首选方案。
- 统一组织架构选择组件必须明确区分“选择人员”“选择部门”和“选择部门下人员”，并按业务场景输出 CRM 用户 ID、企业微信 userid、CRM 部门 ID 或企业微信部门 ID；需要保存 CRM 主体的场景必须禁用未完成 CRM 映射的企业微信成员或部门，避免把企业微信 ID 误写入 CRM 授权字段。
- 若某个新增业务确实不能复用统一组织架构选择组件，必须在设计说明或 OpenSpec 变更中写明原因，例如选择对象并非企业微信/CRM 组织对象，或现有组件无法表达该业务的特殊约束；不得仅因实现方便而绕开统一组件。
- 旧 `.pen` 文件仅用于追溯历史页面范围，不得作为阻止新视觉、新布局或新信息架构的依据。
- 工作台、模板治理、审计检索、合同审核等页面可以按 `DESIGN.md` 重设布局分区，但必须继续满足一期主规格定义的功能边界、权限说明、审计留痕和验收场景。

## 交互体验要求

- 功能页默认保持简洁、简约、好操作。除表单校验、权限阻断、空表格占位和必要的业务失败提示外，不得用大块说明、规则解释、状态描述或“如何使用”文案长期占据主操作区；低频解释应收进悬浮提示、弹窗或帮助入口。
- 分析结果、趋势预测、风险提示和经营建议的文案必须可读可信：每个数值必须说明对应指标、时间范围或业务对象；不得只展示孤立数字、内部状态词或算法术语，例如“低置信”“方向预测”“大概率落在 X 到 Y”这类表达必须转写为业务用户能理解的含义、用途和限制。
- 分析结果页不得堆叠重复文案。摘要、关键发现、预测、关注事项和建议之间必须各自提供不同价值：摘要回答“这次查到了什么”，关键发现回答“最值得看什么”，预测回答“预测的是哪个指标、区间是多少、能用来做什么”，边界说明回答“为什么只能作参考”，建议回答“下一步怎么处理”。
- 所有 AI 生成或程序拼接的用户可见文案都必须在展示前做可读性审查，过滤内部枚举、置信等级、fallback、schema、任务名等实现细节；确需表达不确定性时，应说明“不确定在哪里、会影响什么、用户下一步如何复核”。
- 所有页面、抽屉、弹窗、消息提示、表格、标签、导出文件和机器人回复中的用户可见内容，必须先把接口字段名、数据库字段名、枚举值、状态码、视图类型、动作类型、权限键、模块名等内部代码转换为面向业务用户的中文表达；不得直接显示 `BAR_CHART`、`TABLE`、`RANKING_TABLE`、`ACTIVE`、`QUERY_BLOCKED`、`analysis.use`、`template.view` 这类未解释代码。确需辅助排障时，应只在治理审计或开发诊断入口展示，并同时给出中文含义。
- 查询类列表、表格和筛选区统一采用“查询条件在前，`查询`按钮紧跟查询框或筛选控件之后”的布局；不得把同一语义的按钮命名为“刷新”“刷新数据”“刷新列表”“刷新报表”。凡是可输入查询条件的控件，必须支持按 `Enter` 执行同一次查询。
- 前端页面必须补齐关键交互反馈，至少覆盖鼠标悬浮、点击、聚焦、禁用等常见状态，避免只有静态界面而缺少可感知反馈。
- 所有接口请求都应提供清晰的用户反馈，至少覆盖加载中、成功、失败、空状态、无权限、超时或异常等场景，必要时提供重试入口或下一步指引。
- 新增或修改表单、筛选、导出、复制、切换、提交等操作时，必须同步设计成功提示、失败原因提示和处理建议，避免只在控制台输出错误。
- 所有面向用户的错误提示必须使用亲切友好的中文，先说明“发生了什么”，再明确说明“下一步怎么做”；禁止只返回“失败”“异常”之类没有行动指引的提示。
- 除非页面明确面向治理排障或开发诊断，否则错误提示中不得直接暴露数据表名、字段名、SQL 片段、内部 ID、堆栈、错误码、运行时常量名等技术细节；应统一翻译成业务可理解的提示。
- 所有位于页面右上角的悬浮提示，默认应在 `3` 秒后自动消失；若需要长时间保留，必须有明确业务理由，并提供关闭入口。
- 页面中的低频说明、字段释义、口径解释和规则提示，默认优先使用 Element Plus 现成的悬浮提示能力，如 `ElTooltip`、`ElPopover`，不要把这类帮助文案长期平铺在正文里。
- 操作成功、失败和警告类瞬时反馈，默认优先使用悬浮消息，如 `ElMessage`，除非该提示必须持续占据页面上下文才能完成任务。
- 动效应服务于理解和操作反馈，重点用于面板切换、列表刷新、结果出现、按钮响应、消息提示等场景；禁止使用影响效率或造成干扰的过度动画。
- 用户体验优化应优先保证可读性、响应速度和反馈确定性，避免出现“操作已执行但界面无反馈”“请求失败但用户无感知”等问题。

## 后端实现约束

- 分析问数、结果导出补查、企业微信目录同步、跟进写回等 CRM 执行链路，必须按各自规格声明受控查询、受控写回或受控同步边界，不得再把 CRM 链路统一假设为单一路径；任何读写能力都不得绕过白名单、确认、审计与权限约束。
- 涉及 CRM 的新增功能和既有非认证能力改造，默认先评估并优先复用 CRM 官方 API；只有官方 API 不满足对象、字段、权限或稳定性要求时，才允许自建接口或操作数据库。
- 采用自建接口或数据库路径时，必须同时写清 API 缺口原因、表字段白名单、权限继承来源、审计留痕、失败降级和回退方案，避免把“查库更快”当作默认理由。
- 后续方案评审默认按 `docs/architecture/crm-api-first-governance.md` 中的“官方 API 可用性检查模板”和“方案评审检查清单”执行，未补齐材料视为未完成设计输入。
- 企业微信身份识别优先复用现有映射表：`wx_user_maps`、`wx_users`、`wx_organization_maps`、`wx_departments`、`wx_user_department_maps`。
- 权限判断优先复用现有 CRM 用户、角色、组织、部门和主体归属关系，不允许另起一套独立权限模型。
- 常用查询、最近查询、导出和审计行为必须遵循当前实时权限，不能复用旧授权快照直接放行。
- 用户认证、登录、会话建立与统一鉴权链路当前视为稳定能力，本轮及后续同类梳理默认不动；若必须调整，需单独立规格，不与其他 CRM 接入改造混做。

### 企业微信 Web 登录门户回跳约束

- 2026-05-22 已修复“门户进入系统后企业微信扫码成功但页面停在 `302 Found`”的问题。根因是生产门户会隐藏外层代理上下文，扫码初始化请求可能只能拿到内层 IP 形态的 `webBaseUrl` / `returnUrl`，且完全拿不到门户所需的 `GratuitousProxy`。
- 该场景下企业微信换票和共享 Cookie 写入已经成功，用户手动浏览器返回会进入已登录页面；因此后端在共享 Cookie 直登成功后，如果成功回跳地址缺少 `GratuitousProxy`，必须返回自动 `window.history.back()` 的登录完成页，回到扫码前的门户上下文。只有成功回跳地址已明确包含 `GratuitousProxy` 时，才允许继续直接 `302` 到业务页。
- 后续修改 `backend/src/modules/auth/auth.controller.ts`、`backend/src/modules/auth/wecom-web-login.service.ts`、前端登录页、路由守卫、退出登录、`APP_WEB_BASE_URL` / `APP_WEB_ALLOWED_BASE_URLS` / `APP_WEB_SHARED_COOKIE_DOMAIN` 相关逻辑时，必须保留这个兜底，不得把缺少门户代理参数的场景改成强制直跳门户业务页。
- 涉及该链路的改动必须同步运行并维护 `backend/test/auth.controller.spec.ts` 中的门户回跳用例，至少覆盖“带 `GratuitousProxy` 时直跳业务页”和“缺少 `GratuitousProxy` 时自动返回扫码前页面”两类行为。

### 自然语言输入 AI-first 约束

- 对企业微信机器人、Web 智能分析和合同审核后续自然语言交互中的业务文本输入，默认执行顺序必须是“固定安全前置检查 -> 统一 AI 理解层 -> 固定受控程序执行”，不得继续把关键词表、状态机或局部规则判断作为主理解链路。
- 固定安全前置检查只包含签名校验、来源校验、登录鉴权、文件类型与大小校验、权限装载、白名单和危险动作阻断；这些步骤必须由程序先做，不能交给 AI。
- 规则逻辑只允许作为 fallback 和安全兜底存在，不能继续通过追加词表、短句特判或模糊字符串判断去承担主要语义理解职责。
- 后续遇到“长正文被误判成确认短句”“`不补充` 被误判成取消”“缺项提示不贴合真实缺口”“原文里已明确出现的问题/计划没有抽到对应字段”这类问题时，默认修复顺序必须是“先优化提示词 / few-shot / 负例 / 结构化 contract，再补 fixture 和回归测试，最后才评估是否需要程序边界兜底”；不得一上来继续加关键词和状态分支。
- AI 应优先负责上下文理解、意图识别、字段抽取和补充提示内容；程序只负责固定安全前置检查、权限边界、状态门闩、对象唯一性、白名单和审计留痕。除非涉及安全或执行边界，否则不得把语义问题长期沉淀成程序规则。
- 针对企业微信活跃任务、跟进四段草稿、候选确认和解释追问，提示词设计必须显式覆盖正例、反例和边界例：至少包含“长正文补充 vs 确认短句”“跳过可选字段 vs 取消任务”“只缺单个字段时的补充提示”“问题/阻塞/下一步计划的字段归属”。
- fallback 命中时必须显式记录 `fallbackReason`，并且不得把短回复、口语确认词或模糊文本当成新的实体名、业务正文或可执行动作继续推进。
- 后续新增或修改任何企业微信 / Web 自然语言能力时，若实现方案仍以规则识别为主、AI 仅作补充，视为不满足当前架构约束；必须先说明为什么不能接入统一 AI 理解层。
