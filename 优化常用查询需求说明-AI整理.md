# 优化常用查询需求说明（AI 整理版）

## 1. 背景与目标

当前 CRM 智能分析系统已经具备 Web 自由问数、常用查询、最近查询、查询模板治理、模板 SQL 校验、模板执行审计和最近查询重跑能力。现有实现中，Web 智能分析工作台的“常用查询”和“最近查询”均通过右侧抽屉打开；治理后台已有“查询模板”页面，可新增、编辑、删除固定 SQL 模板，并支持 SQL 校验与预览。

本次优化目标是把“常用查询”从辅助入口升级为高频查询资产中心，让业务用户能够更快复用高价值问数结果，并让具备 SQL 能力的用户或管理员能把可信查询沉淀为可持续治理的模板资产。

核心目标包括：

- 自由问数结果可保存为查询模板，沉淀有价值问题与对应受控 SQL。
- 常用查询支持“我的模板”和“其它模板”分层管理。
- 模板支持分类或标签，用于搜索、筛选和后续治理。
- 常用查询直接展示在查询页左侧，不再作为抽屉内的低频入口。
- 最近查询保持现状，继续放在抽屉中。
- 支持大规模模板列表下的搜索、筛选、虚拟滚动和按点击率排序。
- 具备 SQL 能力的用户可手动新增完整查询模板，并补充分类。

## 2. 当前项目理解

### 2.1 已有能力

- Web 工作台已有自由问数主链路，入口位于 `frontend/src/pages/analysis/AnalysisWorkbenchPage.vue`。
- 常用查询面板组件为 `frontend/src/components/analysis/CommonQueryPanel.vue`，当前通过“常用查询”按钮打开抽屉展示。
- 最近查询面板组件为 `frontend/src/components/analysis/RecentQueryPanel.vue`，当前也通过抽屉展示。
- 前端查询状态集中在 `frontend/src/stores/analysis-query.store.ts`，已支持加载模板、执行模板、加载最近查询、重跑最近查询和结果导出。
- 后端查询模板能力集中在 `backend/src/modules/query-assets/`，已支持模板列表、模板执行、治理列表、新增、更新、删除、校验和预览。
- 当前数据模型中的 `CommonQueryTemplate` 已包含模板名称、说明、默认问题、默认筛选、固定 SQL、参数定义、展示配置、可见角色、展示顺序、点击量、命中率、状态、维护人、校验快照等字段。

### 2.2 当前差距

- 常用查询当前位于抽屉中，不满足“直接展示在查询页面最左侧”的要求。
- 常用查询当前没有“我的模板 / 其它模板”两个标签页。
- 模板数据模型当前没有明确的“分类 / 标签”字段，也没有用户个人复制、收藏或归属关系模型。
- 自由问数结果页当前没有“保存为模板”动作。
- 自由问数链路虽然能返回结果与执行快照，但需要确认是否能稳定回传可保存的受控 SQL 或执行计划摘要。
- 模板列表当前主要按 `displayOrder` 排序，不满足按点击率优先排序的要求。
- 治理后台新增模板当前支持 SQL、默认条件、展示配置，但尚未提供“分类”字段。
- 当前工作台常用查询列表没有针对几百到上千条模板的虚拟滚动方案。

## 3. 用户与权限范围

### 3.1 业务用户

业务用户是使用 Web 智能分析工作台进行问数和复用查询的人。

业务用户应支持：

- 在左侧常用查询区查看“我的模板”和有权访问的“其它模板”。
- 搜索模板名称、说明、问题文本和标签。
- 按标签筛选模板。
- 执行模板并生成结构化分析结果。
- 将一次自由问数结果保存为“我的模板”。
- 将“其它模板”复制或添加到“我的模板”。

### 3.2 具备 SQL 能力的用户

具备 SQL 能力的用户是获得模板创建或模板治理权限的人，可以在受控边界内新增完整查询模板。

此类用户应支持：

- 在查询模板治理页或受控新增入口中手动填写模板名称、问题、说明、SQL、默认条件、展示配置和分类。
- 选择已有分类，或自由输入新分类。
- 新增模板时必须通过只读 SQL、白名单、范围治理和预览校验。

### 3.3 管理员或治理人员

管理员或治理人员负责模板治理、分类治理、权限可见性和审计复核。

此类用户应支持：

- 查看全量模板，包括个人模板、公共模板、停用模板和待优化模板。
- 调整模板分类、状态、说明、SQL、展示配置、可见范围和治理校验状态。
- 通过审计中心追踪模板保存、复制、执行、编辑、删除、校验失败和越权阻断行为。

## 4. 功能需求

### 4.1 自由问数结果保存为模板

当用户通过自由问数得到结果后，如果认为本次查询有复用价值，页面应提供“保存为模板”按钮。

保存内容至少包括：

- 原始问题或标准化问题。
- 本次实际执行的受控 SQL 或可复现的受控执行计划。
- 当前使用的筛选条件。
- 结果展示配置，例如主视图类型、表格列、图表维度和指标字段。
- 模板名称，默认可由问题自动生成，用户可编辑。
- 模板说明，默认可由 AI 根据问题和结果摘要推荐，用户可编辑。
- 分类或标签，用户必填或选择，AI 可提供小字推荐。
- 模板归属人，默认为当前用户。
- 创建来源，标记为“自由问数保存”。

保存前必须校验：

- 当前结果必须成功返回，不能保存失败、阻断、超时或空执行快照。
- 保存的 SQL 或执行计划必须继续满足只读、白名单、范围治理和危险语句拦截规则。
- 保存模板不得绕过当前权限，后续执行必须按执行人实时权限重新收口。
- 自由问数保存优先使用后端实际执行的受控 SQL 快照；如果当前链路只保存执行计划，后端应在保存时把执行计划编译为参数化 SQL 模板并重新校验；如果既没有可复现 SQL，也无法从执行计划安全编译，则提示“当前查询暂不支持保存为模板”，不得保存不可执行的半成品模板。
- 普通业务用户保存的模板默认全员可见，可被其他用户搜索、添加到“我的模板”和分享；执行时仍必须按执行人当前实时权限收口。

### 4.2 模板分类与标签

模板需要支持分类能力，用于业务查找和治理归档。

分类规则：

- 保存模板、新增模板、编辑模板时均应支持分类字段。
- 用户可以选择已有分类。
- 具备权限的用户可以输入新分类。
- AI 可根据问题、SQL、结果摘要或业务主题推荐分类，但推荐内容只能作为辅助，不自动覆盖用户选择。
- 分类名称应使用中文业务表达，例如“商机分析”“合同转化”“客户提单”“团队预测”“个人常用”。
- 分类不得直接展示内部枚举、数据库表名或权限键。

分类与标签实现口径：

- 首期同时支持主分类和多标签。
- `category` 表示模板主分类，用于一级筛选和治理归档。
- `tags` 表示多标签集合，用于补充业务主题、指标口径、部门场景和高频用途。
- 用户保存模板时至少选择或填写主分类；多标签支持选择已有标签，也支持自由输入新标签。
- AI 可以推荐标签，但推荐标签必须由用户确认后才写入模板。
- 标签名称应使用中文业务表达，避免直接保存表名、字段名、权限键或内部枚举。

### 4.3 常用查询分层：我的模板与其它模板

工作台常用查询区应分为两个标签页：

- 我的模板。
- 其它模板。

“我的模板”包含：

- 当前用户通过自由问数保存的模板。
- 当前用户手动创建的模板。
- 当前用户从“其它模板”添加到我的模板后生成的个人副本。

“其它模板”包含：

- 当前用户有权查看但不属于当前用户个人模板的启用模板。
- 管理员发布的公共模板。
- 其他用户保存或发布且当前用户有权访问的模板。

默认展示规则：

- 进入分析页后优先展示“我的模板”。
- 如果“我的模板”为空，可以显示简洁空状态，并提供切换到“其它模板”的入口。
- 不应长期展示大块教学说明，低频解释可通过悬浮提示提供。

### 4.4 从其它模板添加到我的模板

用户可以通过两种方式把其它模板保存到“我的模板”。

方式一：常规保存路径。

1. 用户在“其它模板”中点击模板并执行查询。
2. 查询成功后，用户点击结果页“保存为模板”。
3. 系统将本次执行的问题、参数和 SQL 保存为用户自己的模板。

方式二：列表快捷添加。

1. 用户在“其它模板”列表中点击“添加到我的模板”。
2. 系统直接把该模板添加到“我的模板”。
3. 添加成功后给出 3 秒自动消失的中文成功提示。

快捷添加采用个人副本策略：

- 用户在“其它模板”中点击“添加到我的模板”时，系统复制一条新的模板记录，归属到当前用户。
- 个人副本需要保留 `sourceTemplateId`、复制时间和来源模板名称，便于后续追溯。
- 添加后的模板就是当前用户自己的模板，允许在具备对应权限时继续编辑名称、说明、分类、标签和 SQL。
- 个人副本后续不自动跟随来源模板更新；来源模板被修改、停用或删除时，不应静默影响已经保存到“我的模板”的个人副本。
- 普通用户保存的新模板默认全员可见，同时也归属到保存人的“我的模板”。

### 4.4A 固定部门模板的权限改写规则

跨部门复用模板时，系统必须区分“查询口径复用”和“原部门数据复用”。用户可以复用模板的查询口径，但不能因为保存、复制或添加模板而获得来源模板中的部门数据权限。

典型场景：

- 山东领导保存了一个查询山东区经营数据的模板。
- 北京领导希望使用同一查询口径。
- 如果北京领导没有山东区数据权限，系统不得按原模板查询山东数据。
- 系统可以辅助把模板改写为北京领导当前可见范围，但不能静默改写后直接执行。

固定部门范围识别规则：

- 模板问题文本中写死部门名称，例如“山东区”“山东销售”。
- 默认筛选条件中写死部门 ID、部门名称、负责人组或区域名称。
- SQL 中写死部门 ID、部门名称、部门路径、团队枚举或固定 `UNION` 团队清单。
- SQL 中虽然没有明显部门字段，但展示维度、筛选条件或硬编码别名实际绑定了特定部门。

执行规则：

- 如果执行人覆盖固定部门范围，允许按原模板执行，但仍需执行实时权限校验、范围治理和审计。
- 如果执行人不覆盖固定部门范围，系统必须阻断原模板执行，不能先查原部门再做结果过滤。
- 系统可以给出 AI 辅助改写建议，例如“该模板原范围为山东区，你当前无权查看山东数据。是否改为查询你当前可见范围：北京区？”
- 用户确认后，系统才允许将固定部门条件替换为当前用户可见范围，再重新执行 SQL 白名单、AST 安全校验、范围注入、预览或执行审计。
- 如果无法可靠识别或替换固定部门条件，系统必须阻断执行，并提示联系管理员将模板治理为通用范围模板。

改写边界：

- AI 只负责识别固定部门表达、生成面向用户的改写建议和辅助生成候选改写方案。
- 程序必须负责权限校验、范围替换、SQL 安全校验、白名单校验、执行前确认和审计留痕。
- 不允许 AI 静默把“山东”改成“北京”后直接执行。
- 不允许把执行人的当前部门自动写回来源模板；如需保存改写结果，应保存为当前用户个人模板副本或个人参数配置。

### 4.5 搜索与筛选

“我的模板”和“其它模板”两个标签页都必须支持搜索。

搜索能力包括：

- 模糊搜索模板名称。
- 模糊搜索模板说明。
- 模糊搜索默认问题。
- 按分类或标签筛选。
- 按用户筛选，至少在“其它模板”中支持按创建人或维护人筛选。

交互要求：

- 查询条件在前，“查询”按钮紧跟筛选控件之后。
- 搜索输入框必须支持按 `Enter` 执行查询。
- 搜索、筛选、切换标签页后不得清空当前分析结果。
- 空结果应说明“当前条件下没有匹配模板”，并展示已用筛选条件摘要。

### 4.6 大列表与虚拟滚动

模板数量可能达到几百到上千条，常用查询列表需要支持大列表体验。

要求：

- “我的模板”和“其它模板”均应支持虚拟滚动或等效的大列表性能方案。
- 列表滚动、搜索、标签筛选和添加到我的模板操作不应造成页面明显卡顿。
- 单条模板卡片高度应稳定，避免动态内容导致滚动定位跳动。
- 列表加载中、加载失败、空状态和无权限状态必须有明确中文反馈。

### 4.7 按点击率排序

不论“我的模板”还是“其它模板”，默认排序都应按使用热度优先。

排序建议：

- 优先按历史累计点击量或执行次数倒序。
- 点击量相同的模板，可按最近使用时间倒序。
- 再相同可按更新时间倒序。
- 保留治理后台的展示顺序字段，但工作台默认以点击率为主。

点击统计要求：

- 模板被执行时应增加点击或使用统计。
- 从“其它模板”添加到“我的模板”不应直接等同于执行点击。
- 统计数据应进入审计或可追踪运行态，避免无法解释排序变化。

### 4.8 常用查询左侧常驻展示

常用查询是高频入口，应直接展示在 Web 查询页面最左侧，不再默认放入抽屉。

布局要求：

- 分析页形成“左侧常用查询区 + 右侧查询与结果区”的结构。
- 左侧常用查询区默认给定固定宽度。
- 支持点击拉伸到更宽宽度。
- 支持点击收缩回默认宽度。
- 支持用户自由拖拽调整宽度。
- 宽度应设置最小值和最大值，避免遮挡查询结果或导致内容不可读。
- 小屏幕下可折叠为顶部按钮或抽屉，但桌面端默认必须常驻显示。

建议宽度：

- 默认宽度：`320px`。
- 最小宽度：`280px`。
- 最大宽度：`520px`。
- 宽屏可允许用户拖拽到更宽，但不应超过主内容可用宽度的 40%。

### 4.9 最近查询保持抽屉

最近查询保持现状，继续通过抽屉展示。

要求：

- 不改变最近查询的个人历史属性。
- 最近查询重跑继续按当前实时权限重新执行。
- 最近查询入口可以保留在查询输入区操作按钮中。
- 常用查询左侧常驻后，不应把最近查询也强行塞入左侧常驻区。

### 4.10 SQL 用户手动新增模板

如果用户具备 SQL 模板能力，且现有模板不能满足需要，应允许其自由新增完整查询模板。

新增入口可复用现有“查询模板”治理页面，并在当前基础上补充“分类”字段。

权限边界：

- SQL 模板能力只授予“编写和保存查询 SQL”的最小权限，不等同于完整模板治理、权限治理或审计治理权限。
- 如果现有权限中心没有细粒度能力点，建议新增 `template.sql.write` 或等价中文权限“查询模板 SQL 编写”，仅允许进入受控 SQL 新增、校验、预览和保存流程。
- 该权限不能绕过只读校验、白名单、范围治理、执行审计和导出再次鉴权。

新增模板字段至少包括：

- 模板名称。
- 模板说明。
- 分类。
- 默认问题。
- 查询 SQL。
- 默认筛选条件。
- 参数定义。
- 展示配置。
- 启停状态。

校验要求：

- SQL 必须是只读查询。
- SQL 必须命中允许表字段白名单。
- SQL 必须通过 AST 安全校验和危险语句拦截。
- SQL 必须能按当前用户权限自动收口，或声明范围并通过覆盖校验。
- 预览只能返回当前用户有权查看的数据。
- 保存、预览、执行、导出都必须审计。

## 5. 数据模型影响

建议扩展 `CommonQueryTemplate` 或新增关联实体。

### 5.1 模板主表字段扩展

建议为模板增加以下字段：

| 字段 | 说明 | 必填 | 规则 |
| --- | --- | --- | --- |
| `category` | 模板主分类 | 是 | 中文业务分类，可选择已有分类或新增 |
| `tags` | 模板标签集合 | 否 | 首期支持多选已有标签、自由输入新标签、搜索筛选和 AI 推荐 |
| `sourceType` | 模板来源 | 是 | `GOVERNANCE_CREATED`、`FREE_QUERY_SAVED`、`USER_SQL_CREATED`、`COPIED_FROM_TEMPLATE` |
| `sourceQueryId` | 来源分析请求 ID | 否 | 自由问数保存时填写 |
| `sourceTemplateId` | 来源模板 ID | 否 | 从其它模板复制为个人副本时填写 |
| `ownerUserId` | 模板创建人 | 是 | 普通用户保存或 SQL 用户创建时填写 |
| `visibilityType` | 可见性类型 | 是 | `PRIVATE`、`SHARED`、`GOVERNED_PUBLIC` |
| `lastUsedAt` | 最近使用时间 | 否 | 用于工作台排序 |
| `usageCountTotal` | 历史累计使用次数 | 是 | 工作台默认排序主字段，可由现有点击统计迁移 |

### 5.2 个人模板副本关系

“我的模板”采用个人副本模式。点击“添加到我的模板”会创建一条新的模板记录，并通过来源字段追溯原模板，不再只保存用户与模板的关联关系。

| 字段 | 说明 | 必填 | 规则 |
| --- | --- | --- | --- |
| `id` | 新模板 ID | 是 | 全局唯一 |
| `sourceTemplateId` | 来源模板 ID | 是 | 必须存在于复制时刻的模板列表 |
| `ownerUserId` | 个人副本所属用户 | 是 | 当前 CRM 用户 |
| `sourceSnapshot` | 来源模板摘要快照 | 是 | 至少包含来源模板名称、分类、标签和 SQL 版本 |
| `createdAt` | 添加时间 | 是 | 系统生成 |
| `lastUsedAt` | 个人最近使用时间 | 否 | 排序辅助 |

个人副本创建后作为独立模板管理。来源模板后续变更不自动覆盖个人副本，但治理后台和审计需要能追溯它来自哪条模板。

### 5.3 分类字典

分类可以首期直接使用模板上的字符串字段；如果需要治理分类，则新增分类字典。

| 字段 | 说明 | 必填 | 规则 |
| --- | --- | --- | --- |
| `id` | 分类 ID | 是 | 全局唯一 |
| `name` | 分类名称 | 是 | 中文业务名称 |
| `createdBy` | 创建人 | 是 | CRM 用户 ID |
| `status` | 状态 | 是 | `ACTIVE`、`INACTIVE` |
| `createdAt` | 创建时间 | 是 | 系统生成 |

## 6. 接口影响

### 6.1 工作台模板列表接口

现有接口：`GET /api/v1/analysis/templates`。

建议扩展查询参数：

- `scope`：`mine` 或 `others`。
- `keyword`：关键词。
- `category`：分类。
- `ownerUserId`：创建人或维护人。
- `page`：页码。
- `pageSize`：每页数量。
- `sort`：默认 `usage_desc`。

响应建议增加：

- `items`：模板列表。
- `page`、`pageSize`、`total`。
- `categories`：当前条件下可用分类摘要。
- `scope`：当前列表范围。

### 6.2 保存自由问数为模板

建议新增接口：

`POST /api/v1/analysis/queries/{queryId}/templates`

请求字段：

- `name`。
- `description`。
- `category`。
- `tags`。
- `visibilityType`，默认 `SHARED`。
- `defaultFilters`。
- `renderConfig`。

后端根据 `queryId` 读取已成功执行的受控 SQL 或执行计划，不允许前端直接提交自由拼接 SQL 来保存自由问数模板。若只有执行计划，后端需要先编译为参数化 SQL 模板并通过完整安全校验；无法安全编译时返回不可保存提示。

### 6.3 添加到我的模板

`POST /api/v1/analysis/templates/{templateId}/copy-to-mine`

首期使用 `copy-to-mine` 表达“添加到我的模板”。该接口复制模板定义快照并生成当前用户自己的模板记录，同时写入来源模板 ID、来源模板名称、来源 SQL 版本和复制时间。

### 6.4 分类列表接口

建议新增接口：

`GET /api/v1/analysis/template-categories`

用于工作台和治理页获取已有分类。

如果首期分类只来自模板聚合，也可以由模板列表接口返回分类摘要，避免新增接口。

### 6.5 治理模板接口

现有接口：

- `GET /api/v1/governance/query-templates`
- `POST /api/v1/governance/query-templates`
- `PUT /api/v1/governance/query-templates/{templateId}`

需要扩展请求与响应字段：

- `category`。
- `tags`。
- `sourceType`。
- `ownerUserId`。
- `visibilityType`。
- `sourceTemplateId`。

## 8. 后端治理与审计要求

新增或调整行为必须记录审计：

- 自由问数保存为模板。
- 保存失败。
- 添加到我的模板。
- 从其它模板复制为个人模板。
- 个人模板执行。
- 公共模板执行。
- 模板分类新增或修改。
- SQL 模板新增、校验、预览、执行和删除。
- 模板执行被权限或白名单阻断。
- 固定部门模板被识别、被阻断、被用户确认改写或改写失败。

审计中至少记录：

- 操作人。
- 模板 ID。
- 来源查询 ID 或来源模板 ID。
- 分类。
- 操作类型。
- 当前权限范围快照。
- 固定部门识别结果和改写前后的范围摘要。
- 执行结果。
- 失败或阻断原因。

模板保存和执行必须继续遵循：

- 当前实时权限。
- 数据白名单。
- 只读 SQL。
- AST 安全校验。
- 范围注入或声明范围覆盖校验。
- 导出再次鉴权。
- 敏感字段屏蔽。

## 9. 验收标准

### 9.1 自由问数保存

- 给定用户成功执行一次自由问数，当点击“保存为模板”并填写分类后，模板出现在“我的模板”中。
- 给定自由问数结果没有可复现 SQL，且执行计划也无法安全编译为参数化 SQL，当用户点击保存时，系统给出明确中文提示，不创建不可执行模板。
- 给定用户保存模板后再次执行，结果按当前用户实时权限重新计算，不复用旧授权快照。
- 给定普通业务用户保存模板后，其他用户可在“其它模板”中搜索到该模板，并可添加到自己的“我的模板”。

### 9.2 我的模板与其它模板

- 给定用户进入分析页，默认看到左侧常用查询区和“我的模板”标签页。
- 给定用户没有个人模板，页面展示简洁空状态，并允许切换到“其它模板”。
- 给定用户在“其它模板”中点击“添加到我的模板”，该模板出现在“我的模板”中。
- 给定用户在“其它模板”中点击“添加到我的模板”，系统创建当前用户自己的模板副本，并记录来源模板 ID。
- 给定用户执行其它模板后再保存，系统创建一条新的全员可见模板，并自动添加到保存人的“我的模板”。

### 9.2A 固定部门模板跨部门复用

- 给定山东领导保存了写死山东范围的模板，当北京领导没有山东权限并尝试执行原模板时，系统必须阻断原范围执行。
- 给定系统能识别固定山东范围且能安全替换，当北京领导确认“改为当前可见范围”后，系统按北京领导当前权限范围重新校验并执行。
- 给定系统无法可靠替换固定部门条件，当北京领导尝试改写执行时，系统必须阻断并提示联系管理员治理为通用模板。
- 给定北京领导确认改写后，系统不得修改来源模板，只能生成本次执行参数、个人模板副本或个人配置。
- 给定固定部门模板发生阻断或改写执行，审计记录必须能看到来源模板、原固定范围、执行人当前范围、用户确认状态和最终结果。

### 9.3 搜索、分类和大列表

- “我的模板”和“其它模板”均支持按关键词搜索。
- 两个标签页均支持按分类筛选。
- 两个标签页均支持按多标签筛选。
- 保存或编辑模板时，用户可以选择已有标签，也可以自由输入新标签。
- “其它模板”支持按创建人或维护人筛选。
- 输入搜索条件后按 `Enter` 可以执行同一次查询。
- 模板达到 1000 条时，滚动和筛选仍能稳定使用。

### 9.4 排序

- 模板默认按历史累计点击或执行次数倒序排列。
- 执行次数变化后，重新加载列表时排序能反映最新热度。
- 点击“添加到我的模板”不计入模板执行点击量。

### 9.5 左侧常驻区

- 桌面端常用查询直接展示在分析页左侧，不再需要打开抽屉。
- 左侧区域支持点击展开、点击收缩和拖拽调整宽度。
- 拖拽宽度不会导致右侧查询输入、结果表格、图表或按钮文字重叠。
- 最近查询仍在抽屉中展示。

### 9.6 SQL 新增模板

- 具备权限的用户可以在查询模板页面新增模板并填写分类。
- 用户可以选择已有分类，也可以输入新分类。
- SQL 未通过安全校验时不能保存为可执行模板。
- SQL 预览和执行结果均遵循当前用户权限范围。

## 11. 已确认事项

以下事项已按当前产品口径确认，后续实现以此为准：

- “我的模板”采用个人副本模式，点击“添加到我的模板”会复制模板定义快照并生成当前用户自己的模板记录。
- 普通业务用户保存的模板默认全员可见，允许其他用户搜索、添加到“我的模板”和分享；执行仍按执行人当前实时权限收口。
- “有 SQL 能力的用户”只授予受控查询 SQL 编写能力，不等同于完整模板治理权限；如现有权限无法表达，新增最小权限点 `template.sql.write` 或等价中文权限。
- 自由问数保存优先复用后端实际执行的受控 SQL；如果只有执行计划，则在保存时由后端编译为参数化 SQL 并重新校验；无法安全编译时不允许保存。
- 分类首期支持“主分类 + 多标签”，其中多标签必须支持选择已有标签和自由输入新标签。
- 点击率统计按历史累计点击或执行次数排序，历史点击最多的模板优先展示。
