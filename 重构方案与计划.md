# CRM-Agent 重构方案与计划

> 日期：2026-06-23
> 用途：基于联软渠道 CRM 业务说明与两个标杆案例，明确本项目重构的保留边界、改造重点、新增能力和分期计划
> 对标案例：`广州办渠道下单汇总分析.html`、`全国代理商发展运营数据看板_20260522.html`
> 原则：稳定能力不动，可视化与分析链路是改造重点，看板能力是新增核心；联软 CRM 交互式写入仅预留边界，本期不实现

## 1. 项目现状诊断

### 1.1 已经能用的部分（保留）

| 能力 | 现状 | 判断依据 |
|------|------|----------|
| AI 网关与模型治理 | OpenAI 兼容 HTTP adapter 已通，支持 Profile 加密、健康检查、激活回滚 | 用户确认"对接 AI 没问题" |
| 企业微信机器人接入 | 验签、身份映射、消息编排、模板卡片、只读 HTML 报告链接已通 | 用户确认"对接企微能对的上" |
| 联软 CRM 标准 OpenAPI | `crm-standard-api` 模块已接通 partners/registrations/opportunities/quotes/orders，统计接口（business-overview/funnel/contribution/profile）全通 | 后端探查确认 |
| 认证与鉴权 | CRM 账号密码、企微扫码、会话、统一鉴权稳定 | AGENTS.md 明确"认证链路不动" |
| 治理审计与导出 | 权限中心、字段白名单、导出限制、审计留痕完整 | spec.md 与后端模块确认 |
| 后端结果数据结构 | `AnalysisResultRecord` 已含 metricCards/chartBlocks/tableBlocks/sections/keyFindings/insights | 结构化程度足够，无需重造 |
| 设计系统 DESIGN.md | Stripe 风格浅色工作台，色彩、排版、组件、状态语义完整 | 可直接承接看板视觉 |
| 技术栈 Vue3 + Element Plus | 前端基础组件与图标来源已统一 | 符合 AGENTS.md 约束 |

### 1.2 有但不够的部分（改造）

| 能力 | 当前问题 | 影响 |
|------|----------|------|
| 前端图表 | 全仓无 ECharts/Chart.js，`ResultChartView`、`BarRankingBlock`、`TrendChartBlock` 全部 CSS 自绘，仅支持折线、横向/纵向排名条形三种 | 无法渲染案例中的地图、分组柱、堆叠图、集中度图 |
| block 渲染器 | `AnalysisSectionCanvas` 把 sectionType 静态映射到 blockType，不可扩展 | 新增图表类型必须改映射代码，无法按需注册 |
| 结果表格 | `ResultTableView` 用 el-table 但未启用 sortable/filterable；`DetailTableBlock` 是纯静态 table | 案例要求的"可排序全渠道排名""可筛选合同明细"做不到 |
| 分析主链 | 强制走 OpenAPI Markdown 快照（`ensureFormalAnalysisRouteEnabled` 仅允许 OPENAPI 路线），SQLite/MySQL/受控 SQL 已停用 | 看板型问题实时性受限，统计接口仅用于刷新快照，问答阶段不实时调远端 |
| 查询模板 | 固定 SQL 硬编码在 `query-template.repository.ts` 的 `LEGACY_QUERY_TEMPLATE_PATCHES` | 无法在治理后台灵活维护，新增看板需改代码 |
| 企微 HTML 报告 | 已有指标卡 + 表格 + 地图省份弹窗，但图表能力同前端受限 | 企微打开的报告仍是简单表格，达不到案例看板效果 |

### 1.3 完全没有的部分（新增）

| 能力 | 案例要求 | 当前状态 |
|------|----------|----------|
| 看板型结果能力 | KPI 矩阵 + 集中度分析 + 多图联动 + 明细表 + 地图 | 后端结果结构有 block，但缺看板级组装与集中度/洞察生成器 |
| 看板模板配置中心 | "渠道下单汇总""全国代理商看板"可一键生成、复用 | 不存在，模板只有固定 SQL 形态 |
| 集中度与洞察生成 | TOP5/10/20 占比、单次合作渠道占比、自动洞察文案 | 后端有 `AnalysisRichReportService` 但未覆盖集中度计算 |
| 地图与分组图组件 | 31 省覆盖地图、4 大区分组柱对比 | 前端无 geo 组件、无分组柱 |
| 标杆看板沉淀 | 两个案例作为可复用看板模板 | 不存在 |

## 2. 保留 / 改造 / 新增 决策

### 2.1 保留（冻结，不重构）

1. **AI 网关与模型治理**：OpenAI 兼容 HTTP adapter、AI Profile 加密、健康检查、激活回滚。重构期间只调用，不改实现。
2. **企业微信机器人接入链路**：验签、身份映射、消息编排、模板卡片发送、会话上下文。结果内容会升级，但接入链路不动。
3. **联软 CRM 标准 OpenAPI 接入**：`crm-standard-api` 模块的 client 与 adapter，统计接口已全通，直接复用。
4. **认证与鉴权**：CRM 登录、企微扫码、会话、统一鉴权。AGENTS.md 明确不动，触及需单独立规格。
5. **治理审计与导出控制**：权限中心、字段白名单、导出限制、审计留痕。看板模板会接入治理，但治理引擎本身不动。
6. **后端结果数据结构**：`AnalysisResultRecord` 的 metricCards/chartBlocks/tableBlocks/sections/keyFindings/insights 字段保留，在其上扩展看板 block。
7. **设计系统 DESIGN.md**：Stripe 风格浅色工作台直接承接看板视觉，图表色板、状态语义、组件尺寸已定义。
8. **Vue3 + Element Plus 技术栈**：基础组件与图标来源不变。

### 2.2 改造（有基础，升级）

1. **前端图表升级 ECharts**：引入 ECharts 5，替换 CSS 自绘图表。新增地图、分组柱、堆叠柱、雷达等能力。
2. **block 渲染器可注册化**：把 `AnalysisSectionCanvas` 的静态 sectionType→blockType 映射改为可注册的渲染器表，新增 blockType 无需改核心代码。
3. **表格排序与筛选**：`ResultTableView` 启用 el-table 的 sortable + filterable；`DetailTableBlock` 重写为 el-table，支持列排序、搜索、筛选。
4. **分析链路放开统计直连**：对"看板型/统计型"意图，允许直连联软统计接口（不走 Markdown 快照），由新增的"看板结果组装器"输出 dashboard-grade 结果块。
5. **查询模板外置配置**：把硬编码 SQL 从 repository 外置到配置文件或 DB，治理后台可维护。
6. **企微 HTML 报告看板化**：企微只读 HTML 报告升级为 ECharts 看板，与 Web 端共享渲染组件。

### 2.3 新增（能力空白，从零建）

1. **看板型结果能力**：后端新增"看板结果组装器"，把多个统计接口结果组装为 KPI 矩阵 + 集中度 + 多图 + 明细表的看板结构。
2. **看板模板配置中心**：治理后台新增看板模板维护页，支持配置看板的 block 组合、数据源、筛选条件、适用角色。
3. **集中度与洞察生成器**：后端新增集中度计算（TOP5/10/20 占比、单次合作占比）和自动洞察文案生成。
4. **地图与分组图组件**：前端新增 `GeoMapBlock`（31 省覆盖地图）、`GroupedBarBlock`（大区对比）、`KpiMatrixBlock`（KPI 矩阵）、`ConcentrationBlock`（集中度 + 洞察）。
5. **标杆看板沉淀**：把两个案例沉淀为可复用看板模板，支持一键生成和条件改写。

## 3. 分期实施计划

### 第 0 期 · 基线确认（约 1 周）

目标：冻结稳定能力，确认数据口径，建立回归基线。

- 冻结"保留"清单，明确哪些模块重构期间不得改动。
- 实测联软 OpenAPI 统计接口可用性：`/analytics/business-overview`、`/analytics/funnel/registration-opportunity-order`、`/analytics/partners/contribution`、`/analytics/regions/contribution`、`/analytics/owners/contribution`。
- 确认两个案例的数据口径来源：渠道下单汇总的"合同对方不为空、排除评审和定稿"口径；全国代理商看板的"大区/团队/合作级别/证书"口径。
- 建立回归基线：现有问数、企微、合同审核、治理审计的主链路用例，明确重构后必须仍然通过。
- 产出：基线确认报告。

### 第 1 期 · 可视化升级（核心，约 2-3 周）

目标：Web 端具备富看板渲染能力，能跑通两个标杆案例的静态数据版本。

- 引入 ECharts 5 作为前端图表库（与 DESIGN.md 图表色板对齐）。
- 扩展 block 渲染器为可注册表，新增 blockType：
  - `geo-map`：31 省覆盖地图，支持已覆盖/未覆盖着色、省份点击弹窗。
  - `grouped-bar`：分组柱状图（大区对比、分年对比）。
  - `stacked-bar`：堆叠柱状图。
  - `kpi-matrix`：KPI 指标卡矩阵。
  - `concentration`：集中度分析 + 自动洞察文案。
  - `sortable-table`：可排序、可筛选的明细表。
- 重写 `ResultChartView`，废弃 CSS 自绘部分，统一走 ECharts。
- 重写 `DetailTableBlock` 为 el-table，启用 sortable + filterable。
- 沉淀两个标杆看板组件（先用案例静态数据验证渲染效果）：
  - `ChannelOrderSummaryDashboard`：渠道下单汇总分析看板。
  - `AgentDevelopmentDashboard`：全国代理商发展运营看板。
- 产出：Web 端富看板渲染能力 + 两个标杆看板静态演示。

### 第 2 期 · 分析链路放开（约 1-2 周）

目标：看板型问题能实时拉取联软数据并组装为看板。

- 后端新增"看板结果组装器"（`DashboardReportComposer`），识别看板型意图后直连联软统计接口，组装为看板 block 结构。
- 放开分析主链对"看板型/统计型"意图的统计接口直连权限（不走 Markdown 快照），保留对明细查询的白名单与权限约束。
- 查询模板外置：把 `LEGACY_QUERY_TEMPLATE_PATCHES` 硬编码 SQL 迁移到配置文件或 DB，治理后台可维护。
- 看板模板接入：两个标杆看板配置为看板模板，支持条件改写（如改区域、改时间范围）。
- 产出：实时看板问数闭环（Web 端输入"广州办渠道下单汇总"→ 返回完整看板）。

### 第 3 期 · 企微机器人能力升级（约 2 周）

目标：基于企微官方 9 个开发者文档对照，补齐 7 项关键能力差距，让企微端从"能对话"升级为"能看板、能反馈、能补问、能导出"。

> 参考文档（企微开发者中心）：
> - 概述 `path/101039`、接收消息 `path/100719`、接收事件 `path/101027`
> - 被动回复 `path/101031`、模板卡片类型 `path/101032`、加解密 `path/101033`
> - 主动回复 `path/101138`、长连接 `path/101463`、文档能力 `path/101468`

#### 3.1 现状对照（基于代码探查）

| 能力 | 企微官方支持 | 当前实现 | 差距 |
|------|-------------|---------|------|
| 接入模式 | Webhook 短连接 + WebSocket 长连接 | WebSocket 长连接为主 + HTTP 兼容入口 | 已具备，不动 |
| 接收消息类型 | text/image/mixed/voice/file/video/quote | 仅放行 text，其他被硬拦截 | 需放开 |
| 流式回复 | stream + stream_refresh（6 分钟超时） | stream 已实现，无 refresh，无超时兜底 | 需补 |
| 模板卡片 | 5 种类型 + update_template_card | 仅 text_notice，无更新 | 需补 |
| 流式+卡片组合 | stream_with_template_card | 未使用 | 需补 |
| 反馈事件 | feedback_event（准确/不准确+原因） | 完全未接入 | **关键缺口** |
| feedback.id | 回复时可设置触发反馈 | 未设置 | **关键缺口** |
| 主动回复 | response_url（1 小时有效）+ aibot_send_msg | 解析了 response_url 但未调用 | 需补 |
| 文档能力 | create_doc/edit_doc/smartsheet（MCP） | 未接入 | 可选增强 |

#### 3.2 升级内容（7 项）

**3.2.1 接入企微原生反馈事件（feedback_event）——学习闭环的关键输入**

这是本次对照最重要的发现：企微原生支持用户对机器人回复进行反馈，反馈类型包括"准确/不准确/取消"，不准确时还可附带原因（与问题无关/内容不完整/内容有错误/数据分析错误）和用户输入的反馈内容。

改造内容：
- 在所有流式回复和模板卡片回复中设置 `feedback.id`（值为本次查询的 queryId），触发企微原生反馈入口
- 新增 `WecomFeedbackEventService` 处理 `feedback_event` 回调：
  - type=1（准确）→ 记录正面反馈，进审计（`ANALYSIS_RESULT_FEEDBACK`，USEFUL）
  - type=2（不准确）+ inaccurate_reason_list → 记录负面反馈和原因，进审计（`ANALYSIS_RESULT_FEEDBACK`，NOT_USEFUL + 原因）
  - type=2 + content → 用户自述的口径问题，进审计（`ANALYSIS_RESULT_FEEDBACK`，CALIBRATION_ISSUE）
  - type=3（取消）→ 忽略
- 反馈数据直接对接学习闭环第 2 层（显式反馈），**无需在企微端自建反馈按钮**，Web 端仍保留自建反馈入口

影响：学习闭环设计的"企微端反馈"部分简化，直接复用企微原生能力，体验更好、用户更愿意点。

**3.2.2 看板在企微的展示策略：stream_with_template_card**

企微会话不适合展示完整 ECharts 看板（企微内置浏览器对 ECharts 兼容性有限、会话内嵌 HTML 渲染能力弱）。采用"流式过程 + 卡片摘要 + Web 链接"三段式：

- 流式阶段：用 stream 输出分析进度和关键结论文字（"正在查询广州办渠道下单数据…已获取 326 条…关键发现：TOP3 渠道占比 47%"）
- 卡片阶段：用 `stream_with_template_card` 在流式结束时附带 `text_notice` 模板卡片，展示 KPI 摘要（emphasis_content）+ 关键发现（horizontal_content_list）+ 跳转链接（jump_list 指向 Web 看板页）
- Web 链接：卡片 jump_list 的 URL 指向 `/analysis/results/:queryId`，用户点击在企微内置浏览器打开完整 ECharts 看板

约束：
- 卡片 horizontal_content_list 最多 6 行，只放最关键的 KPI
- jump_list 最多 3 个，默认放"查看完整看板"和"导出"
- 流式 content 支持 markdown，但单条不超过 20480 字节，超限按字节拆分

**3.2.3 流式刷新与 6 分钟超时兜底**

企微流式消息从用户发消息起最多等待 6 分钟，期间会不断推送 stream_refresh 回调。当前未处理 refresh，长查询有超时风险。

改造内容：
- 新增 `WecomStreamRefreshHandler` 处理 stream_refresh 回调：收到 refresh 时返回当前进度文字（"仍在执行，已完成数据查询，正在生成图表…"）
- 新增 6 分钟超时兜底：若分析任务在 5 分 30 秒仍未完成，主动结束流式（finish=true），返回卡片摘要+"任务仍在后台执行，完成后会主动推送结果"
- 后台任务完成后，用主动推送（aibot_send_msg 或 response_url）把最终结果卡片推给用户
- 已有的分阶段进度反馈（thinking→querying→reporting）保留，与 refresh 配合

**3.2.4 模板卡片升级：补齐 button_interaction 和 vote_interaction**

当前只用 text_notice，无法支撑补问选择和口径收敛投票。

新增：
- `button_interaction`（按钮交互卡片）：用于补问场景。当 AI 识别到歧义需要补问时，用按钮卡片让用户直接点击选择（如"您想看哪个区域的？[广州办] [北京办] [全国]"），替代纯文字补问，降低用户输入成本
- `vote_interaction`（投票选择卡片）：用于学习闭环的口径收敛。当检测到口径冲突时，向相关角色发起投票（如"'新增商机'应按创建时间还是更新时间？[创建时间] [更新时间]"），投票结果作为管理员裁定参考
- `update_template_card`（卡片更新）：用户点击 button_interaction 按钮后，用 update_template_card 把卡片更新为"已选择：广州办，正在查询…"，替代重新发一条消息

约束：
- button_list 最多 6 个按钮，补问选项超过 6 个时改用文字补问
- task_id 必须唯一，用 queryId + 时间戳生成
- 投票卡片 checkbox 选项最多 20 个，口径收敛场景足够

**3.2.5 非文本消息接收：放开 voice 和 image**

当前硬拦截非 text 消息，但企微用户在移动端常用语音提问。

改造内容：
- 放开 `voice` 消息接收：企微已将语音转为文本（voice.content），直接作为自然语言输入送入分析链路，无需自建 ASR
- 放开 `image` 消息接收（后置，低优先级）：用户可能发截图（如旧报表截图）让 AI 解读，需接入多模态模型，本期仅预留接口不实现
- `mixed`（图文混排）、`file`、`video`、`quote` 暂不放开，无明确业务场景

约束：
- voice 消息仅支持单聊，群聊仍只支持 text
- 放开后需验证 FR-014A 的上下文语义区分（语音转的文字不能被误判为"确认短句"）

**3.2.6 主动回复与异步推送**

当前 response_url 解析了但未调用，长查询超时后无法把结果推给用户。

改造内容：
- 启用 response_url 主动回复：超时兜底场景（3.2.3）中，后台任务完成后用 response_url 推送最终结果卡片（response_url 1 小时有效，覆盖分析任务典型耗时）
- response_url 失效后降级为 aibot_send_msg（长连接主动推送）：适用于超 1 小时的极端长任务
- 主动推送场景补充：日报定时推送、治理待办提醒（口径冲突待收敛、候选审核待办）

约束：
- response_url 每个只能用 1 次，任务完成后立即调用
- aibot_send_msg 需控制频率，避免骚扰用户

**3.2.7 企微文档导出（可选增强）**

企微 API 模式机器人支持创建文档和智能表格（create_doc/edit_doc_content/smartsheet_*），可把看板导出为企微智能表格，方便用户在企微内直接二次编辑。

改造内容：
- 看板导出新增"导出到企微文档"选项（与现有 Excel 导出并列）
- 调用 create_doc 创建智能表格 → smartsheet_add_fields 添加字段 → smartsheet_add_records 写入数据
- 返回文档链接给用户

约束：
- 需用户授权机器人文档使用权限（授权有效期 7 天）
- 机器人只能编辑自己创建的文档
- 本项为可选增强，优先级低于 3.2.1-3.2.6，视工期决定是否纳入本期

#### 3.3 企微端用户旅程（升级后）

```
用户在企微单聊发语音："本季度各区域新增商机金额排名"
  → 机器人收到 voice 消息，voice.content = "本季度各区域新增商机金额排名"
  → AI 识别为看板型问题，进入流式回复
  → stream 阶段："正在查询本季度各区域商机数据…已获取 156 条…"
  → 若 5 分 30 秒未完成：finish=true，卡片提示"任务后台执行中，完成后推送"
  → 后台完成后用 response_url 主动推送结果卡片
  → stream_with_template_card 结束：卡片展示 KPI 摘要 + "查看完整看板"跳转链接
  → 卡片带 feedback.id = queryId
  → 用户点"不准确" + 选择"数据分析错误"
  → feedback_event 回调 → 写入审计 → 触发学习闭环沉淀
```

预留约束：

- 涉及报价、订单等复杂对象，以联软系统现有计价与状态流转逻辑为准，AI-agent 不复刻生产计价。

## 5. 风险与不变量

### 5.1 不能破坏的主线

1. 认证、登录、会话、统一鉴权链路不动。
2. AI 网关与模型治理引擎不动，只调用不改实现。
3. 联软 OpenAPI client 的只读边界不动，不绕过权限。
4. 治理审计与导出控制引擎不动，看板接入治理但不改治理逻辑。
5. DESIGN.md 设计系统为视觉最高入口，ECharts 配色必须对齐图表色板。
6. 现有问数、企微、合同审核、治理审计的主链路回归用例必须持续通过。

### 5.2 容易出问题的点

1. ECharts 引入后包体积变大，需按需引入模块，避免拖慢首屏。
2. 看板型意图识别要准，不能把普通问数也走看板组装器，导致响应变慢。
3. 统计接口直连要遵守联软 token 绑定的权限范围，不能把局部数据写成全局结论。
4. 看板模板外置后，旧固定 SQL 模板要兼容并存，不能突然失效。
5. 企微 HTML 报告升级 ECharts 后，要验证企微内置浏览器兼容性。
6. 企微流式 6 分钟超时是硬限制，必须有兜底，不能让用户等到超时后无响应。
7. 企微 feedback_event 只支持回复空包（不能同时回复新消息），处理逻辑要兼容这一约束。
8. 企微模板卡片的 task_id 不可重复，用 queryId + 时间戳生成，避免冲突。
9. 放开语音消息后，语音转文本可能包含口语化表达，需验证 FR-014A 上下文语义区分不被误判。
10. response_url 1 小时有效且只能用 1 次，长任务要先判断是否过期，过期降级为 aibot_send_msg。
11. 企微文档能力需用户授权（7 天有效），授权过期需引导用户重新授权，不能静默失败。
