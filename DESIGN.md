# CRM 智能分析系统设计系统

## 设计定位

本文件是 CRM 智能分析系统的根级设计系统入口，面向后续 UI 设计、前端实现、代码生成、设计评审和代理协作使用。后续新增或重构 Web 界面时，视觉方向、页面结构、组件规则、状态反馈和前端实施口径优先遵循本文件；功能边界、权限规则、接口字段和验收口径仍以 `specs/001-crm-intelligent-analytics/` 下的主规格为准。

本系统不再采用“深色 AI 指挥舱”作为默认视觉叙事，而是重构为一套面向经营分析、治理审计和 AI 辅助决策的企业级工作台。整体体验应当像 Stripe 一样清晰、可信、克制、流畅，但又保留 CRM 智能分析系统自身的业务判断节奏和风险治理表达。

## 主参考方向

本设计系统以 Stripe 风格作为主参考，不直接复制其品牌资产、专有插画、原始文案或完整产品外观。参考吸收分为两个层面：

- `Stripe 官网`：用于首页气质、留白、浅色基底、柔和渐变、标题层级和品牌精致感。
- `Stripe Dashboard / Docs`：用于工作台结构、信息分区、表格与表单秩序、状态反馈和治理后台密度控制。

本系统最终目标不是做成 Stripe 的复制品，而是借用其“清晰、可信、克制”的方法，构建更适合 CRM 经营分析、合同审核和治理审计的一套企业级视觉语言。

同时需要吸收成熟企业安全、终端治理、策略管控类系统的优点：清晰的左侧导航、紧凑的页头、可扫描的表格、明确的等级标签、可识别的模块图标、适度的业务插画和成熟的抽屉表单。参考这些系统时只吸收信息组织与视觉完成度，不复制其品牌、配色或过度装饰。

## 与旧原型的关系

- `docs/prototype-ui/crm-intelligent-analytics-ui.pen` 是旧版原型，仅用于理解历史页面范围，不再作为新视觉风格、布局结构、卡片形态或尺寸的强约束。
- `docs/prototype-ui/UI规范.md` 是旧版白底蓝灰后台规范，不再作为新界面的视觉基线。
- 新版允许围绕首页、智能分析工作台、结果页、合同审核页和治理审计页重新组织信息架构，但不得突破一期主规格定义的功能边界、权限边界、审计留痕和验收场景。
- 如果后续需要产出新的可读尺寸原型，必须生成新的 `.pen` 文件，并与本设计系统保持一致，不能继续在旧原型上做局部修补。

## 核心体验原则

- **经营工作台优先**：页面首先是一套经营分析与治理工作台，其次才是 AI 对话入口。AI 能力必须融入工作台，而不是统治整个页面结构。
- **结论与边界同时可见**：摘要、图表、表格、风险、权限范围、时间范围、数据更新时间和审核依据必须共同组成主信息结构，不能把边界说明缩成脚注。
- **浅色主导，重点强化**：页面默认使用浅色基底与白色内容面板，深色只用于代码片段、运行态强调、局部对比或高聚焦模块，不再默认整页深色。
- **可信比炫技更重要**：视觉应优先建立专业感和判断确定性，而不是通过发光、霓虹和大面积动画制造“智能感”。
- **信息密度可调**：管理层先看结论，分析人员继续下钻，治理人员需要检索与复核。不同页面密度可以不同，但必须共享同一视觉语法。
- **状态反馈必须可执行**：加载、补问、拒答、降级、阻断、失败、无权限、空状态都要告诉用户当前发生了什么，以及下一步能做什么。

## 品牌气质与视觉语言

### 气质关键词

- 清晰
- 可信
- 克制
- 流动感
- 面向经营决策

### 视觉方向

- 使用大面积浅色背景、白色卡片、细边框、轻阴影和明确留白建立专业感。
- 使用少量柔和渐变作为品牌层、入口层和重点强调层，不把渐变当成整页底噪。
- 通过标题层级、内容分区和对齐秩序建立高级感，而不是依赖高对比黑底或高饱和发光。
- 将“AI”表达为一项可信的增强能力，而不是具有强压迫感的舞台主视觉。
- 通过业务图形、模块图标、等级标签、数据可视化和空状态插画建立产品识别，不允许只依赖白色卡片、灰色文字和单一品牌色完成页面。
- 在关键首屏、空状态、结果态和治理态中保留适量视觉锚点，例如状态胶囊、轻量图形底纹、指标趋势、进度环、风险等级图标和模块徽标，避免页面显得粗糙或像未完成的后台模板。

### 视觉丰富度底线

后续新增或重构页面时，必须同时满足以下底线：

- 每个一级页面至少有一处清晰的业务视觉锚点，例如模块徽标、页面状态条、核心指标图形、业务插画或数据可视化，不允许整页只由表单、表格和普通白卡组成。
- 首屏必须能看出当前模块属于“经营分析、合同审核、治理审计、权限配置、模板管理、连接策略、AI 配置”中的哪一类，而不是仅靠导航高亮或页标题判断。
- 空状态、加载态、失败态、无权限态和降级态必须有状态图标或轻量插画，不能只有一行文字。
- 表格行、风险项、模板项、合同任务和审计事件必须使用至少一种非文字识别方式，例如状态点、等级色、事件图标、对象类型图标、平台图标或风险徽标。
- 页面中允许出现柔和图形背景、细线几何纹理、圆环进度、迷你趋势线和等级图标，但必须服务于业务识别和状态理解，不得成为无意义装饰。

### 禁止事项

- 禁止默认整页纯黑或近黑背景。
- 禁止霓虹式高饱和光效、长时间脉冲或大面积发光描边。
- 禁止把输入框做成营销页主视觉后，牺牲状态说明、权限摘要和结果阅读效率。
- 禁止在同一页面混用过多视觉语法，造成“多品牌拼接感”。
- 禁止为了“丰富”而无规则堆叠彩色卡片、图标和渐变；所有新增视觉元素必须能回答“它在帮助用户识别什么业务对象、状态或优先级”。
- 禁止继续使用旧版白底蓝灰后台作为视觉上限；白色卡片可以保留，但必须通过层级、图形、状态和数据表达补足完成度。

| 一级卡片 | `#FFFFFF` | 默认内容面板底色 |
| 次级卡片 | `#F9FBFF` | 用于次级信息、筛选区、嵌套面板 |

| 分隔线 | `#E6EBF1` | 边框、分隔线、输入框边界 |

### 品牌与状态色

| 用途 | 色值 | 说明 |
| --- | --- | --- |
| 品牌主色 | `#635BFF` | 主按钮、关键入口、主操作强调 |
| 品牌辅助蓝 | `#3A6FF7` | 链接、选中态、信息型强调 |
| 品牌浅蓝 | `#8BC5FF` | 柔和图形、渐变辅助、高亮底纹 |
| 成功 | `#0A7F5A` | 成功状态、可用、已完成 |
| 警告 | `#B76E00` | 待确认、限制、降级提醒 |
| 风险 | `#C23D4B` | 高风险、阻断、失败 |
| 信息 | `#0A5BD8` | 帮助提示、信息反馈、说明标签 |

### 业务辅助色

业务辅助色用于模块识别、图表序列、等级标签和局部图形，不替代品牌主色。使用时必须遵循“主色克制、辅助色分工明确”的原则。

| 业务语义 | 色值 | 推荐用途 |
| --- | --- | --- |
| AI 分析紫 | `#635BFF` | 智能分析、AI 配置、主操作、生成中状态 |
| 经营蓝 | `#2563EB` | 经营报表、趋势图、信息提示、链接 |
| 治理青 | `#0E9F8A` | 权限治理、连接策略、健康状态、通过态 |
| 合同绿 | `#16A34A` | 合同审核通过、任务完成、有效状态 |
| 审计靛蓝 | `#4F46E5` | 审计检索、事件追踪、系统证据链 |
| 预警琥珀 | `#D97706` | 待确认、口径提醒、阈值接近风险 |
| 高风险红 | `#C23D4B` | 阻断、高风险、审核不通过、失败 |
| 中性灰蓝 | `#64748B` | 停用、归档、普通元信息、次级标签 |

业务辅助色要求：

- 同一页面默认最多使用 `1` 个模块主辅助色、`1` 个品牌主色和必要状态色；不得把所有业务辅助色同时铺满页面。
- 左侧导航、页头徽标、模块空状态、图表主序列和模块标签应共享同一个模块辅助色，形成稳定识别。
- 风险红只能用于真实风险和失败；预警琥珀用于“需要注意但仍可继续”的状态，二者不能混用。
- 治理、审计、合同和经营页面可以使用不同辅助色建立模块差异，但整体背景、文字、边框、圆角和阴影必须保持同一设计系统。

### 图表色板

图表不得长期只使用品牌渐变。后续图表组件默认使用以下序列，并按业务语义覆盖：

| 序号 | 色值 | 用途建议 |
| --- | --- | --- |
| 1 | `#635BFF` | 主指标、AI 结果、核心系列 |
| 2 | `#2563EB` | 次指标、趋势对比 |
| 3 | `#0E9F8A` | 健康、增长、正向变化 |
| 4 | `#D97706` | 预警、待确认、临界值 |
| 5 | `#C23D4B` | 高风险、阻断、负向变化 |
| 6 | `#7C3AED` | 模型、规则、智能推荐 |
| 7 | `#0891B2` | 来源、渠道、连接 |
| 8 | `#64748B` | 其他、中性、未分类 |

图表使用规则：

- 趋势图优先使用线条、区域浅填充和异常点标记，不使用粗重渐变柱堆满画面。
- 排名、排行、Top 类图表优先使用横向榜单条形，左侧保留排名序号，右侧保留数值和单位。
- 占比类图表可以使用圆环或堆叠条，但必须同时给出文字摘要，避免只靠颜色辨识。
- 风险等级图表必须按 `阻断 / 高风险 / 待复核 / 提示 / 正常` 的语义顺序排序，不按随机接口返回顺序展示。
- 图表色彩必须与标签、摘要卡和表格状态保持一致；同一业务状态在不同组件中不能换色。

### 语义状态层

页面状态不得再只靠 `success / warning / danger / info` 四类粗粒度颜色兜底，后续 Web 工作台、治理后台、审计中心和合同审核统一使用以下语义状态层：

| 语义状态 | 背景 | 描边 | 文字 / 图标 | 典型场景 |
| --- | --- | --- | --- | --- |
| 在线 | `#ECFAF4` | `rgba(10, 127, 90, 0.18)` | `#0A7F5A` | 服务在线、权限实时生效、健康可用 |
| 处理中 | `#EDF4FF` | `rgba(10, 91, 216, 0.18)` | `#0A5BD8` | 加载中、排队中、生成中、刷新中 |
| 成功 | `#ECFAF4` | `rgba(10, 127, 90, 0.18)` | `#0A7F5A` | 已完成、已返回、可导出、审核通过 |
| 提示 | `#EDF4FF` | `rgba(10, 91, 216, 0.18)` | `#0A5BD8` | 普通说明、范围提示、数据说明、信息型提示 |
| 警告 | `#FFF6E7` | `rgba(183, 110, 0, 0.2)` | `#B76E00` | 待确认、限制说明、口径提醒 |
| 降级 | `#FFF1E0` | `rgba(155, 94, 0, 0.22)` | `#9B5E00` | 规则快审、fallback、能力降级、补充审核失败保留结果 |
| 阻断 / 失败 | `#FFF0F2` | `rgba(194, 61, 75, 0.22)` | `#C23D4B` | 阻断、失败、高风险、禁止继续 |
| 离线 / 停用 | `#EEF2F8` | `rgba(107, 124, 147, 0.18)` | `#5C6B80` | 离线、停用、归档、不可用 |
| 中性 | `#F4F7FB` | `rgba(213, 222, 234, 0.9)` | `#425466` | 用户标签、对象属性、ID、普通元信息 |

实施要求：

- 状态标签、状态卡、提示块、页头 chip、摘要徽标和按钮内状态图标，必须先声明语义状态，再映射到统一 token；禁止在页面局部直接写死色值。
- “在线”和“成功”可以共享绿色色相，但在线更偏系统健康，成功更偏动作完成；文案和图标必须同时区分，不能只靠同一颗绿点通吃。
- “降级”必须与普通“警告”分开，明确表达“系统已给出兜底结果，但当前不是完整主链能力”。
- “阻断 / 失败”只用于真正不能继续或高风险失败，不得把普通提醒做成阻断红。

### 渐变规则

- 品牌渐变：`linear-gradient(135deg, #635BFF 0%, #7A73FF 45%, #8BC5FF 100%)`
- 柔和背景渐变：`linear-gradient(180deg, rgba(99,91,255,0.08) 0%, rgba(139,197,255,0.06) 100%)`
- 风险提醒渐变仅用于重点横幅或摘要条，不允许成为审计页或合同页默认背景。

### 使用原则

- 颜色首先服务于结构和状态，其次才服务于品牌记忆。
- 品牌色只用于关键入口、选中态、关键图表序列和重点说明，不应在同一页面反复大面积出现。
- 风险红只用于真正高风险、阻断和失败，不得把一般提醒做成红色。
- 深色不再作为默认基底，只允许出现在局部高对比容器、代码区域或需要强聚焦的运行态模块中。
- 如果页面被评价为“颜色单一”，优先检查业务辅助色、状态色、图表序列和模块视觉锚点是否缺失；不得简单把背景、按钮和卡片全部改成彩色。
- 如果页面被评价为“颜色杂乱”，优先收敛到模块辅助色 + 品牌主色 + 状态色三类，不允许继续新增局部色值。

## 字体与排版

### 字体建议

- 中文优先使用 `HarmonyOS Sans SC`、`Source Han Sans SC`、`PingFang SC`，缺失时回退到系统无衬线字体。
- 英文与数字优先使用中性现代无衬线字体，可回退到 `Aptos`、`Inter`、`system-ui`。
- 数据数字建议启用等宽数字特性，确保金额、比例、排名和时间易于纵向比较。

### 排版原则

- 标题追求干净、准确和层次清晰，不再使用“指挥舱大标题”制造压迫感。
- 页面标题、区块标题、摘要文字、表格正文之间必须有稳定层级，不允许相邻层级过于接近。
- 正文、表格、说明和筛选项应优先保证可读性，不为追求“高级感”牺牲信息识别效率。
- 长文本场景，如合同正文、审核说明、审计详情，应以阅读舒适为第一目标。
- 字体层级必须拉开“页面标题、区块标题、关键数字、正文、辅助说明、标签”之间的视觉距离，不能所有文字都落在相近字号和字重。
- 数字、金额、比例、排名、耗时、日期等可比较内容必须使用等宽数字特性，必要时右对齐或按小数位对齐。
- 表格和列表中的文字间距必须优先服务扫描效率，行高不能过密导致字段粘连，也不能过松导致首屏信息不足。

### 推荐层级

后续内容页必须优先使用以下统一层级，不得在不同页面随意放大或缩小标题字号：

| 层级 | 尺寸 | 字重 | 用途 |
| --- | --- | --- | --- |
| 品牌/营销主标题 | 34px / 42px | 700 | 登录辅助区、品牌导语、少数高聚焦入口 |
| 页面标题 | 22px / 30px | 700 | 工作台、结果页、治理页、合同审核页第一主标题 |
| 一级区块标题 | 17px / 24px | 650 | 问数区、结果区、治理区、审计区、详情主卡片标题 |
| 二级区块标题 | 16px / 22px | 650 | 侧栏摘要、表格标题、表单区标题、说明块标题 |
| 强调正文 | 15px / 24px | 500 | 结论摘要、关键说明、重点提示 |
| 正文 | 14px / 22px | 400 | 普通说明、字段信息、表格正文、表单文案 |
| 辅助文字 | 13px / 20px | 400 | 口径、时间、权限、帮助说明 |
| 标签文字 | 12px / 16px | 600 | 状态标签、来源标签、风险标签 |

实施约束：

- 同一页面的第一主标题只能使用一个“页面标题”层级，不得同时出现多个 `22px` 以上标题互相竞争。
- 大多数内容页的卡片标题应落在 `16px` 到 `17px` 区间，不得随意跳到 `20px`、`22px` 或更大。
- 描述文案默认使用 `14px / 22px`，避免出现同屏正文有的 13px、有的 16px 且无业务原因的情况。
- 数字型指标可大于正文，但其标题、单位和辅助说明必须回到统一层级，不得整页到处出现超大字号。
- 核心指标数字建议使用 `28px` 到 `32px`，二级指标数字建议使用 `22px` 到 `24px`，表格内数字保持 `14px` 到 `15px`。
- 页面一级标题与页头状态标签之间应保持 `12px` 到 `16px` 水平或纵向间距，不得贴得过近。
- 卡片标题与正文默认间距 `8px` 到 `12px`；区块标题与内容默认间距 `12px` 到 `16px`；表单字段组之间默认间距 `16px` 到 `20px`。

## 图标、图形与可视化语言

### 图标体系

本项目统一使用 `@element-plus/icons-vue` 作为基础图标来源。若基础图标无法准确表达业务语义，可以通过项目级封装组合“图标底板 + 语义色 + 状态点”增强识别，不得临时使用 emoji、首字母、无语义字符或来源不明的图片。

图标使用分层：

- **导航图标**：用于左侧导航和模块入口，必须覆盖智能分析、经营报表、智能合同、权限中心、查询模板、连接策略、AI 配置、审计中心。
- **操作图标**：用于搜索、筛选、刷新、上传、下载、导出、返回、关闭、展开收起、删除、编辑、查看详情等高频动作。
- **状态图标**：用于成功、失败、警告、处理中、降级、无权限、离线、阻断等状态。
- **业务对象图标**：用于客户、负责人、合同、模板、策略、连接、审计事件、数据表、图表、企业微信来源等对象。
- **等级图标**：用于合同风险、数据敏感等级、审计风险等级和治理事件优先级。

图标尺寸建议：

| 场景 | 尺寸 | 说明 |
| --- | --- | --- |
| 导航图标 | `18px` 到 `20px` | 收起态仍需清晰可辨 |
| 按钮图标 | `14px` 到 `16px` | 与文字 `6px` 到 `8px` 间距 |
| 状态标签图标 | `14px` | 与标签文字垂直居中 |
| 空状态 / 模块图标 | `48px` 到 `72px` | 可放入浅色圆形或圆角底板 |
| 页面视觉锚点图标 | `72px` 到 `120px` | 只用于首页、空状态或关键摘要 |

### 业务图形与插画

业务图形必须轻量、低饱和、与模块语义绑定，不得使用泛化 AI 插画或与业务无关的装饰图。

推荐图形类型：

- **智能分析**：问数输入、数据流、趋势线、AI 生成状态、结果引用链。
- **经营报表**：趋势折线、指标矩阵、漏斗、排行、经营异常点。
- **合同审核**：文档、批注、风险标记、审核通过 / 阻断状态。
- **权限治理**：用户、角色、部门、权限矩阵、策略开关。
- **查询模板**：模板库、分类树、收藏、最近使用、等级标签。
- **连接策略**：数据源、连接状态、同步链路、失败重试。
- **AI 配置**：模型、Profile、上下文策略、健康检查。
- **审计中心**：事件时间线、证据链、风险等级、复核流转。

插画使用规则：

- 空状态插画必须配合一句原因说明和一个下一步动作，不允许只展示图形。
- 插画色彩应使用当前模块辅助色的浅色系，并搭配中性色，不得使用高饱和大面积填充。
- 插画不应替代真实数据；有数据时优先展示数据，有空状态或引导态时才使用插画。
- 同一模块的空状态、加载态和失败态应共享同一视觉家族，避免每个状态都像不同产品。

### 数据可视化语法

数据可视化必须比普通卡片更有信息含量，而不是为了装饰增加图形。

- 指标卡至少包含：指标名、核心值、单位或口径、变化方向或更新时间中的两项。
- 经营概览卡建议增加迷你趋势线、同比 / 环比标签、异常标记或数据新鲜度。
- 风险摘要卡必须展示风险等级、影响对象、原因摘要和建议动作，不得只展示红色数字。
- 排名卡必须保留排名序号、对象名称、关键数值和相对条形，方便横向比较。
- 进度类信息可以使用圆环或进度条，但必须同时展示分子、分母和业务含义。
- 图表卡必须提供标题、时间范围、口径说明和空状态；不能只放图形本体。

### 前端落地组件与 token 映射

当前前端实现已沉淀以下稳定出口，后续页面改造应优先复用，不得在页面局部重新写一套临时视觉语法：

| 类型 | 前端出口 | 使用范围 |
| --- | --- | --- |
| 模块视觉配置 | `frontend/src/ui/visual-language.ts` 的 `businessModuleVisuals` | 页面徽标、空状态、状态条、模块色 |
| 图表色板 | `chartColorTokens`、`resolveChartColor()` | 分析图表、经营报表趋势、排行、指标卡 |
| 模块徽标 | `BusinessVisualAnchor.vue` | 智能分析、经营报表、模板治理、权限、连接、AI 配置、合同、审计首屏 |
| 业务空状态 | `BusinessEmptyState.vue` | 未提问、无数据、无权限、加载中、失败或降级入口 |
| 对象图标标签 | `ObjectIconLabel.vue` | 模板、合同、角色、审计事件、数据表、负责人等主列 |
| 状态摘要条 | `StatusSummaryStrip.vue` | 权限边界、导出能力、连接阈值、审核状态、审计摘要 |
| 风险徽标 | `RiskLevelBadge.vue` | 合同风险项、审计风险、阻断 / 高风险 / 中风险 / 低风险 |
| 图表说明头 | `ChartLegendHeader.vue` | 经营报表趋势、排行和后续复杂图表说明 |

落地要求：

- 页面级视觉锚点默认使用 `BusinessVisualAnchor`，并绑定 `analysis / management / template / permission / connection / aiProfile / contract / audit` 之一。
- 自绘图表和指标卡必须通过 `resolveChartColor()` 读取图表色板，避免继续写死单一蓝色或紫色渐变。
- 表格主列优先使用 `ObjectIconLabel`，状态列优先使用 `badge + status-tone--*` 或 `RiskLevelBadge`，避免退回纯文字表格。
- 空状态和加载态优先使用 `BusinessEmptyState` 或统一 `.empty-state / .loading-state` 图形底纹，文案必须包含“发生了什么”和“下一步怎么做”。
- 只展示图标的按钮必须有 `aria-label` 或 `title`，并保留键盘焦点态；带文字按钮必须使用语义一致的 Element Plus 图标。

## 页面骨架与滚动规则

### 内容页默认骨架

除登录页外，桌面端内容页默认统一采用 `左侧导航 + 顶部吸顶栏 + 主内容区` 的强统一结构。右侧抽屉仅在明确存在详情下钻、临时补充信息或局部解释时按需出现，不再将最右侧常驻信息区作为默认骨架的一部分。

桌面端默认参数如下：

| 区域 | 默认尺寸 | 定位要求 | 滚动要求 |
| --- | --- | --- | --- |
| 左侧导航 | `248px`，复杂导航最多可放宽到 `264px` | 固定在应用壳层左侧 | 导航区可独立纵向滚动，不得出现横向滚动 |
| 顶部栏 | 高度 `72px` | 吸顶，始终位于主内容区顶部 | 顶部栏本身不滚动 |
| 主内容列 | `minmax(0, 1fr)` | 位于页面中心主轨道 | 主内容列是桌面端默认唯一主要纵向滚动区 |
| 右侧抽屉 | 宽度 `400px`，复杂详情最大 `440px` | 固定在右侧覆盖层 | 抽屉内部滚动，打开时页面背景锁定 |

### 骨架强约束

- 桌面端内容页默认高度按 `100vh` 组织，应用壳层负责布局与滚动分发。
- 桌面端 `body` 默认不承担主要纵向滚动；滚动应委托给主内容区、左侧导航区、右侧信息区或明确声明的局部容器。
- 顶部栏必须吸顶，吸顶后仍要保留清晰分隔线或背景层，不允许内容滚动时与正文混在一起。
- 左侧导航必须常驻，但允许在桌面端收起为图标轨道，不允许在桌面端完全隐藏成不可感知入口。
- 右侧补充信息默认应内聚到主内容列的次级卡片、折叠区或按需抽屉中，不鼓励为了放“提示”“边界说明”“统计摘要”而长期占用最右侧常驻栏位。
- 主内容区必须承担主任务、主结果、主说明和主要辅助信息；只有需要上下文切换或临时下钻时，才允许打开右侧抽屉。

### 左侧导航收起规则

- 桌面端左侧导航允许在 `248px` 常规宽度与 `76px` 图标轨道宽度之间切换。
- 收起后的导航必须仍可见主图标、当前激活项和展开入口，不得完全退出屏幕。
- 展开 / 收起必须使用过渡动画，建议时长 `180ms` 到 `220ms`，缓动建议使用 `cubic-bezier(0.2, 0.8, 0.2, 1)`。
- 收起后文案可隐藏，但悬浮或聚焦时必须能通过 tooltip 或浮层看到完整导航名称。
- 收起 / 展开按钮建议放在品牌区或顶部栏品牌信息右侧，默认仅保留图标按钮，不应再占用额外文字标签空间。

### 滚动条规则

- 桌面端默认禁止整页出现多个并列大滚动条。以下区域可以独立出现纵向滚动条：
  - 左侧导航
  - 主内容区
  - 常驻右侧信息区
  - 抽屉内部
  - 明确声明为“表格容器”“长列表容器”“正文预览容器”的局部区域
- 默认禁止页面级横向滚动。横向滚动只允许出现在以下明确容器中：
  - 宽表格容器
  - 代码块容器
  - 对比矩阵容器
- 表格内部横向滚动不应带动整个页面横向位移。
- 打开弹窗、抽屉、全屏预览时，背景主内容区应锁定，避免双层滚动导致焦点丢失。

### 滚动条视觉要求

- 滚动条宽度建议为 `10px`，最小不可低于 `8px`。
- 滚动条轨道保持透明或使用极浅底色，不得使用高饱和品牌色。
- 滚动条滑块默认色建议使用 `#C7D2E3`，悬浮时使用 `#B3C0D4`，按下时使用 `#9FAFCA`。
- 滚动条圆角建议使用 `999px`，确保轻量、安静，不抢正文注意力。
- 局部滚动区必须通过阴影、遮罩或标题固定提示用户“这里可以继续滚动”，不能让用户自行猜测。

### 分析结果展示规则

- 常用模板、自由查询、结果详情必须共用同一套分析结果展示判断，不得为单个模板写局部特例。
- 表格方向优先按数据基数决定：维度值或结果行较多时，维度必须纵向作为行展示，指标横向作为列展示；只有维度值较少、与图表配套对照且估算宽度可控时，才允许转置为“指标 × 少量维度值”的横向对照表。
- 表格不得撑开页面产生页面级横向滚动。宽表只允许在 `.table-wrap` 这类明确表格容器内横向滚动，并必须保留可见滚动条、边界或阴影提示。
- 排名、排行、Top 类图表，以及维度项较多的柱状图，必须按从上到下的榜单条形展示；横向卡片条带只用于少量时间点或少量类别的紧凑对比。
- 图表与表格同时存在时，区块可上下全宽排列，但是否转置表格必须独立判断；不能因为“有图表”就强制把多区域、多负责人或多客户结果横向铺成列。
- 移动端和窄屏下，分析结果默认一列排布；图表榜单和表格滚动容器必须保持键盘可聚焦、文本不重叠、数字可比较。

### 登录页专项规范

登录页默认以当前系统登录页 `frontend/src/pages/auth/LoginPage.vue` 为唯一布局与文案基线，只允许在现有信息结构上升级视觉、层级、状态反馈和响应式表现，不允许重新发明登录页文案体系或改造成全新的营销式登录页。

桌面端登录页要求如下：

| 区域 | 默认宽度占比 | 内容要求 |
| --- | --- | --- |
| 左侧辅助区 | `44%` 到 `48%` | 沿用当前系统品牌口径，并补充精简的系统能力说明，不得偏离现有系统职责 |
| 登录面板区 | 登录卡最大宽度 `460px` | 账号登录、企业微信登录切换、表单、扫码区、管理员联系方式 |

登录页文案与结构约束：

- 系统名称必须沿用当前文案：`CRM 智能业务平台`。
- 系统副标题必须沿用当前文案：`统一客户经营入口`。
- 登录卡标题必须沿用当前文案：`登录 CRM 智能业务平台`。
- 账号输入占位必须沿用当前文案：`请输入 CRM 账号或手机号`。
- 密码输入占位必须沿用当前文案：`请输入登录密码`。
- 企业微信扫码提示必须沿用当前文案：`请使用已绑定 CRM 的企业微信扫码`。
- 管理员信息模块必须保留当前管理员名单展示口径。
- 登录页可以在左侧辅助区增加精简能力说明，但不得新增与当前系统职责无关的大段说明。
- 账号登录与企业微信登录必须是同层级切换，不得把二维码入口藏在二级说明里。
- 登录帮助、管理员联系方式、绑定异常说明必须固定存在于登录卡内或卡片底部，不能依赖悬浮后才出现。
- 桌面端登录页默认不出现整页纵向滚动；窄屏或移动端允许转为纵向堆叠并恢复页面级滚动。

登录页禁止事项：

- 禁止把登录页改造成独立营销页。
- 禁止新增与现有登录页不一致的系统名、副标题、登录标题或功能宣传文案。
- 禁止左侧放过多虚构业务数字、大面积装饰性图表或过多能力卡，掩盖登录动作本身。
- 禁止把“联系管理员”“绑定异常如何处理”“企业微信未开通怎么办”藏在 hover-only 交互中。
- 禁止登录按钮、扫码区、帮助信息三者层级混乱，导致用户需要反复寻找入口。

登录页视觉强化要求：

- 登录页允许在左侧背景区域增加一块品牌辅助区，但必须沿用现有系统名称与定位，不得改造成独立营销页。
- 左侧辅助区文案必须精简，并符合系统整体功能；推荐只保留 `经营问数`、`合同审核`、`治理审计` 三类能力提示，每条说明不超过 `16` 个字。
- 左侧辅助区应使用低饱和浅蓝紫渐变、半透明底或轻描边，不得堆叠多张纯白卡片，也不得使用突兀高饱和背景。
- 左侧辅助区只用于增强品牌与系统能力识别；右侧登录区仍以现有登录流程为主，不应增加额外功能宣传模块。
- 登录页左右区块在桌面端必须保持稳定比例与舒适间距：
  - 左侧辅助区推荐最大宽度 `440px` 到 `520px`
  - 登录面板区推荐宽度 `440px` 到 `474px`
  - 两区之间推荐水平间距 `56px` 到 `96px`
  - 不允许左侧块过小像装饰挂件，也不允许两区间距大到失去整体性
- 登录页左右区块不得表现为两个彼此无关的独立漂浮卡片，必须共享同一视觉舞台：
  - 左右两区应落在同一背景画布或统一构图区域中
  - 两区需有共同的对齐基线、共同的背景气氛或柔和的视觉桥接
  - 左侧区块必须明显弱于右侧登录面板，不能与右侧争夺同等视觉权重
  - 左侧区块不建议使用与右侧完全同等级的厚重阴影和完整卡片边框

### 间距与容器基线

- 桌面端内容页主内容区左右内边距默认使用 `32px`，上下内边距默认使用 `24px`。
- 页面一级区块之间的默认间距使用 `24px`，同一区块内部子模块默认间距使用 `16px`。
- 页面头部摘要区与正文区之间的间距不得小于 `20px`。
- 卡片默认圆角使用 `16px`；高层级品牌卡可放宽到 `20px`；抽屉、弹窗、局部容器圆角不低于 `14px`。
- 卡片与面板不允许无限放大阴影，默认使用轻阴影和细边框建立层级。
- 所有交互控件默认统一使用 `12px` 圆角，包括主按钮、次按钮、图标按钮、输入框、下拉框、搜索框、文本域、筛选控件和登录切换标签；不得在同一页面混用 `10px`、`14px`、`20px` 等多个控件圆角。

### 响应式断点基线

全页面必须支持响应式布局，不允许只做桌面端静态展示。默认断点建议如下：

| 断点 | 范围 | 默认策略 |
| --- | --- | --- |
| `>= 1440px` | 大桌面 | 使用完整骨架，允许常驻右侧信息区 |
| `1200px - 1439px` | 桌面 | 保持左主右辅结构，必要时压缩右侧区宽度 |
| `768px - 1199px` | 中小屏 | 左侧导航可收起为图标轨道或移到顶部，右侧常驻区退化为抽屉或下层区块 |
| `< 768px` | 移动端 | 采用单列布局，顶部栏简化，所有右侧区块折叠进主流中 |

响应式强约束：

- 所有页面必须保证在 `360px` 到 `1600px` 区间内可读、可点、可滚动，不允许出现水平溢出依赖用户缩放。
- 小屏下优先保护主标题、主操作、主结果和关键边界信息，不重要信息必须折叠或后移。
- 顶部 chip、状态标签、按钮组在小屏下允许换行到多行，但单个标签内部仍不得换行。
- 表格在中小屏下允许横向滚动或转为卡片化摘要，但主操作和筛选不可因此消失。
- 登录页在小屏下必须转为单列布局：左侧辅助区上移或下沉为简版说明块，登录面板保持完整。

## 核心页面结构

### 内容页统一结构

桌面端内容页第一屏必须同时满足以下四件事：

1. 看见页面标题与当前场景
2. 看见主操作或主任务入口
3. 看见主结果、主状态或主指标之一
4. 看见当前权限、时间范围、数据状态等关键边界信息之一

如果一个页面需要滚动两屏以上才能同时理解“我在哪、我能做什么、当前结果是什么”，则说明页面主次结构设计失败。

### 内容页紧凑页头

除分析首页和登录页外，后台内容页默认应采用“紧凑页头”而不是大 Hero。

紧凑页头规则：

- 结构固定为：页面标题区 + 一行说明 + 同行主操作 / 状态标签。
- 总高度默认控制在 `64px` 到 `88px` 之间，不得为了说明文案再单独做大面积头图式白卡。
- 页面说明默认最多 `2` 行，超出的帮助说明、边界说明、使用提示应下沉到正文首卡或折叠区。
- 刷新、保存、创建、导出、返回等操作优先放在页头右侧同行，不应再单独占据第二层横幅。
- 审计、治理、连接策略、模板管理、合同审核、结果详情等后台页必须优先使用紧凑页头；只有分析首页和登录页可以保留较强的品牌导语区。

### 登录页

- 登录页结构必须以当前系统登录页为基线：左上品牌标识与文案、右侧登录面板、账号登录 / 企业微信登录切换、管理员信息、扫码区。
- 登录面板顺序固定为：登录标题 -> 登录方式切换 -> 表单或二维码 -> 登录按钮 -> 管理员联系方式 / 异常反馈。
- 登录页不允许出现与登录无关的大体量图表、伪仪表盘或多余营销区块。
- 登录页第一屏内必须完整呈现系统名称、至少一种登录方式和帮助路径。
- 登录页左侧可增加一块低饱和品牌辅助区，右侧登录面板必须保持现有系统登录流程和文案口径。

### 首页与智能分析工作台

首页和工作台应当表现为一套经营分析工作台，而不是单一巨大输入框。建议采用以下一级区块：

1. **页面头部摘要区**
   展示页面标题、当前角色、权限摘要、数据新鲜度、AI 服务状态和今日治理提醒，先建立边界与可信感。
2. **核心工作区**
   展示 AI 问数输入、快捷问题、继续追问、最近进行中的任务。问数入口仍然重要，但不再做唯一视觉主角。
3. **经营概览区**
   展示关键指标卡、趋势摘要、异常提醒和重点变化，让页面在未提问时也具有经营信息价值。
4. **资产复用区**
   展示常用查询、最近查询、推荐分析入口或模板治理反馈，强化“可复用的工作平台”而不是“一次性问答工具”。

首页与工作台减噪规则：

- 工作台首屏禁止出现大段解释性文案、重复能力说明和多层“页面用途介绍”。
- 入口支持范围、权限边界、数据新鲜度优先做成短标签、短说明或折叠提示，不得在主任务区展开成长段正文。
- 如果首页的主价值在图表、表格或关键指标上，则这些内容必须在首屏稳定可见，不得被宽大 hero、说明区或高占用侧栏挤到下方。
- 右侧信息区在工作台首页最多保留 `2` 个主卡片；其余信息应折叠、合并或延后到下屏。
- 工作台首屏总原则是“先让用户开始做事，再解释系统是什么”，不是“先解释系统，再让用户做事”。

### 结果页

结果页必须采用分区式解释结构，而不是简单堆叠摘要、图表和表格：

- 结果标题与问题上下文
- 结论摘要与关键判断
- 指标卡与主视图
- 明细表格或排行视图
- 口径、权限范围、时间范围、数据更新时间、来源说明
- 后续动作：继续追问、重跑、导出、复制链接

解释层必须与结果层并列存在，不能缩在页面最底部的小字说明中。

### 治理与审计页

治理与审计页应优先采用 Stripe Dashboard / Docs 的秩序感，而不是告警大屏或技术日志视图：

- 顶部先展示聚合摘要卡，先回答“今天发生了什么、哪里有风险、哪里需要处理”。
- 中部展示筛选区和事件列表，强调检索、比对和复核效率。
- 右侧或下层使用详情面板、抽屉或页内展开，承接事件详情。
- 风险、拦截、导出、降级、待复核必须有明确等级和状态语言，但整体页面仍需保持克制和专业。

### 合同审核页

合同审核页应当把“风险优先”和“阅读体验”同时拉齐：

- 上传入口与任务列表简洁清楚，不堆叠无关运营信息。
- 风险总结、总体结论和审核依据并列展示，但始终以风险结论优先。
- 合同正文与批注区域使用高可读浅色阅读面板，避免深色背景影响长文阅读。
- `规则快审`、`AI 补充审核`、`降级快审`、`阻断失败` 必须在标签、颜色、说明文案和状态摘要上同时区分。

### 企业微信输出体验

- 企业微信场景优先返回摘要块、关键指标块、结论块和必要的结果摘要块。
- 当完整结果不适合会话展示时，应明确提示用户前往 Web 工作台查看完整结果。
- 企业微信中的失败、拒答、补问、降级和超时提示必须与 Web 端保持同一语义体系，但表达更简洁直接。

## 页面级视觉改造基线

### 智能分析工作台

当前工作台不能只表现为“大输入框 + 结果白卡”。新版应形成“AI 问数 + 经营概览 + 可复用资产 + 可信边界”的组合。

- 页头使用紧凑经营状态条：包含当前角色、权限范围、数据更新时间、AI 服务状态和今日治理提醒。
- 问数区保留主输入，但输入区旁边或下方必须有快捷问题、常用模板、最近查询或继续追问入口。
- 未提问时展示经营概览，不得留出大面积空白；可使用关键指标、趋势摘要、异常提醒和推荐分析入口。
- 结果出现后，摘要、关键发现、图表、表格、口径边界和后续动作必须形成稳定阅读路径。
- 图形强化重点：AI 模块徽标、权限状态胶囊、迷你趋势线、关键发现图标、空状态插画、图表色板。

### 经营报表页

- 报表页应优先展示周期、负责人、部门、客户范围和更新时间，避免用户只看到孤立数字。
- 指标区必须区分核心经营指标、趋势指标、质量指标和异常指标，不能全部用同一种指标卡。
- 图表优先使用趋势线、排行条形、漏斗和矩阵表；每种图表必须有明确业务标题和口径。
- 经营异常应使用琥珀或风险色标签，但不能把整页做成告警大屏。

### 查询模板与模板治理页

- 模板治理页应参考成熟方案库结构，采用“左侧分类 / 中间列表 / 右侧详情或抽屉”的高效结构。
- 模板等级、来源、使用频率、更新时间、权限范围必须可扫描，不能只显示模板名称。
- 分类树、模板列表和详情区必须使用统一图标与等级色；分类、模板、规则、行业包不能混成同一种视觉。
- 导入、克隆、编辑、下线等操作必须明确主次，高风险操作进入确认层。

### 权限中心与连接策略页

- 权限中心应强调“谁拥有什么能力、当前是否生效、影响哪些入口”，而不是单纯配置表单。
- 角色、用户、部门、菜单、动作权限需要使用不同对象图标或标签，便于快速辨识。
- 连接策略页必须突出连接状态、同步健康、最近失败、回退方案和影响范围。
- 配置型页面使用紧凑页头 + 表格 / 列表 + 右侧抽屉，不使用大面积说明卡。

### 合同审核页

- 合同审核工作台首屏应突出上传入口、当前任务、审核模式和最近结果。
- 审核详情页必须形成“三层阅读”：总体结论、风险项清单、原文证据 / 审核依据。
- 风险项必须使用等级图标、风险色、影响条款和建议动作；不能只靠长文本说明。
- 合同正文应使用阅读面板和批注标记，风险批注与正文位置保持可对应。

### 审计中心

- 审计中心应表现为“事件复核工作台”，而不是日志表。
- 首屏先展示今日查询量、拦截量、导出量、敏感命中、待复核高风险等摘要。
- 审计事件列表必须同时显示事件类型图标、风险等级、行为人、对象、时间、处理状态和复核入口。
- 事件详情优先使用抽屉，展示原因、影响范围、请求上下文、审计证据和下一步动作。

### AI 配置页

- AI 配置页应把 Profile、模型、上下文策略、健康检查和切换历史分层展示。
- 当前激活配置必须有明显状态标记；测试失败、降级和未验证状态要有可执行提示。
- 技术字段可以保留，但必须放入诊断区或详情区，默认列表优先展示业务可理解的信息。

## 页面主次与交互显隐

### 页面主次规则

- 每个页面在第一屏只能有一个主标题、一个主任务焦点和一个主结果焦点，不允许多个区块同时争夺主视觉。
- 同一屏中，高强调色块或高饱和高亮区不应超过 `2` 个。
- 每个主工作区只允许存在一个实心主按钮；其余操作应降级为描边按钮、文字按钮或放入更多菜单。
- 主指标卡、主图表或主结论必须位于内容主列上半区，不能被次要说明、低频筛选或帮助模块挤到下方。
- 权限说明、时间范围、数据更新时间、审核模式等边界信息必须靠近主结果展示，而不是沉底。

### 首屏高度压缩规则

- 内容页首屏目标是优先暴露主任务与主结果，不得让说明性文案、低频帮助块和装饰性概述压缩主内容区高度。
- 页面一级导语、说明区、摘要横幅必须控制高度，默认建议如下：
  - 内容页头部标题区总高度建议不超过 `120px`
  - 摘要说明文字默认不超过 `2` 行
  - 非首页场景不鼓励使用大面积 hero 区块
- 指标卡在表格或主图表优先的页面中，应优先采用紧凑布局、左右布局或小卡片布局，避免把主表格挤到首屏以下。
- 筛选区应优先做成一行或两行紧凑布局，按钮与条件尽量同行，不允许把轻量筛选做成高占用大卡片。
- 低频说明、页面目的、设计解释类文字不得长期占据首屏主视觉，应删除、折叠或转入帮助说明。

### 显隐与降噪规则

- 次级解释、历史记录、低频筛选、附加说明优先使用以下方式降噪：
  - 右侧信息区
  - 右侧抽屉
  - 折叠区
  - 页内展开
  - 悬浮提示或帮助气泡
- 不允许把所有说明文字直接平铺在主区下方，导致页面失去主次。
- 复杂筛选默认只展示当前最必要的 `3` 到 `5` 项，其余条件进入“更多筛选”抽屉或折叠区。
- 长表格的低频列应优先考虑列显隐、行展开或详情抽屉，而不是无上限扩宽表格。
- 低频操作如“复制口径”“查看原始块”“下载结构化结果”应进入次级按钮、下拉菜单或详情区，不得抢占主操作区。

### 悬浮提示与帮助规则

- 悬浮提示只用于解释短概念、字段含义或状态差异，单条提示正文不应超过 `2` 行。
- 页面中的低频帮助说明、规则口径、字段释义和“为什么这样配置”类提示，默认优先使用 `ElTooltip` 或 `ElPopover` 这类悬浮提示承载，不应长期占据正文主区域。
- 复杂说明不得塞进 tooltip，必须进入抽屉、详情面板或页内说明区。
- 所有 tooltip 必须有明确触发点，不能让用户猜测某段文字是否可悬浮。
- 悬浮提示背景建议使用 `#0A2540`，文字使用 `#FFFFFF`，圆角 `10px`，内边距 `10px 12px`。
- 操作成功、失败、警告等瞬时反馈，优先使用浮层消息 `ElMessage` 或等价悬浮反馈，不应默认插入大面积常驻提示条挤占页面首屏。
- 页面中的“页面目的”“设计说明”“为何这样布局”这类只服务设计评审的文字，不应进入正式产品首屏；若保留，只能进入折叠帮助区或设计注释版本。

- 标签内容过长时，优先缩短文案、增大容器宽度或调整布局，不允许把标签压窄到自动换行。
- 标签本身应 `inline-flex` 居中显示，并设置 `flex-shrink: 0`，避免在紧凑布局中被压缩变形。
- 如果标签文案超过一行会影响视觉，则必须改文案，不允许保留“两行标签”。
- 所有带图标的标签、chip、页头状态块和摘要徽标必须使用 `inline-flex + align-items: center + gap: 6px` 到 `8px`，图标与文字共享同一水平线并垂直居中。
- 状态标签、chip 和带图标按钮默认必须同时具备背景、描边、文字 / 图标三层语义，不得只保留填充色而缺少描边，导致状态层级发虚。
- 标签与 chip 的图标尺寸建议 `14px` 到 `16px`，图标容器不得额外占用不稳定高度；禁止出现图标偏上、文案偏下或图标贴边的情况。

### 表格

| 项目 | 规格 |
| --- | --- |
| 表头高度 | `44px` |
| 默认行高 | `52px` |
| 紧凑行高 | `44px` |
| 单元格内边距 | `12px 14px` |
| 表头字体 | `13px / 20px`，字重 `600` |
| 正文字体 | `14px / 22px`，字重 `400` |
| 表头背景 | `#F7FAFF` |
| 行分隔线 | `#E6EBF1` |
| 行悬浮背景 | `#F8FBFF` |
| 行选中背景 | `#EEF4FF` |

表格规则：

- 表头在有局部滚动容器时必须支持吸顶。
- 关键主列允许吸附在左侧，但吸附列数量不得超过 `2` 列。
- 金额、比例、日期、状态列必须按可比较方式对齐，不允许杂乱排版。
- 低频字段优先进入详情抽屉，不鼓励把所有字段平铺到表格中。
- 表格空状态最小高度建议为 `160px`，必须同时说明“为什么为空”和“下一步能做什么”。
- 表格首列或主列应提供对象识别线索，例如对象图标、平台图标、类型标签、风险等级或状态点，避免列表只剩纯文字。
- 行内状态必须使用“文字 + 状态色 + 图标或状态点”组合，不得只显示 `ACTIVE`、`FAILED` 等内部枚举或只靠颜色判断。
- 表格操作列默认固定在右侧，操作数量超过 `2` 个时使用“更多”菜单收纳低频动作。
- 长列表优先提供紧凑筛选、列显隐、排序和详情抽屉；不得靠增加表格宽度解决所有信息展示。

### 抽屉与详情面板

| 组件 | 默认宽度 | 头部高度 | 底部操作区 |
| --- | --- | --- | --- |
| 标准详情抽屉 | `420px` | `64px` | 固定在底部，高度 `72px` |
| 扩展详情抽屉 | `520px` | `64px` | 固定在底部，高度 `72px` |
| 常驻右侧信息区 | `336px` | 与页面头部对齐 | 可无固定底部 |

抽屉要求：

- 抽屉标题、状态标签和关闭动作必须固定在头部。
- 抽屉内容独立滚动，背景页面锁定。
- 高风险确认、导出确认、写回确认必须在抽屉或确认层中展示原因与影响，不得直接瞬时执行。

下钻与详情统一规则：

- 所有“在当前页面上下文内继续查看详情”的二级内容，统一优先使用右侧抽屉承接，不再优先新开独立详情页。
- 适用场景至少包括：
  - 表格行详情
  - 审计事件详情
  - 合同风险项详情
  - 分析结果中的对象下钻
  - 模板、策略、连接配置等对象详情
- 下钻抽屉默认使用 `420px` 标准详情抽屉；当详情包含长文本、对比信息、批注预览或较多字段分组时，升级为 `520px` 扩展详情抽屉。
- 抽屉内容区默认内边距使用 `20px 20px 24px`；区块间距默认使用 `16px`。
- 抽屉头部必须固定，至少包含：标题、关键状态、关闭动作。
- 抽屉底部存在操作按钮时，底部操作区必须固定，正文区单独滚动。
- 桌面端详情下钻禁止用连续多层弹窗叠加；一层抽屉未关闭前，不得再开第二层同方向抽屉。
- 只有以下场景允许升级为独立页面，而不是抽屉：
  - 需要长时间阅读的全文内容
  - 需要复杂编辑或多步骤提交
  - 需要并排对比多个对象
  - 需要保留独立 URL 以便分享或刷新恢复
- 移动端和窄屏下，右侧抽屉退化为全屏详情层或底部上滑详情层，但信息结构、固定头部和固定底部操作区规则保持一致。

## 关键组件规则

### AI 问数模块

- AI 问数模块是核心功能模块，但不是首页唯一视觉中心。
- 模块默认应包含：输入框、支持范围提示、当前权限摘要、数据更新时间或服务状态、快捷问题或最近延续任务。
- 提交后必须进入清晰的过程反馈，展示补问、排队、执行、降级、完成或失败等阶段，不允许只显示一个抽象加载动画。
- 在布局上，输入区、状态区、快捷入口和最近任务应当是同一模块内的有序分层，而不是多个互相竞争的焦点。

### 指标卡与结果卡

- 指标卡优先展示当前值、比较基准、变化方向和说明标签。
- 结果卡优先展示结论标题、结论摘要、关键条件和可执行动作。
- 卡片的视觉重点来自层级、留白、对齐和少量强调色，不来自厚重阴影、渐变铺底或大面积高饱和背景。
- 同一屏内不允许出现大量风格差异过大的卡片，避免视觉系统失控。

### 表格、筛选与详情面板

- 表格默认采用浅底、高对比文字、细线边界和清晰行态，优先保证比较效率。
- 筛选区应当克制、轻量、易读，不得做成厚重工具条或复杂仪表盘。
- 同一业务上下文内的详情下钻默认优先使用侧边抽屉，页内展开只用于极短补充信息；不得把详情优先做成新页面。
- 明细区必须支持空状态、无权限脱敏、异常提示、导出限制提示和一致的分页行为。

### 风险与审核组件

- 风险级别主要依靠标签、标题、说明、分区和状态文案表达，颜色仅为辅助。
- 风险红只用于真正高风险、阻断和失败，不得铺满整页。
- 审核依据是可信说明的一部分，但不能压过风险结论本身。
- 降级快审、规则快审、AI 补充审核和阻断失败必须在视觉和文案层面同时区分，不允许混用模糊提示。

### 操作组件

- 主按钮使用品牌主色，强调明确主动作。
- 次按钮使用白底描边或浅底填充，保持克制但可见。
- 文本按钮和链接优先用于轻操作、查看详情和解释性跳转，不应承担高风险主动作。
- 导出、删除、阻断确认、写回确认等高敏感操作必须伴随说明、确认和结果反馈。
- 图标不是默认装饰物，不能为“看起来更丰富”而给每个按钮都加图标；但上传、下载、导出、刷新、筛选、搜索、返回、关闭、展开收起、删除、编辑、查看详情、风险确认和成功反馈等高频动作必须优先配图标。
- 同一操作区内应优先保证文字先可读，再决定是否需要图标；高频工具按钮和图标按钮必须配 tooltip，低频文字按钮可不加图标，避免图标泛滥造成噪声。
- 按钮图标与文字之间必须保留稳定间距，默认建议 `6px` 到 `8px`，图标尺寸建议 `14px` 到 `16px`；图标不得紧贴文字，也不得因为图标挤压按钮文案。
- 所有带图标按钮必须保证图标和文字在同一水平线、垂直居中、共享一致行高；禁止出现图标视觉中心高于文字、或文字被按钮高度挤压到偏下的情况。
- 表单输入框只有在能明显帮助字段识别时才添加前缀图标，优先用于账号、密码、搜索、上传等高频识别字段；普通业务字段默认不加前缀图标。
- 内容页默认优先使用 `36px` 工具栏按钮，只有主任务按钮或需要更强点击性的关键动作才升级到 `38px` 页面主按钮；不得在普通治理页、筛选区或摘要卡中滥用更高按钮尺寸。
- 只展示图标不展示文字的按钮必须满足三项要求：语义明确、具备 `aria-label` 或 tooltip、悬浮和聚焦状态清晰。
- 表格行内的“详情”“编辑”“克隆”“停用”等操作优先使用文字链接或文字 + 图标组合；危险操作不得仅用一个红色图标承载。

## 状态反馈与动效

### 状态反馈要求

所有关键组件至少覆盖以下状态：

- 默认
- 悬浮
- 聚焦
- 按下
- 禁用
- 加载
- 成功
- 失败
- 空状态
- 无权限
- 超时
- 降级

状态变化必须优先通过布局变化、标签、标题、说明文案和下一步操作建议来传达，颜色只是辅助，不允许只换颜色就视为状态设计完成。
- 在线、处理中、成功、提示、警告、降级、阻断 / 失败、离线 / 停用和中性元信息，必须作为统一状态语义层贯穿全系统；不得在不同页面把同一业务状态画成不同颜色。

### 动效规则

- 动效仅用于区块进入、数据刷新、面板展开、状态切换和提示出现。
- 建议使用 `120ms` 到 `220ms` 的短动效，以透明度变化和轻微位移为主。
- 禁止持续闪烁、长时间流光扫描、大面积脉冲、背景粒子和干扰表格阅读的循环动画。
- 加载态应当传递“正在处理什么”，而不是只传递“系统在忙”。
- 左侧导航展开 / 收起、右侧抽屉开合、页面切换进入、列表刷新、按钮提交都必须有明确过渡，不允许“瞬间跳变”。
- 页面切换建议使用轻量进入动画，推荐 `140ms` 到 `220ms` 的淡入 + `8px` 到 `16px` 轻位移，不得使用大幅飞入或缩放。
- 鼠标悬浮必须反馈可点击性，至少通过背景、边框、阴影、颜色或位移中的一项变化体现。
- 鼠标按下必须有按压反馈，不能只有 hover 态。
- 交互反馈必须服从业务优先级：主按钮最明显，次按钮适中，低频操作克制。

### 请求与加载反馈

- 所有会触发请求的操作都必须显式进入加载状态，包括但不限于：
  - 登录
  - 提交问数
  - 应用筛选
  - 导出
  - 上传合同
  - 打开详情抽屉
- 加载状态至少包含以下要素中的两项：
  - 按钮进入 loading
  - 文案从静态动作改为进行中动作
  - 局部 skeleton 或占位块
  - 结果区域的刷新提示
- 禁止用户点击后界面无变化；如果响应时间可能超过 `300ms`，必须立即给出可感知反馈。
- 列表刷新、筛选重载、详情切换优先使用局部加载，不得用全页遮罩打断整体操作。

## 数据展示与可解释性

- 结果展示遵循“结论优先、证据随后、边界持续可见”的原则。
- 关键指标、图表、表格、明细和导出内容必须共享同一组条件与口径，不允许相互矛盾。
- 时间范围、筛选条件、权限范围、数据更新时间、审核依据等信息属于主信息结构的一部分，不能缩成难以发现的脚注。
- 对管理层用户，应优先展示摘要、变化和风险；对分析型用户，应明确提供继续下钻、筛选、重跑和追问路径。
- 图表应减少无意义装饰，优先强调趋势、对比、排序和异常点。
- 表格和图表必须能互相印证，必要时提供文本摘要或数据表兜底。

## 治理审计与风险表达

- 审计页面强调“事件治理工作流”，而不是纯技术事件流。
- 聚合摘要卡优先回答今日查询量、拦截量、敏感字段命中、导出量和待复核风险事件数。
- 事件列表应便于筛选、复核和上下文跳转，不得退化成只供技术人员阅读的日志表。
- 风险表达建议遵循统一等级：`阻断` > `高风险` > `待复核` > `提示`。
- 每一级风险都应同时展示标签、标题、原因、影响范围和建议动作。
- 治理页面的专业感必须来自信息组织和处理路径，而不是来自大量红色或高压视觉噪声。

## 无障碍与可读性

- 正文与背景对比度必须满足 WCAG AA。
- 正文、筛选项、标签和表格都应有明确的最小字号下限，避免将 Stripe 风格误用为“小字加浅色”。
- 颜色不能成为唯一状态信号，风险、成功、处理中、降级和无权限都必须同时使用文字或图标说明。
- 键盘焦点必须清晰可见，表单、上传、筛选、导出、提交和抽屉操作都必须支持键盘完成核心路径。
- 图表必须提供文字摘要或表格兜底，确保截图传播、审计复核和低可视环境下仍然可理解。
- 合同正文、风险说明、治理详情等长文本区必须优先保证阅读舒适度，而不是追求视觉戏剧性。

## 文案语气

- 面向业务用户的文案统一使用中文，语气专业、直接、明确。
- 补问文案应简洁说明“还缺什么条件”，而不是泛泛要求用户“补充更多信息”。
- 拒答、无权限、降级和失败提示必须说明原因与下一步，不责备用户。
- 合同审核结果必须明确区分正式审核、规则快审、降级快审和阻断失败，不允许使用含混措辞。
- 企业微信文案可以更简短，但必须和 Web 端保持同一业务语义与边界口径。

## 前端实施要求

- 新页面默认采用浅色主框架，不得无理由回退到全页深色或深色外壳。
- 品牌渐变只能作为重点装饰、入口强调和局部高光，不能充满背景层。
- 页面应优先复用统一卡片、表格、标签、提示、状态、抽屉和表单语法，不鼓励为单个页面重新发明样式体系。
- 首页、分析页、治理页和合同审核页可以拥有不同信息密度，但必须共享同一色彩逻辑、排版体系、间距节奏和状态语言。
- 后续前端组件库统一选用 `Element Plus`，并配套使用 `@element-plus/icons-vue` 作为默认图标来源；不得继续用文字首字、临时字符或无语义装饰替代关键导航、操作、状态和风险图标。
- Element Plus 只能作为基础交互组件与图标能力来源，不能直接套用默认主题覆盖本文件定义的色彩、圆角、间距、状态语义和页面骨架；必须通过主题变量、全局 CSS 覆盖或本项目封装组件保持当前浅色、克制、可信的工作台风格。
- 重构现有页面时应按“基础组件接入 -> 主题变量覆盖 -> 页面逐步替换 -> 交互状态与图标补齐 -> 回归验证”的顺序推进，禁止一次性大面积替换导致功能绑定、权限说明、审计留痕或响应式结构失控。
- 图标使用必须服务于识别和状态说明，优先覆盖左侧导航、主操作、加载状态、成功/失败/警告/风险标签、上传区、导出、刷新、筛选、搜索、返回、关闭、展开收起和审计事件类型；不得使用与业务语义不匹配的装饰图标。
- 每个新页面上线前至少检查以下问题：
  - 是否清晰展示当前权限和边界
  - 是否补齐加载、失败、空状态、无权限、降级和超时反馈
  - 是否保持摘要、图表、表格和导出的一致性
  - 是否仍符合“浅色主导、克制可信、Stripe 化工作台”的总方向
  - 是否具备至少一个业务视觉锚点，而不是纯白卡片堆叠
  - 是否正确使用模块辅助色、状态色和图表色板，避免单色或乱色
  - 是否补齐关键图标、对象图标、状态图标和空状态图形
  - 是否在 `360px`、`768px`、`1200px`、`1440px` 等宽度下保持文字不重叠、按钮不溢出、标签不换行

### 设计评审检查清单

设计稿、原型或前端页面提交评审前，必须按以下顺序自查：

1. **业务识别**：不看导航，只看主内容区，是否能判断当前页面属于哪个业务模块。
2. **主次关系**：第一屏是否只有一个主任务、一个主结果焦点和一个主操作。
3. **色彩控制**：是否使用模块辅助色 + 品牌主色 + 状态色，是否存在无语义色值。
4. **图标图形**：导航、主操作、状态、空状态、风险和对象类型是否有足够图形识别。
5. **数据表达**：关键数字是否有单位、口径、时间范围或对比基准；图表是否能与表格互相印证。
6. **文字层级**：页面标题、区块标题、正文、辅助说明和标签是否有清晰层级。
7. **交互状态**：悬浮、聚焦、按下、加载、禁用、成功、失败、空状态、无权限和降级是否完整。
8. **响应式稳定**：窄屏是否可读可点，表格是否有局部滚动或卡片化策略。
9. **低频降噪**：说明、口径、诊断信息、低频操作是否被收纳到 tooltip、Popover、折叠区或抽屉。
10. **业务边界**：权限、数据范围、审计留痕、降级原因和下一步动作是否可见且不暴露内部技术细节。

## 后续实施约束

- 新增或重构 UI 页面前，必须先检查本文件、主规格和对应 OpenSpec 变更。
- 若只是调整视觉风格，不需要沿用旧 `.pen`；若要新增新的可视化原型，应输出新的 `.pen` 文件并与本设计系统保持一致。
- 若变更页面功能边界、核心入口、接口字段、数据白名单或验收场景，必须同步检查 `specs/001-crm-intelligent-analytics/` 下相关文档。
- 所有页面与接口命名、状态枚举和字段含义必须继续与主规格和接口契约保持一致，不允许出现视觉系统已经切换，但业务语义与接口口径分离的情况。

## 结语

本设计系统的目标不是把 CRM 智能分析系统做成 Stripe 的外观复刻版，而是借用 Stripe 的清晰、可信和克制，重构一套更适合经营分析、治理审计和 AI 辅助决策的企业级工作台视觉语言。后续所有新页面、新组件和新交互，都应以此为共同基线。
